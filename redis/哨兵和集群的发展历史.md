Redis 的高可用与分片方案，是分两步演进的：**先有哨兵（Sentinel），后有集群（Cluster）**。下面按时间线和架构设计的思路，为你完整还原这段历史。

---

## 一、最早期的 Redis：单机 + 手动复制（2009–2010）

Redis 在 2009 年刚诞生时，只是一个单机内存数据库，连主从复制都没有。  
到 **2010 年的 1.2 版本**，Redis 才引入主从复制，支持读写分离和数据备份。但此时存在一个很明显的问题：

> **如果主节点挂了，怎么办？**

当时社区的普遍做法是：
- 手动把一台从节点提成主节点；
- 用 Keepalived 做 VIP 漂移；
- 自己写脚本监控，做切换。

这些方法缺乏官方支持，而且很难保证切换的可靠性和一致性。**自动故障转移**成为 Redis 的第一个重大架构缺口。

---

## 二、哨兵的出现：先解决“高可用”，不碰“分片”（2012–2013）

### 1. 为什么 Redis 团队要自己做一个哨兵？
2012 年，**Salvatore Sanfilippo（antirez）** 认为，Redis 已经有很多生产部署在用主从复制，但缺乏一个官方、可靠、分布式的自动故障转移方案。  
外部工具（Keepalived、ZooKeeper 等）要么太简单，要么太重，要么会把 Redis 的运维复杂度提高几个量级。  
于是 antirez 设计了一个**内置于 Redis 生态系统**的组件：**Redis Sentinel**。

### 2. 哨兵的设计思想
Sentinel 的设计目标非常纯粹：
- **只解决高可用，不解决数据分片**；
- 独立于 Redis 服务器进程运行，是一个专门做监控和故障转移的“守护者”；
- 必须**去中心化**，用多个 Sentinel 节点组成分布式系统，避免单点故障；
- 使用类似 Raft 的**选举算法**来决定哪个 Sentinel 来执行故障转移；
- 同时充当**服务发现**的角色，客户端可以从 Sentinel 拿到当前的主节点地址。

架构图感：
```
 客户端 → Sentinel 集群（多个 Sentinel 进程）
               ↓ 监控、故障转移
         主从复制组（1 主 + N 从）
```

这样，Redis 使用者只要部署一套 Sentinel，就能在主节点宕机时自动将从节点提升为新的主节点，并通知客户端更新连接。

### 3. 开发与发布节奏
- **2012 年**：antirez 在博客公布 Sentinel 的设计草案。
- **Redis 2.4 时期**：开始有一些早期实验性代码。
- **Redis 2.6**（2012 年 10 月）：Sentinel 作为实验特性首次出现，官方不建议生产使用。
- **Redis 2.8**（2013 年 11 月）：**Sentinel 基本成熟稳定**，可以用于生产环境。

至此，Redis 的单主从架构终于有了一个官方、可靠的高可用方案。

## 三、数据量暴增：单机瓶颈催生“Redis 集群”需求（2011–2015）

### 1. 哨兵解决了高可用，但容量问题依然无解
哨兵 + 主从复制模式下，**所有写请求和全量数据还在一个节点上**。  
当数据量超过单机内存、写入 QPS 超过单 CPU 能力时，这套架构就无能为力。唯一办法是把数据**水平切分**到多台机器上，也就是需要“集群”。

### 2. 当时已有的非官方方案
在 Redis 官方推出集群前，社区已经在用各种方案做分片：
- **客户端分片**：在客户端自己按 key 做 hash 路由，不同的 key 发到不同的 Redis 实例。缺点是逻辑全在客户端，非常笨重；
- **中间代理分片**：Twemproxy（Twitter 开源）、Codis（豌豆荚开源）等，增加一层代理层来做路由和扩缩容。虽然屏蔽了客户端复杂度，但引入了额外的代理组件和维护成本。

Redis 团队分析后认为：
- 代理会多一层网络开销和单点/集群复杂度；
- 客户端分片对使用方侵入太大；
- 一个**官方、无中心、高性能、内建高可用的集群方案**才是最终解。

### 3. Redis Cluster 的设计哲学
早在 **2011 年**，antirez 就发表了《Redis Cluster: a first spec》，开始公开讨论集群设计。  
整个设计的目标可以总结为：

- **去中心化，无代理**：客户端直接与 Redis 节点通信，不依赖中间层；
- **哈希槽分片**：将整个 key 空间划分为 16384 个哈希槽，每个节点负责一部分槽，key 的归属由 `CRC16(key) % 16384` 决定；
- **节点间 Gossip 协议**：用于传播节点状态、槽位分配信息，完全去中心化，没有类似 ZooKeeper 的外部协调器；
- **自动故障转移内建于集群中**：集群节点互相监控，当发现主节点宕机，从节点可以自动发起选举并接管，这个过程不依赖外部 Sentinel；
- **客户端重定向（MOVED / ASK）**：当客户端向错误节点发请求时，节点会返回正确的节点地址，客户端需实现重定向逻辑（智能客户端）。

这个设计的精髓在于：**把复杂度尽量封装在服务端和客户端协议里，而对运维和中间件尽量做减法**。Redis Cluster 既做到了水平扩展，又把 Sentinel 的能力“内置”了进去。

### 4. 漫长而谨慎的开发过程
- **2011 年**：发布设计草案。
- **2012–2014 年**：长时间在 `unstable` 分支开发和迭代，反复修改哈希槽迁移、节点选举、Gossip 通信等细节。
- **2015 年 4 月 1 日**：**Redis 3.0.0 正式发布**，宣告 Redis Cluster 进入生产可用状态。这个版本整整打磨了近 4 年，antirez 在发布时特别强调“经过大量的测试与生产验证”。

## 四、两者定位的区分与并存（2015–至今）

Redis 团队从来没有用 Cluster 去“取代” Sentinel，两者的定位完全不同：

| 方案 | 适用场景 | 数据容量 | 高可用方式 |
|------|--------|--------|----------|
| **Sentinel + 主从** | 单分片，数据量可被单机容纳 | 受单机内存限制 | 外部 Sentinel 集群 |
| **Redis Cluster** | 需要水平扩展，超单机内存/吞吐 | 多节点分片 | 集群节点内建故障转移 |

直到现在（2026 年），这两种方案依然长期共存，很多业务如果数据量不大，直接一套 Sentinel + 主从就够用了；当数据规模突破单机限制时，才会迁移到 Redis Cluster。

---

## 总结：Redis 团队的“两步走”架构演进

1. **早期**：单机 → 主从复制，但缺官方高可用；
2. **先迈出一步（2012–2013）**：用 **Sentinel** 解决自动故障转移，让主从架构变成真正可用的高可用方案，但数据容量仍是单机瓶颈；
3. **再迈一步（2011 设计，2015 发布）**：推出 **Redis Cluster**，使用无中心的哈希槽分片 + 内建故障转移，一步到位解决水平扩展和高可用问题。

整个演进中，Redis 团队一直坚持 **“简单、高性能、自包含”** 的理念：尽量不依赖外部组件，把分布式复杂度吸收到 Redis 自身的协议和实现中。这种设计哲学使 Redis 在今天依然是全球最流行的内存数据库。
