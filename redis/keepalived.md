好的，我们先聚焦 **Keepalived**。

你在 Redis 的早期高可用方案中提到它，是因为在 Sentinel 出现之前，很多运维人员确实就是用 Keepalived 来给 Redis 主从做“自动”故障转移的，虽然它原本并不是为 Redis 设计的。

---

## Keepalived 是什么？

**Keepalived** 是一个 Linux 系统下的轻量级高可用软件，最初是为 **LVS（Linux Virtual Server，Linux 虚拟服务器）** 设计的，主要用来监控 LVS 后端真实服务器的健康状态，后来演变为一个通用的 VRRP 协议实现，常被用来给任意单主服务（如 Nginx、MySQL、Redis 主从）提供一个**虚拟 IP 的自动漂移**能力。

简单说，它的核心功能是：**让多台机器共享一个虚拟 IP（VIP），当主机器宕机时，备用机器能自动接管这个 IP，从而实现故障转移，对客户端来说几乎无感知。**

---

## Keepalived 的核心原理：VRRP 协议

Keepalived 工作的基石是 **VRRP（Virtual Router Redundancy Protocol，虚拟路由冗余协议）**，这是一个标准的 IETF 协议（RFC 3768）。VRRP 本质上解决的是**局域网内默认网关的高可用问题**，但 Keepalived 把它扩展到了应用层。

### VRRP 是如何工作的？
- 多台机器组成一个“VRRP 组”，组内通过选举产生一台 **Master（主用）** 和若干台 **Backup（备用）**。
- 这个组对外共用一个**虚拟 IP（VIP）**和一个虚拟 MAC 地址。
- Master 会定期（默认每秒）发送 **VRRP 通告报文（心跳）** 给组播地址（224.0.0.18），告诉 Backup “我还活着”。
- 如果 Backup 在连续几个周期（通常3秒）内收不到 Master 的心跳，就会认为 Master 挂掉，于是触发**选举**，拥有最高优先级的那台 Backup 会将自己提升为新 Master，并在自己本机网卡上接管（绑定）这个 VIP，同时发送免费 ARP 更新局域网内的交换机/路由器，让流量切过来。

### 优先级与抢占
- 每台机器都有一个优先级（priority），范围 0–255，越大越优先。
- 通常会让期望成为 Master 的机器优先级设高一点（比如 100），备机设低一点（比如 90）。
- 如果开启了 `nopreempt`（非抢占模式），那么即使网络恢复后原来的 Master 重新上线，也不会抢回 VIP，避免来回切换。

## Keepalived 如何用于 Redis 主从架构？

在没有 Sentinel 的年代，典型的 Redis 主从 + Keepalived 架构如下：

```
              客户端
               │
            VIP: 192.168.1.100
               │
      ┌────────┴────────┐
   Master             Backup
  (Redis主)          (Redis从)
  192.168.1.10       192.168.1.20
  优先级 100          优先级 90
  运行 Keepalived     运行 Keepalived
```

- **VIP 始终挂在当前的 Redis 主节点上**，所有写请求通过 VIP 进来。
- 当主节点宕机（比如 Redis 进程崩溃或整机挂掉），Keepalived Master 不再发心跳，备机上的 Keepalived 检测到后，优先级高的备机（如果配了多台）会接管 VIP，成为新的 Master。
- **问题来了**：此时只漂移了 IP，**备机上的 Redis 仍然是 Slave 角色，需要手动将它提升为 Master**。因此通常还要配合一个 **`notify_master` 脚本**，在 Keepalived 切换到 Master 状态时，自动执行 `SLAVEOF NO ONE` 命令，把本机 Redis 提主。同样，原主恢复后，可能需要脚本把它变成 Slave 重新加入。
- 同时还要考虑数据一致性：如果原主宕机时有一部分数据还没同步到从库，那么提主后这些数据就丢了，且原主恢复后需要正确处理冲突（通常直接丢弃旧数据，做全量同步）。

## Keepalived 方案的优缺点（与后来的 Redis Sentinel 对比）

| 角度 | Keepalived 方案 | Redis Sentinel 方案 |
|------|----------------|---------------------|
| **心跳检测** | 基于 VRRP 组播心跳，只能判断机器存活（网络通不通、进程在不在），无法感知 Redis 内部状态（如主从复制是否正常、是否即将死锁） | Sentinel 直接与 Redis 实例通信，周期性发送 PING，能获取详细的 INFO 信息，判断更准确（比如主观下线和客观下线） |
| **故障转移** | 单纯 VIP 漂移，需要额外脚本执行 SLAVEOF 命令。切换逻辑依赖运维人员编写的脚本，健壮性差 | Sentinel 内建完整的故障转移流程：选出新主、通知其他从库复制新主、更新配置、通知客户端，全自动化且经过充分测试 |
| **选主逻辑** | 无 Redis 层面的“最优从库”选择，通常只能按预设优先级选一个固定的备机，可能选到数据落后的从库 | Sentinel 会根据复制偏移量、运行状态等多个条件，选出数据最新、最健康的从库作为新主 |
| **集群复杂度** | 需要自己维护脚本，且 VIP 绑定在单点上，无法分布式仲裁，容易出现脑裂（如果心跳网络隔离，两台机器都可能认为自己是 Master，都绑 VIP） | Sentinel 是分布式系统，使用多节点投票（客观下线）和选举算法，能有效防止脑裂，并保证只有一个 Sentinel 执行故障转移 |
| **客户端发现** | 客户端直连 VIP，无法动态获知新主地址（切换后 VIP 不变，这一点其实对客户端透明，是优点；但无法做读写分离的精确控制） | Sentinel 充当服务发现，客户端可以问 Sentinel 获取当前主从地址，支持更复杂的路由策略 |

**为什么最终 Redis 官方抛弃了 Keepalived 路线，自研了 Sentinel？**
- Redis 作为有状态的数据库，高可用不仅仅是 IP 漂移，必须结合数据复制状态做决策。
- Keepalived 的检测太底层，无法理解“主从复制断开”“从库数据太旧”这类领域知识。
- 大规模部署时，需要一套**分布式、去中心化**的监控裁决系统，而 Keepalived 只是单对单的心跳，不适合多个从库的复杂选举场景。
- 官方希望提供一个开箱即用、与 Redis 深度融合的完整方案，而不是让每个用户自己去拼凑脚本。

这就是 Keepalived 的完整面貌。它在特定场景（无状态服务或简单主备）下仍然非常优秀，但在数据库高可用上，Sentinel 和后来的 Cluster 是更专业的选择。
