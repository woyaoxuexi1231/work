# 技术点一：Linux 本身是什么，如何跑起来的？

## 1. 苦难时代：没有操作系统，你就是人肉调度器

想象一下没有操作系统的年代。你有一块电路板，上面有 CPU、内存条、硬盘。你想让电脑做点事，比如算个加法，你就得用机器码写一堆指令，直接告诉 CPU：去这个地址取数，加那个地址的数，存回另一个地址。如果你还想同时放个音乐，那更惨——你得自己写程序在两个任务之间来回切换，手动保存 CPU 状态。那时候，每个程序员都是人肉操作系统，效率极低，而且换个硬件，程序就得重写。

这就是“裸机编程”的蛮荒。所有硬件资源都在你面前，但没有管家，一切都要你亲力亲为。

## 2. 思想破局：给硬件雇一个大管家

操作系统的本质，就是一个**永远在运行的大管家**。你把他想象成一栋大楼的物业经理。

- **CPU** 是楼里的工人（计算力）。物业经理负责安排工人干活：谁先用，用多久，时间到了就换下一个（进程调度）。
- **内存** 是大楼里的储物柜。物业经理负责分配：张三家几个柜子，李四家几个，不能互相乱翻（内存管理）。
- **硬盘** 是大楼的地下仓库。物业经理负责存取东西，还得记着哪个文件放哪个货架（文件系统）。
- **网卡、显卡、键盘鼠标**，都是大楼的对讲机、窗户、门铃。物业经理提供统一的通话方式（驱动），你不用直接去拨弄电线。

Linux 就是这个物业经理。它在计算机通电后，由引导程序加载到内存，然后**抢过 CPU 的控制权**，从此再也不放手。它成了一个永恒的中间层：上面运行的任何程序（比如你的 Python 脚本），都不能直接去碰硬件，必须通过 Linux 内核提供的“窗口”来申请服务。这个窗口，就是**系统调用**。

## 3. 原理浅剖：内核空间与用户空间

Linux 把世界一分为二：

- **内核空间**：大管家自己的办公室。这里拥有最高权限，能直接操作硬件、管理内存。普通程序禁止入内。
- **用户空间**：给住户（普通程序）用的区域。每个住户有自己的小隔间（进程），以为自己独占世界，但实际上每次想用硬件，都得敲管家办公室的窗户，喊一嗓子：“管家，帮我读个文件！”（系统调用 `open`, `read`）。

管家接到请求，自己去仓库取货，然后把结果递给住户。这样，一个住户就算发疯，也只能砸自己的隔间，砸不到管家的办公室，更弄不坏大楼。

## 4. 血泪案例：没有内存保护的单片机

我年轻时玩过一个单片机，没有 MMU（内存管理单元），所有程序共用一块物理内存。我写的一个循环意外写穿了数组边界，直接覆盖了另一个任务的代码区，整个系统死得莫名其妙。后来调试半天才发现是内存写穿。而 Linux 这样的系统，用 MMU 给每个进程虚拟地址空间，你的程序就算数组越界，也只会触发 Segment Fault 杀掉自己，绝不连累其他进程。这就是管家对大楼的保护。

## 5. 架构反思：一切隔离的源头

理解 Linux 是管家，后面的一切就好懂了。**所有容器技术的隔离，都是管家内部提供的“特殊服务”，而不是在管家之外再雇一个管家。** 这个认知是我们进入 Namespace 和 Cgroups 的钥匙。

# 技术点二：Linux 的“障眼法”和“紧箍咒”是专门为 Docker 准备的吗？

## 1. 苦难时代：内核里长期存在“共享太多，隔离不够”的痛

远在 Docker 2013 年横空出世之前，Linux 内核社区就已经被一个问题困扰很久了：**怎么把一台物理机安全地切成几份，租给不同的人用，而互不影响？**

这需求最初来自超算中心、虚拟主机服务商。比如一个大学买一台大型机，要分给二十个实验室，每个实验室都想当 root，装自己的软件，还不希望隔壁实验室搞崩系统。传统的做法是加一层虚拟机，但那太重了，开销大。

于是内核社区的大神们开始琢磨：能不能不跑两个内核，只用一个内核，却能给每个用户一种“我有独立系统”的错觉？这就催生了两个机制：**Namespace** 和 **Cgroups**。它们完全是通用目的的设计，跟容器没有半点关系，只是后来被 Docker 捡起来用到了极致。

## 2. 思想破局：不盖新楼，而是在同一栋楼里加隔断和分电表

我们再回到那栋大楼。

- **Namespace（障眼法）**：就是**隔板、毛玻璃、门牌号**。物业经理可以在一个大平层里，用轻质隔板分出多个小办公室。你在小办公室 A 里，能看到窗户、书架，但看不到隔壁的小办公室 B。你大吼一声“我是 root！”，也只在小办公室内有回响。
- **Cgroups（紧箍咒）**：就是**分户电表、水表、限流阀**。物业给你房间装个独立电表，你可以用，但设定了上限 10A。你一开热得快，电流超了，电表跳闸，只停你这间。不会把整栋楼的保险丝烧了。

这些机制不是给 Docker 发明的，而是内核自己在问：**“我作为一个管家，为什么不能提供更精细的租户管理服务呢？”** 早在 2006 年，Google 就贡献了 Cgroups 的初版（当时叫进程容器 process containers），用于限制他们搜索爬虫等任务的资源。2008 年合并进内核主线。Namespace 也是类似，UTS Namespace 2006 年就有了，PID Namespace 2008 年引入，User Namespace 2013 年才完善。

所以，**Docker 是那个天才的装修公司，它发现大楼已经预留了隔间框架和分电表接口，于是它设计了一套标准化的“精装小公寓”，让任何人都能一键入住。**

## 3. 原理浅剖：内核如何实现“隔间”

Namespace 的实现，是在内核的数据结构里给进程打上标签。比如 `struct task_struct`（内核里代表进程的结构体）里有个 `nsproxy` 指针，指向一组 Namespace 结构。当你 `clone()` 一个新进程，并传入 `CLONE_NEWPID | CLONE_NEWNET` 等标志时，内核不会给这个新进程复用父进程的 Namespace，而是创建一套新的，装进去。这样，新进程看到的 PID 列表、网络设备列表就是私有的。

Cgroups 则是通过一个虚拟文件系统（`/sys/fs/cgroup`）来控制的。你创建一个目录，内核就自动在里面生成一堆配置文件。你往 `memory.max` 写个数字，内核就记录了这个资源限制，并在这个 cgroup 内所有进程的上下文中生效。这种“文件即配置”的设计，让任何用户态工具都能通过读写文件来管理资源，非常优雅。

## 4. 血泪案例：LXC 的蛰伏与 Docker 的爆发

其实在 Docker 之前，已经有工具试图打包这些内核特性来用，叫做 LXC（Linux Containers）。但 LXC 太复杂了，它没有镜像的概念，你得自己装系统、配网络，跟管虚拟机差不多累。所以它一直不温不火。

Docker 的出现，用“镜像分层”和“一行命令启动”彻底降低了门槛。人们用上 Docker 之后才惊叹：“原来内核早就有这本事了！” 这个案例告诉我们：**伟大的技术，一半功劳在于实现，一半在于把复杂变成简单。**

## 5. 架构反思：内核是武器库，Docker 是精装兵器谱

Namespace 和 Cgroups 从来不是为 Docker 而生。它们是 Linux 内核在漫长演进中，为“资源隔离”这个普适需求锻造的通用武器。Docker 是那个把这些武器组合起来，打造出一套“一键开箱”兵器谱的人。这个关系，就像物理定律和火箭技术：定律从来就在那里，火箭是把它们工程化到极致。

# 技术点三：Docker 又是如何跑起来的？

## 1. 苦难时代：安装环境是连环噩梦

我们再回到那个让你抓狂的场景：你做了一个 Web 应用，用 Python 3.10 + Django + PostgreSQL 驱动 + Redis + Pillow 图像库。你要把它部署到服务器。服务器是 CentOS 7，Python 还是 2.7。你在上面编译 Python 3.10，缺依赖；装依赖，又冲突；好不容易搞好了，Pillow 的图像编解码库版本又不对。整整两天就搭环境了。每个新环境都是一场灾难。

这就是“环境地狱”。人们发明了配置文件管理工具、虚拟机、打包工具，但都无法实现“一次打包，到处运行”。

## 2. 思想破局：把厨房打包成快递

Docker 的创始人 Solomon Hykes 当时在做一个 PaaS 平台 dotCloud，饱受环境不一致之苦。他想：**我们能不能学物流业？不再寄菜谱，而是直接把做好的菜，连盘带锅，原封不动寄过去？**

于是 Docker 做了一件革命性的事：**定义了镜像（Image）这个标准。** 镜像就是一个自包含的、不可变的文件包，里面装着一个完整的最小化文件系统 + 你的应用 + 所有依赖。然后 Docker 利用 Linux 的 Namespace 和 Cgroups，为每个镜像启动的实例，创建一个几乎感觉不到负担的隔离环境——也就是容器。

Docker 的工作流程可以这样理解：
1. **你写一份 Dockerfile**：这是“厨房打包清单”。里面写：“从 Ubuntu 20.04 底料开始；跑 apt-get install python3；复制我的代码；设定启动命令。”
2. **`docker build`**：Docker 按照清单，一层层地把材料抓下来、装好，最后压成一个镜像文件。这个文件可以搬到任何地方。
3. **`docker run`**：到达目的地后，Docker 找到这个镜像，呼叫 Linux 内核：“大哥，请按这个镜像开一个隔离舱，CPU 限 1 核，内存限 512M，只给它看自己的文件系统，网络给个独立 IP，然后启动里面的进程。” Linux 内核照办，一个容器就诞生了。

## 3. 原理浅剖：Docker 引擎的三位一体

Docker 本身是一个 C/S 架构的程序，包含：
- **客户端（docker CLI）**：你敲的命令。
- **守护进程（dockerd）**：在后台运行的管家，监听命令，管理镜像、容器、网络。
- **containerd 与 runc**：这是后来从 Docker 剥离出来的、更专注的容器运行时。runc 是真正去调用内核 Namespace/Cgroups 的“最后一道工序”。

当你 `docker run` 时，大概流程：
1. dockerd 收到请求，先检查本地有没有镜像，没有就拉。
2. 把镜像的层解压、合并，准备好一个 rootfs 目录。
3. 调用 containerd，containerd 再调 runc。
4. runc 用 `clone()` 系统调用创建新进程，附带各种 Namespace 标志，配置 Cgroups，执行 `pivot_root` 把文件系统根切到准备好的目录。
5. 最后执行你指定的命令（如 `nginx -g daemon off`）。

容器进程就在内核的严密监护下跑起来了。它在里面看，自己是 PID 1，有独立的 /etc/hostname，有独立的网卡 eth0。但一回头，内核微笑看着它，知道它只是众多隔离小房间中的一个。

## 4. 血泪案例：权限不足的端口绑定

有个新手常犯的错误：他在 Dockerfile 里写 `EXPOSE 80`，然后 `docker run -p 8080:80`，以为这就万事大吉。但容器内应用启动时却因为非 root 用户无法绑定 80 端口而崩溃。为什么？因为 `EXPOSE` 只是个文档性声明，并不改变容器内权限。容器里的进程还是受容器内 Linux 用户权限限制。你需要在容器内用非特权端口，或者给容器添加 `NET_BIND_SERVICE` 能力。这个教训让我们明白：**Docker 只是隔离，它不改变你应用运行的基本规则。** 容器里还是那个真实的 Linux 环境，只是被限定了视角和资源。

## 5. 架构反思：标准化的伟大胜利

Docker 真正的贡献，是**标准化**。它定义了一种通用的打包格式（OCI 镜像）和运行时接口。从此，任何工具，只要符合这个标准，都能跑 Docker 镜像。Kubernetes 后来才得以轻松地调用 containerd 或 CRI-O 来启动容器，而不必绑定 Docker 本身。

现在，Docker 之于容器，就像 USB 接口之于外设。你没准要关心 USB 里面的电路，只要插上就能用。Docker 把 Linux 内核那些复杂的隔离能力，封装成了一个“USB 插口”，让我们这些开发者，再也不用和内核的参数搏斗，只需要写写 Dockerfile，整个世界的服务器就都成了你的计算机。

好了，这三个问题讲完了。现在我们有了一个清晰的逻辑链：

- **Linux** 是那栋大楼的物业经理，管理所有硬件。
- **Namespace 和 Cgroups** 是物业经理本来就有的“隔间”和“电表”能力，不是专门为 Docker 发明，但 Docker 把它们用活了。
- **Docker** 是一个天才的精装修公司，它把“隔间”和“电表”打包成标准化的小公寓（镜像），然后一键入住。
