# RSA 学习项目 v1 ~ v4 升级过程完整解析

## 这份文档是干什么的

这 4 个项目不是 4 个互不相干的例子，而是一条**循序渐进的学习路线**：

- `v1`：先把最小闭环跑通，先知道“为什么要 RSA + AES 混合加密”
- `v2`：在 v1 基础上补“完整性保护”，解决“密文可能被篡改”的问题
- `v3`：在 v2 基础上补“IV”这个概念，解决 ECB 模式本身不安全的问题
- `v4`：在 v3 之上继续往生产靠，开始处理“密钥轮换、版本篡改、防降级”这些更真实的安全问题

这份文档的核心不是“背算法名词”，而是回答下面这几个问题：

1. 为什么一开始不能直接上最复杂的方案？
2. v1 到底解决了什么？
3. v1 为什么不够？
4. 升级到 v2 后修了什么坑？
5. v2 为什么还不够？
6. 为什么 v3 一定要引入 `iv`？
7. v4 为什么要引入 `keyVersion`、`token`、多版本密钥？

---

## 先记住一个总原则

在这 4 个版本里，核心思路始终没变：

- **RSA 不适合直接加密大段业务数据**
- **AES 适合加密真正的业务数据**
- 所以实际方案通常都是：
  - 客户端随机生成一个 AES key
  - 用 AES key 加密业务数据
  - 再用 RSA 公钥去加密这个 AES key
  - 服务端拿 RSA 私钥先解出 AES key
  - 再拿 AES key 去解业务数据

这就叫**混合加密**，也叫 **Hybrid Encryption**。

为什么要这么做？

- 如果全用 RSA：
  - 慢
  - 能加密的数据长度有限
  - 不适合业务里真正的大字段
- 如果只用 AES：
  - AES key 怎么安全地下发给服务端，会变成大问题
- 所以：  
  **RSA 负责安全地“包”住 AES key，AES 负责高效地加密真实业务数据。**

## 一张总表先看懂

| 版本 | 主要目标 | 新增能力 | 解决了什么 | 还遗留什么问题 |
|---|---|---|---|---|
| v1 | 跑通最小闭环 | RSA 包 AES key，AES-ECB 加密数据 | 先理解混合加密的基本流程 | 没签名、没验签、ECB 不安全 |
| v2 | 解决“响应被篡改” | RSA 私钥签名，前端公钥验签 | 解决响应密文被改、用户看不出来的问题 | 还是 ECB，没有 iv，模式本身弱 |
| v3 | 解决 ECB 不安全 | AES-CBC + iv，并对 `(iv + 密文)` 签名 | 解决“相同明文容易暴露模式”的问题，也保护 iv 不被篡改 | 还没处理密钥轮换、防 keyVersion 篡改、防降级 |
| v4 | 更接近生产 | 多版本密钥、token、防降级骨架 | 开始解决“客户端上报的版本不可信”的问题 | 当前项目还是骨架，OAEP/GCM/完整闭环还没全部补齐 |

## v1：先把最小闭环跑通

### v1 想解决什么

v1 最重要的目标不是“安全性最强”，而是：

> 先把最基本的数据流跑通，让你看懂“客户端怎么加、服务端怎么解、服务端怎么再加回去”。

如果一上来就把 `OAEP`、`GCM`、`AAD`、`keyVersion`、`token`、`多版本密钥轮换` 全塞进来，虽然更像生产，但学习上会非常痛苦，因为你还没看懂最基本的闭环。

所以 v1 做了一件非常克制的事：

- 只保留混合加密最核心的部分
- 不加签名
- 不加 iv
- 不谈 token
- 不谈 keyVersion
- 不谈多私钥轮换

### v1 的流程

1. 前端请求公钥
2. 前端随机生成 AES key
3. 前端用 AES-ECB 加密业务明文
4. 前端用 RSA 公钥加密 AES key
5. 前端把 `encryptedKey + encryptedData` 发给服务端
6. 服务端用 RSA 私钥解 `encryptedKey`
7. 服务端拿到 AES key 后，再解 `encryptedData`
8. 服务端处理业务后，用同一个 AES key 再加密响应
9. 前端收到后，用 AES key 解密响应

### v1 解决了什么

v1 至少把下面 3 个关键问题说明白了：

#### 1. 业务数据为什么不用 RSA 直接加

你会看到，真正加密业务明文的是 AES，不是 RSA。  
这让你理解“RSA 是用来包住密钥，不是直接用来加业务数据”的真实用途。

#### 2. AES key 为什么要每次随机生成

如果 AES key 是固定写死在前端里，那就失去了很大一部分意义。  
v1 让你理解：**每次请求生成一个新的对称密钥**，是实际方案里非常常见的做法。

#### 3. 服务端为什么能把响应也加密回去

因为这次请求的 AES key，前后端都已经知道了：

- 前端自己生成的
- 服务端通过 RSA 私钥解出来的

所以服务端完全可以拿这个 AES key 再把响应加密回去。

这一步对很多初学者来说是个“顿悟点”。

### v1 为什么故意用了 AES-ECB

因为它最简单。

ECB 的特点是：

- 不需要 iv
- 代码最短
- 前后端最容易对上

对于学习最小闭环来说，它的优点是：

- 少一个新概念
- 少一个字段
- 少一个排错点

也就是说，**v1 牺牲的是安全性，换来的是理解成本最低。**

### v1 会出现什么问题

v1 跑通了，但它很快暴露出两个明显问题。

#### 问题 1：响应密文可能被篡改，前端发现不了

在 v1 里，服务端只是把响应加密回来了，但没有签名。

这意味着：

- 假设中间人把响应里的密文换掉了
- 前端收到后还是会去尝试解密
- 前端并不知道这个响应是不是服务端原样发出来的

也就是说，v1 只有“保密性”的一部分，但没有“完整性校验”。

更直白地说：

- `加密` 解决的是“别人看不懂”
- `签名` 解决的是“别人不能偷偷改”

v1 缺的是第二个。

#### 问题 2：ECB 模式本身不安全

这不是“代码写错了”，而是**模式本身就不够安全**。

ECB 最大的问题是：

> 相同的明文块，在同一个 key 下，总会加密成相同的密文块。

这会导致密文暴露出某种“模式特征”。  
虽然在你这个教学项目里不一定马上肉眼看出来，但从安全设计上讲，这已经是明显短板。

#### 所以 v1 升级到 v2 的动机是什么

不是先去修 ECB，而是先修更容易理解、也更常见的那个问题：

> **先补签名，先解决“响应可能被篡改”的问题。**

---

## v2：在 v1 上补“签名验签”

### v2 为什么先升级这个

从学习顺序上说，`签名/验签` 比 `iv/CBC/GCM` 更适合第二步学。

因为它回答的是一个非常直观的问题：

> 我怎么知道这个响应，真的是服务端发的，而且中途没被改过？

这比“ECB 与 CBC 的差异”更容易先理解。

### v2 引入了什么

v2 新增的核心只有一件事：

- 服务端用 **RSA 私钥** 对响应密文做签名
- 前端用 **RSA 公钥** 验签

签名算法使用的是：

- `SHA256withRSA`

注意这里非常关键：

- 加密时：
  - 前端用 RSA 公钥加密 AES key
  - 服务端用 RSA 私钥解 AES key
- 签名时：
  - 服务端用 RSA 私钥签名
  - 前端用 RSA 公钥验签

也就是说，同一对密钥既参与了“包 AES key”，也参与了“签名验签”，但用途不一样。

### v2 具体解决了 v1 的什么问题

v2 解决的是：

> **响应密文被篡改后，前端无法识别的问题。**

在 v1 里，如果有人动了响应内容，前端最多是在解密时报错，或者解出一些奇怪的内容；但前端不知道“这是被改过的”。

而在 v2 里：

1. 服务端生成响应密文
2. 服务端对这个密文字节做签名
3. 前端收到后，先验签
4. 验签通过，才去解密
5. 验签失败，直接拒绝处理

于是前端就从“被动挨打”升级成了“先做完整性校验，再决定要不要信任这份数据”。

### v2 为什么说“签名”不是“加密”

这是很多人第一次学时最容易混的地方。

#### 加密的目的

让别人看不懂内容。

#### 签名的目的

让接收方确认两件事：

1. 数据确实是持有私钥的人发出来的
2. 数据中途没被改过

所以 v2 其实是在 v1 的基础上，补上了“可信性”。

### v2 的签名到底签了什么

v2 签的是：

- **响应 AES 密文的原始字节**

不是 Base64 字符串，不是 JSON 全文，不是明文。

这个口径非常重要，因为跨前后端时，很多 bug 都来自于：

- Java 这边签的是字节
- JS 这边验的是字符串

口径一旦不一致，就会出现“明明都是同一个数据，为什么验签死活不通过”的问题。

### v2 还会出现什么问题

虽然 v2 修掉了“篡改不可见”的问题，但它保留了 v1 的另一个大坑：

> **AES 还是 ECB。**

也就是说，v2 的完整性更好了，但保密性模式依然弱。

换句话说：

- v1 的问题：既没签名，也用了 ECB
- v2 的改进：补了签名
- v2 的遗留：还是 ECB

### 为什么 v2 不能就停在这里

因为 v2 解决的是“别人改你数据”的问题，没解决“你的加密方式本身是否足够好”的问题。

所以接下来就必须升级到 v3：

> **把 ECB 换掉，引入 iv，进入 CBC。**

---

## v3：引入 iv，把 AES 从 ECB 升级到 CBC

### v3 的升级动机

v2 最大的问题是：

> 虽然现在响应不能被随便篡改了，但 AES-ECB 本身仍然是弱设计。

因此 v3 的重点不再是签名，而是：

- 引入 `iv`
- 把 AES 模式从 `ECB` 改成 `CBC`
- 同时让签名范围升级，避免 iv 被篡改

### 先用人话理解 iv

很多人第一次看到 `iv` 会觉得很抽象，其实你可以把它理解成：

> 每次加密前额外放进去的一段随机扰动值。

它的作用是：

> 就算明文相同、key 相同，只要 iv 不同，加密结果也会不同。

这就是 v3 比 v2 更安全的关键原因。

### v3 解决了 v2 的什么问题

v3 解决的是：

> **ECB 容易暴露重复模式的问题。**

在 CBC 模式下：

- 每次都要有 iv
- 同样的明文，不会稳定地产生同样的密文模式

这让密文更不容易泄露结构特征。

### v3 为什么不只是“加个 iv”这么简单

因为如果只是把 `iv` 和密文一起传过去，但**不保护 iv**，那还是不够。

原因是：

- iv 虽然不是秘密
- 但它仍然影响解密结果

如果攻击者能改 iv，就可能让解密结果发生变化。

所以 v3 做了一个非常重要的升级：

> 签名不再只签密文，而是签 `(iv + 密文)`。

这意味着：

- 改密文，不行
- 改 iv，也不行

于是完整性保护的范围更完整了。

### v3 比 v2 多了哪些新字段

#### 请求新增

- `iv`

#### 响应新增

- `iv`

这是因为：

- 请求用 AES-CBC 加密，需要请求 iv
- 响应用 AES-CBC 加密，也需要响应 iv

所以 v3 的“交互报文”会比 v1/v2 多一个字段。

### v3 的签名口径为什么又变了

v2 签的是：

- `encryptedData`

v3 签的是：

- `ivBytes + encryptedBytes`

为什么必须变？

因为 v3 的解密依赖两部分：

1. `iv`
2. `encryptedData`

那你要保护完整性，就不能只保护其中一部分。

### v3 现在已经够生产了吗

还不够。

v3 已经比前两个版本像样很多了，但它还缺少一个非常真实的生产问题：

> **客户端上报的“我用了哪个公钥版本”这件事，本身不可信。**

这就引出 v4 的主题：

- 多版本密钥
- keyVersion
- token 绑定
- 防降级攻击

---

## v4：开始进入“生产里的版本安全问题”

### 为什么 v1~v3 都还不够

前面 3 个版本主要解决的是“单次加解密链路”里的问题：

- 怎么加密
- 怎么解密
- 怎么验签
- 怎么引入 iv

但是在真实系统中，还会多出一个新问题：

> **密钥不是永远不变的。**

生产里通常要做：

- 密钥轮换
- 老密钥废弃
- 新密钥上线
- 某个版本吊销

一旦有了多版本密钥，就会产生一个很危险的问题：

> 客户端上报的 `keyVersion` 到底可信吗？

### 什么叫“降级攻击”

举个例子：

- 现在服务端最新公钥版本是 `v2`
- 旧版本 `v1` 已经准备废弃，甚至安全性更差

如果客户端本来应该用 `v2`，但有人篡改请求，让服务端误以为它在用 `v1`，那就可能把系统“拉回旧版本逻辑”。

这就叫**降级攻击**。

本质上是：

> 明明该走更安全的版本，却被强行拉回更旧、更弱的版本。

### v4 想解决什么

v4 的目标是让“版本信息”本身也变得可信：

- 不再只是下发公钥
- 还要下发 `keyVersion`
- 再下发一个服务端签发的 `token`
- token 里把 `keyVersion + fingerprint + 过期时间` 绑在一起

这样客户端上报时，服务端就能做校验：

- 你说你用的是哪个版本
- 这个版本是不是和我发给你的 token 能对上
- 这个 token 有没有过期
- 这个公钥指纹是不是对得上

如果对不上，就拒绝。

### v4 目前已经引入了什么

从当前项目代码看，v4 已经把以下能力搭起来了：

#### 1. 多版本密钥模型

有 `KeyVersion` 这个模型，里面包含：

- version
- publicKey
- privateKey
- fingerprint
- status

这意味着 v4 已经不是“只有一对固定密钥”的简单世界了。

#### 2. 多版本密钥管理

`KeyManager` 已经维护了：

- `keyVersion=1`
- `keyVersion=2`

并且设置了状态，例如：

- `DEPRECATED`
- `ACTIVE`

这已经具备了“轮换演示”的骨架。

#### 3. token 机制

`TokenService` 已经会签发和校验 token，里面绑定了：

- keyVersion
- fingerprint
- expireAt

这一步非常关键，因为它把“版本声明”变成了“服务端签名确认过的版本声明”。

#### 4. 控制器已经开始校验 token

当前 v4 的控制器已经不是单纯地“给公钥就完事了”，而是：

- `GET /api/key`：
  - 返回 `keyVersion`
  - 返回 `publicKey`
  - 返回 `token`
- `POST /api/secure/echo`：
  - 要求客户端带上 `keyVersion`
  - 要求客户端带上 `token`
  - 服务端先校验 token，再决定是否处理

这说明 v4 虽然还没完全进化成最终版，但“防 keyVersion 篡改”的主线已经出来了。

### v4 目前还没完全解决的问题

这里要实话实说：

> 当前仓库里的 v4，还是一个“往生产升级中的版本”，还不是最终完成态。

从当前代码看，v4 还存在下面这些“待升级点”：

#### 1. 加解密主体还没从 v1 风格完全升级掉

当前 `CryptoService` 还是：

- RSA-PKCS1
- AES-ECB
- 无签名

也就是说，v4 目前重点先落在了：

- 多版本密钥
- token 防降级

但还没有把密码套件主体同步升级成：

- RSA-OAEP
- AES-GCM
- 响应签名 / AEAD 校验

#### 2. 还没真正进入 GCM 这一步

真正更现代的做法，一般会继续升级到：

- `RSA-OAEP`
- `AES-GCM`

为什么？

- OAEP 比 PKCS1Padding 更现代
- GCM 同时提供加密和认证能力

但这一步如果太早上，会让学习难度明显增加，所以现在项目里先把“版本安全”这条线搭起来是合理的。

### v4 解决了 v3 的什么问题

v4 解决的不是“CBC 不够安全”这种问题，而是一个更高层的问题：

> **v3 假设客户端上传的版本信息是可信的，但这在真实世界里不成立。**

v4 就是在补这个坑：

- 客户端说自己用了哪个版本，不算数
- 要和服务端签发的 token 对上，才算数

这是从“密码算法安全”进一步升级到“协议流程安全”。

---

## 从 v1 到 v4，本质上每一步都在补什么

如果你把这 4 个版本抽象成一句话，其实就是：

### v1 补的是“能不能加解密成功”

先别管太多，先把最小闭环通了。

### v2 补的是“结果有没有被改”

加上签名验签，让响应变得可验证。

### v3 补的是“加密模式本身是不是靠谱”

引入 iv，让 AES 从 ECB 升级到 CBC。

### v4 补的是“协议里的版本信息可信吗”

引入 keyVersion、token、多版本密钥、防降级思路。

---

## 你可以怎么理解这 4 个版本的学习顺序

一个非常适合记忆的方式是：

### 第一层：先能跑

这就是 v1。

### 第二层：再防篡改

这就是 v2。

### 第三层：再把对称加密模式升级掉

这就是 v3。

### 第四层：再进入真实业务里的密钥轮换和版本安全

这就是 v4。

也就是说，这 4 个版本不是“谁替代谁”，而是：

> 每一版都在前一版的基础上，补上一个新的现实问题。

> 我做了 4 个独立版本的 RSA 学习项目，不是一次性上最复杂方案，而是按真实问题逐步升级。

> 但 v2 虽然能防篡改，AES 模式还是 ECB，本身不够安全。
>
> 到 `v3` 我把 AES 升级成 CBC，并引入 iv，同时把签名范围从“只签密文”升级成“签 iv + 密文”，解决了 ECB 模式和 iv 可篡改的问题。
>
> 到 `v4` 我开始处理更接近生产的协议层安全问题：多版本密钥、keyVersion、token、防降级。  
> 也就是说，客户端上传的版本号本身不再默认可信，而是必须和服务端签发的 token 绑定校验。

如果你能这么讲，面试官会觉得你不是在背 API，而是真的理解“为什么版本要一步一步升级”。
