v4：keyVersion + token（演示）

⚠️ v3 的问题：keyVersion 由客户端自报，攻击者可以伪造它实施降级攻击
假设服务端有两套密钥：keyVersion=1（RSA-1024，因为历史原因还在，已标记废弃）和 keyVersion=2（RSA-2048，当前推荐）。正常情况下，客户端请求 GET /api/key 拿到 v2 的强公钥，加密请求时带上 keyVersion: 2。但攻击者 Mallory 在链路上截获请求后，把 JSON 里的 "keyVersion": 2 改成 "keyVersion": 1。服务端收到后，以为客户端在用 v1，于是用 RSA-1024 的私钥去解密——1024 位的 RSA 在现代算力下已经可以被分解，Mallory 事后拿到密文就有能力暴力破解出 AES key，进而解密全部业务数据。这就是降级攻击（Downgrade Attack）：攻击者不直接攻破强密码，而是把通信双方"拉回"到更弱的旧版本。类似 TLS 的 POODLE 漏洞（强制降级到 SSL 3.0）和 LOGJAM 漏洞（强制降级到 512-bit DH），都是这一思路。
✅ v4 的解决：服务端签发 token，绑定 keyVersion + 公钥指纹 + 过期时间
现在 GET /api/key 不仅返回公钥和 keyVersion，还返回一个服务端签发的 token。这个 token 里用 HMAC 把 (keyVersion + 公钥指纹 + 过期时间) 绑在一起。客户端请求时必须同时上传 keyVersion 和 token，服务端先验证 token 是否合法、是否过期、里面的 keyVersion 和公钥指纹是否跟请求里的一致——只要 Mallory 改 keyVersion，token 就对不上，服务端直接拒绝。客户端"声称"的版本号不再被信任，一切以服务端签发的 token 为准。
