MySQL EXPLAIN 参数详解

SQL 性能优化必备手册

读懂执行计划，让查询飞起来

EXPLAIN 是 MySQL 提供的用于分析查询语句执行计划的工具。 通过它，我们可以了解 MySQL 如何使用索引、如何处理连接以及扫描的行数，从而精准定位性能瓶颈。

id

执行顺序

查询序列号。表示查询中执行 SELECT 子句或操作表的顺序。

  • 相同： 从上往下执行
  • 不同： 值越大优先级越高
  • NULL： 最后执行 (通常是 UNION 结果集)

select_type

查询类型

主要用于区别普通查询、联合查询、子查询等的复杂查询。

SIMPLE 简单查询 (无子查询/UNION)
PRIMARY 最外层查询
SUBQUERY SELECT/WHERE 中的子查询
DERIVED FROM 列表的子查询 (衍生表)
UNION UNION 后的第二个 SELECT

table

数据来源

显示这一行的数据是关于哪张表的。

有时不是真实的表名，而是别名或衍生表标记 (如 <derived2>)。

type

核心性能指标

显示查询使用了何种访问类型。这是判断查询性能好坏的关键指标。 从最优到最差排序如下：

system 表只有一行记录 (系统表)
const 主键/唯一索引匹配，最多一行
eq_ref 多表 JOIN 时，被驱动表使用主键/唯一索引
ref 非唯一性索引扫描，返回匹配某个单独值的所有行
range 只检索给定范围的行 (BETWEEN, IN, >)
index 全索引扫描 (比全表快，但依然慢)
ALL 全表扫描 (Full Table Scan) ❌

possible_keys

备选索引

显示可能应用在这张表上的索引（一个或多个）。

注意：如果为 NULL，则表示没有相关的索引。在这种情况下，可以通过检查 WHERE 子句看是否引用了某些列，从而创建适当的索引来提高性能。

key

实际使用

实际使用的索引。如果为 NULL，则没有使用索引。

在某些情况下，MySQL 可能会选择一个不在 possible_keys 中的索引（例如覆盖索引）。

key_len

索引长度

表示索引中使用的字节数。

在不损失精确性的情况下，长度越短越好。这有助于判断联合索引中有几个列被用到了。

ref

关联字段

显示索引的哪一列被使用了，或者是一个常数 (const)。

如果是多表连接，这里通常会显示另一个表的列名。

rows

预估行数

估算为了找到所需的行而要读取的行数。

这个值越小越好。它是基于统计信息的估算值，不一定完全准确。

Extra

额外信息 / 优化提示

包含不适合在其他列中显示但十分重要的额外信息。

Using index

极好！ 相应的 SELECT 操作中使用了覆盖索引 (Covering Index)，避免了访问表的数据行，效率很高。

Using index condition

搜索条件里虽不能完全利用索引，但会先用索引过滤一遍数据，减少回表次数 (ICP优化)。

Using where

表示存储引擎返回的记录并不是所有的都满足WHERE条件，需要在Server层进行过滤。

Using temporary

表示需要使用临时表来存储结果集，常见于 ORDER BY 和 GROUP BY。

Using filesort

需要优化！ 说明 MySQL 会对数据使用一个外部的索引排序，而不是按照表内的索引顺序进行读取。
