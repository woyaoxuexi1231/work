覆盖索引 (Covering Index)

EXPLAIN 中的 Using index

什么是覆盖索引？

当查询所需的所有列（SELECT、WHERE、ORDER BY 等）都包含在某个索引中时，MySQL 可以直接从索引树中获取数据，而不需要回表 (Table Lookup) 去读取聚簇索引（主键数据行）。

💡 核心优势： 避免了随机 I/O，将“回表”操作转化为纯粹的索引扫描，性能提升通常在 10 倍以上。
SQL Example ● Optimized
-- 假设表 user(id, name, age, email)
-- 建立联合索引: idx_name_age(name, age)
SELECT name, age FROM user
WHERE name = 'Alice';
✅ SELECT 的列 (name, age) 都在索引 idx_name_age 中。
✅ WHERE 条件也在索引中。
➡️ 触发覆盖索引，无需回表。
Visual Flow
B+ Tree (idx_name_age)
[ ... | Alice | ... ]
↓
[Name: Alice, Age: 25, PK: 101]
索引叶子节点已包含 Age，直接返回！
✅ 直接返回结果 (No Table Lookup)
X 不需要回表
Clustered Index (Data Row)

索引下推 (ICP)

EXPLAIN 中的 Using index condition · MySQL 5.6+

工作原理对比

❌ 未开启 ICP (旧版)
Index Scan
(name='Alice')
↓
回表取整行数据
↓
Server层过滤
age > 20
每次匹配name都要回表
✅ 开启 ICP (5.6+)
Index Scan
(name='Alice')
+ 存储引擎层过滤 age>20
↓
仅对满足条件的行回表
↓
Server层 (几乎无过滤工作)
大幅减少回表次数!

为什么需要 ICP？

在联合索引 (name, age) 中，如果查询是 WHERE name LIKE 'A%' AND age > 20：

  • ✕ 没有 ICP： 索引只能用到 name LIKE 'A%'。所有名字以 A 开头的人，无论年龄大小，都要逐个回表检查。
  • ✓ 有了 ICP： 虽然 age 不能用于索引查找（因为前面是范围查询），但索引里存有 age 的值。存储引擎在遍历索引时，直接在索引层面判断 age > 20，不满足的直接跳过，只有满足的才回表。
Trigger Condition
SELECT * FROM user
WHERE name LIKE 'A%'
  AND age > 20;
🔑 关键点：
1. 使用了联合索引
2. 索引中有无法用于搜索但可用于过滤的列 (age)
3. 不是覆盖索引 (SELECT *)
⚠️

注意： ICP 仅适用于 InnoDB 和 MyISAM 引擎。对于 InnoDB，ICP 仅适用于二级索引（非聚簇索引），因为主键索引的叶子节点就是完整数据行，不存在“减少回表”的概念。
