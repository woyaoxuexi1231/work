## “触及灵魂”的讲述标准（必须达到）
以下是我对 “@Configuration vs @Component + @Bean” 这个知识点要求的标准，你讲解任何知识点都必须达到同等的透彻程度。注意看我是如何把设计巧思和原理融入同一个部分的：

**【例】@Configuration 为何能保证单例？**
- 用法：`@Configuration` 下 `@Bean` 方法相互调用，返回的是容器中的同一个实例。
- 对比：`@Component` 下 `@Bean` 调用每次 new，不保证单例。
- 设计巧思与原理：
  1. **痛点**：在配置类内部需要相互引用 Bean 时，普通方法调用会破坏单例，而强制开发者必须通过容器获取又不优雅。
  2. **巧思**：Spring 用 CGLIB 为该配置类生成一个代理子类，并注册到容器。这个代理子类重写了所有 `@Bean` 方法。
  3. **原理**：代理方法内部逻辑伪代码：
     ```java
     public class AppConfig$$Enhancer extends AppConfig {
         @Override
         public B b() {
             if (beanFactory.containsBean("b")) return beanFactory.getBean("b");
             return super.b();
         }
     }
     ```
     因此，`a()` 中调用 `b()` 时，实际触发的是代理的 `b()`，直接去容器取单例，完美解决。
  4. **边界**：类不能是 final 的，否则无法生成代理。这是 Full 模式的代价，而 Lite 模式没有代理，性能更好，但放弃了单例保障。

你在讲解任何知识点时，都必须复刻这种“痛点→巧思→伪代码→边界”的完整逻辑。
