## 一、给定 GC 日志快速诊断

你提供的是一份 JDK 17 + G1 的 GC 日志，堆大小被死死限定在 **32 MB**。日志展示了从启动到几乎 OOM 的全过程，核心问题可概括为一句话：

> 堆太小，而 HashMap 扩容又产生了大量长期存活的对象和巨型对象，导致老年代持续膨胀，触发频繁 Young GC、多次 **To‑space exhausted** 以及一连串 **Full GC**。

### 关键危险信号一览

| 信号 | 具体表现 | 严重性 |
|------|---------|--------|
| **Old 区单向增长** | Old regions 从 0 → 8 → 11 → … → 29，几乎每次 GC 后都增加 | 🔴 对象大量存活，无法回收 |
| **Humongous Allocation 触发 GC** | `GC(2) Pause Young (Concurrent Start) (G1 Humongous Allocation)` | 🟡 有大对象（≥512KB）频繁分配，触发了 GC 甚至并发标记 |
| **并发标记周期多次启动但被 Abort** | `Concurrent Mark Abort` 多次出现 | 🟡 Full GC 中断了并发标记 |
| **To‑space exhausted** | `GC(16) To-space exhausted` 等 | 🔴 没有空闲 Region 可供复制，堆即将耗尽 |
| **连续 Full GC** | GC(17)、GC(21)、GC(25)、GC(26)… 连续的 `Pause Full` | 🔴🔴 极度危险，应用基本停滞 |
| **Full GC 后内存几乎不下降** | GC(17) 32M→27M，后续 Full GC 几乎都是 31M→31M 或 31M→29M | 🔴🔴 对象都是强引用活，GC 无能为力 |
| **尝试清理软引用** | `Attempting maximum full compaction clearing soft references` | 🟡 JVM 已到绝望阶段 |

### 根因分析

1. **堆太小（32 MB）**：而你的代码要向 HashMap 连续插入 5 万个 `Payload`（每个 256 字节 + Node 开销 ≈ 280 字节），光存活数据就超过 14 MB。
2. **HashMap 反复扩容**：默认初始容量 16，负载因子 0.75，随着数据量增大，Node 数组从 16 一路翻倍。当 `Node[]` 长度达到 131072 时，其大小约 512 KB（压缩指针），触发了 G1 的 **巨型对象** 分配，直接进入老年代。
3. **巨型对象常驻老年代**：日志中始终有 3 个 Humongous regions，且 Old 区持续扩大，说明这些数组和老年代中其他对象都无法被回收。
4. **GC 无法腾出连续空间**：当堆接近满时，没有足够的空闲 Region 供复制存活对象，导致 `To-space exhausted`，进而退化为 STW 的 Full GC。但 Full GC 也只是整理碎片，回收不了强引用对象，所以内存几乎不降。
5. **恶性循环**：Full GC 暂缓后，应用又立即分配新对象，马上再次触发 `G1 Preventive Collection`（预防性 GC），然后又出现 `To-space exhausted`，再次 Full GC，最后甚至尝试 Clean Soft References。

**结论**：这是典型的小堆 + 大 HashMap 扩容引起的 **内存过载**，最终表现为 **GC 风暴**。解决方式很简单：① 增大堆；② 为 HashMap 指定足够的初始容量，避免扩容尤其是巨型数组。

## 二、GC 日志分析完全教程（让你从此一眼看穿问题）

以下教程基于 G1 收集器（JDK 9+ 统一日志格式 `-Xlog:gc*`），但分析思想适用于所有收集器。

### 1. 先看懂单行日志：格式与含义

以你的日志为例：
```
[2026-05-09T18:54:19.832+0800][0.207s][95944] GC(0) Pause Young (Normal) (G1 Evacuation Pause) 16M->11M(32M) 3.690ms
```
- **时间戳** `[2026-05-09T18:54:19.832+0800]` 和 **相对时间** `[0.207s]`：进程启动后的秒数。
- **线程 ID** `[95944]`：产生日志的线程。
- **GC 编号** `GC(0)`：第几次 GC。
- **GC 事件类型**：`Pause Young (Normal) (G1 Evacuation Pause)` —— 正常的年轻代回收停顿。
- **堆大小变化**：`16M->11M(32M)` —— GC 前堆占用 16M，GC 后降为 11M，总堆 32M。
- **停顿时间**：`3.690ms`。

### 2. G1 日志关键组成部分

#### 2.1 内存 Region 分布
每行详细日志常包含：
```
Eden regions: 14->0(4)
Survivor regions: 0->2(2)
Old regions: 0->8
Humongous regions: 2->2
Metaspace: 649K(832K) ...
```
- **Eden / Survivor / Old**：分别表示 GC 前后各区的 Region 数量及 Eden/Survivor 的目标上限（括号内）。
- **Humongous regions**：巨型对象（≥ Region 大小的一半）占用的 Region 数量。**巨型对象直接分配在老年代，几乎无法移动**。
- **变化趋势**：如果每次 GC 后 **Old regions 都在增长**，说明有内存泄漏或大量对象生命期太长。

#### 2.2 并发标记周期 (Concurrent Mark Cycle)
```
GC(3) Concurrent Mark Cycle
  Concurrent Scan Root Regions
  Concurrent Mark From Roots
  Pause Remark
  Pause Cleanup
```
- **触发原因**：G1 在 **堆使用量超过 InitiatingHeapOccupancyPercent（默认 45%）** 时启动并发标记，为后续的 Mixed GC 做准备。
- **正常顺序**：开始 → 扫描根 → 并发标记 → 最终标记(Remark) → 清理(Cleanup)。如果中间出现 **Concurrent Mark Abort**，往往是 Full GC 打断了它。

#### 2.3 Full GC (G1 Compaction Pause)
```
GC(17) Pause Full (G1 Compaction Pause)
  Phase 1: Mark live objects
  Phase 2: Prepare for compaction
  Phase 3: Adjust pointers
  Phase 4: Compact heap
```
- **Full GC** 是 STW（Stop‑The‑World）的，代价极高，应尽量避免。
- **出现 Full GC 意味着**：疏散失败（To‑space exhausted）、分配不了新对象、Metaspace 满等。
- **分析要点**：**看 Full GC 后的内存是否明显下降**。如果几乎没有下降（如 31M→31M），说明老年代对象全为强引用存活，只能从代码或堆大小入手。

### 3. 五大“一眼致命”模式及排查方法

#### 🔴 模式 1：Old 区持续增长且不减少
**表现**：每次 Young GC 后 `Old regions` 一直在增加。
**原因**：对象晋升过快（Survivor 区存不下直接进老年代）、内存泄漏（对象被无意持有）。
**排查**：

- 连续观察 5‑10 次 Young GC 的 `Old regions` 变化。
- 如果趋于平缓则是对象慢慢消亡，如果线性增长就是泄漏。
- 通过 `jmap -histo:live` 或 HeapDump 查看占据内存的对象。

#### 🔴 模式 2：To‑space exhausted
**表现**：`GC(16) To-space exhausted` 后紧跟着 Full GC。
**原因**：没有空闲 Region 来承接要复制的存活对象（例如从 Eden/Survivor 复制到 Old）。通常是堆已被用满。
**排查**：
- 立刻查看堆使用率是否已接近 100%。
- 若是，**必须增大堆大小**或减少对象创建。
- 注意 Humongous 对象会吞掉大量 Region，且难以移动。

#### 🔴 模式 3：Full GC 无法回收内存
**表现**：连续的 Full GC，每次停顿 8‑10 ms，但内存几乎不变（如 31M→31M）。
**原因**：几乎所有对象都是强引用存活的，GC 无垃圾可收。
**排查**：
- 用 `-XX:+HeapDumpOnOutOfMemoryError` 抓堆快照。
- 分析对象图，找到是谁持有了海量对象（HashMap、缓存等）。
- 如果不是泄漏，只是容量不足，就加大堆或设计更省内存的数据结构。

#### 🟡 模式 4：Humongous Allocation 频繁触发 GC
**表现**：`Pause Young (Concurrent Start) (G1 Humongous Allocation)`。
**原因**：分配了 ≥ Region 一半大小的大对象（如大数组、大 ByteBuffer）。
**影响**：巨型对象直接进老年代，容易导致老年代碎片化。
**排查**：
- 找出大对象的来源（HashMap 的 `Node[]` 扩容数组是常见元凶）。
- 尽量避免创建大对象，或把他们放回常规堆（例如分多次分配）。
- 必要时增大 Region 大小（`-XX:G1HeapRegionSize`），但一般不推荐。

#### 🟡 模式 5：Concurrent Mark Abort
**表现**：`Concurrent Mark Abort` 出现在 Full GC 附近。
**原因**：Full GC 会中断并覆盖并发标记。若频繁 Abort，说明系统一直在 Full GC 和并发标记间摇摆。
**排查**：本质是 Full GC 频繁，先解决 Full GC 问题。

### 4. 通用分析四步法（拿去就用）

**第一步：看整体趋势**
- 使用 `gceasy.io` 或直接用文本拉一张表格：GC 次数、平均暂停时间、堆峰值。
- 关注 GC 累计时间占总运行时间的比例。若 > 10%，GC 压力很大。

**第二步：找最“贵”的 GC**

- 找出停顿时间最长的 GC（一般是 Full GC）。
- 检查触发原因：`Humongous Allocation`？`To-space exhausted`？`Metadata GC Threshold`？

**第三步：揪出内存元凶**
- 对比每次 GC 前后各代大小；画出 Old 区增长曲线。
- 判断是**内存泄漏**（Old 一直涨，Full GC 后也不降）还是**峰值流量**（Old 有升有降）。

**第四步：调优/修复**
- **优先调整代码**：避免无界缓存、指定集合初始容量、缩短对象生命周期。
- **调整 GC 参数**：
    - 增大堆：`-Xmx` 至少为稳定存活数据的 1.5‑2 倍。
    - 若必须用小堆，减少 Eden 大小让 Young GC 更频繁但快速（前提对象朝生夕灭）。
    - 提高 `-XX:InitiatingHeapOccupancyPercent` 让并发标记晚点开始（慎重）。
- **验证**：重复测试，观察 GC 日志是否改善。

### 5. 快速工具箱

| 用途 | 命令/工具 |
|------|----------|
| 产生 GC 日志 | `-Xlog:gc*=info:file=gc.log` (JDK9+) |
| 图形化分析 | 在线 `gceasy.io`，或 `GCViewer` |
| 实时监控 | `jstat -gcutil <pid> 1000` |
| 堆快照 | `jmap -dump:live,file=heap.hprof <pid>` |
| HeapDump 分析 | Eclipse MAT / JProfiler / VisualVM |
| 查看 JVM 默认配置 | `java -XX:+PrintFlagsFinal -version \| grep G1` |
