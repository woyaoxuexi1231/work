P14

反射与泛型擦除的获取边界

擦除本质 获取边界 TypeReference 残酷真相

泛型信息并没有完全消失
它只是换了个地方藏起来

Java 泛型在编译期执行类型检查后，会在字节码层面进行擦除（Erasure）。 但这并不意味着所有泛型信息都消失了。Method.getGenericParameterTypes() 和 Field.getGenericType() 依然能从 Signature 属性 中还原出完整的泛型签名。

核心 API 对比

同一个方法，两个截然不同的视角

JVM 规范允许在 class 文件中保留 Signature 属性用于反射。 这就是为什么我们能在运行时"看到"泛型——但仅限于声明处。

Raw getParameterTypes()

返回擦除后的原始类型。这是 JVM 实际执行时看到的类型。

List<String> → java.util.List
Map<String, Integer> → java.util.Map
Gen getGenericParameterTypes()

返回包含泛型信息的 Type。如果参数是泛型，返回 ParameterizedType；否则退化为 Class。

List<String> → java.util.List<java.lang.String>
Map<String, Integer> → java.util.Map<java.lang.String, java.lang.Integer>
ReflectionDemo.java 
public class UserService {
    public void save(List<String> names,
                     Map<String, Integer> scores) {}
}

Method m = UserService.class
    .getMethod("save", List.class, Map.class);

// ❌ 擦除后的原始类型
Class<?>[] raw = m.getParameterTypes();
// [interface java.util.List, interface java.util.Map]

// ✅ 带泛型签名的完整类型
Type[] generic = m.getGenericParameterTypes();
// [java.util.List<java.lang.String>,
//  java.util.Map<java.lang.String, java.lang.Integer>]

// 安全提取泛型参数
if (generic[0] instanceof ParameterizedType) {
    ParameterizedType pt = (ParameterizedType) generic[0];
    Type[] args = pt.getActualTypeArguments();
    // args[0] = java.lang.String ✅
}

获取边界：能 vs 不能

这是理解泛型反射最关键的一张表

场景 能拿到 拿不到 原因
字段声明
private List<String> names; 		✓ Signature 属性保留在字段元数据中
方法参数/返回值
void save(List<String> list) 		✓ Signature 属性保留在方法元数据中
类/接口继承声明
class StrList extends ArrayList<String> 		✓ GenericSuperclass / GenericInterfaces 保留
局部变量
List<String> list = new ...; 		✗ 局部变量表不存储 Signature，编译后彻底擦除
new 表达式
new ArrayList<String>() 		✗ 对象实例上不携带任何泛型信息
纯泛型类型参数
<T> void process(T item) 		✗ T 被擦除为 Object（或上界），调用时的实际类型不可知
数组元素的泛型
List<String>[] arrays 		✗ Java 禁止创建泛型数组，即使通过绕过手段也无法反射获取
💡

一句话总结边界规则

凡是写在"声明处"（字段、方法签名、类继承）的泛型 → 能拿到。
凡是出现在"使用处"（局部变量、new、方法调用传参）的泛型 → 拿不到。
因为 Signature 属性只存在于类文件的结构元数据中，不存在于字节码指令和对象实例中。

经典模式

TypeReference：用匿名子类"偷渡"泛型

Jackson、Gson、Fastjson 等序列化框架的核心技巧。既然 new ArrayList<String>() 无法携带泛型信息，那就创建一个匿名内部类来继承一个带具体泛型的父类—— 这样泛型信息就被"固化"在了子类的 GenericSuperclass 中。

原理拆解

  1. 1 定义抽象类 TypeReference<T>，构造函数中通过 getClass().getGenericSuperclass() 提取 T
  2. 2 用户创建 new TypeReference<List<String>>() {}（注意末尾的 {}）
  3. 3 这个 {} 生成了一个匿名子类，其 GenericSuperclass = TypeReference<List<String>>
  4. 4 从 ParameterizedType.getActualTypeArguments()[0] 中提取出 List<String>
⚠️

关键：末尾的 {} 绝对不能省略！ 没有它就不是匿名子类，而是直接实例化父类，getGenericSuperclass() 返回的是 TypeReference<T>（T 未被绑定），提取失败。

TypeReference.java 
public abstract class TypeReference<T> {
    private final Type type;

    protected TypeReference() {
        Type superType = getClass()
            .getGenericSuperclass();

        if (superType instanceof
                ParameterizedType) {
            ParameterizedType pt =
                (ParameterizedType) superType;
            this.type = pt
                .getActualTypeArguments()[0];
        } else {
            throw new IllegalArgumentException(
                "必须用匿名子类: new TypeReference<X>(){}"
            );
        }
    }

    public Type getType() {
        return type;
    }
}

// ✅ 正确使用
TypeReference<List<String>> ref =
    new TypeReference<List<String>>() {};

ref.getType();
// → java.util.List<java.lang.String> ✅

// ❌ 错误使用（缺少 {}）
TypeReference<List<String>> bad =
    new TypeReference<List<String>>();
// → 抛出 IllegalArgumentException!

残酷真相：拿到了也不是你想象的那种"拿到"

即使 getGenericParameterTypes 成功返回，也必须清醒认识到以下限制

1

拿到的是"描述"，不是"类型本身"

ParameterizedType 只是一个元数据描述对象， 它不是 List<String>.class（这种东西根本不存在）。 你不能用它做 instanceof、不能 newInstance()、 不能传给需要 Class<T> 的方法。

// 你能做的：查询、比较、序列化框架内部使用
// 你不能做的：用它创建实例或做运行时类型检查
2

嵌套泛型的递归地狱

对于 Map<String, List<Set<Integer>>>， getActualTypeArguments() 返回的每个元素可能又是 ParameterizedType、 WildcardType、GenericArrayType 或 TypeVariable。 你必须写递归解析器才能完整还原类型树。

Type → ParameterizedType? WildcardType?
        → GenericArrayType? TypeVariable?
        → Class? ← 只有这一种能直接用
3

通配符 ≠ 具体类型

当声明为 List<? extends Number> 时， 你拿到的是 WildcardType，只能查到上界 Number。 你永远无法知道运行时实际传入的是 Integer 还是 Double。

// WildcardType.getUpperBounds() → [Number]
// 但实际值可能是 Integer, Double, Float...
// 反射无法区分它们
4

TypeVariable 是"占位符"，不是"答案"

对于 <T> void process(T item)， getGenericParameterTypes() 返回的是 TypeVariable（名为 "T"）， 而不是调用时传入的实际类型。调用处的类型信息在编译后就永远丢失了。

// TypeVariable.getName() → "T"
// TypeVariable.getBounds() → [Object]
// 调用 process("hello") 时，"String" 已丢失

⚖️ 最终结论

能拿到的： 声明处的完整泛型签名（字段、方法参数/返回值、类继承关系）。这些信息以 Signature 属性的形式保存在 class 文件中，通过 getGenericXxx() 系列 API 可以完整还原为 Type 对象树。

拿不到的： 使用处的泛型信息（局部变量、new 表达式、方法调用时的实际类型参数）。这些在编译后被彻底擦除， 不存在于字节码中，也不存在于对象实例中。没有任何反射手段可以恢复它们。

拿到了也不等于： 拿到了一个可用的 Class 对象。你拿到的是一个类型描述符（Type Descriptor）， 它只能用于查询和框架内部的类型推导（如 JSON 反序列化、ORM 映射、依赖注入）， 不能用于运行时类型检查、实例创建或任何需要真正 Class<T> 的场景。

💡 TypeReference 的本质：利用"匿名子类将泛型固化到类继承声明中"这一特性，把原本属于"使用处"的类型信息， 巧妙地转化为"声明处"的类型信息，从而绕过擦除限制。这是一种优雅的 Hack，但不是魔法。
