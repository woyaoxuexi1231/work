第一代 已弃用大部分方法

java.util.Date

JDK 1.0 · 1996年1月

诞生背景

Java 诞生之初，急需一个表示"时间点"的基础类。java.util.Date 被设计为 Java 中最基础的日期时间类，本质上是对 Unix 时间戳（毫秒数）的简单封装。 它模仿了 C 语言的 time_t 概念，同时加入了日期解析、格式化、比较等功能—— 将所有职责塞进了一个类中。

解决了什么

  • • 提供了 Java 中第一个时间表示方式，基于 UTC 1970-01-01 的毫秒偏移量
  • • 支持基本的日期创建、比较（before/after）和简单运算
  • • 内置了 toString() 可直接输出可读字符串

优点

  • 概念简单直观，底层就是毫秒时间戳
  • 实现了 Serializable 和 Comparable
  • 被所有后续 API 兼容

缺点

  • 可变对象，线程不安全
  • 月份从 0 开始（0=一月）
  • 年份从 1900 开始偏移
  • 大量方法被 @Deprecated
  • 无时区概念，国际化差
  • 职责混乱：同时承担日期+时间+格式化
DateDemo.java 
// ❌ 反人类的月份从0开始
Date date = new Date(2024 - 1900, 0, 1);
// 2024年1月1日 → 年份要减1900，月份写0

// ❌ 可变对象 → 线程不安全
Date d = new Date();
d.setYear(120);  // 被外部修改！Deprecated!
d.setMonth(5);  // 6月（不是5月！）

// ❌ getYear() 返回距1900的偏移
Date now = new Date();
System.out.println(now.getYear());
// 输出: 126 (而不是 2026)

// ❌ 职责混乱：一个类干了太多事
date.getHours();    // 时间？
date.getDate();     // 日期中的"日"？
date.getDay();     // 星期几？（0=周日）
date.toLocaleString(); // 格式化？Deprecated!
⚠️

历史评价：在一次社区投票中，Date/Calendar 被评为 "Java 最烂 API" 第二名（仅次于 XML/DOM）。 JDK 1.1 就弃用了 Date 的大部分方法，堪称"出道即翻车"。

CalendarDemo.java 
// 获取一个 Calendar 实例（工厂方法）
Calendar cal = Calendar.getInstance();

// ❌ 月份依然从 0 开始！
cal.set(2024, 0, 15); // 2024年1月15日
cal.set(2024, Calendar.JANUARY, 15); // 稍微好点

// ❌ API 极其繁琐
cal.add(Calendar.DATE, 7);     // +7天
cal.add(Calendar.MONTH, -1);   // -1月
cal.roll(Calendar.DAY_OF_MONTH, 3);

// ❌ 取值要用 get()，魔法数字满天飞
int year  = cal.get(Calendar.YEAR);
int month = cal.get(Calendar.MONTH) + 1; // +1!
int dow   = cal.get(Calendar.DAY_OF_WEEK);

// ❌ 可变 + 非线程安全
Calendar shared = Calendar.getInstance();
// 多线程共享 shared 并调用 set/add → 数据错乱!
第二代 遗留代码

java.util.Calendar

JDK 1.1 · 1997年2月

诞生背景

Date 的大部分方法在 JDK 1.1 就被标记为 Deprecated， Sun 引入了 Calendar 作为替代方案。其核心目标是： 将"时间瞬间"（Date）和"日历系统"（Calendar）解耦，同时提供国际化的日历支持（如日本和历、佛历等）。

解决了什么

  • • 提供了日历计算能力：add()、roll()、日期字段提取
  • • 支持多日历系统（GregorianCalendar 只是其中一种）
  • • 引入了 TimeZone 和 Locale 支持
  • • 提供了工厂方法 getInstance()

相比 Date 的进步

  • 支持日历运算（加减天/月/年）
  • 支持多日历系统和本地化
  • 时区感知（可设置 TimeZone）
  • Date/Calendar 职责分离

新的问题

  • 可变 + 非线程安全（变本加厉）
  • 月份依然从 0 开始
  • API 极其繁琐，大量魔法常量
  • 性能差：初始化重量级对象
  • 日期/时间字段无法直接格式化
  • 星期从周日开始（1=SUNDAY）
配套工具 线程不安全

SimpleDateFormat

java.text · JDK 1.1 · 1997年

诞生背景

随着 Date 和 Calendar 剥离了格式化能力（Date 的 toString/toLocaleString 被废弃）， Java 需要一个专门的日期格式化/解析类。SimpleDateFormat 作为 DateFormat 的具体实现被引入 java.text 包。 它通过模式字符串（如 "yyyy-MM-dd HH:mm:ss"）实现灵活的格式化与解析。

解决了什么

  • • 将格式化/解析职责从 Date 中分离，实现单一职责
  • • 支持自定义模式字符串，灵活的日期格式化
  • • 支持字符串到 Date 对象的双向转换（parse / format）
  • • 支持 Locale 本地化格式化

优点

  • 模式字符串直观灵活
  • 支持双向转换（format + parse）
  • 支持本地化与自定义时区

致命缺陷

  • 非线程安全！这是最臭名昭著的 Bug 来源
  • static SDF 实例在多线程下会静默产生错误日期
  • 只能配合 Date 使用，不能独立表示日期
  • 解析错误时抛异常而非返回 Optional

💀 经典线上事故模式

// 这是无数线上事故的根源 ↓
private static final SimpleDateFormat SDF
    = new SimpleDateFormat("yyyy-MM-dd");

// 多线程调用 → 随机返回错误日期！
public String format(Date d) {
    return SDF.format(d); // 线程不安全！
}
SimpleDateFormatDemo.java 
// 基本用法：格式化
SimpleDateFormat sdf = new SimpleDateFormat(
    "yyyy-MM-dd HH:mm:ss"
);

Date now = new Date();
String str = sdf.format(now);
// "2026-05-26 13:13:00"

// 基本用法：解析
Date date = sdf.parse("2024-01-15 10:30:00");

// 本地化格式化
DateFormat df = DateFormat.getDateInstance(
    DateFormat.LONG,
    Locale.CHINA
);
// "2024年1月15日"

// ⚠️ 常见的"安全"用法（但很繁琐）
ThreadLocal<SimpleDateFormat> safe =
    ThreadLocal.withInitial(() ->
        new SimpleDateFormat("yyyy-MM-dd")
    );
// 每个线程一个实例，终于安全了…但好累
JodaTimeDemo.java 
// ✅ 终于正常了！月份从 1 开始
LocalDate date = new LocalDate(2024, 1, 15);
// 2024年1月15日 — 符合人类直觉!

// ✅ 不可变 → 线程安全
LocalDate tomorrow = date.plusDays(1);
// date 不变，返回新对象

// ✅ 流式 API，极其直观
DateTime dt = new DateTime()
    .plusYears(1)
    .minusMonths(2)
    .plusDays(7)
    .withHourOfDay(14);

// ✅ 日期间隔计算
Days days = Days.daysBetween(date1, date2);
Period period = new Period(date1, date2);

// ✅ 丰富的格式化器
DateTimeFormatter fmt =
    DateTimeFormat.forPattern("yyyy-MM-dd");
String str = fmt.print(date);
💡

趣闻：Joda-Time 的作者 Stephen Colebourne 后来主导了 JSR-310 规范的制定，将 Joda-Time 的设计思想直接带入了 JDK 8 的 java.time 包——可以说是"社区倒逼官方改革"的经典案例。

第三方库 社区之光

Joda-Time

Third-party · 2002年首发 · Stephen Colebourne

诞生背景

经过近5年的实战洗礼，Java 开发者对 Date/Calendar 的不满达到顶点。Stephen Colebourne 在 2002 年创建了 Joda-Time——一个完全重新设计的日期时间库。 它从底层重新思考"什么是日期"、"什么是时间"，引入了 不可变性、类型安全、 领域驱动设计 等现代理念，迅速成为 Java 生态中最受欢迎的第三方库之一。

解决了什么

  • • 不可变设计：所有核心类都是不可变的，天然线程安全
  • • 类型分离：LocalDate、LocalTime、DateTime、Interval、Period、Duration 各司其职
  • • 流式 API：plusDays().minusMonths() 链式调用
  • • 月份从 1 开始：终于符合人类直觉
  • • 强大的时区与夏令时支持

革命性进步

  • 不可变 + 线程安全
  • API 设计优雅直观
  • 丰富的类型体系
  • 性能远优于 Calendar
  • 完善的文档和测试

局限

  • 第三方依赖，需额外引入 JAR
  • 与 JDK 原生 Date 互操作需转换
  • 使用 Chronology 系统增加复杂度
  • null 处理不够优雅（Java 8 前无 Optional）
第三代 · 终极方案 当前标准

java.time (JSR-310)

JDK 8 · 2014年3月 · Stephen Colebourne 主导

诞生背景

Joda-Time 在社区的巨大成功证明了：Java 需要一个全新的日期时间 API。 2007 年，Stephen Colebourne 提交了 JSR-310（新的日期时间 API 规范）， 经过 7 年的打磨，最终随 JDK 8（2014年3月）发布。 它继承了 Joda-Time 的精华，同时修正了其中的设计缺陷（如 null 处理、Chronology 过度复杂等）， 并充分利用了 Java 8 的新特性（Lambda、Optional 等）。

解决了什么（彻底解决所有历史问题）

  • • 所有核心类不可变且线程安全
  • • 清晰的类型分离：LocalDate / LocalTime / LocalDateTime / ZonedDateTime / Instant / OffsetDateTime 各司其职
  • • 流式 API + Temporal 接口体系：统一的 plus/minus/with 方法
  • • DateTimeFormatter 线程安全，且支持预定义格式
  • • 完善的时区支持：ZoneId + ZoneRules + 夏令时处理
  • • Period（日期差）和 Duration（时间差）分离
  • • TemporalAdjuster：灵活的日期调整器（"下个周一"、"月末最后一天"）
  • • 与旧 API 的互操作桥梁（Date.from() / .toInstant()）

相比所有前辈的优势

  • JDK 内置，零依赖
  • 不可变 + 线程安全
  • API 设计堪称教科书
  • 类型体系完备且清晰
  • TemporalAdjuster 扩展性极强
  • DateTimeFormatter 线程安全
  • 完善的时区与日历系统

仍有不足

  • 类数量多，学习曲线较陡
  • 与旧代码互操作需要转换
  • 不支持纳秒以下精度
  • JDBC/ORM 框架兼容曾滞后
JavaTimeDemo.java 
// ✅ LocalDate — 纯日期，不可变
LocalDate today = LocalDate.now();
LocalDate birthday = LocalDate.of(1995, 6, 15);
LocalDate nextWeek = today.plusWeeks(1);

// ✅ 流式计算
LocalDateTime meeting = LocalDateTime.now()
    .plusDays(3)
    .withHour(14)
    .withMinute(30);

// ✅ TemporalAdjuster — "下个周一"
LocalDate nextMonday = today.with(
    TemporalAdjusters.next(DayOfWeek.MONDAY)
);

// ✅ 线程安全的格式化
DateTimeFormatter fmt =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm");
String str = meeting.format(fmt);
LocalDateTime parsed = LocalDateTime.parse(str, fmt);

// ✅ 时区处理
ZonedDateTime tokyoTime = ZonedDateTime.now(
    ZoneId.of("Asia/Tokyo")
);

// ✅ Period vs Duration 语义清晰
Period age = Period.between(birthday, today);
Duration elapsed = Duration.between(start, end);

// ✅ 与旧 API 互操作
Instant instant = oldDate.toInstant();
Date legacy = Date.from(instant);
🎉

最终定论：java.time 是目前 Java 中处理日期时间的 唯一推荐方案。 所有新项目都应直接使用 java.time 包，老项目应逐步迁移。Spring Boot、MyBatis、Hibernate 等主流框架已全面支持。

java.time 核心类速查

选对类型，事半功倍

LD

LocalDate

纯日期，无时间无时区

2024-01-15

场景：生日、节假日、到期日

LT

LocalTime

纯时间，无日期无时区

14:30:00

场景：闹钟、营业开始时间

LDT

LocalDateTime

日期+时间，无时区

2024-01-15T14:30

场景：本地事件、日志时间

ZDT

ZonedDateTime

日期+时间+时区

2024-01-15T14:30+08:00[Asia/Shanghai]

场景：跨时区会议、航班时刻

I

Instant

时间戳（机器视角）

2024-01-15T06:30:00Z

场景：记录事件、API 时间戳

P/D

Period / Duration

时间间隔

P1Y2M3D / PT2H30M

场景：年龄计算、耗时统计

迁移速查：旧 → 新

Date → Instant / LocalDateTime 
// Date → Instant
Instant instant = oldDate.toInstant();

// Date → LocalDateTime
LocalDateTime ldt = oldDate.toInstant()
    .atZone(ZoneId.systemDefault())
    .toLocalDateTime();

// Instant → Date
Date date = Date.from(instant);
Calendar → ZonedDateTime / LocalDateTime 
// Calendar → ZonedDateTime
ZonedDateTime zdt = ZonedDateTime.ofInstant(
    calendar.toInstant(),
    calendar.getTimeZone().toZoneId()
);

// Calendar → LocalDateTime
LocalDateTime ldt = zdt.toLocalDateTime();
SimpleDateFormat → DateTimeFormatter 
// ❌ 旧方式：线程不安全
SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd");
String str = sdf.format(new Date());

// ✅ 新方式：线程安全，可声明为 static final
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd");
String str = LocalDate.now().format(fmt);
java.sql.Date / Timestamp → LocalDate / Instant 
// java.sql.Date → LocalDate
LocalDate ld = sqlDate.toLocalDate();

// java.sql.Timestamp → LocalDateTime / Instant

Instant instant = sqlTimestamp.toInstant();

// 反向转换
java.sql.Date sd = java.sql.Date.valueOf(localDate);
Timestamp ts = Timestamp.valueOf(localDateTime);
