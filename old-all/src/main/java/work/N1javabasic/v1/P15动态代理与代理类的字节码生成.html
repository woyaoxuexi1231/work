代理模式架构概览

动态代理允许在运行时创建代理对象，而无需为每个被代理类手动编写代理类。Java 提供了两种主要的动态代理实现方式，它们在字节码生成机制上有着本质的区别。

JDK 动态代理

基于接口实现，使用 ProxyGenerator 生成标准字节码，生成 $Proxy0 类。

CGLIB 代理

基于子类继承，使用 ASM 库直接操作字节码，生成目标类的子类。

JDK PROXY

JDK 动态代理

JDK 动态代理是 Java 标准库提供的代理机制，<要求目标类必须实现至少一个接口。 运行时通过 java.lang.reflect.Proxy 生成名为 $Proxy0 的代理类字节码。

HelloService.java 
public interface HelloService {
    void sayHello(<String name);
}

public class HelloServiceImpl implements HelloService {
    public void sayHello(<String name) {
        System.out.println(<"Hello, " + name);
    }
}
JdkProxyDemo.java 
HelloService proxy = (HelloService) Proxy.<newProxyInstance(
    HelloServiceImpl.<class.<getClassLoader(),
    new Class[]{<HelloService.<class},
    (proxy1, method, args) -> {
        System.out.println(<"Before...");
        Object result = method.<invoke(target, args);
        System.out.println(<"After...");
        return result;
    }
);

ProxyGenerator 字节码生成流程

1. 生成类头信息
public final class $Proxy0 extends Proxy implements HelloService
2. 生成构造方法
$Proxy0(InvocationHandler h) { super(h); }
3. 生成接口方法实现
sayHello(String) → h.invoke(this, m3, new Object[]{name})
4. 生成静态字段
private static Method m1, m2, m3...

核心限制

必须实现接口
目标类必须实现至少一个接口，否则抛出 IllegalArgumentException
只能代理接口方法
无法代理目标类中未在接口定义的方法

CGLIB

CGLIB 动态代理

CGLIB (Code Generation Library) 基于 ASM 字节码操作框架， 通过生成目标类的子类来实现代理。 无需接口，但无法代理 final 类或 final 方法。

UserService.java (无接口) 
public class UserService {
    public void saveUser(<String name) {
        System.out.println(<"Saving: " + name);
    }

    public final void deleteUser(<Long id) {
        // final 方法无法被代理!
    }
}
CglibProxyDemo.java 
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(<UserService.<class);
enhancer.setCallback((<MethodInterceptor) (obj, method, args, proxy) -> {
    System.out.println(<"CGLIB Before...");
    Object result = proxy.<invokeSuper(<obj, args);
    System.out.println(<"CGLIB After...");
    return result;
});

UserService proxy = (UserService) enhancer.create();
proxy.saveUser(<"Alice");

ASM 字节码生成流程

1
生成子类头
class UserService$$Enhancer extends UserService
2
重写非 final 方法
生成 saveUser(String) 代理逻辑
3
插入拦截逻辑
methodProxy.invokeSuper() 回调
4
使用 ClassWriter 输出
直接写入 JVM 字节码格式

核心限制

无法代理 final 类
final 类不可被继承，因此无法生成子类代理
无法代理 final 方法
final 方法不可被重写，代理类中无法插入拦截逻辑
目标类不能是 private
代理类需要继承目标类，无法访问 private 类

ASM 直接字节码操作原理

底层实现

CGLIB 底层依赖 ASM 库直接操作字节码，不通过编译器生成 .java 文件，而是直接在内存中构建 Class 文件格式的字节数组。

ClassReader ClassWriter MethodVisitor Opcodes
// ASM 生成子类核心逻辑
ClassWriter cw = new ClassWriter(<ClassWriter.COMPUTE_FRAMES);
cw.visit(V1_8, ACC_PUBLIC,
"UserService$$Enhancer", null,
"UserService", null);
// 生成重写方法的字节码...
MethodVisitor mv = cw.visitMethod(...);
mv.visitCode();
// 直接插入 ALOAD, INVOKEVIRTUAL 等指令

JDK 动态代理 vs CGLIB

全面对比两种代理机制的实现差异与适用场景

特性 JDK 动态代理 CGLIB
实现基础 接口实现 子类继承
字节码生成工具 ProxyGenerator (标准 JDK) ASM 库
生成类名 $Proxy0, $Proxy1... Target$$EnhancerByCGLIB$$...
要求目标类 必须实现接口 不能是 final 类
方法限制 只能代理接口中的方法 不能代理 final / private 方法
性能 反射调用，稍慢 使用 FastClass 机制，更快
依赖 JDK 内置，无额外依赖 需要引入 cglib/asm 依赖
典型应用 Spring AOP (有接口时) Spring AOP (无接口时)

Spring AOP 的自动选择策略

Spring Framework 默认优先使用 JDK 动态代理（当目标类实现了接口时）。 如果目标类没有实现接口，Spring 会自动回退到 CGLIB 代理。 可以通过配置 @EnableAspectJAutoProxy(proxyTargetClass = true) 强制使用 CGLIB 代理。

CGLIB ASM 生成过程

Enhancer 关键源码逻辑
class Enhancer extends AbstractClassGenerator {
protected void generateClass(<ClassVisitor v) {
ClassEmitter ce = new ClassEmitter(v);
// 声明类: extends 目标类
ce.begin_class(..., superclass, ...);
// 生成构造方法
emitConstructors(ce);
// 重写所有非 final 方法
emitMethods(ce);
ce.end_class();
}
}
生成的方法拦截逻辑
// 重写 saveUser(String)
public void saveUser(String name) {
MethodInterceptor interceptor = CGLIB$CALLBACK_0;
if (interceptor != null) {
interceptor.intercept(
this, // obj
CGLIB$saveUser$0$Method, // method
new Object[]{name}, // args
CGLIB$saveUser$0$Proxy // proxy
);
} else {
super.saveUser(name);
}
}

CGLIB FastClass 性能优化机制

1. 索引分配

为每个方法分配唯一整数索引，避免反射查找 Method 对象的开销。

2. 直接调用

通过 switch(index) 直接分发到目标方法，绕过反射 invoke。

3. 双重生成

为目标类和代理类各生成一个 FastClass，实现双向高效调用。
