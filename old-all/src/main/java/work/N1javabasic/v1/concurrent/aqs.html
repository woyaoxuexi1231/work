AbstractQueuedSynchronizer

独占锁 & 共享锁 · CLH 队列 · 状态管理 · 条件变量 · 大厂高频面试深度解析

核心三要素

  • statevolatile int state，表示同步状态（例如锁的可重入计数、信号量剩余许可）
  • CLH队列FIFO双向链表，存放等待线程的Node节点，实现公平/非公平获取锁
  • CAS + LockSupport原子更新state，线程挂起/唤醒基于Unsafe.park/unpark
面试高频：AQS 是 JUC 同步器的基石，ReentrantLock、Semaphore、CountDownLatch 均继承AQS。

模板方法模式

AQS 定义好骨架，子类只需实现：

• tryAcquire(int arg) — 独占式获取
• tryRelease(int arg) — 独占式释放
• tryAcquireShared(int) — 共享式获取
• tryReleaseShared(int) — 共享式释放
• isHeldExclusively() — 是否独占
AQS 提供 acquire/release 等 final 方法，内部调用上述钩子 + 队列维护。

Node 等待状态

CANCELLED = 1
线程超时/中断，节点取消
SIGNAL = -1
后继节点需要被唤醒
CONDITION = -2
节点处于条件队列
PROPAGATE = -3
共享模式下传播释放
面试问：为何用 int 表示状态？负值表示有效等待，正值表示无效。

🔗 CLH 队列（FIFO 双向链表）

🔥 高频考点

AQS 内部维护一个虚拟的双向队列（CLH 变种），每个线程被包装成 Node，通过 CAS 尾插法入队。head 节点为当前持有锁的线程，后继节点不断自旋/park 等待唤醒。

head (持有锁)
thread=null
waitStatus=SIGNAL?
Node A
thread=Thread-1
waitStatus=-1
prev → head
Node B
thread=Thread-2
waitStatus=-1
prev → NodeA
tail (尾节点)
next = null
CLH 队列示意图：head 指向当前锁占用者，tail 指向最新等待者，入队通过 CAS 保证线程安全。
面试题：CLH队列优点？—— 无锁化入队，避免线程竞争，扩展性强；每个节点自旋检测前驱节点状态。
独占锁获取流程：acquire → tryAcquire失败 → addWaiter → acquireQueued (自旋/park)。
独占模式获取锁 (acquire)面试常考
1️⃣ tryAcquire(arg) 尝试直接获取 state，成功则直接返回，否则进入下一步。
2️⃣ addWaiter(Node.EXCLUSIVE) 将当前线程封装成独占Node，CAS插入队列尾部。
3️⃣ acquireQueued(node, arg) 在队列中“自旋”获取锁：判断前驱是否为head，若是则再次tryAcquire；成功则设置head为当前节点，失败则 shouldParkAfterFailedAcquire 将前驱状态设为SIGNAL，并 LockSupport.park() 阻塞。
4️⃣ 若线程在阻塞期间被中断， acquireQueued 返回 true，并在 acquire 方法中自我中断恢复。
伪代码: public final void acquire(int arg) { if (!tryAcquire(arg) && acquireQueued(addWaiter(Node.EXCLUSIVE), arg)) selfInterrupt(); }
独占模式释放锁 (release)唤醒后继
🔓 tryRelease(arg) 子类实现：释放同步状态（例如 state - 1 变为0时返回true）。
🔓 若释放成功，检查头节点（head）状态，如果 waitStatus != 0，则 unparkSuccessor(h) 唤醒后继节点中的线程。
🔓 后继节点从 park() 醒来，继续自旋尝试 acquire，最终获得锁并成为新 head。
注意：ReentrantLock 中 state 表示重入次数，完全释放后唤醒等待队列。
面试点：唤醒操作从队列头部开始，保证公平性；非公平锁在 tryAcquire 时可能插队。

共享模式 (Shared)

Semaphore / CountDownLatch

共享模式下，多个线程可以同时获取同步状态（如信号量许可证）。acquireShared 内部调用 tryAcquireShared，返回值 >=0 表示成功；若 <0 则进入共享队列等待，并且唤醒传播机制（PROPAGATE）可依次唤醒多个后继节点。

public final void acquireShared(int arg) {
 if (tryAcquireShared(arg) < 0)
  doAcquireShared(arg);
}
释放时 releaseShared 会尝试唤醒多个等待线程，实现并发执行。
面试热点：CountDownLatch 基于共享锁实现，state 初始为 N，每调用 countDown 就释放，直到 0 唤醒所有等待线程。

可重入性实现原理

ReentrantLock 通过 AQS 的 state 字段记录重入次数。同一线程再次获取锁时，tryAcquire 判断当前线程 == exclusiveOwnerThread，执行 state+1 并返回 true。释放锁时 state-1，直到 state=0 才真正释放锁并唤醒后继。

// 伪代码：ReentrantLock.Sync.tryAcquire
Thread current = Thread.currentThread();
int c = getState();
if (c == 0) { 尝试CAS加锁 }
else if (current == getExclusiveOwnerThread()) {
   int nextc = c + acquires; setState(nextc); return true; }
非公平锁在 tryAcquire 时会立即抢占，公平锁则检查队列是否有前驱。
Condition 条件队列 (await/signal)
面试高频: 与Object wait/notify区别
await() 核心流程
  • 当前线程必须持有锁 (独占模式)。
  • 将线程包装成 Node（状态 CONDITION）加入条件队列。
  • 完全释放锁 (fullyRelease) 并唤醒同步队列中的后继节点。
  • 线程 park 阻塞，直到被 signal 或中断。
  • 被唤醒后重新竞争锁 (acquireQueued)，恢复执行。
每个 Condition 拥有独立的条件队列（单向链表），与 AQS 同步队列分离。
signal() / signalAll()
  • 检查当前线程是否持有锁，否则抛异常。
  • 将条件队列中首节点转移到同步队列（CLH队列尾），修改 waitStatus 为 0 或 SIGNAL。
  • signalAll 遍历整个条件队列移动所有节点。
  • 被转移的线程在同步队列中重新尝试获取锁。
面试题：Condition 与 wait/notify 优势：支持多个等待队列，可中断/不可中断，超时等待，更灵活。

🔥 大厂面试高频题 · 深度剖析

Q1：AQS 的核心思想是什么？为什么 JUC 基于它实现？必问

AQS 提供了一个 FIFO 等待队列 和 volatile int state 的框架，子类只需实现状态获取/释放的逻辑，而复杂的线程排队、阻塞唤醒、中断处理全部由 AQS 完成。这分离了“同步状态管理”与“线程调度”，使得 ReentrantLock、Semaphore 等同步器可以快速构建。

Q2：AQS 如何避免“惊群效应”？在共享模式中怎么处理？扩展

独占模式释放时只唤醒 head 的后继节点，不会有多个线程同时唤醒。共享模式通过 doReleaseShared 循环唤醒，并且使用 PROPAGATE 状态传播唤醒，避免所有等待线程同时被唤醒导致的竞争。同时 AQS 的 unparkSuccessor 只唤醒一个线程，共享模式下唤醒后节点会继续尝试唤醒下一个，从而串行化。

Q3：公平锁和非公平锁在 AQS 中的区别是什么？高频对比

公平锁的 tryAcquire 会先检查 hasQueuedPredecessors()，如果队列中有等待线程则放弃获取，保证顺序。非公平锁直接 CAS 抢占，可能造成新线程插队。性能上非公平更高，但可能引起线程饥饿。AQS 通过子类实现两种策略，基类提供队列管理。

Q4：AQS 中 Node 的 waitStatus 如何影响锁的传递？源码级别

waitStatus = SIGNAL (-1) 表示当前节点释放锁时必须唤醒后继节点。当节点被取消（CANCELLED = 1）会被移除队列。在 shouldParkAfterFailedAcquire 中会将前驱节点状态从0更新为 SIGNAL。CONDITION 状态用于条件队列，PROPAGATE 用于共享模式传播。理解这些状态是读懂 acquireQueued 和 release 源码的关键。

Q5：AQS 如何支持中断响应和超时获取？

AQS 提供 acquireInterruptibly 和 tryAcquireNanos 等方法。当线程被中断时，会抛出 InterruptedException 而不是忽略。超时获取通过 LockSupport.parkNanos 实现，在循环中检查时间并跳出，同时处理中断标识。这是 AQS 可扩展性的体现。
