ReentrantReadWriteLock 实现原理

读锁共享 · 写锁独占 · 32位 state 拆分 · 锁降级 · 公平与非公平策略

state 拆分设计

AQS 的 state 是 32 位 int，高 16 位表示读锁重入次数（共享计数），低 16 位表示写锁重入次数（独占计数）。通过 c >>> 16 获取读计数，c & 0xFFFF 获取写计数。

面试要点：为什么用 32 位？单个锁重入深度理论 65535，足够实际应用。

读锁 · 共享模式

  • 共享锁，多个线程可同时持有
  • 每个线程维护自己的读锁重入计数（ThreadLocal HoldCounter）
  • 获取条件：无写锁且无写请求（或公平策略下不插队）

写锁 · 独占模式

  • 独占锁，同一时刻只有一个线程能获取
  • 获取条件：无读锁 && 无其他写锁
  • 支持重入（低16位计数）

核心设计
高16位：读锁计数 (shared count)
每次读锁重入 +1，支持最多 65535
低16位：写锁计数 (exclusive count)
写锁重入次数，最多 65535
示例：state = (读计数 << 16) + 写计数。通过 sharedCount(c) 和 exclusiveCount(c) 快速分解。
面试金句：读写锁用一个原子 int 同时管理两种锁状态，避免了额外的字段或锁竞争，是 AQS 精巧设计的典范。
写锁获取 (tryAcquire) 逻辑独占模式
1️⃣ 获取当前 state 和写锁计数。如果 state != 0，进入下一步；否则尝试 CAS 直接获取写锁。
2️⃣ 如果写锁计数为 0 但有读锁（state≠0 且写计数=0），则获取失败（读锁优先于写锁，避免写饥饿）。
3️⃣ 如果写锁计数不为 0，判断当前线程是否是写锁持有者：是则重入（state + 1），否则失败。
4️⃣ 非公平模式下直接 CAS 竞争；公平模式下需检查队列中是否有前驱节点，有则失败排队。
关键源码：if (w == 0 && current != getExclusiveOwnerThread()) return false; // 读锁存在时阻塞写锁
读锁获取 (tryAcquireShared) 逻辑共享模式
1️⃣ 如果写锁被其他线程持有，则获取失败（写锁排他）。
2️⃣ 如果当前线程持有写锁，允许获取读锁（锁降级的关键）。
3️⃣ 检查是否需要阻塞（公平模式下队列有等待者则阻塞，非公平模式可能会让读锁插队）。
4️⃣ CAS 更新高16位 (读计数+1)，成功则记录 HoldCounter（线程本地重入计数），失败则重试。
注意：每个线程读锁重入数存在 ThreadLocal 中，不占用 state 高16位（state高16位是总读锁重入次数）。

⬇️ 锁降级 (Lock Downgrade)

超高频面试题

锁降级是指 写锁 → 读锁 的转换：当前线程持有写锁，再获取读锁，然后释放写锁，最终降级为读锁。这种机制允许在写操作后继续持有读锁，保证数据可见性，避免其他写线程干扰。

// 标准锁降级代码示例 ReentrantReadWriteLock rwl = new ReentrantReadWriteLock(); rwl.writeLock().lock(); // 获取写锁 try { // 修改数据... rwl.readLock().lock(); // 获取读锁 (降级) } finally { rwl.writeLock().unlock(); // 释放写锁，此时降级为读锁 } // 继续持有读锁进行后续读操作 try { // 读取数据... } finally { rwl.readLock().unlock(); }

⚠️ 注意：不支持锁升级（读锁 → 写锁），因为读锁被多个线程持有时，升级为写锁会造成死锁风险。

面试追问：锁降级的价值？保证写操作之后的数据一致性，避免释放写锁后其他写线程修改导致读数据不一致。常见于缓存读写场景。

公平读写锁

  • 线程按照请求顺序获取锁，避免写饥饿
  • 读锁和写锁都遵循 FIFO 队列
  • 性能相对非公平稍低，但公平性有保障
  • 实现：调用 hasQueuedPredecessors() 检查队列中是否有等待更久的线程

非公平读写锁

  • 写锁、读锁均可插队，提高吞吐量
  • 可能导致写线程长时间等待（写饥饿）
  • 读锁插队：如果队列头节点不是写锁等待，读锁可以直接获取
  • 默认策略，适用于读多写少、短暂竞争场景

重入计数：state 低位 + ThreadLocal HoldCounter

写锁重入

写锁重入次数直接记录在 state 低 16 位，因为写锁是独占的，一个线程可多次重入，释放时逐次递减。

state += 1; // 每次重入增加写计数

读锁重入

每个线程的读锁重入数通过 HoldCounter 存储于 ThreadLocal，state 高16位记录全局读锁重入总数（便于快速判断有无读锁）。

// 每个线程的 HoldCounter 缓存读重入次数
static final class HoldCounter {
int count; long tid;
}
为什么要分离？因为读锁是共享的，如果每个线程的重入次数都占用 state 位会导致状态爆炸。ThreadLocal 方式高效且准确。

写锁释放 tryRelease

将 state 低16位减1，若写计数变为0则释放独占锁，设置 exclusiveOwnerThread = null，并返回 true。然后唤醒 AQS 队列中的后继节点。

int nextc = getState() - releases; if (exclusiveCount(nextc) == 0) { setExclusiveOwnerThread(null); } setState(nextc);

读锁释放 tryReleaseShared

state 高16位减1，同时当前线程的 HoldCounter 计数减1（若减至0则移除）。如果读计数变为0，表示无任何读锁持有，返回 true 可能唤醒等待的写锁。

int c = getState(); int nextc = c - (1 << 16); if (nextc < 0) throw ...; // 同时清理 HoldCounter

🔥 大厂高频面试题 · ReentrantReadWriteLock 专项

Q1：读写锁为什么能同时提高并发性能？底层如何保证读-写互斥？必问

读写锁允许多个读线程并发，提高读多写少场景的吞吐量。互斥通过 AQS 的 state 区分：写锁获取时必须判断读锁计数为0且无其他写锁；读锁获取时必须判断写锁未被其他线程持有。底层使用 CAS 更新 state 实现原子性检查和修改。

Q2：什么是锁降级？ReentrantReadWriteLock 支持锁升级吗？为什么？核心

锁降级是写锁→读锁的过程，支持。但锁升级（读→写）不支持，因为如果有多个读线程同时持有读锁，任何线程尝试获取写锁都会导致死锁（互相等待释放读锁）。JVM 无法安全实现锁升级，会直接死锁或抛异常。

Q3：读写锁的公平/非公平模式下，读锁插队行为有何不同？细节

非公平读锁可以插队：如果当前同步队列头节点不是写锁等待线程，则读锁可直接获取，导致写锁饥饿。公平模式下，读锁必须排队，即使没有写锁等待也要遵循 FIFO，避免写饥饿。因此公平读写锁更适用需要严格公平的场景。

Q4：读写锁如何避免写线程饿死？

JDK 非公平读写锁存在写饥饿风险，但可以通过 fair=true 公平模式避免。此外，内部实现中，如果队列头节点是写锁等待，新的读锁无法插队（会排队），这也是缓解写饥饿的一种设计。推荐读多写少且允许短暂写饥饿时使用非公平，写频率较高时用公平。

Q5：读写锁的重入计数如何实现？HoldCounter 的作用？

写锁重入用 state 低16位。读锁重入需要记录每个线程的读锁重入次数，因为读锁是共享的且一个线程可能多次重入，若只用 state 高16位无法区分线程。因此通过 ThreadLocal<HoldCounter> 存储每个线程的重入数，HoldCounter 还缓存了线程ID以优化性能。释放时减少对应计数。

Q6：ReentrantReadWriteLock 与 StampedLock 的区别？进阶

StampedLock 提供乐观读锁，在读多写极少场景性能更高，但不支持重入，且不支持条件变量。ReentrantReadWriteLock 支持重入、公平性选择、锁降级，API 简单。面试中可提及 StampedLock 的乐观读避免了写锁阻塞读的问题，但重入不足。
