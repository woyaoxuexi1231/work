ForkJoinPool 工作窃取

每个线程独立双端队列 · 本地 LIFO · 窃取 FIFO · 分治任务 · 适用递归/归并场景

双端队列 (Deque)

每个工作线程拥有自己的 双端队列（WorkQueue），存放待执行的 ForkJoinTask。队列支持两端操作：本地线程从一端（底端）进行 push/pop（LIFO），而其他线程（窃取者）从另一端（顶端）进行 poll（FIFO）。

工作窃取 (Work-Stealing)

空闲线程主动从其他繁忙线程的队列尾部（顶端）“窃取”任务，窃取采用 FIFO 顺序。这种机制平衡负载，减少竞争，充分利用 CPU 资源。

优势：无需全局任务队列，降低锁竞争；适应分治任务中动态产生的子任务。

RecursiveTask / RecursiveAction

RecursiveTask<V> 有返回值的分治任务；RecursiveAction 无返回值。核心方法 compute() 中拆分任务，通过 fork() 和 join() 进行子任务调度。

面试要点：双端队列设计是工作窃取的核心，本地 LIFO 利用任务局部性（新创建的子任务往往很快被执行），窃取 FIFO 减少了窃取者与本地线程的竞争。ForkJoinPool 默认线程数 = CPU 核心数，充分利用并行计算。
RecursiveTask 实现（有返回值）
class SumTask extends RecursiveTask<Long> {
    private final int[] array;
    private final int start, end;
    private static final int THRESHOLD = 10_000;

    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) sum += array[i];
            return sum;
        }
        int mid = (start + end) / 2;
        SumTask leftTask = new SumTask(array, start, mid);
        SumTask rightTask = new SumTask(array, mid, end);
        leftTask.fork();        // 异步执行子任务
        long rightResult = rightTask.compute(); // 当前线程继续执行右边任务
        long leftResult = leftTask.join();     // 等待左边结果
        return leftResult + rightResult;
    }
}
关键点：fork() 将任务提交到当前线程的队列，join() 阻塞等待结果。避免对子任务直接调用 fork 两次导致过多任务创建，典型写法是 leftTask.fork(); rightTask.compute(); leftTask.join();
分治执行流程
1️⃣ 任务被提交给 ForkJoinPool，创建根任务。
2️⃣ 在 compute() 中，若任务足够小，直接计算；否则拆分为两个子任务。
3️⃣ 调用 leftTask.fork() 将子任务异步放入当前线程队列。
4️⃣ 当前线程继续执行 rightTask.compute()（或再次 fork）。
5️⃣ 调用 leftTask.join() 等待子任务完成并获取结果。
6️⃣ 工作窃取机制：空闲线程会从其他线程队列末尾窃取任务，分担负载。
fork() 不会立刻产生新线程，而是将任务加入队列；join() 会在等待时帮助执行其他任务（work-stealing 优化）。

⚖️ ForkJoinPool 与 ThreadPoolExecutor 核心对比

对比维度ForkJoinPoolThreadPoolExecutor
任务队列类型每个线程独立的双端队列 (WorkQueue)所有线程共享一个阻塞队列 (BlockingQueue)
负载均衡策略工作窃取 (work-stealing)，空闲线程窃取其他线程队列任务任务在共享队列中竞争，空闲线程从共享队列获取任务
适用场景分治算法、递归任务、任务频繁生成子任务（如排序、矩阵运算）独立任务、I/O 密集型或 CPU 密集型但任务间无依赖
任务类型ForkJoinTask（RecursiveTask / RecursiveAction）Runnable / Callable
工作线程数通常设置为 CPU 核心数（默认并行度）根据任务类型灵活配置（CPU 密集型 = 核心数，I/O 密集型 = 更大）
任务窃取开销窃取时 CAS 操作，竞争较小共享队列存在锁竞争（尤其高并发时）
面试总结：ForkJoinPool 通过工作窃取避免了传统线程池中共享队列的竞争瓶颈，特别适合分治场景。而 ThreadPoolExecutor 在独立任务、无依赖场景下更简单高效。
📊 经典案例：归并排序 / 数组求和 / 斐波那契数列

ForkJoinPool 天生适用于可以递归分解的任务。下面是一个使用 RecursiveAction 实现的并行归并排序片段：

class MergeSortTask extends RecursiveAction {
    private final int[] array;
    private final int lo, hi;
    private static final int THRESHOLD = 1000;

    protected void compute() {
        if (hi - lo <= THRESHOLD) {
            insertionSort(array, lo, hi);
            return;
        }
        int mid = (lo + hi) >>> 1;
        MergeSortTask left = new MergeSortTask(array, lo, mid);
        MergeSortTask right = new MergeSortTask(array, mid, hi);
        invokeAll(left, right);   // fork 两个子任务并等待完成
        merge(array, lo, mid, hi);
    }
}
invokeAll() 是批量 fork 并 join 的便捷方法，内部利用工作窃取提升效率。并行分治框架在排序、流式计算、图像处理等领域广泛应用。

🔥 大厂高频面试题 · ForkJoinPool 专项

Q1：工作窃取算法的核心是什么？为何本地 LIFO 而窃取 FIFO？必问

核心是每个线程拥有独立任务队列，空闲线程从其他队列尾部取任务。本地 LIFO 保证最近创建的子任务优先执行（利用任务局部性，减少内存和递归开销）；窃取 FIFO 防止与本地线程竞争队列同一端，降低 CAS 冲突。同时，LIFO 适用于深度优先遍历，而窃取大块任务有助于负载均衡。

Q2：ForkJoinPool 如何避免线程饥饿和死锁？深度

ForkJoinPool 实现了 join() 的“等待-帮助”机制：当线程调用 join() 等待子任务时，不会阻塞，而是执行其他可窃取的任务（包括自己队列中的任务或其他队列的任务）。这避免了死锁，并充分利用线程。另外，默认的并行度等于 CPU 核心数，避免过多线程竞争。

Q3：ForkJoinPool 与 Java 8 并行流 (parallelStream) 的关系？常用

并行流底层使用 ForkJoinPool.commonPool()（公共池）。但需要注意，commonPool 是全局共享的，大型阻塞任务可能影响其他并行流。可以通过 ForkJoinPool 自定义池隔离任务。parallelStream 适用于无状态、CPU 密集型的聚合操作（如 reduce、map）。

Q4：ForkJoinTask 的 fork() 和 submit() 有什么区别？

fork() 将任务异步提交给当前线程的队列，不保证立刻执行；submit() 是 ForkJoinPool 的方法，将任务提交到池中，可能被任意线程窃取。在 RecursiveTask 内部通常使用 fork()，而外部使用 pool.submit(rootTask)。

Q5：ForkJoinPool 的最大并行度如何设置？为什么通常设为 CPU 核心数？

ForkJoinPool 并行度默认为 Runtime.getRuntime().availableProcessors()。因为 ForkJoinTask 多为 CPU 密集型计算，线程数超过核心数反而会导致上下文切换开销。如果需要执行阻塞 I/O，可以通过构造器增加并行度或使用 ManagedBlocker 接口。

⚠️ 使用 ForkJoinPool 的注意事项

避免在 ForkJoinTask 中使用阻塞操作（如 Thread.sleep、网络IO），否则会导致线程耗尽。必要时使用 ManagedBlocker 适配。
不要手动调用 ForkJoinTask.adapt(Runnable) 包装阻塞任务，可能会影响窃取效率。
尽量确保任务拆分足够细但不过细（阈值经验约为 100-10000 次操作），避免任务数量巨大导致开销上升。
使用 invokeAll() 优于手动 fork/join，它内部会优化任务调度。
自定义 ForkJoinPool 时，记得在 finally 中调用 shutdown()，或使用 try-with-resources (Java 19+ 支持 AutoCloseable)。
监控池状态：getStealCount() 可查看窃取任务总数，getRunningThreadCount() 查看活动线程。
