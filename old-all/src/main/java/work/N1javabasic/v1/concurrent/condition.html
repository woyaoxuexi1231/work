Condition 等待通知机制

独立等待队列 · await释放锁 · signal转移节点 · 多等待集 · 比wait/notify更强大的线程协作

独立等待队列

每个 Condition 对象内部维护一个 单向条件队列 (firstWaiter → lastWaiter)，与 AQS 同步队列完全分离。一个 Lock 可以创建多个 Condition，实现多路等待/通知。

面试亮点：解决了 Object.wait/notify 无法区分等待条件的痛点。

await 核心步骤

  • 当前线程必须持有锁（否则抛异常）
  • 将当前线程封装为 Node (CONDITION) 加入条件队列
  • fullyRelease 完全释放锁（包括重入次数）
  • park 阻塞，等待 signal 或中断
  • 被唤醒后重新竞争锁（acquireQueued）

signal 核心步骤

  • 持有锁调用 signal，检查条件队列非空
  • 将条件队列首节点 (或所有节点) 转移到 AQS 同步队列尾部
  • 修改节点状态：waitStatus 从 CONDITION → 0 或 SIGNAL
  • 同步队列中的节点随后可参与锁竞争

面试核心：Object.wait/notify 只有一个等待队列，而 Condition 允许一个 Lock 创建多个等待队列（例如生产者-消费者模型中分别有 notFull 和 notEmpty），实现更精细的线程调度。
Condition.await() 详细流程释放锁 + 阻塞
1️⃣ 当前线程必须已经持有锁（AQS 独占模式），否则抛出 IllegalMonitorStateException。
2️⃣ 将当前线程包装成 Node，状态设为 CONDITION = -2，并通过 addConditionWaiter() 加入条件队列尾部。
3️⃣ 调用 fullyRelease(node)：释放当前线程持有的所有锁（state 归零），并唤醒 AQS 同步队列中的后继节点。
4️⃣ 进入阻塞循环：LockSupport.park(this)，直到被 signal 或 中断。如果被中断则抛出 InterruptedException 或记录中断状态。
5️⃣ 被 signal 唤醒后，节点从条件队列中移除，并通过 acquireQueued(node, savedState) 重新竞争锁，成功返回后，await 方法返回。
源码要点：addConditionWaiter 中若 lastWaiter 被取消则遍历清除；fullyRelease 支持重入锁完全释放。
Condition.signal() 详细流程转移节点
1️⃣ 检查当前线程是否持有锁，未持有则抛异常。
2️⃣ 从条件队列中取出 firstWaiter（不为空）。
3️⃣ 调用 transferForSignal(node)：将节点从条件队列中移除，并尝试将其添加到 AQS 同步队列尾部。
4️⃣ 修改节点状态：waitStatus 从 CONDITION 改为 0，再将前驱节点状态设为 SIGNAL（若前驱取消则自旋）。
5️⃣ 成功入队后，如果前驱节点的 waitStatus > 0 或设置 SIGNAL 失败，则直接 LockSupport.unpark(node.thread) 唤醒线程。
6️⃣ signalAll 则遍历整个条件队列，对每个节点重复上述转移。
注意：signal 不会立即唤醒等待线程，只是将节点移到同步队列；被转移的线程仍需在 acquireQueued 中自旋竞争锁。

💡 典型应用：使用两个 Condition 实现有界阻塞队列生产者-消费者

通过 notFull 和 notEmpty 两个 Condition，分别控制队列不满和队列不空的条件，避免不必要的线程唤醒。

class BoundedQueue<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Object[] items;
    private int putIndex, takeIndex, count;

    public void put(T item) throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == items.length)
                notFull.await();      // 队列满，等待 notFull 条件
            items[putIndex] = item;
            if (++putIndex == items.length) putIndex = 0;
            count++;
            notEmpty.signal();        // 唤醒一个等待 notEmpty 的消费者
        } finally { lock.unlock(); }
    }

    public T take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == 0)
                notEmpty.await();     // 队列空，等待 notEmpty 条件
            T item = (T) items[takeIndex];
            if (++takeIndex == items.length) takeIndex = 0;
            count--;
            notFull.signal();         // 唤醒一个等待 notFull 的生产者
            return item;
        } finally { lock.unlock(); }
    }
}
面试延伸：通过两个 Condition 精准控制，比使用 notifyAll 避免“惊群效应”且提高效率。

📌 大厂面试高频题 · Condition 深度剖析

Q1：一个 Lock 可以创建多个 Condition，它们之间有什么关系？节点能否跨条件队列转移？必问

每个 Condition 拥有独立的条件队列（单向链表），不同 Condition 的节点完全隔离。调用 signal 只能将当前 Condition 队列中的节点转移到 AQS 同步队列，绝不可能将其他 Condition 的节点误转移。这样实现了 等待集分离，例如生产者线程只等待 notFull，消费者只等待 notEmpty。

Q2：await 释放锁时，如果锁是可重入的（如 ReentrantLock），如何完全释放？源码细节

AQS 提供的 fullyRelease 方法会循环调用 tryRelease，直到 state 变为 0，并返回保存的重入次数。被唤醒后，在 acquireQueued 中会使用该保存的状态重新获取锁（恢复重入次数）。这保证了 await 前后锁状态一致性。

Q3：signal 操作是立即唤醒等待线程吗？被唤醒的线程能立刻执行吗？容易混淆

signal 只是将节点从条件队列转移到同步队列，并修改节点状态（如果前驱是取消状态则可能直接 unpark）。但线程要真正执行，必须先通过 acquireQueued 竞争锁成功。所以被唤醒的线程不会立即运行，而是进入同步队列排队等待锁，符合锁的公平/非公平策略。

Q4：Condition 的 await 方法为什么必须放在 while 循环中？最佳实践

原因有三：① 防止虚假唤醒（spurious wakeup），线程可能在未被 signal/notify 的情况下醒来；② 从 await 返回后，条件可能仍未满足（例如队列再次被其他线程写满）；③ 多条件协作时，signal 不保证条件立刻成立，必须重检查。因此官方文档强制要求 while 循环。

Q5：Condition 的等待队列中节点如何被清理（取消等待）？中断如何影响？

在 await 时如果线程发生中断，且方法支持中断（如 await() 会抛出 InterruptedException），节点会从条件队列中移除并转移到同步队列，然后抛出异常。另外，在条件队列遍历时（如 signal 前），会检查并跳过 waitStatus 为 CANCELLED 的节点。同时 awaitNanos 超时后也会自动将节点转移到同步队列并返回 false。

🔧 Condition 内部实现关键点总结

条件队列为单向链表，由 firstWaiter, lastWaiter 指针维护，节点使用 nextWaiter 连接。
Node 的状态 CONDITION (-2) 仅用于条件队列，表示节点正在等待条件。
transferForSignal 将节点从条件队列移除后，调用 enq 入同步队列，并设置前驱为 SIGNAL。
await 支持超时和中断：awaitNanos, awaitUntil, awaitUninterruptibly 提供灵活变种。
JUC 阻塞队列 (ArrayBlockingQueue, LinkedBlockingQueue) 底层均使用 Condition 实现。
ReentrantLock.newCondition() 返回 ConditionObject，它是 AQS 内部类。
