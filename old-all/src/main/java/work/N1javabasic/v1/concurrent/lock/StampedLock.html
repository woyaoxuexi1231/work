StampedLock 深度解析

乐观读 · 悲观读 · 写锁 · 锁转换 · 不可重入 · 与 ReentrantReadWriteLock 全面对比

乐观读 (Optimistic Read)

tryOptimisticRead() 获取一个邮戳(stamp)，不加锁，直接读数据。随后通过 validate(stamp) 检查期间是否有写锁发生。无竞争时可获得极高读性能，适用于读多写极少场景。

⚠️ 注意：乐观读不阻塞写锁，但必须验证，失败后需升级为悲观读锁。

悲观读锁 (Read Lock)

readLock() 类似于 ReentrantReadWriteLock 的读锁，共享锁，阻塞写锁。支持重入？不支持重入！ 同一个线程再次获取读锁会死锁。不可重入是 StampedLock 的重要特征。

写锁 (Write Lock)

writeLock() 独占锁，互斥于所有读锁（包括乐观读，因为乐观读验证会失败）。不支持重入，同一线程再次获取写锁会死锁。释放时使用对应的 stamp。

⚡ 乐观读模式 · 无锁读 + 验证机制

StampedLock 精髓

乐观读不获取锁，直接读取数据副本，然后通过 validate(stamp) 检查 stamp 有效性。若期间有写锁发生过，则验证失败，需重试或升级为悲观读锁。这种模式彻底消除了读锁对写锁的阻塞，极大提升并发读性能。

面试高频：乐观读的底层实现基于 state 变量上的版本号（邮戳），写锁每次获取会递增一个奇数位，读锁和写锁操作都会改变 state 的二进制位，validate 比较 stamp 与当前 state 是否匹配，从而检测是否有写锁介入。

🔄 锁转换 (Lock Conversion)

特色能力

StampedLock 支持从乐观读 → 悲观读锁、悲观读锁 → 写锁等转换，无需释放锁重新获取，提高效率。常用方法：tryConvertToWriteLock(stamp)、tryConvertToReadLock(stamp)、tryConvertToOptimisticRead(stamp)，转换成功返回新 stamp，失败则返回 0。

long stamp = sl.readLock();
try {
    // 读操作，发现需要写操作
    long ws = sl.tryConvertToWriteLock(stamp);
    if (ws != 0L) {   // 转换成功
        stamp = ws;
        // 执行写操作
    } else {
        sl.unlockRead(stamp);
        stamp = sl.writeLock();
        // 写操作
    }
} finally {
    sl.unlock(stamp);
}
面试亮点：锁转换避免了锁的释放和重竞争，尤其在读后写的场景下提高性能。但注意转换失败时需要手动释放原锁再重新获取。

📌 典型示例：使用 StampedLock 实现线程安全的 Point 坐标乐观读 + 悲观读 + 写锁
class Point {
    private double x, y;
    private final StampedLock sl = new StampedLock();

    public void move(double dx, double dy) {
        long stamp = sl.writeLock();   // 获取写锁
        try {
            x += dx;
            y += dy;
        } finally {
            sl.unlockWrite(stamp);
        }
    }

    public double distanceFromOrigin() {
        long stamp = sl.tryOptimisticRead();   // 乐观读
        double currentX = x, currentY = y;
        if (!sl.validate(stamp)) {            // 验证失败
            stamp = sl.readLock();            // 升级为悲观读锁
            try {
                currentX = x;
                currentY = y;
            } finally {
                sl.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }
}
以上代码展示了乐观读的经典用法：先乐观读，若验证失败则退化为悲观读锁，保证数据一致性且性能最大化。

🔥 StampedLock 大厂面试高频题

Q1：StampedLock 的乐观读到底有没有锁？如何保证数据一致性？必问

乐观读完全无锁，通过 stamp 版本号机制保证一致性。写锁获取时会递增一个奇数的 stamp，乐观读开始时复制当前 stamp，读取数据后调用 validate(stamp) 比较当前 stamp 是否改变。若没变则数据一致；若变了则表明有写锁介入，需要重新读取（通常退化为悲观读锁）。

Q2：StampedLock 为什么不可重入？如果误用会怎样？易错点

设计上为了避免重入带来的复杂性和性能开销，简化实现。如果同一个线程在没有释放读锁或写锁的情况下再次获取相同锁，会直接导致死锁（线程永久阻塞）。所以使用时必须严格避免重入调用，通常通过代码规范或使用 tryConvertToXxx 替代。

Q3：何时使用 StampedLock 代替 ReentrantReadWriteLock？场景选择

当读操作极其频繁、写操作极少，且对锁重入、条件变量无要求时，使用 StampedLock 可获得更高吞吐量（乐观读避免了读锁对写锁的阻塞）。在写多或需要重入、公平锁的场景，应选择 ReentrantReadWriteLock。另外，StampedLock 不支持 Condition，也无法在写锁中调用 await。

Q4：StampedLock 如何避免写饥饿？与读写锁相比有何不同？

StampedLock 内部基于队列（CLH）和自旋，写锁请求时，如果当前有乐观读或悲观读，写锁会阻塞等待，但乐观读不阻塞写锁（乐观读期间写锁可获取，导致 validate 失败）。在写锁等待时，新的读锁请求（悲观读）可能会被阻塞以避免写饥饿，但非公平性依然存在。相比 ReentrantReadWriteLock 的非公平模式，StampedLock 对于写锁的公平性稍好一些（但没有公平选项），总体上仍然建议在读极少写时使用。

Q5：StampedLock 锁转换（tryConvertToWriteLock）实现原理？失败后如何处理？

转换方法检查当前 stamp 对应的锁状态，如果满足升级条件（如当前持有读锁且无其他线程持有读锁，或当前持有乐观读且无写锁发生），则直接更新 stamp 并返回新 stamp，避免了释放锁再重新获取的竞争。如果转换返回 0，表示转换失败，此时需要手动释放原锁，再获取目标锁。失败原因通常是其他线程同时持有读锁导致无法升级为写锁。

🔧 底层实现简述 & 注意事项

stamp 是一个 long 类型，高 7 位表示锁状态，低 57 位作为版本计数器；写锁每次获取会使版本递增，读锁操作会记录数量。
悲观读锁与写锁共享一个等待队列，采用 CLH 变体，自旋 + park 管理。
重要：StampedLock 不是 ReentrantLock 的替代品，不支持重入，务必在 finally 块中释放。
线程中断可能会导致锁状态损坏，建议使用可中断方法时小心处理。
总结：StampedLock 是并发读写锁的性能杀手锏，但需要开发者更精细地管理锁的生命周期，适合中级以上工程师在特定场景使用。面试中如果能完整讲出乐观读原理和对比 ReentrantReadWriteLock 的优劣，是加分项。
