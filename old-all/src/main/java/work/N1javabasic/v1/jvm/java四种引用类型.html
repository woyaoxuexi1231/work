强引用 Strong Reference

默认引用类型 · 永不回收

概念

强引用是 Java 中最普遍的引用类型，只要强引用存在，垃圾收集器永远不会回收被引用的对象。即使内存不足，JVM 宁愿抛出 OutOfMemoryError 也不会回收强引用对象。

实际例子：你日常写的 Object obj = new Object() 就是强引用。只要 obj 这个变量还在作用域内，或者没有被显式置为 null，对象就不会被 GC。

💡 面试话术

"强引用就是默认的引用类型，代码里直接 new 出来的对象都是强引用。只要引用还在，GC 永远不会动它。内存不够就 OOM，也不会回收强引用对象。"

软引用 Soft Reference

OOM 前回收 · 缓存场景

概念

软引用用来描述一些还有用但非必需的对象。当系统内存不足时（即将发生 OOM 前），JVM 会把软引用对象列入回收范围进行二次回收。如果这次回收后内存仍然不够，才会抛出 OOM。

实际例子：图片缓存。App 加载了很多大图，用软引用包装。内存充足时图片都在；内存紧张时，JVM 自动回收这些图片对象释放内存，App 只需要在获取时判断是否为 null，为 null 则重新加载。

💡 面试话术

"软引用适合缓存场景。比如图片缓存，内存够的时候图片留着，内存不够了 JVM 自动回收这些缓存对象。通过 get() 获取，返回 null 就说明被回收了，需要重新加载。"

弱引用 Weak Reference

下次 GC 即回收 · WeakHashMap / ThreadLocal

概念

弱引用也是描述非必需对象，但它的强度比软引用更弱。被弱引用关联的对象只能生存到下一次垃圾收集发生为止。当 GC 工作时，无论内存是否足够，都会回收只被弱引用关联的对象。

实际例子 1：WeakHashMap。它的键是弱引用的。当某个键不再被外部强引用持有时，下次 GC 时这个键值对会被自动从 Map 中移除。非常适合做自动清理的缓存。

实际例子 2：ThreadLocal。ThreadLocalMap 中的 Entry 继承自 WeakReference，key（ThreadLocal 对象）是弱引用。当 ThreadLocal 的强引用被置为 null 后，下次 GC 会回收 key，但 value 还是强引用，可能造成内存泄漏——这就是为什么使用 ThreadLocal 后必须调用 remove()。

💡 面试话术

"弱引用比软引用更脆弱，每次 GC 都会回收。WeakHashMap 的键就是弱引用，外部不再引用这个键时，GC 后 Map 里对应的条目自动消失。ThreadLocal 也用了弱引用做键，但值是强引用，所以用完必须 remove，否则 value 会泄漏。"

虚引用 Phantom Reference

随时回收 · 仅用于跟踪对象回收 / 堆外内存清理

概念

虚引用是最弱的一种引用关系。一个对象是否有虚引用的存在，完全不会对其生存时间构成影响，也无法通过虚引用来取得一个对象实例。为一个对象设置虚引用关联的唯一目的就是能在这个对象被收集器回收时收到一个系统通知（通过 ReferenceQueue）。

实际例子：DirectByteBuffer 的堆外内存清理。NIO 的 DirectByteBuffer 分配的是堆外内存（不受 JVM 堆 GC 管理）。当 DirectByteBuffer 对象被 GC 回收时，需要同时清理其对应的堆外内存。这个清理动作就是通过虚引用 + ReferenceQueue 机制实现的——Cleaner 对象（虚引用）感知到 DirectByteBuffer 被回收后，调用 unsafe.freeMemory() 释放堆外内存。

💡 面试话术

"虚引用形同虚设，get() 永远返回 null，它的存在就是为了跟踪对象什么时候被回收。最经典的例子是 NIO 的 DirectByteBuffer——对象在 JVM 堆里，但数据在堆外内存。当 DirectByteBuffer 被 GC 回收时，虚引用感知到，触发 Cleaner 去释放堆外内存，防止堆外内存泄漏。"

面试高频追问

Q1

"Java 有哪几种引用类型？分别什么特点？"

Java 有四种引用类型，强度从高到低：强引用（默认，永不回收）、软引用（内存不足时回收，适合缓存）、弱引用（下次 GC 即回收，WeakHashMap/ThreadLocal）、虚引用（形同虚设，仅用于跟踪回收通知，如 DirectByteBuffer 堆外内存清理）。

Q2

"WeakHashMap 是怎么实现自动清理的？"

WeakHashMap 的 Entry 继承自 WeakReference，它的 key 是弱引用。当外部对 key 的强引用被断开，key 只剩下 WeakHashMap 内部的弱引用。下次 GC 时，key 对象被回收，对应的 Entry 会被放入 ReferenceQueue。WeakHashMap 在每次 get/put/size 等操作时检查 ReferenceQueue，把 key 已经被回收的 Entry 从 Map 中移除，实现自动清理。

Q3

"ThreadLocal 为什么用弱引用做 key？有什么内存泄漏风险？"

ThreadLocalMap 的 Entry 中，key 是弱引用（指向 ThreadLocal 对象），value 是强引用（指向用户存储的对象）。

用弱引用做 key 的目的是：当 ThreadLocal 的强引用被置为 null（比如 tl = null），下次 GC 时 key（ThreadLocal 对象）能被回收，避免 ThreadLocal 对象本身泄漏。

但问题是：value 仍然是强引用。如果线程是线程池里的线程（长期存活），而 key 被 GC 回收后变成了 null，这个 Entry 就变成了 (null → User对象) 的状态——value 永远无法被访问到，但又不会被回收，造成内存泄漏。

解决方案：使用完 ThreadLocal 后，必须调用 tl.remove() 手动清理 value。或者使用 try-finally 块确保 remove 被执行。

Q4

"虚引用有什么用？get() 为什么永远返回 null？"

虚引用的设计目的就是跟踪对象被回收的事件，而不是获取对象。如果 get() 能返回对象，那用户拿到对象后又可能建立强引用，导致对象无法被回收，违背了虚引用的设计初衷。

典型应用是 NIO 的 DirectByteBuffer。DirectByteBuffer 对象在 JVM 堆里，但它指向的内存是堆外内存（不受 JVM GC 管理）。当 DirectByteBuffer 被 GC 回收时，JVM 通过虚引用机制感知到，触发 Cleaner 去调用 unsafe.freeMemory() 释放堆外内存。如果没有虚引用，堆外内存就会泄漏。
