核心机制概览

反射调用的性能问题

Java 反射 Method.invoke() 看似方便，但底层存在显著性能开销。主要来源于两个方面：

  • 1 JNI 边界开销：Java 代码调用 Native 代码（C/C++）时，需要在 JVM 和 Native 之间切换上下文，涉及参数转换、栈帧切换等成本。
  • 2 运行时检查：反射调用需要运行时校验访问权限（Accessible）、参数类型匹配、方法查找等，这些都是直接调用没有的额外工作。

JVM 的优化策略

JDK 采用了一种"懒加载 + 热点优化"的策略来解决反射性能问题：

阶段一 前 15 次调用

通过 NativeMethodAccessorImpl 走 JNI 调用，保证启动速度和正确性。

阶段二 超过 15 次后

JVM 动态生成 GeneratedMethodAccessor 字节码，消除 JNI 边界，性能接近直接调用。

NativeMethodAccessorImpl

JNI 调用 · 首次/低频调用

工作原理

NativeMethodAccessorImpl 是反射调用的默认实现。当通过 Method.invoke() 调用方法时，首次会走到这个类，它通过 JNI（Java Native Interface） 调用 JVM 底层的 Native 代码来执行目标方法。

为什么先用 JNI？因为 JNI 实现简单、稳定，不需要动态生成字节码。对于只调用几次的反射方法，JNI 的开销可以接受，避免了生成字节码的编译成本。

JNI 开销具体在哪？

  • •参数封箱/拆箱：基本类型需要包装成 Object（如 int → Integer）
  • •数组拷贝：参数数组需要从 Java 堆拷贝到 Native 栈
  • •上下文切换：JVM 执行状态 ↔ Native 执行状态的切换
  • •安全检查：每次都要检查访问权限
💡 面试话术

"反射首次调用走 NativeMethodAccessorImpl，通过 JNI 调到底层 C++ 代码。JNI 有边界开销：参数要封箱、数组要拷贝、还要做权限检查。但对于低频调用，这个开销可以接受，JVM 不会一上来就生成字节码。" 

// sun.reflect.NativeMethodAccessorImpl 简化源码
class NativeMethodAccessorImpl extends MethodAccessorImpl {
    private final Method method;
    private int numInvocations = 0;

    NativeMethodAccessorImpl(Method m) {
        this.method = m;
    }

    public Object invoke(Object obj, Object[] args)
        throws IllegalArgumentException, InvocationTargetException {

        // 每次调用计数 +1
        if (++numInvocations > 15) {
            // 超过阈值，生成字节码版本
            MethodAccessor acc = generateMethodAccessor();
            // 替换当前的 accessor
            setMethodAccessor(acc);
            return acc.invoke(obj, args);
        }

        // 走 JNI 调用
        return invoke0(method, obj, args);
    }

    private static native Object invoke0(Method m, Object obj, Object[] args);
}

GeneratedMethodAccessor

动态字节码 · 消除 JNI 开销

工作原理

当同一个反射方法的调用次数超过阈值（默认 15 次）后，JVM 会触发 动态字节码生成机制。通过 sun.reflect.ClassFileAssembler 在内存中直接拼装出一个新的类，这个类实现了 MethodAccessor 接口，内部直接调用目标方法，完全绕过 JNI。

这个生成的类没有 JNI 调用，没有参数封箱，没有权限检查（在生成时已验证过），本质上就是一段"直接调用目标方法"的字节码，性能几乎和直接调用一样。

💡 面试话术

"当调用超过 15 次，JVM 会用 ClassFileAssembler 动态生成一个 GeneratedMethodAccessor 类。这个类直接调用目标方法，不走 JNI，没有参数封箱和权限检查，性能和直接调用几乎一样。类名通常是 sun.reflect.GeneratedMethodAccessorXXX，存在堆外内存的 Metaspace 里。" 

// JVM 动态生成的类（类似下面这样）
// 类名: sun.reflect.GeneratedMethodAccessor1
public class GeneratedMethodAccessor1
    extends MethodAccessorImpl {

    // 生成的 invoke 方法：直接调用目标方法
    public Object invoke(Object obj, Object[] args) {
        // 直接强转为目标类，无 JNI 过渡
        UserService target = (UserService) obj;

        // 直接解包参数（无需封箱）
        String arg0 = (String) args[0];

        // 直接调用！没有 JNI 边界开销
        target.sayHello(arg0);

        return null;
    }
}

阈值机制：为什么是 15 次？

sun.reflect.inflationThreshold

阈值参数

JVM 通过 -Dsun.reflect.inflationThreshold=15 参数控制反射调用的"膨胀阈值"。默认值为 15，表示同一个反射方法被调用 15 次后，会从 NativeAccessor 切换到 GeneratedAccessor。

为什么是 15？这是一个经验值，平衡了两个因素：

  • •不能太小：如果阈值是 1，那几乎每次反射调用都要生成字节码，编译成本反而更高
  • •不能太大：如果阈值是 1000，那高频反射场景要忍受 1000 次 JNI 开销

15 次是一个"大多数低频调用不会触发，高频调用很快受益"的折中值。你可以通过启动参数调整： 

// 调整阈值为 0：第一次调用就生成字节码（不推荐，启动慢）
java -Dsun.reflect.inflationThreshold=0 MyApp

// 调整阈值为 100：更慢切换，适合极低频调用场景
java -Dsun.reflect.inflationThreshold=100 MyApp

// 关闭 inflation（始终走 JNI）
java -Dsun.reflect.noInflation=true MyApp

实际压测代码

// 反射调用性能测试
public class ReflectBenchmark {
    public static void main(String[] args) throws Exception {
        Method method = TargetClass.class.getMethod("sayHello", String.class);
        TargetClass target = new TargetClass();

        // 热身：触发 inflation
        for (int i = 0; i < 20; i++) {
            method.invoke(target, "test");
        }

        // 正式测试
        long start = System.nanoTime();
        for (int i = 0; i < 1000000; i++) {
            method.invoke(target, "test");
        }
        long reflectTime = System.nanoTime() - start;

        // 直接调用对比
        start = System.nanoTime();
        for (int i = 0; i < 1000000; i++) {
            target.sayHello("test");
        }
        long directTime = System.nanoTime() - start;

        System.out.println("反射: " + reflectTime / 1000000 + " ns/op");
        System.out.println("直接: " + directTime / 1000000 + " ns/op");
        // 结果：热身后反射性能接近直接调用
    }
}

面试高频追问

Q1

"反射为什么慢？JVM 是怎么优化反射性能的？"

反射慢主要有两个原因：一是 JNI 边界开销，Java 调用 Native 代码需要上下文切换、参数封箱、数组拷贝；二是运行时检查，每次都要校验访问权限、参数类型匹配。

JVM 的优化策略是"Inflation"机制：前 15 次调用走 NativeMethodAccessorImpl（JNI），超过 15 次后动态生成 GeneratedMethodAccessor 字节码类，直接调用目标方法，完全消除 JNI 开销。热身后的反射性能接近直接调用。

Q2

"GeneratedMethodAccessor 是怎么生成的？存在哪里？"

当调用次数超过阈值（默认 15），MethodAccessorGenerator 会使用 ClassFileAssembler 在内存中直接拼装字节码，生成一个继承 MethodAccessorImpl 的类。

生成的类通过 Unsafe.defineClass() 直接加载到 Metaspace（方法区），不需要磁盘 .class 文件。类名格式为 sun.reflect.GeneratedMethodAccessorXXX（XXX 是递增数字）。

由于这些类是动态生成的，如果反射调用非常频繁且方法很多，Metaspace 可能会被撑大，极端情况下会抛出 OutOfMemoryError: Metaspace。

Q3

"反射调用的阈值可以调整吗？什么场景需要调整？"

可以通过 JVM 参数 -Dsun.reflect.inflationThreshold=N 调整阈值。默认是 15。

调大阈值（如 100）：适合反射调用极少的场景，避免不必要的字节码生成，节省 Metaspace。

调小阈值（如 0）：适合反射调用非常频繁的场景，第一次调用就生成字节码，但启动时会有编译开销。

关闭 inflation：-Dsun.reflect.noInflation=true 让所有反射始终走 JNI，适合对动态生成类有安全限制的环境（但性能差）。

Q4

"Method.setAccessible(true) 对性能有什么影响？"

setAccessible(true) 的作用是跳过 Java 语言访问检查（如 private 方法的访问控制）。但它本身不会直接提升调用性能。

真正的性能提升来自Inflation 机制生成的 GeneratedAccessor。生成的字节码在拼装时已经确认了访问权限，所以后续调用不需要再做权限检查。

如果不调 setAccessible(true) 就去调用 private 方法，会抛出 IllegalAccessException。所以 setAccessible(true) 是"能不能调用"的问题，不是"快不快"的问题——但配合 Inflation 后，确实能达到接近直接调用的性能。
