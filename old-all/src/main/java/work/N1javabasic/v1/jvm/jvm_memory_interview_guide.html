程序计数器

Program Counter Register · 线程私有

THREAD PRIVATE
Q

"程序计数器具体是什么？能举个实际例子说明吗？"

你可以这样回答：程序计数器是一块很小的内存空间，可以看作是当前线程所执行的字节码的行号指示器。它记录着线程下一步要执行哪一条指令。如果执行的是 Java 方法，计数器记录的是正在执行的虚拟机字节码指令的地址；如果执行的是 Native 方法，这个计数器值则为空（Undefined）。

实际例子：想象你在看一本很厚的书（字节码），每页都有页码。程序计数器就是夹在书里的书签，记录你现在看到第几页。CPU 时间片轮转到这个线程时，JVM 看一眼书签，就知道该从哪一行字节码继续执行。线程切换出去时，把书签放好；切换回来时，按书签位置继续读。

💡 面试话术（口语化版）

"程序计数器就像多线程环境下的书签。每个线程都有自己的书签，记录自己执行到字节码的哪一行。线程切换时靠它恢复上下文，所以它是线程私有的。也因为它只是存一个地址指针，空间非常小，所以不会发生 OutOfMemoryError，这是它最特别的地方。"

虚拟机栈

JVM Stack · 线程私有

THREAD PRIVATE
Q

"虚拟机栈具体存什么？能举个实际例子吗？StackOverflowError 怎么产生的？"

你可以这样回答：虚拟机栈描述的是 Java 方法执行的线程内存模型。每个方法被执行的时候，JVM 都会同步创建一个栈帧（Stack Frame）压入栈顶。方法执行完毕，栈帧出栈。栈帧里存储了局部变量表、操作数栈、动态链接、方法返回地址等信息。

实际例子：你写了一个 main() 方法调用 a()，a() 调用 b()。JVM 会依次压入三个栈帧：main 帧 → a 帧 → b 帧。b 执行完后弹出，回到 a；a 执行完弹出，回到 main。这就像叠盘子，最后放的先拿（LIFO）。

💡 面试话术

"虚拟机栈是方法执行的舞台。每个方法对应一个栈帧，里面存局部变量、操作数栈等。方法调用就是压栈，方法结束就是弹栈。如果方法 A 调 A 自己无限循环，栈帧会一直压入直到超过栈容量，就会抛出 StackOverflowError。如果栈空间申请不到内存，还会抛出 OutOfMemoryError。"

本地方法栈

Native Method Stack · 线程私有

THREAD PRIVATE
Q

"本地方法栈具体是什么？能举个实际例子吗？和虚拟机栈有什么区别？"

你可以这样回答：本地方法栈与虚拟机栈作用非常相似，区别只是服务对象不同。虚拟机栈为 Java 方法（字节码）服务，而本地方法栈为 Native 方法服务。Native 方法是用 C/C++ 等语言编写、通过 JNI（Java Native Interface）调用的方法。

实际例子：Java 的 Thread.start() 底层最终调用了 start0() 这个 Native 方法；System.currentTimeMillis()、FileInputStream.read() 等很多底层 IO 和系统交互，最终都会走到 Native 方法。这些 Native 方法执行时，就在本地方法栈中创建栈帧。

// ... 前置检查 ...

// 调用 native 方法，进入本地方法栈

// 实际调用链:
// Java 层: thread.start() → start0()
// ↓ JNI 过渡
// C++ 层: JVM_StartThread() → 创建操作系统线程
// 这段 C++ 代码的执行上下文就保存在本地方法栈中
虚拟机栈
  • •服务对象：Java 方法（字节码）
  • •栈帧内容：局部变量表、操作数栈等
  • •异常：StackOverflowError / OOM
本地方法栈
  • •服务对象：Native 方法（C/C++）
  • •栈帧内容：Native 方法的局部状态
  • •异常：StackOverflowError / OOM
💡 面试话术

"本地方法栈和虚拟机栈是孪生兄弟，一个服务 Java 方法，一个服务 Native 方法。比如 Thread.start() 最终调用 start0() 这个 Native 方法，这时线程的执行上下文就从虚拟机栈转移到了本地方法栈。HotSpot VM 中这两块栈在实现上是合二为一的，但规范层面仍是独立概念。"

堆

Heap · 线程共享

THREAD SHARED
Q

"堆具体存什么？能举个实际例子吗？为什么需要分代？"

你可以这样回答：堆是 JVM 管理的最大一块内存，被所有线程共享。它的唯一目的就是存放对象实例和数组。几乎所有的对象实例都在这里分配内存。堆是垃圾收集器（GC）管理的主要区域，因此也被称为 "GC 堆"。

实际例子：当你写下 new User() 时，这个 User 对象就诞生在堆中。对象的引用（比如 User u）可能放在栈的局部变量表里，但真正的对象数据——字段值、对象头信息——都在堆里。多个线程可以同时持有指向同一个堆中对象的引用。

🔄 为什么分代？实际 GC 场景

你可以这样回答：分代的依据是"弱代假说"——绝大多数对象都是朝生夕灭的（比如方法内的临时对象），而熬过多次 GC 的对象很可能会长期存活。把堆划分为年轻代（Young）和老年代（Old），可以针对不同生命周期的对象采用不同的回收策略，提高 GC 效率。

Survivor

Old

💡 面试话术

"堆是 JVM 里最大的一块共享内存，专门放 new 出来的对象和数组。比如 new User()，引用在栈里，对象本体在堆里。因为对象生命周期不同，堆被分成了年轻代和老年代——新对象先在 Eden 区，经历 Minor GC 存活下来就移到 Survivor，足够老就晋升老年代。如果对象一直创建且不被回收，最终会抛出 OutOfMemoryError: Java heap space。"

方法区（元空间/永久代）

Method Area / Metaspace · 线程共享

THREAD SHARED
Q

"方法区具体存什么？能举个实际例子吗？永久代和元空间有什么区别？"

你可以这样回答：方法区存储的是类级别的数据，也就是"类本身的信息"，而不是对象实例。它存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。方法区在逻辑上是堆的一部分，但规范上把它描述为"堆的一个逻辑部分"，然而为了区分，通常又叫"非堆"。

实际例子：当你写一个类 public class UserService，里面定义了字段、方法、静态常量 static final String VERSION = "1.0"。类加载器加载这个类后，类的全限定名、父类名、实现的接口列表、字段描述、方法描述、静态常量池等信息都会存入方法区。所有线程执行这个类的方法时，都要来方法区读取这些元数据。

📊 永久代 vs 元空间：面试必考点
对比项 永久代 PermGen（JDK 7-） 元空间 Metaspace（JDK 8+）
存储位置 JVM 堆内存中 本地内存（Native Memory）
默认上限 固定且较小（通常几十 MB） 无上限（受限于物理内存）
OOM 场景 加载类过多很容易 OOM 类加载过多仍可能 OOM，但概率降低
参数 -XX:PermSize / MaxPermSize -XX:MetaspaceSize / MaxMetaspaceSize

💡 面试话术

"方法区存的是类的元数据，不是对象。比如类名、字段列表、方法字节码、静态变量、常量池都在这。所有线程共享同一份。JDK 8 之前叫永久代，存在 JVM 堆里，很容易因为动态生成类太多而 OOM；JDK 8 改成元空间，挪到本地内存，默认不受 JVM 堆大小限制，但加载类太多还是会报 OutOfMemoryError: Metaspace。"
