可达性分析

深入理解为什么 JVM 选择可达性分析，而非引用计数。

核心概念

🌳

可达性分析算法

Reachability Analysis，也叫追踪式垃圾收集（Tracing GC）。

基本思路是：通过一系列称为 "GC Roots" 的根对象作为起始节点集，从这些节点开始，根据引用关系向下搜索，搜索过程走过的路径称为 "引用链"（Reference Chain）。

当一个对象到 GC Roots 没有任何引用链相连时（即从 GC Roots 到这个对象不可达），则证明此对象是不可能再被使用的，可以被回收。

🎯

判定对象死亡的两次标记

1
第一次标记：可达性分析

对象没有与 GC Roots 相连的引用链，被第一次标记。

2
第二次标记：finalize() 筛选

对象覆写了 finalize() 且未执行过，放入 F-Queue 队列，由 Finalizer 线程执行。若 finalize() 中重新建立引用，则移除标记。

3
最终回收

两次标记都通过的对象，被垃圾收集器回收。

引用计数 vs 可达性分析

🔢

引用计数法

Reference Counting

在对象中添加一个引用计数器，每当有一个地方引用它时，计数器值 +1；当引用失效时，计数器值 -1。计数器为 0 的对象就是不可能再被使用的。

实现复杂度 简单
回收及时性 即时
循环引用处理 无法解决 ❌
额外开销 较高（计数器更新）
代表语言
Python Objective-C Swift（ARC）
🌿

可达性分析

Reachability Analysis

通过一系列 GC Roots 作为起始节点，根据引用关系向下搜索，搜索走过的路径称为引用链。当一个对象到 GC Roots 没有任何引用链相连时，证明此对象不可达。

实现复杂度 较复杂
回收及时性 滞后（需 GC 周期）
循环引用处理 完美解决 ✅
额外开销 较低（周期性遍历）
代表语言 / 平台
Java C# Go V8（Chrome）
🔄

循环引用：引用计数的致命缺陷

引用计数法最大的问题是无法解决对象之间的循环引用。当两个对象互相引用，但没有任何外部 GC Root 指向它们时，它们的引用计数器都不为 0，导致永远无法被回收，造成内存泄漏。

而可达性分析从 GC Roots 出发，只要对象不在引用链上，无论对象之间如何互相引用，都会被判定为不可达并回收。

为什么 JVM 不用引用计数？

正是因为循环引用问题。Java 对象图结构复杂，循环引用非常常见（如双向链表、树结构的父子互相引用）。可达性分析能优雅地解决这一问题。

面试高频追问

Q1

"GC Roots 是什么？能举几个实际的例子吗？"

标准回答：GC Roots 是一组必须活跃的对象引用，垃圾收集器从这些根节点出发进行可达性分析。常见的 GC Roots 包括：

  • ①栈帧局部变量：方法执行时局部变量表里的引用，比如 Object obj = new Object() 中的 obj
  • ②静态变量：类中 static 引用的对象，如单例模式中的实例
  • ③常量：static final 引用的对象，如字符串常量池中的引用
  • ④JNI 引用：Native 方法中持有的 Java 对象引用
  • ⑤活跃线程：Thread 对象本身
  • ⑥同步锁对象：被 synchronized 持有的对象

记忆口诀："栈静常本锁" —— 栈帧局部变量、静态变量、常量、本地方法（JNI）、线程本身、同步锁对象。

Q2

"可达性分析和引用计数有什么区别？为什么 JVM 选择可达性分析？"

核心区别：引用计数为每个对象维护一个计数器，记录被引用的次数，为 0 时回收；可达性分析从 GC Roots 出发遍历引用链，不在链上的对象回收。

JVM 选择可达性分析的根本原因：引用计数法有一个致命的缺陷——无法解决循环引用。比如对象 A 引用 B，B 引用 A，但外部没有任何 GC Root 指向它们。引用计数法下 A 和 B 的计数器都是 1，永远不会被回收，造成内存泄漏。

而可达性分析不关心对象之间的引用关系，只关心对象是否能从 GC Roots 到达。A 和 B 虽然互相引用，但没有 GC Root 指向它们，所以都会被判定为不可达，被正确回收。

Q3

"对象不可达了就一定会被回收吗？finalize() 方法了解吗？"

不一定。对象被判定为不可达后，还要经历一次 finalize() 的筛选过程。

具体流程：对象没有引用链 → 第一次标记 → 检查是否覆写了 finalize() 且未执行过 → 若有，放入 F-Queue 队列 → Finalizer 线程执行 finalize() → 若 finalize() 中重新建立了与 GC Roots 的引用链，则移除标记，对象"复活"；否则第二次标记 → 最终被回收。
