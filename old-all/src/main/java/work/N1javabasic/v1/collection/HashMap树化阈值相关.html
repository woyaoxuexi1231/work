HashMap 树化阈值 8 · 退化阈值 6 · 缓冲差值 2

为什么是 8、6、2？

这三个数字不是拍脑袋定的。8 来自泊松分布的概率极限， 6 是为了避免频繁转换的性能抖动， 2 是两者之间的滞后缓冲区（Hysteresis）。

1

为什么树化阈值是 8？—— 泊松分布说"几乎不可能"

HashMap 的设计假设是：在良好的 hashCode 下，桶中元素数量服从参数 λ≈0.5 的泊松分布。 JDK 源码注释中直接给出了这个分布的概率表：

0: 0.60653066 ← 60% 的桶是空的
1: 0.30326533 ← 30% 的桶只有 1 个
2: 0.07581633
3: 0.01263606
4: 0.00157952
5: 0.00015795
6: 0.00001316
7: 0.00000094
8: 0.00000006 ← 千万分之六！

当链表长度达到 8 时，在正常哈希分布下发生的概率仅为 0.000006%。 这意味着：如果触发了树化，几乎一定是 hashCode 出了问题（如恶意攻击或极差的哈希函数）。 树化是一种防御性兜底机制，而非正常使用路径。

💡

为什么不选 7 或 9？
7 的概率是 8 的 15 倍（0.00000094 vs 0.00000006），还不够"极端"； 9 虽然更安全，但会让 O(n) 退化的窗口多暴露一步。 8 是概率足够小与防护足够早的最佳平衡点。

为什么退化阈值是 6 而不是 8？—— 防止"乒乓效应"

如果树化和退化的阈值都是 8，考虑以下灾难场景：

❌ 假设阈值同为 8 时的"乒乓震荡"

size=8
→ 树化！链表变红黑树
remove→7
← 退化！红黑树变回链表
put→8
→ 又树化！重新构建红黑树
remove→7
← 又退化！再次拆解为链表
∞ 循环
💥 每次 put/remove 都在做 O(n) 的结构转换！

转换代价不对称

链表→树：需要遍历链表 + 构建红黑树，O(n)。
树→链表：需要中序遍历 + 重建链表，也是 O(n)。
两者都不便宜。如果在临界点反复触发，性能会比单纯的 O(n) 查找还差。

✅ 设为 6 的效果

树化在 8 触发，退化在 6 触发。
即使 size 在 7~8 之间波动，也不会触发任何转换。
只有当元素真正持续减少到 6 以下，才确认"不再需要树结构"。

3

差值 2 的作用 —— 滞后缓冲（Hysteresis）

数轴上的状态机：滞后区间 [6, 8]

链表
← 缓冲区 →
红黑树
0 ≤6 退化 7 ≥8 树化 16+
← remove 方向
put 方向 →

在缓冲区 [7] 内，保持当前状态不变。状态切换取决于你从哪个方向进入。

🔧

工程术语：Hysteresis

"滞后"是控制系统中的经典概念。空调不会在 26°C 立刻停机、25.9°C 立刻启动， 而是设置 26°C 停 / 24°C 启。HashMap 的 8/6 就是同一个思想。

📐

为什么恰好是 2？

1 太小：size=7 时一次 put+remove 就会触发双向转换。
3+ 太大：退化门槛过低，树结构在 size=5 时才拆除，浪费内存（TreeNode 是 Node 的 2 倍大小）。
2 是最小的安全间距。

🧮

额外约束：MIN_TREEIFY_CAPACITY

即使链表 ≥8，如果总容量 < 64，HashMap 会选择扩容而非树化。 因为小容量下的长链更可能是容量不足导致的，扩容比建树更高效。

4

完整决策流程

putVal() 中的树化判断逻辑（简化）
// 插入新节点后，检查是否需要树化
if (binCount >= TREEIFY_THRESHOLD - 1) {  // -1 是因为计数从0开始

    // 🔒 前置条件：容量必须 ≥ 64
    if (tab.length < MIN_TREEIFY_CAPACITY) {
        resize();          // 容量不够 → 先扩容，不树化
        break;
    }

    // ✅ 容量够 + 链表够长 → 树化
    treeifyBin(tab, hash);
}

// ────────────────────────────────────────
// removeNode() 中，检查是否需要退化
if (root != null && root.size <= UNTREEIFY_THRESHOLD) {
    untreeify(map);   // ≤ 6 → 拆树为链表
}
常量 值 含义 设计依据
TREEIFY_THRESHOLD 8 链表 → 红黑树 泊松分布 P(X≥8) ≈ 6×10⁻⁸，极端异常才触发
UNTREEIFY_THRESHOLD 6 红黑树 → 链表 低于树化阈值 2，提供滞后缓冲防震荡
Buffer Gap 2 8 − 6 = 滞后区间 最小安全间距，避免 size 在临界点波动时反复转换
MIN_TREEIFY_CAPACITY 64 允许树化的最小容量 小容量优先扩容；64 = 8 × 8，与负载因子 0.75 配合

⚖️ 一句话总结

8 不是一个"性能最优"的数字，而是一个"统计学上几乎不可能发生"的安全阈值。 它的存在不是为了优化正常场景，而是为了在 hashCode 失效时，将最坏情况从 O(n) 兜底到 O(log n)。

6 不是一个独立的数字，它是 8 的伴生值。 它的唯一使命是与 8 之间制造一个宽度为 2 的滞后带， 确保数据结构不会因为元素数量的微小波动而在两种形态之间反复横跳。

2 是这个滞后带的最小安全宽度。 它保证了至少需要连续 2 次反向操作（如 put→remove）才能穿越缓冲区触发状态切换， 从而将"乒乓震荡"的可能性降到工程可接受的范围。

💡 这三个数字共同体现了 Java 集合框架的设计哲学：正常路径极致轻量，异常路径有兜底，状态转换有阻尼。
