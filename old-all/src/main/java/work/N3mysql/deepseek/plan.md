## 第1天：MySQL 连接与执行流程探源 – 覆盖原理点：[P1]

### 编码探源题

#### 题目1：编写 JDBC 程序，观察连接创建与线程状态
**要求**：通过 JDBC 连接 MySQL，分别在连接建立前、后、关闭后执行 `SHOW STATUS LIKE 'Threads_connected'` 和 `SHOW PROCESSLIST`，打印变化日志。证明连接池复用线程的行为。

**🔍 原理反思提问**：MySQL 的连接管理使用了哪几种线程？连接认证失败会占用线程资源吗？`wait_timeout` 和 `interactive_timeout` 对连接的影响是什么？
**💬 面试官可能追问**：如果客户端连接池设置最大 200，MySQL 端 max_connections 是 150，会出什么问题？请设计一个脚本触达此边界。

#### 题目2：跟踪一条简单 SQL 在 MySQL 内部的执行流程
**要求**：开启 `performance_schema` 或使用 `SHOW PROFILE` (在 5.7 测试)，执行 `SELECT * FROM t WHERE id=1`，将分析结果输出为 JSON 格式，识别出“连接”“解析”“优化”“执行”各阶段耗时。对于 8.0 使用 `EXPLAIN ANALYZE`。

**🔍 原理反思提问**：优化器在这一阶段做了什么？它将 SQL 改写成了多少种等价计划？请使用 optimizer trace 复现。
**💬 面试官可能追问**：如果同样 SQL 执行 10 万次，哪一阶段会变成瓶颈？为什么 8.0 移除了查询缓存？给出你的实验结论。

#### 题目3：编写脚本验证 MySQL 用户认证过程与权限缓存
**要求**：创建用户并授权，在另一个会话中连续执行查询，同时用管理员会话 `FLUSH PRIVILEGES`，观察权限变更的生效时机。用代码打印每次查询的结果和错误日志。

**🔍 原理反思提问**：MySQL 的权限缓存存储在哪里？为什么要 FLUSH PRIVILEGES？如果跳过这一步可能会出现什么问题？
**💬 面试官可能追问**：修改了最大连接数的全局变量，新连接生效吗？现有连接会受影响吗？请用实验证明。

#### 题目4：模拟连接协议交互（高级/源码级）
**要求**：不借助 JDBC 驱动，用 Socket 编程模拟 MySQL 客户端握手认证过程，至少完成解码服务端 greeting 包、发送登录请求、接收 OK/ERR 包，并打印交互的十六进制数据。指定 MySQL 版本 8.0。阅读源码 `sql/sql_connect.cc` 中的 `check_connection` 函数。

**🔍 原理反思提问**：8.0 的 `caching_sha2_password` 与 `mysql_native_password` 在网络包上有何差异？你的代码如何处理 public key 请求？
**💬 面试官可能追问**：如果中间人攻击修改了认证包，MySQL 如何防御？请结合 SSL/TLS 设置回答。

#### 题目5：连接打满事故复现与线程池行为（地狱级）
**要求**：编写多线程程序，在短连接模式下疯狂建立/关闭连接，不做 wait，观察 MySQL 出现 `Too many connections` 错误的时间点。调整 `max_connections`、`back_log` 参数，对比不同设置下的吞吐曲线（使用脚本统计成功/失败数）。对连接耗时做 p99 统计。

**🔍 原理反思提问**：连接队列溢出的内核参数 `somaxconn` 在此处扮演什么角色？thread_cache_size 如何影响短连接性能？请用系统级监控数据说明。
**💬 面试官可能追问**：如果线上突然出现 Too many connections，除了调大 max_connections 外，还有哪些抢救手段？你会优先 kill 掉哪些连接？

### 核心实现题：从零实现一个数据库连接池
**要求**：
- 支持最大连接数、最大空闲连接数、连接超时、空闲回收。
- 提供接口：`getConnection()`、`releaseConnection()`、健康检查（`SELECT 1` 心跳）。
- 暴露运行时监控数据：活跃数、空闲数、等待队列长度。
- 单元测试：模拟并发获取连接，观察等待队列长度和超时抛出的异常。
  **🔍 原理反思提问**：连接池的“公平”模式和“非公平”模式在锁竞争上有什么不同？如何设计无锁队列以降低延迟？
  **💬 面试官可能追问**：如果数据库发生了主从切换，连接池怎么感知并重建所有连接？你设计的连接池如何集成到 TDDL 这类中间件中？

## 第2天：InnoDB 内存结构：缓冲池与 Change Buffer – 覆盖原理点：[P2, P3]

### 编码探源题

#### 题目1：监控缓冲池命中率与脏页比例
**要求**：编写脚本插入大量随机数据，同时在循环中采集 `SHOW ENGINE INNODB STATUS` 中的缓冲池命中率和脏页比例，以秒级日志输出到文件。绘制命中率随时间变化的曲线（用 Excel 或简单 Web 页面）。

**🔍 原理反思提问**：缓冲池命中率突然下降通常是什么原因？LRU 链表中的 young 区和 old 区如何划分？参数 `innodb_old_blocks_pct` 的作用？
**💬 面试官可能追问**：全表扫描会污染缓冲池吗？如何通过设置参数防止 buffer pool 污染？请用实验证明设置前后的命中率差异。

#### 题目2：验证 Change Buffer 对二级索引写入的加速
**要求**：建一张有多个二级索引的表，分别开启和关闭 `innodb_change_buffering`，批量插入 100 万行，对比耗时和磁盘写入次数。使用 `SHOW ENGINE INNODB STATUS` 观察 Change Buffer 的使用量。

**🔍 原理反思提问**：Change Buffer 合并时机是什么？如果此时数据库突然宕机，Change Buffer 中的数据会丢失吗？它是持久化的吗？
**💬 面试官可能追问**：为什么唯一索引的插入不能使用 Change Buffer？在你的代码里如何体现这一限制？

#### 题目3：模拟数据页与索引页的存储结构（源码级）
**要求**：通过 `innodb_ruby` 或自定义 C/Java 程序解析一个 ibd 文件，读出 Page 0（FSP_HDR）的信息，包括页类型、段、区信息。打印出 B+树 root 页和叶子页的偏移量。阅读源码 `storage/innobase/include/page0page.h`。

**🔍 原理反思提问**：InnoDB 页大小默认 16KB，你能通过解析 page header 验证吗？行格式 COMPACT 和 DYNAMIC 有什么本质区别？
**💬 面试官可能追问**：如果一行数据超长（溢出页），数据是如何存储的？请从你解析的 ibd 文件中找到溢出页的证据。

#### 题目4：自适应哈希索引（AHI）行为探针
**要求**：构建一个等值查询频繁的表，观察 `SHOW ENGINE INNODB STATUS` 中自适应哈希索引的命中次数和占用空间。通过改变查询模式（范围查询）使 AHI 失效，观察性能下降，并写脚本对比 QPS。

**🔍 原理反思提问**：AHI 在什么条件下会触发构建？为什么高并发下 AHI 可能成为瓶颈（锁竞争）？你的实验中有没有出现这种情况？
**💬 面试官可能追问**：如果 AHI 出现 hash 冲突严重，你会怎么优化？直接关闭 AHI 是否是合理方案？请结合 SSB 测试给出结论。

#### 题目5：地狱级：自定义 Buffer Pool 页面淘汰算法
**要求**：编写一个简化的 InnoDB Buffer Pool 模拟器（Java/Python），实现 LRU 链表和脏页列表，模拟磁盘读写延迟。实现不同的淘汰策略（LRU、CLOCK、LRU-K），对比命中率。模拟 MySQL 的“midpoint insertion”机制。

**🔍 原理反思提问**：MySQL 的 LRU 如何防止扫描污染？为什么设置 midpoint 为 5/8？如果你的模拟器将 midpoint 设为 1/4，性能会怎样？
**💬 面试官可能追问**：假设 Buffer Pool 中有很多脏页，需要刷盘，你怎么调度才能最大化吞吐？请引入 doublewrite buffer 的磁盘代价。

### 核心实现题：实现一个简版 LRU 缓存管理器
**要求**：
- 实现固定容量的 LRU 缓存，支持 `get`、`put` 操作，O(1) 复杂度。
- 扩展为分段 LRU（类似 young/old），实现 midpoint 插入策略。
- 加入“脏页”标记，统计脏页数量，并提供 `flushDirtyPages(max)` 方法按顺序刷脏。
- 单元测试验证不同访问模式下的命中率。

**🔍 原理反思提问**：如何保证并发访问下 LRU 链表的一致性？MySQL 使用了哪些锁来保护 Buffer Pool？
**💬 面试官可能追问**：如果缓存容量极大，锁竞争上升，你会如何拆分 Buffer Pool 实例？MySQL 的 `innodb_buffer_pool_instances` 是怎么工作的？

## 第4天：联合索引、最左匹配与索引下推 – 覆盖原理点：[P5]

### 编码探源题

#### 题目1：编写实验证明最左前缀匹配规则
**要求**：建表 `t(id, a, b, c, primary(id), key(a,b,c))`，用 Explain 分别执行 `WHERE a=1`、`WHERE a=1 AND b=2`、`WHERE b=2`、`WHERE a=1 AND c=3`、`WHERE b=2 AND c=3`，记录 key_len 和 Extra，使用日志输出是否用了索引。

**🔍 原理反思提问**：`WHERE a=1 AND c=3` 的 key_len 是多少？为什么 b 部分没有用上但仍然可以用索引？索引条件下推发生了什么？
