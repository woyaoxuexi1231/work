## 每日一句话概览
- 第1天：在 Spring Boot 中集成 Redis，使用 `StringRedisTemplate` 完成第一次 set/get。
- 第2天：利用 String 类型实现文章阅读计数、全局唯一 ID 生成，并设置过期时间。
- 第3天：使用 Hash 缓存用户信息，实现单字段更新与全量存储的灵活操作。
- 第4天：通过 List 实现消息队列生产消费，体验左进右出与范围分页查询。
- 第5天：使用 Set 存储文章标签，完成交并差集计算（共同好友、可能认识的人）。
- 第6天：使用 ZSet 构建积分排行榜，利用 score 实现按时间排序的延迟任务。
- 第7天：设置 key 过期时间，观察自动删除行为，配合 KeyExpirationEventMessageListener 监听过期事件。
- 第8天：使用 MULTI/EXEC 实现转账事务，配合 WATCH 实现乐观锁防止并发修改。
- 第9天：通过 `executePipelined` 批量发送命令，对比逐条与管道执行的耗时差异。
- 第10天：使用 `convertAndSend` 与 MessageListener 实现服务间解耦通知。
- 第11天：配置 `Jackson2JsonRedisSerializer`，让对象存取自动完成 JSON 转换。
- 第12天：调整 Lettuce 连接池参数，开启连接监控，观察等待与空闲连接状态。
- 第13天：体验 Bitmaps 签到、HyperLogLog 统计 UV、GEO 计算附近的人。
- 第14天：使用 `@Cacheable`、`@CacheEvict` 声明式管理缓存，避免手动编码。
- 第15天：综合实现用户模块：缓存用户详情、用 ZSet 做积分榜、用 Set 存标签。
- 第16天：剖析 Redis 单线程事件循环，压测不同并发下的 QPS 并验证 IO 多路复用优势。
- 第17天：探究 String 的 SDS 动态字符串和 Hash 的 dict、ziplist 编码，对比内存占用。
- 第18天：分析 List 的 quicklist 和 Set 的 intset、dict 编码，压测不同元素数量下的性能。
- 第19天：深入 ZSet 的 skiplist 与 ziplist 实现，压测排行榜更新的耗时与内存分配。
- 第20天：执行 bgsave 触发 RDB 持久化，分析 dump.rdb 文件结构与恢复流程。
- 第21天：配置 AOF 三种 fsync 策略，对比其吞吐量与安全级别，执行 AOF 重写。
- 第22天：开启混合持久化，模拟断电场景，验证 RDB+AOF 恢复数据的完整性与速度。
- 第23天：通过源码或实验验证惰性删除与定期删除的执行时机，观察内存占用变化。
- 第24天：设置不同内存淘汰策略，填充数据触发淘汰，对比 allkeys-lru 与 volatile-ttl 的行为。
- 第25天：搭建一主多从，观察全量同步与部分重同步过程，验证 slave-read-only 配置。
- 第26天：部署 Sentinel，模拟主节点故障，验证自动故障转移与 Spring Boot 连接哨兵。
- 第27天：搭建 Redis Cluster，观察 CRC16 哈希槽分配、MOVED/ASK 重定向过程。
- 第28天：模拟缓存穿透（布隆过滤器）、击穿（互斥锁）、雪崩（随机过期），压测各自方案。
- 第29天：用 Redisson 实现分布式锁，解读看门狗续期机制与可重入加锁源码。
- 第30天：用 Lua 脚本实现原子性抢购，对比 Pipeline 的批量执行性能与原子性差异。
- 第31天：设计本地 Caffeine+Redis+MySQL 三级缓存，解决热 key 并保障最终一致性。
- 第32天：实现延时双删策略与基于 binlog 的异步更新方案，解决并发下的缓存脏数据。
- 第33天：设计百万级点赞计数器，利用 Redis 缓存点赞数并异步批量写入 MySQL。
- 第34天：基于 ZSet+Lua 实现滑动窗口限流器，支持动态规则调整与失败降级处理。
- 第35天：对比 Redis ZSet 轮询与 Redisson DelayedQueue 的延时任务方案，给出适用场景。
- 第36天：设计红包拆分算法+Redis Hash 预分配+Lua 抢包原子操作，保证金额一致性。
- 第37天：为社交 Feed 设计推拉结合的时间线缓存，利用 List/ZSet 缓存用户动态。
- 第38天：设计 Cluster 在线扩容方案，模拟 slot 迁移过程中的客户端重定向与数据一致性。
- 第39天：使用 slowlog、bigkeys、hotkeys 诊断性能瓶颈，制定内存爆增的应急扩容流程。
- 第40天：设计秒杀库存预热、Redis Lua 原子扣减、排队异步下单的完整高并发架构。
- 第41天：准备面试：从线程模型、数据结构、内存、IO 角度阐述 Redis 高性能的原因。
- 第42天：准备面试：全面梳理缓存一致性策略、内存淘汰算法选择与集群横向扩展方案。
- 第43天：准备面试：解释 Redlock 算法流程、Redisson 看门狗实现，以及红锁争议点。
- 第44天：准备面试：给出 RDB vs AOF 的选择建议，说明不同业务场景下的持久化策略。
- 第45天：毕业设计：完成商品详情、库存、用户会话的高并发缓存架构，整合上述核心原理。
