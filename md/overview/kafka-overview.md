## 每日一句话概览
- 第1天：在 Spring Boot 中配置 Kafka，使用 KafkaTemplate 发送第一条消息并消费。
- 第2天：体验同步发送等待结果、异步发送回调以及发后即忘的单向发送。
- 第3天：使用 @KafkaListener 监听主题，获取消息内容、分区、偏移量等元数据。
- 第4天：配置 JSON 序列化器，尝试 Avro 与 Schema Registry 的基础集成。
- 第5天：通过消息 key 控制分区，并实现自定义 Partitioner 按业务字段路由。
- 第6天：启动多个消费者实例，观察消费者组内消息分配与再均衡现象。
- 第7天：关闭自动提交，掌握手动同步/异步提交偏移量的时机与 AckMode 配置。
- 第8天：编写生产者与消费者拦截器，实现消息发送前后的日志记录与监控埋点。
- 第9天：通过 KafkaAdminClient 在代码中动态创建主题、增加分区、查看集群信息。
- 第10天：使用 SeekToCurrentErrorHandler 与 DeadLetterPublishingRecoverer 实现重试与死信。
- 第11天：设置批量消费与并发监听，对比单条与批量、单线程与多线程的消费表现。
- 第12天：在生产者中设置自定义 Header，在消费者端通过 @Header 注解获取。
- 第13天：使用 Kafka Streams 构建简单的 word count 流处理，输出到新主题。
- 第14天：基于 EmbeddedKafka 编写单元测试，验证消息的发送与接收逻辑。
- 第15天：搭建订单服务、库存服务、通知服务，通过 Kafka 实现异步解耦调用。
- 第16天：分析 acks=0/1/all 时生产者的发送流程，压测不同确认级别下的吞吐和延迟。
- 第17天：跟踪消费者加入组、SyncGroup、心跳维持的过程，理解再均衡触发条件。
- 第18天：切换分区分配策略，对比 Range、RoundRobin、Sticky 和 Cooperative 的分配结果。
- 第19天：查看 Kafka 数据目录，分析日志段的 .log、.index、.timeindex 文件结构。
- 第20天：配置基于时间和大小的日志清理策略，验证压缩主题的去重效果。
- 第21天：搭建多副本集群，监控 ISR 列表变化，验证落后副本被踢出的条件。
- 第22天：通过修改日志使 Leader Epoch 递增，验证高水位与 Leader Epoch 对一致性的保护。
- 第23天：追踪 Zookeeper 或 KRaft 中的控制器选举，理解 Controller 在分区 Leader 选举中的职责。
- 第24天：开启幂等性和事务，压测事务消息的吞吐损耗，理解幂等生产者如何避免重复。
- 第25天：对比 Gzip、Snappy、LZ4、Zstd 压缩算法对消息吞吐与压缩率的影响。
- 第26天：调整 fetch.min.bytes 和 fetch.max.wait.ms，观察消费者拉取延迟的变化。
- 第27天：探究 Kafka 的零拷贝与页缓存机制，通过系统调用追踪 sendfile 的实现。
- 第28天：配置副本拉取限流与客户端 Quota，验证限流对消费和同步速率的控制效果。
- 第29天：分析 Kafka Streams 的状态存储与 changelog 恢复机制，测试故障后的数据恢复。
- 第30天：全维度压测：结合 acks、压缩、批量大小、linger.ms 等参数找到最佳吞吐配置。
- 第31天：设计应用日志→Kafka→Logstash/ES 的异步缓冲链路，设定分区数与保留策略。
- 第32天：用 Kafka 缓存秒杀请求，消费者使用并发控制和拉取节流实现削峰填谷。
- 第33天：利用事务消息实现交易和账务系统的最终一致性，解决转账与记账的状态同步。
- 第34天：通过 Debezium 捕获 MySQL binlog 发送至 Kafka，实现多系统数据实时同步。
- 第35天：基于 Kafka Streams 计算实时点击数、转化率，并通过 WebSocket 推送大屏展示。
- 第36天：模拟消息积压，演练增加消费者、转储中间主题、降级非关键处理的应急方案。
- 第37天：设计 MirrorMaker 2.0 方案完成跨数据中心的多活同步，处理偏移量映射与冲突。
- 第38天：通过 JMX + Prometheus + Grafana 监控消费者 Lag、I/O 线程、ISR 变化并告警。
- 第39天：设计基于消息 Key 的幂等消费方案，利用 Redis 或数据库去重表实现精准去重。
- 第40天：从存储、性能、事务、生态角度对比 Kafka、RabbitMQ 与 RocketMQ，明确选型边界。
- 第41天：准备面试回答：从 acks、副本同步、高水位、消费者提交角度阐述消息零丢失方案。
- 第42天：准备面试回答：分析重复消费场景，阐述幂等性与事务如何实现精确一次语义。
- 第43天：准备面试回答：从顺序 I/O、页缓存、零拷贝、批量处理解释 Kafka 高吞吐的原因。
- 第44天：准备面试回答：讲解幂等生产者、事务协调器如何协作实现 EOS。
- 第45天：毕业设计：设计包含下单、库存扣减、风控检查、物流通知的完整电商消息架构。
