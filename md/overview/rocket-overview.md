## 每日一句话概览
- 第1天：在 Spring Boot 中集成 RocketMQ starter，发送并消费第一条普通消息。
- 第2天：体验同步发送等待确认、异步发送回调、单向发送不关注结果的三种模式。
- 第3天：配置集群消费和广播消费，观察相同消费者组下消息的分发行为。
- 第4天：发送顺序消息，在消费者端按队列顺序处理消息，理解局部顺序性。
- 第5天：发送延时/定时消息，掌握 different delay level 和精确时间的区别。
- 第6天：批量发送多条消息，使用 Tag 进行简单的消费端过滤。
- 第7天：实现事务消息半消息的发送，编写本地事务执行器与回查接口。
- 第8天：人为导致消费失败，观察自动重试以及消息进入死信队列的过程。
- 第9天：使用 RocketMQTemplate 设置消息的 Keys、Tags 和自定义 Header，为追踪做准备。
- 第10天：用 Spring Cloud Stream 抽象绑定 RocketMQ，减少样板代码。
- 第11天：在消费者端配置 SQL92 表达式过滤消息属性，体验服务器端过滤。
- 第12天：开启 ACL，配置 AccessKey/SecretKey，实现生产者和消费者权限隔离。
- 第13天：开启消息轨迹功能，在 RocketMQ Console 中查看消息从生产到消费的完整链路。
- 第14天：通过 Console 或命令行重置消费进度，观察消费位点回溯与重复消费现象。
- 第15天：设计下单通知流程：订单服务发送→通知服务消费，整合多种消息类型。
- 第16天：剖析 NameServer 的无状态路由表维护机制，探究 Broker 心跳与路由注册。
- 第17天：分析 CommitLog 文件结构，验证 ConsumeQueue 索引的生成逻辑与定位方式。
- 第18天：修改 broker.conf 的刷盘配置，压测同步刷盘与异步刷盘的吞吐与延迟差异。
- 第19天：搭建一主一从，测试同步复制下 master 宕机后 slave 的数据完整性和切换过程。
- 第20天：验证生产者发送失败时的故障规避机制，理解延迟与重试队列的关系。
- 第21天：切换不同的 AllocateMessageQueueStrategy，查看队列分配结果与均衡性。
- 第22天：追踪消费者 Offset 的存储位置与提交时机，模拟手工提交与自动提交的差异。
- 第23天：分析消息重试的 topic %RETRY% 以及死信 topic %DLQ% 的消息流，掌握延时级别表。
- 第24天：深入顺序消息的 Broker 端锁定机制，验证同一队列串行消费的实现方式。
- 第25天：分析延时消息的内部定时任务与时间轮调度，理解默认延迟级别的局限性。
- 第26天：跟踪半消息写入、Op 消息写入以及回查触发流程，绘制事务消息状态图。
- 第27天：对比 Tag 过滤（在 Broker 内存过滤）与 SQL 过滤的性能开销，分析其本质。
- 第28天：部署 DLedger 模式的 RocketMQ 集群，观察基于 Raft 的 commitlog 复制和选主过程。
- 第29天：调整消费者 pull 阈值，通过压测验证不同流控参数对消费速率和延迟的影响。
- 第30天：全维度压测，对比同步/异步刷盘、同步/异步复制、批量大小对 TPS 和延迟的影响。
- 第31天：设计下单后异步扣库存、发短信、发积分的最终一致性消息方案。
- 第32天：用 RocketMQ 缓存秒杀请求，后台限流消费，削峰填充并结合消费端并发控制。
- 第33天：实现支付系统回调后的事务消息，保证付款状态与订单状态最终一致。
- 第34天：用 Canal+RocketMQ 实现 MySQL binlog 到其他数据源的异步同步，解决数据延迟问题。
- 第35天：为证券交易系统设计基于 RocketMQ 的订单簿顺序消费，保证单产品严格有序。
- 第36天：模拟大促消息积压，演练应急扩容消费者、临时转存消息、降级非核心处理的方案。
- 第37天：设计跨 Region 的 RocketMQ Connector 同步方案，实现异地多活的消息复制。
- 第38天：集成 Prometheus+Grafana 监控消费 TPS、延迟、死信量，设置报警规则。
- 第39天：设计基于消息 Key 的幂等方案，利用 Redis/DB 去重表防止重复消费。
- 第40天：从存储、性能、事务、生态角度对比 RocketMQ、Kafka 与 RabbitMQ，明确选型边界。
- 第41天：准备面试回答：从生产端、Broker 端、消费端三环节阐述如何保证消息零丢失。
- 第42天：准备面试回答：详述事务消息的回查机制，以及回查失败后的补偿策略。
- 第43天：准备面试回答：顺序消息的实现原理与使用限制，如何避免热点队列。
- 第44天：准备面试回答：CommitLog 为何采用顺序写？与 Kafka 的分区存储相比有何优势？
- 第45天：毕业设计：设计包含下单、库存、支付、物流通知的完整消息驱动电商架构。
