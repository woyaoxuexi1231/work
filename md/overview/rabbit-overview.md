## 每日一句话概览
- 第1天：在 Spring Boot 中连接 RabbitMQ，发送并接收第一条消息，熟悉基础配置。
- 第2天：使用 DirectExchange 实现点对点路由，理解 routing key 的精确匹配。
- 第3天：使用 FanoutExchange 实现广播式消息，忽略 routing key。
- 第4天：使用 TopicExchange 实现模式匹配路由，掌握通配符 `*` 和 `#`。
- 第5天：了解 HeadersExchange 并通过 header 匹配发送消息，对比四种交换机应用场景。
- 第6天：通过 RabbitTemplate 发送带自定义 headers、priority 等属性的消息对象。
- 第7天：在消费者端使用 @RabbitListener 注解的不同参数形式，获取消息体、头信息及 Channel。
- 第8天：体验 autoAck 与手动 ack/nack 的行为差异，不深入原理，只观察效果。
- 第9天：配置 Jackson2JsonMessageConverter，实现 Java 对象与 JSON 的自动序列化传输。
- 第10天：设置单条消息的过期时间和队列默认 TTL，观察消息自动删除行为。
- 第11天：配置死信交换机和死信路由键，让被拒绝或过期的消息进入死信队列。
- 第12天：安装延迟消息插件，通过 x-delayed-type 实现基本的延迟发送。
- 第13天：修改 concurrency 和 prefetch 参数，观察多线程消费行为。
- 第14天：使用 RabbitAdmin 在代码中动态创建交换机、队列和绑定关系。
- 第15天：综合搭建订单通知解耦小系统，完成多队列多交换机的异步场景练习。
- 第16天：开启 Publisher Confirm 和 Return 回调，通过压测对比事务与确认模式的性能。
- 第17天：深度探究 manual ack 与 nack 的 requeue 策略，配合消息重入队实验。
- 第18天：对比持久化/非持久化交换机和队列的写入性能与恢复行为。
- 第19天：验证消息 TTL 与队列 TTL 同时存在时的优先级规则与删除时间点。
- 第20天：完整实现死信产生→死信交换→死信队列的链路，分析死信原因 header。
- 第21天：用死信队列模拟延迟消息 vs 延迟插件，进行吞吐量和精确度压测对比。
- 第22天：通过不同 prefetch 值压测，找出最佳并发消费配置并解释 QoS 原理。
- 第23天：开启消息优先级排序，测试不同优先级消息的出队顺序与性能开销。
- 第24天：模拟消息积压触发流控报警，理解连接/信道/队列三级流控机制。
- 第25天：基于数据库本地消息表 + 双确认，实现高可靠投递并验证故障恢复。
- 第26天：分别用数据库唯一约束和 Redis setnx 实现消费端幂等，对比方案优劣。
- 第27天：测试单队列顺序发送与接收的保证，同时设计分区顺序消息方案并验证局限性。
- 第28天：配置指数退避重试拦截器，将最终失败消息转入死信并监控。
- 第29天：搭建测试集群，创建镜像队列和仲裁队列，对比数据同步机制差异。
- 第30天：全维度压测：对比不同确认模式、持久化、prefetch 组合下的性能与可靠性平衡。
- 第31天：设计订单 T+1 超时自动取消方案，利用死信队列结合数据库状态回查。
- 第32天：设计前端请求→队列缓冲→后端平滑消费的秒杀异步削峰架构，给出限流参数。
- 第33天：用 RabbitMQ 实现交易与账务系统的最终一致性，解决脏读和幂等问题。
- 第34天：设计服务日志→队列→Logstash→ES 的异步缓冲链路，设定背压保护。
- 第35天：模拟消息积压场景，制定扩容消费者、转储死信、跳过非关键消息等应急方案。
- 第36天：实现微博动态时间线顺序方案，保证单个用户动态严格有序，写出分区键设计。
- 第37天：基于仲裁队列设计高可用 RabbitMQ 集群，分析节点分布和磁盘/网络故障应对。
- 第38天：设计跨机房消息同步方案，借助 Federation/Shovel 插件实现异地多活。
- 第39天：在支付回调场景中集成幂等+发送方确认+消费者手动确认，形成端到端不丢不重。
- 第40天：选型 Prometheus + Grafana 监控队列长度、确认速率、流控状态，设置告警规则。
- 第41天：准备面试回答：从生产者→ Broker →消费者全链路阐述消息不丢失的机制。
- 第42天：准备面试回答：出现重复消费时的完整解决路径（幂等键、Redis、DB 去重表）。
- 第43天：准备面试回答：从延迟队列需求出发，逐步推演延迟插件/死信/TTL 方案的设计思路。
- 第44天：准备面试回答：对比 RabbitMQ、Kafka、RocketMQ 的特性，给出 RabbitMQ 最佳场景。
- 第45天：毕业设计：整合订单、库存、通知三大服务，基于 RabbitMQ 完成异步可靠消息架构。
