Java Generics Core Principle

PECS 原则

Producer Extends, Consumer Super

这是 Java 泛型通配符设计的核心指导原则。它不是建议，而是类型安全的数学必然。 理解它，就理解了为什么 ? extends T 不能写入， 为什么 ? super T 读取只能是 Object。

ExtendsDemo.java 
// 一个"生产者"方法：从列表中读取数据
public static double sum(
        List<? extends Number> source) {

    double total = 0;
    for (Number n : source) {  // ✅ 安全读取为 Number
        total += n.doubleValue();
    }
    return total;
}

// 调用方可以传入任何 Number 子类的列表
List<Integer> ints = List.of(1, 2, 3);
List<Double> dbls = List.of(1.1, 2.2);

sum(ints);  // ✅ List<Integer> 匹配 ? extends Number
sum(dbls);  // ✅ List<Double> 匹配 ? extends Number

// ═══ 为什么不能写入？ ═══
void broken(List<? extends Number> list) {
    list.add(new Integer(1)); // ❌ 编译错误!
    // 如果 list 实际是 List<Double>，
    // 你就往 Double 列表里塞了 Integer!
    // 编译器无法验证安全性，直接拒绝。

    list.add(null); // ✅ null 是唯一例外
}
📖

PECS 之 Producer Extends

当你只需要从集合中读取数据（它是数据的生产者），使用 ? extends T。 你可以安全地以 T 类型读取，但不能写入任何非 null 值， 因为编译器无法确定实际的子类型是什么。

SuperDemo.java 
// 一个"消费者"方法：向列表中写入数据
public static void addNumbers(
        List<? super Integer> dest) {

    dest.add(1);      // ✅ Integer 安全写入
    dest.add(2);      // ✅ Integer 安全写入
    dest.add(3);      // ✅ Integer 安全写入
}

// 调用方可以传入 Integer 或其任何父类型的列表
List<Integer> ints = new ArrayList<>();
List<Number> nums = new ArrayList<>();
List<Object> objs = new ArrayList<>();

addNumbers(ints); // ✅ List<Integer> 匹配 ? super Integer
addNumbers(nums); // ✅ List<Number> 匹配 ? super Integer
addNumbers(objs); // ✅ List<Object> 匹配 ? super Integer

// ═══ 为什么读取只能得到 Object？ ═══
void read(List<? super Integer> list) {
    Object o = list.get(0);  // ✅ 只能是 Object

    Integer i = list.get(0); // ❌ 编译错误!
    // 如果 list 实际是 List<Object>，
    // get(0) 可能返回 String，强转 Integer 会炸!

    Number n = list.get(0);  // ❌ 也不行!
    // List<Object> 里的元素不一定是 Number
}

PECS 之 Consumer Super

当你只需要向集合中写入数据（它是数据的消费者），使用 ? super T。 你可以安全地写入 T 及其子类型，但读取时只能当作 Object， 因为编译器无法确定实际的父类型是什么。

底层原理：为什么必须这样？

这不是设计偏好，而是类型安全的数学约束

? 为什么 ? extends T 不能写入？

假设允许写入，考虑以下场景：

List<Double> doubles = new ArrayList<>();
List<? extends Number> list = doubles;
list.add(new Integer(1)); // 💥
Double d = doubles.get(0); // ClassCastException!

? extends Number 是一个 存在类型（Existential Type）： "存在某个具体的子类型 X，使得 list 是 List<X>"。 编译器知道 X 存在，但不知道 X 是什么。 既然不知道 X，就无法验证任何写入是否与 X 兼容。

形式化表达：∀ value ≠ null, ¬∃ proof(value : ?)
对于任何非 null 值，都不存在它是"未知类型"实例的证明。

? 为什么 ? super T 读取只能是 Object？

假设允许以 T 类型读取，考虑以下场景：

List<Object> objs = new ArrayList<>();
objs.add("hello");
List<? super Integer> list = objs;
Integer i = list.get(0); // 💥 String→Integer!

? super Integer 的含义是 "存在某个父类型 Y ⊇ Integer，使得 list 是 List<Y>"。 Y 可能是 Integer、Number、 Object。 这些类型的公共交集只有 Object。

形式化表达：⋂{Y | Y ⊇ T} = Object
所有 T 的超类型的交集 = Object（Java 类型系统的顶元素）

🎓 类型理论术语

协变（Covariance）：? extends T
子类型关系保持方向：若 A ≤ B，则 List<? extends A> ≤ List<? extends B>。
适合只读场景（输出位置）
逆变（Contravariance）：? super T
子类型关系反转方向：若 A ≤ B，则 List<? super B> ≤ List<? super A>。
适合只写场景（输入位置）

经典案例：Collections.copy()

JDK 源码中 PECS 最完美的体现

java.util.Collections (JDK Source) 
public static <T> void copy(
        List<? super T> dest,    // ← Consumer: 写入目标
        List<? extends T> src   // ← Producer: 读取来源
) {
    for (int i = 0; i < src.size(); i++) {
        dest.set(i, src.get(i)); // 从 src 读 T，写入 dest
    }
}

// ═══ 实际使用 ═══
List<Integer> src = List.of(1, 2, 3);

List<Number> dest1 = new ArrayList<>();
Collections.copy(dest1, src); // ✅ Integer → Number

List<Object> dest2 = new ArrayList<>();
Collections.copy(dest2, src); // ✅ Integer → Object

List<Double> dest3 = new ArrayList<>();
// Collections.copy(dest3, src); // ❌ Double 不是 Integer 的超类

src: List<? extends T>

生产者。copy 需要从 src 中 get() 元素。 用 ? extends T 保证读出的元素至少是 T 类型， 同时允许传入 List<Integer> 等更具体的列表。

dest: List<? super T>

消费者。copy 需要向 dest 中 set() 元素。 用 ? super T 保证 dest 能容纳 T 类型， 同时允许传入 List<Number> 等更宽泛的列表。

速查卡片

打印贴在工位上

通配符 角色 读取 写入 记忆口诀
? extends T Producer ✅ 得到 T ❌ 仅 null "向上看，读安全；不知具体类型，写危险"
? super T Consumer ⚠️ 仅 Object ✅ T 及子类 "向下看，写安全；不知上限类型，读受限"
<T> (精确) Both ✅ 得到 T ✅ 写入 T "既读又写时用精确类型参数"

🧭 决策流程

1

这个方法/字段是否需要同时读写同一个泛型集合？
→ 是：使用精确类型参数 <T>，不用通配符。

2

只需要从中读取（它是数据源/生产者）？
→ 用 ? extends T。最大化调用方的传入灵活性。

3

只需要向其中写入（它是数据汇/消费者）？
→ 用 ? super T。最大化调用方的传入灵活性。

4

既不读也不写，只做结构性操作（如 size()、clear()）？
→ 用 ?（无界通配符），最大灵活性。

💡 Bloch's Rule（Effective Java Item 31）："Don't use wildcard types as return types." 通配符应该出现在方法参数中以增加灵活性，而不应出现在返回值中——那只会给调用方制造麻烦。
