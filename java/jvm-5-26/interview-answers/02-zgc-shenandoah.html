GC 回收机制 第二问

ZGC / Shenandoah：为什么放弃分代？

分代收集统治了 JVM 二十多年，但 ZGC 和 Shenandoah 说"不"。这不是任性——这是一场 GC 理念的根本性转变：从"怎么让停顿变短"到"怎么让停顿与堆大小无关"。读屏障、颜色指针、并发重定位——这些技术才是现代 GC 的核心。

1 分代不是免费的——代价清单

分代 GC 的隐性成本

代价 1

卡表维护开销：每次引用字段赋值（putfield, aastore），JIT 都需要插入写屏障代码来标记卡表脏卡。在大对象图频繁修改的场景（如缓存更新），这个开销不可忽视——ZGC 和 Shenandoah 仍需写屏障但用于 SATB 或并发标记，而非卡表维护。

代价 2

记忆集（Remembered Set）膨胀：G1 为每个 Region 维护一个记忆集来记录"谁引用了我"。在 Region 数量很大（几百个）且跨 Region 引用频繁时，记忆集会占用大量内存（可能达到堆的 10-20%）。

代价 3

Young GC 必须 STW：即使是 G1 的 Young GC 也必须 Stop-The-World。在堆很大的时候（几百 GB），哪怕只回收新生代，Eden + Survivor 扫描也需要几十毫秒的停顿——对于延迟敏感系统（如高频交易、实时推荐）这是不可接受的。

代价 4

晋升阈值调优的负担：MaxTenuringThreshold、TargetSurvivorRatio 等参数需要在每个系统上调优。上一问提到的 Survivor 溢出问题就是分代带来的额外复杂度。

💡 核心理念转变：分代 GC 的前提是"停顿时间与存活对象数量成正比"——所以需要把对象按年龄分开，只回收新生代来缩短停顿。 ZGC 和 Shenandoah 的答案是：让停顿时间与存活对象数量无关，只与 GC Roots 数量有关。 这是通过"并发标记 + 并发重定位"来实现的——连最繁重的对象移动都在并发阶段完成。

2 颜色指针（Colored Pointers）：ZGC 的核心魔法

ZGC 最颠覆性的设计是把 GC 状态信息编码进对象的64 位指针中，而不是存在对象头里。在 x86-64 架构上，虚拟地址实际只用了 48 位——剩下的 16 位就是 ZGC 的"画布"。 

ZGC 64 位指针布局（Linux/x86_64）：
┌───────┬───┬───┬──────────────────────────────────────┐
│ 未使用 │颜色│保留│          44 位地址空间               │
│ 16位  │4位│4位 │       (16TB 虚拟地址范围)            │
└───────┴───┴───┴──────────────────────────────────────┘

颜色位的 4 种状态（2^4 可表示 16 种，ZGC 用了其中几种）：
┌──────────┬─────────────────────────────────────────┐
│  Remapped │  指针指向对象的当前地址（已重定位完成）   │
│  Marked0  │  标记阶段：对象已被标记为存活             │
│  Marked1  │  下一次 GC 的标记位（双缓冲，避免复位）   │
│  Finalizable│  对象有 finalize 方法待处理            │
└──────────┴─────────────────────────────────────────┘

关键洞察：同一对象可以被多个不同"颜色"的指针指向！
当 mutator 线程读到一个 Marked0 指针，读屏障会检测到"这个指针指向的
对象的地址可能已经变了（被重定位了）"，然后触发修正操作。

颜色指针如何解决碎片化？

碎片化的本质是对象移动后，所有指向它的指针都要更新。传统 GC（G1）在 STW 阶段扫描引用关系来更新指针——这很慢。 ZGC 的做法是：延迟指针修正。对象被移动到一个新的连续区域后，原来的指针只是被标记为"旧颜色"。 当 mutator 下一次通过这个旧指针访问对象时，读屏障（Load Barrier）会检测到颜色不匹配，自动将指针修正为新地址。

💡 这就是"自愈"：不需要 STW 去全局更新引用。引用被"懒更新"——只有在真正被使用时才修正。对于短期不再使用的指针，根本不修正。这极大减少了 GC 停顿中指针更新的工作量。

3 读屏障（Load Barrier）：JIT 注入的隐形守卫

读屏障 vs 写屏障：为什么读屏障更关键？

G1 和 CMS 主要依赖写屏障（SATB 或卡表标记），因为分代 GC 需要追踪"谁修改了引用"。但写屏障只能追踪引用变更，不能解决"对象移动后引用过期"的问题。

ZGC 选择读屏障——每次从堆中加载引用时，检查指针的颜色。如果颜色表示对象已被移动，读屏障会透明地：

  1. 从 forwarding table 查到对象的新地址
  2. 更新当前持有的指针为新颜色 + 新地址
  3. 返回新地址（这个过程对 Java 代码完全透明）

⚠️ 读屏障的性能开销：Shenandoah 的读屏障是用一个显式的条件分支实现的（Brooks Pointer：每个对象头存一个 forwarding pointer），每次读引用都要检查。 ZGC 把状态编码进指针位，通过指针位的位运算来判断——在某些情况下比分支预测更稳定。 两种方案都通过 JIT 的屏障消除优化来降低开销：如果 JIT 能证明某个引用在当前 GC 周期中不会变化，就直接去掉了读屏障。

4 并发重定位：在业务线程眼皮底下"搬家" 

ZGC 并发重定位的四个阶段：
Phase 1: Concurrent Mark（并发标记）
  GC 线程并发扫描对象图，标记存活对象
  期间 mutator 线程正常运行

Phase 2: Concurrent Prepare for Relocate（并发准备重定位）
  选择要回收的 Region，计算重定位目标地址
  建立 forwarding table（旧地址 → 新地址）

Phase 3: Concurrent Relocate（并发重定位）★ 核心阶段
  GC 线程将对象从旧 Region 复制到新 Region
  mutator 线程读到旧地址的指针时，读屏障自动修正
  ★ 这里没有 STW！对象在被"搬运"的同时，业务代码可以继续读写

Phase 4: Concurrent Remap（并发重映射）
  修正所有还指向旧地址的指针
  整个阶段也是并发的

唯一的 STW 阶段：
  - Pause Mark Start（< 1ms）
  - Pause Mark End （< 1ms）
  - Pause Relocate Start（< 1ms）
  每个 STW 阶段只做 GC Roots 扫描，与堆大小无关！

一个具体场景：对象在并发重定位中被读写

// 时间线：对象 A 正在从 Region-1 拷贝到 Region-2
// 此时 mutator 线程执行：
MyObj ref = globalRef;          // ① 读取引用（触发读屏障）
int val = ref.getField();        // ② 访问字段
ref.setField(newValue);          // ③ 修改字段

// 读屏障的处理逻辑（伪代码）：
if (ptr.color != REMAPPED) {
    // 对象已被或正在被移动
    newPtr = forwardingTable.get(ptr.address);
    if (newPtr == null) {
        // 还没拷贝完，自己拷贝（Self-Healing）
        newPtr = copyObject(ptr.address);
    }
    ptr = newPtr.withColor(REMAPPED); // 修正指针颜色
    globalRef = ptr;                   // 回写纠正
}
return ptr;  // 返回正确的新地址

这就是 ZGC 的 "Self-Healing" 机制——mutator 不仅仅是"等待 GC 修好指针"，而是主动参与修正。 这使得单个 GC 线程拷贝速度跟不上也没关系，mutator 会帮忙修正。

5 G1 vs ZGC：生产选型决策

维度 G1 ZGC
典型停顿10-50ms（堆 < 32GB）< 1ms（任何堆大小）
堆大小支持4MB ~ 几十 GB8MB ~ 16TB
分代分代（Young + Mixed）不分代（JDK 21+ 支持分代 ZGC）
吞吐量较好（Young GC 是 STW 但很快）稍低（读屏障有 < 5% 开销）
内存开销Remembered Set 占 10-20%Forwarding Table 占 3-5%
JDK 版本JDK 7+ 可用，JDK 9+ 默认JDK 15+ 生产可用

💡 选型建议：如果堆 ≤ 32GB 且停顿几十毫秒可以接受 → G1 足够好，成熟稳定。 如果停顿必须 < 1ms（如交易系统、实时竞价），或者堆 > 100GB → ZGC。 JDK 21 开始 ZGC 也引入了分代支持（Generational ZGC），结合了分代的吞吐优势和 ZGC 的低停顿——这是未来的方向。
