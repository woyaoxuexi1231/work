GC 回收机制 第一问

GC 算法与分代收集的深层原理

标记-清除、标记-整理、复制算法不只是教科书上的几句描述。它们的本质差异根植于硬件特性——CPU 缓存行、TLB 命中率、内存分配器的 bump-the-pointer 优化。分代收集不是简单的"对象寿命不同"，而是一套完整的概率模型。

📐 标记-清除 📐 标记-整理 📐 复制算法 🧬 分代假设 ⚡ bump-the-pointer 🔗 跨代引用

1 三大 GC 算法的本质差异

维度 标记-清除 标记-整理 复制
核心操作 标记存活对象 → 清除未标记 标记存活 → 向一端滑动 → 更新引用 标记存活 → 拷贝到新空间 → 整体回收旧空间
碎片问题 严重 — 不连续空闲链表 无 — 对象紧密排列 无 — 新空间连续
空间开销 低 — 仅标记位 低 — 仅标记位 + 滑动成本 高 — 需要 2× 空间
移动对象？ 否 是 — 大量移动 是 — 全部存活对象移动
适用场景 存活率极高的老年代 存活率较高的老年代 存活率极低的新生代

标记-清除：为什么"简单"也有价值

标记-清除是 GC 算法的鼻祖。它的核心思路：从 GC Roots 出发，递归遍历引用图，标记所有可达对象；然后线性遍历堆，回收未标记的块。 碎片是它最大的代价，但它有一个被严重低估的优势：不移动对象。

💡 资深视角：对象不移动意味着 指针地址不变。在 CMS 并发标记阶段，如果同时移动对象，所有正在运行的 mutator 线程持有的指针都会变成悬垂指针。 这就是为什么 CMS 在并发清除时选择"标记-清除"——它的老年代回收与业务线程并发执行，不能动对象地址。 当然，这也带来了碎片化，所以 CMS 在碎片严重时不得不回退到 Serial Old 做一次标记-整理。

复制算法：快的秘密不只是"复制"

复制算法的核心优势常被简单归因于"只需要扫描存活对象"。但这只说对了一半。 真正的杀手锏是分配器的极致优化：bump-the-pointer。 

// Bump-the-pointer 分配：只需一条指令
// 空闲指针 free_ptr，每次分配只需：
//   1. 读取 free_ptr 当前值
//   2. 比较 free_ptr + size <= end_of_space
//   3. CAS free_ptr = free_ptr + size
// 对比标记-清除的 free-list 分配：
//   遍历空闲链表 → 找到合适大小的块 → 分裂 → 更新链表
//   这涉及多次内存访问和指针操作

// 复制算法 + TLAB (Thread Local Allocation Buffer)
// 每个线程在 Eden 区预先分配一块私有的 TLAB
// 分配时只需 bump 线程本地指针——无锁竞争
// 这才是新生代分配效率远超老年代的根本原因

⚠️ CPU 缓存友好性：复制算法将存活对象紧凑地排列在新空间（Survivor），这意味着相关的对象在物理内存上也是相邻的。 后续访问这些对象时，它们很可能在同一缓存行（cache line，通常 64 字节）中，极大提升 L1/L2 缓存命中率。 标记-清除的碎片化会导致对象散落在堆各处，CPU 预取几乎失效。

标记-整理：昂贵但必要的"搬家"

标记-整理解决了碎片的根本问题。它把存活对象全部向内存一端"滑动"，然后回收边界之外的所有空间。 代价是所有存活对象的引用都要更新——每个被移动的对象，所有指向它的指针都需要重写。

在老年代，存活率高（通常 90%+），标记-清除几乎没什么可回收，碎片累积慢。但当碎片确实严重时（CMS 的 concurrent mode failure），需要进行一次 stop-the-world 的标记-整理——这就是 Serial Old 做的事。

💡 引用的更新成本：现代 JVM 不会真的逐个更新引用。Parallel Old 使用"分区并行整理"——将堆分成多个 region，每个线程负责一个 region 的存活对象滑动，然后用一个全局的 offset 表（forwarding pointer table）来统一调整引用。 G1 则进一步把堆分成了更小的 region，混合回收时只整理"垃圾最多"的几个 region——这就是"Garbage-First"名字的来源。

2 分代收集：不只是"对象存活率不同"

分代假设有两个子假设： 弱分代假设（Weak Generational Hypothesis）：绝大多数对象都是朝生夕死的； 强分代假设（Strong Generational Hypothesis）：熬过越多次 GC 的对象越不容易死亡。 这两个假设已经在大量生产系统中被验证——通常 98% 的对象在第一次 Young GC 时就死了。

新生代用复制算法：四个维度的完整解释

① 内存分配效率（最关键）

新生代分配频率极高——每秒 2000 个支付请求，每个产生 8KB 临时对象，每秒就要分配 ~16MB。 复制算法 + bump-the-pointer 让分配降到 O(1) 的指针加法。如果用空闲链表分配（标记-清除的分配器），分配一次就要扫描链表——在高频分配下性能差距是数量级的。

② 空间碎片（直接原因）

新生代对象大量死亡——98% 都活不过一次 GC。如果使用标记-清除，回收后的空间是大量不连续的小洞，后续分配要么找不到合适大小的连续块，要么碎片越来越严重。 复制算法把 2% 的幸存者紧凑复制到 Survivor，整个 Eden 一次性清空，始终保持 Eden 是一个完美的连续空间。

③ CPU 缓存友好性

复制到 Survivor 的对象是按遍历顺序排列的——这意味它们大概率按访问热度聚集在一起。 再加上 Survivor 区本身很小（通常 Eden 的 1/8），整个 Survivor 有可能被 L3 缓存容纳一部分。后续从老年代引用新生代对象时（跨代引用），这些对象在缓存中的命中率更高。

④ 跨代引用处理

跨代引用（老年代对象引用新生代对象）是分代 GC 的"阿克琉斯之踵"——如果每次 Young GC 都要扫描整个老年代来找跨代引用，分代就失去了意义。 JVM 使用卡表（Card Table）来解决：把老年代分成 512 字节的"卡片"，当老年代对象被修改（引用字段赋值），对应的卡片变"脏"。 Young GC 时只需扫描脏卡片指向的老年代对象，而不是整个老年代。 因为新生代用复制算法，对象会频繁移动地址，卡表也必须频繁更新——但如果新生代对象本身就少（仅 2% 的幸存者跨过 Survivor），卡表的维护开销是可控的。

老年代用标记-整理或标记-清除：取舍逻辑

老年代存活率极高（通常 > 90%）

如果老年代用复制算法，需要把 90% 的存活对象全部复制一遍——而回收的只有不到 10%。这个复制成本太高了。 标记-整理虽然也要移动对象（更新引用），但老年代 GC 频率低得多（可能几小时甚至几天一次），摊销下来是可以接受的。

CMS：先用标记-清除，碎片严重才整理

CMS 追求低停顿，并发清除阶段不能移动对象（否则并发访问的 mutator 指针悬垂）。 所以 CMS 默认用标记-清除，但预留了碎片整理的退路：-XX:+UseCMSCompactAtFullCollection。

Parallel Old：标记-整理最大化吞吐

如果追求吞吐量（批处理场景），Parallel Old 直接做标记-整理——虽然单次停顿长，但回收后内存完全规整，后续分配效率最高，长期吞吐量最大。

Young GC 的触发条件：Eden 区满时触发。存活对象从 Eden + 一个 Survivor 复制到另一个 Survivor。 每复制一次，对象头中的 GC 年龄 +1。年龄达到 MaxTenuringThreshold（默认 15）或 Survivor 空间不足时直接晋升老年代。
