volatile 内存屏障与 happens-before 验证

volatile 写 happens-before 后续 volatile 读——这条规则的实现靠的是 JVM 在 volatile 访问前后插入的四种内存屏障。StoreLoad 是最重的屏障，因为它要同时完成 Store Buffer 刷新和缓存行失效。用代码验证这个规则，关键是要暴露指令重排序。

1 四种内存屏障的排列顺序 

volatile 写 的内存屏障排列：
┌──────────────────────────────┐
│  普通写操作                   │
│  ...                         │
│  【StoreStore 屏障】          │  ← 禁止上面的普通写和下面的 volatile 写重排
│  volatile 写                 │
│  【StoreLoad 屏障】  ← 最关键  │  ← 禁止 volatile 写和后面的读/写重排
│  【？？？这里不能再有写】       │
│  后续读/写操作                │
└──────────────────────────────┘

volatile 读 的内存屏障排列：
┌──────────────────────────────┐
│  普通读/写操作                │
│  ...                         │
│  【LoadLoad 屏障】            │  ← 禁止上面的普通读和下面的 volatile 读重排
│  【LoadStore 屏障】           │  ← 禁止上面的普通写和下面的 volatile 读重排
│  volatile 读                 │
│  【LoadLoad 屏障】            │
│  【LoadStore 屏障】           │
│  后续读/写操作                │
└──────────────────────────────┘

四种屏障的功能：
  StoreStore  — 确保屏障前的写操作先于屏障后的写操作
  StoreLoad   — 确保屏障前的写操作先于屏障后的读操作（最重的屏障）
  LoadLoad    — 确保屏障前的读操作先于屏障后的读操作
  LoadStore   — 确保屏障前的读操作先于屏障后的写操作

🔥 为什么 StoreLoad 是最重的屏障？ 它要完成两件事：① 刷新 Store Buffer（让屏障前的写对所有 CPU 可见）； ② 使其他 CPU 的缓存行失效（让屏障后的读能读到最新值）。 在 x86 上，StoreLoad 屏障通过 lock 前缀或 mfence 指令实现， 这两个都会锁住总线或缓存，开销很大。其他三种屏障在 x86 上天然被硬件保证（x86 是强内存模型）， 所以 x86 上 volatile 写后的 StoreLoad 是唯一的实际屏障——这也是 volatile 比 synchronized 轻量但仍不够"免费"的原因。

2 用纯 Java 验证 volatile 禁止指令重排序 

/**
 * 暴露指令重排序的经典代码。
 * 如果没有 volatile 或内存屏障，x=0, y=0 的组合是可以出现的。
 * 要复现需要跑几十万次，用 -XX:-TieredCompilation 加速 JIT。
 */
public class ReorderDemo {
    static int x = 0, y = 0;
    static int a = 0, b = 0;
    
    // ★ 去掉 volatile 后，x=0, y=0 偶尔会出现
    // ★ 加上 volatile 后，x=0, y=0 永远不会出现
    // static volatile int x = 0, y = 0;  // 修复版本
    
    public static void main(String[] args) throws InterruptedException {
        int count = 0;
        while (true) {
            x = 0; y = 0; a = 0; b = 0;
            count++;
            
            Thread t1 = new Thread(() -> {
                a = 1;      // ① 普通写
                x = b;      // ② 读 b，写 x
                // 如果没有屏障，①和②可能被重排！
                // CPU 可能先执行 x=b (读到 b=0)，
                // 再执行 a=1
            });
            
            Thread t2 = new Thread(() -> {
                b = 1;      // ③ 普通写
                y = a;      // ④ 读 a，写 y
                // 同样，③和④可能被重排
            });
            
            t1.start(); t2.start();
            t1.join(); t2.join();
            
            // 如果 x=0 且 y=0，说明发生了重排序
            if (x == 0 && y == 0) {
                System.out.println("重排序发生！第 " + count + " 次迭代");
                System.out.println("x=" + x + ", y=" + y);
                break;
            }
        }
    }
}

// 理论上的结果：
// 如果没有重排序，可能的 (x, y) 组合：
//   (0, 1): t1 先执行完
//   (1, 0): t2 先执行完
//   (1, 1): 交替执行
// 
// 如果出现 (0, 0)：说明 t1 的 x=b 在 a=1 之前执行了
//                   且 t2 的 y=a 在 b=1 之前执行了
//   → 指令重排序被证实

// ★ 如何增大复现概率：
//   1. 加上 -XX:-TieredCompilation -Xcomp 强制编译
//   2. 关闭分层编译，让 JIT 更激进地优化
//   3. 多核 CPU 上跑（核心越多，Store Buffer 竞争越明显）

💡 为什么要跑很多次才能复现？现代 x86 CPU 是"强内存模型"（TSO: Total Store Order）， 它天然禁止了 LoadLoad、LoadStore、StoreStore 的重排序。但StoreLoad 重排序在 x86 上仍然存在—— CPU 的 Store Buffer 可能导致一个核心的写在另一个核心看来是"晚到的"。 这个 demo 就是利用 Store Buffer 的异步特性来暴露重排序。 如果把 x 和 y 都声明为 volatile，就插入了 StoreLoad 屏障，这个问题就消失了。

⭐ 面试要点

✅ 屏障排列（必须脱口而出）

  • ▸ volatile 写前：StoreStore
  • ▸ volatile 写后：StoreLoad（最重）
  • ▸ volatile 读前：LoadLoad + LoadStore
  • ▸ volatile 读后：LoadLoad + LoadStore
  • ▸ x86 TSO 下只有 StoreLoad 需要真正插入屏障

🔬 验证思路

  • ▸ 四变量 demo：a, b 普通变量；x, y 读取结果
  • ▸ x=0, y=0 出现 → 重排序被暴露
  • ▸ Store Buffer 是 StoreLoad 重排序的硬件根源
