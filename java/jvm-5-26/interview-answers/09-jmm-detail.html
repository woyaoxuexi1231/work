JMM 主内存/工作内存与8 个原子操作

"屏蔽硬件差异"太虚了。JMM 的主内存对应物理上的主存（RAM），工作内存对应 CPU 寄存器和 Cache Line。那 8 个原子操作——lock/unlock/read/load/use/assign/store/write——才是 JMM 的底层"机器码"。

1 物理映射：主内存 ≠ JVM 堆

JMM 概念物理对应是 JVM 的哪块区域？
主内存 (Main Memory) 物理主存（RAM / DRAM） 堆 (Heap) + 元空间（类元数据）
工作内存 (Working Memory) CPU 寄存器 + L1/L2/L3 缓存 线程栈 (Stack) 中的局部变量表
8 个原子操作 JVM 的抽象，映射到具体的 CPU 指令 由 JIT 编译器生成对应的机器码

工作内存的精确对应

CPU 寄存器

线程对变量的操作（use/assign）直接在寄存器中完成——这是最快的一级"工作内存"。 比如执行 a = a + 1，CPU 把 a 从 L1 缓存加载到寄存器 → 在寄存器中加 1 → 写回 L1 缓存。

CPU Cache

L1/L2 是每个核心私有的，L3 是多核心共享的。现代 x86 CPU 的缓存一致性协议（MESI/MESIF）保证了多核心看到的缓存数据最终一致——但"最终"意味着有延迟。 这就是 volatile 需要插入内存屏障的原因——强制刷新缓存行。

Store Buffer

CPU 内部还有一个 Store Buffer——写入不会立即到 L1，而是先到这个 buffer，异步刷新到缓存。 这是指令重排序的硬件来源之一。volatile 的 StoreLoad 屏障会强制刷新 Store Buffer。

8 个原子操作分为两组：

■ 主内存操作（线程间交互）：
  lock   — 标识变量为"线程独占"状态
  unlock — 释放独占状态
  read   — 从主内存读取变量的值，传输到工作内存
  write  — 把工作内存中的值写回主内存

■ 工作内存操作（线程内）：
  load   — 把 read 传输过来的值放入工作内存的变量副本
  use    — 把工作内存的变量值传给执行引擎（用于运算）
  assign — 把执行引擎的结果赋给工作内存的变量
  store  — 把工作内存的变量值传送到主内存（准备 write）

原子操作之间的规则限制

规则 1：read 和 load、store 和 write 必须成对出现

不允许单独 read 不 load，或单独 store 不 write。但不必连续——中间可以插入其他操作。

规则 2：use 和 store 之前必须有 load 和 assign

线程使用变量前必须先从主内存加载；线程将变量值存回主内存前必须先在工作内存中赋值。 这条规则保证了"变量值不会凭空出现"。

规则 3：不允许线程丢弃最近的 assign

线程在工作内存中修改了变量后，必须最终同步回主内存。但"最终"是多长时间？——普通变量没有时间限制，volatile 强制立即同步。

★ StoreLoad 屏障确保：
  Thread-1 的所有写入在 Thread-2 读取之前对所有 CPU 可见
  
★ 实现方式（x86）：
  StoreLoad 屏障 = lock addl $0,0(%rsp) 或 mfence 指令
  — 强制刷新 Store Buffer 到缓存，并让其他 CPU 的缓存行失效

⭐ 面试要点

✅ 核心概念

  • ▸ 主内存 = 物理 RAM + JVM 堆
  • ▸ 工作内存 = CPU 寄存器 + L1/L2 Cache + Store Buffer
  • ▸ read/load 成对，store/write 成对
  • ▸ volatile 写后必须 store+write，读前必须 read+load
  • ▸ StoreLoad 屏障在 x86 上映射为 lock 前缀或 mfence
