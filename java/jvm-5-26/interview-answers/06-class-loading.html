类加载三阶段与 <clinit> 细节

加载 → 链接（验证 → 准备 → 解析）→ 初始化。准备阶段对 static final 的处理经常被误解。<clinit> 的多线程安全不是靠 synchronized——而是 JVM 内部的一把锁，clinit 死锁是真实存在的线上事故。

1 类加载的三个阶段全景图 

┌──────────────────────────────────────────────────────────────┐
│                    类加载生命周期                              │
├─────────┬─────────┬───────────────────────┬──────────────────┤
│ 加载    │ 链接 (Linking)                   │ 初始化            │
│ Loading │ 验证 → 准备 → 解析               │ Initialization   │
├─────────┼─────────┼───────────┼───────────┼──────────────────┤
│ 从字节流│ 格式检查 │为静态变量  │符号引用   │ 执行 <clinit>   │
│ 生成    │ 语义检查 │分配内存并 │转换为     │ 静态变量赋初值   │
│ Class   │ 字节码   │赋零值     │直接引用   │ 静态代码块执行   │
│ 对象    │ 验证    │★重点     │(可延迟)  │                  │
└─────────┴─────────┴───────────┴───────────┴──────────────────┘

2 准备阶段：static 变量的零值陷阱

static int vs static final int

public class InitDemo {
    static int a = 10;            // 场景 1
    static final int B = 10;      // 场景 2
    static final Object C = new Object(); // 场景 3
}

// 准备阶段的值：
// a = 0        ← 只分配内存并赋零值，10 在初始化阶段才赋值
// B = 10       ← 关键区别！static final 且是编译时常量
//                 在准备阶段就已确定了值！
// C = null     ← static final 引用类型，即便是 final
//                 准备阶段也是 null，初始化阶段才 new

为什么 B = 10 在准备阶段就是 10？

因为 static final int B = 10 是编译时常量（Compile-Time Constant）。 javac 在编译时就将 B 的值写入常量池的 ConstantValue 属性。JVM 在准备阶段根据 ConstantValue 直接赋值，无需等到初始化。 这也是为什么 switch-case 中可以使用 static final int 的原因——编译时就能确定。

为什么 C = null（场景 3）即使是 final 也不行？

new Object() 不是编译时常量——对象的创建必须在运行时发生在堆上。 javac 无法在编译时确定其值，因此在准备阶段 C 只能是 null。 只有基本类型和 String 的字面量常量才可以在准备阶段确定值。

3 <clinit> 方法的生成与多线程安全

<clinit> 是怎么生成的？

// 源代码：
public class Example {
    static int x = 1;
    static {
        x = 2;
        System.out.println("static block");
    }
}

// javac 编译后生成的 <clinit> 方法（字节码伪代码）：
static {};  // 方法名就是 <clinit>
  descriptor: ()V
  flags: ACC_STATIC
  Code:
    0: iconst_1
    1: putstatic  #2  // x = 1  (静态变量赋值)
    4: iconst_2
    5: putstatic  #2  // x = 2  (静态代码块)
    8: getstatic  #3
   11: ldc        #4  // "static block"
   13: invokevirtual #5  // println
   16: return

// 关键：<clinit> 包含所有静态变量赋值和静态代码块，
// 按源代码中的顺序合并成一个方法。

如果没有静态变量赋值也没有静态代码块，javac 不会生成 <clinit> 方法——类加载更快。

<clinit> 的多线程安全：不是 synchronized

JVM 保证一个类的 <clinit> 在多线程环境下只被执行一次。 但不是通过简单的 synchronized——而是 JVM 内部在类加载时对 Class 对象加了一把初始化锁（Init Lock）。 

多线程同时触发类初始化的流程：

Thread-1                    Thread-2
  │                           │
  │ 尝试获取 Init Lock        │
  ├─ 获得锁 ✓                 │
  │                           │ 尝试获取 Init Lock
  │                           ├─ 锁被 T1 持有 → 等待
  │ 执行 <clinit> 方法        │
  │ （可能耗时很长）           │ 阻塞中...
  │                           │
  │ <clinit> 执行完毕         │
  │ 释放 Init Lock            │
  │ 设置 class.state =        │
  │   fully_initialized       │
  │                           ├─ 获得锁 ✓
  │                           │ 检查 class.state = fully_initialized
  │                           │ 直接跳过 <clinit>，返回！
  │                           │
  ▼                           ▼

⚠️ <clinit> 死锁——真实线上事故

// 类 A 的 <clinit> 中引用了类 B
class A {
    static { B.getValue(); }  // 触发 B 的初始化
}
// 类 B 的 <clinit> 中引用了类 A
class B {
    static { A.getValue(); }  // 触发 A 的初始化
}

// Thread-1 初始化 A，持有 A 的 Init Lock，等待 B
// Thread-2 初始化 B，持有 B 的 Init Lock，等待 A
// → 死锁！两个线程永远阻塞在 Init Lock 上

// 排查方法：
// jstack 中看到线程阻塞在 Class.forName 或直接引用类时
// thread dump 显示 "waiting to lock <0x...> (a java.lang.Class)"

避免方案：静态初始化块中尽量避免触发其他类的初始化，或者将相互依赖的初始化逻辑移到静态方法中延迟调用。

⭐ 面试要点

✅ 必须掌握的细节

  • ▸ static final 基本类型常量在准备阶段就有值
  • ▸ static final 引用类型准备阶段仍是 null
  • ▸ <clinit> 由 javac 自动生成，包含所有静态赋值和静态块
  • ▸ JVM 通过 Init Lock 保证 clinit 单次执行
  • ▸ clinit 中的循环依赖会导致死锁
