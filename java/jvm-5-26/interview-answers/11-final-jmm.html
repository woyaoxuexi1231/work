final 的内存语义与String 不可变性

final 在 JMM 里不仅仅是"不能修改"——它有特殊的 freeze 语义。构造函数中对 final 字段的写入，与随后通过其他线程读取该对象引用之间，JMM 保证没有重排序。但如果 this 在构造函数中逸出，final 的保证就失效了。

1 final 字段的 freeze 语义

JMM 对 final 的两条特殊规则

规则 1：构造函数内 final 字段的写入，与对象的引用赋值给其他变量之间，禁止重排序

// JMM 保证：
class SafeObj {
    final int x;
    SafeObj() {
        x = 42;            // ① 写入 final 字段
    }                      // ② 构造函数结束 → ★ freeze 操作
}
// freeze 操作插入 StoreStore 屏障！
// 确保 x=42 在对象引用被发布之前对所有 CPU 可见

SafeObj obj = new SafeObj();  // ③ 把引用赋给 obj

// ★★ JMM 保证：任何线程通过 obj 读取 obj.x 一定看到 42
// 不会看到 0（默认值），即使没有 volatile 或 synchronized

// 对比非 final 字段：
class UnsafeObj {
    int x;
    UnsafeObj() { x = 42; }
}
// 另一线程可能看到 x = 0（默认值）！
// 因为 x = 42 和 UnsafeObj 引用赋值之间可能被重排序

规则 2：读 final 字段时，JMM 保证不会看到"未初始化"的值

对于 final 引用类型字段：不仅引用本身不可变，JMM 保证在构造函数中通过该引用写入的其他对象字段，也能在 final 字段被读取时一并可见。 这就是所谓的 "final 字段的传递性保证"。

2 this 引用逸出：final 保证失效 

// ❌ 危险代码：this 在构造函数中逸出
public class DangerousObj {
    final int x;
    
    public DangerousObj(Registry registry) {
        registry.register(this);  // ← this 逸出！在构造函数完成之前
        // 此时 x 还没有被"freeze"
        this.x = 42;
    }
    // 构造函数结束时 freeze 操作才执行
}

// 另一个线程通过 registry 拿到 DangerousObj 引用：
// DangerousObj obj = registry.get();
// int val = obj.x;  // 可能看到 0！！！final 保证失效了！
// 因为 freeze 操作还没有发生

🔥 this 逸出的根本原因：final 字段的 freeze 操作发生在构造函数正常结束时。 如果 this 在构造函数结束前被其他线程获得，那个线程就能在 freeze 之前访问 final 字段——看到零值。

常见的 this 逸出场景： ① 构造函数中启动新线程（新线程可以访问 this） ② 构造函数中注册回调/监听器 ③ 构造函数中把 this 放入静态集合

3 String 不可变性与 final 的内存语义

String 不可变的三层保证

① final 字段 + 无 setter

public final class String {
    private final char value[];  // JDK 8 及之前
    // 或
    private final byte[] value;  // JDK 9+ (Compact String)
    private final int hash;      // 懒计算，但缓存后不可变
    // ...
}

value 和 hash 都是 final 的——JMM 保证任何线程看到的 String 对象，其 value 数组引用不会变。

② 类本身是 final 的

String 类不能被继承——不能通过子类覆盖方法来破坏不可变性。

③ 所有方法返回新对象，不修改自身

substring()、concat()、replace() 等都返回新的 String。 JDK 7 之后 substring 也复制数组（而不是 JDK 6 的"共享数组"），彻底杜绝了通过 substring 修改原 String 的可能性。

💡 String 的不可变性和 final 内存语义的关系： 假设 String 的 value 不是 final 的——那么在多线程环境下，一个线程创建了一个 String，另一个线程可能看到 value 还是 null。 final 的 freeze 语义保证了 String 对象一旦被安全发布（safe publication），所有线程看到的 value 都是完整的。 这是 String 可以用来做 HashMap 的 key 而不担心并发问题的底层原因——hashCode 一旦计算就可以安全缓存， 因为 value 数组不会变化，hash 值也就不会变化。

⚠️ String 的"不可变"能被反射打破：通过反射拿到 value 字段并把 final 修饰符去掉，可以修改 String 的内容。 这种"魔法"在一些框架中被用于性能优化（如修改从外部接收的密码字符串为全 0，防止内存 dump 泄漏）。 但 JDK 12+ 对此做了限制，反射修改 final 字段的行为已被禁止。

⭐ 面试要点

✅ final 的 JMM 语义

  • ▸ freeze 操作在构造函数结束时执行
  • ▸ freeze = StoreStore 屏障，保证 final 字段写入可见
  • ▸ this 逸出导致 freeze 未执行，final 保证失效
  • ▸ final 引用类型的传递性保证
  • ▸ String 不可变 = final class + final fields + 防御性复制

⚠️ 常见 this 逸出场景

  • ✗ 构造函数中 start() 线程
  • ✗ 构造函数中注册监听器
  • ✗ 构造函数中发布到静态集合
  • ✓ 用工厂方法 + 私有构造函数避免逸出
