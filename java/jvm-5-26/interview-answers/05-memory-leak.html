内存泄漏排查与Heap Dump 分析

老年代满 → Full GC → 还是满 → OOM。Heap dump 拉下来 8GB。MAT 的直方图和支配树视图各解决什么问题？如何从 GC Root 路径定位 ThreadLocal 泄漏的精确位置？这不是工具操作手册，这是取证分析的方法论。

1 工具选型：8GB Heap Dump 用什么分析？

工具适用场景8GB Dump 表现推荐度
Eclipse MAT深度分析、泄漏定位需要 12-16GB 内存打开⭐⭐⭐⭐⭐
jhat (过时)简单浏览8GB 直接 OOM，不推荐⭐
JProfiler实时监控 + Dump 分析需商业许可，表现良好⭐⭐⭐⭐
arthas + heapdump线上快速诊断不直接分析，配合 MAT⭐⭐⭐

💡 大 dump 处理技巧：MAT 打开 8GB dump 需要 -Xmx16g。 如果本机内存不够，先用 jcmd <pid> GC.heap_dump -all=false dump.hprof 只 dump 存活对象， 减少约 30-50% 体积。也可以在 dump 前先触发一次 Full GC（jcmd <pid> GC.run），这样垃圾对象不会进入 dump。

2 直方图 vs 支配树：两个视图，两种思路

直方图（Histogram）

功能：按类统计实例数量和总占用内存（Shallow Heap + Retained Heap）。

解决的问题："哪个类占的内存最多？" — 快速定位"吃内存大户"。 比如发现 byte[] 占了 6GB，然后是 HashMap$Node 占了 1GB。

Shallow Heap：对象自身占用的内存（不含引用对象）。 Retained Heap：这个对象及其"独占"引用的所有对象的总和——如果这个对象被回收，能释放的内存。这才是泄漏分析的核心指标。

支配树（Dominator Tree）

功能：按对象引用关系构建树状结构，展示"谁支配（持有）谁"。

解决的问题："是谁持有了这些大对象，让它们无法被回收？" 从树的根节点向下展开，可以看到内存大户的持有链。 

支配树示例：
class MyCache (Retained: 3.2GB)
  └─ HashMap$Node[] (Retained: 3.2GB)
       └─ HashMap$Node (Retained: 1.6GB)
            └─ byte[] (Retained: 1.6GB)
       └─ HashMap$Node (Retained: 1.6GB)
            └─ byte[] (Retained: 1.6GB)
// 两个 byte[] 之所以占了 3.2GB 且无法回收，
// 是因为 MyCache 的 HashMap 持有它们

💡 核心区别：直方图告诉你"嫌疑人是谁"（哪个类占了最多内存），支配树告诉你"犯罪动机"（是什么引用链导致它无法被回收）。 排查顺序永远是：先看直方图找到内存大户 → 再看支配树追溯到持有者 → 沿着 GC Root 路径确认泄漏根因。

3 如何从支配树证明"这条引用链就是泄漏根因"？

证明四步法

Step 1：在支配树中找到存活对象的持有者

比如支配树中，一个巨大的 HashMap$Node 被 StaticCacheManager 的静态字段 cacheMap 持有。

Step 2：追溯 GC Root

在 MAT 中选中对象 → Path to GC Roots → exclude weak/soft/phantom references（排除虚引用干扰）。 如果引用路径从 <system class> 出发，经过静态字段到集合，再到目标对象——这就是一条永不断开的强引用链。

Step 3：验证"应该被回收但没被回收"

结合业务逻辑：这个 cacheMap 里存的是"用户登录会话"对象——用户已经登出/会话过期了，但这些对象仍然在 Map 里。 没有过期清理机制 + 静态持有 = 确认泄漏。

Step 4：用 retain size 量化泄漏规模

如果 cacheMap 的 retained heap 是 5.6GB，且这些对象在业务上已经过期——那么 5.6GB 就是泄漏的量化规模。 对比 Full GC 后老年代仍占用的内存（如 6GB），确认吻合。

4 ThreadLocal 泄漏：从 GC Root 到具体 Entry

ThreadLocal 的引用链结构

ThreadLocal 内存结构（关键：理解引用类型）：
                   
Thread (GC Root)
  └─ threadLocals: ThreadLocal.ThreadLocalMap
       └─ table: Entry[]  ← Entry 继承 WeakReference<ThreadLocal>
            └─ Entry[0]
                 ├─ referent: WeakReference → ThreadLocal 对象（弱引用）
                 └─ value: Object  ← 这就是你 set 进去的值（强引用！）
            └─ Entry[1]
                 ├─ referent: WeakReference → ThreadLocal 对象
                 └─ value: Object

泄漏条件：
  ThreadLocal 对象在外部被置为 null（或 ThreadLocal 变量出了作用域）
  → Entry 的 referent（弱引用）被 GC 回收 → key 变成 null
  → 但 Entry 本身和 value 仍在 table 中（强引用）
  → 只要 Thread 还活着，value 永远无法被回收
  
★ 这就是为什么线程池中使用 ThreadLocal 必须 remove()！

MAT 中的定位步骤

① Histogram 视图 → 找到内存大户

搜索 java.lang.ThreadLocal$ThreadLocalMap$Entry，如果 Entry 实例数异常多，且 retain size 很大 → ThreadLocal 泄漏嫌疑。

② Merge Shortest Paths to GC Roots

找到具体的 Entry 对象 → Merge Shortest Paths to GC Roots（选 exclude weak references）。 这条路径会显示：Thread → threadLocals → table → Entry → value。

③ 定位具体线程

在 GC Root 路径中可以看到是哪个 Thread 对象持有了这个 Entry。 MAT 会显示线程名称（如 http-nio-8080-exec-25），这就定位到了具体线程。

④ 查看 Entry 的 key 和 value

在 Object Query Language (OQL) 中： SELECT e.referent, e.value FROM ThreadLocal$ThreadLocalMap$Entry e WHERE e.referent = null 这会列出所有 key 为 null 的 Entry——即所有泄漏的 value。查看 value 的类型和内容确认泄漏的业务数据。 

// MAT OQL — ThreadLocal 泄漏定位神器

// 1. 找出所有 key 为 null 的 ThreadLocal Entry（确认泄漏）
SELECT e.value.toString() FROM 
  java.lang.ThreadLocal$ThreadLocalMap$Entry e 
  WHERE e.referent = null

// 2. 按线程分组，看哪个线程泄漏最多
SELECT t.name, COUNT(e) FROM 
  java.lang.Thread t, 
  java.lang.ThreadLocal$ThreadLocalMap$Entry e 
  WHERE (t.threadLocals.table = e.@objectAddress) 
  AND (e.referent = null)
  GROUP BY t.name

// 3. 找出占用内存最大的泄漏 value
SELECT OBJECTS dominators(e.value) FROM 
  java.lang.ThreadLocal$ThreadLocalMap$Entry e 
  WHERE e.referent = null

🔥 ThreadLocal 泄漏的经典场景：线程池中的线程被复用，ThreadLocal 变量在任务结束后没有调 remove()。 多个任务复用同一线程，value 不断被新值覆盖，旧的 value 变成 key=null 的 Entry，永久滞留在 ThreadLocalMap 中。 修复方案：在 finally 块中调用 ThreadLocal.remove()。

⭐ 面试回答框架

✅ 内存泄漏排查五步法

  1. 确认泄漏：Full GC 后老年代仍然满 → 真泄漏
  2. Histogram 定位可疑类（按 Retained Heap 排序）
  3. Dominator Tree 追溯持有者引用链
  4. Path to GC Roots 确认泄漏链路（排除虚引用）
  5. OQL 量化泄漏规模并定位业务代码

🛠️ 关键 JVM 参数

  • -XX:+HeapDumpOnOutOfMemoryError
  • -XX:HeapDumpPath=/dump/
  • -XX:+ExitOnOutOfMemoryError
  • jcmd <pid> GC.heap_dump dump.hprof
  • jmap -dump:live,format=b,file=dump.hprof
