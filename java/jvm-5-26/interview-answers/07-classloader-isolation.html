双亲委派的破坏者：Tomcat / SPI / TCCL

双亲委派不是金科玉律。Tomcat 需要隔离不同 Web 应用的类，SPI 需要从子加载器反向查找实现类——两者都"破坏"了双亲委派。理解这些破坏不是为了炫技，而是为了理解类加载器设计的本质：解决"谁加载谁"的问题。

1 loadClass：双亲委派的代码实现 

// java.lang.ClassLoader.loadClass(String, boolean) — 核心逻辑
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // Step 1: 检查是否已经加载过
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // Step 2: ★ 先委托给父加载器（双亲委派的核心）
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // 没有父加载器就用 Bootstrap ClassLoader
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器找不到，没关系，继续
            }
            if (c == null) {
                // Step 3: 父加载器找不到，自己来加载
                c = findClass(name);  // ★ 破坏双亲委派就是改写这里的逻辑
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}

2 两种破坏双亲委派的姿势

姿势一：重写 loadClass

直接把 loadClass 的逻辑改了——"不向上委派，我自己先加载"。 

@Override
public Class<?> loadClass(String name) {
    // 不管父加载器，自己先加载
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        try {
            c = findClass(name); // 自己找
        } catch (ClassNotFoundException e) {
            c = super.loadClass(name); // 找不到再兜底
        }
    }
    return c;
}

Tomcat 的 WebappClassLoader 就是这么干的。

姿势二：线程上下文类加载器

不修改 loadClass，而是通过 TCCL "绕过"父加载器。 

// SPI 的 ServiceLoader 就是这么用的：
ClassLoader cl = Thread.currentThread()
    .getContextClassLoader();
// 用 TCCL 加载实现类，而不是用
// ServiceLoader 自身的类加载器（Bootstrap）

SPI 和 JDBC 驱动的加载依赖这种方式。

3 Tomcat 的 WebappClassLoader：隔离与共享

Tomcat 的类加载器层次

               Bootstrap ClassLoader
                        │
               Platform ClassLoader (JDK 9+)
                        │
               Common ClassLoader    ← 加载 tomcat/lib/*.jar（共享库）
                  ┌────────┴────────┐
                  │                  │
    WebappClassLoader-A    WebappClassLoader-B
    (应用 A 的 /WEB-INF)    (应用 B 的 /WEB-INF)
    
    ★ 每个 Web 应用有自己独立的 WebappClassLoader
    
    加载顺序（WebappClassLoader 的 loadClass）：
    1. 先检查自己缓存
    2. 如果之前委托过 → 再试一次委托
    3. 自己加载（/WEB-INF/classes → /WEB-INF/lib/*.jar）★ 优先自己！
    4. 如果没找到 → 委托给父加载器（Common → Platform → Bootstrap）
    
    ★ 这与标准双亲委派相反！先自己加载，找不到才委托。

↔ 隔离原理

应用 A 的 com.example.User 由 WebappClassLoader-A 加载， 应用 B 的同名类由 WebappClassLoader-B 加载。 JVM 认为不同类加载器加载的同名类是不同类——天然的隔离。

↕ 共享原理

当 WebappClassLoader 找不到某个类时，会向上委托给 Common ClassLoader。 Common 里的类（如 servlet-api.jar）对所有 Web 应用可见且由同一个加载器加载——保证了类型一致。

4 SPI 为什么必须用 TCCL？

SPI 的困境

以 JDBC 为例：java.sql.DriverManager 是核心类，由Bootstrap ClassLoader 加载。 但具体的驱动实现（如 com.mysql.cj.jdbc.Driver）是在应用的 classpath 中，由App ClassLoader 加载。 

SPI 加载链路图：

Bootstrap ClassLoader 加载：
  java.sql.DriverManager  ← 核心接口
  java.util.ServiceLoader ← SPI 加载器（也在 rt.jar 中！）

App ClassLoader 加载：
  com.mysql.cj.jdbc.Driver  ← 实现类

问题：
  ServiceLoader 由 Bootstrap 加载，它"看"不到 App ClassLoader 的类！
  按照双亲委派：Bootstrap → Platform → App → 自定义
  只能从上往下找，不能从下往上。
  
  ServiceLoader 需要"向下"找到实现类 → 但双亲委派只允许"向上"委托 → 矛盾！

解决方案：线程上下文类加载器（TCCL）
  Thread.currentThread().getContextClassLoader()
  → 返回的是 App ClassLoader（在 main 线程中默认设置）
  → ServiceLoader 通过 TCCL 可以"向下"加载到 App ClassLoader 中的实现类

💡 TCCL 的本质：它是一个"反向通道"。双亲委派的模型是自顶向下（Bootstrap → App → Custom）， 而 TCCL 允许被 Bootstrap 加载的类（如 ServiceLoader）反过来访问应用层的类加载器。 这是对双亲委派模型的"破坏"，但也是 SPI 这种设计模式的必然要求。
