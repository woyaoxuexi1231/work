GC 回收机制 第三问

GC 日志分析与Full GC 诊断

GC 日志是 JVM 的"心电图"。Full GC 不是病因，是症状。能读懂 GC 日志的每一行、每一个括号里的触发原因，才能从症状反推根因。Allocation Failure、Ergonomics、Concurrent Mode Failure——每个关键词背后都是一条独特的排查路径。

1 GC 日志格式完全拆解 

// JDK 9+ 统一 GC 日志格式 (启用: -Xlog:gc*:file=gc.log:time,uptime,level,tags)
[2024-01-15T10:23:45.123+0800][info][gc,start     ] GC(1234) Pause Young (Allocation Failure)
[2024-01-15T10:23:45.125+0800][info][gc,heap      ] GC(1234) Eden: 1638400K->0K(1638400K) Survivor: 204800K->102400K(204800K)
[2024-01-15T10:23:45.127+0800][info][gc,heap      ] GC(1234) Old: 512000K->524288K(1048576K)
[2024-01-15T10:23:45.127+0800][info][gc,metaspace ] GC(1234) Metaspace: 65536K->65536K(131072K)
[2024-01-15T10:23:45.128+0800][info][gc           ] GC(1234) Pause Young (Allocation Failure) 1500M->600M(4000M) 5.123ms
[2024-01-15T10:23:45.128+0800][info][gc,cpu       ] GC(1234) User=0.02s Sys=0.01s Real=0.01s

// 关键指标解读：
// GC(1234)       — 第 1234 次 GC
// Pause Young    — Young GC (STW 暂停)
// Allocation Failure — 触发原因：Eden 区满，无法分配新对象
// 1500M->600M   — GC 前堆占用 → GC 后堆占用
// (4000M)        — 堆总大小
// 5.123ms        — STW 停顿时间 ★ 最重要指标

三个必看指标

① STW 停顿时间（Pause Time）

这是直接面向用户的指标。如果 Young GC 停顿超过 50ms（G1）或 100ms（Parallel），说明新生代太大或存活对象太多。 如果 Full GC 停顿超过 2 秒，这就是生产事故级别的信号。

② GC 前后的堆占用变化

如果 Young GC 后 Eden 不是 0，或者 Survivor 几乎没有释放——说明大量对象"活过了"Young GC，可能存在过早晋升问题。 如果 Full GC 后老年代依然接近满（如 95% → 93%），说明这些对象真的都是存活的——这不是 GC 参数的问题，是内存泄漏。

③ GC 触发原因（Cause）

这是整个日志中最有诊断价值的信息。同一个 Full GC 停顿，原因不同，排查方向完全不同。

2 Full GC 的两种触发原因：完全不同的排查方向

[Full GC (Allocation Failure)]

含义

尝试分配对象到老年代时，老年代确实没有连续空间了。JVM 必须先做一次 Full GC 来回收空间，才能完成分配。

根因方向

  • ▸ 大规模晋升：Young GC 后大量对象要晋升，老年代装不下
  • ▸ 大对象直接分配：超过了 -XX:PretenureSizeThreshold 的大对象直接进老年代
  • ▸ 老年代碎片化：CMS 回收后碎片太多，找不到连续空间

排查步骤

  1. 查看 Full GC 前的 Young GC 日志：每次晋升了多少对象？
  2. 查看老年代占用趋势：是缓慢增长（内存泄漏）还是突然暴涨（大对象）？
  3. 查看 Humongous Allocation（G1）或大对象直接分配（Parallel/CMS）的日志

[Full GC (Ergonomics)]

含义

JVM 的自适应策略（Ergonomics）判定"现在做一次 Full GC 对整体性能有利"，主动触发。 不是空间真的不够了，而是 JVM 预测需要整理碎片或扩大某个区域。

常见触发场景

  • ▸ G1 的 Mixed GC 跟不上分配速度，回退到 Full GC
  • ▸ CMS 碎片化严重，JVM 决定做一次压缩
  • ▸ AdaptiveSizePolicy 认为需要调整各代大小

应对策略

  1. 如果是 G1 的退化 Full GC：增大 -XX:G1ReservePercent（默认 10%）
  2. 如果是 AdaptiveSizePolicy：关闭自适应策略 -XX:-UseAdaptiveSizePolicy
  3. 增加 -XX:G1HeapWastePercent 让 G1 更"容忍"垃圾

3 CMS 的两大经典故障

Concurrent Mode Failure（并发模式失败）

发生在 CMS 的并发标记和并发清除期间。CMS 在回收老年代时是和业务线程并发进行的， 但如果在 CMS 还没回收完之前，老年代就被新晋升的对象填满了——CMS 只能放弃，回退到 Serial Old 做一次单线程 STW Full GC。 

// CMS GC 日志中的典型表现：
[GC (CMS Initial Mark) ...]
[GC (CMS Concurrent Mark) ...]
[GC (CMS Concurrent Preclean) ...]
[GC (CMS Concurrent Abortable Preclean) ...]  ← CMS 还在并发工作
// ↑ 业务线程同时也在疯狂分配对象并晋升到老年代 ↑
[Full GC (Concurrent Mode Failure) ...]       ← 老年代不够用了！
    CMS: 放弃并发回收，回退到 Serial Old
    停顿时间：可能 5-30 秒（单线程！）

🔥 发生瞬间的物理过程：CMS 的并发标记阶段需要遍历整个老年代的对象图。 但这个过程中，业务线程还在创建新对象（Eden 满 → Young GC → 对象晋升到老年代）。 如果晋升速度 > CMS 回收速度，老年代就会在 CMS 完成前被填满。此时 CMS 还没标记完——它甚至不知道哪些对象是垃圾——所以只能 STW。

排查方向

  • ▸ 增大老年代大小，给 CMS 更多缓冲空间
  • ▸ 降低 CMS 触发阈值：-XX:CMSInitiatingOccupancyFraction=65（更早开始并发回收）
  • ▸ 检查 Young GC 频率：如果 Young GC 太快，对象晋升太快，CMS 追不上
  • ▸ 根本解决：迁移到 G1 或 ZGC

Promotion Failed（晋升失败）

发生在Young GC 期间。Young GC 要把 Survivor 中活下来的对象晋升到老年代，但老年代没有足够的连续空间（是碎片问题，不一定是满的！）。 

// GC 日志中典型表现：
[GC (Allocation Failure) ... 
    [ParNew (promotion failed): 204800K->204800K(204800K)]  ← 晋升失败！
    204800K->204800K(204800K), 0.1234567 secs]
[Full GC (Allocation Failure) ...]  ← 紧接着触发 Full GC 来整理碎片

// promotion failed 的核心：老年代总空闲可能还很多，
// 但全都是碎片，找不到一块连续的足够大的空间！

💡 如何从日志区分 CMF vs PF： Concurrent Mode Failure 的 Full GC 之前一定有一串 CMS 并发阶段日志（CMS Initial Mark → Concurrent Mark → ...）。 Promotion Failed 的 Full GC 之前是 ParNew 的 Young GC，且带有 "(promotion failed)" 标记。 看时序就能定位。

排查方向

  • ▸ 老年代碎片化严重 → 增大 CMS 触发频率或迁移到 G1
  • ▸ Survivor 太小 → 对象过早晋升 → 增大 SurvivorRatio
  • ▸ 单个对象太大 → 使用 -XX:PretenureSizeThreshold 直接进老年代

4 Full GC 排查的完整框架 

Step 1 — 看触发原因（Cause）
  [Full GC (Allocation Failure)]  → 空间真的不够
  [Full GC (Ergonomics)]          → JVM 自适应策略触发
  [Full GC (System.gc())]         → 代码里显式调了 System.gc()
  [Full GC (Metadata GC Threshold)] → 元空间满了触发的 Full GC
  [Full GC (G1 Evacuation Pause)] → G1 退化 Full GC

Step 2 — 看堆占用变化
  Full GC 后老年代: 90% → 90%  → 内存泄漏（这些对象真活着）
  Full GC 后老年代: 90% → 30%  → 晋升太快，非泄漏
  Full GC 后老年代: 90% → 88%  → 泄漏 + 碎片掺杂

Step 3 — 看时间序列
  Full GC 每 5 分钟一次 → 晋升太猛了
  Full GC 每 30 秒一次 → 内存泄漏逼近 OOM，赶紧 dump
  Full GC 前有 CMS 阶段 → Concurrent Mode Failure
  Full GC 前有 ParNew(promotion failed) → 晋升失败/碎片

Step 4 — 关联业务指标
  Full GC 时间段的 QPS 是否突增？
  Full GC 时间段的 TPS 下降了多少？
  是否有大对象分配的对应业务操作？

⭐ 面试回答框架

✅ GC 日志诊断要点

  • ▸ 三个必看指标：停顿时间、堆变化、触发原因
  • ▸ Allocation Failure vs Ergonomics 的不同排查方向
  • ▸ Concurrent Mode Failure：CMS 回收速度跟不上晋升速度
  • ▸ Promotion Failed：老年代碎片，不是空间满
  • ▸ 四步排查框架：原因 → 占用 → 时序 → 业务关联
