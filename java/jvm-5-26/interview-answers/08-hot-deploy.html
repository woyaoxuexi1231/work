热部署机制：类卸载、元空间与 ThreadLocal 陷阱

"换掉 ClassLoader"只是热部署的一句话描述，但真正的挑战在类卸载的四个必要条件、ThreadLocal 如何阻止 GC、以及元空间 OOM 的排查。这才是热部署的工程难点。

1 一个类被卸载的四个必要条件

条件 ①：该类的所有实例都已被回收

堆中不能再有任何该类的对象。这包括你直接 new 的对象，也包括反射创建的对象，以及序列化反序列化出来的对象。

条件 ②：加载该类的 ClassLoader 实例已被回收

这是最关键的条件。Java 中，类是由 ClassLoader 加载的，ClassLoader 对象持有所有它加载的类的引用。 只要 ClassLoader 对象还活着，它加载的所有类都不能被卸载。 反过来：只要 ClassLoader 被 GC 了，它加载的类就可以被卸载。

条件 ③：该类的 java.lang.Class 对象没有任何引用

不能有反射引用（如缓存的 Method/Field 对象持有 Class 引用）、不能有 Class.forName 产生的引用、 也不能有别的类的静态字段指向这个 Class 对象。

条件 ④：该类的静态字段引用的对象都已被回收

如果这个类有一个静态的 static Map<String, Object> cache 引用了大量数据， 这些数据必须先被回收，然后类的元数据才能从元空间中移除。

实际操作：创建一个新的 ClassLoader，用新 ClassLoader 加载新版本的类，然后丢弃旧 ClassLoader 的所有引用。 当 GC 发生时，旧 ClassLoader 和它加载的所有类都会被回收——这就是"换掉 ClassLoader"的完整含义。

2 ThreadLocal：热部署的头号杀手

为什么 ThreadLocal 会阻止类卸载？

引用链路：Thread → ThreadLocalMap → Entry → Value (某个类的实例)

                                    旧 ClassLoader 加载的类 A
                                           ↑
                                    类 A 的实例（value）
                                           ↑
Thread (GC Root) → ThreadLocalMap → Entry → value (强引用)
                     ↑
                Entry.referent → ThreadLocal (弱引用，但 value 是强引用！)

问题：
  即使 ThreadLocal 对象本身被回收（弱引用），Entry 里的 value 仍然被
  Thread → ThreadLocalMap → Entry 这条强引用链持有。
  只要这条链不断，value 对象（旧 ClassLoader 加载的类 A 的实例）就永远
  不会被回收 → 类 A 不能被卸载 → 旧 ClassLoader 不能被 GC。
  
★ 一条 ThreadLocal 泄漏，整个 ClassLoader 和它加载的所有类都跟着陪葬。

🔥 为什么热部署场景特别容易中招：Web 应用中常用 ThreadLocal 存 RequestContext、用户信息等。 热部署时，旧的 WebappClassLoader 被丢弃，但线程池中的线程（如 Tomcat 的 http-nio-8080-exec-* ）是复用的——它们持有的 ThreadLocalMap 中还保留着旧 Entry。 这些 Entry 的 value 引用了旧 ClassLoader 加载的类的实例 → 旧 ClassLoader 永远不被 GC → 元空间逐渐被旧类的元数据占满。

✅ 解决方案

  • ▸ 在 ServletContextListener.contextDestroyed() 中清理所有已知的 ThreadLocal
  • ▸ 使用 ThreadLocal.remove() 而不是 set(null)
  • ▸ 扫描 Thread 对象的 threadLocals 字段，反射清理（Arthas 的 tt -d 就是干这个的）
  • ▸ 监控元空间使用率，设置 -XX:MaxMetaspaceSize

3 元空间 OOM：热部署的定时炸弹

Metaspace 和 PermGen 的核心区别

维度PermGen (JDK 7-)Metaspace (JDK 8+)
存储位置堆内（受 -Xmx 限制）堆外（直接内存）
默认上限约 82MB无限制（直到物理内存耗尽）
类卸载条件苛刻，很少发生GC 时自动回收废弃类元数据

热部署导致元空间 OOM 的排查思路

Step 1：确认是元空间 OOM

java.lang.OutOfMemoryError: Metaspace
// 不是 Java heap space，是 Metaspace

Step 2：查看元空间使用趋势

jstat -gc <pid> 1000 查看 MU (Metaspace Used) 列。 如果每次热部署后 MU 增长但不回落 → 旧类的元数据没有被回收。

Step 3：追踪 ClassLoader 泄漏

在 MAT 中搜索旧的 WebappClassLoader 实例。 Path to GC Roots 找到是谁持有了 ClassLoader（通常是 ThreadLocal → value → 某对象 → 某类 → ClassLoader）。

Step 4：修复并验证

清理所有已知的 ThreadLocal → 重新部署 → 观察 MU 是否回落。 设置 -XX:MaxMetaspaceSize=256m 作为安全上限， 同时配置 -XX:+HeapDumpOnOutOfMemoryError 在 OOM 时自动 dump。

⭐ 面试要点

✅ 热部署核心要点

  • ▸ 类卸载四条件：实例回收 → ClassLoader 回收 → Class 对象无引用 → 静态字段回收
  • ▸ ThreadLocal 的 value 是强引用，阻止 ClassLoader 回收
  • ▸ 元空间默认无上限 → 热部署必须设置 MaxMetaspaceSize
  • ▸ 排查用 jstat MU 列 + MAT ClassLoader 引用链
  • ▸ 在 contextDestroyed 中清理 ThreadLocal

🛠️ 推荐 JVM 参数

  • -XX:MaxMetaspaceSize=256m
  • -XX:+MetaspaceSize=128m
  • -XX:+CMSClassUnloadingEnabled
  • -XX:+TraceClassUnloading
