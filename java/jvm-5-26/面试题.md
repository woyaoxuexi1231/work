## 一、GC 回收机制——从算法到调优实战

**第一问**：你简历里写着了解 GC。那先把常见的几种垃圾回收算法（标记-清除、标记-整理、复制）的特点、适用场景、各自的劣势说一遍。然后告诉我，分代收集理论为什么能把它们组合起来？为什么新生代用复制，老年代用标记-整理或清除？这个“为什么”如果只回答“因为对象存活率不同”，那就太肤浅了。我要你从**内存分配效率、空间碎片、CPU 缓存、跨代引用**这些角度给一个完整的解释。

**第二问（打断，追问）**：现在有没有不分代的垃圾回收器？ZGC 和 Shenandoah 就是。既然分代那么好，为什么它们要冒天下之大不韪，彻底放弃分代？**停顿时间和吞吐量的权衡在这里发生了什么根本性的理念转变？** 如果一个系统对停顿极度敏感，你会选择 G1 还是 ZGC？别给我背官方参数，说说**读屏障、颜色指针、并发重定位**这些实际是怎么解决碎片化和停顿的。

**第三问（推向实操）**：  
你的生产环境，有一天突然开始频繁 Full GC，每次停顿超过 2 秒。你没有盲猜，而是先看了 GC 日志。告诉我，**你在日志里会重点看哪几个指标？** 怎么根据 `[Full GC (Ergonomics)]` 还是 `[Full GC (Allocation Failure)]` 判断触发原因？  
如果是 CMS 的 `concurrent mode failure`，你知道它发生的瞬间发生了什么吗？`promotion failed` 又是怎么回事？从日志中怎么分辨是这两种情况？

**第四问（压力场景，需要当场设计）**：  
我给你一个系统：每秒 2000 次支付请求，每个请求会在内存里产生约 8KB 的临时对象，存活时间极短。堆设了 4G，新生代 2G，Eden 区 1.6G，两个 Survivor 各 0.2G。每隔几个小时，就会出现一次长达 2 秒的停顿，GC 日志显示是 Young GC 时间暴涨，而且伴随一次 “promotion failed”，然后触发 Full GC。
- 这个场景的根因到底是什么？**给我指出 Survivor 区大小和对象年龄晋升阈值的隐藏陷阱。**
- 为什么动态年龄判断机制在这里会帮倒忙？
- 你打算怎么调整参数？如果让你现场写一段能够复现这个“Survivor 溢出导致过早晋升”的代码，你思路是什么？

**第五问（内存泄漏排查）**：  
一个服务运行了几天后，老年代满，Full GC 后还是满，最终 OOM。Heap dump 拉下来有 8GB。你用什么工具分析？MAT 里直方图和支配树视图分别解决什么问题？当你发现一个对象被某个类的静态集合持有，怎样从支配树里**证明这条引用链就是泄漏根因**？给你一个 ThreadLocal 泄漏的场景，你打算如何通过 GC Root 路径来定位到具体线程、具体 `Entry` 和 `value`？

**第六问**：类加载的三个阶段：加载、链接、初始化，链接又分验证、准备、解析。**准备阶段**会为静态变量分配内存并赋零值，如果静态变量是 `static final` 的常量，会发生什么？`static int a = 10` 和 `static final int B = 10` 在准备阶段的值分别是多少？初始化阶段 `<clinit>` 方法是怎么生成的？JVM 如何保证一个类的 `<clinit>` 在多线程环境下只被执行一次？画出多线程同时触发类初始化的完整流程。

**第八问（逼近极限）**：
