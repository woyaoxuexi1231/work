synchronized 锁升级全解剖：
从 Mark Word 到安全点的每一纳秒

锁升级不是抽象概念，而是对象头里 64 个 bit 的精确状态机。 偏向锁的撤销必须暂停目标线程，轻量级锁的膨胀依赖 CAS 失败检测， 而电商系统为什么要 -XX:-UseBiasedLocking？ 答案藏在安全点的 STW 代价里。

1

Mark Word 状态机：64 bit 里的四重人格

每个 Java 对象的对象头由 Mark Word（8字节/64bit） + Klass Pointer + 数组长度（可选）组成。 synchronized 的所有锁状态都编码在 Mark Word 中。以下是 64 位 JVM 下的完整布局：

无锁 / 未锁定 tag = 01
unused (25)
hashCode (31)
0
age (4)
01
偏向锁 tag = 01, biased = 1
Thread ID (54)
Epoch (2)
1
age (4)
01

同一线程重入时，只需比较 Thread ID，零原子操作

轻量级锁 tag = 00
指向栈帧 Lock Record 的指针 (62)
00

CAS 将 Mark Word 复制到 Lock Record，自旋等待，不阻塞线程

重量级锁 tag = 10
指向 ObjectMonitor 的指针 (62)
10

OS Mutex + EntryList/CXQ，竞争线程挂起（park/unpark）

💡 关键约束

锁升级是单向不可逆的（偏向→轻量→重量）。 一旦膨胀为重量级锁，即使竞争消失也不会降级回轻量级或偏向。 这是因为 ObjectMonitor 的维护成本已经付出，降级反而引入额外复杂度。 （注：JDK 15+ 默认禁用偏向锁，JDK 18+ 彻底移除。）

2

每一步升级的精确触发条件

①

无锁 → 偏向锁

触发条件：第一个线程执行 monitorenter，且对象处于无锁状态（tag=01, biased=0），且 JVM 开启了 -XX:+UseBiasedLocking。

操作：CAS 将当前线程 ID 写入 Mark Word 的 Thread ID 字段，同时将 biased 标志位置 1。
如果 CAS 成功 → 获得偏向锁。
如果 CAS 失败 → 说明已有其他线程抢先偏向，直接进入轻量级锁竞争。

延迟偏向：对象创建后不会立即启用偏向，需等待 BiasedLockingStartupDelay（默认 4s）。这是为了避免启动阶段大量临时对象的无效偏向开销。

②

偏向锁 → 轻量级锁（偏向撤销）

触发条件：第二个线程尝试获取已被偏向的锁。
即 monitorenter 时发现 Mark Word 中 Thread ID ≠ 当前线程。

⚠️ 这不是简单的 CAS 替换！必须先撤销偏向（Revoke Bias），这是一个需要 Safepoint 的操作（详见下一节）。

撤销后：对象变为无锁状态，然后两个线程通过 CAS 竞争轻量级锁。

③

轻量级锁的自旋与膨胀

加锁流程：线程在栈帧中创建 Lock Record，CAS 将 Mark Word 复制到 Lock Record 并将 Mark Word 替换为指向 LR 的指针。

CAS 成功 → 获得轻量级锁，进入临界区。

CAS 失败 → 说明有竞争。开始自适应自旋（Adaptive Spinning）：
• 自旋次数由 JVM 根据历史成功率动态调整
• 上次自旋成功 → 本次多转几圈
• 上次自旋失败 → 本次少转或不转

自旋仍失败 → 锁膨胀为重量级锁。

④

轻量级锁 → 重量级锁（膨胀）

触发条件：自旋超过阈值（或自适应自旋判定继续自旋无意义），或者已经有第三个线程来竞争（轻量级锁只支持两方竞争）。

操作：
① 向 OS 申请 ObjectMonitor 对象
② 将 Mark Word 中的 Lock Record 指针替换为 ObjectMonitor 指针，tag 改为 10
③ 将原来 Lock Record 中的 Displaced Mark Word 保存到 ObjectMonitor._header
④ 当前线程进入 ObjectMonitor._EntryList，调用 park() 挂起

代价：涉及内核态切换、线程上下文保存/恢复。一次 park/unpark ≈ 1~10μs，比轻量级锁慢 100~1000 倍。

偏向锁撤销：Safepoint 里到底做了什么？

为什么撤销偏向必须暂停持有者线程？

偏向锁的核心优化是持有者线程读写 Mark Word 时不做任何同步。 这意味着 Mark Word 中的 Thread ID 可能被持有者随时读取作为"我拥有这把锁"的证据。 如果另一个线程直接 CAS 修改了 Mark Word，而持有者恰好在同一时刻读取它， 就会产生数据竞争（data race），导致持有者错误地认为自己仍持有锁。 因此，必须在持有者不在执行任何代码时才能安全修改 Mark Word——这就是 Safepoint。

偏向撤销的完整时序

1
请求撤销的线程发起 VM Operation

线程 B 发现锁被偏向于线程 A，向 VMThread 提交一个 VM_RevokeBias 操作。 该操作被放入 VMThread 的任务队列。

2
VMThread 触发全局 Safepoint（STW）

VMThread 设置 safepoint 标志 → 所有应用线程在执行到下一个安全点时暂停。 注意：这里暂停的是所有线程，不仅仅是线程 A。 这是偏向锁最大的隐性代价。

3
遍历线程 A 的栈帧

VMThread 检查线程 A 是否仍然存活且在运行。然后遍历线程 A 的所有栈帧：

  • • 查找所有正在使用该对象的 monitorenter/monitorexit 块
  • • 如果线程 A 的某个栈帧仍持有该对象的偏向锁 → 将该锁升级为轻量级锁（在栈帧中创建 Lock Record，复制 Mark Word）
  • • 如果线程 A 没有在栈帧中使用该对象 → 直接将 Mark Word 重置为无锁状态
4
更新 Epoch + 恢复所有线程

递增该类的 prototype_header 中的 Epoch 值。 这样即使有其他对象仍带有旧的 Thread ID，下次访问时也会因 Epoch 不匹配而自动失效（批量重偏向优化）。 最后解除 Safepoint，所有线程恢复执行。

HotSpot VM · BiasedLocking::revoke_at_safepoint (简化)
void BiasedLocking::revoke_at_safepoint(Handle obj) {
  // 前提：已在 Safepoint，所有 Java 线程已暂停

  markWord mark = obj->mark();
  JavaThread* biased_thread = mark->biased_locker();

  if (biased_thread != NULL && biased_thread->is_alive()) {
    // 遍历持有者线程的所有栈帧
    for (StackFrameStream fst(biased_thread); !fst.is_done(); fst.next()) {
      if (fst.current()->is_locked_for(obj)) {
        // 该栈帧仍持有偏向锁 → 升级为轻量级
        upgrade_to_lightweight(fst.current(), obj);
        return;
      }
    }
  }

  // 持有者已不在使用该对象 → 直接重置为无锁
  obj->set_mark(mark->clear_bias());

  // 递增 Epoch，使同类其他对象的旧偏向自动失效
  klass->prototype_header()->incr_biased_epoch();
}

电商场景：为什么高争抢锁必须关闭偏向锁？

🔥 问题：偏向撤销引发的 STW 风暴

电商核心链路（库存扣减、订单创建、支付回调）通常有以下特征：

  • •
    多线程高频竞争同一把锁
    例如 Redisson 本地缓存的 synchronized 保护、连接池的获取/归还、序列化器的共享 buffer
  • •
    对象生命周期短但复用频繁
    线程池中的任务对象被不同 worker 线程交替处理
  • •
    对 P99/P999 延迟极度敏感
    大促期间 SLA 要求 TP99 < 50ms

灾难链条：
对象被线程 A 偏向 → 线程 B 竞争 → 触发偏向撤销 → 全局 Safepoint STW → 所有 200+ 业务线程暂停 → 遍历线程 A 栈帧 → STW 持续 5~50ms → P99 毛刺

更糟的是：如果多个对象同时发生偏向撤销， STW 会叠加，形成毫秒级的延迟尖峰。

✅ 解决方案与权衡

方案一：关闭偏向锁（推荐）
-XX:-UseBiasedLocking

跳过偏向阶段，直接进入轻量级锁竞争。 牺牲了单线程无竞争的 ~2ns 优势， 但彻底消除了偏向撤销的 STW。 对于高竞争场景，这个 trade-off 稳赚不赔。

方案二：延迟偏向 + 批量重偏向
-XX:BiasedLockingStartupDelay=10000
-XX:+OptimizeStringConcat

增大启动延迟，让热身在偏向生效前完成。 利用 Epoch 机制实现批量重偏向，减少逐个撤销的次数。 治标不治本，适合中等竞争场景。

方案三：升级到 JDK 15+

JDK 15 (JEP 374) 默认禁用偏向锁。 JDK 18+ 完全移除相关代码。 长期来看这是唯一正确答案。

📊 真实线上数据：关闭偏向锁前后对比

指标 开启偏向锁 关闭偏向锁 变化
TP99 45ms 12ms ↓ 73%
TP999 380ms 28ms ↓ 93%
Safepoint 次数/min 120 8 ↓ 93%
平均 Safepoint 耗时 12ms 0.3ms ↓ 97%
吞吐量 (QPS) 12,400 12,800 ≈ 持平

数据来源：某电商平台订单服务压测报告，JDK 11，32C64G，200 线程池。 关闭偏向锁后吞吐量几乎不变，尾延迟大幅改善。

🏁 终极总结

偏向锁是一个为"单线程反复进入"设计的极致优化， 但它的前提假设在现代服务端应用中经常不成立。 它的撤销代价（全局 STW + 栈帧遍历）远大于它在无竞争时省下的那几次 CAS。

轻量级锁才是真正的主力。 配合自适应自旋，它在低中度竞争下避免了内核态切换， 而 CAS + Lock Record 的设计让它不需要额外的堆分配。

重量级锁是最后的兜底。 ObjectMonitor 的 EntryList/CXQ/_owner 三件套保证了公平性和正确性， 但 park/unpark 的微秒级代价意味着：如果你的代码频繁走到这一步，该优化的不是锁，而是你的架构。

💡 实践法则： JDK < 15 的服务端应用，启动参数加上 -XX:-UseBiasedLocking。 JDK ≥ 15 无需操心。永远不要因为"教科书说偏向锁快"就保留它—— 那是为桌面 GUI 单线程事件循环设计的，不是为你的 200 线程电商服务设计的。
