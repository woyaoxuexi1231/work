ThreadPoolExecutor · Unbounded Queue Trap · Pre-crash Monitoring

别再背八股了：
线程池在生产环境是怎么崩的

面试官问你"核心线程→队列→最大线程→拒绝策略"的流程，你倒背如流。 但线上真正出事故时，问题从来不是"流程记错了"，而是参数配对了但语义理解错了。 本篇用真实故障视角重讲交互流程，拆解无界队列下 maxPoolSize 的荒谬性， 最后给出一套在 OOM 之前就能救命的监控方案。

1

提交流程的"真实版"：不是流程图，是决策树

教科书画的是线性流水线。但 execute() 源码里是一个三层 if-else 决策树， 每一层的失败才会触发下一层。理解这个"失败驱动"的逻辑，才能理解为什么参数顺序如此反直觉。

workerCount < corePoolSize?
YES → addWorker(core=true)
workQueue.offer(task)?
true → 入队成功，结束
workerCount < maxPoolSize?
YES → addWorker(core=false)
reject(task) 💥

🟢 第一层：创建核心线程

注意：这里用的是 < 不是 <=。 当 workerCount == corePoolSize 时直接跳过，进入第二层。

实战经验：addWorker 内部有双重 CAS（ctl + workerSet）， 高并发提交时大量线程会 CAS 失败然后重试。这不是瓶颈，但会产生短暂的 CPU 尖峰。

🔵 第二层：尝试入队（关键转折点）

这是整个流程最反直觉的地方： 只有当队列满了（offer 返回 false）时，才会走到第三层创建非核心线程。

这意味着：只要队列还有空间，即使当前活跃线程数远小于 maxPoolSize，也不会创建新线程。 这就是为什么"有界队列 + 合理的 maxPoolSize"才是正确搭配——队列满得越快，非核心线程创建得越早。

🟣 第三层：创建非核心线程

走到这里说明两件事同时成立：
① 队列已满（或 offer 失败）
② 当前线程数还没到上限

实战经验：非核心线程的空闲存活时间由 keepAliveTime 控制。 在高波动负载下，这些线程会被反复创建和销毁，带来额外的 GC 压力。 如果波动频繁，考虑适当增大 corePoolSize 而非依赖弹性。

🔴 第四层：拒绝策略

走到这里意味着系统已经过载。 AbortPolicy 抛异常只是表象，真正的信号是：你的容量规划出了问题。

生产建议：永远不要用默认的 AbortPolicy。 至少用 CallerRunsPolicy 做背压，或者自定义策略记录指标 + 降级。

2

无界队列下 maxPoolSize 还有意义吗？

答案：完全没有意义。它是一个死参数。

回顾决策树第二层：workQueue.offer(task)。 LinkedBlockingQueue / ArrayBlockingQueue(cap=Integer.MAX_VALUE) 的 offer() 永远不会返回 false。 因此执行流永远走不到第三层，maxPoolSize 形同虚设。

🤡

可笑配置解剖

// 某项目线上真实配置（已脱敏）
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    10,                          // corePoolSize
    20,                          // maxPoolSize ← 写了个寂寞
    60, TimeUnit.SECONDS,         // keepAliveTime ← 也是寂寞
    new LinkedBlockingQueue<>()   // 默认容量 Integer.MAX_VALUE ≈ 21亿
);
这个配置实际等价于什么？
  • ①
    永远只有 10 个线程干活
    无论提交多少任务，线程数永远不会超过 corePoolSize=10。maxPoolSize=20 这行代码从未被执行过。
  • ②
    keepAliveTime=60s 完全无效
    因为没有非核心线程会被创建，所以没有线程会因为空闲而被回收。
  • ③
    拒绝策略永远不会触发
    队列容量 21 亿，按每秒 1 万任务算，需要 59 小时才能填满。在此之前 JVM 早就 OOM 了。
灾难链条

T+0s: 下游服务变慢，任务处理时间从 10ms → 500ms

T+5s: 10 个线程全部阻塞，新任务开始堆积。队列以 2000/s 速度增长

T+30s: 队列积压 6 万任务。每个任务持有请求上下文 ~2KB → 堆内存额外占用 120MB

T+2min: 队列 24 万任务 → 480MB。GC 频率飙升，STW 增加 → 处理更慢 → 恶性循环

T+5min: Full GC 无法回收（任务仍被引用）→ OutOfMemoryError → 进程崩溃 💀

✅ 正确做法
// 方案 A：有界队列 + 合理 max（推荐）
new ThreadPoolExecutor(10, 20, 60, SECONDS,
    new ArrayBlockingQueue<>(1000),
    new CallerRunsPolicy());  // 队列满时调用者自己干，天然背压

// 方案 B：如果确实需要缓冲大量任务
new ThreadPoolExecutor(10, 20, 60, SECONDS,
    new ResizableCapacityLinkedBlockingQueue<>(5000),  // 自定义可动态调整容量的队列
    new LoggingRejectPolicy());  // 拒绝时打日志+指标，不丢任务

为什么这种配置如此普遍？

1. Executors 工厂方法的毒害 Executors.newFixedThreadPool(n) 底层就是 core=n, max=n, LinkedBlockingQueue(∞)。 很多人复制粘贴后改了 core/max 却没改队列。
2. "怕拒绝"的心理 开发者担心有界队列导致任务被拒，于是用无界队列"保险"。 殊不知把"显式拒绝"变成了"隐性 OOM"——后者更难排查、破坏力更大。
3. 不理解决策树顺序 以为 maxPoolSize 是"队列满了之后的兜底"， 实际上它是"队列满了之后才有机会生效的参数"。 无界队列让这个前提永远不成立。
3

崩溃前感知：线程池监控的正确设计

大多数团队只监控"活跃线程数"和"队列大小"。这两个指标太滞后了—— 当你看到队列暴涨时，OOM 可能只剩 30 秒。你需要的是领先指标。

📡

第一层：领先指标（提前 30s~5min 预警）

这些指标在队列明显堆积之前就会发出信号

① 任务排队等待时间 (Queue Latency)

在任务包装器中记录 submitTimestamp， 线程开始执行时计算 now - submitTimestamp。

告警阈值：P99 > 500ms → WARNING | P99 > 2s → CRITICAL
为什么领先：即使队列绝对值不大，等待时间突增说明处理能力已跟不上提交速率。

② 队列增长率 (Queue Growth Rate)

ΔqueueSize / Δtime，每秒采样。

告警阈值：连续 3 个采样周期 > 0 → WARNING
为什么领先：队列绝对值 500 不可怕，但如果每秒涨 200，10 秒后就 2500。 增长率是"加速度"，比"位置"更早反映趋势。

③ 线程饱和度 (Saturation Ratio)

activeCount / maximumPoolSize

告警阈值：> 80% 持续 10s → WARNING | > 95% → CRITICAL
为什么领先：所有线程都在忙 = 下一个任务必然排队。 这是队列增长的前置条件。

④ 拒绝次数 / 背压触发率

自定义 RejectedExecutionHandler 中计数。

告警阈值：> 0 → CRITICAL（拒绝本身就是事故）
为什么重要：如果你用了 CallerRunsPolicy，没有拒绝但有背压， 需要监控调用者线程的执行耗时作为替代指标。

第三层：JVM 上下文（区分"线程池问题"还是"全局问题"）

Heap Usage % 队列堆积 + 堆使用率高 → 确认是内存风险。
队列堆积 + 堆正常 → 可能是下游超时，不是内存问题。
GC Pause Time STW 期间所有线程暂停 → 恢复后任务瞬间堆积。
这种堆积是自愈的，不应触发 CRITICAL 告警。
Thread Count (JVM) 如果多个线程池同时饱和 → 全局性问题（如下游挂了）。
单个池饱和 → 局部问题（如某个接口流量突增）。
MonitorableThreadPool.java · 可观测线程池封装
public class MonitorableThreadPool extends ThreadPoolExecutor {

    private final MeterRegistry registry;
    private final Timer queueLatencyTimer;
    private final AtomicLong rejectCount = new AtomicLong();

    public MonitorableThreadPool(..., MeterRegistry registry) {
        super(...);
        this.registry = registry;
        this.queueLatencyTimer = Timer.builder("pool.queue.latency")
            .register(registry);

        // 注册 Gauge（实时采集，非推送）
        Gauge.builder("pool.saturation", this,
            p -> (double) p.getActiveCount() / p.getMaximumPoolSize())
            .register(registry);

        Gauge.builder("pool.queue.size", this,
            p -> p.getQueue().size())
            .register(registry);
    }

    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        if (r instanceof TimedTask tt) {
            // ✅ 领先指标：排队等待时间
            queueLatencyTimer.record(
                System.nanoTime() - tt.submitNanos, NANOSECONDS);
        }
        super.beforeExecute(t, r);
    }

    @Override
    public void execute(Runnable command) {
        // 包装任务，注入提交时间戳
        super.execute(new TimedTask(command, System.nanoTime()));
    }

    // ✅ 自定义拒绝策略：计数 + 降级
    public class AlertingRejectPolicy implements RejectedExecutionHandler {
        @Override
        public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
            rejectCount.incrementAndGet();
            log.error("Pool rejected! active={} queue={} size={}",
                e.getActiveCount(), e.getQueue().size(), e.getPoolSize());
            // 降级：CallerRuns 或丢弃并记录
            if (!e.isShutdown()) r.run();
        }
    }
}

🚨 分级告警策略示例

级别 触发条件 动作 响应时间
INFO saturation > 60% 持续 30s Dashboard 标黄，Slack 通知 观察即可
WARNING queueLatency P99 > 1s 或 growthRate > 0 连续 5 周期 On-call 群消息 + 自动扩容（若支持） < 5min

🏁 工程守则

关于流程：不要背"核心→队列→最大→拒绝"的口诀。 记住它是失败驱动的决策树：每一层都是因为上一层"搞不定"才触发的。 理解了这一点，参数的含义自然清晰。

关于无界队列： core=10, max=20, LinkedBlockingQueue() 是一个伪装成弹性池的固定大小池 + 定时炸弹。 max 和 keepAlive 是死代码，OOM 是唯一终点。 如果你在线上看到了这个配置，现在就去改。

关于监控： 队列大小是"车速表"，排队等待时间和增长率才是"加速度计"。 等看到车速表爆表再踩刹车就晚了——你需要在加速阶段就感知到危险。 好的监控不是在崩溃后告诉你"它崩了"，而是在崩溃前告诉你"它快要崩了"。

💡 一句话总结： 线程池的参数决定了它的"形状"，队列类型决定了它的"命运"， 而监控决定了你能不能在它走向命运之前把它拉回来。
