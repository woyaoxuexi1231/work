AQS 的灵魂：
CLH 队列入队唤醒 & 共享传播机制

ReentrantLock 的公平/非公平、CountDownLatch 的一次性、Semaphore 的限流， 底层全靠 AQS 的 CLH 变体队列。本篇从节点级 CAS 操作画出入队唤醒全流程， 重点拆解共享模式下 doReleaseShared 如何像多米诺骨牌一样唤醒一整串节点， 最后用线上事故讲透 CDL 和 Semaphore 的工程陷阱。

ReentrantLock 下 CLH 队列的入队与唤醒

AQS 的 CLH 是一个FIFO 双向链表。每个节点封装了等待线程、模式（独占/共享）和前驱/后继指针。 以下是 ReentrantLock.lock() 失败后的完整入队 + park/unpark 流程：

📥 入队流程 (addWaiter + enq)

①
创建节点：new Node(EXCLUSIVE, thread)
②
快速尝试：CAS 将新节点设为 tail。若成功且原 tail 为 null（首次），同时初始化 head。
③
自旋入队：CAS 失败则进入 enq() 无限循环：
• 读取当前 tail → 设置 node.prev = tail
• CAS(tail, oldTail, node)
• 成功后设置 oldTail.next = node
⚠️ 注意：prev 在 CAS 前设置，next 在 CAS 后设置。这保证了从 tail 向前遍历总是安全的。
④
acquireQueued：入队后自旋检查前驱是否为 head。若是，再次 tryAcquire。成功则将自己设为新 head（清除 thread 引用帮助 GC）。否则 shouldParkAfterFailedAcquire → park()。

🔗 CLH 队列快照（3个线程等待）

HEAD thread=null 已获锁/哨兵
Node₁ T-A (EXCLUSIVE) ← pred=head ✓
Node₂ T-B (EXCLUSIVE) waitStatus=SIGNAL
TAIL T-C (EXCLUSIVE) 刚入队
💡 关键设计：head 是哨兵节点，不关联线程。 只有 head 的后继才有资格 tryAcquire。 其他节点通过 waitStatus=SIGNAL(-1) 告知前驱"释放时记得唤醒我"。

⏰ 唤醒流程 (unlock → unparkSuccessor)

①
release：tryRelease(state-1) 成功（state=0）
②
unparkSuccessor(head)：
• 将 head.waitStatus 置 0
• 从 tail 向前找第一个非 CANCELLED 节点 s
• LockSupport.unpark(s.thread)
③
被唤醒线程：回到 acquireQueued 自旋，发现 pred==head，tryAcquire 成功，setHead 将自己变为新 head。
private void unparkSuccessor(Node node) {
  // 清除 signal 状态
  int ws = node.waitStatus;
  if (ws < 0)
    compareAndSetWaitStatus(node, ws, 0);

  // 从 tail 向前找有效后继
  // ⚠️ 为什么从后往前？因为 next
  // 指针可能还没设置好（入队
  // CAS 成功但 next 赋值前）
  Node s = node.next;
  if (s == null || s.waitStatus > 0) {
    s = null;
    for (Node t = tail; t != null && t != node; t = t.prev)
      if (t.waitStatus <= 0) s = t;
  }
  if (s != null)
    LockSupport.unpark(s.thread);
}

共享模式的"传播唤醒"：为什么需要、JDK17 是否还在

🔬 关于 JDK17 的事实：

流传"JDK17 移除了传播唤醒"的说法是错误的。以下是 JDK17 AQS 中仍然存在的完整传播机制要素：

  • Node.PROPAGATE = -3 — 仍然在 Node 类中定义
  • doReleaseShared() — 仍然在 AQS 中，CAS 0→PROPAGATE 分支健在
  • setHeadAndPropagate(node, r) — 仍然检查 h.waitStatus < 0 来捕获 PROPAGATE

真正发生过的是：JDK6 早期版本中确实没有 PROPAGATE，这曾是 JDK-6801020 已知 bug。 JDK 6u18 及以上版本（包括 JDK8/11/17/21）都包含这个修复。不是"移除"，而是一直存在至今。

独占模式下，release 只唤醒一个后继。但在共享模式（Semaphore/CDL/ReadWriteLock.read）下， 一个 release 可能需要唤醒一连串连续的共享节点。这就是"传播"机制。 但传播的具体作用比直觉更微妙——它不是为了"多个节点同时获取许可"（那是 Semaphore 本身的语义）， 而是为了解决时序竞争导致唤醒丢失的问题。

💀 没有 PROPAGATE 时的经典 Bug：唤醒丢失

场景：Semaphore(1)，队列 [S₁(SIGNAL) → S₂(SIGNAL)]，state=0

T1
release(1) → state=1 → doReleaseShared → CAS head ws=SIGNAL→0 → unpark S₁
S₁
被唤醒 → tryAcquireShared(1) → remaining=0 → setHeadAndPropagate(node, r=0)
T2
同时 release(1) → state=1 → doReleaseShared
→
T2 的 doReleaseShared 读 head — 此时 head 已被 S₁ 设为新节点（S₁ 的旧节点），
新 head.waitStatus = 0（S₁ 还没设置 SIGNAL），
T2 发现 ws == 0，早期 JDK 直接 break 了！→ 唤醒信号丢失！
→
同时 S₁ 的 setHeadAndPropagate 检查 propagate=0 → 也不传播。
❌ 结果：S₂ 永远 park，即使有可用许可也没人唤醒它！

🔧 修复前的 setHeadAndPropagate（JDK6 早期）

// ❌ 只有 propagate > 0 才传播
private void setHeadAndPropagate(Node node, int propagate) {
  setHead(node);
  if (propagate > 0) {
    Node s = node.next;
    if (s == null || s.isShared())
      doReleaseShared();
  }
}

// 当最后一个许可被消费（propagate=0），
// 并发 release 的信号就在这里丢失。

🔧 修复后（JDK6u18+ ~ JDK17）

private void setHeadAndPropagate(Node node, int propagate) {
  Node h = head;  // 记录旧 head
  setHead(node);
  if (propagate > 0 || h == null || h.waitStatus < 0
      || (h = head) == null || h.waitStatus < 0) {
    Node s = node.next;
    if (s == null || s.isShared())
      doReleaseShared();
  }
}

setHeadAndPropagate — JDK17 源码逐行分析

private void setHeadAndPropagate(
    Node node, int propagate) {
  // 🅐 记录旧 head，用于后续检查
  Node h = head;
  // 🅑 将当前节点设为新 head（清除 thread）
  setHead(node);

  // 🅒 以下任一条件满足时继续传播：
  if (propagate > 0        // 1) 还有剩余许可
      || h == null
      || h.waitStatus < 0   // 2) 旧 head 被标记了 SIGNAL/PROPAGATE
      || (h = head) == null
      || h.waitStatus < 0) { // 3) 新 head 也被标记了
    Node s = node.next;
    if (s == null || s.isShared())
      doReleaseShared();
  }
}

条件 ① propagate > 0：tryAcquireShared 返回正数——还有剩余许可。这是最直接的"有资源可分配"信号。

条件 ② 旧 head.waitStatus < 0：旧 head 被标记为 SIGNAL 或 PROPAGATE，说明有并发 release 已经发生或正在发生，需要继续传播。

条件 ③ 新 head.waitStatus < 0：在 setHead 到检查之间，又有新的 release 发生了，另一个线程 CAS 了新 head 的状态。

这些条件共同保证了"只要有任何可能还有未处理的 release 信号，就继续传播"。

doReleaseShared — JDK17 源码逐行分析

private void doReleaseShared() {
  for (;;) {
    Node h = head;
    if (h != null && h != tail) {
      int ws = h.waitStatus;
      if (ws == SIGNAL) {
        // 🅐 有后继在等 → 清除 SIGNAL 并唤醒
        if (!compareAndSetWaitStatus(
            h, SIGNAL, 0))
          continue;
        unparkSuccessor(h);
      }
      else if (ws == 0 &&
        !compareAndSetWaitStatus(
            h, 0, PROPAGATE))
        // 🅑 无后继在等，但标记 PROPAGATE
        // 告诉将来被唤醒的节点："继续传播"
        continue;
    }
    if (h == head) break;
    // 🅒 head 变了 → 有节点被唤醒并设头
    // → 继续循环传播！
  }
}

🅐 SIGNAL 分支：最常见的路径。CAS 将 SIGNAL 清 0，然后 unparkSuccessor 唤醒后继。这是实际的唤醒动作。

🅑 PROPAGATE 分支：核心修复。如果 head 已经是 0（刚被清过 SIGNAL），CAS 设为 PROPAGATE(-3)。这个标记会被 setHeadAndPropagate 捕捉到，作为"必须继续传播"的信号。

🅒 循环检查：如果 head 变了（被唤醒的线程设为新 head），继续 loop 传播。

🎯 为什么 Semaphore 特别需要传播？——从 JDK17 源码角度

// JDK17 Semaphore.Sync.tryReleaseShared
// ⚠️ 注意：永远返回 true！
protected final boolean tryReleaseShared(int releases) {
  for (;;) {
    int current = getState();
    int next = current + releases;
    if (next < current)
      throw new Error("Maximum permit count exceeded");
    if (compareAndSetState(current, next))
      return true;  // ← 无条件 true！
  }
}

对比 CountDownLatch：

CountDownLatch 的 tryReleaseShared 只在 state 到 0 时返回 true，只触发一次 doReleaseShared。 那次 doReleaseShared 需要唤醒所有等待者——所以必须传播。

Semaphore 有所不同：

每次 release() 都返回 true，每次都调用 doReleaseShared。 所以每个 release 都会至少唤醒一个等待者。

传播的需求在这里更微妙——是因为并发 release 的时序交错： 线程 A release(1) 唤醒 S₁，S₁ 醒来取走最后一个许可。 线程 B 同时 release(2)——它的信号必须通过 PROPAGATE 接力给 S₂、S₃。

关键观察：当 S₂ 醒来但拿不到许可时，它不会调用 setHeadAndPropagate——所以 传播在第一次 tryAcquireShared 返回负值时立即停止。这意味着 HTML 旧版本中"S₂ 虽然拿不到许可但仍传播给 S₃"的场景在有 JDK6u18+ 修复后不会发生。 真正的传播只发生在获取成功的线程的 setHeadAndPropagate 中， 或者新 release 触发的 doReleaseShared 中。

⚡

传播唤醒：辟谣与本质

✅ 传播的真实作用

  • 防止唤醒丢失：并发 release 时，一个释放信号可能被另一个线程的 setHead 掩盖。PROPAGATE 标记确保信号不被丢弃。
  • 多许可级联：release(5) 的 5 个许可需要唤醒多个等待线程。在 doReleaseShared 的循环中逐个唤醒。
  • 独占节点阻断：传播遇到独占节点 (isShared()=false) 自动停止——这是 ReadWriteLock 的重要设计。

❌ 常见的误解

  • 误解："传播让所有共享节点都拿到许可" — 错。每个节点拿到许可后必须通过 setHeadAndPropagate 主动触发下一轮传播。拿不到的节点直接 park 回去。
  • 误解："JDK17 移除了传播" — 错。PROPAGATE(-3)、doReleaseShared、setHeadAndPropagate 在 JDK17 中全部存在。
  • 误解："Semaphore 不需要传播" — 错。Semaphore 的 tryReleaseShared 永远返回 true，每次 release 都触发 doReleaseShared，正是传播的核心使用者。

🔄 传播唤醒的完整调用链（JDK17）

→ r < 0? (没许可)
→ park 回去，不传播

CountDownLatch 陷阱：不可重用与虚假完成

💥

陷阱一：不可重用

CDL 的 state 从 N 递减到 0 后就永久停留在 0。 之后所有 await() 立即返回，countDown() 变成空操作。 没有 reset() 方法。 

// ❌ 错误：复用 CDL 做循环屏障
CountDownLatch cdl = new CountDownLatch(3);
for (int round = 0; round < 10; round++) {
    submitTasks(cdl);  // 第2轮起 await 秒过
    cdl.await();       // ⚠️ 不再阻塞！
}

✅ 修复：需要可重用屏障请用 CyclicBarrier。 它内部有 generation 概念，每轮完成后自动重置。

💥

陷阱二：异常吞没导致永久阻塞

如果某个子任务抛异常而未执行 countDown()， 主线程的 await() 将永远阻塞。 CDL 不感知任务成功/失败，只计数。 

// ✅ 安全写法：始终在 finally 中 countDown
executor.submit(() -> {
    try { process(data); }
    finally { cdl.countDown(); }
});

✅ 最佳实践：始终在 finally 中 countDown。 或使用带超时的 await(timeout, unit) 兜底。

⚠️

陷阱三：countDown 与 await 的顺序竞态

如果在提交任务之前就调用了 await，或者任务执行极快在主线程 await 之前就全部 countDown 完毕， 逻辑上没问题（await 会立即返回）。但如果 count 初始化为 0，await 永远不阻塞——这可能是 bug 而非预期行为。

💡 调试技巧：生产环境建议封装 CDL，记录 countDown 调用栈和 await 超时信息， 便于排查"为什么 latch 没拦住"或"为什么 latch 卡住了"。

Semaphore 陷阱：为什么不能替代 CyclicBarrier？

🧬 JDK17 Semaphore 源码分析：为什么 tryReleaseShared 永远返回 true？

// Semaphore.Sync (JDK17)
protected final boolean tryReleaseShared(int releases) {
  for (;;) {
    int current = getState();
    int next = current + releases;
    if (next < current) // 溢出检查
      throw new Error("Maximum...");
    if (compareAndSetState(current, next))
      return true; // ← 区别于 CDL！
  }
}

// 为什么要永远返回 true？
// 因为 Semaphore 的 release 语义是"归还许可"
// 不论有没有线程在等，release 都应该增加许可计数
// 而不像 CDL 只有最后一次 countDown 才有意义

这对传播链的影响：

每次 release 都走 releaseShared → doReleaseShared 全路径。 所以 Semaphore 的传播链被每一个 release 调用触发，而不是只在最后一次。

对比 CountDownLatch：

CDL 的 tryReleaseShared 只在 state→0 时返回 true，一次 doReleaseShared 需要把队列里所有等待者都唤醒。

Semaphore 的 tryReleaseShared 每次返回 true，每次 release 都触发 doReleaseShared，逐个唤醒。

核心区别：信号量 ≠ 屏障

Semaphore 的语义
  • • 许可计数器：控制同时访问资源的线程数
  • • acquire/release 完全解耦：不要求配对，不要求同一线程
  • • 无"轮次"概念：没有 generation，无法区分"这一批"和"下一批"
  • • 允许超额释放：release(100) 合法，permits 可以超过初始值
CyclicBarrier 的语义
  • • 同步点：N 个线程必须全部到达才能继续
  • • await 是原子集合操作：要么全过，要么全等
  • • 有 generation：每轮完成后自动重置，支持循环使用
  • • 支持 barrierAction：所有线程到齐后执行一次汇总逻辑

💥 用 Semaphore 模拟 Barrier 的经典翻车

// ❌ 试图用 Semaphore 做 3 线程屏障
Semaphore sem = new Semaphore(0);

// 每个 worker:
doWork();
sem.release();    // 签到
sem.acquire(3);   // 等齐3个
sem.release(3);   // 放行所有人
doNextPhase();

问题 1：竞态条件

T1 release → permits=1
T2 release → permits=2
T3 release → permits=3
T1 acquire(3) 成功 → T1 独自通过！T2/T3 永远阻塞

问题 2：第二轮污染

release(3) 后 permits=3。第二轮线程提前 release 导致 permits 累积到 6→acquire(3) 不等齐就通过。

✅ 正确做法：直接用 CyclicBarrier(3)。

⚠️ 泄漏陷阱

acquire 后未在 finally 中 release → 许可永久减少 → 最终所有线程阻塞。 比 ReentrantLock 更危险，因为 Semaphore 没有 owner 概念， JVM 无法检测"谁持有了许可但没还"。

⚠️ 公平性陷阱

默认非公平模式下，刚 release 的许可可能被刚到的线程抢走， 而非给队列头部等待最久的线程。高负载下可能导致线程饥饿。 对延迟敏感的场景务必 new Semaphore(n, true)。

🏁 终极总结

CLH 队列的精妙在于：prev 先于 CAS 设置保证反向遍历安全， next 后于 CAS 设置避免读到未初始化节点。唤醒时从 tail 向前找后继正是为了应对这个时序窗口。

共享传播通过 PROPAGATE 状态和 setHeadAndPropagate 的循环检查， 解决了"唤醒丢失"这个困扰早期并发库的经典难题。代价是偶尔的无效唤醒（spurious wakeup）， 但正确性优先于性能——这是 Doug Lea 一贯的设计哲学。

关于 JDK17 的澄清：传播机制（PROPAGATE(-3)、doReleaseShared、setHeadAndPropagate）在 JDK17 中完整保留。 该机制最初在 JDK6u18 中引入以修复 JDK-6801020（共享模式唤醒丢失 bug），后续所有版本包括 JDK17 和 JDK21 均继承此修复。 所谓"JDK17 移除传播"是一个流传的误解。

CountDownLatch 是一次性的倒计时器，不是屏障。 Semaphore 是许可计数器，不是同步点。 混淆它们的语义是并发编程中最常见的工程事故之一。 记住：选错工具比不用工具更危险，因为它看起来"能跑"，直到某天在高负载下悄然崩塌。
