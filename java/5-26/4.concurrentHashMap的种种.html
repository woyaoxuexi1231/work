为什么放弃 Segment？

1. 锁粒度不够细 Segment 数量在构造时确定（默认16），即使数据量增长到百万级，并发写入上限仍然是 16。JDK8 的锁粒度跟随数组大小，理论上并发度可达数万。
2. 二次哈希开销 JDK7 需要先 hash 定位 Segment，再 hash 定位桶。JDK8 只需一次 hash 直接定位桶，减少了 CPU 指令。
3. 数据结构统一 JDK8 的 CHM 与 HashMap 底层结构一致（Node+链表+红黑树），代码复用度高，且能享受 synchronized 在 JVM 层面的持续优化红利。

synchronized "足够轻量" 是谁保证的？—— JVM 锁升级机制

JDK 6 之前，synchronized 是"重量级锁"，每次加锁都要操作系统互斥量（Mutex），涉及用户态↔内核态切换。 JDK 6 引入了锁升级（Lock Escalation），由 HotSpot VM 的 ObjectMonitor 子系统保证： 在低竞争场景下，synchronized 的性能接近甚至优于 ReentrantLock。

为什么 CHM 特别适合轻量级锁？

CHM 的 synchronized 只锁单个桶的头节点。 在高并发但 key 分布均匀的场景下，多个线程几乎不会竞争同一个桶。 这意味着绝大多数加锁操作都停留在偏向锁或轻量级锁阶段， 永远不会膨胀为重量级锁。这正是 Doug Lea 敢于用 synchronized 替代 ReentrantLock 的底气。

ReentrantLock 为什么不行了？

ReentrantLock 基于 AQS，始终需要维护一个 volatile state + CLH 队列。 即使在无竞争时也有 CAS 操作和对象分配开销。 而 synchronized 的偏向锁在稳态下连 CAS 都不需要， 只是一次普通的内存读取比较线程 ID。对于 CHM 这种"大量短临界区"的模式，优势明显。

size() 的进化：从 Segment 求和到 LongAdder 思想

JDK 7: 三次不加锁尝试 + 全段加锁兜底

// 尝试 3 次不加锁求和
for (int k = 0; k < RETRIES_BEFORE_LOCK; k++) {
    sum = 0;
    for (Segment seg : segments)
        sum += seg.count;  // volatile read
    if (modCount == lastModCount)
        return sum;        // 一致则返回
}
// 3 次都不一致 → 锁全部 Segment
for (Segment seg : segments) seg.lock();
sum = ...;
for (Segment seg : segments) seg.unlock();

问题：高并发写入时 modCount 永远在变， 3 次尝试几乎必定失败，最终退化为锁住所有 Segment， 相当于全局锁，吞吐量断崖式下降。

JDK 8: baseCount + CounterCell[]

public int size() {
    long n = sumCount();
    return ((n < 0L) ? 0 :
            (n > Integer.MAX_VALUE) ? Integer.MAX_VALUE :
            (int)n);
}

final long sumCount() {
    CounterCell[] as = counterCells;
    long sum = baseCount;  // 基础值
    if (as != null) {
        for (CounterCell a : as)
            if (a != null)
                sum += a.value; // 累加各cell
    }
    return sum;
}

优势：借鉴 LongAdder 分段累加思想。 写入时分散竞争到不同 Cell，读取时完全无锁遍历求和。 弱一致性，但永不阻塞。

addCount() 写入路径：CAS 竞争感知 + 动态扩容 Cell 数组

①
快速路径：CAS 更新 baseCount。如果成功，结束。 低竞争时只有这一次原子操作。
②
竞争检测：CAS 失败说明有竞争。检查 counterCells 是否已初始化。 若未初始化，通过 cellsBusy CAS 创建初始大小为 2 的数组。
③
分散写入：用 ThreadLocalRandom.getProbe() & (n-1) 随机选一个 Cell 做 CAS 累加。失败则换下一个 Cell 重试。
④
自适应扩容：如果连续失败，说明 Cell 数量不够。 将 counterCells 数组扩容为 2 倍（最大不超过 NCPU），进一步降低冲突概率。

CounterCell 的伪共享防护：从手动填充到 @Contended

什么是伪共享（False Sharing）？

CPU 缓存以 Cache Line（通常 64 字节） 为单位读写。 如果两个线程频繁读写的变量恰好落在同一条 Cache Line 上， 即使它们操作的是不同变量，也会导致缓存行在两个核之间反复传递（称为 Cache Line Bouncing）。

CounterCell 只有一个 long value（8字节）。如果不做处理， 数组中相邻的 8 个 Cell 会挤在同一条 Cache Line 里， 多线程分别更新不同 Cell 时依然会产生严重的伪共享。

JDK 7 LongAdder: 手动 Padding

hacky but effective 
// 通过继承链强制每个 Cell 占满一个 Cache Line
class Cell extends PaddedAtomicLong { ... }

class PaddedAtomicLong extends AtomicLong {
    volatile long p0, p1, p2, p3, p4, p5, p6; // 7×8=56 bytes padding
}
// AtomicLong.value (8B) + padding (56B) = 64B = 1 Cache Line
// ⚠️ 依赖 JVM 不重排字段顺序，非官方保证

JDK 8+: @sun.misc.Contended 注解

official JVM support 
@sun.misc.Contended
static final class CounterCell {
    volatile long value;

    CounterCell(long x) { value = x; }
}

// JVM 看到 @Contended 后自动在该对象前后插入 padding
// ✅ 不再依赖字段排列 hack
// ✅ 语义明确，编译器/JIT 不会优化掉 padding
// ⚠️ 需要 -XX:-RestrictContended 才能用于非 JDK 内部类

💡 关键细节：CounterCell 本身不是数组元素

counterCells 是一个引用数组（CounterCell[]）， 每个元素是指向堆上独立对象的引用。@Contended 作用于 CounterCell 对象本身， 确保每个 CounterCell 实例独占 Cache Line， 而不是让数组中的引用彼此隔离。这与 JDK7 中把 padding 字段直接嵌入数组元素的做法有本质区别。

🏁 一句话总结

JDK 7 的 Segment 是软件层面的分片思想，受限于固定并发度和二次哈希； JDK 8 将并发控制下沉到硬件友好的粒度—— CAS 解决无竞争路径，JVM 优化的 synchronized 解决低竞争路径， CounterCell + @Contended 解决统计路径的伪共享。

这不是简单的"换锁"，而是从算法设计到 JVM 运行时再到 CPU 缓存架构的全栈协同优化。 理解这三层联动，才算真正读懂了 ConcurrentHashMap。
