扩容时链表是怎么拆分的？
(e.hash & oldCap) == 0
到底利用了什么数学特性？

这行代码是 HashMap 扩容的灵魂。 它用一个位与操作替代了取模运算，将链表元素精确地分成"低位链"和"高位链"。 理解它，就理解了为什么 HashMap 的容量必须是 2 的幂。

1

核心数学：为什么 & oldCap 能替代 % newCap？

设旧容量 oldCap = 16，新容量 newCap = 32。 扩容后，一个元素在新表中的位置只有两种可能：

📍 位置不变（低位链）

新索引 == 旧索引

hash % 16 == hash % 32
⟺ hash 的第 4 位（oldCap 对应位）为 0

📍 位置 + oldCap（高位链）

新索引 == 旧索引 + 16

hash % 32 == hash % 16 + 16
⟺ hash 的第 4 位（oldCap 对应位）为 1

🔑 数学本质

因为容量始终是 2 的幂，hash % cap 等价于 hash & (cap - 1)。 当 cap 翻倍时，(newCap - 1) 比 (oldCap - 1) 恰好多了一个最高有效位。 这个新增位就是 oldCap 本身！ 所以 e.hash & oldCap 就是在检查这个"新增位"是 0 还是 1。

java.util.HashMap · resize() 核心拆分逻辑
// 遍历旧桶中的每个节点 e
if ((e.hash & oldCap) == 0) {
    // ✅ 低位链：新索引 == 旧索引
    if (loTail == null) loHead = e;
    else loTail.next = e;
    loTail = e;
} else {
    // ✅ 高位链：新索引 == 旧索引 + oldCap
    if (hiTail == null) hiHead = e;
    else hiTail.next = e;
    hiTail = e;
}

// 拆分完成后，直接挂到新表
newTab[j] = loHead;           // 低位链 → 原位置
newTab[j + oldCap] = hiHead;  // 高位链 → 原位置 + oldCap

负载因子调参：时间 vs 空间的残酷权衡

LF = 0.5

激进扩容
空间利用率≤ 50%
查询性能极佳
扩容频率频繁

一半空间永远浪费。适合对延迟极度敏感、数据量小且可预知的场景。

LF = 0.75

默认值 ✓
空间利用率≤ 75%
查询性能良好
扩容频率适中

时间与空间的最佳平衡点。Knuth 在 TAOCP 中证明开放寻址法在 α=0.75 时期望探测次数 ≈ 2。

LF = 2.0

极端节省
空间利用率可达 100%+
查询性能严重退化
扩容频率极少

链表平均长度翻倍，O(1) 退化为 O(n)。除非内存极度紧张且只写不读，否则不要用。

线上事故与调参实战

💥

事故：LF=2.0 导致接口 P99 从 5ms 飙到 800ms

某团队为节省内存将缓存服务的 HashMap LF 调到 2.0。上线后 GC 正常，但查询延迟飙升。 原因：平均链表长度接近 2，最坏情况超过 20。加上部分 key 的 hashCode 分布不均， 某些桶链表长达 50+，O(1) 彻底退化为 O(n)。

修复：改回 0.75，内存增加 4MB/实例，P99 恢复到 3ms。
教训：省下的内存不值钱，多花的 CPU 才贵。
✅

合理调参：已知数据量的批量导入场景

ETL 任务需要将 500 万条记录加载到内存做去重。数据一次性写入、写完才读。 使用 new HashMap<>(capacity, 1.0f)， 避免中间扩容，同时充分利用空间。

关键：这里调的不是 LF 本身，而是初始容量。
LF=1.0 只在"写入密集、读取稀疏、数据量已知"时才安全。
⚠️

陷阱：LF=0.5 并没有让查询快一倍

有人以为 LF 减半 = 冲突减半 = 速度快一倍。实际上，当 LF 从 0.75 降到 0.5， 期望比较次数只从 ~2.0 降到 ~1.5（提升 25%），但内存开销增加了 50%。 在现代 CPU 上，cache miss 的代价远大于多一次链表遍历。 更大的数组意味着更多的 cache miss，实际收益可能为零甚至为负。

5

为什么默认 0.75 没人动它？

🎯 理论最优区间

Knuth《TAOCP Vol.3》证明：对于链地址法，当负载因子 α ∈ [0.5, 0.75] 时， 成功查找的期望比较次数 ≈ 1 + α/2，失败查找 ≈ 1 + α²/2。 0.75 是这个区间内空间效率最高的端点。

对于开放寻址法（如 Python dict），0.75 更是硬性上限； Java 用链地址法可以容忍更高，但 0.75 已经是经验验证的安全边界。

🏭 工程惯性

  • • JDK 源码注释明确说 "this is a tradeoff between time and space costs"
  • • 几乎所有框架（Spring、MyBatis、Netty）都基于默认值做性能测试
  • • 改 LF 的收益不确定，但引入 bug 的风险确定
  • • 真正需要优化时，正确的做法是预设初始容量而非调 LF

⚖️ 终极建议

扩容拆分利用了"2 的幂容量下，翻倍只新增一个有效位"的数学特性， 用一次位与操作完成了原本需要取模才能做的分桶决策。这是 HashMap 高性能的基石之一。

负载因子是一个已经被理论和实践充分验证的参数。 除非你能精确描述你的场景为什么偏离了"通用假设"，并且有 benchmark 数据支撑， 否则不要修改它。

💡 如果你发现自己想调 LF，先问自己：我是不是应该改用 new HashMap<>(expectedSize * 4 / 3 + 1) 来避免扩容？ 99% 的情况下，这才是正确答案。
