### 一、集合——连环追问

**第一问：HashMap 红黑树与阈值**

引入红黑树是为了解决哈希冲突严重时链表过长导致查询退化为 O(n) 的问题。

- 为什么不用二叉搜索树？因为无序插入可能形成单边树，退化为链表。
- 为什么不用 AVL 树？AVL 追求严格平衡，插入、删除时的旋转开销比红黑树更高。红黑树的不严格平衡能兼顾插入和查询，代价更低。
- 树化阈值为什么是 8？源码注释说明，在负载因子 0.75 下，哈希冲突遵循泊松分布，链表长度达到 8 的概率低于千万分之一，此时树化可以在极低风险下有效防御恶意哈希攻击导致的性能退化。
- 退化阈值 6 是为了引入一个“缓冲区间”，避免元素个数在 8 附近频繁增删导致树与链表的反复转换，带来不必要的性能开销。

**第二问：扩容拆分与负载因子**

- `(e.hash & oldCap) == 0` 利用了 HashMap 容量始终为 2 的幂这一特性。扩容后容量翻倍，新的索引只比原来多一位高位判断。`oldCap` 恰是那个新增的最高位，用 `e.hash & oldCap` 检测该位是 0 还是 1：为 0 则留在原索引位置，为 1 则移至“原索引 + oldCap”位置。本质是将一条链表按哈希值的新增 bit 拆分为高链与低链。
- 负载因子调大（如 2）会节省空间但碰撞加剧，链表或红黑树增多，查询时间变长；调小（如 0.5）则扩容频繁，空间浪费且元素分布过散。默认 0.75 是大量实测得出的时空折中。线上我曾遇过预估对象量大且生命周期短的场景，手动调大负载因子和初始容量来避免频繁扩容引起的 STW（Stop-The-World）与 CPU 飙升，但必须谨慎评估。

**第三问：HashMap 多线程成环与数据覆盖**

- JDK 7 使用头插法扩容，多线程下 `resize` 时容易出现链表反转，如果两个线程同时操作同一条桶链表，会形成循环引用（a->b, b->a），后续 `get` 陷入死循环。用笔画出线程一挂起后线程二完成扩容、线程一恢复继续用旧链头插导致指向错误的过程。
- JDK 8 改用尾插法，不会出现链表成环。但并发问题依然存在：多个线程同时 `put` 时，`if ((p = tab[i = (n - 1) & hash]) == null)` 判断后直接赋值可能被覆盖，导致数据丢失；`size` 自增非原子也会导致计数不准，甚至触发不了扩容。

**第四问：ConcurrentHashMap 演进**

- JDK 7 采用分段锁（Segment），每个 Segment 继承 `ReentrantLock`，维护自己的哈希表，并发度取决于段数。
- JDK 8 放弃分段锁，改用 CAS + synchronized 锁桶的头节点。主因是：JDK 8 对 synchronized 进行了锁升级优化，性能已与 `ReentrantLock` 持平；去除 Segment 层级简化了数据结构，降低内存开销，扩容也可单桶迁移，并发度进一步提升。
- `size()` 方法在 JDK 8 采用 `baseCount` + `CounterCell` 数组分散并发更新。失败时线程会尝试 CAS 更新 `baseCount`，若仍失败则随机选择一个 `CounterCell` 累加，求和时累加 `baseCount` 与所有 `CounterCell` 的值。为解决 `CounterCell` 的伪共享问题，源码中使用 `@Contended` 注解或在前后填充 padding 字段，保证不同 `CounterCell` 不在同一缓存行。

**第五问：ArrayList 与 LinkedList 插入成本**

- `ArrayList` 尾插均摊 O(1)，扩容搬移为间歇性开销；中间插入 O(n)，但依靠 `System.arraycopy` 做批量内存块复制，利用 CPU 缓存行预取，常数时间远小于逐元素移动。缓存命中率高，因为底层是连续数组。
- `LinkedList` 头尾插入 O(1)（维护双端指针），但每次插入都要新建 Node，内存分配开销大且链表节点跳跃导致 CPU 缓存未命中；中间插入需要遍历，理论 O(n)，但寻址慢，实际性能往往不如 `ArrayList` 的 `arraycopy`。
- `System.arraycopy` 快在它是本地方法，直接访问内存，可使用向量化指令批量复制，避免 JVM 循环解释执行的开销。

### 二、多线程——压力测试

**第六问：synchronized 锁升级**

- 偏向锁：第一次获取时记录线程 ID，后续同一线程进入无需 CAS 加锁。当另一线程尝试获取时，JVM 到达安全点，暂停持有偏向锁的线程，检查其是否仍执行同步块，否则撤销偏向（变为无锁或轻量级锁）。升级过程：偏向锁 -> 轻量级锁（CAS 尝试获取，失败自旋）-> 重量级锁（自旋超时或冲突大，挂起线程）。
- 电商等高竞争场景，线程竞争频繁，偏向锁的撤销在安全点会产生 STW，且无谓消耗撤销代价，关闭偏向锁能减少延迟抖动，让锁直接进入轻量级竞争。

**第七问：AQS 队列与共享模式**

- CLH 队列变体：节点入队时设置前驱信号，`ReentrantLock` 独占模式下，获取锁失败的线程被包装成 Node 加入尾端，前驱节点释放时唤醒后继。唤醒流程：`unparkSuccessor` 从尾向前找到第一个有效节点。
- 共享模式：`tryAcquireShared` 返回负值则入队；`doReleaseShared` 在释放时会唤醒头节点后的节点，该节点会传播式地唤醒后续共享等待节点，形成“传播唤醒”。
- `CountDownLatch` 不可重用；`Semaphore` 可以释放许可证，若试图用它模仿 `CyclicBarrier` 的循环使用，需额外维护重置计数逻辑，容易因异常提前释放导致逻辑错乱，不如直接用 `CyclicBarrier`。

**第八问：线程池细节与监控**

- 提交任务：核心线程未满则新建；满了入队；队列满了且最大线程未达则新建；达到最大线程则执行拒绝策略。
- 使用无界队列时，最大线程数形同虚设，线程池永远维持在核心线程数，无法扩容，导致任务堆积，可能 OOM。
- 配置核心 10、最大 20 且无界队列，意味着希望线程波动却又堵死了扩容通道，矛盾可笑，典型误用。
- 监控：通过继承 `ThreadPoolExecutor` 覆写 `beforeExecute`/`afterExecute` 记录任务排队时间、执行时间、队列长度；定期拉取 `getQueue().size()`、`getActiveCount()`；结合拒绝策略计数，当队列逼近阈值时告警。

**第九问：ThreadLocal 内存泄漏与对策**

- ThreadLocal 底层 Entry 的 key 是弱引用，GC 后 key 可能被回收变成 null，但 value 强引用仍存在，线程存活则无法回收，导致内存泄漏。
- 线程池最佳实践：每次使用完在 `finally` 块中调用 `remove()`。
- `InheritableThreadLocal` 在线程复用时，上下文只在创建线程那一刻从父线程拷贝一次，后续无法同步更新，导致脏数据。解决方案用 TransmittableThreadLocal（TTL），通过装饰 `Runnable`/`Callable`，在提交时捕获、在执行时重放上下文，并在任务结束后恢复原状，适用于线程池。

### 三、泛型与反射

**第十问：泛型擦除与反射获取**

- 擦除：泛型在编译后替换为裸类型，但在字节码的 `Signature` 属性中保留了泛型声明；`LocalVariableTypeTable` 也保存局部变量的泛型信息。因此可通过反射读取。
- 获取 `HashMap<String, Integer>` 的 key 类型：需通过匿名子类获取父类泛型参数，例如 `new HashMap<String,Integer>(){}.getClass().getGenericSuperclass()` 得到 `ParameterizedType`，调用 `getActualTypeArguments()[0]` 得到 `String` 类型。`ParameterizedType` 描述参数化类型，`TypeVariable` 代表类型参数本身（如 E、K）。

**第十一问：PECS 与类型安全**

- `List<? extends Number>`：只能取不能加（除了 null），因为列表可能是 `List<Integer>`，若允许 `add` 则可能塞入 `Double`，破坏类型安全。编译器用捕获转换禁止。
- `List<? super Integer>`：可以添加 `Integer` 及其子类，因为确定容器至少能接纳 `Integer` 类型，但取出时只能视为 `Object`，因为不知道实际下界。

**第十二问：反射性能优化与方法句柄**

- 反射开销：访问检查、参数装箱拆箱、方法遍历查找；早期 MethodAccessor 由本地实现逐步“膨胀”（inflate）为动态生成字节码的直接实现，越过安全检查后提速。
- 优化：缓存 Method 对象；使用 `MethodHandle` 预解析，其 `invokeExact` 提供更精准的类型签名，避免包装；可通过 `LambdaMetafactory` 把反射方法转为函数接口实例，调用与直接调用几乎无异。
- `MethodHandle` 并不能完全替代反射，它对访问权限检查要求签名精确，灵活性较低；但在高频调用路径中能提供接近原生调用的性能。

**第十三问：简易 DI 容器与循环依赖**

思路：
- 定义 `BeanDefinition` 存类信息、是否单例等。
- 扫描包获取带 `@Inject` 注解的字段。
- 维护三级缓存：一级 `singletonObjects`（完全成品），二级 `earlySingletonObjects`（暴露的早期引用），三级 `singletonFactories`（对象工厂）。
- 创建 Bean A 时提前暴露工厂到三级缓存，然后填充字段 B；若 B 依赖 A，从工厂获取早期 A 并放入二级缓存，以此解开循环。
- 循环检测：维护一个“正在创建”集合，发现递归重建同一 Bean 时判定循环。

### 四、设计模式与模块化

**第十四问：DCL 与单例安全**

- DCL 需要 `volatile` 禁止初始化阶段指令重排（分配内存 -> 初始化 -> 引用赋值），否则可能返回半初始化对象。
- 枚举单例通过 `Enum` 类的序列化机制天生防序列化破坏，且反射调用 `newInstance` 时会检查是枚举而抛异常。
- 不用枚举防止反射：在私有构造器内判断静态实例是否已存在并抛异常，但可以通过反射修改标志绕过，因此最安全仍是枚举。

**第十五问：支付系统设计**

- 使用策略模式，定义 `PaymentStrategy` 接口（含支付、回调、查询），每个通道一个实现类；工厂模式根据通道标识生成实例。
- 通过 SPI 或配置扫描，新增通道只需加入新实现并注册，主业务只依赖接口，符合开闭原则。
- 组合降级：使用责任链，每个支付策略实现可持有下一个备选通道，失败时调用下一个；或引入支付路由引擎，根据规则编排多个策略，采用模板方法控制流程。
- 零侵入编排：把降级规则抽象为可配置的责任链，利用 `@Order` 或 XML 编排执行顺序，主业务只调用链的入口，内部动态组装。

**第十六问：模块划分与约束**

- 交易系统模块：订单模块、支付模块、商品模块、用户模块。各模块分为 `api`（对外暴露接口、DTO、枚举）和 `impl`（内部实现）。API 层清晰定义契约。
- 反向依赖靠构建工具（如 Maven 模块依赖单向）保证 `impl` 只能依赖 `api`，不能反向。使用 ArchUnit 编写测试规则检查包依赖，禁止 `impl` 被其他模块的 `impl` 直接引用，也可利用 Java 9 模块化 `exports ... to` 约束暴露范围。
- 在 Code Review 中指出订单 `impl` 内部 DTO 泄露到支付模块会形成强耦合，造成修改连锁反应；必须以 `api` 模块的数据传输对象交互，并用 ArchUnit 规则确保编译失败。

### 五、综合实战

**第十七问：线程安全 LRU 实现**

（白板快速写关键点）

```java
public class LRUCache<K, V> {
    private final int capacity;
    private final Map<K, Node> map;
    private final Node head, tail;
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    class Node { K key; V value; Node prev, next; }

    public LRUCache(int capacity) {
        this.capacity = capacity;
        map = new HashMap<>();
        head = new Node(); tail = new Node();
        head.next = tail; tail.prev = head;
    }

    public V get(K key) {
        lock.readLock().lock();
        try {
            Node node = map.get(key);
            if (node == null) return null;
            moveToHead(node);
            return node.value;
        } finally { lock.readLock().unlock(); }
    }

    public void put(K key, V value) {
        lock.writeLock().lock();
        try {
            Node node = map.get(key);
            if (node != null) {
                node.value = value; moveToHead(node);
            } else {
                Node newNode = new Node(key, value);
                map.put(key, newNode); addToHead(newNode);
                if (map.size() > capacity) {
                    Node removed = removeTail();
                    map.remove(removed.key);
                }
            }
        } finally { lock.writeLock().unlock(); }
    }
    // ... 双向链表操作
}
```

- 锁粒度瓶颈：读写锁在高并发写时退化到互斥。优化方向：模仿 `ConcurrentHashMap` 的桶级锁将哈希表分段，但 LRU 链表需要同步。可采用类似于 `ConcurrentLinkedHashMap` 的方法，使用多队列或 CAS 维护 LRU 顺序，如通过多个读队列与定时的写队列合并来减少竞争，或直接使用无锁队列。

**第十八问：高吞吐消息处理模块**

核心思路：基于 **Disruptor 模式** 的环形缓冲区，无锁并发。

- 结构：预分配 RingBuffer，生产者通过 CAS 申请序号写入消息，消费者使用序号屏障等待并批量拉取。
- 解析、校验、入库分解为多个 EventProcessor（消费者链），构成流水线，各阶段可独立并行驶，无锁竞争。
- 保证不丢消息：在返回 ACK 前，消息必须持久化到本地 Mapped 文件顺序写（类似 commit log），刷盘策略可设为组提交以兼顾延迟和可靠性。
- 50ms 延迟：内存操作在微秒级，异步批量写文件和入库；流量尖峰由 RingBuffer 大小缓冲，配合背压策略拒绝溢出或阻塞生产者。
- 可扩展性：流水线任意阶段可以增加并行消费者（多线程），通过序号依赖控制顺序；如果需要横向扩展，再引入一致性哈希分片，但单机首先用无锁设计榨干资源。
