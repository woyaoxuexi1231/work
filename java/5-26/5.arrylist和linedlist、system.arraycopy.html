忘掉大 O：
ArrayList vs LinkedList
在真实硅片上的 add() 代价

教科书说 LinkedList 头插 O(1)、中间插 O(n)，ArrayList 尾插均摊 O(1)、中间插 O(n)。 但在现代 CPU 上，常数因子差了 10~100 倍。 这篇不讲渐进复杂度，只讲纳秒、Cache Miss、malloc 和 SIMD 指令。

1

尾部插入：ArrayList 的绝对统治区

ArrayList.add(e)

~5ns 稳态
①
容量检查：一次整数比较 size < elementData.length。分支预测命中率 > 99.9%（因为绝大多数时候不需要扩容）。代价：~0.3ns
②
写入元素：elementData[size] = e。由于数组连续分配，新位置几乎总在 L1 Cache 中。代价：~1ns（L1 hit）
③
size++：寄存器自增。代价：~0.1ns

扩容时呢？ Arrays.copyOf → malloc 新数组 + memcpy 旧数据。 但按 1.5x 增长策略，N 次插入总共触发 O(log N) 次扩容， 均摊到每次插入的额外拷贝成本 < 2 个元素。 所以均摊代价仍然是 ~5ns。

LinkedList.add(e)

~30-80ns 波动大
①
new Node()：JVM 对象分配。TLAB 内分配 ~10ns；TLAB 耗尽需同步申请 ~100ns+。Node 对象头 12B + prev/next/item = 32B（压缩指针下）。代价：10~100ns
②
链接指针：修改 last.next、node.prev、this.last。三次内存写入，且目标地址是上一次 new 出来的——大概率不在 L1 Cache 中。代价：3~15ns（取决于 cache miss）
③
size++：同 AL。代价：~0.1ns

核心瓶颈：不是算法，是 malloc/new。 每个节点都是一次独立的堆分配。即使 JVM 有 TLAB 优化， 大量小对象仍然会造成 GC 压力和内存碎片。

2

头部插入：LinkedList 唯一的理论优势区……真的吗？

LinkedList.addFirst(e)

操作：new Node + 修改 first.prev + this.first。

代价：与尾插相同，~30-80ns。瓶颈仍然是 new Node()。

✅ 不需要移动任何已有元素。这是 LL 相对 AL 的唯一算法优势。

ArrayList.add(0, e)

操作：System.arraycopy(data, 0, data, 1, size) + 写入 data[0]。

代价：与当前 size 线性相关。

size=10~15ns
size=100~40ns
size=1000~200ns
size=10000~1500ns

⚖️ 临界点在哪里？

当 size < 50~100 时，ArrayList 头插仍然快于或等于 LinkedList， 因为 arraycopy 搬几十个引用只需几纳秒，而 LL 的 new Node() 固定要 30ns+。 只有当列表很长且频繁头插时，LL 才有优势。 但如果你真的需要频繁头插，你应该用 ArrayDeque（环形数组，头插 O(1) 且缓存友好）。

💥 反直觉结论

即使在中间插入，ArrayList 通常仍然比 LinkedList 快一个数量级。 原因是 LL 的"O(n/2) 遍历"中每一步都可能 cache miss（节点散落在堆各处）， 而 AL 的 arraycopy 虽然也是 O(n)，但它是连续内存的顺序读写， CPU 预取器（Prefetcher）能完美预测访问模式，实际吞吐量接近内存带宽上限。

5

System.arraycopy 为什么敢搬数据？它到底快在哪？

System.arraycopy 不是一个普通的 Java 方法。 它是一个 native intrinsic，JIT 编译器会将其替换为平台最优的机器码。 它的快体现在四个层面：

⚡

SIMD 向量化拷贝

HotSpot JIT 会将 arraycopy 编译为 SSE/AVX/NEON 指令。 在 x86-64 上，使用 VMOVDQU 一次搬运 32 字节（AVX2）甚至 64 字节（AVX-512）。

; 搬 1000 个引用 (8KB)
; 传统: 1000 × MOV = 1000 条指令
; AVX2: 250 × VMOVDQU = 250 条指令
; 吞吐: ~30 GB/s (接近 L1 带宽上限)
🛡️

零边界检查

普通 Java 循环 for(i...) a[i]=b[i] 每次迭代都要做两次边界检查。 arraycopy 作为 intrinsic，JIT 只在入口处做一次范围校验， 内部循环完全消除边界检查。

; 普通循环每元素: CMP + JAE + MOV = 3 ops
; arraycopy 每元素: VMOVDQU = 1 op
; 指令数减少 3×，流水线利用率翻倍
📐

对齐感知 + 自适应策略

JVM 内置的 arraycopy stub 会根据源/目标地址的对齐情况选择不同路径：

  • • 小数组 (<32B)：展开为标量 MOV，避免 SIMD 启动开销
  • • 对齐拷贝：直接用 aligned load/store，更快
  • • 非对齐：先对齐头部，中间 SIMD，尾部标量处理
  • • 重叠拷贝：自动检测方向，从前往后或从后往前，保证正确性
♻️

批量写屏障（Write Barrier）

GC 需要在引用写入时记录脏页（Card Table）。 普通循环每次赋值都触发一次写屏障。 arraycopy 将写屏障合并为一次： 拷贝完成后只对覆盖的 Card Table 区间做一次标记。 对于 G1/ZGC 这类分代收集器，这减少了大量元数据操作。

// JIT 编译时，System.arraycopy 被替换为以下 stub：

if (len < 32) {
    // 小数组：展开为标量 MOV，无循环开销
    switch (len) {
        case 8: *(dst+7) = *(src+7);
        case 7: *(dst+6) = *(src+6);
        ...
        case 0: return;
    }
} else if (aligned(src) && aligned(dst)) {
    // 对齐大数组：纯 SIMD 循环
    for (; len >= 32; len -= 32)
        VMOVDQU [dst], [src];  // AVX2: 32 bytes/cycle
    // 尾部标量处理...
} else {
    // 非对齐：head-align → SIMD body → tail-scalar
    ...
}

// 最后：批量更新 GC Card Table
mark_cards(dst, len);  // 一次调用替代 N 次写屏障

🏁 一句话总结

ArrayList 的 add() 快，不是因为算法更优，而是因为它的内存布局与现代 CPU 的缓存层次、预取器、SIMD 单元完美契合。 System.arraycopy 是 JVM 送给它的"硬件加速器"，把 O(n) 的常数压到了纳秒级。

LinkedList 的 add() 慢，不是因为算法差，而是因为每个节点都是一次独立的堆分配 + 不可预测的内存访问。 在现代 CPU 上，一次 cache miss (~100ns) 的代价足以抵消 O(1) 相对于 O(n) 的所有算法优势。

💡 实践法则：默认用 ArrayList。需要双端操作用 ArrayDeque。 只有在"频繁在已知位置插入/删除 + 列表极大 + 已 profiling 确认瓶颈"时，才考虑 LinkedList—— 而且大多数时候你会发现，真正该换的是数据结构本身（如 SkipList、B+Tree、或自定义 buffer pool）。
