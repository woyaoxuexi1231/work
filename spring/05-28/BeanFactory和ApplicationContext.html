当需要编写自定义Starter、实现热部署、或在非Spring环境嵌入IOC容器时，只有BeanFactory级别的API才能救你。

"既然Spring Boot都封装好了，你觉得还有必要区分BeanFactory和ApplicationContext吗？"

The Story Behind the Code

Spring 容器进化史：不是设计出来的，是逼出来的

想真正理解 BeanFactory 和 ApplicationContext，必须回到 2002 年—— 那个 J2EE 开发者被 EJB 折磨得生不如死的年代。

2000-2003: EJB 2.x 噩梦

1. 写一个 EJB 需要 3 个文件：Home 接口、Remote 接口、Bean 实现类，缺一不可。

2. 必须实现 SessionBean / EntityBean 接口，连 ejbActivate() / ejbPassivate() 这些空方法都得写上。

3. 部署必须打成 EAR 包，扔进 WebLogic / WebSphere / JBoss 等重量级应用服务器，启动一次 5 分钟起步。

4. 单元测试？（苦笑）你必须把整个 EAR 部署到服务器上才能跑——TDD 根本不可能。

5. 一个简单 CRUD 项目，代码量是同等功能 Servlet+JDBC 的 3~5 倍。

—— Rod Johnson, Expert One-on-One J2EE Design and Development, 2002

Rod Johnson 的核心洞察

"EJB 的复杂性和重量级并不是企业级开发的必需品。"

他提出三个激进观点：

▸ IoC 容器不需要应用服务器 — 一个普通 Java SE 程序就够了。

▸ POJO 可以替代 EJB — 普通的 Java 对象，不需要实现任何框架接口。

▸ 依赖注入 > JNDI 查找 — 对象不该自己去"找"依赖，容器应该主动"注入"。

这套理念就是 Spring 的 DNA。而承载这套理念的第一个容器 —— 就是 BeanFactory。

2003 Spring Framework 0.9 第一个公开版本

👶 BeanFactory 是唯一的主角

第一个公开版本里，XmlBeanFactory 就是整个框架。 你写一个 XML 配置文件描述 bean，框架帮你创建和管理它们。 就这么简单，没有 AOP，没有事件，没有国际化。

🤔 当时的设计哲学

"最小可用"——比 EJB 轻 100 倍，不需要应用服务器，普通 main() 就能跑。 这是故意为之：BeanFactory 就是那个"只有 IoC 的极简内核"。

// 2003 年的 Spring 代码 —— 只有 BeanFactory
Resource res = new ClassPathResource("beans.xml");
BeanFactory factory = new XmlBeanFactory(res);
MyService svc = (MyService) factory.getBean("myService");
2004.03 Spring 1.0 GA ⭐ 里程碑

🏛️ ApplicationContext 诞生

1.0 正式版里 ApplicationContext 已经存在。 但它是作为 BeanFactory 的"增强版"推出的： ClassPathXmlApplicationContext、 FileSystemXmlApplicationContext。

🔑 为什么要分两层？

Spring 团队的远见：不是所有场景都需要全部能力。 ApplicationContext 继承了 BeanFactory，额外提供：

  • MessageSource — 国际化
  • ApplicationEventPublisher — 事件机制
  • ResourceLoader — 统一资源加载
  • 自动 BeanPostProcessor 注册

💡 关键认知： Spring 1.0 就同时有 BeanFactory 和 ApplicationContext。 不是"先有 BeanFactory，后来才加 ApplicationContext"—— 它们从 Spring 诞生之初就是分层设计的两个抽象层次。

2006 ~ 2009 Spring 2.0 → 2.5 🔥 爆发期

2.0 (2006)

  • XML Schema 替代 DTD
  • AspectJ 集成
  • <aop:config>
  • JMS / JMX 支持

2.5 (2007)

  • @Autowired 诞生
  • @Component / @Service
  • 注解驱动开始普及
  • Classpath 自动扫描

对容器的影响

注解扫描让 BeanFactory 的"手动注册"模式显得过时。 ApplicationContext 的自动装配能力成为标配。 开发者和 BeanFactory 的直接接触越来越少。

2009 ~ 2014 Spring 3.0 → 4.0 🚀 现代化

3.0 (2009): JavaConfig 革命

@Configuration + @Bean 的出现， 意味着 XML 不再是定义 Bean 的唯一方式。

// 纯 Java 配置 Spring 容器
ApplicationContext ctx = new
  AnnotationConfigApplicationContext(AppConfig.class);
// 底层依然是 DefaultListableBeanFactory

4.0 (2013): Java 8 时代

@EventListener 让事件发布更简单。 @Async / @Scheduled 的普及， 使得 ApplicationContext 提供的基础设施（事件、线程池、调度）成为业务开发的必需品。

此时 BeanFactory 已经彻底退居幕后。 但每次你写 @EventListener， ApplicationContext 都在背后调用 BeanFactory 来查找所有匹配的 Listener。

2014 → 现在 Spring Boot 1.0 → 3.x 👑 王者时代

Spring Boot 把 ApplicationContext 的创建完全自动化： SpringApplication.run() 内部根据 classpath 自动选择合适的 ApplicationContext 实现类（Servlet / Reactive），并完成所有配置注入。

但这并没有消灭 BeanFactory—— 打开 AnnotationConfigServletWebServerApplicationContext 源码， 你会发现它继承链的终点是 DefaultListableBeanFactory。

📌 本页面的价值就在于此：你用 Spring Boot 3.x 写的每一个 @RestController、 每一个 @Transactional， 最终都是在操作一个被 ApplicationContext 层层包装的 DefaultListableBeanFactory。

终极拷问：既然 ApplicationContext 这么好， 为什么还要保留 BeanFactory？

接口隔离原则 (ISP)

BeanFactory 定义的是"IoC 容器的最小契约"——就是 getBean + 依赖查找。 如果你只需要一个轻量级 DI 引擎，为什么强迫你依赖 AOP、事件、国际化等 10+ 个模块？

设计模式：Interface Segregation

分层架构的基石

BeanFactory → HierarchicalBeanFactory → ConfigurableBeanFactory → ConfigurableListableBeanFactory → DefaultListableBeanFactory。 这 5 层继承链，每一层增加一个"正交"的能力维度。

设计模式：Template Method + Decorator

可扩展性

如果 ApplicationContext 是一个不可拆分的整体，你怎么替换它的国际化实现？ 怎么注入自定义的事件广播器？分离接口意味着每一层都可以被替换。

设计模式：Strategy + SPI

资源受限场景

Spring 1.0 时代的目标包括 Applet 和移动设备。 今天，Android 应用、Serverless 函数、IoT 设备， 这些场景对启动速度和内存极度敏感——BeanFactory 的懒加载和极低开销无法替代。

设计目标：Write Once, Run Anywhere

面试时这样回答，直接让面试官眼前一亮

"BeanFactory 是 Spring 的 IoC '内核'，ApplicationContext 是面向企业开发的'完整运行时'。 这不仅仅是设计和实现上的分层，更反映了 Spring 团队从 2002 年 Rod Johnson 提出 IoC 理念以来一直坚持的哲学： 框架应该分层提供能力，而不是把所有东西揉成一个整体。"

"从 Spring 1.0 到 Spring Boot 3.x，经历了从 XML 到注解、从 EJB 替代品到云原生基础设施的进化， 但 DefaultListableBeanFactory 始终是那个干活的底层引擎。 ApplicationContext 是对它的增强门面，而 Spring Boot 是对 ApplicationContext 的自动装配封装。 三层抽象，各司其职，从未改变。"

核心差异速查表

特性 BeanFactory ApplicationContext Spring Boot 现状
Bean加载时机 懒加载 (getBean时创建) 预加载 (启动时全部创建) 可通过 @Lazy 切换
国际化 (i18n) 不支持 MessageSource Web应用必备
事件发布 不支持 ApplicationEventPublisher @EventListener 依赖此机制
AOP 集成 需手动配置 自动代理 声明式事务/AOP基石
内存占用 极低 较高 嵌入式/IoT场景选Factory

何时退回 BeanFactory?

  • 开发自定义 Spring Boot Starter
  • 实现动态Bean注册/销毁
  • Android / IoT 等资源受限环境
  • 编写单元测试仅需轻量容器
  • 分析启动性能瓶颈时

何时坚守 Context?

  • 所有标准Web/API应用
  • 需要使用 @Transactional
  • 使用 @Async / @Scheduled
  • 多语言国际化支持
  • 基于事件的模块解耦

Spring Framework — 从 2003 年到今天，三层架构从未改变

BeanFactory (内核) → ApplicationContext (门面) → Spring Boot (自动装配)

Remember: Frameworks change, but architectural thinking lasts.
