BeanFactory vs ApplicationContext
面试官到底在考什么？

"现在都用Spring Boot了，谁还手动new BeanFactory啊？"
如果你也这么想，请先看完这份关于框架设计哲学的深度解剖。

面试官的3层潜台词

1

架构抽象能力

你是否理解Spring如何通过接口将"IOC容器核心"与"企业级基础设施"解耦？这决定了你能否设计出高内聚低耦合的系统。

2

源码级认知深度

你是否知道 ApplicationContext 内部其实持有一个 DefaultListableBeanFactory？它只是门面，干活的还是工厂。

3

极端场景应对力

当需要编写自定义Starter、实现热部署、或在非Spring环境嵌入IOC容器时，只有BeanFactory级别的API才能救你。

"既然Spring Boot都封装好了，你觉得还有必要区分BeanFactory和ApplicationContext吗？"
❌ 普通回答： "没必要吧，反正我都用@Autowired，没区别。"
✅ 高分回答： "业务开发确实无感，但架构上必须区分。BeanFactory是IOC的最小内核，负责Bean的生命周期；而Context是面向企业的Facade，整合了AOP、MessageSource、EventPublisher等。Spring Boot的 AnnotationConfigApplicationContext 底层依然是委托给 DefaultListableBeanFactory 完成Bean注册的。理解这个分层，才能在写自定义Starter或排查启动性能问题时，知道该在哪一层动手。"

特性 BeanFactory ApplicationContext Spring Boot 现状

何时退回 BeanFactory?

  • 开发自定义 Spring Boot Starter
  • 实现动态Bean注册/销毁
  • Android / IoT 等资源受限环境
  • 编写单元测试仅需轻量容器
  • 分析启动性能瓶颈时

何时坚守 Context?

  • 所有标准Web/API应用
  • 需要使用 @Transactional
  • 使用 @Async / @Scheduled
  • 多语言国际化支持
  • 基于事件的模块解耦
