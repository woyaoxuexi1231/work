计算机科学式推演

Spring 三级缓存：
从一个 Map 开始的推演之旅

这不是一篇"三级缓存是什么"的文章。
这是一次思维实验——假设你穿越回 2004 年， 是 Spring 团队的一员，面对"循环依赖 + AOP 代理"这个死结，你会怎么办？

0 先忘掉三级缓存 —— 回到一切开始的地方

2004 年 Spring 1.0 发布时，它的核心卖点就一句话：一个不需要 EJB 服务器的轻量级 IoC 容器。 本质是什么？就是一个管理对象创建和依赖关系的工厂。

如果让你从零开始写一个最简 IoC 容器，你的第一反应是什么？

🧪 最简 IoC 容器 —— 版本 0 v0.0 
// 一个 Map 就是最简单的 IoC 容器
Map<String, Object> singletonObjects = new HashMap<>();

// "创建" A：发现需要 B
Object a = singletonObjects.get("A");
if (a == null) {
    Object b = singletonObjects.get("B"); // 需要 B
    if (b == null) {
        b = new B(); // 创建 B 需要 A → 💥 死循环!
    }
    a = new A(b);
    singletonObjects.put("A", a);
}
💥 问题：循环依赖导致无限递归 A 需要 B → 创建 B → B 需要 A → 创建 A → A 需要 B → ... StackOverflowError。

1 第一次突破：把"创建"和"装配"分开

循环依赖的本质是什么？A 说"我需要完整的 B"，B 说"我需要完整的 A"——互相等待对方先完成。 但如果 A 愿意说"我先把我的壳给你，你拿去用，内容我等下再填"呢？

🔑 核心洞察：对象的"出生"可以分两步

第 1 步：实例化 (Instantiation)

new A() —— 构造器执行完毕，对象在堆上有了地址。此时属性全是 null，但引用已经有效。

第 2 步：初始化 (Initialization)

填充属性、执行 @PostConstruct、Aware 回调、AOP 代理 —— 把对象变成"可用状态"。

这就引出了二级缓存的雏形：

一级缓存

singletonObjects

存完全初始化的 Bean

二级缓存

earlySingletonObjects

存刚实例化、未初始化的 Bean

✅ 二级缓存方案 —— 适用于纯 POJO 无 AOP 时完美工作 
// 创建 A 的流程
Object a = new A();                           // 1. 实例化
earlySingletonObjects.put("A", a);       // 2. 放入二级缓存！B 现在可以看到 A

// 填充属性：发现需要 B → 去创建 B
b = new B();                                   // B 实例化
// B 填充属性时发现需要 A → 从 earlySingletonObjects 拿到 a
b.setA(a);                                     // ✅ B 注入 A 的引用

a.setB(b);                                     // A 拿到 B
a = initializeBean(a);                          // A 初始化完成
singletonObjects.put("A", a);               // 从二级移到一级
earlySingletonObjects.remove("A");
✅ 对于纯 POJO（没有 AOP 代理），二级缓存完美工作。 A 实例化后暴露引用 → B 注入了 A → A 后续初始化修改的字段，B 通过同一个引用能读到。因为 Java 的对象引用指向的是堆上的同一块内存。

2 风暴降临：AOP 代理打破了平衡

⏳ 先回答一个关键历史问题：Spring AOP 和 IoC 容器是同一时期设计的吗？

是的，它们诞生于同一时期（2003~2004），但作为两个独立模块设计。 Spring 1.0（2004.03）同时包含了 spring-beans（IoC）和 spring-aop（AOP）。 但 Rod Johnson 从一开始就刻意把它们设计为解耦的两个模块：IoC 容器通过 BeanPostProcessor 接口暴露扩展点， AOP 模块实现这个接口，作为"插件"接入。IoC 的代码里没有一行 import org.springframework.aop。

三级缓存呢？它不是在 Spring 1.0 就设计好的。 它是在 AOP 和循环依赖这两个独立设计产生冲突后，作为"修补方案"被引入的。 如果你去看 Spring 1.0 的源码，会发现没有 singletonFactories 这个 Map。 这是后来（大约是 Spring 2.0 前后）为了解决具体 Bug 才加上去的。

有了这段历史背景，我们再来看：为什么 AOP 打破了平衡？

冲突场景：A(@Transactional) 互相依赖 B

A 上加了 @Transactional，Spring 需要为 A 创建事务代理。 A 依赖 B，B 依赖 A。

① A 实例化 → 得到 原始 A（new A()）
② 原始 A 放入二级缓存 earlySingletonObjects

此时 AOP 代理还远没到创建的时机（它在第 ⑦ 步）

③ A 填充属性，发现需要 B → 创建 B
④ B 实例化、填充属性，发现需要 A
⑤ B 从二级缓存拿到 原始 A

⚠️ 此时 A 的 AOP 代理还没创建！

⑥ B 持有的引用是 原始 A，不是代理 A

💥 B 调用 A 的方法 → @Transactional 不生效 → 事务静默失效！

等一下——如果二级缓存不放"原始 A"，改放"提前创建好的代理 A"，不就解决了？

这似乎是所有人的第一反应。让我们认真地推演一遍，看看到底哪里会出问题。

3 穷举法：逐一推演所有替代方案

我们不假设三级缓存是"唯一正确"的答案——而是把每条路都走一遍，看看到底哪里会撞墙。

A

方案 A：把二级缓存从"放原始对象"改为"放提前创建的代理"

"A 实例化后立刻创建代理，把代理放进 earlySingletonObjects，B 直接拿到代理 A。"

我们先诚实回答一个关键问题：代理对象的 target 存的是引用，如果后续 populateBean() 给原始 A 填充了属性，代理能通过 target 引用看到这些属性吗？

🔬 实验：CGLIB 代理能否看到"后来才填充"的属性？
// CGLIB 代理本质是一个子类：
class A$$EnhancerByCGLIB extends A {
    // 它不"持有"一个 target 引用，它 IS-A A。
    // 所有拦截器织入到重写的子类方法里。
    // this.getName() 读取的是 this.name 字段——
    // 就是代理对象自己的字段（继承自父类 A）。
}

// 所以如果你：
A proxy = new A$$EnhancerByCGLIB();  // 创建代理
proxy.setName("hello");                // 通过代理的 setter 设置字段
proxy.getName();                        // → "hello" ✅ 当然看得到！
🤔 诚实地说：如果只是"字段最终能否读到"这个问题，答案是——能读到。

CGLIB 代理是一个子类实例，它继承并持有父类的所有字段。后续 populateBean() 通过反射给这些字段赋值，代理对象自己的字段就被填充了。如果只是这个维度，方案 A 似乎没问题。

但方案 A 的真正问题不在"字段能否读到"，而在下面三个更深层的问题：

问题 1：BeanPostProcessor 拿到的不再是原始对象

Spring 的初始化链是：populateBean() → @PostConstruct → InitializingBean.afterPropertiesSet() → BeanPostProcessor.postProcessBeforeInitialization() → BeanPostProcessor.postProcessAfterInitialization()。 这条链上所有的处理器都预期收到的是原始 Bean 实例，而不是一个 CGLIB 子类。

举个例子：Spring 的 CommonAnnotationBeanPostProcessor 负责处理 @PostConstruct 和 @Resource。 它通过反射在目标类上查找方法。如果传来的已经是一个 CGLIB 子类，反射扫描会扫到 CGLIB 生成的大量 CGLIB$xxx 方法， 以及被 final 修饰的原始方法（CGLIB 不能代理 final 方法）。这会导致行为不确定——有些注解可能被处理，有些可能被跳过。

问题 2：如果 A 同时有多个切面怎么办？

假设 A 同时有 @Transactional 和 @Cacheable。 Spring 的 AOP 是按顺序创建多层代理的：先创建事务代理包装原始对象，再创建缓存代理包装事务代理（或者反过来）。 如果你在属性填充之前就创建了代理，你只创建了一层。 后续的 BPP 还要再套一层——结果就是 B 拿到了单层代理，而最终容器里的是双层代理，两者不一致。

问题 3：代理创建的逻辑本身依赖 Bean 的完整元数据

AOP 代理的创建（AbstractAutoProxyCreator.wrapIfNecessary()）需要知道：这个 Bean 有哪些 Advisor 匹配？ 哪些方法需要被拦截？在属性填充之前，Bean 的某些元数据可能还不完整（比如通过 @Value 注入的配置还没到位）。 虽然大多数情况下类级别的注解已经可用，但有些动态切面（比如基于 Bean 名称的匹配）在早期阶段可能得到错误的结果。

💡 所以方案 A 的真正死因不是"代理读不到字段"——这个直觉其实是错的。真正的问题在于：提前创建代理会扰乱 BeanPostProcessor 链的正常执行顺序。Spring 的整个初始化管线是围绕"原始对象"设计的，你把一个 CGLIB 子类塞进去，就像给一个预期接收苹果的流水线塞了一个菠萝——有的工位还能处理，有的直接罢工。

B

方案 B：让 IoC 容器自己去调 AOP 的 API，在 doCreateBean() 里硬编码创建代理

"既然 AOP 模块的 API 是现成的，IoC 直接调用它，不就绕开了 BPP 时序问题？"

这确实能解决功能问题。但让我们看看代价：

❌ 方案 B 的 doCreateBean() —— IoC 硬编码 AOP 耦合灾难 
protected Object doCreateBean(String beanName, ...) {
    Object bean = createBeanInstance();

    // ← 从这里开始，IoC 核心代码被迫认识每一个切面模块：

    if (transactionManager.needProxy(bean)) {
        bean = transactionManager.createProxy(bean);      // 事务
    }
    if (cacheManager.needProxy(bean)) {
        bean = cacheManager.createProxy(bean);             // 缓存
    }
    if (asyncExecutor.needProxy(bean)) {
        bean = asyncExecutor.createProxy(bean);            // 异步
    }
    if (retryTemplate.needProxy(bean)) {
        bean = retryTemplate.createProxy(bean);            // 重试
    }
    // 每加一个切面，就要在这里加一个 if...

    earlySingletonObjects.put(beanName, bean);
    populateBean(beanName, mbd, instanceWrapper);
    initializeBean(beanName, exposedObject, mbd);
}
🔥 致命缺陷：恢复了 EJB 时代的"单体噩梦"

Rod Johnson 设计 Spring 的核心理念之一就是：核心容器不感知任何切面。 事务、缓存、异步、重试、日志、监控——这些都是通过同一个 BeanPostProcessor 接口插入的。 每个切面模块是一个独立的 jar 包，你引入它就生效，不引入就没有。 这个架构让 Spring 可以在二十年间从 5 个模块扩展到 50+ 个模块，而核心代码几乎不需要修改。

但这是否过于理想主义了？为了架构纯洁性就搞出三级缓存这么复杂的东西？

好问题。这就是工程上的价值判断。如果 Spring 只有两个切面（事务和缓存），那方案 B 可能就用了。 但当你知道这个框架会被全世界数百万开发者用 20 年、在上面构建无数你想象不到的扩展时—— 你也会选择多写 50 行代码来保全架构设计。这就是"设计"和"堆功能"的区别。

C

方案 C：禁止 AOP Bean 参与循环依赖，检测到就抛异常

"这个极端场景太罕见了，直接不让用。"

这个方案最"省事"，但有两个实际问题：

1. A(@Service) ⇄ B(@Service) 的循环在大型项目中极其常见，不是边缘 case。UserService 需要 OrderService，OrderService 需要 UserService——这种业务上的双向调用太正常了。

2. 开发者很难理解为什么加上 @Transactional 就报错了。 从业务视角看，"我在 Service 上加了事务注解"和"出现了循环依赖"这两个现象之间没有任何直观的因果关系。这会是 Spring 被骂得最多的设计。

3. 实际上 Spring 确实在某些情况下会抛异常（比如构造器注入的循环依赖），那是因为构造器注入的循环依赖数学上不可能解决。而 Setter 注入的循环依赖是可以解决的——所以 Spring 选择解决它，而不是禁止它。

4 三级缓存的真正精妙之处

好了，你看到了——方案 A 会扰乱 BPP 管线，方案 B 会破坏架构解耦，方案 C 太粗暴。 那么 Spring 团队是怎么找到出路的？

🔑 破局思路：不是"提前创建代理"，而是"推迟决定要不要提前"

方案 A 的最大问题是：它无条件地提前创建了代理。即使没有发生循环依赖（大多数情况），代理也被提前了。 这就像你为了防止偶尔堵车，每天凌晨 4 点就出门——代价太大了。

Spring 的答案是：不要在 A 实例化后就创建代理。而是存一个"如果 B 来找 A，到时候再创建代理"的承诺。 这个承诺就是 ObjectFactory（一个 lambda）。

如果没有发生循环依赖，这个 lambda 永远不会被调用，A 的代理仍然按正常流程在初始化末尾创建。 如果发生了循环依赖，B 来 getBean(A) 时触发这个 lambda，提前创建代理给 B。 两全其美。

这就是三级缓存：它不是存"代理对象"，也不是存"原始对象"，而是存一个能够生产早期引用的工厂。

一级: singletonObjects

完全就绪的 Bean

二级: earlySingletonObjects

从工厂生产出来的早期 Bean
（可能是原始对象，也可能是代理）

三级: singletonFactories

ObjectFactory ——
一个"按需生产"的 lambda

✅ Spring 真实源码逻辑（精简） AbstractAutowireCapableBeanFactory 
protected Object doCreateBean(String beanName, ...) {
    Object bean = createBeanInstance();   // 1. 实例化

    // 2. 放入三级缓存：一个 lambda，不是对象本身！
    addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

    Object exposedObject = bean;
    populateBean(beanName, mbd, instanceWrapper);  // 3. 属性填充
    exposedObject = initializeBean(beanName, exposedObject, mbd); // 4. 初始化

    // 5. 检查早期引用和最终对象是否一致
    Object earlySingletonReference = getSingleton(beanName, false);
    if (earlySingletonReference != null) {
        if (exposedObject == bean) {
            exposedObject = earlySingletonReference; // 用早期引用替换
        }
    }

    return exposedObject;
}
🎯 关键：lambda 内部做了什么？

getEarlyBeanReference() 遍历所有的 SmartInstantiationAwareBeanPostProcessor。 AOP 模块的 AbstractAutoProxyCreator 实现了这个接口。当它被调用时， 它判断这个 Bean 是否需要代理，如果需要就提前创建并缓存。 注意：决定"是否创建代理"的逻辑仍然在 AOP 模块内部——IoC 容器依然不知道 AOP 的存在。

那如果没有循环依赖，lambda 什么时候被清理掉？

Bean 初始化完成后，Spring 会调用 addSingleton() 把最终 Bean 放入一级缓存。 这个操作同时会从三级缓存移除 factory、从二级缓存移除早期引用。lambda 被 GC 回收。 整个生命周期中，这个 lambda 可能从未被调用——这正是设计的目的。

5 临门一脚：B 从缓存里拿到的到底是个什么？

这是九成人脑子最乱的地方——"A 还没创建完呢，A 还依赖 B 呢，B 怎么就拿到了 A？
拿到的 A 是半成品吗？那后来 A 的字段被填上了，B 手里的引用还能看见吗？"

📦 案例设定

两个互相依赖的 Service
@Service
public class OrderService {
    @Autowired
    private UserService userService;  // A 依赖 B

    @Transactional  // ← 需要 AOP 代理!
    public void createOrder() { ... }
}

@Service
public class UserService {
    @Autowired
    private OrderService orderService;  // B 依赖 A
}

我们将分别跟踪两种场景下每一个对象的内存身份（identity）：

场景 ①：A 没有 @Transactional 纯 POJO 循环依赖，二级缓存就能搞定
场景 ②：A 有 @Transactional A 需要 AOP 代理，三级缓存介入

场景 ①：无 AOP —— 二级缓存足够

在这个场景下，只有一个 Java 对象——rawA（OrderService 的实例）。 从出生到进容器，它的堆地址从未变过。

对象 rawA (OrderService@0x1234) 的生命线

T1 实例化 — rawA = new OrderService()

rawA.userService = null （字段默认值）

T2 放入二级缓存 — earlySingletonObjects.put("orderService", rawA)
T3 A 填充属性 — 发现需要 B → 触发 B 创建
T4 B 填充属性 — 发现需要 A → 走缓存查询

→ L1 miss → L2 HIT! → b.orderService = rawA → B.orderService 指向 @0x1234

T5 A 继续初始化 — rawA.userService = b （setter 注入）

现在 rawA 的字段被填上了！rawA.userService = B@0x5678

T6 A 初始化完成 — @PostConstruct 执行（rawA 身上）
T7 移入一级缓存 — singletonObjects.put("orderService", rawA)

从二级缓存移除。一级缓存里的是同一个 rawA @0x1234。

✅ 结论：B 拿到的 rawA 和容器最终持有的 rawA，是同一个 Java 对象。
  • T4 时 B 把 rawA 的引用存到了自己的 orderService 字段里。
  • T5 时 Spring 把 B 的引用写入了 rawA.userService。
  • 这是同一个堆对象——它的字段从 null 变成了有值，但地址没变。B 手里的引用天然能看到这些变化。
  • 不需要任何"同步"或"回填"——Java 的引用语义天然保证了这一点。

场景 ②：有 AOP —— 三级缓存介入

这次，有两个 Java 对象：rawA（OrderService 实例）和 proxyA（CGLIB 代理实例）。 proxyA.target = rawA。

两个对象，一个引用链：proxyA (OrderService$$Enhancer@0xAABB) → rawA (OrderService@0x1234)

T1 实例化 — rawA = new OrderService() (@0x1234)

rawA.userService = null

T2 放入三级缓存 — singletonFactories.put("orderService", lambda)

注意：这里放的是 lambda 工厂，不是 rawA 本身！

T3 A 填充属性 — 发现需要 B → 触发 B 创建
T4 B 填充属性 — 发现需要 A → 走缓存查询

→ L1 miss → L2 miss → L3 HIT!

T5 🔥 关键时刻：调用 lambda.getObject()

→ lambda 内部调用 getEarlyBeanReference("orderService", rawA)

→ 遍历 SmartInstantiationAwareBeanPostProcessor

→ AbstractAutoProxyCreator 拦截："A 有 @Transactional，需要代理！"

→ proxyA = createProxy(rawA) — 创建 CGLIB 代理 @0xAABB

→ 记录到 earlyProxyReferences："A 的早期代理已创建"

→ 返回 proxyA（不是 rawA！）

T6 proxyA 放入二级缓存 + B 获取

earlySingletonObjects.put("orderService", proxyA)
b.orderService = proxyA → B.orderService 指向 proxyA @0xAABB

T7 A 继续初始化 — 在 rawA 身上执行！

rawA.userService = b （setter 注入到 rawA，不是 proxyA）
rawA.init() （@PostConstruct 在 rawA 身上执行）

T8 AOP BPP 后置处理

postProcessAfterInitialization(rawA, "orderService")
→ 检查 earlyProxyReferences.contains("orderService") → TRUE
→ 不再创建新代理，直接返回 rawA
（阻止了第二个代理的产生！）

T9 最终判定

exposedObject == bean (rawA == rawA) → true
earlySingletonReference = getSingleton("orderService") → proxyA
→ exposedObject = proxyA // 用早期代理替换！

T10 移入一级缓存 — singletonObjects.put("orderService", proxyA)

一级缓存里的是 proxyA @0xAABB。
rawA @0x1234 作为 proxyA 的内部 target 存活，但不直接暴露给外部。

🤔 追问：T7 是在 rawA 身上 populate，那 proxyA 能"看见" rawA 被填充的属性吗？

能。因为 CGLIB 代理的每一个方法调用最终都委托给 rawA（target 引用）。 proxyA 自己不存业务字段——它只是一个拦截器壳。

B 调用 A 的方法时发生了什么
// B 的某段代码：
orderService.createOrder();

// orderService 指向 proxyA @0xAABB（CGLIB 代理）
// CGLIB 拦截：
//   1. 开启事务（@Transactional advice）
//   2. 调用 target.createOrder() → rawA.createOrder() @0x1234
//   3. rawA.createOrder() 内部访问 this.userService → rawA.userService → B@0x5678 ✅
//   4. 提交/回滚事务
//   5. 返回结果

🤔 再追问：rawA 不是在 T1 就创建了吗，为什么 T7 才 populate？那 T1~T7 之间 proxyA 如果被 B 调用了，rawA 的字段还是 null，不就出问题了吗？

这是一个时序问题，答案很简单：B 在 T4~T6 期间只是"持有"了 proxyA 的引用，并没有立即调用 A 的方法。 循环依赖解决的是"注入"（把引用赋给字段），不是"调用"。

Spring 的设计假设：在容器完全启动之前，不会有业务方法被调用。所有 Bean 的依赖注入全部完成后， 容器才发启动完成事件，业务线程才开始访问。在此之前，rawA 的字段已经从 null 变成了有效值（T7 完成）。

6 源码走读：三个 Map 到底存在哪

所有三级缓存都定义在 org.springframework.beans.factory.support.DefaultSingletonBeanRegistry 中。 这是 Spring IoC 最底层的类之一。

📍 DefaultSingletonBeanRegistry.java 三个 Map 的声明 
/** 一级缓存：完全就绪的单例 Bean */
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** 三级缓存：单例工厂（lambda），按需生产早期引用 */
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

/** 二级缓存：从工厂生产出来的早期 Bean 引用 */
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

获取单例的核心方法 —— 这就是三级缓存查询的完整逻辑：

📍 DefaultSingletonBeanRegistry.getSingleton() 完整的三级查询 
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 第一级：从 singletonObjects 查
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        synchronized (this.singletonObjects) {
            // 第二级：从 earlySingletonObjects 查
            singletonObject = this.earlySingletonObjects.get(beanName);
            if (singletonObject == null && allowEarlyReference) {
                // 第三级：从 singletonFactories 取出工厂, 调用 getObject()
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject(); // ← 这里触发 lambda!
                    this.earlySingletonObjects.put(beanName, singletonObject); // 升级到二级
                    this.singletonFactories.remove(beanName); // 从三级移除
                }
            }
        }
    }
    return singletonObject;
}
📌 注意数据流向：

三级缓存的数据是单向流动的：
singletonFactories（被触发）→ 生产出对象 → 放入 earlySingletonObjects → 最终 Bean 就绪后 → 放入 singletonObjects。
一级不会回退到二级，二级不会回退到三级。这是"承诺兑现"后就不可逆的语义。

7 最后的升华：这到底教会我们什么

1. 三级缓存不是一个"feature"，它是一个"补丁"

它不是 Spring 团队在 2004 年画在白板上的天才设计。它是当 IoC 的分步创建 和 AOP 的延迟代理 这两个独立设计的优雅特性，在循环依赖这个边界条件下发生冲突时，团队找到的代价最小的和解方案。 这也是为什么 Spring 官方文档几乎不提三级缓存——它更像一个实现细节，而非一个"功能"。

2. 理解框架，要理解它的"边界条件"

很多开发者知道三级缓存的三个 Map 叫什么名字，但不知道什么时候会触发三级缓存查询、什么时候 lambda 会被跳过。 真正理解一个框架，不是记住它的 API，而是能推导出它在极端输入下的行为。 这就是"背答案"和"真理解"的区别。

3. 架构设计的本质是管理"张力"

Spring 的设计中充满了这种张力：

  • IoC 想要"简单"（一个 Map），AOP 想要"灵活"（动态代理）→ 三级缓存是妥协
  • 核心容器想要"纯粹"（不知道切面），企业开发想要"方便"（@Transactional 一行搞定）→ BeanPostProcessor 是妥协
  • 框架想要"通用"（支持任何扩展），开发者想要"简单"（开箱即用）→ Spring Boot 是妥协

好的架构不是没有矛盾，而是让矛盾在正确的抽象层被化解。
