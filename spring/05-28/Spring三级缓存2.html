Advanced Spring Internals

为什么二级缓存“能”解决，
Spring 却偏要用三级？

抛弃“二级存原始、三级存代理”的肤浅回答。本教程将从 Spring 生命周期设计原则 与 BeanPostProcessor 扩展点架构 两个维度，彻底揭示三级缓存存在的历史必然性。

核心矛盾：你的方案 vs Spring 的原则

你的方案 (二级缓存)

“发现循环依赖，立刻创建代理放入二级”

A 实例化后，在填充属性发现依赖 B。此时立刻调用 AOP 逻辑生成 A 的代理对象，存入二级缓存。B 从二级缓存拿到代理，完美解决。

致命缺陷：破坏了生命周期 Spring 的设计原则是：AOP 代理应该在 Bean 生命周期的最后一步（初始化后）创建。如果所有 Bean 在实例化阶段就提前判断并创建代理，等于把 AOP 逻辑硬编码到了 IOC 的早期阶段。
Spring 的方案 (三级缓存)

“存入工厂，延迟到不得不创建时再创建”

A 实例化后，只把“创建代理的工厂 (ObjectFactory)”放入三级缓存。只有当 B 真的去获取 A 时，才触发工厂的 getObject() 方法，此时才提前创建代理。

架构优势：延迟计算 (Lazy Evaluation) 如果没有循环依赖，工厂根本不会被调用，A 的代理依然会在正常的生命周期末尾创建。三级缓存完美保全了 Spring “尽可能延迟代理创建”的设计哲学。

Bean 生命周期与代理创建时机对比

1. 实例化 (Instantiation)

调用构造器，生成 Raw Object

三级缓存动作

putSingletonFactory(A)

二级缓存方案的动作

if(isAop) createProxy(A)

❌ 破坏原则：过早创建代理

2. 属性填充 (Populate)

注入依赖，可能触发循环依赖

3. 初始化 (Initialization)

执行 Init 方法，Aware 接口回调

4. 后置处理 (PostProcess)

AOP 代理在此刻创建 (正常流程)

✅ 三级缓存：若无循环依赖，代理在此创建

架构因果：BeanPostProcessor 的尊严

Spring 的核心魅力在于 IOC 与 AOP 的解耦。AOP 不是硬编码在容器里的，而是通过 BeanPostProcessor (后置处理器) 扩展点插入的。
如果采用二级缓存方案，Spring 必须在“实例化”阶段就去判断是否需要 AOP，这会导致 IOC 核心代码直接耦合 AOP 逻辑，彻底毁掉 Spring 的扩展架构。

❌ 假设只有二级缓存 (破坏架构) 硬编码 
// AbstractAutowireCapableBeanFactory.doCreateBean()

Object bean = createBeanInstance(); // 1. 实例化

// 2. 填充属性前，被迫提前处理 AOP
// IOC 容器必须知道 AOP 的存在！
if (aopAdvisorRegistry.hasAdvisors(bean)) {
    bean = AopProxyFactory.createProxy(bean);
}
earlySingletonObjects.put(beanName, bean);

populateBean(bean); // 3. 属性填充
initializeBean(bean); // 4. 初始化
✅ Spring 的三级缓存 (架构优雅) 解耦 
// AbstractAutowireCapableBeanFactory.doCreateBean()

Object bean = createBeanInstance(); // 1. 实例化

// 2. 放入工厂，IOC 完全不知道 AOP 的存在
addSingletonFactory(beanName, () -> {
    // 延迟调用：只有发生循环依赖时才执行
    return getEarlyBeanReference(bean);
});

populateBean(bean); // 3. 属性填充 (可能触发工厂)
initializeBean(bean); // 4. 初始化

getEarlyBeanReference 的魔法

getEarlyBeanReference() 方法会遍历所有的 SmartInstantiationAwareBeanPostProcessor。
当 AOP 模块的 AbstractAutoProxyCreator 拦截到这个调用时，它会判断：“哦，现在需要提前暴露引用，那我就提前把代理对象创建出来返回”。
结论： 三级缓存的 Factory 只是一个“信使”，它把“是否创建代理”的决定权，优雅地交还给了 AOP 模块自己，没有破坏 IOC 容器的纯粹性。

终极总结：三级缓存的本质

延迟计算

将代理对象的创建尽可能推迟，保全正常 Bean 的生命周期完整性。

架构解耦

IOC 容器无需感知 AOP 的存在，通过 Factory 接口与后置处理器完美协作。

一致性保证

确保 B 注入的 A，与最终容器中的 A，在内存地址和代理状态上绝对一致。
