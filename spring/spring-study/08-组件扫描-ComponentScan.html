Part 1 · 基础认知

四大派生注解 — 功能完全相同，语义不同

注解语义额外能力
@Component通用 Spring Bean—
@Service业务逻辑层—（语义标记）
@Repository数据访问层自动翻译持久层异常（PersistenceExceptionTranslationPostProcessor）
@ControllerWeb 层被 RequestMappingHandlerMapping 识别为请求处理器

四个注解都被 @Component 元注解标注，所以扫描时一视同仁。@Repository 的"翻译持久层异常"是唯一有实际功能的派生注解。

@ComponentScan 常用配置

// 默认扫描启动类所在包及子包
@SpringBootApplication  // 内部包含 @ComponentScan

// 显式指定扫描范围
@ComponentScan(basePackages = {"com.example.service", "com.example.dao"})

// 类型安全扫描
@ComponentScan(basePackageClasses = {UserService.class, OrderService.class})

// 排除某些类
@ComponentScan(excludeFilters = @ComponentScan.Filter(
    type = FilterType.REGEX, pattern = "com.example.legacy.*"))

// 自定义包含规则
@ComponentScan(includeFilters = @ComponentScan.Filter(
    type = FilterType.ANNOTATION, classes = MyCustomAnnotation.class))

Part 2 · 触及灵魂的设计原理

🤔 假如没有组件扫描，怎么注册 Bean？

回到 XML 时代。每写一个 Service，都要在 XML 中加一行 <bean class="..." />。200 个 Service = 200 行 XML。新增一个类 → 改 Java 文件 + 改 XML 文件。重命名一个类 → 改 Java + 改 XML。两个事实来源，永远同步不上。

组件扫描把"注册 Bean"这件事从配置文件搬到了类本身。加一个 @Service → 自动注册。代码即配置，单一事实来源。

💡 设计巧思：ClassPathBeanDefinitionScanner 的扫描策略

巧思 1：默认扫描启动类所在包。Spring Boot 启动类通常放在项目根包（com.example.demo），这样所有子包自动被覆盖。不需要显式配置扫描路径——又是一个"约定优于配置"的实例。

巧思 2：excludeFilters 让你排除第三方的 @Component。如果引入了一个 jar，它的内部类也被 @Component 标注了，但你不想让它注册 → excludeFilters 可以按注解、正则、类名排除。

// ClassPathBeanDefinitionScanner 的核心逻辑（伪代码）
for (String basePackage : basePackages) {
    // 1. 扫描 classpath 下所有 .class 文件
    Set<BeanDefinition> candidates = findCandidateComponents(basePackage);
    for (BeanDefinition bd : candidates) {
        // 2. 检查 excludeFilters
        if (excludeFilters.match(bd)) continue;
        // 3. 检查 includeFilters
        if (includeFilters.match(bd)) register(bd);
    }
}
🔬 @Indexed — Spring 5 的编译期加速

组件扫描默认在运行时遍历 classpath——项目越大越慢。Spring 5 引入了 @Indexed 注解：编译期生成 META-INF/spring.components 索引文件，列出所有候选 Bean。扫描时直接读索引，不用遍历所有 class。

// spring.components 内容示例（编译期自动生成）
com.example.service.UserService=org.springframework.stereotype.Component
com.example.dao.UserRepository=org.springframework.stereotype.Repository

Spring Boot 2.3+ 默认启用了索引扫描。如果你的项目有 5000 个类但只有 200 个 @Component → 扫描时间从 2 秒降到 200 毫秒。

⚠️ 边界与陷阱

陷阱1：扫描范围过大。

@ComponentScan("com.example") 连第三方包一起扫了——启动暴增 10 秒。

陷阱2：同名 Bean 覆盖。

两个包下都有 UserService，后扫描的覆盖先扫描的——silent override，不报错。

陷阱3：@Component 和 @Bean 冲突。

同一个类既被 @Component 扫描，又被某个 @Configuration 的 @Bean 方法手动注册 → 两个 BeanDefinition，可能冲突。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Component 和 @Bean 的区别？什么时候用哪个？

@Component 标注在你自己写的类上，让扫描器自动发现。@Bean 标注在方法上，用于那些你不能或不该改源码的类（第三方库类、需要复杂初始化的对象）。一句话：自己能改源码用 @Component，不能改用 @Bean。

A
💡 基础题但很多人混淆——核心判据不是"哪个好"，是"这个类你能不能改"。
Q

@Service、@Repository 和 @Component 功能一样，为什么要分三个？

两个原因。① 代码可读性：扫一眼就知道这个类是 Controller 还是 Service。② @Repository 有特殊功能：PersistenceExceptionTranslationPostProcessor 会给 @Repository 的 Bean 创建代理，把 Hibernate/JPA 的异常翻译为 Spring 的 DataAccessException。

A
Q

Spring Boot 的默认扫描范围是什么？如果第三方包不在扫描范围怎么办？

默认扫描启动类所在包及子包。第三方包不在这个范围内 → 需要显式指定：① @ComponentScan(basePackages = {...}) 添加路径 ② 或者自己写 @Configuration + @Bean 手动注册 ③ 或者用 @Import 导入第三方配置类。

A
Q

如果两个包下都有 UserService.class 且都被 @Service 标注，容器中会发生什么？

如果它们用不同的 Bean 名（类名不同或手动指定了不同名字）→ 共存。如果 Bean 名相同 → 后扫描的覆盖先扫描的，不报错、不警告——silent override。这就是为什么生产会莫名其妙出现"明明写了 @Service 但注入的是另一个类"的故障。

A
Q

🔧 场景题：多数据源组件中，你有一个 DataSourceAutoConfiguration（自己写的，不是 Boot 的）被 @Configuration 标注。如何确保它被扫描到，又不把整个第三方包都扫进来？

最佳方案：不要扩大扫描范围。用 spring.factories（或 Boot 3.x 的 AutoConfiguration.imports）注册自动配置类。这样 Spring Boot 在自动配置阶段加载它，不需要扫描。或者用 @Import(MyDataSourceConfig.class) 在启动类上显式导入——精确可控，不引入多余的 Bean。

A
Q

🔧 场景题：智能拓客多渠道系统中，渠道适配器分布在多个子模块中。启动时发现部分渠道的 @Service 没被扫描到，怎么排查？

① 开启 debug 日志：logging.level.org.springframework.context=DEBUG 看扫描了哪些包。② 检查没有 @Service 的那些类所在的包是否在扫描范围内。③ 如果是多模块项目，确认子模块的 classpath 是否正确。④ 最终方案：要么调整 @ComponentScan 的 basePackages，要么把渠道适配器的注册方式从"扫描"改为"spring.factories 自动配置"——更可靠、不受包路径约束。

A
Q

@SpringBootApplication 的 exclude 属性和 @ComponentScan 的 excludeFilters 有什么区别？

exclude 排除的是自动配置类（AutoConfiguration），控制的是 @EnableAutoConfiguration 加载的类。excludeFilters 排除的是组件扫描发现的类。两个作用在不同的阶段：前者在自动配置加载时生效，后者在包扫描时生效。

A

📋 面试官总结

1. 四个 @Component 派生注解的功能差异 — 只有 @Repository 有额外功能（异常翻译）

2. @Component vs @Bean 的选择判据 — "能不能改源码"

3. 扫描范围与性能 — @Indexed 加速原理、扫描过大的后果

4. 同名 Bean 的 silent override — 没有报错，排查极其痛苦

⚠️ 大厂高频陷阱：exclude 属性排除的不是扫描的类；spring.factories 注册的 Bean 不受扫描范围影响
