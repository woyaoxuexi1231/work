Part 1 · 基础认知

三种注入方式速览

方式写法推荐度
构造器注入private final UserRepo repo;
public UserService(UserRepo repo){this.repo=repo;}		⭐⭐⭐ 首选
Setter 注入@Autowired public void setRepo(UserRepo r){this.repo=r;}⭐⭐ 可选依赖时
字段注入@Autowired private UserRepo repo;❌ 不推荐

与 @Autowired、@Resource 的关系

依赖注入是"怎么把依赖给到对象"，而 @Autowired / @Resource 是"标记哪些依赖需要注入"。两者正交：你可以用 @Autowired 做构造器注入，也可以做字段注入。

· @Autowired → 默认 byType，支持 required=false，支持集合注入，可配合 @Qualifier

· @Resource → JSR-250，先 byName 再 byType，不支持构造器注入

· @Inject → JSR-330，与 @Autowired 类似但更弱（无 required、无集合）

📎 关联你的项目

你在恒生多数据源组件中，必然要用构造器注入来接收 DataSource Map——因为你不能让多数据源的标识（lookup key）在运行时被意外修改。构造器注入的 final 不可变性正是这里的正确选择。

Part 2 · 触及灵魂的设计原理

🤔 假如只有字段注入，会发生什么？

痛点 1：不可测试。你要对 UserService 做单元测试：

// 字段注入 → 测试必须启动 Spring 或使用反射
@Autowired private UserRepo repo; // Mockito @InjectMocks 底层也是反射

// 构造器注入 → 直接 new，依赖一目了然
UserService service = new UserService(mockRepo); // ✅ 纯 Java，干净

痛点 2：隐藏依赖。一个类有 10 个 @Autowired 字段，你扫一眼根本不知道它依赖了什么。构造器参数列表就是依赖清单。

痛点 3：不可变性丧失。字段注入不能用 final，依赖可能在中途被替换（无论是有意还是 bug）。

💡 设计巧思：为什么允许单构造器省略 @Autowired？

Spring 4.3（2016年）做了这个改进。逻辑很简单：

// 旧写法（Spring 4.2 及以前）
@Autowired
public UserService(UserRepo repo) { this.repo = repo; }

// 新写法（Spring 4.3+）
public UserService(UserRepo repo) { this.repo = repo; }
// 如果只有一个构造器 → Spring 自动用它注入，@Autowired 可省略

设计者的"啊哈时刻"：如果一个类只有一个构造器，那么创建它的唯一方式就是调用那个构造器——还需要标注 @Autowired 来告诉 Spring 用哪个吗？不需要。这消除了注解冗余，让代码更接近"纯 Java"。

如果有多个构造器，则必须至少标注一个 @Autowired，否则 Spring 不知道该选哪个。

🔬 底层机制：AutowiredAnnotationBeanPostProcessor

依赖注入不是魔法——它由 AutowiredAnnotationBeanPostProcessor（一个 BPP）在 Bean 生命周期的"属性填充"阶段完成：

// doCreateBean() 内部流程：
1. createBeanInstance()      → 实例化（调构造器）
2. populateBean()            → 属性填充 ← 这里执行注入！
   ├─ InstantiationAwareBeanPostProcessor.postProcessProperties()
   │  └─ AutowiredAnnotationBeanPostProcessor
   │     → 扫描 @Autowired / @Value 字段和方法
   │     → beanFactory.resolveDependency() 递归解析依赖
   │     → 反射 set 值
   └─ (Setter 注入也是在这里，因为是 populateBean)
3. initializeBean()          → 初始化（@PostConstruct 等）

构造器注入在哪？更早——在步骤 1 createBeanInstance() 中，Spring 推断构造器时就解析了构造器参数依赖。这意味着构造器注入在属性填充之前就完成了。

为什么这个顺序很重要？因为构造器注入先于属性填充 → 构造器里的代码可以放心使用注入的依赖；而 @PostConstruct 在属性填充之后 → 也安全。只有一种情况危险：在构造器里期望某个字段注入的依赖已经可用——那是不可能的。

⚠️ 边界与陷阱

陷阱 1：构造器注入 + 循环依赖 = 💥

构造器注入在实例化阶段就需要依赖，但被依赖的 Bean 可能还没出生。三级缓存只能救字段/Setter 注入的循环依赖，救不了构造器注入。

陷阱 2：@Autowired(required = false) 的 null 地狱

@Autowired(required = false)
private PaymentService paymentService; // 可能为 null
// 之后到处 if(paymentService!=null) → 代码变丑

替代方案：ObjectProvider<PaymentService> 或 Optional<PaymentService>。

陷阱 3：Setter 注入的顺序不确定性

如果两个 Setter 有隐式依赖关系（setA 必须在 setB 之前调用），Spring 不保证调用顺序——这种行为是脆弱的。用构造器注入消除这个隐患。

Part 3 · 大厂面试官的灵魂追问

Q

构造器注入、Setter 注入、字段注入，你推荐哪种？为什么？

构造器注入。三个理由：① 依赖可以声明为 final，不可变；② 纯 Java 测试，直接 new 就行；③ 依赖列表一清二楚，不会出现"这个类依赖了什么得翻半天"的情况。字段注入唯一的优点就是代码短，但这是一个陷阱。

A
💬 面试官内心戏：80% 的人能说出"构造器注入好"，但只有 20% 能说清"为什么"。提到 final、可测试性、依赖显式化——这是及格线。
Q

既然构造器注入这么好，为什么还有人在用字段注入？

两个原因。历史原因：早期 Spring 教程（包括官方文档的示例）大量使用字段注入，形成了惯性。书写简便：@Autowired 一行搞定，不用写构造器。但这两个"优点"的代价是代码质量下降——所以 IDEA 会给你加警告。

A
💬 追问意图：考察你是否理解技术选型背后的"人性因素"——一项技术好不好，和它流不流行，不完全是一回事。
Q

有一个 Service 有 8 个依赖，构造器参数列表特别长，你怎么看？

这不是注入方式的问题，是类设计的问题——这个类违反了单一职责原则。8 个依赖意味着这个 Service 做了太多事情，应该拆分为 3-4 个更专注的 Service。如果用字段注入，这个问题就被"藏起来"了；构造器注入把问题暴露出来，逼你重构——这是设计上的好处。

A
💬 考核意图：能不能从"技术选择"上升到"设计判断"。好的技术选择会暴露坏设计，坏的技术选择会隐藏坏设计。
Q

回到你的多数据源项目：如果主从切换的逻辑需要拿到 DataSource Map 的同时还要拿到切换策略，你会怎么设计注入？

用构造器注入 Map<String, DataSource> 保证不可变；同时注入一个 RoutingStrategy 接口（策略模式）。构造器的参数列表里，DataSource Map 是第一参数（核心依赖），策略是第二参数（行为依赖）。如果有人想加新的切换策略，只需实现 RoutingStrategy 并通过配置选择——不用改任何注入点。
