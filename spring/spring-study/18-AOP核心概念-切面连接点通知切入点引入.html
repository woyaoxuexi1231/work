Part 1 · 五个术语一张表

术语类比含义
Aspect 切面菜谱通知+切点的集合（@Aspect 标注的类）
Join Point 连接点厨房里所有食材程序执行中所有可插入的点（Spring AOP 中=方法调用）
Pointcut 切入点"哪些菜适用"从所有连接点中筛选出目标方法（表达式匹配）
Advice 通知炒菜步骤做什么+何时做（@Before/@After 等）
Introduction 引入给菜加新配料让目标类多实现一个接口（@DeclareParents）

一句话：切面 = 通知 + 切入点。通知决定"做什么+何时做"，切入点决定"在哪里做"。连接点是所有候选点，切入点是筛选后的。

Part 2 · 触及灵魂

🤔 为什么要把"在哪里做"和"做什么"拆成两个概念？

如果切入点和通知不分离，一个 @Before 只能绑定一个方法 = 100 个方法需要 100 个 @Before。分离后，一个切点表达式匹配 100 个方法 + 一个 @Before = 一个切面搞定。切入点是可复用的匹配规则，通知是可复用的增强逻辑。

💡 设计巧思：织入（Weaving）— Spring 的代理织入 vs AspectJ 的编译织入

织入就是把切面"弄到"目标对象上的动作。Spring 用运行时代理（创建代理对象拦截方法），AspectJ 用编译期/类加载期织入（直接修改字节码）。Spring 的方案更简单（零配置），但有限制（不能增强 private/final）。

⚠️ 边界

Spring AOP 的连接点只限方法调用——不能拦截字段访问、构造器调用。AspectJ 可以，但需要编译器支持。

Part 3 · 面试追问（6 组）

Q

用一句话解释切面、连接点、通知、切入点。

切面是"类"，包含通知和切入点；通知是"做什么+何时做"；切入点是"在哪里做"；连接点是"所有可选的地方"。切面 = 通知 + 切入点。

A
Q

织入（Weaving）有几种方式？Spring 用哪种？

三种：编译期（ajc 编译器）、类加载期（LTW）、运行期代理（Spring 方式）。Spring 用运行期 CGLIB/JDK 代理——不需要额外编译器，但只能拦截方法调用。

A
Q

引入（Introduction）和通知的本质区别？

通知增强行为（在方法前后加逻辑），引入改变类型（让 Bean 多实现一个接口）。前者不改变类结构，后者改变。

A
Q

JoinPoint 和 ProceedingJoinPoint 什么关系？

ProceedingJoinPoint 继承 JoinPoint，多了一个 proceed() 方法。@Before/@After 等用 JoinPoint（只能获取元数据），@Around 用 ProceedingJoinPoint（能获取元数据+执行目标方法）。

A
Q

🔧 场景题：你要给一个类加"审计"能力（记录谁在什么时候做了什么）。但你不能改这个类的源码。用 AOP 的什么概念？

两种方案：① 用通知（@Before/@Around）拦截方法调用，记录参数和调用者。不改类结构，只增强行为。② 用引入（@DeclareParents）让类多实现一个 Auditable 接口，外部可以((Auditable)bean).getAuditLog()。方案①更简单，方案②更类型安全。

A
Q

Spring 为什么只支持方法级 AOP？

因为 Spring AOP 基于代理——代理只能拦截方法调用。字段访问、构造器调用发生在代理对象内部，无法被代理拦截。AspectJ 编译期织入可以做到，但需要 ajc 编译器。Spring 选择了"牺牲能力换零配置"。

A

📋 面试官总结

1. 五个术语的关系 — 切面⊃通知+切入点；连接点⊃切入点

2. 织入方式 — 运行时代理（Spring）vs 编译期（AspectJ）

3. JoinPoint vs ProceedingJoinPoint — 是否有 proceed()

4. Spring AOP 的限制 — 只能拦截方法调用，不能拦截字段/构造器
