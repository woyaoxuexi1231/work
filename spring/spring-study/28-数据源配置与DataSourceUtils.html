🔌 数据源配置与 DataSourceUtils

dataSource.getConnection() vs DataSourceUtils.getConnection() — 事务感知 vs 每次新连接

Part 1Part 2Part 3

Part 1 · 两个获取连接的方式

方法行为适用
ds.getConnection()每次返回新连接非事务场景
DataSourceUtils.getConnection(ds)先查 ThreadLocal 有无绑定连接→有则复用→无则新建并绑定事务场景

Part 2 · 触及灵魂

🤔 不用 DataSourceUtils 会出什么问题？

@Transactional 方法内：JdbcTemplate 自动用 DataSourceUtils 获取连接→同一线程内所有操作共享同一连接。如果你手动调 ds.getConnection()→拿到不同连接→不在同一事务中，回滚不生效。

💡 ThreadLocal 怎么绑定连接？
// TransactionSynchronizationManager 内部
ThreadLocal<Map<Object,Object>> resources; // key=DataSource, value=ConnectionHolder
// DataSourceUtils 先查这个 Map→有就复用→没有就用 ds.getConnection() 新建并放入
⚠️ 陷阱

非事务方法中 DataSourceUtils.getConnection() 也会获取新连接——只有活跃事务期间才会复用。事务结束后连接从 ThreadLocal 移除。

Part 3 · 面试追问（6 组）

Q

DataSourceUtils.getConnection() 和 ds.getConnection() 区别？

DataSourceUtils 先查 ThreadLocal 有没有绑定连接→有则复用（事务内共享同一连接），无则新建。ds.getConnection() 每次都返回新连接。

A
Q

为什么 JdbcTemplate 自动用 DataSourceUtils？

为了保证事务一致性。JdbcTemplate 内部写死了用 DataSourceUtils——不需要开发者手动选择。这就是"框架替你做了正确的事"。

A
Q

🔧 场景题：多数据源场景——主库和从库的事务怎么隔离？

ThreadLocal 的 Map 中 key 是 DataSource 实例——不同的 DataSource 绑定不同的连接。主库事务和从库事务天然隔离。DataSourceUtils.getConnection(masterDs) 和 DataSourceUtils.getConnection(slaveDs) 返回不同连接。

A
Q

连接什么时候从 ThreadLocal 移除？

事务提交/回滚后，PlatformTransactionManager 调 doCleanupAfterCompletion() → 解绑并关闭连接。非事务场景下，每次操作后立即释放。

A
Q

HikariCP 连接池和 DataSourceUtils 怎么配合？

HikariCP 管理连接池的创建和回收（物理连接复用）。DataSourceUtils 管理事务内的连接共享（逻辑连接复用）。两层不冲突——HikariCP 负责"连接从哪来"，DataSourceUtils 负责"同一事务用同一连接"。

A
Q

直接用 ds.getConnection() 在 @Transactional 方法里会怎样？

拿到一个独立连接——不在事务中，setAutoCommit(true)，回滚无效。如果这个连接做了写操作→立即提交→无法回滚。这是生产数据不一致的常见原因。

A

📋 面试官总结

1. DataSourceUtils — 事务感知的连接获取

2. ThreadLocal 绑定 — DataSource→Connection 映射

3. 多数据源天然隔离 — 不同 DataSource 不同 key

4. 手动 ds.getConnection() 在事务中=灾难
