🧩 引入：@DeclareParents

让 Bean 凭空多实现一个接口 — 不改源码的"外挂"能力

Part 1Part 2Part 3

Part 1 · @DeclareParents 是什么

普通 AOP 是增强行为（方法前后加逻辑）。引入更进一步：让 Bean 多实现一个接口——不需要改原类的代码，像外挂一样贴上去。

// 1. 定义接口和默认实现
public interface Auditable { String getAuditLog(); }
public class DefaultAuditable implements Auditable {
    public String getAuditLog() { return "audit: ..."; }
}

// 2. @DeclareParents 引入
@Aspect @Component
public class AuditableIntroduction {
    @DeclareParents(value = "com.example.service.*+",   // 目标
                    defaultImpl = DefaultAuditable.class)
    private Auditable auditable;  // 占位字段
}

// 3. 使用 — 任何 Service 都能强转为 Auditable
@Autowired private UserService userService;
String log = ((Auditable) userService).getAuditLog(); // ✅

Part 2 · 触及灵魂

🤔 如果没有引入，50 个 Service 要加审计能力怎么做？

要么所有 Service 都实现 Auditable（改 50 个类），要么写抽象父类（改继承关系），要么组合模式（每个 Service 加字段）。无论哪种都要碰源码。引入让你零改动给 50 个 Service 贴上一个新接口。

💡 设计巧思：代理同时实现多个接口

JDK 代理本来就支持多接口——Proxy.newProxyInstance 可以传多个接口。引入本质就是让代理多实现一个接口，所有对该接口方法的调用都被转发到 defaultImpl。CGLIB 同理。

⚠️ 边界

只能引入接口，不能引入类（多继承限制）。defaultImpl 必须有无参构造器。被引入的接口方法不能与目标类已有方法重名（冲突时目标类优先）。

Part 3 · 面试追问（6 组）

Q

@DeclareParents 和 @Before 的本质区别？

@Before 增强行为（方法执行前后加逻辑），@DeclareParents 改变类型（让 Bean 多实现接口）。前者是"让方法多做一件事"，后者是"让 Bean 多一种身份"。

A
Q

JDK 和 CGLIB 代理下引入行为一样吗？

一样。JDK 代理天然支持多接口，CGLIB 通过生成子类也可以让代理多实现一个接口。行为一致。

A
Q

🔧 场景题：30 个 Service 需要给监控系统暴露 healthCheck() 方法，但不能改 Service 代码。怎么做？

用 @DeclareParents。定义 HealthCheckable 接口+默认实现（查连接池、Redis 连通性）。一个切面引入到所有 Service。监控系统遍历容器中所有 Bean，if(b instanceof HealthCheckable) 调 healthCheck()。30 个 Service 零改动。

A
Q

defaultImpl 必须有无参构造器吗？为什么？

是的。Spring 在创建代理时需要实例化 defaultImpl，它通过 Class.newInstance()（反射无参构造器）创建。如果 defaultImpl 只有有参构造器→实例化失败。

A
Q

引入的接口方法和目标类同名怎么办？

目标类优先。代理本身是目标类的子类（CGLIB）或接口实现（JDK），目标类已有的方法不会被引入覆盖。这是个潜在坑——你以为引入了新方法，但实际上目标类已经有了同名方法。

A
Q

引入和装饰器模式有什么区别？

装饰器需要手动为每个类写 Wrapper——N 个类 = N 个装饰器。引入只需要一个切面——N 个类一次搞定。引入是批量装饰，装饰器模式是单类装饰。

A

📋 面试官总结

1. 引入 = 批量装饰器模式 — 改类型 vs 增强行为

2. defaultImpl 无参构造器 — 反射实例化要求

3. 同名方法目标类优先 — 引入不覆盖已有方法
