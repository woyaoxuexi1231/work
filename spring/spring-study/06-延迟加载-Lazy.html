Part 1 · 基础认知

@Lazy 的三种使用方式

位置写法效果
Bean 定义上@Component @Lazy该 Bean 只在首次使用时创建
@Bean 上@Bean @Lazy该 @Bean 延迟创建
注入点@Autowired @Lazy注入一个延迟代理，首次调用方法时才真正获取
全局spring.main.lazy-initialization=true所有 Bean 默认延迟（除了显式标注 @Lazy(false) 的）
// 1. 标注在 Bean 上
@Service @Lazy
public class ExpensiveService { }  // 启动时不会创建

// 2. 标注在注入点（解决循环依赖的另一种方式）
@Service
public class UserService {
    @Autowired @Lazy
    private OrderService orderService;  // 注入的是代理，不是真实对象
}

// 3. 全局开启
// application.yml: spring.main.lazy-initialization=true

Part 2 · 触及灵魂的设计原理

🤔 假如没有 @Lazy，会发生什么？

痛点：启动时间线性增长。Spring 默认是"饿汉式"——启动时把所有单例 Bean 全部创建好。100 个 Bean 启动 2 秒，5000 个 Bean 启动 1 分钟。大型微服务项目启动 3-5 分钟是常态。

更隐蔽的痛点：灰度发布时，服务器需要快速切换流量——如果启动 5 分钟，这 5 分钟要么没流量（浪费），要么有流量但 Bean 还没就绪（报错）。

💡 设计巧思：@Lazy 注入点 = 代理假装存在

@Lazy 在注入点上的实现非常巧妙：Spring 不创建真实 Bean，而是注入一个 CGLIB 代理。代理在每次方法调用时，才去容器里拿真实 Bean（此时触发创建）。

// 你写的
@Autowired @Lazy
private ExpensiveService service;

// Spring 实际注入的是：
// new LazyProxy(() -> applicationContext.getBean(ExpensiveService.class))
// 只有当你调 service.doSomething() 时，代理才去真正创建 Bean

"啊哈时刻"：这不只是"晚点创建"——它还能打破循环依赖。A 依赖 B、B 依赖 A，如果 A 对 B 的注入用了 @Lazy，A 在实例化时不需要"此刻就有 B"——它拿到的是一个代理。等 B 真正需要调用 A 时，A 已经创建好了。

🔬 底层机制 
// ContextAnnotationAutowireCandidateResolver.getLazyResolutionProxyIfNecessary()
// 如果注入点标注了 @Lazy → 返回代理对象，而不是真实 Bean

Object lazyProxy = ProxyFactory.getProxy(TargetClass.class, 
    (method, args) -> {
        Object target = beanFactory.getBean(beanName); // 首次调用时才创建
        return method.invoke(target, args);
    });
// 这个代理会被注入到字段中
⚠️ 边界与陷阱

陷阱1：启动报错推迟到运行时

饿汉模式下，Bean 创建失败 → 启动失败，立即知道。懒加载下，创建推迟到首次调用 → 用户请求报错才发现。

陷阱2：第一个请求极慢

如果大量 Bean 都是懒加载，第一个 HTTP 请求可能触发几十个 Bean 的首次创建——请求超时。

陷阱3：@ConditionalOnMissingBean 可能误判

懒加载下 Bean 还没创建 → @ConditionalOnMissingBean 认为"不存在" → 自动配置创建了重复 Bean。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Lazy 标注在 Bean 上和标注在注入点上有区别吗？

有本质区别。标注在 Bean 上 → 容器不会主动创建它，只有 getBean() 时才创建。标注在注入点上 → 不改变 Bean 本身的创建策略，只是注入一个代理；如果其他地方也注入这个 Bean 且没有 @Lazy，它仍然会在启动时被创建。

A
Q

spring.main.lazy-initialization=true 全局开了之后有什么副作用？

三个副作用。① 启动时的 Bean 创建异常推迟到运行时才发现（本来启动就报错的现在用户请求才报错）。② 第一个请求可能成为"请求炸弹"——大量 Bean 同时首次创建，响应时间爆炸。③ @ConditionalOnMissingBean 条件判断可能出错，因为 Bean 还没创建。

A
Q

@Lazy 能解决构造器注入的循环依赖吗？

能。A 的构造器参数 B 标注了 @Lazy → Spring 不会尝试立即获取 B，而是注入一个代理。A 的构造器成功调用后，B 的代理在后续某个时机才触发 B 的真实创建——此时 A 已经在容器中了。

A
Q

🔧 场景题：智能拓客多渠道改造中，有一个"渠道适配器工厂"依赖 5 种不同渠道的 Bean（短信、邮件、Push 等），但每次请求只用到其中一个。怎么用 @Lazy 优化启动？

渠道适配器工厂用 Map<String, ChannelAdapter> 注入所有渠道，但每个 ChannelAdapter 标注 @Lazy。这样启动时只创建工厂本身和代理——5 个渠道的实际 Bean 不会创建。只有第一次用到某个渠道时（比如短信），该渠道的 Bean 才真正初始化。内存和启动时间都大幅减少。

A
💡 考核意图：不是问"@Lazy 怎么用"，而是考察"在什么架构场景下用 @Lazy 最契合"。
Q

🔧 场景题：多数据源中，从库的 DataSource 希望在真正有读请求时才建立连接（避免启动时从库不可用导致整体启动失败）。怎么设计？

从库 DataSource 标注 @Lazy。主库 DataSource 保持饿汉（因为启动就需要）。读请求第一次到达时，从库 DataSource 才建立连接池——如果此时从库不可用，抛出异常但不影响主库已有的连接。配合健康检查和告警，运维可以收到告警后修复从库，而不影响线上服务。

A
Q

懒加载的 @PostConstruct 什么时候执行？

Bean 第一次被使用时（第一次 getBean 或第一次通过代理调用方法时）。不在容器启动时。这意味着：如果你的 @PostConstruct 里有关键初始化（比如加载密钥），这个初始化被推迟了——第一次请求会变慢。

A
Q

@Lazy 和 @DependsOn 同时使用会怎样？

@DependsOn 胜出。@Lazy 说"推迟创建"，@DependsOn 说"必须先创建 B 再创建我"。如果 A 是 @Lazy，但被另一个 Bean 通过 getBean() 触发创建，此时 @DependsOn 仍然生效——先创建 B，再创建 A。@Lazy 只是推迟了触发时机，没有改变创建流程。

A

📋 面试官总结：@Lazy 高频考点

1. 三个使用位置的区别 — Bean上、注入点上、全局配置的行为不同

2. 懒加载的副作用 — 运行时报错、首请求慢、@ConditionalOnMissingBean 误判

3. 打破循环依赖 — @Lazy 是除了三级缓存之外的另一种循环依赖解决方案

4. 与 @PostConstruct 的交互 — 初始化被推迟，关键路径受影响

⚠️ 大厂高频陷阱：全局懒加载在生产环境默认不开——Spring Boot 官方文档建议只在开发环境使用。
