Part 1 · 基础认知

声明式 vs 编程式事务

维度声明式 @Transactional编程式 TransactionTemplate
写法注解代码块
事务边界方法级别精确到代码块
适用场景95% 的场景需要精细控制事务边界
底层AOP @Around模板方法模式
// TransactionTemplate — 编程式事务
@Autowired private TransactionTemplate txTemplate;

public void batchProcess(List<Order> orders) {
    txTemplate.execute(status -> {
        for (Order o : orders) {
            orderMapper.insert(o);
            if (o.getAmount() > 10000) {
                status.setRollbackOnly();  // 手动标记回滚
                return null;
            }
        }
        return null;
    });
}

// TransactionSynchronizationManager — 注册事务回调
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
    @Override public void afterCommit() {
        mqTemplate.send(orderMessage);  // 事务提交后才发 MQ
    }
});

Part 2 · 触及灵魂

🤔 什么时候必须用编程式事务？

三个场景声明式不够用：

① 事务边界小于方法：方法前 50 行不需要事务，后 10 行需要——@Transactional 只能包整个方法。

② 批量处理中部分失败继续：100 条记录，每条独立事务，失败 3 条不影响其余 97 条。

③ 运行时动态决定传播行为：根据配置或请求参数决定用 REQUIRED 还是 REQUIRES_NEW。

💡 设计巧思：TransactionSynchronizationManager — ThreadLocal 的"事务黑板"

它像一个挂在线程上的"事务状态公告板"：

// TransactionSynchronizationManager 的核心 ThreadLocal 变量
ThreadLocal<Map<Object, Object>> resources;       // DataSource → ConnectionHolder
ThreadLocal<Set<Synchronization>> synchronizations; // 事务提交/回滚后的回调
ThreadLocal<String> currentTransactionName;          // 当前事务名
ThreadLocal<Boolean> readOnly;                       // 是否只读
ThreadLocal<Integer> isolationLevel;                 // 隔离级别
ThreadLocal<Boolean> actualTransactionActive;        // 是否有活跃事务

这就是为什么：① 同一线程共享事务（所有变量都是 ThreadLocal）。② @Async 拿不到事务（换了线程）。③ 你可以在任何地方通过静态方法判断"当前是否有事务"。

💡 设计巧思：TransactionSynchronization — 事务生命周期钩子

@TransactionalEventListener 的底层就是靠它：

// 注册一个回调：事务提交后执行
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
    void afterCommit()  { /* 发 MQ */ }
    void afterCompletion(int status) { /* 清理资源 */ }
    void beforeCommit(boolean readOnly) { /* 提交前 */ }
    void beforeCompletion() { /* 完成前 */ }
});
⚠️ 陷阱

没有事务时 registerSynchronization 抛异常

必须在活跃事务中注册。

afterCommit 中抛异常不影响主事务

事务已提交，回调异常只会打日志。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

TransactionTemplate 和 @Transactional 底层是不是同一个东西？

是。两者最终都调用 PlatformTransactionManager.getTransaction() / commit() / rollback()。TransactionTemplate 是编程式入口，@Transactional 通过 TransactionInterceptor（AOP）调用同样的 API。一个代码里写，一个注解里写。

A
Q

TransactionSynchronizationManager 为什么用 ThreadLocal？

因为事务是线程绑定的——一个 HTTP 请求在一个线程中执行，所有 Service 调用共享同一线程。ThreadLocal 是最自然的绑定方式：不需要传参、不需要全局 Map 加锁、不需要上下文对象。这是把 JVM 的线程模型和数据库的事务模型对齐的经典设计。

A
Q

🔧 场景题：批量导入 1 万条数据，要求每条数据独立事务（一条失败不影响其他）。怎么写？

用 TransactionTemplate，在循环内 execute()——每条记录独立事务。不能用 @Transactional 包整个方法（那就是一个事务包 1 万条）。注意：① 不要在循环内注入 TransactionTemplate 依赖——已经是 Bean。② 失败的记录记录到 error list 并 continue。

A
Q

🔧 场景题：订单创建后要发 MQ 通知下游系统。但如果 MQ 发送在事务提交之前——下游系统可能收到消息但订单还没提交。怎么保证一致性？

用 TransactionSynchronizationManager.registerSynchronization() 注册 afterCommit 回调。事务提交后才执行——此时订单一定在数据库中。如果 MQ 必达性要求高 → 配合本地消息表（在事务内插入一条"待发送"消息记录 + afterCommit 触发实际发送 + 定时任务补偿未发送的）。

A
Q

status.setRollbackOnly() 和抛异常回滚有什么区别？

setRollbackOnly() 不抛异常——方法继续执行，方法结束后事务回滚。适合"我知道这个操作失败但不希望中断调用链"的场景。抛异常会中断方法执行。性能上没区别。setRollbackOnly 的优势是语义更明确——"这个事务应该回滚"vs "这里出错了"。

A
Q

afterCommit 回调里抛了异常会怎样？

事务已经提交了——回调异常不影响数据。但后续注册的回调可能不会执行。Spring 会 catch 异常并打日志，不会抛给调用方。如果你的 afterCommit 逻辑需要可靠性（如发 MQ），自己在回调里 try-catch + 重试。

A
Q

TransactionSynchronizationManager 有 MODE_INHERITABLETHREADLOCAL 吗？和 InheritableThreadLocal 什么关系？

没有 MODE_INHERITABLETHREADLOCAL——那是 SecurityContextHolder 的。TransactionSynchronizationManager 用的是普通 ThreadLocal，子线程天然继承不到。InheritableThreadLocal 可以让子线程继承父线程的值，但 Spring 没用它——因为 ① 事务通常不应该跨线程传播 ② 线程池的线程是复用的，InheritableThreadLocal 只在创建时继承一次。

📋 面试官总结 · 数据访问与事务章节完结

1. TransactionTemplate — 精确控制事务边界，批量独立事务的利器

2. TransactionSynchronizationManager — ThreadLocal 是事务隔离的物理基础

3. TransactionSynchronization — 事务生命周期钩子，@TransactionalEventListener 的底层

4. afterCommit 发 MQ — 事务提交后才通知下游，配合本地消息表保证可靠性
