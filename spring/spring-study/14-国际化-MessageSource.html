Part 2 · 触及灵魂

🤔 假如不用 MessageSource，国际化怎么做？

用 JDK 的 ResourceBundle：

// 原始方式：每个地方手写 ResourceBundle 调用
ResourceBundle bundle = ResourceBundle.getBundle("messages", locale);
String msg = bundle.getString("user.notfound");

痛点：每条消息都要获取 bundle → 代码中到处是 ResourceBundle 样板代码。和 Spring IoC 完全割裂——ResourceBundle 不是 Bean，不能被注入，不能被 AOP 拦截。

💡 设计巧思：MessageSource 就是"消息的 BeanFactory"

Spring 的设计思路一以贯之：把一切抽象为 Bean，纳入容器管理。MessageSource 是一个特殊的 Bean（名字固定为 "messageSource"），被 ApplicationContext 自动识别。

// ApplicationContext.getMessage() 内部委托给 MessageSource Bean
public String getMessage(String code, Object[] args, String defaultMessage, Locale locale) {
    return getMessageSource().getMessage(code, args, defaultMessage, locale);
}

"啊哈时刻"：这意味着你可以用 Spring 的一切能力来增强 MessageSource——比如从数据库加载消息（用 JdbcTemplate 实现 MessageSource）、从 Redis 加载、加缓存切面。它不再是一个静态工具类，而是一个可以被代理、被替换的 Bean。

🔬 ReloadableResourceBundleMessageSource 的缓存与刷新 
// 内部用 ConcurrentHashMap 缓存 PropertiesHolder
// key = filename + locale, value = Properties + 最后修改时间
// 每次 getMessage() 时检查文件修改时间 → 变了就重新加载
// cacheSeconds = -1（默认）= 永久缓存；0 = 每次检查；正数 = 秒级过期
⚠️ 边界：MessageSource 的"不存在"处理

陷阱：

getMessage(code, args, locale) 如果找不到 → 直接抛 NoSuchMessageException。必须用 getMessage(code, args, defaultMessage, locale) 来兜底。

Spring Boot 默认配置：

spring.messages.basename=messages。默认查找 messages.properties。

Part 3 · 大厂面试官的灵魂追问（6 组）

Q

MessageSource 和 ResourceBundle 的区别？为什么不直接用 JDK 的？

MessageSource 是 ResourceBundle 的 IoC 化包装。核心区别：它是一个Spring Bean，可以被注入、被代理、被替换为数据库实现。ResourceBundle 只是一个静态工厂方法。Spring 的设计哲学是"一切皆 Bean"——MessageSource 只是其中一个例子。

A
Q

ReloadableResourceBundleMessageSource 和 ResourceBundleMessageSource 的区别？

Reloadable 支持热加载——改了 properties 文件不用重启。内部用 PropertiesHolder + 文件修改时间判断是否需要重新加载，cacheSeconds 控制检查频率。生产环境建议用 Reloadable：产品经理说"这个词改一下"，你能直接改文件生效。

A
Q

怎么实现一个从数据库加载消息的 MessageSource？

实现 MessageSource 接口：getMessage() 中先查缓存，没有就用 JdbcTemplate 查数据库。数据库表结构：code + locale + message_text。注意要加本地缓存（Caffeine/Guava Cache）避免每次请求都查库。然后注册为名为 "messageSource" 的 Bean。

A
Q

🔧 场景题：线上订单状态中文显示"待支付"，产品要求紧急改成"等待付款"。用了 ResourceBundleMessageSource，怎么改最快？

ResourceBundleMessageSource 不支持热加载 → 必须重启。最快的临时方案：如果消息不多，先切到 ReloadableResourceBundleMessageSource（改一个配置），然后改 properties 文件即可生效。长期方案：所有国际化配置都用 Reloadable，cacheSeconds 设 60，生产应急不用重启。

A
Q

🔧 场景题：微服务中有 10 个服务都用同一套国际化文本。现在每个服务都在自己的 classpath 下维护一份 messages_zh_CN.properties，每次改一个词要改 10 个文件。怎么统一管理？

方案一：自定义 MessageSource 从配置中心（Nacos/Apollo）读取，表结构 code+locale+text。所有服务共享这个实现。方案二：把 messages.properties 放在独立的 common jar 包中，所有服务依赖这个 jar，classpath 自动包含。方案三（最轻量）：Redis 存消息文本，自定义 MessageSource 从 Redis 读取。三种方案的共性：实现 MessageSource 接口，利用 IoC 的可替换性。

A
Q

MessageSource 的 Bean 名为什么必须是 "messageSource"？

因为 AbstractApplicationContext.initMessageSource() 中硬编码了 beanName = "messageSource"。它先按类型找，如果容器中有这个名就叫 "messageSource" 的 Bean 就用它；如果没有就创建一个默认的 DelegatingMessageSource（空实现，返回默认值）。这是 Spring 的"惯例优于配置"——名字约定好了就不需要额外配置。

A

📋 面试官总结

1. MessageSource 是 ResourceBundle 的 IoC 包装 — 核心价值是"可替换"

2. Reloadable vs ResourceBundle — 热加载是生产必备能力

3. 自定义实现 — 从数据库/Redis/配置中心加载，实现 MessageSource 接口

4. Bean 名约定 — "messageSource" 是硬编码约定

⚠️ 高频陷阱：getMessage 不带 defaultMessage 参数会抛异常；Spring Boot 默认 Reloadable
