Part 1 · 基础认知

引入（Introduction）— 给 Bean "加接口"

普通的 AOP 是在方法前后加逻辑。引入更进一步：让一个 Bean 凭空多实现一个接口。不需要修改原类的代码。

// 1. 定义一个新接口和默认实现
public interface Auditable { void audit(); }
public class DefaultAuditable implements Auditable {
    @Override public void audit() { log.info("审计记录"); }
}

// 2. 用 @DeclareParents 把 Auditable 引入到所有 Service
@Aspect @Component
public class AuditableIntroduction {
    @DeclareParents(
        value = "com.example.service.*+",  // 目标：所有 Service
        defaultImpl = DefaultAuditable.class
    )
    private Auditable auditable;  // 这个字段只是占位，Spring 会替换为代理
}

// 3. 使用 — 任何 Service 都能强转为 Auditable
@Autowired private UserService userService;
((Auditable) userService).audit();  // ✅ UserService 没实现 Auditable，但能调 audit()！

Part 2 · 触及灵魂

🤔 如果不用引入，想让 50 个 Service 都有审计能力怎么做？

方案 A：让 50 个 Service 都实现 Auditable 接口 → 改 50 个类。方案 B：写一个抽象父类 → 50 个 Service 都要改继承。方案 C：用组合模式 → 每个 Service 里加一个 Auditable 字段。无论哪种，都至少要碰代码。

引入让你完全不碰 Service 代码，像一个外挂接口一样贴上去。

💡 设计巧思：exposeProxy 解决自调用

exposeProxy=true 把当前代理对象放到 ThreadLocal 中。同一个 Service 内部通过 AopContext.currentProxy() 就能拿到代理：

// 自调用失效的解法之一
((UserService) AopContext.currentProxy()).transactionalMethod();

代价：每个请求都要往 ThreadLocal 里写一次代理引用，有小额性能开销。而且引入了对 Spring API 的依赖（不纯粹）。

🔬 AbstractAutoProxyCreator 在生命周期中的位置

在 Bean 生命周期的 postProcessAfterInitialization 阶段执行。此时 Bean 已经初始化完成（@PostConstruct 已执行）。AbstractAutoProxyCreator 检查这个 Bean 是否需要代理（匹配切点）→ 需要就创建代理 → 返回代理替代原始 Bean。

Part 3 · 大厂面试官的灵魂追问（6 组）

Q

@DeclareParents 是做什么的？和普通 @Before 有什么区别？

@Before 是在方法执行前后加逻辑——增强行为。@DeclareParents 是让目标类多实现一个接口——改变类型。前者是"让方法多做一件事"，后者是"让 Bean 多一种身份"。

A
Q

exposeProxy=true 有什么代价？什么时候必须开？

代价：每个代理对象被请求时都要往 ThreadLocal 写一次引用，有小量性能开销；代码里出现 AopContext.currentProxy() 是 Spring 框架耦合。必须开的场景：自调用需要触发 AOP（this.method 不经过代理）。但最佳实践是拆类，不依赖 exposeProxy。

A
Q

@EnableAspectJAutoProxy 在 Spring Boot 中需要手动加吗？

不需要。Spring Boot 的 AopAutoConfiguration 自动配置了它。proxyTargetClass 默认为 true。只有在非 Boot 的纯 Spring 项目中才需要手动加。

A
Q

🔧 场景题：项目中有 30 个 Service，你需要在每个 Service 上加一个 healthCheck() 方法给监控系统调用。不能让 30 个 Service 都去改代码。怎么做？

用 @DeclareParents。定义 HealthCheckable 接口和 DefaultHealthCheckable 实现（查数据库连接池状态、Redis 连通性等）。然后一个切面引入到所有 Service。监控系统调用 ((HealthCheckable) service).healthCheck()。30 个 Service 零改动。

A
Q

🔧 场景题：AOP 代理和循环依赖在一起时，Spring 怎么保证代理一致性？

三级缓存的 getEarlyBeanReference() 在这里起了关键作用。当 B 在填充属性时通过三级缓存获取 A 的早期引用 → getEarlyBeanReference() 被调用 → 如果 A 需要 AOP 代理 → 返回代理对象而不是原始对象。这就保证了 B 拿到的 A 是代理过的。等 A 初始化完成（postProcessAfterInitialization），Spring 发现 A 的早期引用已经是代理了 → 不会再创建新代理 → B 和 其他调用者拿到的 A 是同一个代理。

A
Q

Spring AOP 和 AspectJ 的区别？为什么 Spring 不直接用 AspectJ？

AspectJ 是编译期/类加载期织入（需要 ajc 编译器或 LTW），不依赖代理 → 可以增强 final 类、private 方法、自调用也生效。Spring AOP 是运行时代理织入 → 简单、不需要额外编译步骤，但有代理的局限性。Spring 选择代理方案是"牺牲部分能力，换零配置"的权衡。

A

📋 面试官总结 · AOP 章节完结

1. @DeclareParents — 给 Bean 加接口，不改源码的"外挂"能力

2. exposeProxy — 解决自调用的最后一招，但有框架耦合代价

3. 代理与循环依赖 — getEarlyBeanReference 保证代理一致性

4. Spring AOP vs AspectJ — 运行时代理 vs 编译期织入，能力与简单性的权衡

⚠️ AOP 章节总结：五种通知 → 六种切点指示符 → JDK/CGLIB 代理选择 → @DeclareParents 引入 → expireProxy 自调用 → 循环依赖中的代理一致性
