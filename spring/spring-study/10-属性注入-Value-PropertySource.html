Part 1 · 基础认知

@Value 常用方式

// 1. 注入字符串
@Value("${app.name}")  private String appName;

// 2. 带默认值（冒号后面）
@Value("${app.timeout:5000}")  private int timeout;  // 不存在用 5000

// 3. 注入系统属性
@Value("${server.port}")  private int port;

// 4. SpEL 表达式
@Value("#{ T(java.lang.Math).random() * 100 }")  private double randomNum;
@Value("#{ systemProperties['user.home'] }")      private String homeDir;

@Value vs @ConfigurationProperties

对比@Value@ConfigurationProperties
支持类型基本类型 + String✅ 复杂对象、List、Map、Duration
松散绑定❌ 需要严格一致✅ app-name = appName = APP_NAME
校验❌ 启动时不校验✅ @Validated + JSR 303
IDE 提示❌ 无✅（需 processor 依赖）

Part 2 · 触及灵魂的设计原理

🤔 假如没有 @Value，怎么把配置注入到 Bean？

回到 XML 时代：<property name="url" value="${db.url}" />。每个属性一段 XML，配置和代码分离在两个文件里。改了字段名 → XML 不会自动更新 → 运行时 BeanCreationException。

@Value 把配置绑定从 XML 搬到了字段级别：代码即文档。但它的真正痛点不在这里——而是它的"拼写地狱"：

// 同一个前缀的配置分散在 10 个字段里
@Value("${app.datasource.url}")       // 拼错一个字母 → 启动不报错 → 运行时空指针
@Value("${app.datasource.username}")  // refactor 改名 → 全局搜索替换
@Value("${app.datasource.password}")  // 没有编译期保证

这就是 @ConfigurationProperties 诞生的背景——它用一个类型安全的 POJO 替代分散的 @Value。

💡 设计巧思：PropertySourcesPlaceholderConfigurer

${} 不是 @Value 的专利——它是一种占位符协议。任何被 Spring 管理的 Bean 都可以在字段上使用 ${}。背后的处理者是 PropertySourcesPlaceholderConfigurer（一个 BFPP）：

// PropertySourcesPlaceholderConfigurer 的本质工作：
// 在 Bean 定义加载后、实例化之前，遍历所有 BeanDefinition
// 把 "${xxx}" 替换为 Environment 中的真实值

postProcessBeanFactory(beanFactory) {
    for (String beanName : beanFactory.getBeanDefinitionNames()) {
        BeanDefinition bd = beanFactory.getBeanDefinition(beanName);
        // 遍历 bd 的所有属性值 → 替换 ${...} 占位符
        // 替换来源：application.yml → 环境变量 → 命令行参数 → ...
    }
}

"啊哈时刻"：替换发生在 Bean 创建之前——所以你注入到的字段里拿到的已经是替换后的值了。这就是为什么 @Value 能注入 ${} 表达式——AutowireCandidateResolver 会委托给 PropertySourcesPlaceholderConfigurer 解析。

🔬 @PropertySource 的加载链路 
// @PropertySource 注册一个外部 .properties 文件到 Environment 中
@Configuration
@PropertySource("classpath:database.properties")
@PropertySource(value = "file:/etc/secrets.properties", ignoreResourceNotFound = true)
public class AppConfig { }

// 加载链路：
// @PropertySource → PropertySourceFactory 读取文件
// → 添加到 Environment.getPropertySources() 列表末尾
// → PropertySourcesPlaceholderConfigurer 按优先级遍历所有 PropertySource
// → 1. 命令行参数 → 2. 环境变量 → 3. application.yml 
// → 4. @PropertySource 文件 → 5. 默认值
⚠️ 边界与陷阱

陷阱1：@Value 注入 static 字段 = null

static 字段不属于实例，Spring 不管理。需要用 setter 注入。

陷阱2：@Value 不支持复杂类型

List、Map 注入需要 SpEL 或者改用 @ConfigurationProperties。

陷阱3：@PropertySource 不支持 YAML

只能加载 .properties 文件。YAML 需要额外配置 YamlPropertySourceLoader。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Value 和 @ConfigurationProperties 的区别？你推荐用哪个？

@ConfigurationProperties 完胜。类型安全、支持松散绑定、支持 JSR 303 校验、有 IDE 自动补全。@Value 只有一个优势：适合注入单个简单值（如 @Value("${server.port}")）。如果有一组相关配置 → 一定要用 @ConfigurationProperties。

A
Q

@Value 注入 static 字段可以吗？

不可以。Spring 的注入机制基于实例——static 字段属于类，Spring 不管理。需要用非 static 的 setter：public void setAppName(String name) { MyConst.APP_NAME = name; } 然后 @Value 放在 setter 上。

A
Q

${app.name:default} 里的冒号是什么意思？

默认值语法。冒号后面是 fallback。如果所有 PropertySource 都找不到 app.name → 用 "default"。可以嵌套：${app.name:${user.name:anonymous}}。

A
Q

🔧 场景题：多数据源中，DataSource 的配置（url/username/password）存在配置中心。系统启动时配置中心挂了，但应用必须启动（缓存上次配置），怎么设计？

用 @Value 的默认值机制兜底：@Value("${db.url:jdbc:mysql://localhost:3306/default}")。配置中心不可用时用本地默认值。但要区分"配置中心挂了"和"配置不存在"——前者应该用缓存值，后者才用默认值。更完善的方案：写一个 CompositePropertySource，先查配置中心、失败则读本地缓存文件、再失败用默认值。

A
Q

🔧 场景题：智能拓客中，渠道的 API 密钥通过加密存储。用 @Value 注入加密字符串然后代码解密，这种方式有什么问题？

三个问题：① 解密逻辑和配置耦合，如果加解密算法升级（如 AES 换国密）要改注入代码。② @Value 在 BPP.before 阶段执行——如果你需要在解密之后做其他处理，时机太早。③ 日志打印属性值时可能泄露明文。更好的方案：自定义 PropertySource，在配置加载阶段就完成解密——这样注入到的已经是明文，业务代码无感知。Spring Cloud Config 的加密就是这么做。

A
Q

@PropertySource 能加载 YAML 吗？

默认不支持。@PropertySource 使用 DefaultPropertySourceFactory（只能解析 .properties）。要加载 YAML 需要自定义 factory = YamlPropertySourceFactory.class。但 Spring Boot 项目中根本不需要 @PropertySource 加载 YAML——application.yml 已经在 Environment 中了，直接用 @ConfigurationProperties 绑定即可。

A
Q

${} 和 #{} 的区别？

${} 是属性占位符（Property Placeholder），从 Environment 中取值。#{} 是 SpEL 表达式，可以调用方法、访问对象、做计算。${} 更多用于配置注入，#{} 更多用于运行时表达式。

A

📋 面试官总结

1. @Value 的定位 — 单个简单值注入；复杂的用 @ConfigurationProperties

2. 默认值语法与 fallback 机制 — ${key:default} 和嵌套 fallback

3. PropertySourcesPlaceholderConfigurer 的替换时机 — Bean 实例化之前已完成替换

4. static 字段注入陷阱 — setter 注入是唯一解法

⚠️ 大厂高频陷阱：@PropertySource 不支持 YAML；加密配置的最佳位置是 PropertySource 层而非业务代码层
