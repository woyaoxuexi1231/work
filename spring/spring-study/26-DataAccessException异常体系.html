Part 1 · 异常层次结构

DataAccessException (RuntimeException, 非受检)
├── DataAccessResourceFailureException  // 连接失败
├── DataIntegrityViolationException     // 唯一约束冲突(MySQL 1062)

├── IncorrectResultSizeDataAccessException // 结果数不对

Part 2 · 触及灵魂

🤔 如果没有 DataAccessException，数据库异常怎么处理？

JDBC 只抛出 SQLException —— 不管是连接断了、死锁了、唯一约束冲突了，全都是同一个异常类型。你要靠 getErrorCode() 区分，而且每个数据库的错误码不同（MySQL 1062 ≠ Oracle 1）。代码不可移植。

💡 设计巧思：SQLErrorCodeSQLExceptionTranslator

Spring 维护了每种数据库的错误码映射表（sql-error-codes.xml），自动把 SQLException 翻译为具体的 DataAccessException 子类。你不再需要 switch(errorCode)。

⚠️ 为什么是非受检异常？

Spring 认为数据库异常通常不可恢复（连接断了你让用户重试？），应该是非受检的。这让你不用在每个 DAO 方法上写 throws。

Part 3 · 面试追问（6 组）

Q

为什么 DataAccessException 是非受检的？

Spring 认为数据库异常通常是不可恢复的——连接断了、SQL 写错了、约束冲突了，你 catch 了又能怎样？与其强制 try-catch，不如让它自然上抛到统一异常处理层。

A
Q

MySQL 的 1062 和 Oracle 的 1 怎么变成同一个异常类型？

SQLErrorCodeSQLExceptionTranslator 从 sql-error-codes.xml 读取映射表。MySQL 1062 → DataIntegrityViolationException，Oracle 1 → 同样的异常。每个数据库有独立映射。

A
Q

🔧 场景题：线上 MySQL 报 1213 死锁，你想在 catch 中识别死锁并自动重试。怎么精确识别？

catch(DeadlockLoserDataAccessException e) { retry(); }。不需要 parse errorCode——Spring 已经翻译好了。这就是 DataAccessException 体系的核心价值。

A
Q

EmptyResultDataAccessException 和 IncorrectResultSizeException 的区别？

Empty = 0 条（期望 1 条），IncorrectResultSize = >1 条（期望恰好 1 条）。都发生在 queryForObject。

A
Q

DataAccessException 和 SQLException 的继承关系？

没有继承关系。DataAccessException 继承 RuntimeException，SQLException 继承 Exception。它们是完全独立的体系。Spring 通过 SQLErrorCodeSQLExceptionTranslator 做翻译，而不是包装。

A
Q

自定义异常翻译怎么做？

继承 SQLErrorCodeSQLExceptionTranslator，覆盖 customTranslate() 方法。Spring 的默认翻译在找不到映射时会调 customTranslate 给你兜底。

A

📋 面试官总结

1. 非受检设计 — 数据库异常通常不可恢复，不强制 try-catch

2. 跨数据库错误码映射 — sql-error-codes.xml 是翻译字典

3. 20+ 子类精准捕获 — catch(DeadlockLoserDataAccessException) 比 switch(errorCode) 优雅多了
