Part 1 · 基础认知

四个核心术语

术语翻译含义
Aspect切面横切逻辑的"类" — @Aspect 标注 + 包含通知和切点
Join Point连接点程序执行中可插入的点 — 方法调用、异常抛出等
Advice通知做什么 + 何时做 — @Before / @After / @Around
Pointcut切入点在哪里做 — 表达式匹配哪些方法

五种通知 — 执行顺序与签名

通知执行时机能否拿到返回值能否拿到异常
@Before目标方法前❌❌
@AfterReturning正常返回后✅ returning❌
@AfterThrowing异常抛出后❌✅ throwing
@After方法结束（类似 finally）❌❌
@Around包裹整个方法（最强大）✅ —✅ —
@Aspect @Component
public class LoggingAspect {

    @Before("execution(* com.example.service.*.*(..))")
    public void before(JoinPoint jp) {
        log.info("调用: {}", jp.getSignature().getName());
    }

    @AfterReturning(pointcut = "execution(* com.example.service.*.*(..))", returning = "ret")
    public void afterReturning(JoinPoint jp, Object ret) {
        log.info("返回: {}", ret);
    }

    @AfterThrowing(pointcut = "execution(* com.example.service.*.*(..))", throwing = "ex")
    public void afterThrowing(JoinPoint jp, Exception ex) {
        log.error("异常: {}", ex.getMessage());
    }

    @After("execution(* com.example.service.*.*(..))")
    public void after(JoinPoint jp) { log.info("结束"); }

    @Around("execution(* com.example.service.*.*(..))")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try { return pjp.proceed(); }  // ← 必须调用，否则方法不执行！
        finally { log.info("耗时: {}ms", System.currentTimeMillis() - start); }
    }
}

Part 2 · 触及灵魂

🤔 如果不用 AOP，100 个 Service 加日志怎么写？

每个 Service 的方法都要 copy-paste 日志代码。改日志格式→改 100 个文件。加一个新的 Service→又要 copy 一次。这是横切关注点散落的经典问题。AOP 从设计模式"装饰器模式"和"代理模式"获得了灵感。

💡 设计巧思：@Around 为什么是最强大的？

其他四种通知都只能"观察"——@Before 在方法前执行但无法改变参数、@AfterReturning 能看返回值但不能改。只有 @Around 能"介入"：改参数、改返回值、吞异常、决定是否执行目标方法、执行几次。@Transactional 就是用 @Around 实现的——它在 proceed() 前后开启/提交事务。

🔬 五种通知的执行顺序

@Around 前 → @Before → 目标方法执行 → @AfterReturning / @AfterThrowing → @After → @Around 后

注意 @After 是"finally"语义——不管正常还是异常都执行。

⚠️ 边界：@Around 不调 proceed() 会怎样？

目标方法永远不执行。这是一个常见的初学者错误——写了 @Around 但忘记 pjp.proceed()。更隐蔽的坑：proceed() 调了两次→目标方法执行了两次。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

AOP 的五个术语：切面、连接点、通知、切入点、织入——用一句话解释清楚。

切面是"类"，通知是"做什么+何时做"，切入点是"在哪里做"，连接点是"所有可插入的点"，织入是"把切面弄到目标对象上的动作"。一句比喻：切面是菜谱，通知是步骤，切入点是"哪些菜适用这个步骤"，连接点是厨房里的所有食材。

A
Q

@After 和 @AfterReturning 的区别？

@AfterReturning 只在正常返回时执行，@After 不管正常还是异常都执行（finally 语义）。@AfterReturning 能拿到返回值（returning 属性），@After 不能。

A
Q

@Around 能搞定其他四种通知的所有功能吗？

技术上能。@Around 的 proceed() 前后可以模拟 @Before/@After；try-catch 可以模拟 @AfterThrowing。但能不用就不用——@Around 代码量大、容易出错（忘调 proceed、调两次）。能用 @Before 搞定的就别用 @Around。

A
Q

🔧 场景题：你接手了一个项目，所有 Controller 方法都没有打日志。产品说线上问题定位太难了，要你在不修改 Controller 代码的前提下给所有 public 方法加上入参和出参日志。怎么做？

写一个 @Aspect，切点 = execution(public * com.example.controller..*(..))，@Around 中：打印请求参数（pjp.getArgs()），执行 pjp.proceed()，打印返回值。加一个 @Component 就生效了——200 个 Controller 方法零入侵全部有日志。还可以加上 @ConditionalOnProperty 让运维能通过配置开关关闭日志。

A
Q

🔧 场景题：线上有一个大事务（10秒），你怀疑是某个 Service 方法里的一个外部 HTTP 调用太慢。怎么在不加代码的情况下排查是哪个方法？

写一个 @Aspect，@Around 切所有 Service 方法，记录执行时间。超过阈值（比如 1 秒）的打 WARN 日志（类名+方法名+耗时）。加上这个 Aspect 后重启，不需要改任何业务代码。这比在每个方法里加 StopWatch 快得多。

A
Q

同一个方法上有多个 @Around 切面，执行顺序怎么定？

@Order 控制。数字越小越先执行（最外层）。@Order(1) 的 @Around 最先包裹目标方法，最后返回。像洋葱：Aspect1 → Aspect2 → 目标方法 → Aspect2 返回 → Aspect1 返回。

A
Q

JoinPoint 和 ProceedingJoinPoint 什么关系？

ProceedingJoinPoint 继承 JoinPoint，多了一个 proceed()。@Before/@After 用 JoinPoint（只能获取元数据），@Around 用 ProceedingJoinPoint（能获取元数据 + 执行目标方法）。

A

📋 面试官总结

1. 四个术语的关系 — 切面⊃通知+切入点，通知回答了"做什么、何时做"，切入点回答"在哪做"

2. 五种通知的执行顺序 — @Around前→@Before→方法→@AfterReturning/Throwing→@After→@Around后

3. @After = finally — 不管正常异常都执行，但不能拿返回值

4. @Around 的威力与危险 — 能搞定一切，但忘调 proceed() 是常见翻车现场

⚠️ 高频陷阱：@After 拿不到返回值；@Around 不调 proceed 目标方法不执行
