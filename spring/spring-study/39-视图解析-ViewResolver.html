Part 1 · 三种 ViewResolver

实现逻辑视图名→实际视图场景
InternalResourceViewResolver"users" → /WEB-INF/views/users.jspJSP 渲染
ThymeleafViewResolver"users" → templates/users.html⭐ Thymeleaf
ContentNegotiatingViewResolver根据 Accept 选 JSON/XML/HTML内容协商

Part 2 · 触及灵魂

💡 ContentNegotiatingViewResolver 的决策链

① 看 Accept header → 确定 MediaType。② 遍历所有 ViewResolver，看哪个能处理该 MediaType。③ JSON 返回 MappingJackson2JsonView，HTML 委托给 ThymeleafViewResolver。它不是直接解析视图，而是做路由。

⚠️ REST API 中基本不用 ViewResolver

@RestController 用 HttpMessageConverter 直接写响应体，跳过视图解析。ViewResolver 主要用于需服务端渲染的页面（Thymeleaf）。

Part 3 · 面试追问（5 组）

Q

多个 ViewResolver 同时存在，谁先生效？

按 @Order 排序，数字小的优先。ContentNegotiatingViewResolver 通常排在最前面——它决定是否需要视图解析（API 请求直接跳过）。

Q

ViewResolver 什么时候被跳过？

当 Handler 方法标注了 @ResponseBody 或返回 ResponseEntity 时——直接走 HttpMessageConverter，不需要 ViewResolver。
