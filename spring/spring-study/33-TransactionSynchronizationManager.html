Part 1 · ThreadLocal 变量清单

ThreadLocal<Map<Object,Object>> resources;       // DataSource→ConnectionHolder
ThreadLocal<Set<Synchronization>> synchronizations; // 事务生命周期回调
ThreadLocal<String> currentTransactionName;          // 当前事务名
ThreadLocal<Boolean> readOnly;                       // 是否只读
ThreadLocal<Integer> isolationLevel;                 // 隔离级别
ThreadLocal<Boolean> actualTransactionActive;        // 是否有活跃事务

这就解释了：①同一线程共享事务 ②@Async拿不到事务(换线程) ③任何地方可判断"当前是否有事务"。

Part 2 · 触及灵魂

💡 为什么用 ThreadLocal 而不是全局 Map？

事务是线程绑定的——一个 HTTP 请求在一个线程执行。ThreadLocal 是最自然的绑定方式：不需要传参、不需要全局锁、不需要上下文对象。把 JVM 线程模型和数据库事务模型对齐的经典设计。

💡 TransactionSynchronization — 事务生命周期钩子
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization(){
    void afterCommit() { /* 发MQ */ }
    void afterCompletion(int s) { /* 清理 */ }
    void beforeCommit(boolean readOnly) { /* 提交前 */ }
    void beforeCompletion() { /* 完成前 */ }
});

@TransactionalEventListener 的底层就是它。事务提交后才发MQ——确保下游不会收到"还没提交"的数据。

⚠️ 陷阱

无事务时 registerSynchronization 抛异常；afterCommit 异常被吞——事务已提交，回调异常不影响数据但可能丢MQ。

Part 3 · 面试追问（6 组）

Q

为什么用 ThreadLocal 不是 InheritableThreadLocal？

事务不应跨线程传播。线程池线程复用→InheritableThreadLocal只在创建时继承一次→过期数据。Spring 选择普通 ThreadLocal 是正确的。

A
Q

@Async 里怎么拿到主线程的事务？

拿不到。不同线程有独立的 ThreadLocal。事务应该不跨线程——把需要事务的工作在主线程完成，异步线程只处理非事务逻辑。

A
Q

🔧 场景题：订单创建后要发MQ，但必须在事务提交后才发（确保下游能读到订单）。怎么保证？

registerSynchronization 注册 afterCommit 回调。事务提交后才执行→此时订单一定在DB中。如果MQ必达性要求高→配合本地消息表：事务内插"待发送"记录 + afterCommit实际发送 + 定时任务补偿。

A
Q

afterCommit 里抛异常会回滚事务吗？

不会——事务已经提交了。异常只打日志，后续回调可能不执行。如果 afterCommit 需要可靠性，自己 try-catch+重试。

A
Q

resources ThreadLocal 的 key 是什么？

DataSource 实例。多数据源场景下 masterDataSource 和 slaveDataSource 是不同的 key→连接天然隔离。

A
Q

@TransactionalEventListener 和 registerSynchronization 什么关系？

@TransactionalEventListener 底层就是 registerSynchronization。Spring 在事件发布时检查 listener 是否有 @TransactionalEventListener→包装成 TransactionSynchronization→注册。AFTER_COMMIT 对应 afterCommit 回调。

A

📋 面试官总结 · 数据访问与事务章节完结

1. ThreadLocal 是事务隔离的基石 — 每个线程一份独立状态

2. TransactionSynchronization — afterCommit 是发MQ的正确时机

3. 多数据源天然隔离 — DataSource 作为 ThreadLocal 的 key
