Part 2 · 触及灵魂

🤔 如果没有声明式事务，怎么管理数据库事务？

每个 Service 方法都要手写 6 行样板：

Connection conn = null; try {
    conn = ds.getConnection(); conn.setAutoCommit(false);
    // ... 业务
    conn.commit();
} catch (Exception e) { conn.rollback(); }
finally { conn.close(); }

@Transactional 把 6 行变成 1 个注解——AOP 的 @Around 在 proceed() 前后处理事务。这不仅仅是"少写代码"，关键是传播行为的声明式编排——"这个方法需要独立事务"这种语义写进注解比写进代码清晰得多。

💡 设计巧思：REQUIRES_NEW 的"挂起"是怎么实现的？ 
// AbstractPlatformTransactionManager 内部（伪代码）
if (definition.getPropagationBehavior() == PROPAGATION_REQUIRES_NEW) {
    SuspendedResourcesHolder suspended = suspend(transaction);
    // suspend: 把当前事务的 Connection 从 ThreadLocal 中取出暂存
    // 然后开始一个新事务（新 Connection + setAutoCommit(false)）
    try {
        return startNewTransaction();
    } finally {
        resume(suspended);  // 把暂存的旧事务资源放回 ThreadLocal
    }
}

"啊哈时刻"："挂起"不是数据库级别的操作——数据库并不知道事务被"挂起"了。它只是 Spring 把 ThreadLocal 中的旧连接暂存起来，换了一个新连接。恢复时再把旧连接放回去。

💡 NESTED 的巧思：Savepoint 复用同一连接

NESTED 不需要挂起旧事务——它在同一连接上创建一个 Savepoint（JDBC 的 Connection.setSavepoint()）。内层回滚只需要 conn.rollback(savepoint)，不影响外层。

代价：JPA 不支持 NESTED——EntityManager 不暴露 Savepoint 操作。只有纯 JDBC 或 MyBatis 可以用。

⚠️ 10 大失效场景速查

① 非 public 方法 ② 同类自调用 ③ 异常被 try-catch 吞掉 ④ rollbackFor 不匹配 ⑤ 多线程 ⑥ 数据库引擎不支持 ⑦ final 方法 ⑧ 传播行为配错 ⑨ 类未被 Spring 管理 ⑩ 默认只回滚 RuntimeException

Part 3 · 大厂面试官的灵魂追问（8 组）

Q

REQUIRES_NEW 和 NESTED 的本质区别？

REQUIRES_NEW = 新连接 + 独立物理事务（内外完全隔离）。NESTED = 同一连接 + Savepoint（外层回滚会连内层一起回滚）。本质区别：REQUIRES_NEW 是两个独立事务，NESTED 是一个事务内的子事务。

A
Q

默认只对 RuntimeException 回滚——为什么？

EJB 的历史遗留。EJB 认为 checked exception 是"可恢复的"（如 FileNotFoundException），不应回滚事务。Spring 沿用了这个约定。今天的实践是：始终加 rollbackFor = Exception.class。

A
Q

PlatformTransactionManager 怎么知道该用哪个数据源的事务？

通过 DataSource 或 EntityManagerFactory 的绑定。JpaTransactionManager 持有一个 EntityManagerFactory 引用——它知道该管哪个数据源。如果有多数据源，需要多个 TransactionManager Bean，用 @Transactional("tm2") 指定。

A
Q

隔离级别 READ_UNCOMMITTED 和 SERIALIZABLE 什么时候用？

READ_UNCOMMITTED：几乎不用——脏读风险太高。SERIALIZABLE：金融对账、库存扣减等场景需要绝对数据一致性时用——代价是大量锁等待和死锁。大多数场景用 READ_COMMITTED（默认）或 REPEATABLE_READ（MySQL 默认）。

A
Q

🔧 场景题：日志记录（插入日志表）必须成功，即使业务操作（插入订单）失败了。怎么设计？

日志方法用 @Transactional(propagation = REQUIRES_NEW)。外层订单事务回滚 → 不影响日志事务（日志已经提交了）。但要注意：如果日志方法抛了异常且被外层 catch 了 → 外层不应该受影响。这是 REQUIRES_NEW 的经典场景。

A
Q

🔧 场景题：线上发现一个 @Transactional 方法没回滚——排查发现同类自调用。怎么修复？

三种修复：① 拆类——把事务方法移到一个独立的 Service（最佳实践）。② 注入自己——@Autowired private XxxService self，通过 self.method() 调用（绕过 this）。③ AopContext.currentProxy()（需 expireProxy=true）。推荐拆类——不依赖 Spring 内部 API。

A
Q

🔧 场景题：主从数据库分离 + 读写分离，写操作走主库事务，读操作走从库（不需要事务）。怎么让读操作不走事务？

读方法加 @Transactional(propagation = SUPPORTS, readOnly = true)。SUPPORTS = 外层有事务就加入（写场景），没有就非事务执行（读场景纯查询）。readOnly=true 给 Hibernate/JDBC 驱动一个优化提示（禁用脏检查、flush）。

A
Q

@Transactional 标注在接口上和方法上有区别吗？

JDK 动态代理时接口上的 @Transactional 生效，CGLIB 代理时接口上的不生效（因为代理是类级别）。Spring 推荐标注在具体类的方法上而不是接口上——确保两种代理方式都生效。
