Part 2 · 触及灵魂

🤔 为什么需要 @RestController？

Spring MVC 最初为服务端渲染设计——Controller 返回视图名。REST API 兴起后，每个方法都要加 @ResponseBody 太繁琐。@RestController 让 REST 开发回归直觉：返回什么就输出什么。

💡 redirect: 和普通字符串怎么区分？

startsWith("redirect:") → RedirectView → 302 重定向。startsWith("forward:") → 服务端转发。其他 → ViewResolver 解析为视图路径。

Part 3 · 面试追问（6 组）

Q

@Controller + @ResponseBody 和 @RestController 完全等价吗？

完全等价。@RestController 源码就是 @Controller + @ResponseBody。但类上 @ResponseBody 会让所有方法都返回 JSON——如果某个方法需要返回视图就不行了。

Q

🔧 场景题：一个 Controller 中 90% 方法返回 JSON，10% 返回视图。怎么设计？

用 @Controller（不用 @RestController）。10% 的视图方法不加 @ResponseBody，90% 的 JSON 方法加 @ResponseBody。或者拆成两个 Controller——RestUserController 和 PageUserController。

Q

@Controller 方法返回 "redirect:/xxx" 发生了什么？

ViewNameMethodReturnValueHandler 检测到 startsWith("redirect:")→创建 RedirectView→response.sendRedirect(url)→浏览器收到 302 重新请求新 URL。

Q

🔧 场景题：API 需要同时返回 JSON 和 XML（根据 Accept header）。怎么做？

内容协商。加 jackson-dataformat-xml 依赖→Spring Boot 自动注册 XML 的 HttpMessageConverter。Accept: application/json → JSON；Accept: application/xml → XML。同一个方法不用改。

Q

什么是内容协商？底层怎么做？

客户端的 Accept header（application/json）→ ContentNegotiationManager 确定 MediaType → 遍历 HttpMessageConverter 链找第一个 canWrite(type, mediaType) → 序列化输出。

📋 面试官总结

1. @RestController = @Controller + @ResponseBody

2. redirect: / forward: 前缀决定返回处理方式

3. 内容协商基于 Accept header + HttpMessageConverter
