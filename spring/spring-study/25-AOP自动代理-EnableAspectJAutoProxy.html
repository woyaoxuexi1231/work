Part 1 · 两个关键属性

属性作用Spring Boot 默认
proxyTargetClasstrue=强制CGLIB / false=按接口选true（Boot 2.x起）
exposeProxytrue=代理暴露到 AopContext.currentProxy()false
// Spring Boot 不需要手动加这个注解 — AopAutoConfiguration 自动配置了
@EnableAspectJAutoProxy(proxyTargetClass = true, exposeProxy = false)
@Configuration
public class AopConfig { }

// exposeProxy 用法（自调用的临时解法）
((UserService) AopContext.currentProxy()).method();

Part 2 · 触及灵魂

🤔 @EnableAspectJAutoProxy 到底做了什么？

它向容器注册了一个 AnnotationAwareAspectJAutoProxyCreator（继承 AbstractAutoProxyCreator）。这是一个 InstantiationAwareBeanPostProcessor——在每个 Bean 初始化后检查是否需要创建代理。

💡 exposeProxy 的代价

每次代理对象被调用时，都要往 ThreadLocal 里写一次引用。虽然微秒级，但高并发下累积可观。更重要的代价是代码耦合——AopContext.currentProxy() 需要 import Spring 内部 API，你的代码不再是纯 POJO。

⚠️ 陷阱

exposeProxy=true 后，AopContext.currentProxy() 必须在代理方法的执行线程中调用——异步线程中调用返回 null。

Part 3 · 面试追问（6 组）

Q

Spring Boot 需要手动加 @EnableAspectJAutoProxy 吗？

不需要。AopAutoConfiguration 自动检测 classpath 中有 @Aspect 注解和 aspectjweaver 依赖时自动开启。

A
Q

proxyTargetClass=true 和 false 的区别？

true=强制CGLIB（所有代理都是子类），false=有接口用JDK代理、没接口用CGLIB。Boot 2.x起默认为true，避免 JDK 代理的注入陷阱。

A
Q

🔧 场景题：线上排查半天发现 @Transactional 没生效，最后发现是因为同类自调用。怎么彻底解决？

短期：加 exposeProxy=true + AopContext.currentProxy() 让自调用走代理。长期：拆类——把事务方法移到独立 Service。拆类是最佳实践，不依赖 Spring 内部 API，不需要配置开关。

A
Q

🔧 场景题：项目里有些 AOP 切面在单元测试中不需要（比如监控切面）。怎么在不改代码的情况下控制？

监控切面加 @Profile("!test") 或 @ConditionalOnProperty(name="monitor.enabled", havingValue="true")。测试环境通过 profile 或配置关闭。不需要改切面代码。

A
Q

AopContext.currentProxy() 在 @Async 方法里调用能拿到代理吗？

不能。@Async 换线程 → ThreadLocal 中没有主线程的代理引用 → 返回 null。需要事务+异步的场景，把需要代理的部分在主线程调完，只把结果传进异步方法。

A
Q

如果有多个 @Configuration 类都加了 @EnableAspectJAutoProxy？

AnnotationAwareAspectJAutoProxyCreator 只会注册一次（BeanDefinitionRegistry 去重）。多个 @EnableAspectJAutoProxy 不冲突，但容易造成配置混乱——建议只在主配置类上加一个。

A

📋 面试官总结 · AOP 章节完结

1. proxyTargetClass — Boot 2.x起默认 true，统一CGLIB

2. exposeProxy — 自调用解药但有框架耦合代价

3. 自动代理注册 — AopAutoConfiguration 自动处理

AOP 完整链路：核心概念→五种通知→切入点表达式→引入→AspectJ集成→JDK/CGLIB代理→ProxyFactoryBean→@EnableAspectJAutoProxy
