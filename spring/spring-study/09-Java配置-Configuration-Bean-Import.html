Part 1 · 基础认知

@Configuration vs @Component + @Bean — 一张表搞定

对比维度@Configuration（Full）@Component + @Bean（Lite）
CGLIB 代理✅ 有❌ 无
@Bean 方法相互调用走容器，保证单例普通 Java 调用，每次 new
启动速度较慢（生成代理）较快
final 类/方法❌ 不能是 final✅ 可以
// @Configuration 下的单例保证
@Configuration
public class AppConfig {
    @Bean public A a() { return new A(b()); }  // b() 返回容器中的单例 B ✅
    @Bean public B b() { return new B(); }
}

// @Import：模块化组装配置
@Configuration @Import({DataConfig.class, CacheConfig.class})
public class AppConfig { }  // 三个配置类的 Bean 全在一个容器中

// @Import 也可以直接导入普通类（直接注册为 Bean）
@Import(UserService.class)  // UserService 变成 Bean，无需 @Component

Part 2 · 触及灵魂的设计原理

🤔 假如 @Configuration 不用 CGLIB 代理，会怎样？

真实故障：a() 里调用 b() → 每次调用 new 一个新 B。如果 B 是 DataSource → 连接池被创建了两次，数据库连接数翻倍，线上连接池耗尽。

更隐蔽的是：开发者看不出问题。代码看起来完全合理——"我调了 b() 方法拿到 B，这有什么问题？" 问题在于 Java 方法调用的语义是"每次执行"，但 Bean 容器的语义是"单例"。CGLIB 代理就是这个语义鸿沟的桥梁。

💡 设计巧思：CGLIB 子类拦截 @Bean 方法

"啊哈时刻"：Spring 不为你的 AppConfig 直接注册，而是生成一个 CGLIB 子类 AppConfig$$EnhancerBySpringCGLIB。这个子类重写了所有 @Bean 方法，在方法里拦截调用：

// Spring 生成的代理子类（伪代码）
public class AppConfig$$Enhancer extends AppConfig {
    private BeanFactory beanFactory;

    @Override
    public B b() {
        // 先查容器：如果已有 B → 直接返回（保证单例）
        if (beanFactory.containsSingleton("b")) {
            return beanFactory.getBean("b");
        }
        // 没有 → 调用父类原始方法 new B()
        B b = super.b();
        beanFactory.registerSingleton("b", b);
        return b;
    }
}

为什么这个设计"精妙"？因为开发者完全不用改变编码直觉——配置类中用"方法调用"来表达"我需要那个 Bean"，而 CGLIB 在背后悄悄把这个调用变成了"先去容器查"。代码怎么写怎么读，但实际执行完全不同——这就是代理的终极价值。

💡 @Import 的设计巧思：配置的"乐高积木"

@Import 解决了一个组织问题：如果你把所有 @Bean 写在一个配置类里 → 3000 行的"上帝类"。如果用多个配置类 → 怎么把它们"组装"起来？

@Import 就像乐高的连接点：每个配置类是一个独立模块（DataSourceConfig、CacheConfig、SecurityConfig），通过 @Import 拼在一起。@Import 还支持导入 ImportSelector 和 ImportBeanDefinitionRegistrar——这就是 @EnableAutoConfiguration 实现自动配置的底层机制。

⚠️ 边界：@Configuration 的 final 限制

限制：

类不能是 final（CGLIB 需要继承），@Bean 方法不能是 final 或 private（需要重写）。

权衡：

Full 模式（proxyBeanMethods=true）= 单例保障，但启动有代理开销。Lite 模式（proxyBeanMethods=false）= 无代理、更快、可 final，但 @Bean 方法间调用不保证单例。Spring Boot 2.2+ 的自动配置类大多使用 Lite 模式——因为它们的 @Bean 方法之间通常没有相互调用。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Configuration 下两个 @Bean 方法相互调用，返回的是同一个实例吗？

是。因为 CGLIB 代理拦截了方法调用——代理先去容器里查有没有，有的就直接返回。但 @Component 下不行，那是普通 Java 方法调用。

A
Q

proxyBeanMethods = false 什么时候用？有什么风险？

当 @Bean 方法之间没有相互调用时用——提升启动速度、支持 final 类。风险：如果你误用了 Lite 模式但又依赖了方法间调用的单例保证 → Bean 被重复创建，对于 DataSource 这种持有连接的 Bean 来说后果很严重。

A
Q

@Import 除了导入配置类还能导入什么？

三种：① 普通配置类（@Configuration）② ImportSelector — 根据条件动态返回类名（@EnableAutoConfiguration 的核心）③ ImportBeanDefinitionRegistrar — 直接编程式注册 BeanDefinition（MyBatis 的 MapperScan 就是这样注入 Mapper 代理的）。@Import 是 Spring 扩展体系的核心入口。

A
Q

🔧 场景题：多数据源组件中，你有一个 DynamicDataSourceConfig，里面定义了 master/slave 等 DataSource。slave DataSource 的初始化依赖 master（需要先确认 master 连接正常），怎么在 @Configuration 中保证创建顺序？

@Bean 上不用写 @DependsOn——因为 slave() 方法里调了 master()，这是 CGLIB 代理拦截的方法调用，会自动触发 master 先创建。或者显式地用 @DependsOn("masterDataSource")。但是最佳实践是：不要在 @Bean 方法里做连接验证——把它放到 BeanPostProcessor 的 after 阶段，解耦配置和验证。

A
Q

🔧 场景题：智能拓客多渠道改造中，渠道配置分散在多个 @Configuration 类中。你想统一管理渠道列表，怎么做？

写一个 @Import(ChannelConfigSelector.class)，让 Selector 根据配置文件（或 classpath 中的渠道实现）动态返回需要注册的配置类。这样新增渠道：只需在对应子模块加一个 @Configuration + 在配置中加一行——ChannelConfigSelector 自动加载。不需要改任何已有的 @Import 点。

A
Q

@Bean(initMethod="init") 和 @PostConstruct 有什么区别？

initMethod 在 @PostConstruct 之后执行。核心区别：initMethod 可以指定第三方类的方法（你改不了源码，加不了 @PostConstruct），而 @PostConstruct 必须写在类内部。@Bean(destroyMethod="close") 同理——这是 @Bean 相对于 @Component 的独特优势。

A
Q

ConfigurationClassPostProcessor 在什么时候执行？

在 refresh() 的 invokeBeanFactoryPostProcessors() 阶段。它实现 BeanDefinitionRegistryPostProcessor，被优先执行。它扫描所有 @Configuration 类 → 解析 @Bean/@Import/@ComponentScan → 生成新的 BeanDefinition 注册到容器。这是它的核心工作。
