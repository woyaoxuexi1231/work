Part 2 · 触及灵魂

💡 策略模式 — 为什么不把事务逻辑写死？

如果写死 DataSourceTransactionManager→JPA 项目用不了。如果写死 JpaTransactionManager→MyBatis 项目用不了。PlatformTransactionManager 是接口——实现可替换，上层代码（@Transactional）只依赖接口。这是策略模式的教科书级应用。

🔬 getTransaction() 内部流程
getTransaction(definition) {
    if (当前线程已有事务) {
        根据传播行为决定：加入/挂起+新建/抛异常
    } else {
        新建事务：ds.getConnection()→setAutoCommit(false)→绑定到ThreadLocal
    }
    return TransactionStatus;
}
⚠️ 多数据源需要多个 TransactionManager

每个数据源配一个 TransactionManager Bean。@Transactional("tm2") 指定用哪个。如果不指定→Spring 按优先级选默认的。

Part 3 · 面试追问（6 组）

Q

为什么需要 PlatformTransactionManager 接口？

策略模式——JDBC/JPA/JTA 的事务实现不同，但上层只依赖同一接口。换 ORM 框架不需要改 @Transactional 代码。

A
Q

DataSourceTransactionManager 和 JpaTransactionManager 可以共存吗？

可以，但通常不需要。Spring Boot 有 DataSource 和 JPA 时自动选 JpaTransactionManager（它包含 DataSource 事务能力）。手动共存用于混合场景：JDBC+JPA 各自管理不同数据源。

A
Q

🔧 场景题：项目同时用 JPA 和 MyBatis 操作同一个数据库。事务管理器怎么配？

用 JpaTransactionManager——它管理 EntityManagerFactory 的同时也能管理 JDBC Connection（同一个 DataSource）。不需要两个 TransactionManager。

A
Q

TransactionStatus 里存了什么？

新事务标记、回滚标记（rollbackOnly）、保存点（Savepoint）、是否已完成。它是对当前事务状态的"快照"，commit/rollback 时根据它决策。

A
Q

doBegin() 里做了什么？

DataSourceTransactionManager.doBegin()：获取连接→conn.setAutoCommit(false)→绑定到 TransactionSynchronizationManager（ThreadLocal）→返回 TransactionStatus。这就是事务的物理开端。

A
Q

JtaTransactionManager 什么时候用？

跨多个独立数据源需要两阶段提交（2PC）时。比如一笔转账：减 A 库余额 + 增 B 库余额——两个操作必须原子。单数据源或主从复制不需要 JTA。

A

📋 面试官总结

1. 策略模式 — PlatformTransactionManager 是可替换的事务实现

2. getTransaction 根据传播行为决策 — 新建/加入/挂起

3. doBegin — setAutoCommit(false) 是物理事务起点
