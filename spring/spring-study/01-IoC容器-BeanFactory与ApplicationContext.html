IoC 容器：BeanFactory 与
ApplicationContext

Spring 的心脏——理解了这两个接口的关系，就理解了 Spring 对"对象"这件事的全部态度。

📘 基础认知 🔬 原理深潜 🎤 面试追问

📘 第一部分：基础使用与直观理解

1.1 一句话说清楚

没有 Spring 之前，你需要手写 new UserService() 来创建对象。有了 Spring，你把"如何创建对象"的配方告诉它，它帮你完成创建、组装依赖、管理生命周期。这个"配方管理器"就是 IoC 容器。

Spring 给这个容器设计了两层接口：BeanFactory（底层接口，提供容器最基本的"骨架"）和 ApplicationContext（上层接口，在骨架之上增加了企业级"血肉"）。

1.2 BeanFactory vs ApplicationContext 对比

维度 BeanFactory ApplicationContext
定位 IoC 容器的底层骨架 面向开发者的完整容器
继承关系 独立接口 继承自 BeanFactory + 多个企业级接口
Bean 实例化时机 懒加载：getBean() 时才创建 预加载：启动时创建所有单例 Bean
国际化 (i18n) ❌ 不支持 ✅ MessageSource（支持多语言）
事件发布 ❌ 不支持 ✅ ApplicationEventPublisher
资源加载 ❌ 不支持 ✅ ResourceLoader（classpath/file/URL）
环境抽象 & Profile ❌ 不支持 ✅ Environment + @Profile
BeanPostProcessor 注册 需要手动注册 自动检测并注册
典型实现类 DefaultListableBeanFactory AnnotationConfigApplicationContext
ClassPathXmlApplicationContext

1.3 代码对比：两种容器的使用方式

使用 BeanFactory（底层方式）

// 1. 创建 BeanFactory
DefaultListableBeanFactory factory =
    new DefaultListableBeanFactory();

// 2. 手动注册 BeanDefinition
BeanDefinitionBuilder builder =
    BeanDefinitionBuilder
        .genericBeanDefinition(UserService.class);
factory.registerBeanDefinition(
    "userService", builder.getBeanDefinition());

// 3. 手动注册 BeanPostProcessor
factory.addBeanPostProcessor(
    new AutowiredAnnotationBeanPostProcessor());

// 4. 获取 Bean（此时才创建）
UserService svc = factory.getBean(UserService.class);
svc.doSomething();

使用 ApplicationContext（日常方式）

// 一行代码启动容器
ApplicationContext ctx =
    new AnnotationConfigApplicationContext(
        AppConfig.class);

// Spring Boot 中更简洁
// @SpringBootApplication 启动即可

// 直接获取 Bean（容器启动时就已创建好）
UserService svc = ctx.getBean(UserService.class);
svc.doSomething();

// 额外还能做的事：
ctx.publishEvent(new MyEvent(...)); // 发布事件
ctx.getMessage("user.title", ...);  // 国际化
Resource r = ctx.getResource(...);  // 加载资源

⚡ 关键发现：BeanFactory 只做一件事——管理 Bean。ApplicationContext = BeanFactory + 事件 + 国际化 + 资源加载 + 环境 + 自动 BPP 注册。

1.4 关联你的项目

📋

你在恒生多数据源组件中，之所以能通过 @DS("db1") 切换数据源，底层就是利用 ApplicationContext 的事件机制 + Bean 生命周期管理。每次数据源切换时，组件在 Bean 初始化阶段拦截，根据注解的值从 ApplicationContext 中取出对应的 DataSource Bean——这个"能从容器中按名称取出任意 Bean"的能力，正是 ApplicationContext 提供的基础设施。

🔬 第二部分：原理深潜与设计动机

2.1 假如没有 IoC 容器，会发生什么？

想象你在做一个大型金融系统，有 200 个 Service 类，每个 Service 依赖 3-5 个其他类。没有 IoC 容器的时候，你的代码长这样：

// ❌ 没有 IoC 的世界：每次 new，到处是样板代码
public class TradeService {
    private AccountService accountService = new AccountService(
        new AccountRepository(
            new DataSource(...),   // 连接从哪里来？
            new CacheService(...)  // 缓存怎么共享？
        )
    );
    private RiskService riskService = new RiskService(
        new RiskEngine(...),       // 50 个地方都在 new RiskEngine
        new RedisClient(...)       // 连接池怎么复用？
    );
    // ... 问题：
    // 1. 到处 new，一旦想换实现，改 50 处
    // 2. DataSource 被 new 了 200 次（应该只有一个连接池！）
    // 3. 单元测试要 mock → 只能反射或者重写构造器
    // 4. 想加日志/事务 → 只能手写，无法统一
}

核心痛点不是"麻烦"，而是"不可控"。对象的创建散落在整个代码库中，你无法统一管理它们的生命周期、依赖关系、以及"一个对象在系统中应该只有几份"这个根本问题。

2.2 设计者的破局巧思：从"工厂"到"容器"

Rod Johnson 在 2002 年的《Expert One-on-One J2EE》中就提出了核心思想。但他的"啊哈时刻"不是"用工厂模式"，而是——"让容器本身成为一个巨大的注册表，每个对象在这个注册表里有一个唯一的名字，任何地方都可以按名字或类型取到它。"

💡 设计巧思的核心——三层抽象：

1.

BeanDefinition（配方）

把"如何创建一个对象"变成数据，而非代码。一个 Bean 的类名、scope、依赖关系，全部存为元数据。

2.

BeanFactory（工厂）

根据 BeanDefinition 的配方，真正去 new 对象。类似"厨房"——你给我配方，我给你菜。

3.

ApplicationContext（餐厅）

在"厨房"之外加了"前台"（事件发布）、"翻译"（国际化）、"采购"（资源加载）、"环境"（Profile）。

这个设计的精妙之处：BeanFactory 只做一件事（管理 Bean），且做得极其精简；ApplicationContext 通过继承 + 组合的方式扩展它，而不是重写。这是接口隔离原则（ISP）的完美示范。

2.3 底层机制逐层剥开

层次 1：BeanFactory 的核心数据结构

// DefaultListableBeanFactory 的核心内部结构（精简）
public class DefaultListableBeanFactory extends ... {
    // ① Bean 定义的注册表：beanName → BeanDefinition
    private final Map<String, BeanDefinition> beanDefinitionMap =
        new ConcurrentHashMap<>(256);

    // ② 单例 Bean 的缓存池：beanName → 成品对象（一级缓存）
    private final Map<String, Object> singletonObjects =
        new ConcurrentHashMap<>(256);

    // ③ 别名注册表
    private final Map<String, String> aliasMap =
        new ConcurrentHashMap<>(16);

    // ④ BeanPostProcessor 列表
    private final List<BeanPostProcessor> beanPostProcessors =
        new ArrayList<>();
}

// ApplicationContext 内部组合了 DefaultListableBeanFactory
public class AnnotationConfigApplicationContext extends ... {
    // 核心成员——ApplicationContext 不自己造轮子，
    // 而是持有一个 BeanFactory 的引用！
    private final DefaultListableBeanFactory beanFactory;

    // 新增：事件广播器
    private ApplicationEventMulticaster eventMulticaster;

    // 新增：国际化消息源
    private MessageSource messageSource;
}

层次 2：getBean() 的执行流——容器的心脏跳动

调用链示意图

用户代码:    ctx.getBean("userService")
              │
              ▼
BeanFactory:  doGetBean("userService")
              │
              ├─► ① 查缓存：singletonObjects.get("userService")
              │      └─ 找到 → 返回 ✅（大多数情况走这里，O(1)）
              │
              ├─► ② 没找到 → 查 BeanDefinition：
              │      beanDefinitionMap.get("userService")
              │      └─ 拿到配方：class=UserService, scope=singleton
              │
              ├─► ③ 创建 Bean：createBean(beanDefinition)
              │      ├─ 推断构造器
              │      ├─ 反射 new 对象（Constructor.newInstance）
              │      ├─ 属性填充（@Autowired 依赖注入）
              │      ├─ 初始化（@PostConstruct → afterPropertiesSet）
              │      └─ BeanPostProcessor.after（AOP 代理在此创建）
              │
              └─► ④ 放入缓存 + 返回

层次 3：ApplicationContext 和 BeanFactory 的启动时序差异

二者最本质的行为差异在于：Bean 何时被创建。

// BeanFactory：懒到极致
BeanFactory factory = ...;
// 此时 UserService 还未创建 ⏸️
UserService svc = factory.getBean(UserService.class);
// ↑ 现在才真正 new！

// ApplicationContext：启动时全部就绪
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);
// ↑ 上面的 new 内部调用了 refresh()
//   → 所有非懒加载的单例 Bean 都创建好了
//   → 启动慢，但运行时快（Bean 已经在缓存里）

2.4 边界与限制

限制说明应对
BeanFactory 太底层 需要手动注册 BPP、不支持事件/国际化 绝大多数场景直接用 ApplicationContext
容器本身占用内存 DefaultListableBeanFactory 内部 5-6 个 ConcurrentHashMap，大项目可达几十 MB 合理控制 Bean 数量，避免过度扫描
启动时间 ApplicationContext 启动时创建所有单例 Bean @Lazy 延迟加载、排除不需要的自动配置
不适用所有对象 频繁变化的值对象、请求范围的 DTO 不适合放容器 只用容器管理"有状态"和"有依赖"的服务对象

🧠 设计哲学总结

"BeanFactory 是内核，ApplicationContext 是外壳"——这不是工程上的妥协，而是深思熟虑的分层设计。如果你只需要一个轻量级的 DI 容器（比如嵌入式设备、测试环境），可以直接用 BeanFactory；如果你在做企业应用，ApplicationContext 提供了完整的基础设施。

Rod Johnson 在他的书中写道："一个好的框架应该让你在 80% 的场景下感受不到框架的存在，在剩下 20% 的场景下给你完全的控制力。"——BeanFactory 就是那 20% 的"控制力后门"，ApplicationContext 就是 80% 的"无感体验"。

🎤 第三部分：大厂面试官的灵魂追问

官

第一层 · 基本认知："ApplicationContext 和 BeanFactory 有什么区别？你说 ApplicationContext 继承自 BeanFactory，那它多了哪些能力？

ApplicationContext 继承自 BeanFactory，在此之上多了：事件发布（ApplicationEventPublisher）、国际化（MessageSource）、资源加载（ResourceLoader）、环境抽象（Environment + Profile）。最大的行为差异是：BeanFactory 是懒加载的（getBean 时才创建），ApplicationContext 启动时就预加载所有单例 Bean。

你

👁️ 面试官内心：基础回答，不错。但这不是我想听的——我要看他对"为什么这样设计"有没有自己的理解。

官

第二层 · 内部机制："你说 ApplicationContext 启动时创建所有单例 Bean——那它内部到底是怎么做的？BeanFactory 又为什么不这样做？

ApplicationContext 内部持有一个 DefaultListableBeanFactory 实例。它的 refresh() 方法中第 11 步 finishBeanFactoryInitialization() 会遍历所有 BeanDefinition，逐个调 getBean() 来创建。BeanFactory 没有这个步骤——它是纯「按需供给」，你调 getBean 我才创建，不调就不创建。

设计上的区别：ApplicationContext 选择"启动慢、运行快"，因为企业级应用启动后要应对高并发请求，必须确保所有 Bean 就绪。BeanFactory 选择"启动快、运行时懒"，适合对内存和启动速度敏感的场景。

你

👁️ 面试官内心：他能说出 refresh() 内部第 11 步，说明看过源码。接下来我要问边界了。

官

第三层 · 异常与边界："如果有一个 Bean 在初始化时抛异常，ApplicationContext 启动会怎样？BeanFactory 呢？另外——你在恒生多数据源项目中，有没有遇到过 Bean 初始化顺序的问题？

第一个问题：ApplicationContext 会直接启动失败——因为它在启动时执行了所有单例 Bean 的初始化，任何一个失败都会导致 refresh() 抛出异常，整个容器不会启动。BeanFactory 则不会——你启动时不加载任何 Bean，只有等你真正调 getBean() 发现这个 Bean 有问题时才报错。

这就是为什么 Spring Boot 默认用 ApplicationContext——宁可启动时炸掉，也别上线了才发现某个 Bean 是坏的。这也叫「快速失败」原则。

第二个问题——多数据源组件的场景下，确实有 Bean 初始化顺序问题。因为数据源切换的 AOP 切面需要保证在 DataSource Bean 创建之后、在 Service Bean 创建之前加载，否则切面拿到的是空的数据源。这个问题通过实现 PriorityOrdered 接口来提升切面加载的优先级解决的。

你

👁️ 面试官内心：很好，他能把原理和他实际项目的坑联系起来。这是 5 年经验的差异化水平。

官

第四层 · 架构取舍与项目结合："如果让你重新设计恒生的多数据源组件，你会把 ApplicationContext 用在哪些更巧妙的地方？如果同时有 500 个 DataSource Bean 注册到容器，ApplicationContext 会不会成为瓶颈？

第一个问题：可以用 ApplicationContext 的事件机制做热加载——当运维在配置中心新增一个数据源配置，发布一个 DataSourceAddedEvent，监听器动态向容器注册新的 DataSource Bean（通过 BeanDefinitionRegistry）。现在你们用的可能是重启生效的方式，用事件机制可以在不重启的情况下动态加载新数据源。

第二个问题：500 个 DataSource 确实有潜在的性能问题。因为 singletonObjects 是 ConcurrentHashMap，查询是 O(1)，但启动时要创建 500 个连接池（每个 HikariCP 连接池有一定初始化成本），会拖慢启动。解决方案是把不常用的数据源标记为 @Lazy，或者用 FactoryBean 模式按需创建。

你

👁️ 面试官内心：他不仅理解了容器，还能对实际系统做架构层的推演。这是在考察「架构思维」而非「API 记忆」。

官

附加送命题："你说 BeanFactory 的 getBean 会触发创建——那如果在创建过程中，又触发了另一个 Bean 的 getBean，形成了 A→B→A 的循环，会发生什么？Spring 怎么处理？

这就是经典的循环依赖问题。Spring 用三级缓存来解决：

一级缓存（singletonObjects）存成品；二级缓存（earlySingletonObjects）存半成品早期引用；三级缓存（singletonFactories）存 ObjectFactory 工厂。

创建 A → 发现需要 B → 创建 B → 发现需要 A → 从三级缓存拿到 A 的早期引用 → B 注入 A → B 完成 → A 注入 B → A 完成。

但构造器注入的循环依赖无法解决——因为对象还没创建出来（new 没返回），无法放入任何缓存。
