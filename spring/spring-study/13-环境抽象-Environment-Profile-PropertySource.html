Part 2 · 触及灵魂的设计原理

🤔 假如没有 Environment 抽象，怎么做多环境切换？

真实噩梦：五套环境（dev/test/staging/prod/canary），每套的数据库、Redis、MQ、外部服务 URL 都不同。没有统一抽象时只能靠 Maven Profile + 资源过滤在构建时替换占位符——同一个 jar 包不能在所有环境运行，每个环境都要重新打一个包。

Spring Environment + Profile 把这套机制搬到了运行时：一个 jar，通过启动参数切换环境。

💡 设计巧思：MutablePropertySources — 一个有序列表决定配置优先级

PropertySources 不是一个 Map（Map 无序）——它是一个 CopyOnWriteArrayList，且列表顺序 = 优先级。排在前面（index 小）的 PropertySource 优先被查找。

// Environment 查找配置的流程（伪代码）
public String getProperty(String key) {
    for (PropertySource source : propertySources) {  // 按优先级遍历
        if (source.containsProperty(key)) {
            return source.getProperty(key);  // 找到即返回
        }
    }
    return null;  // 都找不到
}

"啊哈时刻"：这个设计极其简单——没有复杂的优先级算法，没有加权评分。就是一个 for 循环，谁排前面谁优先。配置的来源可以无限扩展（数据库、Consul、Vault），只要 new 一个 PropertySource 插到列表里就行。

💡 Profile 的巧思：不是"分支"，是"条件过滤"

很多人把 Profile 理解为 if-else。但它的真正巧思在于：它不是让代码判断环境，而是让容器决定要不要注册这个 Bean。

// Profile 在底层是 @Conditional(ProfileCondition.class)
// ProfileCondition.matches() 检查 Environment.acceptsProfiles()

// 所以 @Profile("dev") 本质就是：
// 如果 activatesProfiles 包含 "dev" → 注册 Bean，否则跳过

这意味着 条件不满足的 Bean 根本不存在于容器中——不是"存在但不执行"，是"不存在"。这避免了 if-else 在业务代码中到处散布。

⚠️ 边界与陷阱

陷阱1：jar 外部配置的优先级

jar 同目录下的 config/ 文件夹优先级最高——很多人用容器部署时不小心在里面放了一个旧配置文件，覆盖了 jar 内的新配置。

陷阱2：@Profile 和 @ConditionalOnProperty 冲突

两者独立判断，同时标注时"与"关系。两个条件都满足才注册。

陷阱3：EnvironmentPostProcessor 的执行顺序

自定义的 PropertySource 在这个阶段插入，但如果依赖了其他 PropertySource（比如配置中心），要注意 Spring Boot 提供的 DeferredLog。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Profile 激活了多个 profile 时，同一属性在 dev.yml 和 test.yml 都有定义，哪个生效？

活跃 Profile 列表中最后一个的配置生效。因为 PropertySource 链表是按 Profile 列表顺序添加的——后面的覆盖前面的。如果 spring.profiles.active: dev,test，test.yml 会覆盖 dev.yml 的同名配置。

A
Q

@PropertySource 和 application.yml 的加载顺序？

application.yml 在 configurationProperties 阶段加载，排在 PropertySource 链前面；@PropertySource 在 ConfigurationClassPostProcessor 阶段加载，排在后面。所以 application.yml 优先级高于 @PropertySource。

A
Q

@Profile 和 @ConditionalOnProperty 有什么区别？

@Profile 只看"哪个环境被激活"，适用于粗粒度的环境区分（dev/test/prod 整包切换）。@ConditionalOnProperty 看"某个配置的具体值"，适用于细粒度的特性开关（feature.toggle=true 单独控制某个功能）。@Profile 的本质是 @Conditional(ProfileCondition.class)，它只是 @Conditional 的一个特化。

A
Q

🔧 场景题：生产上通过环境变量注入数据库密码。运维不小心在 application.yml 里也写了一个数据库密码（但值是错的）。为什么线上还是连上了正确的数据库？

因为环境变量在 Spring 的 PropertySource 链中排在更前面——优先级高于 application.yml。Spring 先找到环境变量中的正确密码就用它了，application.yml 中错误的密码被跳过了。但这是运气好——如果环境变量和配置文件都有同名字段，配置文件的错误被掩盖，排查起来极其困难。

A
Q

🔧 场景题：线上排查发现一个 Bean 在 prod 环境没有注册（@Profile("dev")），但灰度环境（@Profile("canary")）也没有——新环境需要加支持。怎么让旧代码支持新环境而不改代码？

旧代码 @Profile("dev") 是硬编码。如果不想改旧代码（可能很多处），在启动参数中让 canary 伪装成 dev 环境：spring.profiles.active=canary,dev。这样 @Profile("dev") 的 Bean 在 canary 环境中也会被注册。更好的方案是：以后不要在生产代码中直接写 @Profile("dev")，改用 @Profile("!prod") 或自定义 @Conditional——表达"非生产环境"，而不是"dev 环境"。

A
Q

自定义 PropertySource 怎么做？你实现过吗？

实现 EnvironmentPostProcessor 接口，在 postProcessEnvironment() 中往 environment.getPropertySources() 里 addFirst()（最高优先级）或 addLast()（最低优先级）插入自定义的 PropertySource。JSON 配置、数据库配置、Vault 密钥都是这样接入的。注意要在 META-INF/spring.factories 中注册这个 Processor。

A
Q

Environment 和 @Value 的关系？是先有 Environment 还是先有 @Value？

先有 Environment。refresh() 步骤 1-5 中已经准备好了 Environment，步骤 5 的 BFPP 阶段中 PropertySourcesPlaceholderConfigurer 从 Environment 取值并替换 ${} 占位符，然后步骤 11 才创建 Bean。所以 @Value("${xxx}") 的值在 Bean 创建前就从 Environment 中取到了。

📋 面试官总结

1. PropertySource 优先级链 — 列表顺序 = 优先级，不是 Map

2. Profile 的本质 — @Conditional(ProfileCondition.class)，不是 if-else

3. 多 Profile 激活时的覆盖规则 — 最后一个覆盖前面

4. jar 外部配置 vs jar 内部配置 — 运维可能放了旧文件覆盖新配置

5. EnvironmentPostProcessor — 扩展 PropertySource 的标准入口
