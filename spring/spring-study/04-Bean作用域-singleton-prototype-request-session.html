Part 1 · 基础认知

四种常用作用域速查

作用域存活范围创建时机销毁
singleton整个容器容器启动时（默认）容器关闭
prototype每次获取每次 getBean() / 注入时不自动销毁！
request一次 HTTP 请求请求到达时请求结束时
session一次 HTTP 会话会话创建时会话过期/销毁

基本用法

// singleton — 默认，可以不写
@Service
public class UserService { }

// prototype — 每次获取都 new
@Service @Scope("prototype")
public class ReportGenerator { }

// request — 每个 HTTP 请求一个实例（Web 环境才可用）
@Component @Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { }

// session — 每个用户会话一个实例
@Component @Scope(value = "session", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class ShoppingCart { }

// 常量写法（推荐）
@Service @Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class ReportGenerator { }

Part 2 · 触及灵魂的设计原理

🤔 如果只有 singleton，会发生什么？

痛点 1：有状态的 Bean 互相污染。一个报表生成器被多个用户并发使用，如果共享同一个实例，用户 A 设置的参数会覆盖用户 B 的。

痛点 2：高并发下 singleton 的锁竞争。如果 singleton Bean 内部的某个方法加了 synchronized，所有线程都在等一把锁。

所以 prototype 存在的意义是："我需要一个 Bean 的"款式"，但每次都要一件新的"。

💡 设计巧思：prototype 的"孤儿"困境 — Spring 只生不养

这是 Spring 文档里最容易被忽略的一句话："Spring does not manage the complete lifecycle of a prototype bean."

什么意思？Spring 创建 prototype Bean 后就"放手"了——不再管理它的销毁。@PreDestroy 标注的方法不会被调用，因为 Spring 没有"持有"这个 Bean。

@Component @Scope("prototype")
public class HeavyProcessor implements Closeable {
    @PostConstruct public void init()  { /* 会执行 ✅ */ }
    @PreDestroy    public void clean() { /* 不会执行 ❌ */ }
    
    // 如果持有连接池等资源……内存泄漏！
}

设计者的考量：prototype 可能被创建成百上千次——如果 Spring 要追踪每一个 prototype 的生命周期，内存开销巨大。所以设计者做了一个权衡：交给你自己管。你需要手动调用 Bean 的清理方法，或者注册 destruction callback（这需要你自己持有 Bean 引用）。

💡 设计巧思：Scoped Proxy — 长作用域注入短作用域的秘密

场景：singleton 的 Controller 要注入 request 作用域的 RequestContext。但 singleton 在启动时就创建了，当时根本没有 request！

// 如果不加 proxyMode → 启动时就报错！
@RestController
public class UserController {
    @Autowired
    private RequestContext ctx; // singleton 注入 request → 💥
}

// 解法：proxyMode = ScopedProxyMode.TARGET_CLASS
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { }

// 现在 UserController 注入的不是 RequestContext 本身
// 而是 CGLIB 生成的代理
// 代理在每次调用 ctx.getXxx() 时才去当前线程的 RequestAttributes 里找真正的实例

巧思：代理对象是 singleton 的，但它"假装"自己每次都能拿到 request 作用域的实例。实际上它是通过 RequestContextHolder（ThreadLocal）在运行时获取的。

⚠️ 边界与陷阱

单例 Bean 持有 prototype Bean → prototype 退化为 singleton

因为单例只会被创建一次，它的所有依赖在那一刻就被注入——prototype Bean 只创建了一次，之后再也不会重新创建。

request / session 作用域在非 Web 环境下不可用

在纯 @SpringBootTest 中（不带 webEnvironment），request 作用域会抛异常。需要 @SpringBootTest(webEnvironment = RANDOM_PORT)。

prototype 不会触发 @PreDestroy

这可能导致数据库连接、文件句柄等资源泄漏。

Part 3 · 大厂面试官的灵魂追问

Q

singleton 和 prototype，日常开发中用哪个多？

singleton 占了 95% 以上。因为绝大部分 Bean 是无状态的——Service、Repository、Controller 都只是一个"方法容器"，不持有会话级数据。prototype 只在需要每次不同实例时用。

A
💬 考核意图：看你是不是"为了 prototype 而 prototype"。很多人以为 prototype 更安全就用它，这是浪费内存。
Q

singleton Bean 里注入 prototype Bean，prototype 还有效吗？

失效了。因为 singleton 只创建一次，它的依赖在那一刻就被注入了——prototype 也就变成了"事实上的 singleton"。解法：不要注入 prototype Bean 本身，而是注入 ObjectProvider 或 ApplicationContext，每次使用时手动获取新实例。

A
Q

request 作用域的 Bean 在 @Async 方法里会怎样？

直接炸。request 作用域依赖当前线程的 RequestAttributes，@Async 换到了线程池线程 → ThreadLocal 里没有请求上下文 → 抛 ScopeNotActiveException。解法：要么把需要的数据在主线程提取出来传过去，要么用自定义 TaskDecorator 传递 RequestContextHolder。

A
💬 深挖意图：考察是否理解 ThreadLocal 与作用域的关系。
Q

多数据源场景——切换数据源的 lookup key 是存在 ThreadLocal 里的。这个 ThreadLocal 变量应该放在 singleton Bean 里，还是 prototype Bean 里？

放在 singleton Bean 里。ThreadLocal 的作用域是线程级别，不是 Bean 级别。singleton Bean 内部的 ThreadLocal 字段——每个线程看到的是不同的值，不存在并发冲突。如果放在 prototype 里反而多余：每次新建 Bean，ThreadLocal 也是新的，之前线程的 ThreadLocal 里的值就丢了。

A
