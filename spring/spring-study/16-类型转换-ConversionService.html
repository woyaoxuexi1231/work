Part 1 · 基础认知

三种转换体系对比

体系接口应用层
PropertyEditorPropertyEditor (JDK)XML 配置的 String → Object
ConverterConverter<S, T>⭐ 任意类型互转
FormatterFormatter<T>String ↔ 对象（带 Locale）
// 注册 Converter — 全局生效
@Component
public class StringToLocalDateConverter implements Converter<String, LocalDate> {
    @Override
    public LocalDate convert(String source) {
        return LocalDate.parse(source, DateTimeFormatter.ofPattern("yyyy-MM-dd"));
    }
}

// Spring Boot 会自动把 Converter 注册到 ConversionService
// 之后 Controller 方法的 LocalDate 参数直接从请求参数转换

// 手动使用
@Autowired private ConversionService conversionService;
LocalDate date = conversionService.convert("2025-01-15", LocalDate.class);

// Formatter — 带 Locale
@Component
public class LocalDateFormatter implements Formatter<LocalDate> {
    @Override public LocalDate parse(String text, Locale locale) { ... }
    @Override public String print(LocalDate object, Locale locale) { ... }
}

Part 2 · 触及灵魂

🤔 如果没有统一的类型转换，Spring 怎么处理"请求参数 String → Controller 方法参数"？

没有 Converter 之前，Spring 用 PropertyEditor（JDK 1.0 时代的东西）。PropertyEditor 有两个致命问题：

① 线程不安全：PropertyEditor 有状态（setValue/getValue），Spring 不得不为每个请求创建新实例。

② 只能 String → Object：不能做任意类型互转（比如 Long → LocalDate）。

Spring 3.0 引入 Converter 体系：无状态、线程安全、S→T 通用。现代 Spring 中 Converter 是主力，PropertyEditor 只保留给 XML 配置的向后兼容。

💡 设计巧思：GenericConverter — 一对多的万能转换

普通 Converter<S, T> 只能一对一。但有很多场景需要一个 Converter 处理多种类型（比如 String → 枚举、String → 集合）。GenericConverter 解决了这个问题：

// Spring 内置的 StringToCollectionConverter 就是 GenericConverter
// 它能处理 String → List<T>、String → Set<T>（T 通过递归转换）
Set<ConvertiblePair> getConvertibleTypes() {
    return Set.of(
        new ConvertiblePair(String.class, Collection.class),
        new ConvertiblePair(String.class, List.class),
        new ConvertiblePair(String.class, Set.class)
    );
}

"啊哈时刻"：ConversionService 会为每次转换动态查找最合适的 Converter。如果找不到直接匹配的，它会尝试转换链：String → long → Long → Object。每一步都可能由不同的 Converter 完成。

🔬 Spring Boot 自动注册你的 Converter

只要你的 Converter 实现了 Converter 接口且标注了 @Component → Spring Boot 的 WebMvcAutoConfiguration 会自动发现它并注册到 FormattingConversionService——这是 DefaultFormattingConversionService（实现了 ConversionService + FormatterRegistry）的实例。

⚠️ 边界与陷阱

陷阱1：Formatter 优先于 Converter

FormatterRegistry 在查找时先查 Formatter（有 Locale），再查 Converter。如果你注册了 Formatter，Converter 可能被跳过。

陷阱2：Converter 失败时返回 null

不是抛异常。null 被当作类型不匹配，Spring MVC 会转成 TypeMismatchException → 400。

Part 3 · 大厂面试官的灵魂追问（6 组）

Q

Converter 和 Formatter 的区别？为什么需要两个？

Converter 是通用类型转换（S→T），无状态。Formatter 专门处理 String ↔ Object，带 Locale（用于国际化数字格式、日期格式）。Formatter 在 Web 层（请求参数绑定）中优先级更高，因为它有 Locale 感知。

A
Q

PropertyEditor 和 Converter 的根本区别？

PropertyEditor 有状态（线程不安全）、只能 String↔Object。Converter 无状态（线程安全）、支持任意类型互转。Spring 3.0 引入 Converter 就是为了解决 PropertyEditor 这两个致命缺陷。

A
Q

🔧 场景题：前端传了 "2025-01-15" 字符串，Controller 参数是 LocalDate，类型不匹配报 400。怎么让它自动转换？

Spring Boot 默认配置了 JavaTimeModule（Jackson）用于 JSON 反序列化。但对于 @RequestParam 的字符串参数，需要注册 Formatter：实现 Formatter<LocalDate> + @Component。或者更简单：在 application.yml 中配置 spring.mvc.format.date=yyyy-MM-dd，Spring Boot 会自动注册对应的 Formatter。

A
Q

🔧 场景题：你的系统对外暴露 API 给 5 个不同前端团队，他们对日期格式要求不一致（yyyy-MM-dd、yyyy/MM/dd、dd-MM-yyyy）。怎么用一个 Converter 支持多种格式？

在 Converter 里按顺序尝试多个 DateTimeFormatter 解析：先 try yyyy-MM-dd，失败再 try yyyy/MM/dd，依次类推。但更合理的方案是前端统一格式，后端只接受一种——多格式是接口设计的问题，不是技术问题。如果实在要兼容，建议在新版 API 中强制一种格式并提供迁移文档。

A
Q

如果 Converter 和 Formatter 都能处理同一种转换，谁生效？

Formatter 优先。FormattingConversionService 先查 FormatterRegistry（Formatter），再查 ConverterRegistry（Converter）。因为 Formatter 有 Locale，被认为更适合 Web 场景。

A
Q

ConversionService 能在纯 Java 代码中手动用吗？

可以。@Autowired 注入 ConversionService，然后 conversionService.convert(source, TargetType.class)。这在非 Web 场景（消息消费、定时任务）中也很有用。Spring 的 DataBinder、BeanWrapper 都是基于 ConversionService 做类型转换的。

A

📋 面试官总结

1. PropertyEditor → Converter 的演进 — 有状态/线程不安全 → 无状态/线程安全

2. Formatter vs Converter — Locale 感知 vs 通用转换

3. GenericConverter — 一对多转换的核心抽象

4. 自动注册 — @Component + 实现 Converter 接口 = 全局生效

⚠️ 高频陷阱：Formatter 优先于 Converter；Converter 失败返回 null 而非抛异常
