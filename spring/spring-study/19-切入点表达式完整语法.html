Part 1 · 基础认知

六种指示符一张表

指示符匹配维度示例
execution方法签名execution(* com.ex.service.*.*(..))
within类/包within(com.ex.service.*)
this代理对象类型this(com.ex.UserService)
target目标对象类型target(com.ex.UserService)
args方法参数类型args(Long, String)
@annotation方法上有特定注解@annotation(com.ex.Transactional)
// execution — 最常用，完整语法
execution(修饰符? 返回类型 包.类.方法(参数) throws?)
// 示例：
execution(public * com.example..*.save*(..))  // public、任意返回、save开头
execution(* com.example.service.*.*(Long, ..)) // 第一个参数Long
execution(* com.example..*.*(String))          // 仅一个String参数

// within — 简化版，只匹配类
within(com.example.service.*)                  // service包下所有类

// @annotation — 最灵活，按注解匹配
@annotation(org.springframework.transaction.annotation.Transactional)

// 组合使用（AND / OR / NOT）
@Before("execution(* com.ex..*.*(..)) && !within(com.ex.internal.*)")
@Around("@annotation(Monitor) || @annotation(Timed)")

Part 2 · 触及灵魂

🤔 this 和 target 的区别是什么？为什么需要两个？

this 匹配的是代理对象的类型，target 匹配的是被代理的目标对象的类型。

如果 UserServiceImpl 实现 UserService 接口：

  • JDK 代理时：代理类型 = UserService（接口），目标类型 = UserServiceImpl
  • this(UserServiceImpl) → 不匹配（代理不是 UserServiceImpl）
  • target(UserServiceImpl) → 匹配
  • CGLIB 代理时：代理类型 = UserServiceImpl 子类，目标类型 = UserServiceImpl
  • this(UserServiceImpl) → 匹配（代理是子类实例）
  • target(UserServiceImpl) → 匹配

这个差异在 Spring Boot 2.x 强制 CGLIB 后基本消失了，但理解它有助于理解代理的本质。

💡 设计巧思：切点表达式的"组合力量"

切点表达式的真正威力不是单个指示符，而是 && 和 || 的组合。就像 SQL 的 WHERE 条件：

// 所有 Service 的方法，但排除 InternalService、且排除 @NoLog 标记的
@Pointcut("execution(* com.example.service.*.*(..))")
public void serviceLayer() {}
@Pointcut("!within(com.example.service.InternalService)")
public void notInternal() {}
@Pointcut("!@annotation(com.example.NoLog)")
public void notNoLog() {}

@Before("serviceLayer() && notInternal() && notNoLog()")
public void log() { }
⚠️ 边界：切点表达式的缓存与性能

陷阱：

通配符太宽 → 匹配太多类 → 每个 Bean 创建时都要判断是否匹配这个切点。within(com.example..*) 这种全量通配在生产中要谨慎。

优化：

Spring 会缓存切点匹配结果。同一个类只会评估一次。但初次评估的消耗还是存在的。

Part 3 · 大厂面试官的灵魂追问（6 组）

Q

execution 和 within 的区别？

execution 匹配方法级别（精确到方法签名），within 匹配类型级别（只精确到类）。within 比 execution 更快（粒度更粗）。能用 within 就不用 execution。

A
Q

@annotation 和 @within 有什么区别？

@annotation 匹配方法上有该注解的，@within 匹配类上有该注解的所有方法。比如 @Transactional 在类上 → @within(@Transactional) 会匹配该类的所有方法；@annotation(@Transactional) 只匹配标注在方法上的。

A
Q

this 和 target 在 Spring Boot 2.x CGLIB 默认代理下还有区别吗？

基本没有了。CGLIB 代理是目标类的子类，所以 this 和 target 通常指向同一个类型。但如果你显式配置了 JDK 代理（proxy-target-class=false），区别就又回来了。

A
Q

🔧 场景题：你要给所有标注了 @AuditLog 的方法加上审计日志。切点怎么写？

@annotation(com.example.AuditLog)。加一个 @AuditLog 自定义注解，开发只需要在需要审计的方法上加 @AuditLog，切面自动生效。需要审计信息的话，在切面中通过反射读 @AuditLog 的属性。

A
Q

🔧 场景题：Controller 层 100 个方法，但你想只给前 3 个最核心的加性能监控，之后逐步推广。切点怎么灵活控制范围？

用 @annotation 而不是包路径。先给 3 个核心方法加 @Monitored。切点 = @annotation(Monitored)。后面推广时只需给新方法加注解。如果后面不想用注解了，可以加一个 @ConditionalOnProperty 开关，配置文件中控制切面是否启用。

A
Q

切点表达式中 args 和 execution 的区别？

execution 的括号里匹配的是声明的参数类型，args 匹配的是运行时的实际参数类型。比如参数声明为 Object，但运行时传的是 String → execution(Object) 匹配，args(String) 也匹配。args 更灵活但更慢（运行时判断）。

A

📋 面试官总结

1. execution vs within — 方法级 vs 类级，粒度与性能的权衡

2. this vs target — CGLIB 代理下基本无区别，但理解原理很重要

3. @annotation 的实战价值 — 最灵活的匹配方式，渐进式推广

4. 切点组合 — && / || / ! 是切点的灵魂

⚠️ 高频陷阱：通配符太宽导致性能问题；args 运行时匹配比 execution 慢
