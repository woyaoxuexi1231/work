Part 2 · 触及灵魂的设计原理

🤔 假如没有 BeanPostProcessor，会发生什么？

想象你有 200 个 Service，每个都需要在方法执行时打日志。没有 BPP，每份代码都要加 try-finally：

// 每个 Service 都要这样写 → 200 份重复代码
public void doBiz() {
    log.info("开始");
    try { /* 业务 */ } finally { log.info("结束"); }
}

有了 BPP + AOP：一个切面搞定 200 个 Service，零侵入。BPP 在生命周期中提供了一个"插桩点"，让框架能力（事务、日志、安全）可以在完全不修改业务代码的前提下注入。

💡 设计巧思：为什么 BPP 分 before 和 after？

巧思：before 和 after 不是随便分的——before 让你在 @PostConstruct 之前做预处理（如注入代理不可见的上下文），after 让你在 @PostConstruct 之后做后处理（如创建 AOP 代理）。

如果只有 after，那些需要在初始化之前就完成的逻辑（比如给 @Resource 赋值）就没有地方放了。如果只有 before，AOP 代理就永远在 @PostConstruct 之前创建——但代理对象不一定能正确处理 @PostConstruct 的回调（因为目标对象还在原始状态）。

before → @PostConstruct → afterPropertiesSet → init-method → after

↑ 预处理 ↑ 用户初始化逻辑 ↑ 后处理（AOP代理）

🔬 底层机制：doCreateBean() 源码骨架 
// AbstractAutowireCapableBeanFactory.doCreateBean()
protected Object doCreateBean(String beanName, RootBeanDefinition mbd, Object[] args) {
    // 1. 实例化
    BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);
    Object bean = instanceWrapper.getWrappedInstance();

    // 2. 放入三级缓存（循环依赖用）
    addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

    // 3. 属性填充（依赖注入）
    populateBean(beanName, mbd, instanceWrapper);

    // 4. 初始化
    Object exposedObject = initializeBean(beanName, bean, mbd);
    //   ↓ 内部：
    //   invokeAwareMethods()        → Aware 回调
    //   applyBeanPostProcessorsBeforeInitialization() → BPP.before → @PostConstruct
    //   invokeInitMethods()         → afterPropertiesSet → init-method
    //   applyBeanPostProcessorsAfterInitialization()  → BPP.after → AOP代理

    return exposedObject;
}

为什么 @PostConstruct 里拿不到 AOP 代理？因为 @PostConstruct 在 BPP.before 阶段执行，而 AOP 代理在 BPP.after 阶段创建。代理是"后来者"。

⚠️ 边界与陷阱

陷阱1：@PostConstruct 里调 @Async → 不生效

代理还没创建，this 是原始对象，调的是原始方法。

陷阱2：BeanPostProcessor 之间执行顺序

用 @Order 或实现 PriorityOrdered/Ordered 控制。CommonAnnotationBeanPostProcessor 优先级高于 AbstractAutoProxyCreator，所以 @PostConstruct 一定先于 AOP。

陷阱3：BFPP 和 BPP 名称相似但完全不同

BFPP 操作的是 BeanDefinition（蓝图），BPP 操作的是 Bean 实例（成品）。BFPP 在实例化之前，BPP 在初始化前后。

Part 3 · 大厂面试官的灵魂追问（8 组）

Q

说下 Spring Bean 的完整生命周期。

14 步：BeanDefinition 加载 → BFPP 修改定义 → 实例化 → 属性填充 → Aware → BPP.before → @PostConstruct → afterPropertiesSet → init-method → BPP.after(AOP代理) → 单例池 → 就绪 → @PreDestroy → 销毁。

A
💡 考核意图：80%的人只能说出"实例化→初始化→销毁"三个词。说出 BeanPostProcessor 的 before/after 和 AOP 代理创建时机，才是及格线。
Q

@PostConstruct、afterPropertiesSet、init-method，执行顺序是什么？为什么？

顺序：@PostConstruct → afterPropertiesSet → init-method。源码层面看 initializeBean()：先调 BPP.before（含 @PostConstruct），再调 invokeInitMethods()（含 afterPropertiesSet + init-method）。设计意图：注解优先于接口，接口优先于 XML 配置——越靠近业务代码的越早执行。

A
Q

BFPP 和 BPP 有什么区别？为什么需要两个？

BFPP 改"图纸"（BeanDefinition），BPP 改"成品"（Bean 实例）。设计者的巧思：把配置阶段和运行时阶段严格分离。BFPP 在实例化前执行——因为改图纸要在施工前。BPP 在初始化前后执行。如果混在一起，就会出现"车已经开动了再改路线"的尴尬。

A
Q

Bean 什么时候从二级缓存移到一级缓存？

Bean 完成初始化（initializeBean）后，从二级缓存 earlySingletonObjects 移除，放入一级缓存 singletonObjects。生命周期步骤 10 之后。此时对象已经完整初始化（包括 AOP 代理）。

A
💡 考核意图：把生命周期和循环依赖两个知识点关联起来，考察整体理解。
Q

🔧 场景题：你在恒生做多数据源组件时，需要在每个 DataSource Bean 初始化后验证连接可用性（ping 一下）。你会怎么利用 BeanPostProcessor 实现？

写一个 DataSourceHealthCheckPostProcessor 实现 BPP，在 postProcessAfterInitialization 里判断 bean 是否是 DataSource 类型——如果是，就 try conn.isValid(3) 验证连接。验证不通过记录日志告警，但不要抛异常（避免阻塞其他 Bean 的初始化）。这样所有 DataSource 初始化后自动验证，无需在每个配置类里写 try-catch。

A
💡 考核意图：不只要知道 BPP 的概念，还要能在真实场景中"看到可以用 BPP 解决什么问题"。
Q

🔧 场景题：你的 RSA+AES 加密 Filter 依赖一个密钥管理器 Bean。线上发现请求进来时密钥管理器还没初始化完成（@PostConstruct 里的密钥加载耗时 5 秒），导致 Filter 拿到的是 null。怎么排查和解决？

排查：查启动日志，看密钥管理器的 @PostConstruct 耗时。确认 Filter 是在哪个阶段注册的（Servlet Filter 可能在 Bean 初始化完成前就激活了）。解决：① 在 Filter 中加启动等待逻辑（轮询等待密钥管理器就绪）或 ② 把密钥加载从 @PostConstruct 提前到 BFPP 阶段（用 BeanFactoryPostProcessor 同步加载）或 ③ 把 Filter 改为延迟注入密钥管理器（用 Provider / ObjectProvider）。

A
💡 考核意图：把生命周期和 Web 容器的启动顺序结合起来考察，这是线上真实故障的高发区。
Q

两个 BeanPostProcessor 之间的执行顺序怎么控制？

实现 PriorityOrdered 或 Ordered 接口，或加 @Order 注解。数字越小越先执行。比如 CommonAnnotationBeanPostProcessor（@PostConstruct）的 order 是 LOWEST_PRECEDENCE-3，AbstractAutoProxyCreator（AOP代理）是 LOWEST_PRECEDENCE——所以 @PostConstruct 一定在 AOP 之前。

A
Q

prototype Bean 的 @PreDestroy 会执行吗？

不会。Spring 只管 prototype Bean 的创建，不管销毁。文档写得清楚：prototype bean 的生命周期你自行管理。如果 prototype Bean 持有连接等资源，必须手动释放。这和 singleton 不同——singleton 的 @PreDestroy 会在容器关闭时触发。

A

📋 面试官总结：Bean 生命周期高频考点

1. 生命周期的完整阶段 — 不是背 14 个名字，而是理解"为什么需要这个阶段"

2. BeanPostProcessor 的 before/after 和 AOP 的关系 — @PostConstruct 和代理的先后顺序

3. BFPP vs BPP — 操作对象不同、执行时机不同、经典实现不同

4. 三个初始化回调的顺序 — @PostConstruct → afterPropertiesSet → init-method，以及为什么

5. 循环依赖与三级缓存在生命周期中的位置 — 实例化后→三级缓存→属性填充→初始化→一级缓存

⚠️ 大厂高频挖坑点：@PostConstruct 里调 @Async 不生效、prototype @PreDestroy 不触发、Filter 拿到未初始化完成的 Bean
