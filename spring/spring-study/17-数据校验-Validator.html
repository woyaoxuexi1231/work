Part 1 · 基础认知

Spring Validator vs Bean Validation (JSR-303)

对比Spring ValidatorBean Validation (JSR-303)
接口org.springframework.validation.Validatorjavax.validation.Validator
校验逻辑Java 代码（编写 validate()）注解声明（@NotNull/@Size 等）
分组校验❌✅ @Validated(Group.class)
现状遗留项目、兼容⭐ 主流方案
// Bean Validation — 注解声明式校验
public class UserDTO {
    @NotBlank(message = "姓名不能为空")
    private String name;

    @Email @NotBlank
    private String email;

    @Min(0) @Max(150)
    private int age;

    @NotNull(groups = Update.class)  // 只在更新场景校验
    private Long id;
}

// Controller — @Valid 触发校验
@PostMapping("/users")
public Result create(@Valid @RequestBody UserDTO dto) { ... }

// Spring Validator — 编程式校验（旧风格，很少用）
public class UserValidator implements Validator {
    @Override public boolean supports(Class clazz) { return UserDTO.class.equals(clazz); }
    @Override public void validate(Object target, Errors errors) {
        UserDTO u = (UserDTO) target;
        if (u.getName() == null || u.getName().isEmpty())
            errors.rejectValue("name", "NotEmpty");
    }
}

Part 2 · 触及灵魂

🤔 假如校验逻辑散落在 Controller、Service、DAO 层？

真实噩梦：Controller 里判断 name 是否为空，Service 里判断 age 是否 >0，DAO 里判断 email 格式。同一份数据有三层校验，校验规则改一处其他层不同步。

Spring 的解法：把校验规则放到 DTO 上（注解），然后通过 AOP 在 Controller 层统一触发 @Valid。校验逻辑不散落、不改业务代码、一段 DTO 定义所有规则。

💡 设计巧思：LocalValidatorFactoryBean — Spring 与 Bean Validation 的桥梁

Spring 的 Validator 接口和 JSR-303 的 Validator 不是一回事。Spring 用 LocalValidatorFactoryBean（实现 Spring 的 Validator）来适配 Hibernate Validator（JSR-303 的实现）。

// Spring Boot 自动配置了这个 Bean
// 所以 @Valid 触发的其实是：
// RequestResponseBodyMethodProcessor → DataBinder →
//   → SpringValidatorAdapter（包装了 Hibernate Validator）
//     → 遍历 @NotNull/@Size/@Email 注解执行校验
//       → 结果写入 BindingResult

"啊哈时刻"：Spring 没有重新发明校验——它引入了一个适配层，让 Bean Validation 成为 Spring 的"默认校验引擎"。你声明的是 JSR-303 注解，但 Spring 在 Bean 生命周期中统一触发。

🔬 @Valid 和 @Validated 的本质区别

@Valid 来自 JSR-303，只触发基本校验。@Validated 是 Spring 的包装，额外支持分组校验和方法级校验（标注在类上）。

⚠️ 边界与陷阱

陷阱1：@Valid 只触发一层校验

嵌套对象的校验需要 @Valid 层层标注。DTO 里有 List<Item> → Item 里也需要 @Valid。

陷阱2：校验失败的 Exception 类型不同

@RequestBody → MethodArgumentNotValidException。@RequestParam → ConstraintViolationException（需要 @Validated 在类上）。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Valid 和 @Validated 有什么区别？

@Valid 是 JSR-303 标准注解，触发基本校验。@Validated 是 Spring 的扩展，支持分组校验和类级别的方法参数校验。@Valid 用于 @RequestBody 参数，@Validated 用于 @RequestParam/@PathVariable（需标注在类上）。

A
Q

@RequestBody 校验失败和 @RequestParam 校验失败抛的异常一样吗？

不一样。@RequestBody @Valid → MethodArgumentNotValidException（400）。@RequestParam @Validated（类上） → ConstraintViolationException（500！）。这就是为什么很多人发现 @RequestParam 校验失败返回 500 而不是 400——需要在 @ControllerAdvice 中单独处理 ConstraintViolationException。

A
Q

分组校验怎么用？什么场景需要分组？

同一个 DTO，创建时不需要 ID（@NotNull(groups=Update.class)），更新时需要 ID。@Validated(Update.class) 指定校验分组。分组本质是多套校验规则共享一个 DTO——避免了为每个场景建一个 DTO。

A
Q

🔧 场景题：DTO 中有一个字段只在"审批通过"场景需要校验，其他场景不需要。怎么做？

定义 Approval.class 分组，该字段注解 @NotBlank(groups = Approval.class)。Controller 方法用 @Validated(Approval.class) 触发。如果业务逻辑太复杂（比如需要动态判断），分组不够用 → 用自定义 Validator 或 @AssertTrue 写 SpEL 表达式。

A
Q

🔧 场景题：线上发现有些恶意请求绕过了 Controller 层校验（比如通过内部调用），Service 层没做校验导致脏数据入库。怎么设计多层校验？

Controller 做格式校验（@Valid → 字段不为空、格式正确），Service 做业务校验（重复、权限、状态机）。Controller 校验防参数注入，Service 校验防业务黑洞。不能只用 Controller 校验——任何绕过 Controller 的调用（定时任务、MQ 消费者、内部调用）都会导致脏数据。Service 校验可以用 @Validated + @Valid 自动触发，也可以手动 Validator.validate()。

A
Q

自定义注解校验怎么做？比如校验身份证号。

两步：① 定义 @IdCardNo 注解，标注 @Constraint(validatedBy = IdCardNoValidator.class)。② 实现 ConstraintValidator 接口完成校验逻辑。然后用 @IdCardNo 像 @NotNull 一样用。

A
Q

校验失败后的 BindingResult 和 Exception 的关系？

如果 Controller 方法有 BindingResult 参数 → 校验失败不进 Exception，结果写入 BindingResult（你手动判断 hasErrors()）。如果没有 BindingResult → Spring 自动抛 MethodArgumentNotValidException。推荐不写 BindingResult，让 @ControllerAdvice 统一处理——减少重复代码。

A

📋 面试官总结

1. @Valid vs @Validated — JSR-303 标准 vs Spring 扩展，分组校验是核心差异

2. 校验失败异常类型 — @RequestBody (400) vs @RequestParam (500)，很多人踩坑

3. 多层校验设计 — Controller 格式校验 + Service 业务校验，不能只靠一层

4. BindingResult — 有就不抛异常，没有才抛

⚠️ 高频陷阱：@RequestParam 校验需 @Validated 在类上；只有 @RequestBody 自动触发 400
