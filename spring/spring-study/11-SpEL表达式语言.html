Part 2 · 触及灵魂的设计原理

🤔 假如没有 SpEL，@PreAuthorize 怎么写？

Spring Security 的权限校验需要灵活判断——"当前用户是否是记录的拥有者"、"用户年龄是否大于 18"、"角色是否是 ADMIN 或 SUPER_ADMIN"。没有 SpEL，这些判断必须写成 Java 代码放在 AccessDecisionVoter 或自定义注解里——权限规则和业务代码耦合。

SpEL 让权限规则可配置化：一行表达式替换一个类。表达式的字符串可以从数据库动态加载，实现运行时修改权限规则。

💡 设计巧思：EvaluationContext — 表达式与环境的桥梁

SpEL 的最精妙设计不是语法解析，而是 EvaluationContext：

// EvaluationContext 就是"变量空间"——你把什么放进 Context，表达式就能访问什么
StandardEvaluationContext ctx = new StandardEvaluationContext();
ctx.setVariable("user", user);         // #user 指向这个对象
ctx.setRootObject(order);              // #root 指向订单
ctx.setBeanResolver(beanFactory);      // @beanName 可以引用 Spring Bean

// 表达式字符串 + Context = 完整的运行时计算环境
Expression exp = parser.parseExpression("#user.name + ' bought ' + #root.product");
String result = exp.getValue(ctx, String.class);

"啊哈时刻"：SpEL 解析器不关心变量从哪来——它只管解析表达式。你可以在不同的 Context 中使用同一个表达式，得到不同的结果。Spring Security 往 Context 里塞了 authentication、#id 等；Spring Cache 往 Context 里塞了方法参数。同一套解析器，不同场景。

⚠️ 边界与陷阱

陷阱1：性能

SpEL 每次调用都重新解析表达式（除非缓存）。循环中反复执行同一表达式开销不小。

陷阱2：安全问题

如果允许用户输入的字符串直接当 SpEL 执行 → 注入攻击。T(java.lang.Runtime).exec("rm -rf /") 可以通过 SpEL 执行任意代码。

陷阱3：可读性

SpEL 表达式的调试极其困难——字符串被扔进解析器，没有编译期检查，拼写错误只有运行时才知道。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

${} 和 #{} 的区别？什么时候用哪个？

${} 从 Environment 取值，由 BFPP 在实例化前替换。#{} 是 SpEL 运行时求值，可以做计算、调方法。简单配置注入用 ${}，需要动态计算或有条件逻辑用 #{}。比如 @Value("${server.port}") 用 ${}，@PreAuthorize("#id > 0") 用 #{}。

A
Q

@PreAuthorize 里的 #id 是从哪来的？

SecurityExpressionRoot 的 EvaluationContext 会把方法参数按参数名注入为变量。如果参数是 User user，表达式用 #user；如果是 Long id，表达式用 #id。参数名必须通过编译时 -parameters 标志保留，或者用 @P("id") 显式指定。

A
Q

SpEL 能调用任意 Java 方法吗？有什么限制？

可以，T(全限定类名).静态方法()。但不能调用非静态方法（除非对象已经在 Context 中），不能调用 private 方法。Spring Security 的 EvaluationContext 默认还禁止了部分危险操作——比如不能调 System.exit()。

A
Q

@Cacheable 的 key 里 #user.id 如果 user 为 null 会怎样？

抛 NullPointerException。SpEL 的 ?. 安全导航操作符可以解决：key = "#user?.id"（user 为 null 时不求值 id）。或者用三元表达式：key = "#user != null ? #user.id : 'null'。

A
Q

🔧 场景题：线上一个 Controller 根据请求来源动态切换调用逻辑，目前用 if-else 判断 header 值。主管说这样扩展性差，让你用 SpEL 重构。怎么做？

不要硬上 SpEL——这不是它的最佳场景。if-else 分支逻辑应该用策略模式 + Map 注入，SpEL 的优势是动态表达式（权限规则、缓存 key、条件注解），不是替代 if-else。如果非要动态规则，把分支判断规则存入数据库（如 source=app → handler=AppHandler），用 SpEL 做规则匹配——但依然不如策略模式直观。SpEL 适合"一行表达式描述一个布尔条件"，不适合"这是一个分支调度器"。

A
💡 考核意图：看你是否滥用技术。知道什么时候不用 SpEL 和知道怎么用同样重要。
Q

🔧 场景题：生产上@PreAuthorize 的表达式是硬编码的，运营想动态改。怎么把表达式存到数据库并运行时生效？

自定义 PermissionEvaluator 或 AccessDecisionVoter。实现一个 DatabaseDrivenPermissionEvaluator：从数据库加载 URL-表达式映射并缓存；在 evaluate() 中拿到当前认证信息、请求参数，构建 EvaluationContext，SpelExpressionParser 解析并执行表达式。缓存策略很重要——每次请求都解析表达式太重，需要预编译并缓存 Expression 对象。

A
Q

SpEL 性能怎么样？要不要担心？

单次执行微秒级，不是瓶颈。但两个场景要当心：① 高并发下频繁解析同一个表达式——用 CacheExpression 或预编译 Expression 对象缓存。② 表达式中调用了耗时的 Bean 方法——比如调了远程 RPC，每次表达式求值都阻塞。这种不要写在表达式里，提前算好放进 Context。

A

📋 面试官总结

1. ${} vs #{} 的本质区别 — 一个是配置替换（编译期），一个是运行时求值

2. EvaluationContext 的变量注入机制 — @PreAuthorize 的 #id 从哪来

3. SpEL 的安全边界 — T() 可以执行任意静态方法，用户输入不要直接拼进 SpEL

4. 适用/不适用场景 — 权限规则、缓存 key、条件注解是天然场景；分支调度不是

⚠️ 大厂高频陷阱：SpEL 不是 if-else 的替代品；动态 SpEL 必须预编译缓存 Expression 对象
