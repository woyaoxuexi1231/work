JDK 动态代理 vs CGLIB — 一张表

原理基于接口：生成实现相同接口的代理类基于继承：生成目标类的子类

// DefaultAopProxyFactory.createAopProxy() — 代理选择逻辑（伪代码）
public AopProxy createAopProxy(AdvisedSupport config) {
    if (config.isProxyTargetClass()       // 强制 CGLIB（@EnableAspectJAutoProxy(proxyTargetClass=true)）
        || !config.isOptimize() &&
           config.getProxiedInterfaces().length == 0) {  // 没有实现接口
        return new ObjenesisCglibAopProxy(config);  // → CGLIB
    }
    return new JdkDynamicAopProxy(config);           // → JDK 代理
}

Part 2 · 触及灵魂

🤔 如果 Spring 只用 JDK 动态代理，会出什么问题？

注入灾难：UserServiceImpl 实现了 UserService 接口。JDK 代理生成的是 $Proxy0 类型——它不是 UserServiceImpl 的实例。

// ❌ JDK 代理下这行代码会注入失败
@Autowired UserServiceImpl userService;  // 代理是 $Proxy0，不是 UserServiceImpl

// ✅ 只能按接口注入
@Autowired UserService userService;  // 代理实现了 UserService 接口

在大型项目中，很多人习惯直接注入实现类。JDK 代理让这种写法全部报错。这就是 Spring Boot 2.x 强制 CGLIB 的根本原因——兼容开发者的编写习惯。

💡 设计巧思：Objenesis — 绕过构造器的魔法

CGLIB 通过继承创建代理——这意味着父类构造器会被调用。如果构造器里有复杂逻辑（初始化连接池、加载配置），代理创建时也会触发。Objenesis 是一个小工具库，可以创建对象但不调用构造器。Spring 用 ObjenesisCglibAopProxy 来避免代理创建时执行构造器中的副作用。

🔬 ProxyFactoryBean — 手动创建代理

Spring AOP 的自动代理（AbstractAutoProxyCreator）内部就是用的 ProxyFactoryBean 的逻辑。你可以手动使用它：

ProxyFactoryBean factory = new ProxyFactoryBean();
factory.setTarget(new UserServiceImpl());
factory.addAdvice(new LoggingAdvice());
factory.setProxyTargetClass(true);  // 强制 CGLIB
UserService proxy = (UserService) factory.getObject();
⚠️ CGLIB 的限制：final 的诅咒

final 类

完全不能代理。CGLIB 需要继承，final 禁止继承。

final 方法

增强"静默失效"——不报错，但 AOP 不生效。这是生产中最危险的陷阱之一。

private 方法

子类不可见，不能代理。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

JDK 动态代理和 CGLIB 的根本区别？

JDK 代理基于接口实现，生成的是 $Proxy 类，只能代理接口方法。CGLIB 基于继承，生成的是子类，能代理所有非 final 方法。核心差异在于代理对象的类型：JDK 代理不是原类的实例，CGLIB 是。

A
Q

Spring Boot 2.x 为什么强制 CGLIB？

为了避免 JDK 代理的注入陷阱。很多开发者习惯 @Autowired UserServiceImpl 而不是 @Autowired UserService——JDK 代理下 $Proxy 不是 UserServiceImpl 的子类，注入失败。CGLIB 代理是子类，@Autowired UserServiceImpl 也能注入。统一行为、减少配置、降低上手门槛。

A
Q

CGLIB 代理的对象，getClass() 返回什么？

返回 UserService$$EnhancerBySpringCGLIB$$xxxxx（CGLIB 子类的类名）。不是 UserService 本身。这在某些场景下会出问题——比如通过 getClass().getAnnotation(Xxx.class) 获取类上的注解时，CGLIB 子类没有继承注解（取决于注解的 @Inherited 元注解）。

A
Q

CGLIB 代理的构造器会被调用两次，这是为什么？Objensis 怎么解决的？

CGLIB 先创建目标对象实例（第一次调构造器），再创建子类代理对象（第二次调构造器——父类构造器）。如果构造器里有副作用（建立网络连接），就会执行两次。Objenesis 通过 Unsafe 或 MethodHandle 直接分配内存创建对象，绕过构造器调用——代理创建时不会触发业务构造器逻辑。

A
Q

🔧 场景题：你发现一个 Service 上的 @Transactional 没生效，排查发现方法被声明为 final。为什么没生效？怎么发现其他类似问题？

CGLIB 无法重写 final 方法 → 无法创建代理 → @Transactional 静默失效。排查方法：开启 debug 日志看 AOP 代理是否创建（logging.level.org.springframework.aop=DEBUG）。预防：在代码规范中禁止 Service 方法使用 final，配合 Sonar 规则扫描。IDEA 插件也能检测到。

A
Q

🔧 场景题：线上有一个类被 final 修饰，但又需要 AOP。不改 final 的情况下怎么解决？

如果类实现了接口 → 改回 JDK 动态代理（proxy-target-class=false），JDK 代理不需要继承。如果没有接口 → 拆出一个接口，让 final 类实现它。如果都不能改 → 手工包装（写一个非 final 的 Wrapper 类，继承不了但可以组合）。最后的选择：用 AspectJ 的编译期织入（LTW），不依赖代理。

A
Q

DefaultAopProxyFactory 的代理选择逻辑是什么？

① 如果 proxyTargetClass=true（Spring Boot 2.x 默认 true）→ 返回 ObjenesisCglibAopProxy。② 如果目标类没有实现接口 → 返回 CGLIB。③ 否则 → 返回 JdkDynamicAopProxy。简单说：能 CGLIB 就 CGLIB。

📋 面试官总结

1. JDK vs CGLIB 的本质 — 接口实现 vs 继承，代理对象类型不同

2. Spring Boot 2.x 的决策 — 牺牲 final 兼容性换注入灵活性和零配置

3. final 的静默失效 — 不报错不警告，最难排查的 AOP 问题

4. Objenesis — 绕过构造器创建对象，避免副作用执行两次

⚠️ 高频陷阱：final 方法 AOP 失效无任何提示；CGLIB 代理 getClass() 不是原类名
