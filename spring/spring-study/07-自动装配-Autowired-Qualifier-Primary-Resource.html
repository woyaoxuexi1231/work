// 多 Bean 冲突时的三种解决方式
@Service
public class OrderService {
    // 方式1：@Qualifier 精确指定
    @Autowired @Qualifier("wechatPay") private PaymentService pay;

    // 方式2：@Primary 默认首选
    @Autowired private PaymentService pay;  // 注入标注了 @Primary 的那个

    // 方式3：@Resource name 查找
    @Resource(name = "alipay") private PaymentService pay;
}

// @Qualifier 优先级 > @Primary

Part 2 · 触及灵魂的设计原理

🤔 如果只有 @Autowired byType，没有 @Qualifier 和 @Primary？

痛点：有多个同类型 Bean 时，@Autowired 直接抛 NoUniqueBeanDefinitionException。开发者被迫走歪路：要么不按接口注入（写死实现类），要么把不需要的实现类从容器中排除（破坏可扩展性）。

@Qualifier 和 @Primary 提供了两个不同维度的解决方案：@Qualifier 是"调用方的主动选择"，@Primary 是"定义方的自我推荐"。两者互补，覆盖了所有多实现场景。

💡 设计巧思：@Resource 的 byName→byType 降级链

@Resource 不是单纯的 byName。它的查找逻辑：

// CommonAnnotationBeanPostProcessor 内部逻辑（伪代码）
String beanName = resource.name();  // 有 name 属性就用它
if (beanName.isEmpty()) beanName = field.getName();  // 否则用字段名

// 先按名字找
if (beanFactory.containsBean(beanName)) {
    return beanFactory.getBean(beanName);  // byName 成功 ✅
}
// 名字找不到 → 降级为 byType
return beanFactory.getBean(field.getType());  // byType

"啊哈时刻"：这个降级链让 @Resource 比 @Autowired 更"宽容"——大多数情况下，字段名刚好等于 Bean 名，byName 就命中了；没命中也不会报错，byType 兜底。

🔬 装配决策链：AutowireCandidateResolver

Spring 内部用 ContextAnnotationAutowireCandidateResolver（一个责任链）来决定"该注入哪个 Bean"：

// 装配解析过程（简化）
1. 按类型查找候选 Bean 列表
2. 如果有 @Qualifier → 按名称过滤候选 → 匹配失败抛异常
3. 如果候选只剩 1 个 → 返回
4. 如果候选有多个：
   a. 看有没有 @Primary → 有就返回它
   b. 尝试字段名匹配（byName fallback）
   c. 都不行 → NoUniqueBeanDefinitionException
⚠️ 边界与陷阱

陷阱1：@Resource 不能用在构造器上

JDK 的 @Resource 定位是"字段和 Setter"——Target 不包含构造器。

陷阱2：@Qualifier + @Primary 同时 → @Qualifier 胜出

显式 > 隐式。面试高频。

陷阱3：@Autowired(required=false) 的 null 地狱

如果 Bean 可能不存在，用 Optional 或 ObjectProvider 更安全。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@Autowired 和 @Resource 的区别？

@Autowired 是 Spring 原生，默认 byType，支持 required/集合/构造器。@Resource 是 JSR-250，先 byName 再 byType 降级链，不支持构造器和集合。区分要点不是"byType vs byName"，而是 @Resource 有降级链——这恰恰是很多人答错的地方。

A
Q

@Qualifier 和 @Primary 同时存在谁赢？

@Qualifier 胜出。装配链中先检查 @Qualifier，匹配就直接返回，不会走到 @Primary 的逻辑。显式指定优先于默认推荐。

A
Q

Map<String, PaymentService> 注入所有实现类，怎么和 @Primary 协作？

Map 注入不受 @Primary 影响——它始终注入所有实现。@Primary 只在单值注入时生效（@Autowired PaymentService 选首选）。如果在 Map 的前提下想默认用某个实现，需要业务层自己逻辑选择（如读取配置或注解标记），不能靠 @Primary。

A
Q

@Resource 不指定 name，怎么决定按什么名字找？

用字段名。private UserService userService → 找名为 "userService" 的 Bean。找不到 → 降级为 byType。

A
Q

🔧 场景题：多数据源中，你有 3 个 DataSource Bean（master/slave1/slave2）。JdbcTemplate 默认应该注入哪个？怎么设计？

master 标注 @Primary → 默认 JdbcTemplate 注入 master。读写分离的业务代码中，读操作显式注入 slave DataSource（@Qualifier("slave1")）。这样既不破坏默认行为（大部分代码直接用 @Autowired），又给了精确控制的能力。更优雅的做法是自定义 AbstractRoutingDataSource + 切面动态路由——但 @Primary + @Qualifier 是最小成本的方案。

A
Q

🔧 场景题：智能拓客多渠道改造中，5 个渠道适配器都用 @Service 注册。如何在某个 Controller 中动态选择渠道（而不是写死 @Qualifier("smsChannel")）？

用 Map<String, ChannelAdapter> 注入所有适配器，key 就是 Bean 名。运行时根据请求参数（如 channel=sms）从 Map 中取对应适配器。如果需要默认渠道，在其中一个适配器上加 @Primary，或者用 @ConditionalOnProperty 控制启用哪个。这样新增渠道只需加一个 @Service，不用改任何 Controller 代码——开闭原则。

A
Q

@Autowired 底层由哪个处理器负责？它的 order 是多少？

AutowiredAnnotationBeanPostProcessor。它实现 PriorityOrdered，order 值很低（优先执行）。在属性填充阶段（populateBean）中被调用，扫描 @Autowired 和 @Value 注解进行注入。

A

📋 面试官总结

1. @Resource 不是纯粹的 byName — 降级链是区分点，90% 的人答错

2. @Qualifier > @Primary — 显式指定永远优先于默认推荐

3. Map 注入是实现策略模式的最简方式 — 但 Map 不受 @Primary 影响

4. 场景设计题最爱考 — 多数据源、多渠道、多支付方式，都是"如何优雅选择多实现"的问题

⚠️ 大厂高频陷阱：@Resource 不支持构造器注入；@Autowired(required=false) 不如 ObjectProvider
