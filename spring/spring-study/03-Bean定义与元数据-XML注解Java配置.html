Part 1 · 基础认知

三种方式对比

方式年代Bean 声明依赖注入适用
XML2004<bean id="us" class="..."/><constructor-arg ref="repo"/>第三方类、遗留系统
注解2007@Component + 扫描@Autowired自己写的业务类
Java Config2009@Configuration + @Bean方法参数 / 方法调用第三方类、复杂初始化

XML 配置

<bean id="userRepo"
  class="com.ex.UserRepository"/>

<bean id="userService"
  class="com.ex.UserService">
  <constructor-arg ref="userRepo"/>
</bean>

✅ 类不用改
❌ 极其冗长

注解

@Repository
public class UserRepository { }

@Service
public class UserService {
  @Autowired
  private UserRepository repo;
}

✅ 代码即配置
❌ 类必须能改

Java Config

@Configuration
public class AppConfig {
  @Bean
  public UserRepo repo() {
    return new UserRepo();
  }
  @Bean
  public UserService svc() {
    return new UserService(repo());
  }
}

✅ 类型安全+灵活
❌ 需要 @Configuration

Part 2 · 触及灵魂的设计原理

🤔 如果只有 XML 配置，会发生什么？

痛点：一个中型项目可能有 200+ 个 Bean。每个 Bean 在 XML 中至少 3 行，加上依赖声明 → 一个 applicationContext.xml 轻松上千行。添加一个新 Service 需要同时改 Java 类和 XML，大脑要在两种语言之间来回切换。重命名一个类，XML 里不会自动更新 → 运行时 ClassNotFoundException。

Rod Johnson 在 Spring 2.5（2007）引入 @Component 和 @Autowired 时说过一句经典的话："代码本身就是最好的配置文档"。注解让"配置"回到了代码中——你不需要再维护一份平行的 XML 来描述代码结构。

💡 设计巧思：BeanDefinition — 统一元数据模型

这是 Spring 最精妙的设计之一：无论你用 XML、注解还是 Java Config，最终都会转化成同一种数据结构——BeanDefinition。容器只认 BeanDefinition，不关心它的来源。

// 三者殊途同归：
XML     → XmlBeanDefinitionReader 解析 → BeanDefinition
@Component + 扫描 → ClassPathBeanDefinitionScanner → BeanDefinition
@Bean   → ConfigurationClassPostProcessor 解析 → BeanDefinition
                            ↓
              DefaultListableBeanFactory.beanDefinitionMap
              (ConcurrentHashMap<String, BeanDefinition>)
                            ↓
                  容器统一处理：实例化、注入、初始化

这意味着你可以在一个项目里混用三种方式：自己写的 Service 用 @Component，第三方 DataSource 用 @Bean，遗留的 Struts Action 用 XML。它们和平共处，因为最终都是 BeanDefinition。

🔬 BeanDefinition 里到底存了什么？

BeanDefinition {
    String beanClassName;      // 全限定类名
    String scope;              // singleton / prototype
    boolean lazyInit;          // 是否懒加载
    String[] dependsOn;        // 依赖的 Bean 名
    ConstructorArgumentValues; // 构造器参数
    MutablePropertyValues;     // 属性值
    String initMethodName;     // init-method
    String destroyMethodName;  // destroy-method
    boolean primary;           // 是否首选
    // ...
}

这就是 Bean 的"蓝图"。容器在 getBean() 时读取这份蓝图，按图施工。

⚠️ 边界：XML 和注解定义同名 Bean，谁赢？

取决于加载顺序。XML 通常后加载（ApplicationContext 构造时传入 xml 路径），后加载的会覆盖先加载的同名 BeanDefinition。如果 XML 中 bean id="userService" 和 @Service("userService") 同时存在 → XML 的覆盖注解的。

Spring Boot 中，自动配置的 Bean 用 @ConditionalOnMissingBean 主动退让给用户定义的 Bean——这是一种"自觉"的优先级机制，避免暴力覆盖。

Part 3 · 大厂面试官的灵魂追问

Q

XML、注解、Java Config，现在项目里你推荐用哪种？

混合使用。自己写的业务类用 @Component + @Autowired（最少代码）；第三方类或需要复杂初始化的用 @Bean；XML 基本不用，除非维护遗留系统。没有一种方式能覆盖所有场景。

A
💬 考核意图：看你有没有"非此即彼"的二极管思维。真正做过项目的人知道混用是常态。
Q

@Bean 方法上能加 @Scope 吗？initMethod 和 @PostConstruct 哪个更好？

能，@Scope 可以标在 @Bean 上。initMethod 和 @PostConstruct 不冲突——但如果 @Bean 返回的类本身有 @PostConstruct，两者都执行（@PostConstruct 先）。initMethod 的优势是：你可以对一个第三方类（不能改源码）指定它的初始化方法——@PostConstruct 做不到这一点，因为你要在类里加注解。

A
Q

如果 XML 中定义了一个 Bean，注解扫描也扫到了同名的，到底哪个生效？

后加载的覆盖先加载的。如果 ApplicationContext 的构造是先注册注解扫描再加载 XML → XML 覆盖注解。但这是"无声覆盖"——不会报错、不会警告，排查起来非常痛苦。所以 Spring Boot 默认不用 XML，从根本上规避了这个问题。

A
💬 深挖意图：考察你是否理解 Spring Boot "去 XML 化"不只是赶时髦，而是解决了一个真实的"配置来源冲突"问题。
Q

你的多数据源组件如果用 @Bean 注册了多个 DataSource，怎么区分谁是主谁是备？

用 @Bean("masterDataSource") 和 @Bean("slaveDataSource") 命名区分，配合 @Qualifier 或 Map 注入。切换策略单独抽象（RoutingStrategy），不写在 @Bean 定义中。Java Config 的类型安全让这些设计决策在编译期就可验证——如果用 XML，一个拼写错误要到运行期才知道。

A
