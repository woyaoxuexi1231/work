Part 2 · 触及灵魂的设计原理

🤔 假如没有 Resource 抽象，加载配置有多痛苦？

痛苦 1：开发环境读 classpath 下的文件，生产环境读外部目录——两套代码。

// 没有 Resource 前——代码里写死文件路径和加载方式
InputStream is;
if (env.equals("dev")) {
    is = getClass().getClassLoader().getResourceAsStream("config.yml");
} else {
    is = new FileInputStream("/etc/app/config.yml");
}

痛苦 2：每个资源来源（本地文件、Jar 内、HTTP、FTP、S3）都需要不同的 API。开发者在写"读一个文件"这么简单的事情时，要关注文件到底在哪。

💡 设计巧思：前缀协议 + 策略模式

ResourceLoader 用一行字符串决定策略：

// ResourceLoader.getResource() 的核心逻辑（伪代码）
public Resource getResource(String location) {
    if (location.startsWith("classpath:")) {
        return new ClassPathResource(location.substring(10));
    }
    if (location.startsWith("file:")) {
        return new FileSystemResource(location.substring(5));
    }
    if (location.startsWith("https://") || location.startsWith("http://")) {
        return new UrlResource(location);
    }
    // 没有前缀 → 根据 ApplicationContext 类型推断默认
    return getDefaultResource(location);
}

"啊哈时刻"：Spring 没有发明一个复杂的"资源注册表"或 SPI 机制。它只用了一个最古老的设计模式——前缀匹配 + 策略模式。classpath: 开头 → ClassPathResource；file: 开头 → FileSystemResource。简单得令人发指，但极其有效。

后来 Spring Boot 在 jar 包内读取文件（可执行 jar 的特殊结构）时，又加了一个自定义协议 org.springframework.boot.loader.jar.Handler——同样是前缀匹配的思路。

🔬 ResourceLoader 在容器中的默认实现

ApplicationContext 本身就实现了 ResourceLoader。这意味着：

// ApplicationContext 的继承链
ApplicationContext
  ↳ ResourcePatternResolver  (支持 classpath*: 通配符)
    ↳ ResourceLoader         (基础 getResource)

// 所以任何 ApplicationContext 都可以直接当 ResourceLoader 用
@Autowired ApplicationContext ctx;
Resource r = ctx.getResource("classpath:db/schema.sql");
Resource[] rs = ctx.getResources("classpath*:META-INF/spring.factories");
⚠️ 边界与陷阱

可执行 JAR 中的文件读取

Spring Boot fat jar 有嵌套结构——classpath: 在 jar 内，不能用标准 File API。Spring Boot 自定义了 JarURLConnection 来处理。

getResource vs getResources

getResource 返回第一个匹配；getResources 返回所有匹配（适合 classpath*: 扫描）。

Resource 不保证物理文件存在

Resource.getFile() 可能失败——如果资源在 jar 里。应该用 getInputStream()。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

classpath:、classpath*:、file: 有什么区别？

classpath: 从类路径加载，优先第一个匹配的。classpath*: 扫描所有 jar，返回所有匹配的资源（用于 spring.factories 这类需要合并不同名文件的内容）。file: 从操作系统文件路径加载。不带前缀时由 ApplicationContext 类型决定默认。

A
Q

Resource.getFile() 和 getInputStream() 什么时候用哪个？

能用 getInputStream() 就别用 getFile()。getFile() 要求资源必须在文件系统中——如果资源在 jar 内（比如可执行 jar 的 classpath），getFile() 直接抛 FileNotFoundException。getInputStream() 在任何情况下都能工作。

A
Q

Spring 能读 S3 上的文件吗？怎么做到？

Spring 只内置了 classpath:/file:/http: 协议。但 ResourceLoader 可以扩展——你可以实现自己的 Resource 和 ProtocolResolver，注册后 s3://bucket/file.txt 就能自动识别。Spring Cloud AWS 就是这么做的。

A
Q

🔧 场景题：你们的应用需要同时支持本地配置文件和远程配置中心。本地开发用 classpath:，生产环境优先从配置中心 HTTP 拉取。如何用 ResourceLoader 统一？

自定义一个 CompositeResourceLoader：先尝试从配置中心加载（http://config-server/xxx），失败则降级到 classpath:。还可以加缓存——第一次从配置中心拉取后缓存到本地文件，后续直接用 file: 路径读取。核心思路是利用 ResourceLoader 的统一接口，业务代码只写 getResource("config.yml")，不用关心来源。

A
Q

🔧 场景题：在 Docker 容器中运行 Spring Boot 应用，配置文件想通过环境变量注入而不是挂载文件。Resource 抽象能帮忙吗？

Resource 抽象解决的是"文件在哪"的问题——环境变量不是文件，不应该用 Resource。Docker 场景下应该用 Spring 的 Environment 抽象（@Value / @ConfigurationProperties），环境变量天然在 PropertySource 链中优先级很高。但如果非要通过文件——可以在启动脚本里用环境变量生成一个临时配置文件，然后用 file:/tmp/config.yml 加载，但这是歪路。

A
💡 考核意图：分清楚 Resource 和 Environment 的边界。Resource 管文件加载，Environment 管配置注入。不要用错工具。
Q

classpath*: 扫描所有 jar 的实现原理是什么？面试官想听到"合并"这个词。

classpath*: 不是简单的"匹配多个文件"——它做的是 合并。比如 spring.factories 在每个 jar 里都有一个同名文件，classpath*:META-INF/spring.factories 会找到所有 jar 中的这个文件，然后把它们的内容拼接在一起。SpringFactoriesLoader 依赖这个机制，这也是自动配置能加载 140+ 个配置类的基础。

A
Q

Resource 接口和 Java 的 java.io.File 有什么区别？

java.io.File 只能表示文件系统中的文件——不能表示 classpath 资源、jar 内的文件、HTTP 资源。Resource 是一个更上层的抽象，底层可以是 File、也可以是输入流、URL、byte 数组。而且 Resource 自带 exists()、isReadable()、lastModified() 等元数据方法，File 需要自己处理。

A

📋 面试官总结

1. Resource 接口的设计模式 — 前缀协议 + 策略模式，简单但强大

2. classpath: vs classpath*: — 前者找第一个，后者合并所有

3. getInputStream 优先于 getFile — jar 内文件没有 File 对象

4. Resource vs Environment 的分工 — 管文件加载 vs 管配置注入，不要混淆

⚠️ 大厂高频陷阱：可执行 jar 中 classpath 资源的特殊性；classpath*: 不是 glob 匹配而是合并
