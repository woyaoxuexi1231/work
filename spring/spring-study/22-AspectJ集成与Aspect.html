Part 1 · Spring AOP 与 AspectJ 的关系

维度Spring AOPAspectJ

是否需要特殊编译器❌✅ ajc
能拦截什么public 方法（代理限制）方法/字段/构造器/静态方法

核心事实：Spring AOP 借用了 AspectJ 的注解语法（@Aspect/@Before/@Around 等），但底层用的是自己的代理机制，没有用 AspectJ 的编译器。这是一种"偷师"——用最成熟的 AOP 语法，但保留简单的运行时实现。

Part 2 · 触及灵魂

🤔 为什么 Spring 不直接用 AspectJ？

因为 AspectJ 需要ajc 编译器或LTW（Load-Time Weaving）。ajc 意味着要替换 javac，这在企业级 Maven/Gradle 构建中极难推广。LTW 需要 javaagent 启动参数——运维不愿意加。Spring 的选择是："牺牲部分能力（不能拦截 private/final/自调用），换零配置和零运维成本"。

💡 设计巧思：一套语法两个引擎

你写的 @Aspect 和 @Before 注解，在 Spring 环境中被 AspectJAutoProxyCreator 解析，在 AspectJ 环境中被 ajc 解析。语法是兼容的——但能力不同。这让你可以在需要时从 Spring AOP 升级到 AspectJ，不改切面代码，只改构建配置。这也是为什么 Spring 选择了"兼容"而不是"另起炉灶"。

⚠️ LTW（Load-Time Weaving）是什么？

LTW 是 AspectJ 的一种织入方式——在类加载时修改字节码。Spring 支持 LTW 作为 Spring AOP 的增强模式：启动时加 -javaagent:aspectjweaver.jar。代价是启动变慢（每个类加载都要检查是否需要织入），收益是可以拦截 private/final/自调用。生产环境很少用。

Part 3 · 面试追问（6 组）

Q

@Aspect 是 AspectJ 的还是 Spring 的？

@Aspect 注解本身来自 AspectJ 库（org.aspectj.lang.annotation）。Spring 借用了这个注解和整套语法，但解析和执行引擎是 Spring 自己的（AspectJAutoProxyCreator）。

A
Q

Spring AOP 什么时候该升级到 AspectJ？

三个信号：① 需要对 private 方法做 AOP ② 自调用场景太多拆类不现实 ③ 需要对非 Spring Bean（如领域对象）做增强。但 95% 的项目不需要——Spring AOP 的方法级代理已经够用。

A
Q

🔧 场景题：项目里有一个 final 类需要 AOP。Spring AOP 不支持，改成 AspectJ LTW 的步骤？

① 加依赖 aspectjweaver。② 启动参数加 -javaagent:aspectjweaver.jar。③ 在 META-INF/aop.xml 中声明切面和目标。④ 切面代码不用改。代价：启动时间增加 20%-50%，每个类加载时都要检查是否需要织入。

A
Q

🔧 场景题：自调用 AOP 不生效，除了拆类还有什么解法？

① 注入自己：@Autowired private XxxService self; 通过 self.method() 调用 ② AopContext.currentProxy()（需 exposeProxy=true）③ 升级到 AspectJ LTW。推荐拆类——不依赖 Spring 内部 API。

A
Q

AspectJ 的 execution 和 Spring AOP 的 execution 有区别吗？

语法兼容，但匹配范围不同。AspectJ 的 execution 可以匹配 private 方法、静态方法、构造器——Spring AOP 只能在 public 非 final 方法上生效。同一个表达式在两个环境中可能匹配到不同数量的方法。

A
Q

Spring 能不用 AspectJ 注解做 AOP 吗？

能，但很少用。早期的 Spring AOP 用 XML 配置切面（<aop:config>），或者编程式 ProxyFactory。现在都用 @Aspect 注解——它是事实标准。

A

📋 面试官总结

1. Spring AOP 借用了 AspectJ 注解 — 语法相同，引擎不同

2. 代理 vs 织入 — 运行时代理（简单）vs 编译期织入（能力全）

3. LTW — 升级路径存在但极少用，代价是启动性能

4. 同一套切面代码可在两种引擎间迁移 — 这是设计的远见
