Part 1 · 基础认知

三个核心工厂 Bean

FactoryBean产生什么适用场景
LocalSessionFactoryBeanHibernate SessionFactory纯 Hibernate（不经过 JPA）
LocalContainerEntityManagerFactoryBeanJPA EntityManagerFactory⭐ Spring Boot 默认使用
HibernateJpaVendorAdapter（适配器，不是 FactoryBean）把 Hibernate 适配到 JPA 接口
// Spring Boot 自动配置的内部逻辑（简化）
@Bean @ConditionalOnMissingBean
public LocalContainerEntityManagerFactoryBean entityManagerFactory(
        DataSource dataSource, JpaVendorAdapter jpaVendorAdapter) {
    LocalContainerEntityManagerFactoryBean emfb = new LocalContainerEntityManagerFactoryBean();
    emfb.setDataSource(dataSource);
    emfb.setJpaVendorAdapter(jpaVendorAdapter);  // HibernateJpaVendorAdapter
    emfb.setPackagesToScan("com.example.entity");  // 扫描 @Entity
    return emfb;
}

// HibernateJpaVendorAdapter 做了什么：
// 把 Spring 的配置翻译为 Hibernate 的 JPA 实现属性
// 比如 spring.jpa.hibernate.ddl-auto → hibernate.hbm2ddl.auto

Part 2 · 触及灵魂

🤔 如果没有 Spring 的 ORM 集成，怎么在 Spring 中使用 Hibernate？

你需要手写 Hibernate 的 Bootstrap 代码：

// 原始 Hibernate 初始化
StandardServiceRegistry registry = new StandardServiceRegistryBuilder()
    .applySettings(hibernateProperties).build();
SessionFactory sf = new MetadataSources(registry)
    .addAnnotatedClass(User.class).buildMetadata()
    .buildSessionFactory();

问题：① SessionFactory 不是 Spring Bean——不能注入、不能参与事务管理；② 事务管理需要手动 try-finally；③ 异常翻译需要自己写。Spring 用 FactoryBean 模式把 Hibernate 的核心对象包装成 Bean。

💡 设计巧思：SharedEntityManagerCreator — 每个 @Autowired 的 EntityManager 都是代理

注入到 DAO 中的 EntityManager 不是真正的 EntityManager——它是一个代理（SharedEntityManagerCreator）。每次方法调用时，代理从 TransactionSynchronizationManager 中获取当前线程绑定的真实 EntityManager。这个设计保证了：

  • 同一事务中的所有 DAO 共享同一个 EntityManager（一级缓存一致）
  • 线程安全（每个线程有自己的 EntityManager）
⚠️ Open Session In View (OSIV) 机制

Spring Boot 默认开启 OSIV：一个请求开始时打开 Hibernate Session，请求结束时关闭。这让 Controller 层也能触发懒加载——但会长时间占用数据库连接。在高并发下建议关闭（spring.jpa.open-in-view=false），在 Service 层主动 fetch 所需数据。

Part 3 · 大厂面试官的灵魂追问（6 组）

Q

LocalSessionFactoryBean 和 LocalContainerEntityManagerFactoryBean 的区别？

前者产生 Hibernate 的 SessionFactory（纯 Hibernate API），后者产生 JPA 的 EntityManagerFactory（标准 JPA 接口 + Hibernate 作为实现）。Spring Boot 默认用后者——它是 JPA 标准，换 ORM 实现（如 EclipseLink）不需要改代码。

A
Q

SharedEntityManagerCreator 为什么用代理而不是直接给真实 EntityManager？

因为 singleton DAO 在容器启动时就创建了，但 EntityManager 是每个事务一个（由 TransactionSynchronizationManager 管理）。singleton DAO 不能持有某个特定事务的 EntityManager——代理在每次方法调用时动态获取当前线程的 EntityManager。这和 @Lazy 注入点的代理思路一致。

A
Q

OSIV 为什么建议关闭？关了有什么风险？

OSIV 让 Session 跨了整个请求生命周期——包括视图渲染阶段（Jackson 序列化 JSON）。这期间数据库连接一直被占用。高并发下连接池耗尽。关闭后，懒加载不能在 Controller 层使用——需要在 Service 层用 JOIN FETCH 或 @EntityGraph 主动加载。风险：开发忘记 fetch → LazyInitializationException。

A
Q

🔧 场景题：线上出现 LazyInitializationException: could not initialize proxy - no Session。什么原因？怎么排查？

原因：在 Session 关闭后（@Transactional 结束或 OSIV 关闭后）访问了懒加载关联。排查：① 看异常栈，找到触发懒加载的代码行 → 确认是否在 @Transactional 方法外部。② 如果关了 OSIV → 检查是否在 Controller 层访问未 fetch 的关联。解决：在 Service 层用 JOIN FETCH 或 DTO 投影一次性查出所有需要的数据。

A
Q

🔧 场景题：一个查询关联了 5 张表，用 JPA 的 @OneToMany + lazy loading 产生了 N+1 查询共 500 条 SQL。怎么优化？

三种方案按场景选择：① JOIN FETCH — 一条 SQL 全查出来，但一对多容易笛卡尔积。② @BatchSize — 把 N 条 SQL 合并为 N/100 条 IN 查询，最推荐的"懒人"方案。③ DTO 投影 — 不查 Entity，直接 new DTO 只选需要的列。④ 如果实在复杂 → 别用 JPA，用 MyBatis/JdbcTemplate 手写 SQL。

A
Q

Spring Data JPA 的 JpaRepository 是怎么被 Spring 变成实现类的？

不是编译期生成，也不是 CGLIB/ASM 代理——是 JDK 动态代理。JpaRepositoryFactory 为每个 Repository 接口创建一个 Proxy，所有方法调用都被 JpaRepositoryFactory 拦截，解析方法名（findByNameAndAge）→ 生成 JPQL → 通过 EntityManager 执行。它是通过 @EnableJpaRepositories 和 ImportBeanDefinitionRegistrar 在容器启动时动态注册的。

A

📋 面试官总结

1. FactoryBean 模式 — 所有 ORM 核心对象（SessionFactory/EntityManagerFactory）都是 FactoryBean 产物

2. SharedEntityManagerCreator 代理 — 每次调用动态获取当前线程的 EntityManager

3. OSIV — 方便但有连接池耗尽风险，生产建议关闭

4. N+1 问题 — JOIN FETCH / @BatchSize / DTO 投影 / 直接写 SQL

⚠️ 高频陷阱：LazyInitializationException 是 OSIV 关闭或事务范围不够的信号
