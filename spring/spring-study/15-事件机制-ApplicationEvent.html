Part 1 · 基础认知

三种写法对比

方式写法特点
实现接口implements ApplicationListener<OrderEvent>老派写法，笨重
注解@EventListener⭐ 推荐，灵活
事务监听@TransactionalEventListener事务提交后才触发
// 1. 定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Order order;
    public OrderCreatedEvent(Object source, Order order) { super(source); this.order = order; }
    public Order getOrder() { return order; }
}

// 2. 发布事件
@Service public class OrderService {
    @Autowired private ApplicationEventPublisher publisher;
    @Transactional
    public Order create(OrderDTO dto) {
        Order order = save(dto);
        publisher.publishEvent(new OrderCreatedEvent(this, order));  // 发布！
        return order;
    }
}

// 3. 监听事件
@Component public class OrderEventListener {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent e) { sendSms(e.getOrder()); }
    
    @EventListener @Async  // 异步监听
    public void updateStats(OrderCreatedEvent e) { statsService.incr(); }

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void afterCommit(OrderCreatedEvent e) { mqService.send(e.getOrder()); }
}

Part 2 · 触及灵魂

🤔 如果没有事件机制，Service 之间松耦合怎么做？

紧耦合代码：订单创建后要发短信、更新统计、发 MQ。传统做法是在 OrderService.create() 中依次调用这些依赖 → OrderService 依赖了 SMSService、StatsService、MQService：

// ❌ 紧耦合：新增一个"订单创建后"的动作要改 OrderService
public Order create(OrderDTO dto) {
    Order order = save(dto);
    smsService.send(order);      // 1
    statsService.update(order);  // 2
    mqService.send(order);       // 3
    // 加新的？又要改这里！
    return order;
}

用事件机制：OrderService 只发布事件，谁监听、怎么处理，OrderService 完全不知道。新增一个监听器不需要改 OrderService。

💡 设计巧思：@TransactionalEventListener — 事务感知的观察者

这是 Spring 事件机制最精妙的部分。普通的 @EventListener 在事件发布后立即执行——但此时事务可能还未提交。如果监听器需要读数据库（比如新创建的订单），可能读到旧数据（取决于隔离级别）。

// @TransactionalEventListener 的四种触发阶段：
AFTER_COMMIT      → 事务提交后（默认，最常用）
AFTER_ROLLBACK    → 事务回滚后
AFTER_COMPLETION  → 事务结束后（不管提交还是回滚）
BEFORE_COMMIT     → 事务提交前

内部实现：它通过 TransactionSynchronizationManager.registerSynchronization() 注册一个回调。事务提交时 → 回调触发 → 执行监听器方法。这和 ThreadLocal 事务机制完美配合。

🔬 事件的传播链路 
publishEvent(orderCreatedEvent)
  → SimpleApplicationEventMulticaster.multicastEvent()
    → for (ApplicationListener listener : getListeners(event)) {
          // 默认同步：直接调 listener.onApplicationEvent()
          // @Async → 线程池异步执行
      }
// 默认所有监听器同步串行执行 → 一个监听器慢了，事件发布者也慢
⚠️ 边界与陷阱

陷阱1：默认同步执行

@EventListener 没有 @Async 时，事件发布线程直接调监听器方法。一个慢监听器拖慢整个请求。需要异步时记得加 @Async。

陷阱2：事件在事务内触发

普通 @EventListener 在事务内执行 → 监听器的 DB 操作在同一事务中 → 监听器抛异常会导致业务回滚。用 @TransactionalEventListener(AFTER_COMMIT) 避免。

陷阱3：父类事件匹配

@EventListener 默认也监听子类事件。发布 UserEvent → 监听 ApplicationEvent 的也会触发。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

@EventListener 和 ApplicationListener 接口有什么区别？

@EventListener 是注解驱动，一个类里可以写多个 @EventListener 监听不同事件。ApplicationListener 是接口，一个类只能监听一种事件（泛型参数）。@EventListener 还支持 SpEL 条件过滤、@Async、@Order 排序。

A
Q

多个监听器监听同一个事件，执行顺序怎么控制？

@Order(1)、@Order(2) 控制。数字越小越先执行。但注意：如果监听器加了 @Async，Order 控制的是"提交到线程池的顺序"，不是"实际执行完的顺序"——异步执行不保证结束后顺序。

A
Q

@TransactionalEventListener 怎么做到"事务提交后才触发"？

通过 TransactionSynchronizationManager.registerSynchronization() 注册了一个回调。事务管理器在 commit() 后会遍历所有 Synchronization 回调——afterCommit、afterCompletion 等。回调里再用 ApplicationEventPublisher 重新发布事件给监听器。

A
Q

🔧 场景题：订单创建后要发短信、更新统计、发 MQ。发短信失败了不应该影响订单创建。你怎么设计？

用 @TransactionalEventListener + @Async。发短信的监听器加 @Async + AFTER_COMMIT。事务提交后才触发，且异步线程中的异常不会回滚主事务。发短信失败只记录日志+告警，不影响订单。如果发 MQ 需要保证可靠性——不要用事件，用本地消息表 + 定时任务补偿。

A
Q

🔧 场景题：线上发现一个订单创建请求非常慢（5秒），排查发现统计更新的监听器执行了 4 秒。怎么解决？

默认 @EventListener 是同步的——发布者线程直接执行监听器。统计更新慢 → 拖慢主流程。加 @Async 让统计监听器异步执行，主线程立即返回。还要检查统计监听器慢的根本原因（是不是大表没索引）。短期加 @Async 应急，长期优化统计查询。

A
Q

事件跟 MQ 有什么区别？什么时候用事件、什么时候用 MQ？

Spring 事件是进程内的，MQ 是跨进程的。事件快（内存调用）、MQ 可靠（持久化 + 重试）。同一个 JVM 内的解耦用事件（发短信、更新统计）；跨服务通信用 MQ（订单服务通知库存服务）。不能把事件当 MQ 用——服务器挂了事件全丢。

A
Q

监听器里抛了异常会怎样？

默认同步执行时，监听器异常会直接抛给发布者——后续的监听器不会执行，发布者也可能因此回滚。加了 @Async 的监听器异常在独立线程中，不影响发布者。所以关键监听器要自己 try-catch，不要指望框架兜底。

A

📋 面试官总结

1. 事件 vs MQ — 进程内解耦 vs 跨进程通信，可靠性完全不同

2. @TransactionalEventListener — 事务感知，解决"事务没提交监听器就读数据"的问题

3. 默认同步执行 — 一个慢监听器拖死整个请求

4. 异常传播 — 同步监听器异常影响发布者

⚠️ 高频陷阱：把事件当 MQ 用；@Async 后 Order 只控制提交顺序不保证执行完毕顺序
