Part 1 · 基础认知

DataAccessException 异常层次

DataAccessException (RuntimeException, 非受检)
├── DataAccessResourceFailureException  // 连接失败
├── DataIntegrityViolationException     // 唯一约束冲突
├── DeadlockLoserDataAccessException     // 死锁
├── IncorrectResultSizeDataAccessException  // 结果数不对
├── EmptyResultDataAccessException       // 期望1条结果但0条
├── OptimisticLockingFailureException    // 乐观锁冲突
└── ... (共 20+ 子类)

每个子类对应一种具体的数据库异常场景——不再需要 catch(Exception e) 然后看 errorCode 猜发生了什么。

JdbcTemplate vs 原始 JDBC

操作原始 JDBCJdbcTemplate
获取连接DriverManager.getConnection()自动从 DataSource 获取
创建 Statement手写自动创建 PreparedStatement
异常处理SQLException (受检) + 手写 try-finally自动翻译为 DataAccessException (非受检)
资源释放手动 close (6 行模板代码)自动释放
代码量~15 行/查询~3 行/查询
// JdbcTemplate — 基本查询
@Autowired private JdbcTemplate jdbcTemplate;

// 查单条
User user = jdbcTemplate.queryForObject(
    "SELECT * FROM users WHERE id = ?", new BeanPropertyRowMapper<>(User.class), 1L);

// 查列表
List<User> users = jdbcTemplate.query("SELECT * FROM users WHERE age > ?", rowMapper, 18);

// 更新
int rows = jdbcTemplate.update("UPDATE users SET name = ? WHERE id = ?", "张三", 1L);

// NamedParameterJdbcTemplate — 具名参数（更可读）
@Autowired private NamedParameterJdbcTemplate namedJdbc;
Map<String, Object> params = Map.of("id", 1L, "name", "张三");
namedJdbc.update("UPDATE users SET name = :name WHERE id = :id", params);

Part 2 · 触及灵魂

🤔 如果不用 Spring 的异常翻译，数据库异常有多难处理？

JDBC 只抛出 SQLException——不管是连接断了、死锁了、唯一约束冲突了，全都是同一个异常类型。你要靠 getErrorCode() 或 getSQLState() 来区分，而这些代码每个数据库都不一样（MySQL 的 1062 是唯一约束冲突，Oracle 是 1）。写一次代码只能跑在一个数据库上。

Spring 的 SQLErrorCodeSQLExceptionTranslator 维护了每种数据库的错误码映射表，自动把 SQLException 翻译为具体的 DataAccessException 子类。

💡 设计巧思：模板方法模式 — JdbcTemplate 的灵魂

JdbcTemplate 的核心是模板方法模式——它定义了不变的流程骨架，把"变化的部分"（SQL 和结果映射）留给回调：

// JdbcTemplate.query() 的核心骨架（伪代码）
public <T> T query(String sql, RowMapper<T> rm, Object... args) {
    Connection conn = null; PreparedStatement ps = null; ResultSet rs = null;
    try {
        conn = dataSource.getConnection();           // 1. 获取连接
        ps = conn.prepareStatement(sql);             // 2. 创建 Statement
        for (int i = 0; i < args.length; i++)        // 3. 设置参数
            ps.setObject(i + 1, args[i]);
        rs = ps.executeQuery();                      // 4. 执行
        return rm.mapRow(rs, 1);                     // 5. 映射结果 ← 只有这步是"你的代码"
    } catch (SQLException e) {
        throw translateException(e);                 // 6. 异常翻译
    } finally {
        JdbcUtils.closeResultSet(rs);                // 7. 释放资源
        JdbcUtils.closeStatement(ps);
        JdbcUtils.closeConnection(conn);
    }
}

"啊哈时刻"：15 行样板代码被压缩到框架内部。开发者写一个 lambda 就是一次查询。

⚠️ 边界与陷阱

陷阱1：queryForObject 期望恰好1条结果

0条→EmptyResultDataAccessException；>1条→IncorrectResultSizeDataAccessException。用 query() + Optional 更安全。

陷阱2：JdbcTemplate 不是连接池

它每次操作都从 DataSource 获取新连接。连接池是 HikariCP/DBCP 的事。

Part 3 · 大厂面试官的灵魂追问（7 组）

Q

为什么 Spring 要把 SQLException 翻译成 DataAccessException？

三个原因：① SQLException 是受检异常，强制 try-catch——Spring 认为数据库异常通常是不可恢复的，应该是非受检。② 一个 SQLException 涵盖所有错误——通过 errorCode 映射为 20+ 种具体异常子类。③ 跨数据库一致性——MySQL 的 1062 和 Oracle 的 1 → 都翻译为 DataIntegrityViolationException。

A
Q

JdbcTemplate 用了什么设计模式？

模板方法模式。JdbcTemplate 固定了"获取连接→创建Statement→设置参数→执行→映射结果→异常翻译→释放资源"的骨架，只把"映射结果"这步通过 RowMapper 回调留给开发者。RestTemplate、TransactionTemplate 都是同样的模式。

A
Q

queryForObject 和 query 的区别？什么时候用哪个？

queryForObject 期望恰好 1 条结果——0 条抛 EmptyResultDataAccessException，多条抛 IncorrectResultSize。query 返回 List，由你判断 size。推荐用 query + Optional：更安全、不抛异常。

A
Q

NamedParameterJdbcTemplate 比 JdbcTemplate 好在哪？

? 占位符是按位置绑定——参数顺序错了不报错但数据错乱。:name 是按名称绑定——顺序无关、更可读、重构友好。NamedParameterJdbcTemplate 底层还是 JdbcTemplate，只是多了一层参数名→位置的翻译。

A
Q

🔧 场景题：线上 MySQL 数据库主从切换后，从库连接开始抛 SQLException。你需要在 catch 中区分"连接断开"和"SQL 语法错误"，但 SQLException.getErrorCode() 在不同 MySQL 版本中不一样。怎么设计一个健壮的异常处理？

不要自己 parse errorCode。用 Spring 的 SQLErrorCodeSQLExceptionTranslator——它已经维护了各数据库版本的错误码映射。自己写的翻译逻辑永远不如 Spring 的全。如果 Spring 的翻译不准，可以继承 SQLErrorCodeSQLExceptionTranslator 并覆盖 customTranslate() 补充自定义规则。

A
Q

🔧 场景题：一个查询返回了 50 万条数据，用 JdbcTemplate.query() 全部加载到 List 里导致 OOM。怎么优化？

三种方案：① 改用 RowCallbackHandler——每条记录处理完就丢弃，不攒 List。② 分页查询（LIMIT/OFFSET）。③ 使用流式查询（需设置 fetchSize 和 ResultSet.TYPE_FORWARD_ONLY + CONCUR_READ_ONLY）。JdbcTemplate 默认一次性加载所有结果——大数据量时这是灾难。

A
Q

DataSourceUtils.getConnection() 和 dataSource.getConnection() 有什么区别？

DataSourceUtils.getConnection() 会先检查当前线程是否已经绑定了连接（通过 TransactionSynchronizationManager）——如果有就返回同一个连接（保证事务内共享连接）。dataSource.getConnection() 每次都返回新连接。在 @Transactional 方法内一定要用 DataSourceUtils。

📋 面试官总结

1. DataAccessException 体系 — 非受检 + 20+子类 + 跨数据库错误码翻译

2. JdbcTemplate 模板方法模式 — 固定骨架 + 回调，消灭样板代码

3. NamedParameterJdbcTemplate — 具名参数 > 位置参数，重构友好

4. DataSourceUtils vs DataSource — 事务感知 vs 每次新连接

⚠️ 高频陷阱：queryForObject 0 条/多条都抛异常；大数据量用 RowCallbackHandler 而非 query()
