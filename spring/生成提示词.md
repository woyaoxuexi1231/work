### 第一部分：基础使用与直观理解（扫清认知障碍）
- 用最简洁的代码和表格，展示这个知识点的基本用法、常见配置、API。
- 对比相似但不同的概念，一张表说清区别（如 @Configuration vs @Component）。

### 第二部分：原理深潜与设计动机（直抵灵魂）
这是教程的重中之重，必须包含以下完整逻辑链：
1. **假如没有它，会发生什么？**
   - 描述一个真实痛点：只用基础功能时系统会卡在哪里？比如“如果不用 CGLIB 代理，配置类内部的 @Bean 方法调用会产生多个实例，导致连接池配置单例失效”。
2. **设计者的破局巧思**
   - 详细阐述设计师是用什么“巧劲”解决了这个问题。它可能是一个代理、一层缓存、一种特殊的调用拦截。
   - 强调设计上的“啊哈时刻”，比如“用子类代理对开发者透明地截获方法调用，既保持了代码直觉，又保证了单例”。
3. **底层机制逐层剥开**
   - 给出伪代码或简化的源码逻辑，展示核心的拦截点、数据结构、执行流。
   - 画出一张调用链示意图（用文字描述，我会转成图），比如“a() -> 代理拦截 -> getBean('b') -> singletonObjects.get('b')”。
   - 解释为什么这样就能解决痛点，原理上如何做到。
4. **边界与限制**
   - 这个设计有什么副作用？什么场景下会失效？（比如 final 类无法被 CGLIB 代理）
   - 与其它方案的取舍和权衡。

## “触及灵魂”的讲述标准（必须达到）
以下是我对 “@Configuration vs @Component + @Bean” 这个知识点要求的标准，你讲解任何知识点都必须达到同等的透彻程度。注意看我是如何把设计巧思和原理融入同一个部分的：

**【例】@Configuration 为何能保证单例？**
- 用法：`@Configuration` 下 `@Bean` 方法相互调用，返回的是容器中的同一个实例。
- 对比：`@Component` 下 `@Bean` 调用每次 new，不保证单例。
- 设计巧思与原理：
  1. **痛点**：在配置类内部需要相互引用 Bean 时，普通方法调用会破坏单例，而强制开发者必须通过容器获取又不优雅。
  2. **巧思**：Spring 用 CGLIB 为该配置类生成一个代理子类，并注册到容器。这个代理子类重写了所有 `@Bean` 方法。
  3. **原理**：代理方法内部逻辑伪代码：
     ```java
     public class AppConfig$$Enhancer extends AppConfig {
         @Override
         public B b() {
             if (beanFactory.containsBean("b")) return beanFactory.getBean("b");
             return super.b();
         }
     }
     ```
     因此，`a()` 中调用 `b()` 时，实际触发的是代理的 `b()`，直接去容器取单例，完美解决。
  4. **边界**：类不能是 final 的，否则无法生成代理。这是 Full 模式的代价，而 Lite 模式没有代理，性能更好，但放弃了单例保障。

你在讲解任何知识点时，都必须复刻这种“痛点→巧思→伪代码→边界”的完整逻辑。
