QUESTION 17 · MVC

参数绑定与转换

@RequestParam / @RequestBody / @PathVariable 底层有何不同？

面试时你可以这样回答

🥉 青铜

"@RequestParam 和 @PathVariable 从 URL 获取字符串参数然后 TypeConverter 转换。@RequestBody 走 HttpMessageConverter 根据 Content-Type 解析 HTTP Body。"

🥈 白银

"@RequestParam 和 @PathVariable 不用 HttpMessageConverter——它们直接从 request.getParameter() 或 URL 模板提取字符串，然后用 TypeConverter 做 String→Integer 这样的简单类型转换。@RequestBody 则完全不同——HTTP Body 可能是 JSON/XML/Protobuf，Spring 根据 Content-Type 头遍历 HttpMessageConverter 链：第一个 canRead() 返回 true 的就用来反序列化。FormHttpMessageConverter 处理表单（application/x-www-form-urlencoded），MappingJackson2HttpMessageConverter 处理 JSON。新增格式只需加一个 Converter——Controller 代码不用改。"
为什么分层？URL 参数是简单 key-value → TypeConverter 够用。Body 是结构化数据 → 需要 HttpMessageConverter 链。不用一种方式处理所有——单一职责。

🥇 王者

"参数绑定和返回值处理的设计体现了 Spring MVC 对'输入输出通道'的抽象。想象一下如果没有 HttpMessageConverter——每个 Controller 都要自己判断 Content-Type、自己调 Jackson 或自己解析 XML。那 Controller 就不是 Controller 了——它是一个 HTTP 协议适配器。Spring 把'怎么读请求'和'怎么处理业务'彻底分离：前者交给参数解析器 + Converter 链，后者交给 Controller。返回值同理——@ResponseBody 走 Converter 链，String 走 ViewResolver。这两个通道的抽象让 Controller 只操心业务——Input 已经是 Java 对象，Output 也是 Java 对象。这就是 MVC 中'Model'的真正含义：不是数据库实体，而是脱离 HTTP 协议的纯 Java 对象。"
面试官心理："如果没有这套抽象，每个 Controller 都是 HTTP 协议适配器"——这个对比直接点出了设计的价值。
