异步请求 · 演化史

从"一个请求占一个线程"到"200 个线程能接 10000 个请求"

1998 第0代：同步阻塞——一个请求占一个线程

// Servlet 默认模型：每个请求分配一个 Tomcat 线程直到响应返回 protected void doGet(req, resp) { User user = userService.findById(id); // 查 DB — 线程在等 DB String info = httpClient.get("http://other-api"); // 调外部 API — 线程在等网络 resp.getWriter().write(user.getName() + info); // 返回 // 线程全程被占用——即使大部分时间在"等待" }
❌ 瓶颈不是 CPU，是线程数。Tomcat 默认 200 个线程。如果所有请求都在等外部 API（5 秒响应）——200 个线程全阻塞，第 201 个请求排队。线程绝大多数时间在等待，不是在工作。

2009 第1代：Servlet 3.0 startAsync —— 释放容器线程

Servlet 3.0 引入了 request.startAsync()——容器线程可以提前释放，后台线程处理完后再写响应。

AsyncContext ctx = request.startAsync(); Executors.newCachedThreadPool().submit(() -> { User user = userService.findById(id); // 业务线程执行——Tomcat 线程已释放 ctx.getResponse().getWriter().write(user.getName()); ctx.complete(); // 通知容器：响应写完了，可以发给客户端了 }); // Tomcat 线程在这里已经释放了——可以去接下一个请求
✅ 容器线程释放了——Tomcat 线程只负责"接客"，业务线程池负责"干活"。
⚠️ 问题：代码太啰嗦。要手动管理 AsyncContext、线程池、超时、异常处理。每个 Controller 都是这 10 行样板。

2012 第2代：Callable —— Spring 帮你管理线程

Spring MVC 3.2：Controller 返回 Callable<T>——Spring 自动把 Callable 提交给 TaskExecutor，自动管理 AsyncContext。

@GetMapping("/async") public Callable<String> async() { return () -> { Thread.sleep(3000); // 模拟耗时操作——在 TaskExecutor 线程中执行 return "OK"; }; // ★ Tomcat 线程在这之后立即释放！去接下一个请求 }
✅ 消灭了样板代码。不再手动调 startAsync()、管理线程池、管理 AsyncContext。

2014 第3代：DeferredResult —— 等待外部事件

Callable 只能在"返回一个线程去执行"的场景用。如果你在等 MQ 消息、等另一个用户的 WebSocket 回应——Callable 不行。你需要 DeferredResult。

@GetMapping("/deferred") public DeferredResult<String> deferred() { DeferredResult<String> result = new DeferredResult<>(30000L, "timeout"); // 在 MQ 消费者里——不用管线程！不用管 AsyncContext！ messageQueue.registerCallback(msg -> { result.setResult(msg); // ← 任意线程、任意时间调 setResult——响应立刻返回 }); return result; // Tomcat 线程立即释放 }
✅ 适用场景：长轮询、MQ 回调、多个外部服务并发调用等待最后一个结果。

核心洞察：异步不是为了"快"——是为了用更少的容器线程处理更多的请求。容器线程只做"接客"和"送客"，耗时操作交给业务线程池——200 个 Tomcat 线程可以支撑上万个并发。
