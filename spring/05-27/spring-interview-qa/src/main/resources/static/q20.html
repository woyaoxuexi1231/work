📋 Q20 · 异步请求 — DeferredResult / Callable / SSE。

面
DeferredResult 和 Callable 的处理流程？如何释放容器线程？
我
同步模型：每个请求占用一个 Tomcat 线程直到响应返回——200 个线程全阻塞，第 201 个排队。

Callable：Controller 返回 Callable → Spring 提交给 TaskExecutor → Tomcat 线程立即释放 → Callable 执行完 → 新 Tomcat 线程写响应。适合简单异步。
DeferredResult：更灵活——你可以在任意线程（MQ 消费者、定时任务）中调用 setResult()。适合事件驱动和长轮询。

两者都基于 Servlet 3.0 的 request.startAsync()。SSE 通过 SseEmitter 实现服务器单向推送——比 WebSocket 更轻量（HTTP 协议，自动重连）。
❓ 自问：什么时候同步就够了？
答：大部分业务同步足够。只有当你需要调用外部 API、等待 MQ 消息、或者处理长时间计算时才需要异步。不要为了异步而异步——增加复杂度。
