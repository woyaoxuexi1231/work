QUESTION 22 · 自动装配

自定义 Starter

如果让你写一个 my-starter，需要哪些模块？

面试时你可以这样回答

🥉 青铜

"需要两个模块：autoconfigure 模块（自动配置类、属性类）和 starter 模块（空壳 POM 做依赖聚合）。@ConfigurationProperties 绑定配置前缀，spring-boot-configuration-processor 自动生成 IDE 提示的 metadata.json。"

🥈 白银

"双模块结构不是过度设计。starter 模块：零代码，只定义依赖——用户引入 my-starter 时自动带来自动配置 + 所有传递依赖。autoconfigure 模块：包含 @ConfigurationProperties（配置绑定）、@AutoConfiguration（自动配置类 + @Conditional 条件）、以及业务 Service。为什么分开？因为用户可以不引入 starter 而直接引入 autoconfigure——如果他只想用你的配置模型但自己做依赖管理。spring-configuration-metadata.json 不是手写的——configuration-processor 在编译时扫描 @ConfigurationProperties 的 prefix、字段名、Javadoc，自动生成。"
注册：META-INF/spring/...AutoConfiguration.imports 中写自动配置类全限定名。Spring Boot 启动时 AutoConfigurationImportSelector 扫描。

🥇 王者

"自定义 Starter 的设计揭示了一个平台级框架的核心竞争力：'扩展的标准化'。一个 Starter 本质上是一套约定——自动配置类放在哪里、配置属性怎么命名、条件注解怎么用、如何注册到 spring.factories——所有这些都有明确的规范。这个规范让 Spring Boot 从一个'框架'变成了一个'平台'——任何人都可以写 Starter 来扩展它，而且所有的 Starter 遵循同一套游戏规则。这就是为什么 Spring Boot 生态能如此繁荣：不是 Spring 团队写了所有的 Starter，而是他们定义了写 Starter 的规范，让整个社区来贡献。这种能力是 Spring 二十年来最重要的架构决策：把扩展点标准化，然后把实现交给生态。"
面试官心理："扩展的标准化"——你不是在说怎么写 Starter，你在说 Spring 为什么能成为一个生态平台。
