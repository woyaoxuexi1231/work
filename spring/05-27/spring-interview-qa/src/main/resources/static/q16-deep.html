Spring MVC 请求处理：一段 30 年的演化史

当你理解了前一代的代码有多痛苦，你才能理解后一代的设计为什么是"必然"。

第一代：CGI（1993）

一个 HTTP 请求 = 一个操作系统进程。100 并发 = 100 个进程。
hello.plPerl CGI
#!/usr/bin/perl # 每次请求，Web Server（Apache/Nginx）fork 一个新进程执行这个脚本 # 进程执行完就销毁——下一次请求再来，重新 fork，重新执行 use CGI; my $q = CGI->new; # 手写 HTML，没有任何模板引擎 print $q->header('text/html; charset=UTF-8'); print "<html><body>"; print "<h1>Hello, " . $q->param('name') . "</h1>"; print "</body></html>"; # 如果要连数据库呢？ # 每个请求都要：connect → query → disconnect # 没有连接池——每个进程一条独立连接
❌ 致命缺陷：每个请求 fork 一个进程。进程创建/销毁的开销在并发场景下是灾难性的。100 个并发 = 100 个进程 = 内存爆炸。而且进程之间完全隔离——无法共享数据库连接、无法共享缓存。

第二代：Servlet（1998）

进程 → 线程。一个 Servlet 实例被所有请求共享，每个请求分配一个线程。
UserServlet.javaJava Servlet 2.0
// Servlet 不再为每个请求创建进程——一个 Servlet 实例常驻内存 // 每个请求分配一个线程调用 service() 方法 public class UserServlet extends HttpServlet { // ★ 第一个请求过来时，容器调用 init()——只调一次 public void init() { // 这里可以初始化一次：连数据库、加载配置 // 所有请求共享这个初始化结果——比 CGI 进步太多了 } // ★ 每个请求调这个方法——多线程并发调用同一个实例！ protected void doGet(HttpServletRequest req, HttpServletResponse resp) { // ── 问题 1：参数解析全靠手写 ── String idStr = req.getParameter("id"); // 拿到的永远是 String Long id = Long.parseLong(idStr); // 自己转类型 if (idStr == null || idStr.isEmpty()) { // 自己校验 resp.sendError(400, "id is required"); // 自己写错误响应 return; } // ── 问题 2：业务逻辑和 HTTP 细节混在一起 ── User user = userService.findById(id); if (user == null) { resp.sendError(404, "User not found"); return; } // ── 问题 3：响应全靠手拼 ── resp.setContentType("text/html;charset=UTF-8"); PrintWriter out = resp.getWriter(); out.println("<html><body>"); out.println("<h1>User: " + user.getName() + "</h1>"); out.println("<p>Email: " + user.getEmail() + "</p>"); out.println("</body></html>"); } // ★ 容器关闭时调用 destroy()——清理资源 public void destroy() { // 关闭数据库连接池 } }
web.xmlServlet 2.0 配置
<!-- 每个 Servlet 都要在 XML 里声明 —— 一个项目几十个 Servlet → XML 爆炸 --> <servlet> <servlet-name>userServlet</servlet-name> <servlet-class>com.example.UserServlet</servlet-class> <load-on-startup>1</load-on-startup> </servlet> <servlet-mapping> <servlet-name>userServlet</servlet-name> <url-pattern>/user</url-pattern> </servlet-mapping> <!-- 再加一个 OrderServlet → 又是一个 <servlet> + <servlet-mapping> --> <!-- 再加 20 个...你懂的 -->

💡 进步：进程→线程，共享实例。但产生了新问题：乱码处理、参数解析、类型转换、校验、XML 膨胀——每个 Servlet 都要写一遍。

❌ Servlet 时代没解决的

· 每个 URL 需要一个 Servlet → 类爆炸
· 参数永远是 String → 手动 parseLong
· 校验全靠 if-else → 大量重复代码
· 响应全靠 out.println → 维护噩梦
· URL 映射写在 XML → 修改要重启
· 业务代码和 HttpServletRequest 强耦合

✅ Servlet 已经解决的

· 进程→线程 → 内存效率大幅提升
· init() 一次初始化 → 共享资源
· destroy() 优雅关闭 → 资源释放
· Filter 链 → 统一的横切处理

第三代：Spring MVC 早期（2004-2009）

Controller 不再继承 HttpServlet——但还是要实现 Spring 的 Controller 接口。
UserController.javaSpring MVC 2.x
// Spring MVC 最早的设计：Controller 必须实现 Controller 接口 // 好处：不再需要继承 HttpServlet——你的类只是一个普通 Bean // 坏处：一个 Controller 只能处理一个 URL（和 Servlet 一样的问题！） public class UserController implements Controller { public ModelAndView handleRequest(HttpServletRequest req, HttpServletResponse resp) { // 参数还是手动解析——但有了数据绑定工具 String id = req.getParameter("id"); // 调 Service User user = userService.findById(Long.parseLong(id)); // 返回 ModelAndView——比直接 out.println 好一些，但还是很原始 ModelAndView mv = new ModelAndView("userView"); // 视图名 mv.addObject("user", user); // 数据 return mv; } }
dispatcher-servlet.xmlSpring MVC 2.x 配置
<!-- 虽然 Controller 变成了 Bean，但 URL 映射还是在 XML 里 --> <bean name="/user" class="com.example.UserController"/> <bean name="/order" class="com.example.OrderController"/> <!-- 好了一点：不想写 Servlet 了。但一个 URL 还是一个 Bean → 还是爆炸 -->
❌ 还是没解决：一个 Controller 处理一个 URL——和 Servlet 一样的问题。只是从"继承 HttpServlet"变成了"实现 Controller 接口"。XML 地狱依旧——URL 映射还是在配置文件里。

第四代：@RequestMapping 注解驱动（2009）

一个 Controller 可以处理多个 URL。@RequestParam 自动类型转换。
UserController.javaSpring MVC 3.0
// ★ 革命性变化：@Controller + @RequestMapping // 一个类可以处理多个 URL，不再需要为每个 URL 写一个类！ @Controller public class UserController { // URL "/users/1" 自动映射到这个方法 @RequestMapping(value = "/users/{id}", method = RequestMethod.GET) public String getUser(@PathVariable Long id, Model model) { // ★ @PathVariable 自动从 URL 提取 "1"，自动转成 Long！ // 再也不用 req.getParameter() + Long.parseLong() 了！ User user = userService.findById(id); model.addAttribute("user", user); return "userDetail"; // 返回视图名，不是 HTML 字符串 } // 同一个 Controller 里还可以有 POST 方法 @RequestMapping(value = "/users", method = RequestMethod.POST) public String createUser(@RequestParam String name, @RequestParam Integer age, // ★ 自动类型转换！ Model model) { User user = userService.create(name, age); model.addAttribute("user", user); return "redirect:/users/" + user.getId(); // 重定向也很简单 } }

❌ Spring MVC 3.0 还是没解决的

· 返回的是 JSP 视图名 → 还要写 JSP 模板
· 前后端不分离 → 改 UI 要改后端
· 配置还是在 XML 里（DispatcherServlet 等）
· 要手动配 ViewResolver、Converter

✅ Spring MVC 3.0 已经解决的

· 一个 Controller 处理多个 URL → 类爆炸终结
· @RequestParam 自动类型转换 → 告别 parseLong
· @PathVariable 从 URL 提取参数 → 告别 getParameter
· 不再需要实现任何接口 → 纯 POJO

第五代：Spring Boot + @RestController（2014-至今）

零 XML、零 JSP、自动配置。后端只返回 JSON——前端完全独立。
UserController.javaSpring Boot 3.x
// ★ 最终形态：Controller 就像一个普通的 Java 类 // HTTP 的一切细节全部被框架屏蔽 @RestController // = @Controller + @ResponseBody——所有方法返回 JSON @RequestMapping("/users") public class UserController { // GET /users/1 → {"id":1,"name":"Alice","email":"alice@example.com"} @GetMapping("/{id}") public User getUser(@PathVariable Long id) { return userService.findById(id); // 返回 User 对象 → Spring 自动序列化为 JSON → 写进 response body // 不需要 Model、不需要 ModelAndView、不需要视图名 } // POST /users + JSON body → 201 Created @PostMapping public ResponseEntity<User> createUser( @Valid @RequestBody CreateUserRequest req) { // ★ JSON → Java 对象自动转换 // @Valid: 自动校验 @NotNull/@Size → 不合法抛异常 → @ControllerAdvice 处理 User user = userService.create(req); return ResponseEntity.created(...).body(user); // 精确控制 HTTP 状态码 } }
application.ymlSpring Boot
# 零 XML！一切配置在 yml 里 server: port: 8080 spring: jackson: default-property-inclusion: non_null # null 字段不序列化

1998 Servlet vs 2024 Spring Boot

Servlet: 手写 getParameter → 手写 parseLong → 手写 out.println → 手写 XML 映射 Spring Boot: @PathVariable 自动提取 → @RequestBody 自动反序列化 → @RestController 自动 JSON → 零配置

同样是 GET /users/1
Servlet: 30 行代码，其中 25 行和业务无关
Spring Boot: 3 行代码，全部是业务逻辑

30 年演化，解决了什么？

第1代 CGI：进程→线程。解决了并发效率问题。
第2代 Servlet：引入 Filter、生命周期回调。
第3代 Spring MVC：Bean 化 Controller。摆脱继承。
第4代 注解驱动：一个类多 URL。自动类型转换。
第5代 Spring Boot REST：零 XML。全自动序列化。前后端分离。

每代解决前一代最痛的三个问题，再留给下一代三个新问题。

🧠 为什么 DispatcherServlet 被设计成这样？

① 策略模式大集合

HandlerMapping、HandlerAdapter、ViewResolver、HandlerExceptionResolver——全部是策略接口。因为在 30 年演化中，每种策略都换过至少 3 种实现。如果硬编码，每次换实现都要改框架源码。

② HandlerAdapter 的适配器模式

第3代 Controller 接口和第4代 @RequestMapping 注解——两种 Handler 的调用方式完全不同。没有 Adapter，DispatcherServlet 就得知道每一种 Handler 怎么调用。

③ HandlerMethodArgumentResolver

20+ 个参数解析器，每一个处理一种注解。你要支持新注解？加一个 Resolver 就行。Controller 代码一行不用改。开闭原则的教科书。

④ HttpMessageConverter 链

返回值怎么变成 HTTP Body？不是硬编码 Jackson——而是遍历 Converter 链。今天 JSON，明天 Protobuf——加一个 Converter 就搞定。Controller 完全无感。
