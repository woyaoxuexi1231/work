QUESTION 05 · IOC 容器

作用域

singleton / prototype 的原理？singleton 注入 prototype 如何保证每次拿到新实例？

面试时你可以这样回答

🥉 青铜

"singleton 是容器中唯一实例，prototype 每次 getBean() 创建新实例。singleton 注入 prototype 时只注入一次，后续拿到的是同一个引用。解决方式有 @Lookup、ObjectFactory、ApplicationContext.getBean()。prototype 的 @PreDestroy 不会被 Spring 调用。"

🥈 白银

"singleton 注入 prototype 之所以'失效'，不是 Spring 的 bug——是 Java 引用的基本特性：singleton 只创建一次，依赖注入也只执行一次，prototype 在注入那一刻被创建后，引用就被 singleton 永久持有。@Lookup 通过 CGLIB 生成子类重写方法——每次调用执行 getBean()，这是最透明的方案但依赖 CGLIB。ObjectProvider 是 Spring 4.3 引入的，实际上就是一个懒加载的 getBean() 包装，但比 @Lookup 更灵活——支持 getIfAvailable、getIfUnique、stream。prototype 不管理销毁是职责分离的设计：Spring 是'容器'不是'GC'，如果持有所有 prototype 引用会导致内存泄漏。"
源码：AbstractBeanFactory.doGetBean() 中，scope=prototype 时每次 new 一个 → 不走缓存。singleton 走 getSingleton() → 从 singletonObjects 取。

🥇 王者

"singleton 注入 prototype 的'失效'本质上暴露了 DI 容器的根本性局限：声明式注入是静态的、一次性的，而某些依赖关系天然是动态的、多次的。Spring 提供的三种解法（@Lookup、ObjectProvider、getBean）本质上都是在声明式注入的框架上撕开一个'命令式逃生口'。ObjectProvider 是三者中最优雅的——它把'懒加载'这个概念从框架内部机制（三级缓存的 ObjectFactory）推广到了应用层。你会发现 ObjectProvider 的接口签名和 ObjectFactory 几乎一样——这不是巧合，Spring 在 4.3 引入它时就是在说：'懒加载不仅是框架内部需要的，应用层也有同样的需求'。prototype 不管理销毁则揭示了另一个哲学：框架不应该替你做你不需要的事情。管理 prototype 的生命周期意味着 Spring 要维护一个对所有实例的引用表——这在大多数场景下是一种无谓的内存开销。Spring 的选择是：默认不做，你需要时自己管理。这其实是'约定优于配置'的反面——'不约定也是一种约定'。"
面试官心理："声明式注入上撕开命令式逃生口"——这句话把你和其他候选人瞬间拉开差距。
