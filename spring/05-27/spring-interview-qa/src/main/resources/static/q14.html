QUESTION 14 · 事务

事务实现原理

@Transactional 如何通过 AOP 织入？如何在事务提交后执行自定义逻辑？

面试时你可以这样回答

🥉 青铜

"@Transactional 通过 AOP 代理织入：TransactionInterceptor 在方法前后调用 PlatformTransactionManager 的 getTransaction/commit/rollback。事务提交后执行逻辑用 TransactionSynchronization.afterCommit()。"

🥈 白银

"完整流程：@EnableTransactionManagement → 注册 InfrastructureAdvisorAutoProxyCreator（BPP）+ BeanFactoryTransactionAttributeSourceAdvisor（Pointcut=有@Transactional的方法，Advice=TransactionInterceptor）。Bean 初始化时 BPP 检测到匹配 → 创建代理。方法调用时 TransactionInterceptor.invoke() → 从注解/配置获取 TransactionAttribute → 根据传播行为调用 PlatformTransactionManager.getTransaction() → 执行目标方法 → 成功则 commit，异常则判断 rollbackOn → rollback。TransactionSynchronizationManager 用 ThreadLocal 存储：当前连接、事务名、只读标志、隔离级别、以及 TransactionSynchronization 回调链。afterCommit 在 commit 后、finally 块中触发。"
回调链：beforeCommit → beforeCompletion → afterCommit → afterCompletion。afterCompletion 无论成功失败都会调用。

🥇 王者

"TransactionSynchronization 的设计体现了 Spring 对'事务生命周期的关注点分离'的极致追求。事务的核心操作（开启/提交/回滚）由 PlatformTransactionManager 完成——这是一个 SPI，解耦了具体的事务实现（JDBC/JTA/Hibernate）。而事务的'副作用'（发 MQ、清缓存、记录日志）由 TransactionSynchronization 完成——这是一个回调接口，解耦了'什么时候做'和'做什么'。这两个接口的分离让事务管理的核心逻辑和业务副作用完全解耦——你可以切换 JTA 实现而 afterCommit 的逻辑一行不改。afterCommit 的设计尤其精妙：为什么不在业务代码末尾直接发 MQ？因为 @Transactional 方法返回 ≠ 事务已提交——如果 commit 失败（比如数据库连接断开），消息已经发出去了。afterCommit 保证消息只在实际提交成功后发送——这个时序差异在高并发下是数据一致性的生命线。"
面试官心理："为什么不在业务代码末尾发 MQ？"这个追问本身就能过滤掉 80% 的候选人。
