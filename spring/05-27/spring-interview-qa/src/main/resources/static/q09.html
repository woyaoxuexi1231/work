📋 Q09 · 拦截链 — 多个切面如何排序？proceed() 阻断机制。

面
多个切面作用于同一方法时，执行顺序如何控制？
我
@Order 控制，值越小越先进入——洋葱模型，先进后出。

比如 @Order(1) 日志切面、@Order(2) 权限切面、@Order(3) 事务切面：进入时 1→2→3，退出时 3→2→1。这就形成了一个洋葱——最外层最先进入、最后退出。

@Transactional 的 order 默认是 Integer.MAX_VALUE——所以事务总是在最内层。为什么？因为事务应该最靠近业务代码——如果日志切面执行了 2 秒，事务已经开启了，数据库连接白白被占用 2 秒。最内层意味着最后开启、最先提交。
面
proceed() 如果不调用会怎样？
我
后续所有切面 + 目标方法都不会执行。这就是 @PreAuthorize 的实现基础——权限不通过，抛异常，proceed() 不被调用。

拦截链是递归实现的：ReflectiveMethodInvocation 维护一个拦截器列表和 index。每次 proceed() index++，取出下一个拦截器，调用它的 invoke(this)。如果某个拦截器不调用 mi.proceed()，递归就终止了——后续的全部跳过。
❓ 自问：多个切面 order 相同时执行顺序是什么？
答：不确定的——取决于它们被注册到容器的顺序。应该避免 order 重复超过 2 个。建议用常量类统一管理 order 值。
