QUESTION 09 · AOP

拦截链

多个切面作用于同一方法时，执行顺序如何控制？proceed() 不调用会怎样？

面试时你可以这样回答

🥉 青铜

"多个切面通过 @Order 控制顺序，值越小越先进入（洋葱模型，先进后出）。proceed() 不调用会阻断后续所有切面和目标方法。@Order 和 @Priority 类似，但切面排序用 @Order。"

🥈 白银

"拦截链的核心是 ReflectiveMethodInvocation 的递归结构——每个拦截器在 invoke() 中调用 mi.proceed()，形成递归调用栈。为什么用递归而不是 for 循环？因为递归天然支持'进入做 A，退出做 B'的环绕语义——拦截器在 proceed() 之前写前置逻辑，之后写后置逻辑。proceed() 不调用 = 递归在此终止 = 后续拦截器和目标方法全部跳过。这就是 @PreAuthorize 的实现基础——权限不通过时抛异常，异常在递归的某一层被捕获，后面的都不会执行。@Transactional 默认 order=Integer.MAX_VALUE 不是随便选的——事务必须在最接近业务代码的地方开启和提交，否则外层切面的耗时持有数据库连接是浪费。"
关键：ReflectiveMethodInvocation 维护一个 currentInterceptorIndex，每次 proceed() 递增。当 index == interceptors.size() 时，invokeJoinpoint() 执行目标方法。

🥇 王者

"拦截链的递归设计是一个精彩的'用控制流表达横切关注点'的案例。for 循环只能表达'依次执行'，无法表达'执行一半、回来再继续'的语义。递归的调用栈天然就是一个'进出栈'——进时从外到内，出时从内到外。更深层地看，proceed() 的阻断能力是整个 Spring Security 注解权限体系的基础——它不是 AOP 的 bug，是 intentionally designed interception point。而 @Transactional 默认最低优先级的设定体现了一个更根本的设计原则：'谁离数据最近，谁最后进来、最先出去'。数据访问层的事务应该是最内层的洋葱皮——日志、权限、缓存这些横切面应该在外面。这个优先级设计不是偶然的，它保证了无论你怎么自定义切面，数据一致性永远是最后一道防线。"
面试官心理："数据一致性是最后一道防线"——这句话让你从'会用 @Order'升级到'理解为什么这么设计'。
