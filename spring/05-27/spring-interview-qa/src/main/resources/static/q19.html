统一异常处理 · 演化史

从每个 Controller 里写 15 行 try-catch，到 @ControllerAdvice 一行搞定

1998 第0代：纯 Servlet——每处手写 try-catch

protected void doGet(req, resp) { try { User user = userService.findById(id); // 手拼 JSON 返回…… } catch (IllegalArgumentException e) { resp.setStatus(400); resp.getWriter().write("{\"error\":\"" + e.getMessage() + "\"}"); } catch (Exception e) { resp.setStatus(500); resp.getWriter().write("{\"error\":\"server error\"}"); } }
❌ 每个方法都要写 15 行 try-catch。20 个 Controller 方法 = 20 × 15 = 300 行重复样板。改一下错误格式？全部改一遍。

2004 第1代：@ExceptionHandler —— 一个 Controller 里的异常收敛

Spring 2.x 引入了 @ExceptionHandler。一个 Controller 里的所有异常处理收敛到几个方法。

@Controller public class UserController { @ExceptionHandler(IllegalArgumentException.class) public ResponseEntity handleBadArg(IllegalArgumentException e) { return ResponseEntity.badRequest().body(e.getMessage()); } // Controller 方法不用写 try-catch 了！ @GetMapping("/users/{id}") public User getUser(@PathVariable Long id) { return userService.findById(id); // 抛异常自动被 @ExceptionHandler 捕获 } }
✅ 进步：一个 Controller 的异常处理收敛了。业务方法不再写 try-catch。
⚠️ 还没解决的：如果 UserController 和 OrderController 都要处理 IllegalArgumentException？你要在两个 Controller 里写两遍一模一样的 @ExceptionHandler。还是重复。

2009 第2代：@ControllerAdvice —— 全局异常处理

Spring 3.0 引入了 @ControllerAdvice——它的作用范围不是"一个 Controller"，而是所有 Controller。

@ControllerAdvice // ← 对全局所有 @Controller 生效 public class GlobalExceptionHandler { @ExceptionHandler(IllegalArgumentException.class) public ResponseEntity handleBadArg(IllegalArgumentException e) { return ResponseEntity.badRequest().body(e.getMessage()); } @ExceptionHandler(Exception.class) public ResponseEntity handleAll(Exception e) { return ResponseEntity.status(500).body("server error"); } } // UserController、OrderController、ProductController……全部不用写异常处理了！
✅ 核心洞察：@ControllerAdvice 就是"Web 层的 AOP"。AOP 解决了 Bean 层的横切关注点（事务、日志），@ControllerAdvice 解决了 Controller 层的横切关注点（异常处理、返回值包装）。同一个设计思想，换了一个层面。

2014 第3代：ResponseBodyAdvice —— 正常路径也统一

异常统一了。正常返回呢？每个 Controller 还是返回各自的格式——有的返回 User，有的返回 String。前端收到的格式不一致。

@ControllerAdvice public class ResponseWrapper implements ResponseBodyAdvice<Object> { public boolean supports(...) { return true; } // ★ 在 @ResponseBody 序列化前拦截——所有正常返回值经过这里 public Object beforeBodyWrite(Object body, ...) { ApiResponse r = new ApiResponse(); r.setCode(200); r.setMessage("success"); r.setData(body); return r; // Controller 返回 User → 前端收到 {code:200, data:User} } }
✅ 正常路径 + 异常路径全部统一了。Controller 只返回业务对象，前端收到的永远是 {code, message, data} 的三段式格式。

★ 异常处理流程（DispatcherServlet 内部）

Controller 抛异常 → DispatcherServlet.processHandlerException() → 遍历 HandlerExceptionResolver 链： ① ExceptionHandlerExceptionResolver ← 扫描所有 @ControllerAdvice，找匹配的 @ExceptionHandler ② ResponseStatusExceptionResolver ← 处理 @ResponseStatus 注解的异常 ③ DefaultHandlerExceptionResolver ← 兜底（404/405/415 → 对应的 HTTP 状态码） → 找到匹配 → 反射调用 → 返回值走 HttpMessageConverter → JSON
