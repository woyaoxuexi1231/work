📋 Q03 · 依赖注入

@Autowired vs @Resource、构造器注入 vs 字段注入——Spring 最基础但也最容易答浅的问题。

面
@Autowired 和 @Resource 的查找逻辑有什么不同？
我
@Autowired 默认按类型找，@Resource 默认按名称找。

@Autowired：从容器中找所有匹配类型的 Bean → 只有一个就直接注入 → 多个用 @Qualifier 或 @Primary 决定 → 都没有就用字段名匹配。

@Resource：先按 name 找（如果写了 @Resource(name="xxx") 就用 name，没写就用字段名当 name）→ 找不到再按类型找。

优先级顺序：@Qualifier > @Primary > 字段名匹配 > 类型匹配。越明确的意图优先级越高。
面
字段注入有什么缺陷？为什么官方推荐构造器注入？
我
字段注入三个致命缺陷：

1. 不能写单元测试。你 new Service() 后所有 @Autowired 字段都是 null——调用任何方法都 NPE。必须启动整个 Spring 容器或者用 @InjectMocks。构造器注入可以直接 new Service(mockDao)。
2. 依赖不可变。字段不能是 final——任何代码都可以在运行时修改它。构造器注入的依赖是 final，编译期保证不可变。
3. 隐藏依赖。一个类有 8 个 @Autowired 字段，你第一眼看不出它依赖了多少东西。构造器参数列表一眼看到所有依赖——如果参数太多，说明类该拆了。

Spring 4.3 之后单构造器不需要 @Autowired——比字段注入还省一行注解。
追问："你在团队里怎么强制推行构造器注入？"
答：三管齐下。SonarQube 规则禁止字段注入；Checkstyle 检查；CR 时看到 @Autowired 在字段上直接打回。IDE 模板设置成生成 private final 字段 + 构造器。
❓ 自问：构造器注入有没有缺点？
答：有。如果一个类有 8 个依赖，构造器就有 8 个参数——代码很难看。但这不是构造器注入的问题，是你类违反单一职责的信号——8 个依赖说明这个类做了太多事，该拆了。构造器注入在这里起到了"设计坏味道检测器"的作用。
