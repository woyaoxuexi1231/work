QUESTION 21 · 自动装配

自动装配原理

@SpringBootApplication 包含哪三个注解？条件注解工作原理？

面试时你可以这样回答

🥉 青铜

"@SpringBootApplication = @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan。自动装配通过 spring.factories 加载 ~180 个配置类，用 @ConditionalOnClass、@ConditionalOnMissingBean 等条件注解过滤。"

🥈 白银

"核心流程：@EnableAutoConfiguration → @Import(AutoConfigurationImportSelector) → selectImports() → SpringFactoriesLoader.loadFactoryNames() 从所有 jar 的 spring.factories 中读 EnableAutoConfiguration 的候选类 → 全量 ~180 个 → 逐个检查 @Conditional → 不满足的跳过 → 剩余的就是要加载的配置。条件注解的工作：OnClassCondition 通过 classLoader.loadClass() 判断类路径，OnBeanCondition 通过 beanFactory.getBeanNamesForType() 判断容器中是否已有 Bean，OnPropertyCondition 通过 environment.getProperty() 判断配置值。@ConditionalOnMissingBean 是自动配置能被用户覆盖的关键——用户自己定义了就跳过自动配置。"
Spring Boot 3.x 变化：从 spring.factories 改为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports——一个文件一个目的，语义更清晰。

🥇 王者

"自动装配是 Spring 从'配置框架'到'推断平台'的质变。Spring Framework 时代，一切配置都是显式的：你要 DataSource 就自己配，要 TransactionManager 就自己配。Spring Boot 改变了这个范式：'你引入了 HikariCP 的 jar？那你大概率需要 DataSource——我帮你配一个，但如果你自己配了，我自动退让。'这个'大概率'就是条件注解的精髓——它不是 AI，而是在类路径、已有 Bean、配置项这三个维度上做推断。这种自动化的代价是'透明度下降'——你看到 @SpringBootApplication 一行注解，背后注册了上百个配置类。所以 --debug 模式和 actuator/conditions 端点不是可选的附加功能——它们是自动装配体系的'可观测层'，和自动装配本身同等重要。理解了这一点，你就理解了为什么 Spring Boot 3.x 把 spring.factories 拆成独立的 imports 文件：不是语法糖，是在为未来更复杂的推断模式（如 GraalVM native image 的条件注册）做准备。"
面试官心理："推断平台"、"可观测层"——你不是在背 spring.factories 的路径，你理解了自动装配是一个'推断系统'而不仅仅是'自动配置'。
