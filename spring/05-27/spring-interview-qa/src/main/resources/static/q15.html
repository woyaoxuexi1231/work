QUESTION 15 · 事务

分布式事务

只用 Spring 框架，不引入 Seata，你能怎么实现简单的分布式事务？

面试时你可以这样回答

🥉 青铜

"三种方案：TCC（Try-Confirm-Cancel）手动实现、本地消息表 + 定时任务 + 幂等、JTA（Atomikos/Bitronix 配置 XA 数据源）。本地消息表最常用。"

🥈 白银

"TCC 的核心是把一个操作拆成 Try（预留资源）、Confirm（真正执行）、Cancel（释放资源）三步。挑战是空回滚（Cancel 时 Try 可能没执行）、悬挂（Cancel 比 Try 先到）、幂等（Confirm/Cancel 可能重复调用）——都需要事务 ID 去重。本地消息表的精髓在于把'业务操作 + 写消息'放在同一个本地事务中，利用本地事务的原子性保证消息一定写入，然后用定时任务异步投递，消费方幂等。JTA 最简单但 XA 协议的同步阻塞特性不适合高并发——prepare/commit 两个阶段都要等所有参与者确认。"
CAP 权衡：JTA 追求强一致性（牺牲可用性），本地消息表追求最终一致性（牺牲实时性）。没有银弹。

🥇 王者

"Spring 没有内置分布式事务方案，这个'缺失'恰恰是 Spring 最深刻的设计选择。分布式事务是一个问题空间，不是一个解决方案——CAP 理论约束下，任何方案都是在一致性、可用性、分区容错性之间的取舍。如果 Spring 内置了一种方案（比如 JTA），它就在替所有用户做取舍——这对于一个声称'不绑定方案'的框架来说是不可接受的。Spring 选择了提供 PlatformTransactionManager 这个 SPI——JTA 可以实现它、Seata 可以实现它、你写的 TCC 框架也可以实现它。这才是 Spring 作为平台的本分：定义接口标准，让生态竞争出最好的实现。面试时如果有人问'Spring 怎么解决分布式事务'，最好的回答不是列方案，而是解释为什么 Spring 不解决这个问题——以及这个'不解决'背后的架构智慧。"
面试官心理："为什么 Spring 不解决"比"怎么解决"更能体现架构思维。这题答对了，面试官会开始跟你讨论 CAP 理论。
