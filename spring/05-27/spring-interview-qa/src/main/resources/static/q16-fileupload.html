1"流"到底是什么——一个 10KB 的比喻

你有一个 10GB 的水库，要把水运到另一个城市。你有两种方案：

方案 A：把整个水库装车运过去

你需要一辆能装 10GB 水的卡车。水箱的容量必须 ≥ 10GB。
10GB 文件 → 你需要 10GB 内存 → 卡车造不出来 → OOM。
这就是 byte[] body = request.getInputStream().readAllBytes()。

方案 B：用水桶一桶一桶舀

你只需要一个 8KB 的水桶。舀一桶 → 倒进对面的水池 → 回去再舀一桶。
任何时刻，你手上只有一桶水——不管水库多大，水桶永远是 8KB。
这就是 流。
流的本质 — 8KB 的秘密
byte[] buffer = new byte[8192]; // ← 这个 8KB 数组就是"水桶" int bytesRead; while ((bytesRead = in.read(buffer)) != -1) { // 舀一桶 out.write(buffer, 0, bytesRead); // 倒掉 } // 循环结束，水库搬空了——但内存只用过 8KB

3MultipartResolver：Spring 帮你做的那一步

这是整个流程中最关键的一步——也是很多人不理解"Spring 到底帮了我什么"的地方。

StandardServletMultipartResolver — Spring 的处理逻辑
// 请求进来时，body 长这样： // ------WebKitFormBoundary7MA4YWxk // Content-Disposition: form-data; name="file"; filename="photo.jpg" // Content-Type: image/jpeg // // <二进制 JPEG 数据> // ------WebKitFormBoundary7MA4YWxk-- // MultipartResolver 做的事： // ① 读 boundary 分隔符 —— 找到 file 字段的起止位置 // ② 解析 Content-Disposition 头 —— 拿到 filename=photo.jpg // ③ ★ 核心：读 body 里的文件内容，根据 file-size-threshold 决定放哪： // 小于阈值 → new byte[fileSize] → 存在内存 // 大于阈值 → 创建临时文件 → InputStream → OutputStream 写到磁盘 // ④ 封装成 MultipartFile 对象，塞进 request 的 parameter map // ⑤ Controller 拿到 MultipartFile —— 此时数据已在内存或临时文件里
💡 关键洞察：你在 Controller 里拿到的 MultipartFile 已经不是原始的网络流了——MultipartResolver 已经把文件内容从 HTTP body 中提取出来，放在了内存（小文件）或临时文件（大文件）里。你调 file.getBytes() 时不是去读网卡，是去读内存/临时文件。

6一句话回答"流式传输到底是什么"

流式传输 = 不用把整个文件读进内存，而是用一个小缓冲区（通常 8KB）循环"读一块→写一块"，直到处理完整个文件。任何时刻内存里只有那个小缓冲区的数据——无论文件是 1MB 还是 1GB，内存占用恒定。

Spring 的 MultipartFile.transferTo() 已经帮你做了流式写入目标位置。 但流式的第一步——把文件内容从 HTTP body 中提取出来——是 MultipartResolver 做的。 它根据 file-size-threshold 决定放在内存还是临时文件。 你写的 Controller 代码不需要关心这些——拿到 MultipartFile 时，数据已经就位了。

概念内存占用代表代码
全量读= 文件大小 → 大文件必 OOMbyte[] b = inputStream.readAllBytes()
流式读= 缓冲区大小（恒定 8KB）while (in.read(buf) != -1) out.write(buf, 0, n)
