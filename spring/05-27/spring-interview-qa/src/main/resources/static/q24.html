配置优先级 · 演化史

从"只有一个 application.properties"，到"12 层配置源，同一个 JAR 通吃所有环境"

2004 第0代：只有一个 properties 文件

// application.properties —— 开发环境的值写死在里面 server.port=8080 db.url=jdbc:mysql://localhost:3306/dev // 要上线了？改文件 → 重新打包 → 部署 // 换了测试环境？改文件 → 重新打包 → 部署
❌ 同一个 JAR 不能复用。开发、测试、预发布、生产——四个环境需要四个不同的构建产物。改一行配置就要重新打包。和十二因子应用（12-Factor App）原则违背——配置必须与代码分离。

2014 第1代：外部化配置 + 优先级体系

核心原则：越靠近部署环境，优先级越高。

// 同一个 JAR，不同环境覆盖不同配置： application.yml server.port=8080 （代码里的默认值） application-dev.yml server.port=8081 （开发覆盖） 环境变量: SERVER_PORT=9090 （K8s 注入） 命令行: --server.port=80 （运维在部署时决定） // 最终生效：80 —— 命令行参数优先级最高

12 层优先级（从高到低）：

  1. 命令行参数 — --server.port=80
  2. SPRING_APPLICATION_JSON
  3. Servlet 参数、JNDI
  4. 环境变量 — export SERVER_PORT=80
  5. 系统属性 — -Dserver.port=80
  6. application-{profile}.yml（jar 外部）
  7. application-{profile}.yml（jar 内部）
  8. application.yml（jar 外部）
  9. application.yml（jar 内部）
  10. @PropertySource
  11. SpringApplication.setDefaultProperties()
✅ 核心洞察：同一个构建产物从开发到生产不用重新打包。开发用 application-dev.yml，上线时用环境变量覆盖。Environment 的 PropertySource 是一个有序列表——getProperty("server.port") 遍历它，第一个非 null 的就是最终值。这就是为什么用 List 而不是 Map——Map 不支持同名 key 的覆盖语义。
