📋 Q02 · 循环依赖

三级缓存是 Spring 面试的必考题。关键是讲清楚"为什么必须三级"。

面
Spring 如何解决单例 Bean 的循环依赖？三级缓存分别存什么？
我
三级缓存解决。先说三个缓存放什么——

一级 singletonObjects：完全初始化好的成品 Bean
二级 earlySingletonObjects：已实例化但未初始化的半成品
三级 singletonFactories：一个 ObjectFactory，能生成半成品

流程：A 和 B 互相依赖。A 实例化 → 三级缓存放入 ObjectFactory → A 填充属性时发现需要 B → 创建 B → B 实例化 → B 填充属性时发现需要 A → 从三级缓存取 ObjectFactory → getObject() 拿到 A 的引用 → B 完成 → A 继续填充 → A 完成。
面
为什么必须三级缓存，只用二级行不行？
我
不行。原因两个字：AOP 代理。

AOP 代理在初始化末尾（BPP.after）才生成。但循环依赖发生在属性填充时——比代理生成早。如果只有二级缓存，A 实例化后直接放二级缓存（原始 A）。B 拿到的是原始 A。之后 A 初始化完毕、生成代理——一级缓存放代理 A。同一个 Bean 两个版本，B 手上的原始 A 和一级缓存里的代理 A 不是同一个对象。

三级缓存的 ObjectFactory 解决了这个时间差：getObject() 被调用时判断"需要代理吗？"需要就提前生成，不需要就返回原始对象。无论谁、什么时候拿，拿到的都是同一个版本。
追问："那构造器注入为什么解决不了循环依赖？"
答：循环依赖的前提是 Bean 能先实例化再放入缓存。构造器注入时，A 的构造器需要 B、B 的构造器需要 A——双方连实例化都完成不了，根本没机会进缓存。setter/字段注入先实例化（进缓存），再注入依赖（从缓存取）。
❓ 自问：prototype 的循环依赖 Spring 能解决吗？
答：不能。prototype 每次 getBean() 都新建，没有缓存这个概念。循环依赖变成无限递归，直接抛 BeanCurrentlyInCreationException。遇到 prototype 循环依赖，说明你的设计有问题——该重构了。
