QUESTION 02 · IOC 容器

循环依赖

Spring 如何解决单例 Bean 的循环依赖？三级缓存分别存什么？为什么必须三级缓存？构造器注入为什么无法解决？

面试时你可以这样回答

🥉 青铜（背答案）

"三级缓存是 singletonObjects、earlySingletonObjects 和 singletonFactories。一级存放完全初始化的 Bean，二级存放早期暴露的 Bean，三级存放 ObjectFactory 用于生成代理对象。"

🥈 白银（讲来源）

"二级缓存通过分离实例化和初始化解决了纯 POJO 的循环依赖。但 AOP 代理在初始化末尾才创建，循环依赖发生在属性填充阶段——时间上冲突了。三级缓存的 ObjectFactory 是一个 lazy evaluation：只有真的发生循环依赖时才提前创建代理，否则 lambda 永远不会被调用，代理仍然在正常时机创建。"
流程推演：
1. 创建 A → 实例化 → 三级缓存放入 ObjectFactory(lambda: getEarlyBeanReference)
2. 填充 A → 需要 B → 创建 B
3. B 实例化 → 填充 B → 需要 A → 从三级缓存取 ObjectFactory → getObject()
4. getEarlyBeanReference 内部判断：A 需要代理吗？
   需要 → 提前创建代理 → 放入二级缓存 → 返回代理
   不需要 → 返回原始 A → 放入二级缓存
5. B 完成 → A 继续填充 → AOP 在 after 阶段再次检查：
   代理已在二级缓存？→ 直接取出，不再重复生成

🥇 王者（讲设计哲学）

"三级缓存本质上是 Spring 面对两个正交设计——分步创建和延迟代理——在循环依赖这一边界条件下产生冲突时，做出的最小侵入性修复。它没有选择让 IoC 硬编码 AOP（破坏解耦），也没有选择无条件提前代理（破坏生命周期），而是引入了一个 lambda 工厂作为'承诺'，把'是否提前创建代理'的决定推迟到真正需要的时刻。getEarlyBeanReference 内部通过 SmartInstantiationAwareBeanPostProcessor 仍然走的是 BPP 扩展机制，IoC 容器一行 AOP 代码都不需要知道。这其实就是延迟计算 + 开闭原则的教科书级应用。"
面试官心理：你说出"lambda 工厂作为承诺"、"最小侵入性修复"、"IoC 容器一行 AOP 代码都不需要知道"这三句话时，面试官脑子里只有一个念头：这个人真的读过源码。

追问：构造器注入为什么无法解决？

循环依赖的前提是 Bean 能先实例化再放入缓存。构造器注入的问题在于——A 的构造器需要 B，B 的构造器需要 A，双方都无法完成实例化，根本没有机会进入缓存。setter/字段注入先实例化（放入缓存），再注入依赖（从缓存获取）。
