QUESTION 06 · IOC 容器

Bean 覆盖与冲突

Spring Boot 中同名 Bean 默认允许覆盖吗？多模块冲突怎么排查？

面试时你可以这样回答

🥉 青铜

"Spring Boot 2.1+ 默认不允许 Bean 覆盖，启动会报 ConflictingBeanDefinitionException。可以通过 spring.main.allow-bean-definition-overriding=true 开启。排查时用 --debug 看 AUTO-CONFIGURATION REPORT。"

🥈 白银

"Spring Framework 原生和 Spring Boot 在 Bean 覆盖上的默认行为是不同的——这是很多人踩坑的地方。Spring Framework 的 DefaultListableBeanFactory.allowBeanDefinitionOverriding 默认是 true，后注册的悄悄覆盖前面的。Spring Boot 在 2.1 通过 SpringApplication 把这个值改成了 false。为什么要改？生产事故太多——你引入一个 starter，它内部定义了一个和你同名的 Bean，1.x 时代静默覆盖，没有任何警告，出问题根本不知道是哪个 Bean 被替换了。改成 false 后启动直接报错，fail-fast。"
排查工具链：--debug 启动 → Positive matches（生效的）/ Negative matches（跳过的）→ 看哪个 Condition 不满足。

🥇 王者

"Bean 覆盖的默认值变更揭示了一个更深层的设计哲学演变：Spring Framework 的'宽松'是为了灵活性——它最初是一个框架，开发者应该自己知道在做什么。Spring Boot 的'严格'是为了安全性——它已经变成了一个平台，面对的是数百万开发者，不能假设每个人都理解所有配置的含义。从'true'到'false'的这一个字符的变更，背后是从'框架思维'到'平台思维'的跨越——框架提供能力，平台提供护栏。这和 Java 的 checked exception、Rust 的所有权系统是同一个方向：在编译/启动期暴露错误，而不是在生产环境等着爆炸。你团队里的多模块冲突，解决思路只有两条路：治标的 @Primary/@Qualifier 指定优先级，治本的 @ConditionalOnMissingBean 让自动配置变成'温和注册'——用户不定义我才注册，用户定义了我自动退让。"
面试官心理："从框架思维到平台思维的跨越"——这个概括说明你理解的是设计趋势，不是孤立的配置项。
