拦截器 vs 过滤器 · 演化史

Filter 是 Servlet 规范级别的能力，Interceptor 是 Spring MVC 级别的能力。它们不是在竞争——是不同层次解决不同问题。

1998 第0代：没有 Filter——每个 Servlet 自己干

// 每个 Servlet 都要自己处理编码、自己写日志、自己检查权限 public class UserServlet extends HttpServlet { protected void doGet(req, resp) { req.setCharacterEncoding("UTF-8"); // ← 每个 Servlet 都写一遍！ resp.setContentType("text/html;charset=UTF-8"); System.out.println("收到请求：" + req.getRequestURI()); // ← 日志也每个写一遍 String token = req.getHeader("Authorization"); if (token == null) { resp.sendError(401); return; } // ← 权限还是每个写一遍 // ... 业务逻辑 } }
❌ 痛点：编码设置、日志记录、权限校验——这些横切逻辑在每个 Servlet 里重复了 N 遍。你改了编码配置，要改所有 Servlet。加一个日志，要改所有 Servlet。如果忘记给新的 Servlet 加编码，乱码了查不出来。

2000 Filter 出现——Servlet 2.3 规范

核心思想：在 Servlet 之前加一道拦截链——只需要写一个 Filter 类，配置一次，所有请求自动经过它。

// Filter：一个类处理所有请求的横切逻辑 public class CharsetFilter implements Filter { public void doFilter(req, resp, chain) { req.setCharacterEncoding("UTF-8"); // ← 只写一次，所有 Servlet 生效！ chain.doFilter(req, resp); // ← 放行 } }
✅ Filter 解决的：横切逻辑从 N 个 Servlet 中抽离到 1 个 Filter 里。编码、日志、CORS、XSS 过滤——这些"基础设施"层面的事情，Filter 统一处理。
⚠️ Filter 的局限：
  • 不能注入 Spring Bean：Filter 是 Servlet 容器管理的，不是 Spring 管理的。想用 TokenService？要绕 DelegatingFilterProxy 桥接
  • 精度低：只能按 URL 模式匹配——不能根据 哪个 Controller 方法 决定要不要放行
  • 没有方法级别的后置回调：Filter 只有 doFilter 一个方法，没有 afterCompletion 这种 finally 回调
  • Filter 是"全或无"：不像 Interceptor 有个 preHandle/postHandle/afterCompletion 三个阶段的精细控制

2004 HandlerInterceptor 出现——Spring MVC

核心思想：在 DispatcherServlet 内部（Filter 之后），提供更精细的拦截能力。

// Interceptor：可以注入 Spring Bean，知道 HandlerMethod，三段式回调 public class TokenInterceptor implements HandlerInterceptor { boolean preHandle(req, resp, handler) { // ① 前置（类似 Filter） if (!(handler instanceof HandlerMethod)) return true; // 可以注入 TokenService，可以做注解排除 // false → 请求被拦截，Controller 不会执行 } void postHandle(req, resp, handler, mav) { // ② 后置（Controller 之后，View 之前） // 可以修改 ModelAndView } void afterCompletion(req, resp, handler, ex) { // ③ 最终（类似 finally） // 清理 ThreadLocal 之类的 } }
✅ Interceptor 解决 Filter 解决不了的：
  • 可以 @Autowired：Interceptor 是 Spring 管理的 Bean，直接注入 TokenService
  • 知道是哪个 Controller：handler 参数就是 HandlerMethod——可以做 @NoAuth 注解排除
  • 三段式回调：preHandle（前置）、postHandle（后置，可改 ModelAndView）、afterCompletion（finally）
  • 排除路径精确：registry.addInterceptor().excludePathPatterns("/login")
⚠️ Interceptor 的局限：它运行在 DispatcherServlet 内部——请求已经经过了 Filter，body 可能已经被消费了。你读不到最原始的请求数据（Filter 可以）。

★ 执行顺序：Filter → Interceptor → Controller

Filter.doFilter() 前置
  → DispatcherServlet.doDispatch()
    → HandlerMapping 找到 HandlerExecutionChain（Handler + Interceptors）
    → Interceptor.preHandle() ← 可以注入 Spring Bean
      → Controller 方法执行
    → Interceptor.postHandle() ← 可以改 ModelAndView
    → ViewResolver 解析视图
    → Interceptor.afterCompletion() ← 类似 finally
Filter.doFilter() 后置

★ 实际项目中用 Filter 还是 Interceptor？

场景选哪个为什么
编码设置 (CharacterEncodingFilter)Filter需要在 Spring 解析参数之前就设好编码。Interceptor 太晚了
CORS (CrossOriginFilter)Filter跨域头必须在最外层设置——浏览器/网关层就应该看到的配置
Token 鉴权Interceptor需要注入 TokenService、需要排除登录接口、需要按 HandlerMethod 控制
请求日志（记 URL/参数/耗时）Filter 或 InterceptorFilter 能记原始 body，Interceptor 能记更多上下文
XSS / SQL 注入过滤Filter必须拿到最原始的 request body，Interceptor 拿不到
加密/解密请求体Filter需要在 Spring 解析 body 之前就解密，Interceptor 太晚
权限校验（基于方法注解）Interceptor能通过 HandlerMethod 获取方法上的 @RequiresRole 注解
💡 经验法则：Filter 做"进门前"的事——基础设施，不用 Spring 的能力。Interceptor 做"进门后"的事——业务逻辑，需要 Spring 的能力。
