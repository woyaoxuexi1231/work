QUESTION 11 · AOP

自研简化版 AOP 引擎

如果让你自己实现一个支持前置、后置、环绕通知的 AOP 引擎，核心接口怎么设计？

面试时你可以这样回答

🥉 青铜

"核心接口：Advice 接口族（BeforeAdvice / AfterAdvice / AroundAdvice）、Pointcut 接口（matches 方法）、Aspect 类（绑定 Advice + Pointcut）、AopEngine（注册切面 + 创建代理）。代理通过 JDK 动态代理在 Bean 初始化时织入。"

🥈 白银

"200 行代码的核心架构：Advice 接口族为什么分 Before/After/Around 三个而不是一个？因为不同类型通知有不同的方法签名——Before 不需要返回值，After 需要接收返回值，Around 需要 MethodInvoker 控制链。如果合并成一个接口，每个通知都要写一堆 if-else 判断自己是什么类型。Pointcut 用接口抽象是因为切点的匹配方式多种多样（注解、表达式、正则）——接口让每一种都可以独立实现。MethodInvoker 链是最关键的设计：从目标方法开始，逐层包裹 AroundAdvice，形成洋葱模型。这和 ReflectiveMethodInvocation 的递归是同一个思想——用控制流表达横切关注点的层次。"
对比 Spring：你写的 AopEngine.registerBean() 对应 AbstractAutoProxyCreator.postProcessAfterInitialization()——都是成品阶段织入。

🥇 王者

"自研 AOP 引擎的目的不是造轮子——是通过'自己实现一遍'来暴露出 Spring AOP 中那些'如果我来做，我也会这么设计'的必然性。你会发现三个必然：第一，Advice 必须按类型分接口——这是接口隔离原则，你不可能用一个 doEverything() 覆盖所有通知类型。第二，Pointcut 必须是接口——否则换一种匹配方式就要改引擎源码，违反开闭原则。第三，MethodInvoker 链必须是递归——你尝试用 for 循环实现环绕通知就会立刻撞墙。这些不是你选择的，是问题结构本身决定的。理解了这三个必然性，你就理解了 Spring AOP 的架构不是某个人的偏好，而是这个问题的唯一优雅解。"
面试官心理："问题结构本身决定的"、"唯一优雅解"——不是在背 Spring 的设计，是在推演设计。
