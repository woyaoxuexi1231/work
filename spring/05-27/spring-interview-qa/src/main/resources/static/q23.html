ApplicationContext.refresh() · 源码级详解

12 个步骤，每一步做什么、为什么这么设计、跳过它会出什么问题

refresh() 全景——12 步一览

① prepareRefresh() → 准备：校验配置、初始化 PropertySource
② obtainFreshBeanFactory() → 创建/刷新 BeanFactory
③ prepareBeanFactory() → 给 BeanFactory 塞工具（ClassLoader、SpEL 解析器、Aware 处理器）
④ postProcessBeanFactory() → 留给子类定制的空方法
⑤ invokeBeanFactoryPostProcessors() → ★ 调用所有 BFPP — 解析 @Configuration/@Bean
⑥ registerBeanPostProcessors() → 注册所有 BPP（此时只是注册，还没调用）
⑦ initMessageSource() → 国际化消息源
⑧ initApplicationEventMulticaster() → 事件广播器
⑨ onRefresh() → Spring Boot 在这里创建内嵌 Tomcat
⑩ registerListeners() → 注册事件监听器
⑪ finishBeanFactoryInitialization() → ★ 实例化所有单例 Bean
⑫ finishRefresh() → 启动 WebServer + 发布 ReadyEvent

① prepareRefresh() — 开工前检查

做什么：设置启动时间、标记 active=true、初始化 PropertySource、校验 required 属性。
protected void prepareRefresh() { this.startupDate = System.currentTimeMillis(); this.closed.set(false); this.active.set(true); initPropertySources(); // 留给子类——比如 Web 环境初始化 ServletContext 属性 getEnvironment().validateRequiredProperties(); // 校验 spring.profiles.active 等必须的属性 }
为什么这样设计：如果连 application.yml 里的必须属性都没配（比如数据库 URL），后面的所有步骤都是白费——Bean 创建到一半发现缺配置再报错，浪费了前面所有的初始化时间。这里做的是 fast-fail——开工前把能检查的都查了。

② obtainFreshBeanFactory() — 获取/刷新 BeanFactory

做什么：如果已有 BeanFactory 就销毁重建；新建一个 DefaultListableBeanFactory；加载所有 BeanDefinition。
DefaultListableBeanFactory beanFactory = createBeanFactory(); loadBeanDefinitions(beanFactory); // 扫描 @Component、解析 @Bean 等 → 注册 BeanDefinition
为什么这样设计：BeanFactory 不只是一个 Map——它是整个 IOC 的核心引擎。这一步创建的是一个"空的工厂"——有了骨架，但还没有任何 Bean。加载的 BeanDefinition 只是"蓝图"，不是"成品"。把创建工厂和加载蓝图分开，是为了让后面的步骤可以修改蓝图（BFPP）。

③ prepareBeanFactory() — 给工厂装工具

做什么：设置 ClassLoader、注册 SpEL 解析器、注册 ApplicationContextAwareProcessor（一个 BPP）、忽略一些不应自动注入的接口。
// 注册 ApplicationContextAwareProcessor —— Context 级别的 Aware 回调全靠它 beanFactory.addBeanPostProcessor(new ApplicationContextAwareProcessor(this)); // 告诉 BeanFactory：这些接口不应该被自动注入（它们是容器回调，不是业务依赖） beanFactory.ignoreDependencyInterface(ResourceLoaderAware.class); // 注册一些常用 Bean 的"现成实例"——比如 applicationContext 本身、environment beanFactory.registerResolvableDependency(ApplicationContext.class, this);
为什么这样设计：这些不是"某个特定项目需要的"——是所有 Spring 项目都需要的。ClassLoader、SpEL 解析器、Aware 回调——没有它们，Bean 连最基本的 Spring 能力都用不了。放在 refresh() 的统一流程里，而不是让每个项目自己配，是"约定优于配置"的体现。

④ postProcessBeanFactory() — 空方法，留给子类

做什么：什么都不做。这是一个模板方法——子类（如 Web 环境的 AnnotationConfigServletWebServerApplicationContext）可以覆盖它来添加 Web 特有的配置。
为什么这样设计：模板方法模式。不是所有 ApplicationContext 都一样——Web 环境要注册 ServletContext、Reactive 环境要注册 ReactiveAdapterRegistry。这个空方法让每种 Context 可以插入自己的逻辑，而不影响主流程。

⑤ invokeBeanFactoryPostProcessors() — ★ 解析 @Configuration

做什么：调用所有 BeanFactoryPostProcessor。最核心的：ConfigurationClassPostProcessor 解析 @Configuration 类，把 @Bean 方法注册为 BeanDefinition。
// ★ 核心中的核心：ConfigurationClassPostProcessor.postProcessBeanDefinitionRegistry() // 遍历所有 @Configuration 类 → 解析 @Bean、@ComponentScan、@Import、@PropertySource // → 为每个 @Bean 方法注册一个 BeanDefinition // → 此时，你写的所有 @Bean 才真正被容器"认识" // 同时还调用 PropertySourcesPlaceholderConfigurer——替换 ${server.port} → 8080
为什么设计在⑤而不是①：BFPP 需要操作 BeanFactory（修改 BeanDefinition）。BeanFactory 在②才创建、在③才配置好——所以 BFPP 必须在这之后执行。但必须在⑥之前——因为⑥注册的 BPP 要在 Bean 创建时使用，而 Bean 创建时 BFPP 必须已经完成了所有的 BeanDefinition 修改。

⑥ registerBeanPostProcessors() — 注册 BPP

做什么：扫描所有 BeanPostProcessor 类型的 Bean，按 PriorityOrdered → Ordered → 普通 排序，注册到 BeanFactory。但此时只是注册，还没调用——调用发生在⑪每个 Bean 创建时。
// 注册顺序决定了调用顺序——先注册的先调用 // PriorityOrdered: ApplicationContextAwareProcessor 等 // Ordered: AutowiredAnnotationBeanPostProcessor 等 // 普通: AnnotationAwareAspectJAutoProxyCreator 等
为什么设计在⑥：BPP 要在 Bean 创建时使用——所以必须在 Bean 创建之前注册好。但必须在 BFPP 之后——因为有些 BPP 本身也需要 BFPP 来注册（比如 @EnableAspectJAutoProxy 注册的 AnnotationAwareAspectJAutoProxyCreator，它的 BeanDefinition 是在⑤里由 ConfigurationClassPostProcessor 解析出来的）。

⑦initMessageSource — 国际化

如果没有自定义的 MessageSource Bean，就注册一个默认的（读 messages.properties）。国际化消息的解析全靠它。
为什么这时候初始化：后面的 Bean 创建可能需要国际化消息（比如校验错误信息）。必须赶在⑪ Bean 创建之前准备好。

⑧initApplicationEventMulticaster — 事件广播器

如果没有自定义的，就创建一个 SimpleApplicationEventMulticaster。后面所有的 ApplicationEvent 都通过它广播。

⑨onRefresh() — ★ Spring Boot 在这里创建内嵌 Tomcat

又是一个模板方法——空的。ServletWebServerApplicationContext 覆盖了它：在这里创建内嵌的 Tomcat/Jetty/Undertow 并启动。
// ServletWebServerApplicationContext.onRefresh()： createWebServer(); // → new Tomcat() → 配置端口、Servlet → tomcat.start()
为什么这时候创建：Tomcat 需要访问 Spring 的 Bean（DispatcherServlet 就是一个 Bean）。如果 Bean 还没创建完就启动 Tomcat——请求进来了但 Controller 还没实例化 → 404。所以 Tomcat 的创建在⑨，但真正接收请求是在⑫的 finishRefresh()——那时所有 Bean 都就绪了。

⑩registerListeners() — 注册事件监听器

扫描所有 ApplicationListener Bean，注册到事件广播器。同时把之前积累的 earlyApplicationEvents 全部广播出去。

⑪ finishBeanFactoryInitialization() — ★ 实例化所有单例 Bean

这是整个 refresh() 中最重的一步。调用 beanFactory.preInstantiateSingletons()——遍历所有非懒加载的单例 BeanDefinition，逐个调用 getBean() → createBean() → doCreateBean()。
// 对每个单例 Bean： createBeanInstance() → 构造器反射 newInstance() populateBean() → @Autowired 字段注入 initializeBean() → Aware 回调 → BPP.before → @PostConstruct → BPP.after → AOP 代理 // 这就是 Q01 讲的完整生命周期
为什么设计在最后几步：到这一步时，BFPP 已经完成了 BeanDefinition 的修改（⑤），BPP 已经注册好了（⑥），消息源和事件广播器就绪了（⑦⑧⑩）——创建 Bean 所需的一切基础设施都准备好了。这就是为什么 refresh() 的顺序如此严格——每一步都在为后面铺路。

⑫ finishRefresh() — 收尾：启动 WebServer + 发布事件

清除上下文缓存、初始化 LifecycleProcessor、启动 WebServer（开始接收请求）、发布 ContextRefreshedEvent。
// ServletWebServerApplicationContext.finishRefresh()： super.finishRefresh(); // 发布 ContextRefreshedEvent WebServer webServer = startWebServer(); // ★ 真正开始监听端口、接收请求
为什么 WebServer 在⑨创建但在⑫才启动：⑨创建时 Bean 还没实例化完——Controller 还不存在。⑫启动时所有单例 Bean 都已就绪——DispatcherServlet 可以正常分发请求了。把创建和启动分开，是一种常见的"初始化分离"模式。

★ 为什么是 1→2→3→…→12 这个顺序？

①②③④ 准备工作：环境初始化 → 创建空 BeanFactory → 装工具
   ↓
  ⑤ ★ 修改蓝图：BFPP 解析 @Configuration，注册 BeanDefinition
   ↓
  ⑥ ★ 注册拦截器：BPP 就位（但还没调用）
   ↓
⑦⑧⑨⑩ 补充设施：国际化、事件广播、WebServer 创建
   ↓
  ⑪ ★ 创建成品：实例化所有单例 Bean（调用 BPP）
   ↓
  ⑫ 启动服务：WebServer 开始接收请求
核心原则：每一步都在为后面的步骤铺设前提条件。⑤ 修改 BeanDefinition 必须在 ⑪ 创建 Bean 之前。⑥ 注册 BPP 必须在 ⑪ 使用 BPP 之前。⑨ 创建 Tomcat 必须在 ⑪ Controller Bean 就绪之前（否则启动时没 Controller）。⑫ 启动 Tomcat 必须在 ⑪ 之后（否则请求进来 Controller 还没实例化）。
