QUESTION 23 · 自动装配

启动流程

SpringApplication.run() 内部做了哪些核心工作？

面试时你可以这样回答

🥉 青铜

"8 个阶段：创建实例 → 准备 Environment → 创建 ApplicationContext → 准备上下文 → 加载 BeanDefinition → refresh → 回调 Runner → 发布 ReadyEvent。"

🥈 白银

"最关键的阶段是 refresh()——这是整个 Spring IOC 的核心。其中 invokeBeanFactoryPostProcessors() 解析 @Configuration、registerBeanPostProcessors() 注册 BPP、finishBeanFactoryInitialization() 实例化所有单例 Bean、onRefresh() 在 Web 环境下创建内嵌 Tomcat。在此之前，SpringApplication 先推断应用类型（SERVLET/REACTIVE/NONE），然后从 spring.factories 加载 ApplicationContextInitializer 和 ApplicationListener——这些是启动过程的扩展点。CommandLineRunner 和 ApplicationRunner 的区别：后者参数已解析（getOptionNames/getOptionValues），前者是原始 String[]。"
模板方法：整个 run() 方法是模板方法模式的经典应用——骨架固定，每个阶段都是扩展点。

🥇 王者

"SpringApplication.run() 的设计本质上是一个'启动生命周期的标准化'。和 Bean 生命周期一样，启动过程也被分解为一系列有序的阶段——每个阶段前后都有事件发布，每个阶段都支持 Initializer/Listener 扩展。这种设计让 Spring Boot 的启动过程不是一段无法定制的代码，而是一条可以插入自定义逻辑的流水线。Spring Cloud 的 Bootstrap 上下文、配置中心的配置拉取、链路追踪的 Agent 注入——所有这些都在 SpringApplication 的扩展点上实现，而不是修改 Spring 源码。这就是平台思维：提供的不只是功能，而是让生态在标准化的生命周期上构建自己的能力。"
面试官心理："启动生命周期的标准化"——和 Bean 生命周期做了呼应，体现了系统化的理解。
