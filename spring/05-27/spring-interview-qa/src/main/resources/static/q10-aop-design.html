以 Spring 源码为蓝本，逐层拆解 AOP 的四大接口 + 代理生成 + 调用链执行

1四大核心接口 —— Spring AOP 的骨架

Spring 源码中的对应

Pointcut → AspectJExpressionPointcut（execution(* com.example..*.*(..))）
Advice → MethodInterceptor（环绕）、MethodBeforeAdvice（前置）、AfterReturningAdvice（后置）
Advisor → DefaultPointcutAdvisor（把一个 Pointcut 和一个 Advice 绑在一起）
AopProxy → JdkDynamicAopProxy / CglibAopProxy
MethodInvocation → ReflectiveMethodInvocation

为什么分开设计？

同一个 Advice（比如"记录日志"）可以用不同的 Pointcut 应用到不同的包——Advice 代码一行不改。

AopProxy 和 MethodInvocation 分开，因为代理生成（JDK vs CGLIB）和代理执行（递归链）是两个独立的变化维度。换代理方式不影响拦截链逻辑。

2ReflectiveMethodInvocation — 递归调用链

这是 Spring AOP 的心脏。一个对象持有拦截器列表 + 游标，每次 proceed() 游标+1，取出下一个拦截器，把自身传进去——拦截器内部再调 proceed()，形成递归。
org.springframework.aop.framework.ReflectiveMethodInvocation
public class ReflectiveMethodInvocation implements ProxyMethodInvocation { protected final Object target; // 原始 Bean protected final Method method; // 被调用的方法 protected Object[] arguments; // 方法参数 protected final List<?> interceptors; // ★ 拦截器链 private int currentInterceptorIndex = -1; // ★ 游标 public Object proceed() throws Throwable { // 游标 +1 if (this.currentInterceptorIndex == this.interceptors.size() - 1) { // ★ 所有拦截器执行完毕 → 反射调用目标方法 return invokeJoinpoint(); } // 取出下一个拦截器，把 this 传进去——拦截器内部再调 mi.proceed() → 递归！ Object interceptor = this.interceptors.get(++this.currentInterceptorIndex); return ((MethodInterceptor) interceptor).invoke(this); } protected Object invokeJoinpoint() throws Throwable { return AopUtils.invokeJoinpointUsingReflection(this.target, this.method, this.arguments); } }
💡 面试杀手锏：为什么用递归而不用 for 循环？因为环绕通知需要在 proceed() 前后写逻辑——递归的调用栈天然表达"进入→退出"。for 循环只能线性执行，没法"执行到一半暂停，后面再回来"。

3AopProxy — 代理是怎么生成的

AopProxy 是代理的抽象。JDK 和 CGLIB 是两个实现。它们在 getProxy() 里创建代理对象，在 InvocationHandler.invoke() 里创建 ReflectiveMethodInvocation 并启动链。
org.springframework.aop.framework.JdkDynamicAopProxy
final class JdkDynamicAopProxy implements AopProxy, InvocationHandler { private final AdvisedSupport advised; // 持有：目标对象 + Advisor 列表 + 接口列表 public Object getProxy() { return Proxy.newProxyInstance(classLoader, proxiedInterfaces, this); } public Object invoke(Object proxy, Method method, Object[] args) throws Throwable { // ① 获取匹配该方法的拦截器链 List<Object> chain = this.advised.getInterceptorsAndDynamicInterceptionAdvice(method, targetClass); if (chain.isEmpty()) { return AopUtils.invokeJoinpointUsingReflection(target, method, args); // 无切面 → 直接反射 } // ② 创建调用链对象并启动 ★ MethodInvocation invocation = new ReflectiveMethodInvocation(proxy, target, method, args, targetClass, chain); return invocation.proceed(); // ← 从这里开始递归 } }
org.springframework.aop.framework.CglibAopProxy（关键区别）
// CGLIB 不使用 InvocationHandler——它用 MethodInterceptor（CGLIB 自己的接口） // 但最终同样创建 ReflectiveMethodInvocation 并调 proceed() // 区别在 getProxy()：用 Enhancer 生成子类，而非 Proxy.newProxyInstance() public Object getProxy() { Enhancer enhancer = new Enhancer(); enhancer.setSuperclass(config.getTargetClass()); // 目标类作为父类 enhancer.setInterfaces(config.getProxiedInterfaces()); enhancer.setCallback(new DynamicAdvisedInterceptor(config)); // 拦截逻辑 return enhancer.create(); } // DynamicAdvisedInterceptor.intercept() 内部和 JdkDynamicAopProxy.invoke() 几乎一样： // → 获取拦截器链 → new ReflectiveMethodInvocation → proceed()

5AnnotationAwareAspectJAutoProxyCreator — 谁在背后生成代理

这个类是 Spring AOP 的总调度——它是一个 BeanPostProcessor，在 postProcessAfterInitialization 阶段检查每个 Bean 是否匹配任何 Advisor，匹配就创建代理。
AbstractAutoProxyCreator.postProcessAfterInitialization()
public Object postProcessAfterInitialization(Object bean, String beanName) { if (bean != null) { Object cacheKey = getCacheKey(bean.getClass(), beanName); if (!this.earlyProxyReferences.contains(cacheKey)) { // ★ 检查这个 Bean 是否匹配任何 Advisor return wrapIfNecessary(bean, beanName, cacheKey); } } return bean; } protected Object wrapIfNecessary(Object bean, String beanName, Object cacheKey) { // 获取所有匹配该 Bean 的 Advisor Object[] specificInterceptors = getAdvicesAndAdvisorsForBean(bean.getClass(), beanName, null); if (specificInterceptors != DO_NOT_PROXY) { // 创建代理 Object proxy = createProxy(bean.getClass(), beanName, specificInterceptors, new SingletonTargetSource(bean)); return proxy; } return bean; }
💡 面试关键：为什么代理在 after 阶段生成、不在 before？因为 before 时 @PostConstruct 还没执行，Bean 是半成品。after 时才是成品——代理对象能完整替代原始 Bean。
