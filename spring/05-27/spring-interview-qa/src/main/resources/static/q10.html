QUESTION 10 · AOP

AOP 实现原理

Advisor / Advice / Pointcut 的关系？ReflectiveMethodInvocation 递归链如何工作？

面试时你可以这样回答

🥉 青铜

"Advisor = Advice + Pointcut。Advice 是增强逻辑（做什么），Pointcut 是切点表达式（在哪里做）。ReflectiveMethodInvocation 通过递归调用每个拦截器的 invoke() 方法实现环绕增强。"

🥈 白银

"AspectJExpressionPointcut 的匹配机制基于方法签名——不是运行时判断，而是通过方法的反射元信息（返回类型、类名、方法名、参数类型）做匹配。这是编译期表达式 + 运行时反射的组合。Advisor 把 Pointcut 和 Advice 打包，然后交给 AbstractAutoProxyCreator 在 BPP.after 阶段使用——对每个 Bean，遍历所有 Advisor，Pointcut 匹配成功就创建代理。AOP Alliance 是一个独立的标准组织，定义了 MethodInterceptor 和 MethodInvocation 接口——Spring AOP 的实现遵守这个标准，所以理论上任何 AOP Alliance 兼容的拦截器都可以在 Spring 中运行。"
为什么用 AOP Alliance？Spring 一贯的设计哲学——依赖标准接口而非具体实现。AOP Alliance 定义了 MethodInterceptor/MethodInvocation，Spring 实现它们，Guice 也可以实现它们。

🥇 王者

"Advisor/Advice/Pointcut 的三层分离是 Spring AOP 架构的灵魂——它把'在哪里做'（Pointcut）、'做什么'（Advice）、'两者的组合'（Advisor）彻底解耦。这不是过度设计——正是因为这种解耦，你才能写出一个通用的日志切面（Advice），然后通过不同的 Pointcut 表达式把它应用到不同的包——Advice 的代码一行都不需要改。Pointcut 接口的抽象也极其重要：AspectJExpressionPointcut 只是其中一种实现，你可以实现基于注解的、基于正则的、基于自定义规则的任意 Pointcut。ReflectiveMethodInvocation 的递归链则体现了另一个哲学：状态驱动的控制流。for 循环的控制流由循环变量决定（线性的），而递归的控制流由调用栈决定（树状的）——当拦截器需要决定'执行前做什么、执行后做什么'时，递归的树状结构天然匹配。这两个设计合在一起，构成了 Spring AOP 的理论基础：切面是'做什么 + 在哪里做'的声明式分离，拦截是'进入 + 退出'的控制流抽象。"
面试官心理："声明式分离 + 控制流抽象"——两个短语概括了整个 AOP 的理论框架。
