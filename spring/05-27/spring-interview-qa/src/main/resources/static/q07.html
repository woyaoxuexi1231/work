QUESTION 07 · AOP

代理选择：JDK vs CGLIB

各自的优缺点？Spring Boot 2.x 为什么默认改为 CGLIB？

面试时你可以这样回答

🥉 青铜

"JDK 动态代理基于接口 + Proxy.newProxyInstance()，CGLIB 基于继承生成子类。Spring Boot 2.x 默认改为 CGLIB 因为很多 Service 没有接口，1.x 的 JDK 代理会导致 @Transactional 静默失效。"

🥈 白银

"两种代理的本质区别不在于性能——JDK 17+ 的反射优化已经让性能差距可以忽略。真正的区别在于对代理对象的类型契约的影响。JDK 代理生成的代理类实现了目标接口，但和原始类之间没有继承关系——你不能把代理转型为原始类。CGLIB 代理生成的代理类是原始类的子类——你可以转型，但不能代理 final 类和方法。Spring Boot 2.x 默认改 CGLIB 的核心原因是'最小惊讶原则'——开发者加了 @Transactional 就期望它生效，不在乎底层用的是哪种代理。1.x 时代因为 Service 没接口导致注解静默失效的 bug 太多了。"
关键：AopProxy 接口有两个实现——JdkDynamicAopProxy 和 CglibAopProxy，Spring 通过 DefaultAopProxyFactory.createAopProxy() 根据配置选择。

🥇 王者

"代理选择之争本质上不是技术之争，是'显式 vs 隐式'的设计哲学之争。JDK 代理是显式的——你必须定义接口，这强迫你做面向接口编程，但也意味着多写代码。CGLIB 是隐式的——你不需要接口，框架帮你搞定一切，但如果类被标记为 final 就会静默失败。Spring Boot 从 JDK 转向 CGLIB 默认值，表面上是'方便开发者'，深层是对现代 Java 开发模式的适应：微服务时代，一个 Service 往往只有一个实现，定义接口变成了一种形式主义。CGLIB 默认值让 Spring 对齐了这种现实。但代价也存在：final 类和方法无法代理，这迫使 Spring 生态形成了'不要随意 final'的约定——一个框架的设计选择反过来塑造了整个社区的编码习惯。"
面试官心理：看透表面性能之争，直指"显式 vs 隐式"的哲学分歧——这才是架构师的视角。
