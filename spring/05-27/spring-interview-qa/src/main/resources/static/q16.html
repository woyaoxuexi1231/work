📋 Q16 · MVC 请求处理流程

不考组件名字——考你排查过什么实际问题。每个答案都是真实开发场景。

面
你遇到过 @RequestBody 突然读不到数据的情况吗？是什么原因？怎么查出来的？
我
遇到过。那是一个请求日志 Filter 引发的。

🛑 场景还原：运维要求把所有请求的 body 记到日志里。我写了一个 Filter，在 doFilter 里调 request.getReader().lines().collect() 拿到 body 内容，打了一行 log，然后 chain.doFilter 放行。日志是有了——但后面所有 Controller 的 @RequestBody 全部拿到 null。
✅ 排查和修复：打断点发现在 Filter 之后的 HttpMessageConverter 里，request.getInputStream().read() 直接返回 -1。原因是 HttpServletRequest 的 InputStream 只能读一次——Filter 读完了，流到了末尾，Spring 再读就是空的。

修复很简单：把 request 用 new ContentCachingRequestWrapper(request) 包一层，这个包装类把 body 缓存到内部 byte[]。但这有个代价——它会先把整个 body 读进内存。如果上传的是大文件，内存直接炸。所以最后改成只对 Content-Type 是 application/json 且 Content-Length < 4KB 的请求才记录 body，其余的只记录 Content-Length。
如果面试官追问"为什么不用 Interceptor 而要写在 Filter 里"，答案也很直接——Interceptor 在 DispatcherServlet 之后才执行。那时候如果 body 已经被某个 ArgumentResolver 或者 Converter 消费了，Interceptor 同样读不到。拿到最原始的请求数据必须在 Filter 层。
面
你写了一个 @ControllerAdvice 做全局异常处理，但发现有几种异常就是捕获不到。是哪几种？为什么捕获不到？
我
🛑 两种捕获不到的情况：

第一种：异常在进入 DispatcherServlet 之前就抛了。比如 Filter 里抛了异常——这时候请求还没到 DispatcherServlet，@ControllerAdvice 的 ExceptionHandler 根本不会被触发。返回的是一个 Tomcat 默认的 500 错误页。解决方案是在 Filter 里自己 try-catch，手动写 response。

第二种：响应已经 commit 了才抛异常。比如 Controller 方法返回了一个大文件，OutputStream 写了一半，磁盘满了抛 IOException。但此时 HTTP 响应头已经发出去了（状态码 200 已经在网络上了），你没法再把 200 改成 500。@ControllerAdvice 能捕获异常，但修改响应状态码的代码会抛另一个异常——因为 response 已经 committed。这种情况只能靠日志告警，没法给客户端返回正确的错误码。
✅ 实际项目里我是这样做的：Filter 层的异常直接用 Filter 内部的 try-catch 处理，返回统一 JSON 格式的错误。Controller 层的异常靠 @ControllerAdvice。响应已 commit 的情况——加监控，日志里搜 "response committed" + IOException，靠告警而不是靠代码。
面
你做了前后端分离，Controller 全部返回 JSON。但有一天前端突然说某个接口返回了 XML。怎么排查？
我
🛑 场景：前端 fetch('/users/1') 突然收到 <xml> 而不是 JSON。排查发现前端新来的同事在请求里加了一个 header：Accept: application/xml。浏览器默认的 Accept 是 */*，但加了之后就明确告诉服务端"我想要 XML"。
Spring MVC 有个机制叫内容协商（Content Negotiation）——在 ReturnValueHandler 选 HttpMessageConverter 时，不仅看返回值类型，还看请求的 Accept header。如果 MappingJackson2HttpMessageConverter 发现 Accept 里没有 application/json 但 MappingJackson2XmlHttpMessageConverter 发现 Accept 里有 application/xml——后者就胜出了。如果 classpath 里恰好有 jackson-dataformat-xml 的 jar（可能是一个不相关的依赖带进来的），就会自动注册一个 XML Converter。
✅ 修复：两种方式。治本的方式——在 Spring Boot 配置里关掉内容协商：spring.mvc.contentnegotiation.favor-parameter=false。治标的方式——Controller 方法上加 @RequestMapping(produces = "application/json")，明确告诉 Spring"这个方法只产 JSON"。实际项目我两个都加了：全局关掉内容协商避免意外，关键接口加 produces 做防御。
面
你做了一个全局的 ResponseBodyAdvice 把返回值包成 {code, message, data}。但后来有个接口需要返回原始格式——比如要给第三方回调，他们要求固定的格式。你怎么让这个接口跳过包装？
我
ResponseBodyAdvice 有一个 supports() 方法——它在每次写响应之前被调用，返回 true 才会执行 beforeBodyWrite()。这个 supports 方法可以拿到 MethodParameter——也就是 Controller 方法的反射信息。

✅ 做法：自定义一个注解 @NoWrapper，标在不需要包装的 Controller 方法上。然后在 supports() 里检查方法有没有这个注解：
return !returnType.hasMethodAnnotation(NoWrapper.class);

有 @NoWrapper → supports 返回 false → beforeBodyWrite 不执行 → 返回原始数据。没有 @NoWrapper → supports 返回 true → 走包装逻辑。

比写死 URL 白名单好。用注解的方式，加新接口的人自己决定要不要包装——不用每次都来改全局配置。
面
你的 Controller 接收 @RequestBody User user。但有一天前端传的 JSON 里多了一个字段 "extraField"，后端直接报 400 了。为什么会报 400？你倾向怎么处理——拒绝还是忽略？
我
🛑 原因：Jackson 默认配置里 FAIL_ON_UNKNOWN_PROPERTIES 是 true——JSON 里出现了 Java 类里没有的字段，Jackson 抛 UnrecognizedPropertyException，Spring 的 HttpMessageConverter 把这个异常往上抛，最终变成 400。
✅ 我倾向全局关掉——忽略未知字段：
spring.jackson.deserialization.fail-on-unknown-properties=false

为什么选忽略而不是拒绝？前后端版本不一致是常态。后端先上线，加了新字段 extraField；前端还没更新，多传了这个字段。如果拒绝，旧版本前端直接全线 400——这是不可接受的。忽略未知字段让 API 向后兼容——前端加字段不影响后端，后端加字段不影响前端。这就是鲁棒性原则："对自己发送的东西要严格，对接收的东西要宽容。"
面
你有没有遇到过"加了 @ResponseBody 但返回值还是被 ViewResolver 处理了"的情况？
我
遇到过。那次是 Controller 类上写了 @Controller，方法上写了 @ResponseBody，返回了一个 String。

代码是这样：@ResponseBody public String hello() { return "hello"; }
期望返回纯文本 hello。但实际上 Spring 完美地返回了 hello 纯文本——没有走 ViewResolver。因为 RequestResponseBodyMethodProcessor 检测到了 @ResponseBody 注解，它处理完后标记了 mavContainer.setRequestHandled(true)，后面的 ViewNameMethodReturnValueHandler 看到标记直接跳过。

真正出问题的是另一种情况：方法上没写 @ResponseBody，返回 String。public String hello() { return "hello"; }——Spring 把 "hello" 当成了视图名，交给 ViewResolver，去找 /WEB-INF/views/hello.jsp。找不到 → 404。

这就是 @RestController 的价值——整个类的所有方法默认都是 @ResponseBody。不会出现"忘了加注解，返回值被当成视图名"的低级错误。
