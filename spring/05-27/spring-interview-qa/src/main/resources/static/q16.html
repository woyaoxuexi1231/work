QUESTION 16 · MVC

请求处理流程

从 DispatcherServlet 到返回结果经过了哪些组件？

面试时你可以这样回答

🥉 青铜

"HandlerMapping 根据 URL 找 Handler，HandlerAdapter 适配不同 Handler 类型，ViewResolver 解析视图名。DispatcherServlet 做总调度。"

🥈 白银

"doDispatch() 的核心流程：HandlerMapping.getHandler() 返回 HandlerExecutionChain（Handler + Interceptors）→ HandlerAdapter.handle() 调用 Controller → 参数通过 HandlerMethodArgumentResolver 解析 → 返回值通过 HandlerMethodReturnValueHandler 处理 → @ResponseBody 走 HttpMessageConverter 序列化，String 走 ViewResolver 渲染。HandlerAdapter 是适配器模式——早期 Controller 需要实现 Controller 接口，后来引入 @RequestMapping 注解驱动，两种 Handler 的调用方式不同，Adapter 解耦了这个差异。"
扩展点：想自定义参数解析？实现 HandlerMethodArgumentResolver。想自定义返回值处理？实现 HandlerMethodReturnValueHandler。开闭原则的教科书。

🥇 王者

"DispatcherServlet 的设计是一个策略模式的大集合——HandlerMapping、HandlerAdapter、ViewResolver、HandlerExceptionResolver 全部是策略接口。这种设计让 Spring MVC 从一个'MVC 框架'变成了一个'MVC 扩展平台'——你想换 URL 映射策略？换一个 HandlerMapping 实现。想支持新的视图技术？实现 ViewResolver。想自定义参数绑定？实现 ArgumentResolver。Spring 在 DispatcherServlet 中只做一件事：组装这些策略并按顺序调用。这就是为什么 Spring MVC 能二十年来不被淘汰——它从一开始就不是'某个人的 MVC 设计'，而是'MVC 模式的可插拔实现'。每一个组件都可替换，但整体流程保持稳定。理解了这个，你就理解了为什么 Spring 的文档中到处都是'strategy interface'这个词。"
面试官心理："从 MVC 框架到 MVC 扩展平台"、"策略模式大集合"——这是对 Spring MVC 架构最精准的概括。
