## 五、Spring Boot 自动装配

21. **自动装配原理**  
    `@SpringBootApplication` 包含哪三个注解？`@EnableAutoConfiguration` 是如何通过 `spring.factories` 和 `AutoConfigurationImportSelector` 加载自动配置类的？条件注解 `@ConditionalOnClass`、`@ConditionalOnMissingBean` 的工作原理？

`@SpringBootApplication` 是一个组合注解，它主要包含三个核心注解：

1. **`@SpringBootConfiguration`**  
   本质上是 `@Configuration` 的派生注解，标记当前类为配置类，并允许在其中定义 `@Bean` 方法。

2. **`@EnableAutoConfiguration`**  
   开启 Spring Boot 的自动配置机制，根据类路径中的 jar 包、已定义的 Bean 以及各种条件注解，智能地配置 Spring 应用。

3. **`@ComponentScan`**  
   启用组件扫描，默认会扫描当前类所在包及其子包下的 `@Component`、`@Service`、`@Repository`、`@Controller` 等组件。

---

### `@EnableAutoConfiguration` 如何加载自动配置类

`@EnableAutoConfiguration` 本身又通过 `@Import` 引入了 `AutoConfigurationImportSelector`，核心加载流程如下：

1. **`AutoConfigurationImportSelector` 被 `@Import` 导入**
   `@EnableAutoConfiguration` 的定义：
   ```java
   @Import(AutoConfigurationImportSelector.class)
   public @interface EnableAutoConfiguration { ... }
   ```
   `AutoConfigurationImportSelector` 实现了 `DeferredImportSelector`，其 `selectImports` 方法会在所有配置类解析完成后、`@Conditional` 处理之前被调用。

2. **读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件**
   - 在 Spring Boot 2.7 之前，自动配置类全限定名配置在 `META-INF/spring.factories` 中，键为 `org.springframework.boot.autoconfigure.EnableAutoConfiguration`。
   - 从 Spring Boot 2.7 开始，推荐使用 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件，每行一个自动配置类的全限定名。  
     `AutoConfigurationImportSelector` 会通过 `SpringFactoriesLoader` 或 `ImportCandidates` 读取这些文件，加载所有候选的自动配置类。  
   - 加载时会进行去重和排除（根据 `spring.autoconfigure.exclude` 属性）。

3. **过滤条件注解**
   加载到的自动配置类上通常标有 `@Configuration` 以及各种条件注解（如 `@ConditionalOnClass`、`@ConditionalOnBean` 等）。Spring 容器会在注册这些配置类时评估条件，只有满足条件的配置类及其内部定义的 `@Bean` 才会生效。

4. **执行时机**
   由于实现了 `DeferredImportSelector`，自动配置类的导入被延迟到所有用户自定义 Bean 注册之后，这样 `@ConditionalOnMissingBean` 等条件就能基于用户已有的 Bean 做出正确判断。

---

### 条件注解工作原理

条件注解由 `SpringBootCondition` 及其子类实现，核心机制是 Spring Framework 的 `@Conditional` 配合 `Condition` 接口。以你提到的两个注解为例：

#### `@ConditionalOnClass`
- **作用**：仅当 classpath 中存在指定的类时，条件才匹配。
- **对应条件类**：`OnClassCondition`，继承自 `SpringBootCondition`。
- **评估逻辑**：
  1. 读取 `@ConditionalOnClass` 注解中指定的类名（`value`）或类引用（`name`）。
  2. 使用 `ClassUtils.isPresent()` 或通过类加载器尝试加载这些类。
  3. 如果所有指定类都能加载成功，则条件通过；若任一类缺失（抛出 `ClassNotFoundException` 或 `NoClassDefFoundError`），条件不通过。
- **示例**：
  ```java
  @ConditionalOnClass(name = "org.springframework.web.servlet.DispatcherServlet")
  ```
  仅在 Spring MVC 存在时生效。

#### `@ConditionalOnMissingBean`
- **作用**：当 Spring 容器中不存在指定类型的 Bean 时，条件才匹配。
- **对应条件类**：`OnBeanCondition`，同样继承自 `SpringBootCondition`。
- **评估逻辑**：
  1. 在 `ConfigurationClassPostProcessor` 处理配置类、准备注册新 Bean 之前，`OnBeanCondition` 会检查 `BeanFactory`。
  2. 读取 `@ConditionalOnMissingBean` 注解的 `value`（Class 数组）、`name`（Bean 名称数组）、`annotation`（Bean 上必须有这些注解）等属性。
  3. 从当前的 `BeanDefinitionRegistry` 和单例池中查找匹配的 Bean：
     - 按类型匹配：检查是否有类型兼容的 BeanDefinition 或已实例化的单例。
     - 按名称匹配：检查是否有指定名称的 Bean。
     - 按注解匹配：检查是否存在标注了特定注解的 Bean。
  4. 如果未找到任何匹配的 Bean，条件通过；否则条件不通过，该配置类或 `@Bean` 方法将被跳过。
- **重要细节**：为防止注册顺序影响判断，`OnBeanCondition` 会将当前正在处理的配置类注册为“正在处理中”，避免自己认为自己是缺失的 Bean 而通过条件。

这两个注解都可以使用 `matchIfMissing` 属性（仅限 `@ConditionalOnMissingBean` 无此属性，而 `@ConditionalOnClass` 也没有，但部分条件有），来决定当所有条件都无法确切判断时的默认行为。

---

通过这种“配置类清单 + 条件过滤”的组合，Spring Boot 实现了“按需自动配置”：只有引入了相关 starter 并且用户没有覆盖默认 Bean 时，对应的自动配置才会生效。
