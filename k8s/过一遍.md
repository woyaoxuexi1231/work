hulei@k8s-master:~/k8s$ kubectl get nodes -o wide

### 🔍 指令解析

`kubectl get nodes -o wide` 是 Kubernetes 集群运维中最基础的**节点级巡检命令**。

- `kubectl get nodes`：获取集群中所有 Node（节点）资源的摘要信息。Node 是 K8s 的工作机器，可以是物理机或虚拟机。
- `-o wide`：以“宽输出”格式显示，在默认列的基础上额外展示 **INTERNAL-IP、EXTERNAL-IP、OS-IMAGE、KERNEL-VERSION、CONTAINER-RUNTIME** 这 5 列关键底层信息。如果不加 `-o wide`，你只能看到 NAME、STATUS、ROLES、AGE、VERSION 这 5 列。

**使用场景**：集群刚搭建完成时验证节点是否就绪、日常巡检节点健康状态、排查网络/运行时版本不一致问题、确认节点 IP 规划是否正确。

------

### 📊 结果详解

你的输出显示集群有 **3 个节点（1 Master + 2 Worker）**，全部处于健康状态。下面逐列拆解每个字段的含义和运维价值：

| 列名                  | 你的值                             | 含义与运维意义                                               |
| --------------------- | ---------------------------------- | ------------------------------------------------------------ |
| **NAME**              | k8s-master / k8s-node1 / k8s-node2 | 节点在 K8s 中的唯一标识符。**注意**：K8s 通过主机名识别节点，修改主机名后必须重新加入集群。命名规范建议体现角色和序号，方便快速定位。 |
| **STATUS**            | Ready                              | 节点健康状态。`Ready` = kubelet 正常运行且已向 API Server 汇报心跳。若变为 `NotReady`，说明 kubelet 挂了、容器运行时异常、或网络插件未就绪，需立即排查。这是最核心的告警指标。 |
| **ROLES**             | control-plane / `<none>`           | 节点角色。`control-plane` = Master 节点，运行 apiserver/etcd/scheduler/controller-manager 等控制平面组件；`<none>` = Worker 节点，只运行业务 Pod 和 kube-proxy/calico 等数据平面组件。**生产环境严禁在 Master 上调度业务 Pod**。 |
| **AGE**               | 19m / 17m                          | 节点加入集群的时长。Master 比 Worker 早 2 分钟，符合先初始化控制平面再加入工作节点的正常流程。若某节点 AGE 远小于其他节点，说明它可能是刚恢复或刚扩容的。 |
| **VERSION**           | v1.28.15                           | kubelet 版本号。**三节点版本完全一致，这是最佳实践**。K8s 允许 Master 比 Worker 高一个小版本，但绝不允许 Worker 比 Master 高。版本不一致可能导致 API 兼容性问题。 |
| **INTERNAL-IP**       | 192.168.3.100~102                  | 节点在集群内部通信使用的 IP。kubelet、Pod、Service 之间的流量都走这个 IP。**三节点同属 192.168.3.0/24 网段**，说明之前跨网段问题已彻底解决。若此处出现多网卡选错 IP，会导致 Pod 无法跨节点通信。 |
| **EXTERNAL-IP**       | `<none>`                           | 节点的外部访问 IP（如云厂商公网 IP）。本地自建集群通常为 `<none>`。若在云上部署，此字段会自动填充弹性公网 IP，用于从集群外部直接访问节点。 |
| **OS-IMAGE**          | Ubuntu 22.04.5 LTS                 | 操作系统镜像。**三节点 OS 完全一致**，避免了因内核参数、systemd 版本差异导致的诡异问题。生产环境强烈建议所有节点使用相同 OS 版本。 |
| **KERNEL-VERSION**    | 5.15.0-179-generic                 | Linux 内核版本。containerd 和 Calico 都依赖特定内核特性（如 cgroupv2、eBPF）。内核版本过低可能导致网络插件或存储驱动异常。5.15 是 Ubuntu 22.04 的 HWE 内核，对 K8s 1.28 兼容性良好。 |
| **CONTAINER-RUNTIME** | containerd://1.7.27                | 实际运行容器的引擎及版本。正如之前讨论的，这里显示的是 **containerd 而非 Docker**，印证了 K8s 1.24+ 移除 dockershim 后的标准架构。版本 1.7.27 是 containerd 的稳定版，与 K8s 1.28 官方兼容矩阵匹配。 |

### 💡 从这份输出可以得出的结论

1. **集群健康**：3/3 节点 Ready，无异常。
2. **环境一致性极佳**：OS、内核、K8s 版本、容器运行时四者完全统一，这是稳定运行的基础。
3. **网络规划正确**：内网 IP 连续且同网段，无 External IP 干扰。
4. **架构标准**：1 控制平面 + 2 工作节点，角色分离清晰。

> ⚠️ **一个潜在风险提示**：当前只有 **1 个 Master 节点**。这意味着控制平面是单点故障——如果 k8s-master 宕机，整个集群将无法创建/删除/更新任何资源（已运行的 Pod 不受影响）。若用于生产，建议后续扩展为 3 Master 高可用架构。

### 🔍 指令解析

`kubectl get pods -A` 是 Kubernetes 的**全局 Pod 状态巡检命令**。

- `kubectl get pods`：列出 Pod 资源。Pod 是 K8s 最小调度单元，封装了一个或多个容器，是应用和系统组件的实际载体。
- `-A`（`--all-namespaces`）：查询**所有命名空间**。K8s 用 Namespace 做逻辑隔离，默认只查 `default` 命名空间。加 `-A` 才能看到 `kube-system` 等系统命名空间中的核心组件。

**使用场景**：集群搭建后验证系统组件是否全部就绪、日常排查故障时快速定位异常 Pod、确认部署的应用是否正常启动。**这是 K8s 运维中使用频率最高的命令之一**。

------

### 📊 结果详解

你的输出显示 `kube-system` 命名空间下共 13 个 Pod，涵盖了网络、DNS、控制平面、代理四大核心子系统。逐列拆解如下：

#### 列字段含义

| 列名          | 含义                | 运维关注点                                                   |
| ------------- | ------------------- | ------------------------------------------------------------ |
| **NAMESPACE** | Pod 所属的逻辑分组  | `kube-system` = K8s 自身核心组件所在命名空间，切勿手动删除或修改其中的 Pod |
| **NAME**      | Pod 唯一名称        | 由"组件名+副本集哈希+随机后缀"组成（如 `coredns-6554b8b87f-24bk6`），重建后名称会变，但前缀不变 |
| **READY**     | 就绪容器数/总容器数 | `1/1` = 完全就绪；`0/1` = 容器在运行但未通过就绪探针，**不能接收流量** |
| **STATUS**    | Pod 生命周期阶段    | `Running` = 容器已启动；还需结合 READY 判断是否真正可用      |
| **RESTARTS**  | 容器重启次数        | 0 = 稳定；>0 需警惕，频繁重启通常意味着 OOM、配置错误、健康检查失败 |
| **AGE**       | Pod 创建时长        | 与节点 AGE 对比可判断组件是否在节点加入后立即启动            |

#### 各组件功能与状态分析

| 组件                        | 数量 | 当前状态          | 作用与运维意义                                               |
| --------------------------- | ---- | ----------------- | ------------------------------------------------------------ |
| **calico-kube-controllers** | 1    | ✅ 1/1 Running     | Calico 的中央控制器，负责同步 NetworkPolicy、IPAM 地址分配、BGP 路由宣告。单副本即可，挂了会导致新 Pod 无法分配 IP |
| **calico-node**             | 3    | ⚠️ 1个0/1, 2个1/1  | 每个节点一个 DaemonSet Pod，负责本节点的虚拟网卡创建、iptables/IPVS 规则写入、BGP 邻居建立。**`calico-node-5m6q4` 仍为 0/1**，说明该节点（大概率是 k8s-master）的网络插件尚未完全就绪，可能正在等待 BGP 收敛或 IPAM 初始化 |
| **coredns**                 | 2    | ✅ 1/1 Running     | 集群内部 DNS 服务器，所有 Service 域名解析依赖它。2 副本保证高可用，若全挂则 Pod 间无法通过服务名通信 |
| **etcd-k8s-master**         | 1    | ✅ 1/1, RESTARTS=3 | 集群的分布式数据库，存储所有配置和状态数据。**重启 3 次偏高**，可能是启动初期 etcd 集群未形成法定人数、磁盘 I/O 慢、或内存不足导致。当前已稳定，但建议后续检查日志确认根因 |

| **kube-proxy**              | 3    | ✅ 1/1 Running     | 每个节点一个，维护 Service → Pod 的转发规则（iptables/IPVS）。3 副本对应 3 节点，全部正常 |

#### 💡 关键结论与建议

1. **集群整体健康**：除 1 个 calico-node 外，所有核心组件均已就绪且稳定运行。

2. **控制平面重启历史**：apiserver / controller-manager / scheduler / etcd 的重启次数分别为 3/5/6/3，这是**集群冷启动时的典型现象**——各组件按依赖顺序启动，先启动的组件在后端未就绪时会反复崩溃重试，直到整个链路打通。只要当前 STATUS=Running 且 READY=1/1，就无需担心。
