## 判断标准

你判断“代码演示”还是“操作文档”时，遵循以下原则：

- **适合代码演示的场景**：并发/多线程行为、锁的升级/降级、事务传播与回滚、内存可见性、缓存击穿/穿透的复现、算法实现、数据结构的使用对比、设计模式的应用效果等。这些场景通过代码运行后，能在控制台、日志或HTTP响应中直接看到现象差异。

- **适合操作文档的场景**：JVM参数调优、GC策略对比、数据库索引优化、MySQL主从复制验证、Redis集群配置与故障转移、中间件安装与性能压测、监控工具的使用等。这些场景需要你亲自执行命令、修改配置、观察系统指标变化。

- **可以混合**：如果一道题既有代码层面的核心逻辑，又有需要操作观察的运维层面，可以同时给出代码+操作文档，但需说明先后顺序。

#### 若为代码演示：

**项目骨架**：
- 基于 **Spring Boot + JDK 1.8 + Maven**。
- 只引入与此题核心原理相关的最小依赖。
- 目录结构清晰：Controller（演示入口）、Service（核心逻辑）、Entity/Config（辅助）。
- 所有演示通过 HTTP 接口触发，挂载在 `http://localhost:8080/` 下。
- 数据库使用 MySQL：`192.168.3.100:3306`，`root`/`123456`。
- Redis 环境（如需）：
  - 单机：`192.168.3.100:6379`，密码 `123456`
  - 哨兵：主6379，从6380/6381，哨兵26379/26380/26381，密码 `123456`
  - 集群：7.2.5，端口7000-7005，总线17000-17005，密码 `123456`

**演示场景设计**：
- 设计一个能“赤裸裸”展现此题原理的业务场景，说明这个场景的巧妙之处。

**核心代码**：
- 提供关键类的完整代码，用 `//【重点】` 标注体现原理的行。
- 接口返回的结果中，必须包含能直观观察到原理生效的字段或数据。

**运行指南**：
- 给出具体的接口调用方式（curl 命令或浏览器地址）。
- 设计“对比实验”：先调用一个“错误/反面”版本看到问题现象，再调用“正确”版本看到原理如何解决问题。
- 告诉我应该关注返回结果或日志中的什么信息。

**原理点睛**：
- 结合代码中出现的现象，点出此题涉及的底层原理，并指出这道面试题的回答中，哪段理论刚好被这段代码“实锤”了。

---

#### 若为操作文档：

**操作背景**：
- 用你叙事结构中的“苦难时代”风格，简要说明如果不做这个操作，系统会怎样。

**操作目标**：
- 一句话说明本次操作要达成的效果。

**环境准备**：
- 列出需要的工具、命令、前置条件。

**详细操作步骤**：
1. 每一步包含具体命令或配置修改内容。
2. 每一步说明预期看到的输出或现象。
3. 如果某一步是在验证原理，用 `【原理验证点】` 标注并解释。

**结果对比**：
- 给出操作前和操作后的系统行为差异，最好能用表格对比。

**原理点睛**：
- 点出这些操作步骤背后验证的核心原理，并指出这道面试题的回答中，哪段理论刚好被这个操作“实锤”了。

**面试要诀**：
- 一句话告诉我，如果在面试中被问到这道题，我提到自己做过这个操作，面试官会怎么刮目相看。

- 你的实战内容必须与之前讲解的技术点和面试答案**一脉相承**，体现“理论指导实践，实践验证理论”。
- 代码要直接可用，操作步骤要直接可执行，不允许有“请自行查资料补充”之类的敷衍。
- 语言风格保持你一贯的穿透力，用比喻和故事让操作指南也读起来津津有味。
