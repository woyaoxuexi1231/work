### RabbitMQ 知识点

**1. 核心概念与 AMQP 模型**
- Broker（消息代理）
- Virtual Host（vhost）：逻辑隔离单元
- Connection（连接）：长连接 TCP
- Channel（信道）：轻量级虚拟连接，多路复用
- Producer / Publisher（生产者）
- Consumer（消费者）
- Exchange（交换机）：接收消息并路由到队列
- Queue（队列）：存储消息
- Binding（绑定）：交换机与队列之间的路由规则
- Routing Key（路由键）：生产者发布消息时指定，交换机用它匹配绑定键
- 消息组成：Properties（属性）、Headers（头）、Body（有效载荷）

**2. 交换机类型与路由**
- Direct Exchange：精确匹配路由键
- Topic Exchange：基于模式（`*`、`#`）匹配路由键
- Fanout Exchange：广播到所有绑定队列
- Headers Exchange：基于消息头属性匹配（`x-match: all/any`）
- Default (nameless) Exchange：每个队列自动绑定的空字符串交换机，路由键等于队列名
- Dead Letter Exchange：处理死信的交换机（详见死信部分）

**3. 队列属性与参数**
- 声明队列：`queue.declare`
- 属性：
    - `durable`：队列持久化（服务重启后仍存在）
    - `exclusive`：排他（仅限声明连接使用，连接断开删除）
    - `auto-delete`：自动删除（最后一个消费者取消后删除）
- 可选参数（`x-arguments`）：
    - `x-message-ttl`：队列级别消息 TTL（毫秒）
    - `x-expires`：队列空闲自动删除时间
    - `x-max-length`：最大消息数量限制（溢出行为：`drop-head` 或 `reject-publish`）
    - `x-max-length-bytes`：最大队列总大小（字节）
    - `x-dead-letter-exchange`：死信交换机
    - `x-dead-letter-routing-key`：死信路由键
    - `x-max-priority`：队列最大优先级（0-255），创建后不可更改
    - `x-queue-mode`：队列模式（`default`、`lazy`）
    - `x-single-active-consumer`：单一活跃消费者（仅一个消费者接收消息，其他作为备份）
    - `x-queue-type`：队列类型（经典 `classic`、仲裁 `quorum`、流 `stream`）

**4. 消息属性（AMQP 0-9-1 Properties）**
- `delivery-mode`：持久化（1 非持久，2 持久）
- `content-type`：消息内容类型（如 `application/json`）
- `reply-to`：回复队列名称
- `correlation-id`：关联 ID，用于 RPC 模式
- `expiration`：消息级别 TTL（毫秒）
- `message-id`：消息唯一标识
- `timestamp`：时间戳
- `type`：消息类型标识
- `user-id`：发布者用户 ID（需验证）
- `app-id`：应用程序 ID
- `headers`：自定义头部（可参与 Header Exchange 路由）

**5. 发布与消费**
- 发布消息：`basic.publish`，指定交换机、路由键、mandatory/ immediate 标志（已废弃）
- `mandatory` 标志：若消息无法路由到队列，返回给生产者（需绑定 ReturnListener）
- 消费消息：`basic.consume`（推模式，`Consumer` 接口）、`basic.get`（拉模式，单条获取）
- 消费者标签（consumer tag）：取消订阅时的标识
- `prefetch`（QoS）：`basic.qos` 限制未确认消息数量（prefetch count、prefetch size、global）
- 消费者优先级（`x-priority`）：高优先级消费者优先获得消息（需手动确认模式）

**6. 消息确认机制**
- 生产者确认（Publisher Confirms）：
    - 信道设置为确认模式：`confirm.select`
    - 异步确认：`ConfirmListener` 处理 `handleAck` / `handleNack`
    - 批量确认与序列化号
- 消费者确认：
    - 自动确认（`autoAck = true`）：消息发送后立即视为确认
    - 手动确认：
        - `basic.ack`：单条或多条确认（`multiple` 参数）
        - `basic.nack`：否定确认，可批量，支持 `requeue` 参数
        - `basic.reject`：单条拒绝，支持 `requeue`
    - `requeue` 行为：重新入队则重回队头，可能造成无限重试
- 事务（性能较低，不推荐）：`txSelect`、`txCommit`、`txRollback`

**7. 死信队列与延迟队列**
- 死信（Dead Letter）产生的三种情况：
    - 消息被消费者拒绝（`basic.reject` / `basic.nack`，且 `requeue=false`）
    - 消息 TTL 过期
    - 队列达到最大长度（`x-max-length`）或最大字节数
- 死信交换机与死信路由键配置：`x-dead-letter-exchange`、`x-dead-letter-routing-key`
- 延迟队列实现：
    - 利用死信：创建带 TTL 的队列（或消息 TTL），绑定死信交换机实现延迟投递
    - 插件 `rabbitmq_delayed_message_exchange`：支持在交换机级别延迟投递（`x-delay` 消息头）

**8. 持久化与可靠性**
- 队列持久化（`durable`）
- 消息持久化（`delivery-mode=2`）
- 持久化组合：持久队列 + 持久消息 + 发布确认 → 不丢消息
- 镜像队列（Classic Mirror Queue）：主从节点数据同步，用 `ha-mode`、`ha-params` 等策略控制
- 仲裁队列（Quorum Queue）：基于 Raft 协议，强一致，自动故障转移，替代镜像队列
- 流（Stream）：不可变、持久化的仅追加日志，支持消费者偏移量跟踪和重复消费

**9. 集群**
- 节点类型：磁盘节点（disc）、内存节点（ram），元数据在磁盘节点持久化
- 集群形成：`rabbitmqctl join_cluster`
- 节点发现：经典配置（`cluster_nodes` 参数）、`rabbitmq-peer-discovery` 插件（Kubernetes、Consul、etcd、DNS）
- 队列位置：经典队列元数据所有节点共享，消息内容仅存在于声明队列的节点（单主镜像）；仲裁队列自动分布
- 集群通信：Erlang 分布式协议，Cookie 同步
- 集群节点重启与升级
- 网络分区：处理策略 `cluster_partition_handling`（`ignore`、`pause_minority`、`pause-if-all-down`、`autoheal`）

**10. 高可用与故障转移**
- 镜像队列策略：`ha-mode`（`all`、`exactly`、`nodes`）、`ha-params`、`ha-sync-mode`（`automatic` / `manual`）
- 镜像主节点故障：提升最老从节点为主
- 仲裁队列：自动选主，多数派写入确认，数据安全
- 单一活跃消费者（SAC）：`x-single-active-consumer` 实现消费者侧高可用，活跃消费者断开后备份消费者接管
- 客户端连接故障转移：连接恢复、重试、心跳配置
- 代理不可达时的 `connection.blocked` 通知（资源告警）

**11. 联邦（Federation）**
- 联邦交换机（Federation Exchange）：把上游交换机上的消息复制到下游
- 联邦队列（Federation Queue）：拉取上游队列的消息到下游，消费者可跨数据中心
- 联邦链接（Federation Link）：配置上游 URI、prefetch、重连等
- 适用场景：跨 WAN、跨机房消息同步、分层结构
- 联邦策略（federation-upstream-set）管理

**12. Shovel**
- 动态 Shovel：通过 Runtime Parameters 定义（管理界面或命令行）
- 静态 Shovel：在 `rabbitmq.conf` 中配置
- Shovel 源与目标：可以是 AMQP 地址（队列或交换机），可跨 vhost、跨集群
- 消息转发模式：on publish、on confirm 等
- 自动重连与确认机制

**13. 插件系统**
- 常用内置插件：
    - `rabbitmq_management`：Web UI 与 HTTP API
    - `rabbitmq_management_agent`：管理代理
    - `rabbitmq_mqtt`：MQTT 协议支持
    - `rabbitmq_stomp`：STOMP 协议支持
    - `rabbitmq_web_mqtt` / `rabbitmq_web_stomp`：WebSocket 桥接
    - `rabbitmq_delayed_message_exchange`：延迟消息交换机
    - `rabbitmq_sharding`：分片队列（自动将同一队列名映射到多个内部队列）
    - `rabbitmq_federation` / `rabbitmq_federation_management`
    - `rabbitmq_shovel` / `rabbitmq_shovel_management`
    - `rabbitmq_peer_discovery_k8s` / `consul` / `etcd` 等
    - `rabbitmq_stream`：流核心及管理插件
    - `rabbitmq_consistent_hash_exchange`：一致性哈希交换机（分片用）
- 启用/禁用插件：`rabbitmq-plugins enable/disable`

**14. 流（Stream）**
- 概念：基于日志的不可变数据结构，类似 Kafka topic
- 特性：可重复消费（偏移量跟踪），高性能持久化，仅追加
- 队列声明：`x-queue-type: stream`
- 消费者偏移量管理：初始偏移量 `first`、`last`、`next`、时间戳
- 消费者自动偏移量提交与手动追踪
- 流过滤（Stream Filtering）：基于消息属性过滤消费者接收的消息
- 超级流（Super Stream）：多个分区流的组合（通过绑定键路由）
- 与经典队列和仲裁队列的对比

**15. 认证与授权**
- 默认认证后端：内部数据库
- 外部认证后端：LDAP、HTTP、OAuth 2.0、TLS 证书认证
- 用户管理：`rabbitmqctl add_user`、`set_permissions`、`set_user_tags`
- 虚拟主机权限：`configure`、`write`、`read`
- 主题授权（Topic Authorization）：细粒度控制交换机、路由键等
- 角色标签（Tags）：`management`、`monitoring`、`administrator` 等
- 客户端认证时的密码加密

**16. 管理监控**
- `rabbitmqctl` 命令集：集群状态、队列列表、连接列表、通道列表、虚拟主机操作等
- `rabbitmq-diagnostics`：健康检查、环境信息
- `rabbitmqadmin`：基于 HTTP API 的命令行管理工具
- 管理插件提供的 HTTP API：节点信息、概览统计、队列/连接/通道操作
- Web 管理界面（Overview、Connections、Channels、Exchanges、Queues、Admin）
- 统计信息：消息速率、确认速率、连接数、内存/磁盘使用等
- 追踪（Firehose）：将消息投递到追踪队列，用于调试
- 审计日志（Audit Log）插件

**17. 性能与流量控制**
- 心跳（Heartbeat）：连接存活检测
- 帧大小（`frame_max`）：AMQP 帧最大尺寸
- 信道最大数（`channel_max`）：每连接信道数限制
- 内存告警与磁盘告警：`vm_memory_high_watermark`、`disk_free_limit`
- 基于信用的流控：当发布过快时，连接或通道会被 `blocked`，客户端收到 `Connection.Blocked` 通知
- 队列主节点的分派策略优化
- 惰性队列（Lazy Queue）：消息直接写入磁盘，节省内存，适用于大量消息堆积场景
- 延迟发布确认的影响

**18. 其他特性与最佳实践**
- 消息顺序保证：单一队列、单一消费者可保证顺序；消息重新入队可能打破顺序；使用“消息组”或一致性哈希交换机保持顺序
- 消费者取消通知（Consumer Cancel Notification）：队列被删除或消费者被取消时通知
- 消费者超时（`consumer_timeout`）：一段时间内无确认自动取消消费者
- 死信环路处理
- 优雅关闭与消息补偿
- 生产端必开发布确认，消费端手动确认，合理设置预取
- 队列选择：经典镜像用于兼容，Quorum Queue 用于高可靠低延迟，Stream 用于大数据量重放场景
- 虚拟主机隔离不同环境/应用
- 策略（Policy）：动态应用队列/交换机的可选参数，优于定义时指定
- 运行时参数（Runtime Parameter）：用于联邦、Shovel 等插件配置
