## 二、逐题精讲

### 面试题 1：MySQL 的逻辑架构分为哪几层？
- **你的答案摘要**：不清楚。
- **评分**：0/20  
- **扣分明细**：
  - 基础概念缺失：未提及连接层、服务层、引	擎层、存储层。
- **可吸收的标准答案**：
  > MySQL 逻辑架构分为四层：
  > 1. **连接层**：负责连接管理、授权认证、安全防护。
  > 2. **服务层 (Server 层)**：核心所在，包含解析器、查询优化器、执行器等。所有跨存储引擎的功能（如存储过程、触发器、视图）都在此实现。
  > 3. **引擎层**：插件式架构，负责数据的存储和提取。
  > 4. **存储层**：底层文件系统，存储数据文件、日志等。
- **具体改进建议**：
  
  - 重点记忆 Server 层中“优化器”和“执行器”的作用。

### 面试题 2：InnoDB 和 MyISAM 的核心区别是什么？为什么现在默认使用 InnoDB？
- **你的答案摘要**：索引是 btree，innodb 有行锁，myisam 只有表锁。
- **评分**：10/20  
- **扣分明细**：
  - 核心缺失：**事务 (ACID) 支持**（这是最本质区别） → 扣 5 分。
  - 缺失点：外键支持、崩溃恢复能力 → 扣 3 分。
  - 索引误区：两者默认索引结构都是 B+Tree，InnoDB 是聚簇索引，MyISAM 是非聚簇索引 → 扣 2 分。
- **可吸收的标准答案**：
  
  > 1. **事务**：InnoDB 支持事务，满足 ACID；MyISAM 不支持。
  > 2. **锁粒度**：InnoDB 支持行级锁；MyISAM 仅支持表级锁。
  > 3. **崩溃恢复**：InnoDB 具备 Crash-Safe 能力（依赖 Redo Log）；MyISAM 不具备。
  > 4. **外键**：InnoDB 支持外键约束。
  > 5. **索引实现**：InnoDB 是聚簇索引（索引和数据在一起）；MyISAM 是非聚簇索引。
- **具体改进建议**：
  - 理解“聚簇索引”对查询性能的影响。

### 面试题 3：InnoDB 是如何实现行级锁的？MyISAM 为什么只支持表级锁？
- **你的答案摘要**：描述了 S/X 锁的互斥逻辑。
- **评分**：8/20  
- **扣分明细**：
  - 未触及底层本质：InnoDB 的行锁是**加在索引项（Index Record）上**的，而非物理记录行 → 扣 7 分。
  - 未解释 MyISAM 限制：MyISAM 缺乏 MVCC 机制和复杂的事务日志系统来支撑细粒度并发控制 → 扣 5 分。
- **可吸收的标准答案**：
  
  > 1. **InnoDB 实现**：行锁是通过给索引上的索引项加锁来实现的。如果没有索引，InnoDB 会退化为表锁。
  > 2. **MyISAM 限制**：其存储格式简单，为了追求极致的读取速度，设计之初就未考虑复杂的事务并发冲突，因此采用简单的表锁来确保数据一致性。

### 面试题 5：崩溃恢复场景下 InnoDB 的优势？依赖什么日志？
- **你的答案摘要**：提到了 redo/undo 日志，以及崩溃后的恢复逻辑。
- **评分**：15/20  
- **扣分明细**：
  - 深度不足：未提及 WAL (Write-Ahead Logging) 协议的重要性 → 扣 3 分。
  - 缺失对比：未提及 MyISAM 崩溃可能导致索引文件损坏且无法自动恢复一致性 → 扣 2 分。
- **可吸收的标准答案**：
  > InnoDB 具有 **Crash-Safe** 能力。其核心依赖 **WAL 机制**：
  > - **Redo Log**：记录物理修改，保证已提交事务的持久性（Forward Recovery）。
  > - **Undo Log**：记录逻辑相反操作，保证未提交事务的原子性（Backward Recovery）。
  > 相比之下，MyISAM 在断电或崩溃时，缓冲区的数据会永久丢失，且索引文件极易损坏。

## 三、编程题点评

**原题**：通过 SQL 命令查询当前 MySQL 实例的默认存储引擎、最大连接数以及缓存池大小，并输出结果。

**参考实现**：
```sql
-- 1. 查看默认存储引擎
SHOW VARIABLES LIKE 'default_storage_engine';

-- 2. 查看最大连接数
SHOW VARIABLES LIKE 'max_connections';

-- 3. 查看 Buffer Pool 大小
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
```

**鼓励的话**：
Day 1 只是热身，看到你对 redo/undo 日志有清晰的恢复逻辑认知，这非常棒！这是理解 MySQL 核心的门槛。继续加油，补齐架构分层和引擎特性，你很快就能建立起完整的 MySQL 知识体系。
