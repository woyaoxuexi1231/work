## Day 08: JMM 与并发三大特性

### 1. 什么是指令重排序？`volatile` 是如何禁止重排序的？

#### 指令重排序原理

**一句话总结**：指令重排序是编译器和 CPU 为了优化性能，在不改变单线程执行结果的前提下，重新安排指令执行顺序的优化手段。

**重排序的三种类型**：

```java
// 源代码
int a = 1;      // 语句 1
int b = 2;      // 语句 2
int c = a + b;  // 语句 3

// 编译器重排序（可能）
int b = 2;      // 语句 2 和 1 交换（不影响结果）
int a = 1;
int c = a + b;

// CPU 重排序（可能）
// CPU 发现语句 1 和 2 没有依赖关系
// 可能乱序执行以提高性能
```

**为什么需要重排序**：
```
1. 编译器优化：
   - 减少寄存器使用
   - 提高指令并行度
   
2. CPU 优化：
   - 指令流水线
   - 乱序执行（Out-of-Order Execution）
   - 内存访问延迟隐藏

示例：
语句 1：读取内存（100 周期）
语句 2：CPU 计算（1 周期）

如果不重排序：CPU 等待 100 周期
如果重排序：先执行语句 2，同时执行语句 1
```

**重排序的限制**：
```java
// as-if-serial 语义：
// 无论如何重排序，单线程的执行结果不能改变

int a = 1;      // 1
int b = a + 1;  // 2 - 依赖 1
int c = b + 1;  // 3 - 依赖 2

// 1、2、3 不能重排序，因为有数据依赖

// 但这种情况可以重排序：
int x = 1;      // 1
int y = 2;      // 2 - 不依赖 1
// 1 和 2 可以交换顺序
```

#### volatile 禁止重排序的原理

**核心机制**：内存屏障（Memory Barrier）

```java
// volatile 变量
volatile boolean flag = false;
int a = 0;

// 线程 1
a = 1;              // 普通写
flag = true;        // volatile 写

// 线程 2
if (flag) {
    System.out.println(a);  // 一定输出 1
}
```

**内存屏障的四种类型**：

```
LoadLoad Barrier：
  Load1; LoadLoad; Load2
  确保 Load1 的数据装载先于 Load2

StoreStore Barrier：
  Store1; StoreStore; Store2
  确保 Store1 的数据刷新到内存先于 Store2

LoadStore Barrier：
  Load1; LoadStore; Store2
  确保 Load1 的数据装载先于 Store2 刷新到内存

StoreLoad Barrier：
  Store1; StoreLoad; Load2
  确保 Store1 的数据刷新到内存先于 Load2 装载
  （这是最严格的屏障，开销最大）
```

**volatile 的屏障策略**：

```
在每个 volatile 写操作前插入 StoreStore 屏障
在每个 volatile 写操作后插入 StoreLoad 屏障

在每个 volatile 读操作后插入 LoadLoad 屏障
在每个 volatile 读操作后插入 LoadStore 屏障

示意图：

普通写 A
volatile 写 B
├─ StoreStore Barrier（在 B 之前）
├─ Store B
└─ StoreLoad Barrier（在 B 之后）

volatile 读 C
├─ Load C
├─ LoadLoad Barrier（在 C 之后）
└─ LoadStore Barrier（在 C 之后）
普通读 D
```

**happens-before 规则**：
```
volatile 写 happens-before 后续的 volatile 读

线程 1：
a = 1;          // 普通写
flag = true;    // volatile 写（插入屏障）

线程 2：
if (flag) {     // volatile 读（插入屏障）
    // 保证能看到 a = 1
    System.out.println(a);
}
```

### 2. `volatile` 能保证原子性吗？为什么 `count++` 不是原子操作？

#### volatile 不能保证原子性

**一句话总结**：volatile 只能保证可见性和有序性，不能保证原子性。复合操作（如 count++）仍然可能产生竞态条件。

**详细分析**：

```java
// 错误用法
volatile int count = 0;

// 10 个线程，每个线程 count++ 1000 次
// 预期结果：10000
// 实际结果：通常小于 10000

// 原因：count++ 不是原子操作
```

**count++ 的字节码分析**：

```java
// Java 代码
count++;

// 编译后的字节码
0: aload_0              // 加载 this
1: dup
2: getfield #2          // 读取 count 的值到操作数栈
5: iconst_1             // 加载常量 1
6: iadd                 // 执行加法
7: putfield #2          // 将结果写回 count
```

**三个步骤**：
```
1. READ：读取 count 的值（如 10）
2. MODIFY：执行 +1 操作（10 + 1 = 11）
3. WRITE：写回 count（11）
```

**竞态条件示例**：

```
初始：count = 10

线程 1：READ count = 10
线程 2：READ count = 10     （线程 1 还没写回）
线程 1：MODIFY 10 + 1 = 11
线程 2：MODIFY 10 + 1 = 11  （基于旧值计算）
线程 1：WRITE count = 11
线程 2：WRITE count = 11    （覆盖了线程 1 的结果）

结果：两个线程执行了两次 count++，但 count 只增加了 1
```

**正确的解决方案**：

```java
// 方案 1：synchronized
public synchronized void increment() {
    count++;
}

// 方案 2：AtomicInteger（推荐）
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // CAS 操作，原子性

// 方案 3：ReentrantLock
ReentrantLock lock = new ReentrantLock();
public void increment() {
    lock.lock();
    try {
        count++;
    } finally {
        lock.unlock();
    }
}
```

### 3. 谈谈 `Happens-Before` 的理解

#### 核心原理

**一句话总结**：Happens-Before 是 JMM 定义的一组偏序关系规则，用于判断一个操作的结果是否对另一个操作可见。

**八大规则**：

```
1. 程序次序规则（Program Order Rule）：
   在一个线程内，书写在前面的操作 happens-before 后面的操作
   
2. 监视器锁规则（Monitor Lock Rule）：
   解锁操作 happens-before 后续的加锁操作
   
3. volatile 变量规则（Volatile Variable Rule）：
   对 volatile 变量的写操作 happens-before 后续的读操作
   
4. 线程启动规则（Thread Start Rule）：
   Thread.start() happens-before 线程内的所有操作
   
5. 线程终止规则（Thread Termination Rule）：
   线程的所有操作 happens-before Thread.join() 返回
   
6. 线程中断规则（Thread Interruption Rule）：
   Thread.interrupt() happens-before 被中断线程检测到中断
   
7. 对象终结规则（Finalizer Rule）：
   对象的初始化完成 happens-before 它的 finalize() 方法
   
8. 传递性规则（Transitivity）：
   如果 A happens-before B，B happens-before C，则 A happens-before C
```

**为什么 Happens-Before 比看源码更重要**：

```
1. 抽象层次高：
   - 不需要理解内存屏障、缓存一致性等底层细节
   - 只需记住 8 条规则即可判断可见性
   
2. 跨平台：
   - 不同 CPU 架构的内存模型不同（x86、ARM）
   - Happens-Before 是 Java 层面的抽象，与平台无关
   
3. 指导编程：
   - 如果两个操作没有 happens-before 关系
   - 就必须使用同步机制（锁、volatile、原子类）
```

**示例分析**：

```java
// 场景：线程 1 初始化配置，线程 2 使用配置
class Config {
    private String value;
    private volatile boolean initialized = false;
    
    // 线程 1 调用
    public void init() {
        value = "config_data";     // 1. 普通写
        initialized = true;        // 2. volatile 写
    }
    
    // 线程 2 调用
    public String getValue() {
        if (initialized) {         // 3. volatile 读
            return value;          // 4. 普通读
        }
        return null;
    }
}

// Happens-Before 分析：
// 1 happens-before 2  （程序次序规则）
// 2 happens-before 3  （volatile 变量规则）
// 3 happens-before 4  （程序次序规则）
// 根据传递性：1 happens-before 4
// 所以线程 2 一定能看到 value = "config_data"
```

## Day 09: Synchronized 深度探究

### 1. 对象头结构（Mark Word）

**对象头的组成**：

```
Java 对象在内存中的布局：
┌─────────────────────┐
│  对象头（Header）    │
├─────────────────────┤
│  实例数据（Data）    │
├─────────────────────┤
│  对齐填充（Padding） │
└─────────────────────┘

对象头包含：
1. Mark Word（标记字段）：64 位
2. Klass Pointer（类指针）：64 位（开启压缩后 32 位）
3. 数组长度（仅数组对象）：32 位
```

**Mark Word 的结构（64 位 JVM）**：

```
锁状态  | unused:25 | identity_hashcode:31 | unused:1 | age:4 | biased_lock:1 | lock:2
--------|-----------|---------------------|----------|-------|---------------|-------
无锁    |           | 对象的 hashCode       |          | 分代年龄 | 0           | 01
偏向锁  | 线程 ID:54|  epoch:2             |          | 分代年龄 | 1           | 01
轻量级锁| 指向锁记录的指针 (62 位)                        | 0           | 00
重量级锁| 指向 Monitor 的指针 (62 位)                     | 0           | 10
GC 标记 |                                                 |           | 11
```

**分代年龄**：
```
- 4 位，最大值 15
- 对象在 Survivor 区每熬过一次 GC，年龄 +1
- 年龄达到阈值（默认 15）时，晋升到老年代
```

### 2. 锁升级过程

**升级路径**：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

升级是单向的，不能降级（除了 GC 时）
```

**偏向锁**：

```java
// 场景：只有一个线程访问同步块
synchronized (obj) {
    // 代码
}

// 偏向锁的优化：
// 1. 第一次获取锁时，将线程 ID 记录在 Mark Word 中
// 2. 后续同一个线程获取锁时，只需检查 Mark Word 中的线程 ID
// 3. 无需 CAS 操作，性能极高

// 偏向锁的撤销：
// 1. 其他线程尝试获取锁
// 2. 暂停持有偏向锁的线程
// 3. 检查线程是否还在同步块中
// 4. 如果不在，恢复无锁或升级为轻量级锁
// 5. 如果在，升级为轻量级锁
```

**轻量级锁**：

```java
// 场景：多线程交替执行同步块，无竞争
synchronized (obj) {
    // 代码
}

// 轻量级锁的实现：
// 1. 线程在自己的栈帧中创建 Lock Record
// 2. 使用 CAS 将 Mark Word 替换为指向 Lock Record 的指针
// 3. 如果 CAS 成功，获取锁
// 4. 如果 CAS 失败，说明有竞争，自旋重试
// 5. 自旋超过阈值，升级为重量级锁

// 自旋锁：
// - 线程不放弃 CPU，循环检查锁
// - 适用场景：锁占用时间短
// - JDK 1.6 引入自适应自旋（根据历史情况调整自旋次数）
```

**重量级锁**：

```java
// 场景：多线程竞争激烈
synchronized (obj) {
    // 代码
}

// 重量级锁的实现：
// 1. Mark Word 指向 Monitor 对象
// 2. Monitor 包含：
//    - Owner：当前持有锁的线程
//    - EntryList：等待锁的线程队列
//    - WaitSet：调用 wait() 的线程队列
// 3. 线程获取锁失败，进入 EntryList 阻塞
// 4. 需要内核态切换，性能开销大
```

## Day 10: AQS 原理

### 1. AQS 的内部结构

**核心字段**：

```java
public abstract class AbstractQueuedSynchronizer {
    // 同步状态（volatile 保证可见性）
    private volatile int state;
    
    // CLH 队列的头节点
    private transient volatile Node head;
    
    // CLH 队列的尾节点
    private transient volatile Node tail;
    
    // 节点状态
    static final class Node {
        volatile int waitStatus;  // 节点状态
        volatile Node prev;       // 前驱节点
        volatile Node next;       // 后继节点
        volatile Thread thread;   // 线程
    }
}
```

**CLH 队列**：

```
双向链表结构：

head → [Node1] ↔ [Node2] ↔ [Node3] ← tail

每个 Node 包含：
- thread：等待的线程
- waitStatus：节点状态
  - CANCELLED (1)：线程已取消
  - SIGNAL (-1)：后继节点需要被唤醒
  - CONDITION (-2)：在 Condition 队列中
  - PROPAGATE (-3)：释放锁需要传播
  - 0：初始状态
```

## Day 11: 读写锁与 StampedLock

### 1. 锁降级过程

```java
// 锁降级：写锁 → 读锁
public class Cache {
    private final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
    private final Lock readLock = rwl.readLock();
    private final Lock writeLock = rwl.writeLock();
    private volatile String data;
    
    public void update(String newData) {
        writeLock.lock();
        try {
            // 更新数据
            data = newData;
            
            // 降级为读锁（在释放写锁之前获取读锁）
            readLock.lock();
        } finally {
            writeLock.unlock();  // 释放写锁，但保留读锁
        }
        
        try {
            // 读取数据（其他线程可以获取读锁，但不能获取写锁）
            System.out.println(data);
        } finally {
            readLock.unlock();
        }
    }
}

// 为什么不支持锁升级（读锁 → 写锁）？
// 因为可能导致死锁：
// 线程 1：持有读锁，请求写锁（阻塞，等待其他线程释放读锁）
// 线程 2：持有读锁，请求写锁（阻塞，等待其他线程释放读锁）
// 互相等待，死锁！
```

## Day 12: 线程池核心参数与执行流程

### 1. 7 大参数详解

```java
public ThreadPoolExecutor(
    int corePoolSize,              // 核心线程数
    int maximumPoolSize,           // 最大线程数
    long keepAliveTime,            // 空闲线程存活时间
    TimeUnit unit,                 // 时间单位
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,   // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**执行流程**：

```
任务提交 → 
  1. 当前线程数 < corePoolSize → 创建新线程执行
  2. 当前线程数 >= corePoolSize → 加入队列
  3. 队列满 && 线程数 < maximumPoolSize → 创建新线程执行
  4. 队列满 && 线程数 >= maximumPoolSize → 执行拒绝策略

图解：

[任务] 
  ↓
[corePoolSize 满了吗？]
  ├─ 否 → [创建核心线程]
  └─ 是 → [队列满了吗？]
           ├─ 否 → [加入队列]
           └─ 是 → [maximumPoolSize 满了吗？]
                    ├─ 否 → [创建非核心线程]
                    └─ 是 → [拒绝策略]
```

## Day 13: ThreadLocal 与 CompletableFuture

### 1. ThreadLocal 内存泄漏原因

```java
// ThreadLocalMap 的结构
class ThreadLocalMap {
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;  // 强引用
        
        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // Key 是弱引用
            value = v;  // Value 是强引用
        }
    }
    
    private Entry[] table;
}

// 内存泄漏过程：
// 1. ThreadLocal 对象没有被外部引用
// 2. Key（ThreadLocal）被 GC 回收（弱引用）
// 3. 但 Entry.value 仍然强引用对象
// 4. Value 无法被回收，内存泄漏

// 解决方案：
// 1. 使用完后调用 remove()
threadLocal.remove();

// 2. 使用 try-finally
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove();
}
```

## Day 14: 并发编程模拟面试

### 高效限流器实现

```java
// 基于滑动窗口的限流器
public class SlidingWindowRateLimiter {
    private final int windowSize;  // 窗口大小（秒）
    private final int maxRequests; // 最大请求数
    private final AtomicInteger[] windows;  // 窗口数组
    private volatile int index;    // 当前窗口索引
    
    public SlidingWindowRateLimiter(int windowSize, int maxRequests) {
        this.windowSize = windowSize;
        this.maxRequests = maxRequests;
        this.windows = new AtomicInteger[windowSize];
        for (int i = 0; i < windowSize; i++) {
            windows[i] = new AtomicInteger(0);
        }
    }
    
    public boolean tryAcquire() {
        int current = index;
        int count = windows[current].incrementAndGet();
        return count <= maxRequests;
    }
}
```

## 面试加分技巧

### 并发编程回答模板

**面试官**：synchronized 和 ReentrantLock 的区别？

**回答结构**：
1. **直接回答**："synchronized 是 JVM 内置的关键字，ReentrantLock 是 JDK 提供的类"
2. **对比差异**："synchronized 使用简单，自动释放锁；ReentrantLock 更灵活，支持公平锁、可中断、超时等"
3. **性能对比**："JDK 1.6 之后两者性能差不多，synchronized 做了很多优化"
4. **使用建议**："优先使用 synchronized，需要高级功能时用 ReentrantLock"
