## Day 22: Java IO 模型演进

### 1. Select/Poll/Epoll 的区别

**对比分析**：

| 特性 | Select | Poll | Epoll |
|------|--------|------|-------|
| 支持的文件描述符 | 1024 | 无限制 | 无限制 |
| 数据结构 | 数组 | 链表 | 红黑树 + 双向链表 |
| 每次调用 | 全量传递 fd 集合 | 全量传递 fd 集合 | 只传递变化的 fd |
| 内核检查 | O(n) | O(n) | O(1)（只检查活跃 fd） |
| 工作模式 | LT | LT | LT + ET |
| 性能 | 低（fd 多时） | 中 | 高 |

**Epoll 的两种模式**：

```java
// LT（Level Trigger，水平触发）：
// - 只要 fd 处于就绪状态，每次 epoll_wait 都会通知
// - 如果一次没处理完，下次还会通知
// - 编程简单，不易遗漏事件

// ET（Edge Trigger，边缘触发）：
// - 只有 fd 状态变化时才通知
// - 必须一次处理完所有数据
// - 性能更高，但编程复杂

// 示例：
// 假设 socket 收到 10KB 数据，每次 read 只能读 2KB

// LT 模式：
// epoll_wait 通知 → read 2KB
// epoll_wait 通知 → read 2KB
// ... 重复 5 次

// ET 模式：
// epoll_wait 通知一次 → 必须循环 read 直到 EAGAIN
```

### 2. 零拷贝（Zero-copy）

**传统 IO 的问题**：

```
读取文件并发送的网络的过程：

1. read() 系统调用：
   磁盘 → 内核缓冲区 → 用户缓冲区（CPU 拷贝）
   
2. write() 系统调用：
   用户缓冲区 → 内核 socket 缓冲区（CPU 拷贝）
   内核 socket 缓冲区 → 网卡（DMA 拷贝）

总共：4 次上下文切换，2 次 CPU 拷贝，2 次 DMA 拷贝
```

**零拷贝的实现**：

```java
// 方式 1：mmap + write
// 减少一次 CPU 拷贝

FileChannel fileChannel = new FileInputStream("file.txt").getChannel();
MappedByteBuffer buffer = fileChannel.map(FileChannel.MapMode.READ_ONLY, 0, fileChannel.size());

SocketChannel socketChannel = SocketChannel.open();
socketChannel.write(buffer);

// 过程：
// 磁盘 → 内核缓冲区（DMA 拷贝）
// 内核缓冲区 → 用户缓冲区（mmap 映射，无拷贝）
// 用户缓冲区 → 内核 socket 缓冲区（CPU 拷贝）
// 内核 socket 缓冲区 → 网卡（DMA 拷贝）

// 3 次上下文切换，1 次 CPU 拷贝，2 次 DMA 拷贝
```

```java
// 方式 2：sendfile（真正的零拷贝）
FileChannel fileChannel = new FileInputStream("file.txt").getChannel();
SocketChannel socketChannel = SocketChannel.open();

// 直接从内核缓冲区传输到 socket 缓冲区
fileChannel.transferTo(0, fileChannel.size(), socketChannel);

// 过程（Linux 2.4+）：
// 磁盘 → 内核缓冲区（DMA 拷贝）
// 内核缓冲区 → 网卡（DMA 拷贝，带 scatter-gather）

// 2 次上下文切换，0 次 CPU 拷贝，2 次 DMA 拷贝
```

## Day 23: Netty 基础

### 1. ByteBuf vs ByteBuffer

**对比**：

| 特性 | ByteBuffer | ByteBuf |
|------|------------|---------|
| 指针 | 单指针（position） | 双指针（readerIndex、writerIndex） |
| 扩容 | 固定大小 | 动态扩容 |
| 零拷贝 | 不支持 | 支持（slice、duplicate） |
| 引用计数 | 无 | 有（手动管理内存） |
| 池化 | 无 | 支持（PooledByteBufAllocator） |
| API | 繁琐（flip、rewind） | 简洁 |

**ByteBuf 的双指针设计**：

```
┌─────────────────────────────────┐
│  discarded  │  readable  │  writable  │
│   bytes     │   bytes    │    bytes    │
│             │            │             │
0      readerIndex      writerIndex    capacity

读取数据：
byte b = buf.readByte();  // readerIndex++

写入数据：
buf.writeByte(0x12);      // writerIndex++

无需手动 flip()！
```

### 2. Netty 解决 TCP 粘包/拆包

**问题**：

```
TCP 是流协议，没有边界：

发送方：
send("Hello")
send("World")

接收方可能收到：
recv("HelloWorld")  // 粘包

或者：
recv("Hel")
recv("loWorld")     // 拆包
```

**Netty 的解决方案**：

```java
// 方案 1：固定长度
pipeline.addLast(new FixedLengthFrameDecoder(1024));

// 方案 2：分隔符
pipeline.addLast(new DelimiterBasedFrameDecoder(1024, Delimiters.lineDelimiter()));

// 方案 3：长度字段（最常用）
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    1024,           // 最大长度
    0,              // 长度字段偏移量
    4,              // 长度字段长度
    0,              // 长度调整值
    4               // 跳过的字节数
));

// 数据包格式：
// [4 字节长度][数据]
// [00000005][Hello]
```

## Day 25: 序列化与 SPI 扩展机制

### 1. JDK 序列化为什么不推荐使用？

**问题**：

```
1. 性能差：
   - 序列化速度慢
   - 序列化后的数据体积大
   
2. 安全性问题：
   - 反序列化漏洞（CVE-2015-6420）
   - 可以构造恶意数据执行代码
   
3. 兼容性差：
   - 类修改后可能无法反序列化
   - 需要维护 serialVersionUID
   
4. 跨语言困难：
   - 只有 Java 支持
   - 无法与其他语言交互
```

**替代方案对比**：

| 方案 | 性能 | 体积 | 跨语言 | 易用性 |
|------|------|------|--------|--------|
| JDK 序列化 | 差 | 大 | 否 | 简单 |
| JSON（Jackson） | 中 | 中 | 是 | 简单 |
| Protobuf | 高 | 小 | 是 | 中 |
| Hessian | 高 | 中 | 是 | 中 |
| Kryo | 高 | 小 | 否 | 中 |

### 2. Java SPI 的原理

**实现过程**：

```java
// 步骤 1：定义接口
public interface DatabaseDriver {
    void connect();
}

// 步骤 2：实现接口
public class MySQLDriver implements DatabaseDriver {
    public void connect() {
        System.out.println("MySQL connected");
    }
}

// 步骤 3：创建配置文件
// META-INF/services/com.example.DatabaseDriver
// 文件内容：
com.example.MySQLDriver

// 步骤 4：加载实现
ServiceLoader<DatabaseDriver> loader = ServiceLoader.load(DatabaseDriver.class);
for (DatabaseDriver driver : loader) {
    driver.connect();
}
```

**SPI 的局限性**：

```
1. 全量加载：
   - 一次性加载所有实现类
   - 无法按需加载
   
2. 无法获取实现类名称：
   - 只能通过迭代获取实例
   - 无法根据名称选择实现
   
3. 缓存问题：
   - ServiceLoader 缓存了实现类
   - 动态加载的实现无法生效
   
4. 没有 IOC/AOP 支持：
   - 实现类依赖注入困难
```

### 2. Sealed Classes（密封类）

```java
// Java 17+ 引入
public sealed class Shape permits Circle, Rectangle, Triangle {
    // 只允许这三个类继承
}

public final class Circle extends Shape { }
public final class Rectangle extends Shape { }
public final class Triangle extends Shape { }

// 配合 switch 表达式
public double calculateArea(Shape shape) {
    return switch (shape) {
        case Circle c -> Math.PI * c.radius() * c.radius();
        case Rectangle r -> r.width() * r.height();
        case Triangle t -> 0.5 * t.base() * t.height();
    };
}
```

### 2. 观察者模式

```java
// Java 内置的观察者模式
class MyObservable extends Observable {
    public void doSomething() {
        setChanged();
        notifyObservers("事件数据");
    }
}

class MyObserver implements Observer {
    @Override
    public void update(Observable o, Object arg) {
        System.out.println("收到通知：" + arg);
    }
}

// 使用
MyObservable observable = new MyObservable();
observable.addObserver(new MyObserver());
observable.doSomething();
```
