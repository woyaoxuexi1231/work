## Day 15: JVM 运行时内存区域

### 1. Java 8 为什么要用元空间取代永久代？

#### 核心原因

**一句话总结**：永久代存在 OOM 风险、调优困难、与 HotSpot 其他部分耦合严重，元空间使用本地内存解决了这些问题。

#### 详细分析

**永久代的问题**：

```
1. 内存限制：
   - 永久代大小受 -XX:MaxPermSize 限制
   - 默认只有 64MB（32 位 JVM）或 82MB（64 位 JVM）
   - 加载大量类时容易 OOM
   
2. 调优困难：
   - 开发者很难预估需要多少永久代空间
   - 设置太小容易 OOM，设置太大浪费内存
   
3. GC 效率低：
   - Full GC 才会回收永久代
   - 回收效率低，容易造成内存碎片
   
4. 与其他 JVM 不兼容：
   - JRockit、IBM J9 等其他 JVM 没有永久代
   - 不利于 JVM 统一
```

**元空间的优势**：

```java
// 元空间使用本地内存（Native Memory）
// 不受 JVM 堆内存限制

// JVM 参数对比：

// JDK 1.7（永久代）：
-XX:PermSize=64m        // 初始大小
-XX:MaxPermSize=256m    // 最大大小

// JDK 1.8（元空间）：
-XX:MetaspaceSize=21m   // 初始大小（触发 GC 的阈值）
-XX:MaxMetaspaceSize    // 最大大小（不设置则无限制）

// 元空间自动扩容：
// 1. 初始 MetaspaceSize = 21MB
// 2. 达到阈值时触发 Full GC
// 3. 如果没有类被卸载，提高阈值
// 4. 阈值最高可达 MaxMetaspaceSize
```

**内存结构对比**：

```
JDK 1.7：
┌──────────────────────┐
│      Heap            │
├──────────────────────┤
│  Eden  │  Survivor   │
├──────────────────────┤
│  Old Generation     │
├──────────────────────┤
│  Permanent Gen      │  ← 在堆内
└──────────────────────┘

JDK 1.8：
┌──────────────────────┐
│      Heap            │
├──────────────────────┤
│  Eden  │  Survivor   │
├──────────────────────┤
│  Old Generation     │
└──────────────────────┘
┌──────────────────────┐
│  Metaspace           │  ← 在本地内存
└──────────────────────┘
```

### 2. 栈帧中包含哪些内容？

**栈帧结构**：

```
每个方法调用都会创建一个栈帧：

┌──────────────────────────────┐
│  局部变量表                   │
│  - 方法参数                   │
│  - 局部变量                   │
│  - this 引用（实例方法）      │
├──────────────────────────────┤
│  操作数栈                     │
│  - 临时存储计算结果           │
│  - 方法调用的参数和返回值     │
├──────────────────────────────┤
│  动态链接                     │
│  - 指向运行时常量池的引用     │
│  - 支持方法调用的动态绑定     │
├──────────────────────────────┤
│  方法返回地址                 │
│  - 方法执行完成后返回的位置   │
├──────────────────────────────┤
│  附加信息                     │
│  - 调试信息                   │
│  - 性能优化信息               │
└──────────────────────────────┘
```

**局部变量表**：

```java
public void test(int a, long b) {
    int c = 10;
    double d = 20.0;
}

// 局部变量表：
// Slot 0: this 引用
// Slot 1: int a
// Slot 2-3: long b（long/double 占 2 个 Slot）
// Slot 4: int c
// Slot 5-6: double d

// 为什么局部变量表大小在编译期确定？
// - 编译器可以分析方法中使用的局部变量
// - 确定需要的 Slot 数量
// - 运行时不需要动态调整
```

## Day 16: 对象创建、布局与定位

### 1. 对象在内存中的布局

**对象头详解**：

```
64 位 JVM（开启压缩指针）：

Mark Word（8 字节）：
┌─────────────────────────────────────────────────────┐
│ unused:25 | hashcode:31 | unused:1 | age:4 | lock:2 │
└─────────────────────────────────────────────────────┘

Klass Pointer（4 字节，压缩后）：
┌──────────────────┐
│ 指向类元数据的指针 │
└──────────────────┘

实例数据（变长）：
┌──────────────────┐
│ 对象的字段数据    │
└──────────────────┘

对齐填充（变长）：
┌──────────────────┐
│ 填充到 8 字节倍数 │
└──────────────────┘
```

**示例计算**：

```java
class User {
    int id;        // 4 字节
    String name;   // 4 字节（引用，压缩后）
}

// 对象大小：
// Mark Word：8 字节
// Klass Pointer：4 字节
// id：4 字节
// name：4 字节
// 总计：20 字节
// 对齐填充：4 字节（对齐到 8 的倍数）
// 最终大小：24 字节
```

### 2. TLAB（Thread Local Allocation Buffer）

**原理**：

```
问题：
- 堆内存是共享的
- 多线程分配对象需要同步
- 性能瓶颈

解决方案：
- 每个线程在 Eden 区分配一块私有缓存
- 小对象直接在 TLAB 中分配
- 无需同步，性能提升

配置：
-XX:+UseTLAB              // 开启 TLAB（默认开启）
-XX:TLABSize              // TLAB 大小
-XX:TLABRefillWasteFraction  // 浪费比例阈值
```

## Day 17: 垃圾回收算法

### 1. 哪些对象可以作为 GC Roots？

**GC Roots 包含**：

```
1. 虚拟机栈（栈帧中的局部变量表）中引用的对象
   - 方法的参数
   - 局部变量
   
2. 方法区中类静态属性引用的对象
   - static 字段
   
3. 方法区中常量引用的对象
   - final static 字段
   
4. 本地方法栈中 JNI 引用的对象
   - Native 方法引用的 Java 对象
   
5. Java 虚拟机内部的引用
   - 基本数据类型对应的 Class 对象
   - 常驻异常对象（NullPointException 等）
   - 系统类加载器
   
6. 所有被同步锁（synchronized）持有的对象
```

### 2. 分代收集算法

**新生代 vs 老年代**：

```
新生代（Young Generation）：
- 占堆内存的 1/3
- 分为 Eden、Survivor0、Survivor1
- 比例默认 8:1:1
- 使用复制算法
- 存放新创建的对象
- GC 频繁（Minor GC）

老年代（Old Generation）：
- 占堆内存的 2/3
- 使用标记-整理算法
- 存放存活时间长的对象
- GC 较少（Major GC / Full GC）

对象晋升过程：
1. 新对象在 Eden 区分配
2. 第一次 Minor GC 后，存活对象移到 Survivor0
3. 第二次 Minor GC 后，Survivor0 存活对象移到 Survivor1
4. 每次 GC 后年龄 +1
5. 年龄达到阈值（默认 15），晋升到老年代
```

## Day 18: 垃圾收集器

### 1. CMS 为什么会被废弃？

**CMS 的问题**：

```
1. CPU 敏感：
   - 并发阶段会占用 CPU 资源
   - 如果 CPU 核心数少，性能影响大
   
2. 浮动垃圾：
   - 并发标记期间产生的垃圾无法回收
   - 可能导致 Concurrent Mode Failure
   
3. 内存碎片：
   - 标记-清除算法会产生碎片
   - 大对象可能无法分配
   
4. 复杂的调优：
   - 需要精细调整参数
   - CMSInitiatingOccupancyFraction
   - UseCMSInitiatingOccupancyOnly
   
5. JDK 9 标记为废弃：
   - G1 成为默认收集器
   - ZGC 和 Shenandoah 提供更低延迟
```

---

### 2. ZGC 为什么能做到 10ms 以内的延迟？

**核心技术**：

```
1. 并发标记：
   - 标记过程与应用程序线程并发执行
   - 不需要 STW
   
2. 并发转移：
   - 对象移动也并发执行
   - 使用染色指针技术
   
3. 染色指针：
   - 64 位指针的高 4 位存储标记信息
   - 标记位：Finalizable、Remapped、Marked1、Marked0
   
4. 读屏障（Load Barrier）：
   - 每次读取对象指针时检查标记
   - 如果需要，自动转发到新地址
   
5. 多映射内存：
   - 使用 mprotect 系统调用
   - 同一块物理内存映射到多个虚拟地址
   
性能：
- STW 时间：< 1ms
- 最大暂停时间：< 10ms
- 支持的堆大小：8MB - 16TB
- 性能损耗：< 15%
```

## Day 19: 类加载机制

### 1. 类加载的过程

**五个阶段**：

```
1. 加载（Loading）：
   - 通过类的全限定名获取字节流
   - 将字节流转换为方法区的运行时数据结构
   - 在内存中生成 Class 对象
   
2. 验证（Verification）：
   - 文件格式验证
   - 元数据验证
   - 字节码验证
   - 符号引用验证
   
3. 准备（Preparation）：
   - 为类变量分配内存
   - 设置默认初始值（零值）
   
4. 解析（Resolution）：
   - 将常量池中的符号引用替换为直接引用
   
5. 初始化（Initialization）：
   - 执行类构造器 <clinit> 方法
   - 为静态变量赋初值
   - 执行静态代码块
```

**准备 vs 初始化**：

```java
public class Test {
    public static int value1 = 123;
    public static final int value2 = 456;
}

// 准备阶段：
// value1 = 0（int 的零值）
// value2 = 456（final 变量在编译时确定）

// 初始化阶段：
// value1 = 123（执行赋值语句）
// value2 = 456（不变）
```

### 2. 双亲委派模型

**委派过程**：

```
ClassLoader 层次结构：
Bootstrap ClassLoader（启动类加载器）
  ↑
Extension ClassLoader（扩展类加载器）
  ↑
Application ClassLoader（应用类加载器）
  ↑
Custom ClassLoader（自定义类加载器）

加载过程：
1. 当前类加载器收到加载请求
2. 先委托给父类加载器
3. 父类加载器继续向上委托
4. 如果父类加载器无法加载，再尝试自己加载

代码实现：
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 检查是否已加载
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        try {
            // 2. 委托给父类
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                // 3. 委托给 Bootstrap ClassLoader
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
        }
        
        // 4. 父类无法加载，自己加载
        if (c == null) {
            c = findClass(name);
        }
    }
    return c;
}
```

**为什么设计双亲委派**：

```
1. 安全性：
   - 防止核心 API 被篡改
   - 如 java.lang.String 只能由 Bootstrap 加载
   
2. 避免重复加载：
   - 同一个类只会被加载一次
   - 父类加载器加载过的类，子类不会重复加载
   
3. 一致性：
   - 保证同一个类在所有地方都是同一个 Class 对象
```

## Day 20: 字节码与 JIT 编译优化

### 1. 逃逸分析

**三种优化**：

```java
public void test() {
    User user = new User();  // 对象创建
    user.setName("张三");
}

// 逃逸分析判断 user 是否逃逸：
// - 是否被返回？否
// - 是否被赋值给全局变量？否
// - 是否被其他线程访问？否
// 结论：user 没有逃逸

// 优化 1：栈上分配
// 对象分配在栈上，方法执行完自动销毁
// 无需 GC，性能提升

// 优化 2：标量替换
// 将对象拆解为基本类型
// User user 拆分为：
// String name = "张三";
// int age = 0;

// 优化 3：锁消除
// 如果对象不会逃逸，无需同步
synchronized (user) {
    // 这个锁会被消除
}
```

## 面试加分技巧

### JVM 回答模板

**面试官**：线上 CPU 100% 怎么排查？

**回答结构**：
1. **定位进程**：`top` 找到 CPU 高的 Java 进程
2. **定位线程**：`top -Hp <pid>` 找到 CPU 高的线程
3. **转换线程 ID**：`printf "%x\n" <tid>` 转为十六进制
4. **查看堆栈**：`jstack <pid> | grep <十六进制 tid> -A 50`
5. **分析代码**：找到对应的代码行，分析问题
