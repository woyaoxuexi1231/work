## 🧐 逐题深度点评

### 1. equals 与 hashCode 的约束

- **你的回答**：提到了集合类寻址失败的问题，逻辑正确。
- **面试官纠偏**：
  - 必须提到 **Object 规范（The Contract）**：如果两个对象 `equals` 相等，它们的 `hashCode` 必须相等。
  - **后果描述**：在 `HashSet` 或 `HashMap` 中，会造成“明明对象相等却存了两次”或“存进去却取不出来”的诡异 Bug，导致数据不一致。
  - **加分项**：提到 `hashCode` 是为了快速定位（Bucket），`equals` 是为了解决哈希冲突。

### 2. `new String("abc")` 创建对象数

- **你的回答**：意识到了常量池的存在，分情况讨论，思路清晰。
- **面试官纠偏**：
  - **术语纠正**：不是“静态变量池”，是 **“字符串常量池 (String Constant Pool)”**。
  - **底层细节**：`new` 关键字一定会在**堆 (Heap)** 上创建一个新对象。如果池中没有 `"abc"`，则会在池中也创建一个。

### 3. String 的不可变性

- **你的回答**：提到了安全性和一致性，但性能部分基本靠猜。
- **面试官纠偏**：
  - **性能维度**：
    1. **哈希码缓存**：String 不可变，其 `hashCode` 也可以被缓存（String 类内部有 `hash` 字段），非常适合做 HashMap 的 Key。
    2. **字符串池化**：只有不可变，才能实现池化共享。
  - **安全维度**：除了你说的业务一致性，还有**线程安全**（天生线程安全）和**网络/文件传输安全**。

### 4. 泛型擦除与反射获取

- **你的回答**：比较混乱，没解释清楚“既然擦除了为什么还能拿到”。
- **面试官纠偏**：
  - **为什么要擦除**：核心是为了 **兼容性 (Backward Compatibility)**，让 Java 5 之前的代码也能运行。
  - **反射如何获取**：虽然类内部的泛型被擦除了，但在类文件（.class）的 **Signature 属性**中保留了泛型签名。通过 `getGenericSuperclass()` 等 API 可以获取到。

## 🏆 Day 01 标准答案详解（建议背诵并内化）

### Q1：为什么重写 equals 必须重写 hashCode？

1. **契约要求**：根据 `java.lang.Object` 的规范，如果两个对象通过 `equals()` 方法比较相等，那么它们的 `hashCode()` 必须返回相同的值。
2. **哈希容器逻辑**：`HashMap/HashSet` 是先通过 `hashCode()` 计算哈希值来定位数组下标（Bucket），再通过 `equals()` 确认对象是否真的相等。
3. **不重写的后果**：如果只重写 `equals` 而不重写 `hashCode`，会导致两个“逻辑相等”的对象哈希值不同。它们会被分配到不同的桶中，导致 `Set` 出现重复元素，或者 `Map` 无法通过 key 找到对应的 value。

### Q2：`String s = new String("abc")` 创建了几个对象？

- **1个或2个**。
- **1个对象**：如果字符串常量池中已经存在 `"abc"`，则只在堆上 `new` 一个 String 对象。
- **2个对象**：如果常量池中没有 `"abc"`，则先在常量池创建一个对象，再在堆上创建一个对象。

### Q3：String 为什么要设计成不可变的？

1. **字符串池 (String Pool) 的需求**：只有 String 不可变，才能实现池化共享，节省大量内存。
2. **线程安全**：不可变对象天生线程安全，可以在多个线程间共享而无需同步。
3. **哈希码优化**：String 的哈希码在创建时被缓存，这使得它作为 Map 的 Key 时性能极高。
4. **安全性**：String 经常作为网络连接、文件路径、数据库连接的参数。如果可变，可能会在校验通过后被恶意修改（TOCTOU 攻击）。

### Q4：谈谈泛型擦除及其反射获取。

1. **泛型擦除**：Java 的泛型是“伪泛型”。在编译期，所有的泛型信息都会被擦除，替换为原始类型（如 `Object`）或限定类型。这是为了兼容 Java 1.5 之前的 JVM。
2. **反射获取**：虽然运行时实例不携带泛型信息，但 JVM 将泛型签名信息存储在类文件的 **`Signature`** **属性**中。
3. **API**：可以通过 `ParameterizedType` 接口，配合 `method.getGenericReturnType()` 或 `field.getGenericType()` 等方法在运行时获取泛型类型。

### Q5：Integer 缓存机制详解

1. **缓存范围**：`Integer` 内部维护了一个 `IntegerCache` 静态内部类，默认缓存了 **-128 到 127** 之间的对象。
2. **原理**：当通过 `Integer.valueOf(i)`（自动装箱时调用）创建对象时，会先检查是否在缓存范围内。如果在，直接返回缓存中的对象。
3. **设计模式**：这是典型的 **享元模式 (Flyweight Pattern)**，旨在减少重复创建小对象的开销。
4. **可配置性**：上限 127 可以通过 JVM 参数 `-XX:AutoBoxCacheMax=<size>` 进行调整，但下限 -128 是固定的。

## ⚡ 深度追问补录（追问记录与结论）

### Q6：一万个字符的超长字符串也会在常量池创建吗？

- **结论**：**会，只要在 65535 字节限制内。**
- **底层原理**：
  - Class 文件常量池中字符串以 `CONSTANT_Utf8_info` 存储，其长度字段为 16 位无符号数，最大值为 65535。
  - **编译期限制**：字面量超过 65535 字节会导致编译错误。
  - **运行期限制**：如果是运行期动态拼接的长字符串，存在堆中，除非调用 `intern()` 且常量池有空间。

### Q7：为什么在 `GenericMethod.java` 中反射拿不到 `GenericSimpleObject`？

- **结论**：**你反射的是类定义（Declared Type），而非运行时的具体实例类型（Instance Type）。**
- **深度解析**：
  - **类定义元数据**：`DataTransferReq<S>` 在类定义里字段类型是 `List<S>`。反射 `DataTransferReq.class` 只能拿到 `S` 这个变量。
  - **实例擦除**：`new DataTransferReq<GenericSimpleObject>()` 创建的实例在运行时只知道自己是 `DataTransferReq`，具体的 `GenericSimpleObject` 被擦除了。
- **面试级解决方案**（如何拿到？）：
  - **匿名内部类技巧**：使用 `new DataTransferReq<GenericSimpleObject>() {}`（多出一对花括号）。这会创建一个子类，子类的父类签名中硬编码了 `GenericSimpleObject`。
  - **获取方式**：通过 `this.getClass().getGenericSuperclass()` 即可获取到带参数的具体泛型类型。
