## 二、逐题血淋淋复盘

### 面试题 1：讲讲 AQS 的内部结构。它是如何唤醒后续节点的？
- **你说的**：state 状态值和双向链表，通过 release 拿到 head 的 next 唤醒。
- **得分**：8/20  
- **哪里错了 / 哪里不够**：
  - • **概念模糊**：双向链表里存的是什么？面试官想听到 **Node** 对象，想听到 **WaitStatus**。
  - • **唤醒逻辑过于草率**：如果 head 的 next 为空，或者 next 节点被取消了（waitStatus > 0）怎么办？
  - • **面试官心想**：只知道个大概，真让他写个 Lock 肯定崩。
- **你该补什么**：
  - 去查：AQS 内部 **Node 的几种状态**（CANCELLED, SIGNAL, CONDITION, PROPAGATE）。
  - 去看：`unparkSuccessor` 方法源码，重点看当 next 为 null 或取消时，是如何从 **tail 往前找** 的。

### 面试题 2：ReentrantLock 的公平锁与非公平锁实现差异在哪里？
- **你说的**：公平锁通过 `hasQueuedPredecessors()` 判断当前线程是否持有锁。
- **得分**：12/20  
- **哪里错了 / 哪里不够**：
  - • **严重错误**：`hasQueuedPredecessors()` 是判断**队列中是否有排在自己前面的线程**，而不是判断自己是否持有锁。
  - • **面试官心想**：API 名字都能记错逻辑，这面试直接挂。
  - • **缺失关键点**：非公平锁在调用 `lock()` 的第一时间会尝试直接 **CAS 修改 state**，抢不到才走后续逻辑。
- **你该补什么**：
  - 去看：`NonfairSync` 和 `FairSync` 的 `lock` 方法源码对比。

### 面试题 3：为什么非公平锁的吞吐量更高？
- **你说的**：不需要总是放入队尾等待。
- **得分**：10/20  
- **哪里错了 / 哪里不够**：
  - • **浮于表面**：没提到**线程唤醒的延迟**。
  - • **面试官心想**：背了结论，不懂原理。
- **你该补什么**：
  - 去想：线程 A 释放锁，唤醒线程 B，B 从“唤醒”到“真正开始跑”是有 CPU 时间损耗的。如果此时线程 C 刚好过来，直接抢到锁跑完，是不是利用了这段“空档期”？

## 三、编程题：代码审判级点评

**原题**：阅读 `ReentrantLockTest.java`（你的代码叫 `Test.java`）。

### 0. 死刑宣判
> **第 16-18 行：致命错误。** 
> `lock()` 之后没有接 `try-finally` 块。如果业务代码抛出异常，`unlock()` 永远不会执行，锁永远无法释放。**这是生产环境中最严重的死锁事故源。**

### 1. 正确性——你在写 Hello World 吗？
- 你只是在 main 函数里顺序调用了 `lock()` 和 `unlock()`。
- **核心缺失**：没有**多线程竞争**，没有**并发场景**，你根本没测试出锁的作用。

### 2. 复杂度——无分析
- O(1) 的调用，不需要分析，但你连这个态度都没有。

### 3. 鲁棒性检查——漏了多少地雷
- [ ] try-finally 释放锁：没做 → **必挂项**
- [ ] 多线程验证：没做 → 毫无意义

### 4. 正确的写法（带注释）
```java
public void correctUsage() {
    ReentrantLock lock = new ReentrantLock();
    lock.lock(); // 加锁不要放在 try 块内，防止加锁失败反而执行 unlock 报错
    try {
        // 核心业务逻辑
        System.out.println("执行中...");
    } finally {
        lock.unlock(); // 必须在 finally 块释放，保证即使异常也能解锁
    }
}
```
