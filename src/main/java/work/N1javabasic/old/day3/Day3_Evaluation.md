## 二、逐题精讲

### 面试题 1：1.7 的死循环是怎么产生的？1.8 是如何解决的？
- **你的答案摘要**：认为死循环是多线程在桶位置插入元素冲突产生的；1.7 使用头插法，具体指针变迁过程记不清了。
- **评分**：8/20  
- **扣分明细**：
  - **概念错误**：死循环发生在 **`transfer`（数据迁移）** 阶段，而非普通的 `put` 插入冲突。 → 扣 5 分
  - **细节缺失**：没有提到关键的“链表翻转”逻辑。 → 扣 4 分
  - **逻辑链条断裂**：未描述清楚线程 A 和 B 如何形成环。 → 扣 3 分
- **可吸收的标准答案**：
  > 1. **1.7 死循环原因**：1.7 扩容时采用 **头插法**，迁移后的链表顺序会翻转。
  >    - 假设旧链表为 `A -> B`。线程 T1 准备迁移 A，记录了 `e=A, next=B` 后被挂起。
  >    - 线程 T2 完成扩容，新表顺序变为 `B -> A`。
  >    - T1 恢复，处理 A 时将其指向新表头（此时是 B），导致 `A.next = B`。
  >    - 而原有的 `B.next = A` 依然存在（或在 T1 处理 B 时形成），最终形成 **环形链表**，导致 `get` 时死循环。
  > 2. **1.8 解决方法**：改用 **尾插法**。扩容时保持元素的原始相对顺序，且通过“高低位链表”直接拆分，从根源杜绝了环的形成。
- **具体改进建议**：
  - **建议 1**：在纸上模拟一遍 `e.next = newTable[i]; newTable[i] = e; e = next;` 这三行核心代码在并发下的指针变迁。
  - **建议 2**：对比 1.8 源码中 `TreeNode.split()` 或 `resize()` 里的尾插逻辑。
- **鼓励的话**：你能准确说出“头插法”是祸根，这说明你已经抓住了 1.7 版本的命门。

### 面试题 2：为什么负载因子默认是 0.75？为什么红黑树转换阈值是 8？
- **你的答案摘要**：通过计算得到的均衡点，超过 8 链条太长影响性能。
- **评分**：10/20  
- **扣分明细**：
  - **深度不足**：未提及 **泊松分布（Poisson Distribution）**。 → 扣 6 分
  - **回答太泛**：未说明“均衡”的具体对象（空间 vs 时间）。 → 扣 4 分
- **可吸收的标准答案**：
  > 1. **0.75**：是 **空间成本** 与 **时间成本** 的折中。
  >    - 若设为 1.0，空间利用率高但碰撞剧烈，查询效率降至 O(N)。
  >    - 若设为 0.5，碰撞少但空间浪费严重。
  >    - 0.75 附近是碰撞概率开始剧增的临界点（数学上与 log2 相关）。
  > 2. **阈值 8**：根据 **泊松分布**，在负载因子为 0.75 时，同一个桶内元素达到 8 个的概率仅为 **0.00000006**（千万分之六）。选择 8 是为了在极端碰撞（如哈希算法糟糕）时，利用红黑树保证 O(logN) 的保底性能，同时避免在元素较少时 TreeNode 带来的额外空间开销（TreeNode 占用空间约为普通 Node 的两倍）。
- **具体改进建议**：
  - **建议 1**：阅读 HashMap 源码开头的英文注释，那里面详细解释了泊松分布的计算背景。
  - **建议 2**：记住“千万分之六”这个数字，这在面试中非常具有说服力。
- **鼓励的话**：知道是“计算得到的均衡点”已经比单纯背答案强很多了。

### 面试题 3：扰动函数（hash 方法）的作用是什么？为什么数组长度要是 2 的幂次方？
- **你的答案摘要**：扰动函数是提供随机数证明相等；2 的幂次方是为了好扩容。
- **评分**：5/20  
- **扣分明细**：
  - **常识性错误**：**扰动函数与对象相等判定（equals/hashCode）完全无关**。 → 扣 10 分
  - **回答不全**：漏掉了“位运算优化取模”的核心性能点。 → 扣 5 分
- **可吸收的标准答案**：
  > 1. **扰动函数作用**：`(h = key.hashCode()) ^ (h >>> 16)`。
  >    - 核心目的：**减少哈希冲突**。
  >    - 原理：数组长度 `n` 通常较小，计算索引时 `(n-1) & hash` 只用到了低位。如果哈希值高位特征明显而低位雷同，就会剧烈冲突。扰动函数让 **高 16 位与低 16 位进行异或**，混合了特征，使哈希分布更均匀。
  > 2. **2 的幂次方原因**：
  >    - **性能优化**：当 `n` 为 2 的幂次方时，`hash % n == (n - 1) & hash`。位运算效率远高于除法取模。
  >    - **分布均匀**：扩容后元素位置要么在原位，要么在 `原位 + oldCap`，分布极度均匀且计算简便。
- **具体改进建议**：
  - **建议 1**：彻底搞清楚 `hashCode()`、`equals()` 和 HashMap 内部 `hash()` 扰动函数这三者的调用链条。
  - **建议 2**：手动计算一下 `15 & 10` 和 `10 % 16` 的结果，体会 2 的幂次方的巧妙。
- **鼓励的话**：你能想到“左移扩大两倍”和扩容有关，说明你对二进制运算是有感觉的，只是没用对地方。

public class SimpleHashMap<K, V> {
    static final int DEFAULT_INITIAL_CAPACITY = 16;
    Node<K, V>[] table;

    // 1. 扰动函数：面试必考点
    static final int hash(Object key) {
        int h;
        return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
    }

    public V put(K key, V value) {
        int h = hash(key);
        // 2. 定位公式：(n - 1) & hash
        int index = (table.length - 1) & h;
        // ... 后续链表或红黑树插入逻辑

    }
}
