## 二、逐题血淋淋复盘

### 面试题 1：`ReentrantReadWriteLock` 适合什么场景？存在什么问题（锁饥饿）？
- **你说的**：缓存、白名单；写性能低下；读插队导致写饥饿。
- **得分**：15/20  
- **哪里错了 / 哪里不够**：
  - • **应用场景太笼统**：面试官想听的是 **“读多写少且数据量较大”** 的场景。如果数据量小，synchronized 性能可能更好，因为 RRWL 维护 AQS 队列有开销。
  - • **面试官心想**：知道结论，但不知道权衡。
- **你该补什么**：
  - 去查：RRWL 的 `state` 高 16 位和低 16 位分别存了什么？（面试必考细节）。

### 面试题 2：锁降级的过程是怎样的？为什么不支持锁升级？
- **你说的**：写锁内获取读锁再释放写锁；读升级写会因为互相等待导致死锁。
- **得分**：18/20  
- **哪里对了**：
  - • **逻辑满分**：你对“线程 A 等 B 释放读，B 等 A 释放读”导致死锁的推导非常准确。
- **你该补什么**：
  - 去想：锁降级的**核心意义**到底是什么？（提示：是为了在保证数据可见性的同时，允许其他线程并发读，提高吞吐量）。

### 面试题 3：`StampedLock` 相比 `ReentrantReadWriteLock` 做了哪些改进？
- **你说的**：新增乐观锁；乐观读不加锁；可以避免写饥饿。
- **得分**：12/20  
- **哪里错了 / 哪里不够**：
  - • **遗漏致命点**：StampedLock 是 **不可重入** 的！这是它和 RRWL 最大的区别，用错直接死锁。
  - • **原理太浅**：乐观读为什么能提高性能？因为它不需要修改 state（没有 CAS 操作），只读 state，对 CPU 缓存友好。
- **你该补什么**：
  - 去看：StampedLock 为什么不支持 Condition？
  - 去查：`sl.validate(stamp)` 底层用到了什么指令？（提示：内存屏障）。
