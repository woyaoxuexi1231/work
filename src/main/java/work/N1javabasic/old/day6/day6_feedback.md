## 二、逐题精讲

### 面试题 1：Stream 的并行流（parallelStream）底层实现与坑点
- **你的答案摘要**：底层使用 `ForkJoinPool.commonPool()`。坑点在于：1. 共享线程池可能导致任务阻塞；2. 任务必须独立。
- **评分**：18/20  
- **扣分明细**：
  - 深度不足：未提及并行流在处理“有状态操作”（如 `sorted`, `distinct`）时的巨大性能损耗 → 扣 1 分
  - 忽略线程安全：并行流操作共享变量（如 `ArrayList` 的 `add` 操作）会导致线程不安全 → 扣 1 分
- **可吸收的标准答案**：
  > 1. **底层实现**：并行流底层采用 **ForkJoinPool** 框架，通过“分治”思想将大任务拆分为子任务，利用“工作窃取”算法提高 CPU 利用率。
  > 2. **核心坑点**：
  >    - **共享池风险**：所有并行流默认共享 `commonPool`，若某处任务阻塞，会拖慢全局并行任务。
  >    - **线程不安全**：在 Lambda 内部操作非线程安全容器（如 `ArrayList`）会导致竞态条件。
  >    - **开销问题**：数据量小时，拆分任务的开销可能大于并行带来的收益。
  >    - **有序性**：`forEach` 在并行流下是无序的，若需有序则必须使用 `forEachOrdered`。

### 面试题 2：Optional 真的能完全解决 NPE 吗？正确姿势？
- **你的答案摘要**：一定程度上避免了空指针，通过 `isPresent` 和 `orElseGet` 处理。
- **评分**：15/20  
- **扣分明细**：
  - 概念误区：面试官通常不喜欢 `isPresent + get` 的写法（这和 `if (obj != null)` 没区别） → 扣 3 分
  - 缺失最佳实践：未提及 Optional 不应作为类成员变量或方法参数 → 扣 2 分
- **可吸收的标准答案**：
  > 1. **能否解决**：不能。Optional 是为了**减少**而非完全消除 NPE，它强制调用者面对“值可能为空”的现实。
  > 2. **正确姿势**：
  >    - **拒绝 `get()`**：永远不要直接调用 `get()`，应使用 `orElse`, `orElseGet` 或 `orElseThrow`。
  >    - **链式编程**：利用 `map`, `filter`, `flatMap` 进行流畅的链式处理。
  >    - **避免滥用**：Optional 极耗内存（它是一个包装对象），不要将其用于集合元素、方法参数或类属性。

### 面试题 3：接口默认方法（default method）冲突了怎么办？
- **你的答案摘要**：在实现类中必须重写该方法，并可以使用 `InterfaceName.super.method()` 指定调用哪一个。
- **评分**：20/20  
- **可吸收的标准答案**：
  > Java 遵循“类优先”原则和“显式覆盖”原则：
  > 1. **类优先**：若父类和接口有同名方法，优先调用父类的（实现类不需要重写）。
  > 2. **接口冲突**：若两个接口有同名默认方法，编译器会报错，实现类必须**显式覆盖**该方法，并在方法体内手动指定调用逻辑（或完全重写）。
