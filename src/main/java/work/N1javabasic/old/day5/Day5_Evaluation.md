## 二、逐题精讲

### 面试题 1：ArrayList 扩容是 1.5 倍，为什么不是 2 倍？
- **你的答案摘要**：因为扩容公式是 size < 1 + arraylist 所以是 1.5 倍。
- **评分**：12/20  
- **扣分明细**：
  - **表达不专业**：源码公式是 `oldCapacity + (oldCapacity >> 1)`，利用了位运算。 → 扣 4 分
  - **深度缺失**：未解释为什么 1.5 比 2 好（如：1.5 倍扩容能更好地复用之前的内存碎片）。 → 扣 4 分
- **可吸收的标准答案**：
  > 1. **实现方式**：`newCapacity = oldCapacity + (oldCapacity >> 1)`。采用右移运算效率极高。
  > 2. **为什么 1.5 而非 2**：
  >    - **内存复用**：k=2 时，每次扩容后的容量都大于之前所有容量之和，导致之前的内存永远无法被复用。k=1.5 时，经过几次扩容，新分配的内存块可能落在之前释放的内存碎片中（假设内存是连续分配的）。
  >    - **增长平稳**：避免了一次性申请过大空间导致内存浪费，同时也平衡了频繁扩容带来的复制开销。

### 面试题 2：ArrayBlockingQueue 与 LinkedBlockingQueue 的锁实现有什么区别？
- **你的答案摘要**：Array 一个锁；Linked 两个锁（读写分离）。
- **评分**：18/20  
- **扣分明细**：
  - **细节缺失**：未提到 LinkedBlockingQueue 这种设计带来的“虚假唤醒”处理和锁竞争降低的本质。 → 扣 2 分
- **可吸收的标准答案**：
  > 1. **ArrayBlockingQueue**：使用 **一把锁 (ReentrantLock)** 控制入队和出队。这意味着同一时刻只能有一个线程在操作（要么生产要么消费）。适用于并发量较低或需要强一致性顺序的场景。
  > 2. **LinkedBlockingQueue**：使用 **两把锁 (putLock & takeLock)**。由于是链表结构，头部和尾部操作相对独立，两把锁实现了**生产和消费的并发并行**，极大提升了吞吐量。

### 面试题 3：PriorityBlockingQueue 是如何保证有序的？它的扩容逻辑有何特殊之处？
- **你的答案摘要**：实现 Comparable 或指定 Comparator；扩容就是数组复制。
- **评分**：15/20  
- **扣分明细**：
  - **结构缺失**：未提到底层是 **小顶堆（二叉堆）**。 → 扣 3 分
  - **扩容细节缺失**：扩容时会**先释放锁**再进行数组分配，这是为了不阻塞出队操作。 → 扣 2 分
- **可吸收的标准答案**：
  > 1. **有序性**：底层基于 **数组实现的平衡二叉堆（小顶堆）**。入队时会进行“上浮”操作，出队时进行“下沉”操作，保证堆顶永远是优先级最高的元素。
  > 2. **扩容特殊点**：扩容时（通常是 1.5 倍或 2 倍，取决于当前大小）会**先释放锁**，通过一个自旋锁（allocationSpinLock）来保证只有一个线程在创建新数组，这样在分配大内存期间，其他线程仍然可以进行 `take()` 出队操作。

## 🎓 总教官深度补课：解开你的技术心结

### 1. 为什么 1.5 倍扩容能复用内存，而 2 倍不行？

这是一个经典的数学与工程结合的问题。我们假设内存分配是**连续**的，且每次扩容后，旧的内存会被立刻释放。

#### **当扩容因子 k = 2 时：**
1.  初始容量为 10。
2.  第一次扩容：20（之前释放了 10）。
3.  第二次扩容：40（之前累计释放了 10+20=30）。
4.  第三次扩容：80（之前累计释放了 10+20+40=70）。
*   **结论**：你会发现，**每一次新申请的内存空间，都比之前所有释放的空间总和还要大**。这意味着，旧的内存碎片永远无法拼凑出一块足够大的空间来满足下一次扩容，造成了内存的“永久浪费”。

#### **当扩容因子 k = 1.5 时：**
1.  初始容量为 10。
2.  第一次扩容：15（释放 10）。
3.  第二次扩容：22（释放 15，累计 25）。
4.  第三次扩容：33（释放 22，累计 47）。
5.  第四次扩容：49（释放 33，累计 80）。
*   **结论**：在第 4-5 次扩容左右，**之前累计释放的内存总和就会超过下一次需要申请的空间**。此时，底层内存管理器（如 glibc 的 malloc）就有机会把之前那些零散的内存块重新合并，分配给这次扩容。这极大地提高了内存的利用率。

### 2. 什么是小顶堆（二叉堆）？

`PriorityBlockingQueue` 的灵魂就是**小顶堆**。它不是一个复杂的链表，而是一个用**数组**实现的**完全二叉树**。

#### **核心特征**
1.  **结构特征**：除了最后一层外，每一层都是满的，且最后一层从左向右填充。
2.  **顺序特征**：**父节点的值永远小于或等于其左右子节点的值**。
    *   这意味着：**堆顶（数组下标为 0 的位置）永远是整棵树中最小的元素**。

#### **为什么用数组实现？**
因为完全二叉树的父子节点下标有完美的数学关系：
*   如果父节点下标是 `i`：
    *   左孩子下标 = `2 * i + 1`
    *   右孩子下标 = `2 * i + 2`
*   如果孩子节点下标是 `j`：
    *   父节点下标 = `(j - 1) / 2`

#### **核心操作：如何维持有序？**
1.  **入队（上浮 Sift Up）**：
    *   把新元素插到数组末尾。
    *   不断跟自己的父节点比，如果比父节点小，就交换位置，直到不再比父节点小或到达堆顶。
2.  **出队（下沉 Sift Down）**：
    *   取出堆顶元素（下标 0）。
    *   把数组最后一个元素放到堆顶。
    *   不断跟左右孩子比，如果比孩子大，就跟较小的那个孩子交换位置，直到不再比孩子大或到达叶子节点。

**面试一句话总结**：小顶堆通过位运算定位父子关系，利用“上浮”和“下沉”操作，以 **O(logN)** 的时间复杂度保证了每次取出的都是优先级最高（最小）的元素。
