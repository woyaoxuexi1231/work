#### `synchronized` 加在静态方法和实例方法上有什么区别？

加在静态方法上作用于整个类，在任何地方，整个类的这个方法同一时间仅允许一个线程访问。

加载实例方法上则作用于单个对象，不同的对象可以被不同的线程同时访问，互不干扰。



重新回答：

1. 加载静态方法和示例方法上，通过 javap -v 可以看到，在方法的 flags中，多了一个 ACC_SYNCHRONIZED 的标记。
2. 其次，加载静态方法上等同于 synchronized(Test.class) ，而在示例方法中，等同于 synchronized(this) 

#### 描述一下锁升级的过程。为什么 1.6 之后引入了这么多级别的锁？

synchronized的锁升级过程比较复杂。

使用到了对象的对象头的markword字段。

对象头的结构：

markword(当前类型对象的运行状态信息) + classopoint（一个指针，指向当前对象类型在方法区中的类型数据）

锁的四种状态：

**无锁**

一个对象存在无竞争和有竞争两种情况，即使存在有竞争的情况，那么也应该首先考虑无锁竞争的情况，因为直接加锁带来的开销实在太大。

这个时候引入了cas操作，cas操作允许线程通过改变对象内的某个值来达到加锁的目的，改变成功的线程相当于成功获取了该对象的锁，而没有改变成功的对象将进行自选尝试，直到改变成功获取到该对象的操作权为止。



**偏向锁**

是否支持偏向锁是由markword的倒数第3位决定的，这个是由具体的jvm决定的。

存在这样一种场景，一个对象大部分时间都被同一个线程所持有，但是还是线程安全的前提下，然后此对象支持偏向锁，那么在加锁的过程中，这个同步代码块在首次进入的时候，那么会直接检查倒数2个位，看当前是否处于无锁状态，也就是 01，如果是，那么检查是否支持偏向锁，也就是倒数第3个bit，如果是1，就支持偏向锁，然后检查对象的前23bit，为空则设置当前线程id，拿到锁；如果不为空，但是就是当线程，也拿到锁；但是如果不是当前线程那么就要进行进一步的锁升级了。升级为**轻量级锁**



**轻量级锁**

多个线程进行偏向锁时发生此升级

原偏向线程进入安全点检查时，检查是否依旧在同步块内：

如果在，那么暂停线程，将锁升级为轻量级锁，将该线程的锁记录指针(lock record)写入markword，然后继续执行。竞争线程执行轻量级锁的获取流程

如果已经不在了，那么撤销偏向，回到无锁状态，重新竞争。



线程先建立自己的lock record，然后使用cas，尝试把 markword 替换为指向自己的lockrecord的指针，如果成功，则获取到偏向锁，如果不成功，进行自选。超过一定次数升级为重量级锁。





**重量级锁**

在轻量级锁自选失败后，线程会通过尝试CAS将对象头的指针替换为 objectmonitor (monitor mutex) 操作系统互斥量的指针

一旦成功，后续所有线程在moniterenter的时候进入等待队列，线程被挂起。

持有线程退出同步块的时候monitorexit，会唤醒等待队列中的线程。



至此完成了从 无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁的 升级过程

引入这么多级别的锁，主要还是规避了使用 monitorenter 和 monitorexit 这两个系统级别的操作，这两个操作会带来线程的上下文切换，伴随线程的挂起和唤醒，开销时比较大的，无锁、偏向锁、轻量级锁都大量使用cas操作来规避这种线程的切换操作，大大提升了性能。



重新回答：

1. 无锁状态是一个对象刚被创建的时候的状态。
2. 当一个线程首次进入同步代码块的时候，他首先需要检查对象是否支持偏向锁，如果支持那么就直接cas把markword的前23位改为当前线程的id，如果成功了，那么就直接拿到对象的锁了，如果不成功，那么说明已经由其他线程竞争，进入偏向锁的撤销流程，可能会升级成轻量级锁。
3. 撤销偏向锁的时候，持有线程仍在同步代码块中，那么升级为轻量级锁，此时是通过cas把对象头前30个bit替换为线程的lockrecord指针，替换成功那么就获取到锁了。
4. 轻量级锁自选一定次数后仍失败，那么就需要使用moinitorenter和monitorexit了。



#### 为什么重量级锁需要切换到内核态？性能开销在哪里？

重量级锁之所以必须切换到内核态，是因为它依赖操作系统的互斥量(mutex)

用户态的代码是无法做到让线程休眠和唤醒，这是操作系统级别提供的权利，涉及cpu调度。

当一个线程尝试获取一个已经被占用的重量级锁的时候，jvm会调用操作系统的系统原语，此时用户态将切换到内核态，内核态将该线程移除运行队列，设置为等待，触发线程调度。

当持有锁的线程释放锁的时候，同样会调用系统原语来操作，此时也会进入内核态，内核会进行调度，把等待队列的线程放入运行队列中。

**性能开销**

1. 内核态的切换
2. 线程上问的切换，这设计把需要休眠的线程的数据进行保存，恢复需要运行的线程的数据
3. 线程调度的开销，维护等待队列和运行队列
