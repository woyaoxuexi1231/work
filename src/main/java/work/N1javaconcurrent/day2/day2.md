#### 锁升级过程、AQS 核心架构、Condition 原理。

第1个锁升级过程，锁升级过程是在jdk1.8之后引入的，对于synchronized的优化， synchronized的锁是通过对象的mark word来实现的，当一个对象使用synchronized的枷锁时，他第一时间会去检查mark word中最后一个字节的数字它被分为了无锁和偏向锁。无锁情况下支持通过改变对象的某个值，通过CAS操作来直接进行无锁的操作。第2种是偏向锁，如果加锁的对象被发现它的对象头中的是否偏向锁这一位被标记为了一，那么它会尝试通过CAS操作来改变macworld中的前23个比特，把它修改为当前线程的线程ID来达到加锁的目的，但是如果他通过CAS失败了，那么这个时候锁将由偏向锁改为轻量级锁，轻量级锁会把MAC word的前三十个字段通过CAS改为指向栈中所记录的指针， CS自旋在轻量级锁的情况下， CAS自旋超过失败超过一定次数之后，会转化为重量级锁，重量级锁需要用到系统级的互斥量了，这个时候就是把MAC word中中的前30个字节改为指向互斥量的指针，这个时候就涉及操作系统级别的指令使用了，会涉及到用户态和内核态的切换线程的沉睡唤醒。



AQS是通过一个state字段来标标识它的状态，以及一个双向的链表来作为等待队列实现的阻塞队列的结构。



#### 为什么 `Synchronized` 升级到重量级锁需要切换到内核态？开销具体在哪？

Synchronize升级到重量所，重量级所需要切换到内核态，主要依赖于操作系统提供的互斥量来实现线程的阻塞和唤醒，而互斥量是操作系统内核级的系统调用，必须从用户态切换到内核态具体的开销主要体现在用户态和内核态的切换，因为这样每次调用CPU都需要将用户态的寄存器、线、线程栈等信息全部切换到内核栈，执行完成后再恢复，这个过程本身就会消耗很长的时间。第2个就是线程上下文切换，线程被阻塞的时候，操作系统会将其上下文保存到内存，并把另一个线程的上下文从内存中加载进来，如果线程过多或竞争激烈，那大量的上下文切换就会吃掉CPU的性能。第3个就是维护内核等待队列，重量级锁的等待唤醒操作依赖内核的等待队列等待线程会进入一个等待集合，被唤醒后又要从内核态切换换回用户态重新竞争锁。
