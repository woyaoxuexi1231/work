#### 可见性、原子性、有序性、Happens-Before 原则、Volatile 内存屏障。

第1个可见性，可见性是一个线程对某个变量共享变量的修改，其他线程能够立刻看到修改后的值。第2个原子性是一个或多个操作，要么全部执行完且不被任何因素中断，要么全部不执行。第3个有序性程序执行的顺序，按照代码的先后顺序执行。第1个可见性呢，是因为CPU缓存和指令重排序的问题，现代CPU中每个CPU为了速度更快，每个核心都有自己的缓存共享变量，都是先通过写入本地内存再刷入主内存，其他核心是不会第一时间读取到最新的值的。第2个原则性， Java中的一条语句在编译后，其实可能对应多条字节码指令，比如count加加这种操作，它被翻译成字节码指令之后，分别对应三条指令，第一个是get static,第二个是I add,第三个是put static,其实是被分为了三步操作多线程的情况下呢，由于这一个操作，它非原子性，多个线程同时操作，看了下家，最终结果肯定远远小于理想得到的结果。第3个就是有序性， Java编译器和处理器，为了优化性能，它可能会改变语句在原本编写状态下的先后顺序，改变顺序之后的代码和在单线程运行下的代码，两者的结果其实是一致的，但是多线程下呢，因为指令重排了，会导致不一致的问题。



Happened before,它规定的是两个操作A和操作B,如果操作A happened before操作B,那么操作A的结果对操作B可见，并且A的执行顺序在B之前然后Java中其实是规定了八大happen before原则的啊，其中比较典型的就是程序次第规则啊，在一个线程内书写在前面的操作， happened before书写在后面的操作，然后对一个锁的解锁操作，还有比如说后续对这个锁的加锁操作。 volatile变量规则，对一个volatile变量写操作， happened before,后续对这个volatile变量读操作。



第1个就是这个volatile,保证可见性，当一个线程修改volatile的值之后，它会立即把这个值刷入到储存中，当另一个线程要读取这个volatile的值的时候，那么它会直接从储存中获取这个volatile修饰的变量的最新值。第2个就是有序性， volatile可以直接禁止指令重排序，它会在volatile修饰的变量前后插入屏内存屏障。



#### 什么是 CPU 缓存行（Cache Line）？为什么会有“伪共享”问题？

CPU相较于内存来说，它的运行速度是非常快的，所以后续引进了高速缓存直接加在CPU上，这样CPU的运行速度就不会再被内存这种慢速度给拖拖慢了，但是呢，这个时候会存在一个问题，每个CPU它都有自己的缓存，当这个CPU改变了一个值之后，它存储的数据在自己的缓存上，而另一个CPU它并不能读取到这个CPU内存上的值，所以这就存在了一个数据不一致的问题。



#### `volatile` 是如何保证可见性的？它能保证 `count++` 的安全性吗？

被volatile修饰的变量，他线程修改这个变量值的时候，他会直接把这个值刷入到主存中，线程读取被volunteer修饰的变量的值的时候，它会直接从主存中获取这个变量的最新值，这也就解决了内存可见性的问题。但是呢， volatile它并不能保证count++这种操作的安全性，因为count++在底层，它其实并不是一个原子操作， count++它被分为了三个步骤，但是volatile它并不保证原子性，所以volatiler不能保证count++的安全性。



#### 详细解释一下 `Happens-Before` 原则中的“传递性”和“监视器锁规则”。

Happen before原则中的传递性是指如果一个操作A happened before BB happened before C,那么A一定happened before C,然后第2个是这个监视器锁规则定义的是一个锁的解锁操作一定happen before,后续对这个锁的加锁操作，它保证了在解锁的时候，现存对现存所有的修改都会刷回到主存中，加锁的时候线程强制从主存中重新加载所有的共享变量，从而让解锁前线程的所有修改对加锁后的线程可见。
