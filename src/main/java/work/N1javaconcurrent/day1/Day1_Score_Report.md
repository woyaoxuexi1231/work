### 面试题 1 标准答案：可见性、原子性、有序性、Happens-Before 原则

**结论**：并发编程三大特性是可见性、原子性、有序性。Happens-Before 是 JMM 定义的可见性保证规则，编译器和处理器必须遵守。

**三大特性详解**：

| 特性 | 定义 | 问题原因 | 解决方案 |
|------|------|----------|----------|
| 可见性 | 一个线程修改共享变量，其他线程能立刻看到 | CPU 缓存、寄存器导致工作内存与主内存不一致 | volatile、synchronized、Lock |
| 原子性 | 一个或多个操作作为整体执行，中间不被打断 | 一条 Java 语句可能对应多条字节码指令 | synchronized、Lock、Atomic 类 |
| 有序性 | 程序执行顺序与代码顺序一致 | 编译器优化、指令重排序 | volatile、synchronized、happens-before |

**JMM 内存模型**：

```
┌─────────────────────────────────────────┐
│              主内存（Main Memory）         │
│  存储所有共享变量的副本                   │
└──────────────┬──────────────────────────┘
               │
       ┌───────┴───────┐
       │               │
┌──────▼──────┐ ┌──────▼──────┐
│ 线程 A 工作内存 │ │ 线程 B 工作内存 │
│  - 变量 x 的副本 │ │  - 变量 x 的副本 │
│  - 变量 y 的副本 │ │  - 变量 y 的副本 │
└─────────────┘ └─────────────┘
```

- 每个线程有自己的工作内存（CPU 缓存、寄存器的抽象）
- 线程对变量的操作都在工作内存中进行
- 工作内存与主内存之间通过 read/load/use/assign/store/write 交互

**Happens-Before 规则（8 条）**：

1. **程序次序规则**：单线程内，书写在前 happens-before 书写在后
2. **锁规则**：同一个锁的解锁 happens-before 后续加锁
3. **volatile 变量规则**：volatile 写 happens-before 后续 volatile 读
4. **传递性规则**：A happens-before B，B happens-before C，则 A happens-before C
5. **线程启动规则**：Thread.start() happens-before 线程内操作
6. **线程中断规则**：Thread.interrupt() happens-before 检测到中断
7. **线程终止规则**：线程内操作 happens-before Thread.join() 返回
8. **对象终结规则**：对象构造完成 happens-before finalize() 开始

**as-if-serial 语义**：
- 单线程内，无论怎么重排序，执行结果不能改变
- 编译器和处理器可以对不存在数据依赖的操作重排序

**常见错误**：
- ❌ "volatile 保证原子性" → 错，只保证可见性和有序性
- ❌ "synchronized 只保证原子性" → 错，也保证可见性和有序性
- ❌ "指令重排序会改变单线程结果" → 错，as-if-serial 语义保证单线程结果不变

**一句话总结**：可见性、原子性、有序性是并发编程三大基石，Happens-Before 是 JMM 定义的可见性保证规则，编译器和处理器必须遵守。

### 面试题 2 标准答案：CPU 缓存行（Cache Line）与伪共享问题

**结论**：缓存行是 CPU 缓存的最小单位（64 字节），伪共享是多个线程修改同一缓存行中的不同变量导致缓存行失效，性能急剧下降。

**CPU 缓存架构**：

```
┌──────────────────────────────────────────────┐
│                 CPU Core 1                    │
│  ┌─────────────┐  ┌─────────────┐            │
│  │   L1 Cache  │  │   L2 Cache  │            │
│  │  (32KB-64KB)│  │  (256KB)    │            │
│  └─────────────┘  └─────────────┘            │
└──────────────────────────────────────────────┘
                    │
┌──────────────────────────────────────────────┐
│                 L3 Cache (Shared)             │
│              (8MB-32MB)                       │
└──────────────────────────────────────────────┘
                    │
┌──────────────────────────────────────────────┐
│                 Main Memory                   │
│              (DDR4/DDR5)                      │
└──────────────────────────────────────────────┘
```

**缓存行（Cache Line）**：
- CPU 从内存读取数据时，不是读取单个变量，而是读取整个缓存行
- 缓存行大小通常是 **64 字节**
- 例如：读取一个 long（8B），会连同周围的 56B 一起加载到缓存行

**伪共享（False Sharing）**：

```java
// 伪共享示例
class FalseSharing {
    long a; // 线程 1 修改（假设地址 0x1000）
    long b; // 线程 2 修改（假设地址 0x1008）
    // a 和 b 在同一个缓存行（0x1000 - 0x103F）
}
```

**伪共享的发生过程**：
1. 线程 1 在 Core 1 修改 `a`，Core 1 的缓存行变为 Modified 状态
2. 线程 2 在 Core 2 修改 `b`，发现缓存行在 Core 1 且状态为 Modified
3. Core 2 发起缓存一致性协议（MESI），Core 1 的缓存行失效
4. Core 1 重新从内存加载缓存行，Core 2 的缓存行也失效
5. 两个线程不断使对方的缓存行失效，性能急剧下降

**性能影响**：
- 正常情况：纳秒级访问（L1 Cache）
- 伪共享：百纳秒级访问（需要从内存重新加载）
- 性能差距可达 **100 倍**

**解决方案**：

1. **使用 `@Contended` 注解（JDK 8+）**：
```java
import sun.misc.Contended;

class NoFalseSharing {
    @Contended
    long a; // 线程 1 修改
    
    @Contended
    long b; // 线程 2 修改
}
```
- JVM 会自动填充缓存行，使 `a` 和 `b` 不在同一个缓存行
- 需要添加 JVM 参数：`-XX:-RestrictContended`

2. **手动填充缓存行（Padding）**：
```java
class NoFalseSharing {
    long p1, p2, p3, p4, p5, p6, p7; // 填充 56B
    long a; // 线程 1 修改（占 8B，总共 64B）
    
    long q1, q2, q3, q4, q5, q6, q7; // 填充 56B
    long b; // 线程 2 修改（占 8B，总共 64B）
}
```

3. **使用 `LongAdder`（高并发计数器）**：
- `LongAdder` 内部使用 `Cell[]` 数组，每个 Cell 独立一个缓存行
- 线程分散到不同的 Cell，避免伪共享

**工业级案例**：
- `ConcurrentHashMap` 的 `CounterCell` 使用 `@Contended` 避免伪共享
- `LongAdder` 的 `Cell` 使用填充避免伪共享
- 高性能队列（Disruptor）使用填充避免伪共享

**常见错误**：
- ❌ "伪共享是多个线程修改同一个变量" → 错，是修改同一个缓存行中的**不同**变量
- ❌ "缓存行大小是 32 字节" → 错，现代 CPU 通常是 64 字节
- ❌ "只有 Java 有伪共享" → 错，所有语言都有，是 CPU 架构问题

**一句话总结**：缓存行是 CPU 缓存的最小单位（64B），伪共享会导致缓存行频繁失效，性能下降 100 倍。解决方案是填充缓存行或使用 LongAdder。

### 面试题 3 标准答案：volatile 如何保证可见性？能保证 count++ 的安全性吗？

**结论**：volatile 通过内存屏障保证可见性和有序性，但不保证原子性，不能保证 count++ 的安全性。

**volatile 保证可见性的原理**：

1. **写入时**：
   - 线程修改 volatile 变量后，立即将工作内存中的值刷新到主内存
   - 插入 **StoreStore** 和 **StoreLoad** 内存屏障
   - 使其他线程工作内存中该变量的缓存失效

2. **读取时**：
   - 线程读取 volatile 变量前，从主内存重新加载最新值
   - 插入 **LoadLoad** 和 **LoadStore** 内存屏障
   - 禁止后续读写操作重排序到 volatile 读之前

**内存屏障详解**：

| 屏障类型 | 插入位置 | 作用 |
|----------|----------|------|
| LoadLoad | volatile 读之后 | 保证后续读操作读取到最新值 |
| LoadStore | volatile 读之后 | 保证后续写操作在读取之后执行 |
| StoreStore | volatile 写之前 | 保证前面的写操作完成后再执行 volatile 写 |
| StoreLoad | volatile 写之后 | 保证 volatile 写完成后再执行后续读写（**最昂贵**） |

**JVM 底层实现**：
- x86 架构：使用 `lock` 前缀指令（Lock Addl $0x0, (%rsp)）
- `lock` 指令会锁定总线或缓存行，强制刷新写缓冲区
- 使其他 CPU 核心的缓存行失效（MESI 协议）

**volatile 不能保证 count++ 安全性的原因**：

```java
// count++ 的字节码
getstatic count    // 1. 从主内存读取 count 到工作内存
iconst_1           // 2. 压入常量 1
iadd               // 3. 相加
putstatic count    // 4. 写回主内存
```

**并发执行时序图**：

```
线程 A                  线程 B
  |                      |
getstatic(0)            |
  |                      |
  |                    getstatic(0)  // 读取到旧值 0
  |                      |
iadd(1)                 |
  |                      |
putstatic(1)            |
  |                      |
  |                    iadd(1)
  |                      |
  |                    putstatic(1)  // 覆盖为 1，丢失线程 A 的修改
  |                      |
```

- 最终结果是 1 而不是 2，**丢失了一次修改**
- volatile 只能保证每次读取到最新值，但不能保证复合操作的原子性

**解决方案**：

| 方案 | 原理 | 适用场景 | 性能 |
|------|------|----------|------|
| `synchronized` | 锁保证原子性 | 低并发、复杂逻辑 | 中 |
| `AtomicInteger` | CAS 无锁 | 中等并发、简单累加 | 高 |
| `LongAdder` | 分段累加 | 高并发、计数器 | 极高 |

**AtomicInteger 实现**：
```java
public final int incrementAndGet() {
    return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
}

// CAS 循环
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset); // 读取最新值
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS 更新
    return v;
}
```

**LongAdder 实现**：
```java
// 内部结构
class LongAdder {
    transient volatile Cell[] cells; // 分段数组
    transient volatile long base;    // 基础值
}

class Cell {
    @sun.misc.Contended // 避免伪共享
    volatile long value;
}
```
- 低竞争时累加到 `base`
- 高竞争时分散到 `cells[]`，每个线程操作不同的 Cell
- 最终结果 = base + sum(cells[])

**常见错误**：
- ❌ "volatile 保证原子性" → 错，只保证可见性和有序性
- ❌ "count++ 是原子操作" → 错，分为读、加、写三步
- ❌ "AtomicInteger 比 synchronized 慢" → 错，中等并发下 AtomicInteger 更快

**一句话总结**：volatile 通过内存屏障保证可见性和有序性，但不保证原子性。count++ 应该用 AtomicInteger 或 LongAdder。

### 面试题 4 标准答案：Happens-Before 原则中的"传递性"和"监视器锁规则"

**结论**：传递性是 Happens-Before 的核心推导规则，监视器锁规则保证同一个锁的解锁 happens-before 后续加锁，实现可见性保证。

**传递性规则**：

```
如果 A happens-before B，且 B happens-before C，那么 A happens-before C
```

**传递性的作用**：
- 通过已知的 Happens-Before 关系，推导出更多的可见性保证
- 是 Happens-Before 规则的**核心推导机制**

**传递性示例**：

```java
// 线程 A
x = 1;                    // 操作 1
volatileFlag = true;      // 操作 2（volatile 写）

// 线程 B
while (!volatileFlag);    // 操作 3（volatile 读）
System.out.println(x);    // 操作 4
```

**推导过程**：
1. 操作 1 happens-before 操作 2（程序次序规则）
2. 操作 2 happens-before 操作 3（volatile 变量规则）
3. 操作 3 happens-before 操作 4（程序次序规则）
4. 通过传递性：操作 1 happens-before 操作 4
5. **结论**：线程 B 的操作 4 一定能看到操作 1 的修改（x = 1）

**监视器锁规则**：

```
同一个锁的解锁操作 happens-before 后续对这个锁的加锁操作
```

**关键点**：
- 强调的是**同一个锁**，不是任意锁
- 解锁前线程对工作内存的所有修改，在加锁后对其他线程可见

**监视器锁规则示例**：

```java
// 线程 A
synchronized (lock) {
    x = 1;              // 操作 1
    y = 2;              // 操作 2
}                       // 操作 3：解锁

// 线程 B
synchronized (lock) {   // 操作 4：加锁
    System.out.println(x); // 操作 5：一定能看到 x = 1
    System.out.println(y); // 操作 6：一定能看到 y = 2
}
```

**推导过程**：
1. 操作 1 happens-before 操作 3（程序次序规则）
2. 操作 2 happens-before 操作 3（程序次序规则）
3. 操作 3 happens-before 操作 4（监视器锁规则）
4. 操作 4 happens-before 操作 5（程序次序规则）
5. 操作 4 happens-before 操作 6（程序次序规则）
6. 通过传递性：操作 1 happens-before 操作 5，操作 2 happens-before 操作 6
7. **结论**：线程 B 一定能看到 x = 1 和 y = 2

**监视器锁规则的底层实现**：
- 解锁时：释放锁前，线程对工作内存的所有修改刷新到主内存
- 加锁时：获取锁后，线程从主内存重新加载共享变量到工作内存
- JVM 实现：通过内存屏障（StoreLoad）保证

**传递性与监视器锁规则的组合使用**：

```java
// 线程 A
synchronized (lock1) {
    x = 1;
}

// 线程 B
synchronized (lock1) {
    System.out.println(x); // 能看到 x = 1
    
    synchronized (lock2) {
        y = x + 1;
    }
}

// 线程 C
synchronized (lock2) {
    System.out.println(y); // 能看到 y = 2
}
```

**推导过程**：
1. 线程 A 的 `x = 1` happens-before 线程 A 解锁 lock1
2. 线程 A 解锁 lock1 happens-before 线程 B 加锁 lock1（监视器锁规则）
3. 线程 B 加锁 lock1 happens-before 线程 B 的 `System.out.println(x)`
4. 通过传递性：线程 A 的 `x = 1` happens-before 线程 B 的打印
5. 线程 B 的 `y = x + 1` happens-before 线程 B 解锁 lock2
6. 线程 B 解锁 lock2 happens-before 线程 C 加锁 lock2（监视器锁规则）
7. 通过传递性：线程 B 的 `y = x + 1` happens-before 线程 C 的打印

**常见错误**：
- ❌ "传递性可以推导任意操作" → 错，只能推导存在 Happens-Before 关系的操作
- ❌ "监视器锁规则是任意锁" → 错，必须是**同一个锁**
- ❌ "synchronized 只保证原子性" → 错，也保证可见性和有序性

**一句话总结**：传递性是 Happens-Before 的核心推导规则，监视器锁规则保证同一个锁的解锁 happens-before 后续加锁，两者组合使用可以推导出复杂的可见性保证链条。
