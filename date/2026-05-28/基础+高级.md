### 一、Spring Framework 基础

#### 1. IoC 容器
- **BeanFactory** vs **ApplicationContext**（功能层次、延迟加载、国际化、事件等）
- 常用实现类：`ClassPathXmlApplicationContext`、`AnnotationConfigApplicationContext`
- 容器的启动、刷新、关闭生命周期

#### 2. Bean 定义与注册
- XML 配置、注解配置（`@Component`, `@Service`, `@Repository`, `@Controller`）
- `@Bean` + `@Configuration` 的 Full 模式 vs Lite 模式
- 组件扫描：`@ComponentScan`（includeFilters, excludeFilters）
- 条件注册：`@Conditional` 及其派生注解（`@ConditionalOnClass` 等 Spring Boot 扩展）

#### 3. 依赖注入（DI）
- 注入方式：构造器注入（推荐）、Setter 注入、字段注入（`@Autowired`）
- 循环依赖的解决机制（三级缓存，仅限单例且非构造器注入）
- `@Qualifier`、`@Primary`、`@Resource`（JSR-250）

#### 4. Bean 作用域
- `singleton`（默认）、`prototype`、`request`、`session`、`application`
- 自定义作用域：`Scope` 接口

#### 5. Bean 生命周期
- 实例化 → 属性填充 → BeanNameAware → BeanFactoryAware → 前置处理 → 初始化（`@PostConstruct`, `afterPropertiesSet`, `init-method`）→ 后置处理 → 销毁
- 关键扩展点：`BeanPostProcessor`（前置/后置）、`InstantiationAwareBeanPostProcessor`、`DestructionAwareBeanPostProcessor`

#### 6. AOP（面向切面编程）
- 核心概念：`JoinPoint`、`Advice`（@Before, @After, @Around 等）、`Pointcut`、`Aspect`、`Weaving`
- 代理方式：JDK 动态代理（基于接口）vs CGLIB（基于子类）
- 织入时机：编译期、类加载期、运行期（Spring 使用运行期）
- 自动代理：`@EnableAspectJAutoProxy`，`AopContext.currentProxy()` 解决内部调用

#### 7. Spring 表达式语言（SpEL）
- 基本语法：`#{...}`，支持属性引用、方法调用、类型运算、集合筛选/投影
- 使用场景：`@Value`、`@ConditionalOnExpression`、XML 配置

#### 8. 资源与国际化
- `Resource` 接口（`ClassPathResource`, `FileSystemResource`, `UrlResource`）
- `ResourceLoader` 与 `ResourcePatternResolver`
- 国际化：`MessageSource`（`ReloadableResourceBundleMessageSource`），`LocaleContextHolder`

#### 9. 事件机制
- `ApplicationEvent` + `ApplicationListener`（或 `@EventListener`）
- 同步 vs 异步（`@Async` + `SimpleApplicationEventMulticaster` 配置线程池）
- 事务绑定事件：`@TransactionalEventListener`

#### 10. 数据访问（Spring JDBC / ORM 集成）
- `JdbcTemplate`、`NamedParameterJdbcTemplate`
- 事务抽象：`PlatformTransactionManager`（DataSourceTransactionManager, JtaTransactionManager）
- 声明式事务：`@Transactional`（传播行为、隔离级别、回滚规则）
- ORM 整合：`LocalSessionFactoryBean`（Hibernate）、`@PersistenceContext`（JPA）

## 第二部分：高级深潜（Advanced Folder）

> 目标：从“缺少什么、为什么需要、设计思路、巧思”角度剖析重要特性，建立架构级认知。

### 1. 循环依赖的解决方案 —— 三级缓存的精妙之处

**背景（缺什么？）**  
两个 Bean 互相引用（A→B, B→A），若直接构造注入会导致“先有鸡还是先有蛋”的死锁。Spring 默认支持单例 setter/字段注入的循环依赖，但不支持构造器注入或 prototype 作用域的循环依赖。

**需要什么？**  
一种打破创建顺序死锁的机制：在 Bean 未完全初始化时，提前暴露一个“半成品”给依赖方，待依赖方创建完成后，再回过头来完成当前 Bean 的完整初始化。

**设计思路**  
Spring 使用 **三级缓存**（`singletonObjects`, `earlySingletonObjects`, `singletonFactories`）：
- 一级缓存：完全初始化好的单例 Bean。
- 二级缓存：早期暴露的 Bean 实例（属性未填充完成，但已可引用）。
- 三级缓存：`ObjectFactory`，用于生成早期 Bean 的代理对象（核心巧思）。

**巧思**  
- 为何不用二级缓存就够了？引入三级缓存是为了解决 **AOP 代理** 问题：如果一个 Bean 需要被代理（比如事务切面），那么提前暴露的应该是代理对象，而不是原始对象。三级缓存的 `ObjectFactory` 在调用时才会决定生成代理还是返回原始实例，从而保证最终注入的是正确的代理对象。
- 整个机制仅对 **单例 + 非构造器注入** 有效，因为构造器注入必须获得完整实例，无法提前暴露。
- 这也是为什么官方建议“避免循环依赖，重构代码”的原因——依赖三级缓存会牺牲一些性能和可读性。

### 2. AOP 代理的选择与内部调用失效 —— 设计权衡

**背景（缺什么？）**  
业务中经常需要横向逻辑（日志、事务、权限），传统 OOP 通过继承或装饰器实现，侵入性强。需要一种无侵入、声明式的能力。

**需要什么？**  
一个动态代理框架，能够拦截目标对象的方法调用，在调用前后织入额外逻辑。同时要求对使用者透明，不强制实现接口。

**设计思路**  
Spring AOP 默认使用 JDK 动态代理（基于接口），若目标类没有实现任何接口，则回退到 CGLIB（基于字节码生成子类）。  
- JDK 代理：只能代理接口方法，对象必须实现接口，性能较好，且是 Java 原生支持。  
- CGLIB：可以代理普通类，但无法代理 `final` 方法，生成代理类开销略大。

**巧思 & 难点**  
- **内部调用失效问题**：当你在同一个类的内部调用 `this.method()` 时，方法调用不会经过代理对象，因此切面不会生效。Spring 的解决方案：
  1. 注入自身代理（通过 `AopContext.currentProxy()`），显式调用代理方法。
  2. 重新设计类结构，将内部调用移到不同 Bean。
  3. 使用 `@EnableAspectJAutoProxy(exposeProxy = true)` + `(YourService)AopContext.currentProxy()`。
- 为什么 Spring 不强制使用 CGLIB 让所有方法都能被代理？因为 CGLIB 创建代理更慢，且某些场景（如 `final` 类/方法）会失败。把选择权留给用户：`spring.aop.proxy-target-class=true`。
- 这一设计体现了 **约定大于配置，但允许覆盖** 的哲学。

### 3. Spring 事务管理的设计 —— 从本地事务到分布式事务的抽象

**背景（缺什么？）**  
原始 JDBC 事务需要手动 `conn.setAutoCommit(false); conn.commit(); conn.rollback()`，代码冗长且容易遗漏。更麻烦的是切换不同数据源（JPA、Hibernate、JTA）时，API 完全不同。

**需要什么？**  
一个统一的事务抽象层：无论底层是 `DataSource`、JPA 还是 JTA，上层只需要声明式（`@Transactional`）或编程式（`TransactionTemplate`）控制事务边界，并能管理多资源（如同时操作两个数据库）的两阶段提交。

**设计思路**  
Spring 事务的核心是 `PlatformTransactionManager` 接口：
```java
public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition definition);
    void commit(TransactionStatus status);
    void rollback(TransactionStatus status);
}
```
- `TransactionDefinition`：传播行为（7 种）、隔离级别、超时、只读标志。
- `TransactionStatus`：当前事务状态（是否新事务、是否已完成、是否仅回滚）。

**巧思**  
1. **传播行为的实现**：通过判断当前线程是否已经存在事务绑定（`TransactionSynchronizationManager`），决定挂起、新建或融合。挂起时保存旧事务到栈，恢复时再绑回。这是轻量级线程绑定，避免了复杂的事务管理器间通信。
2. **声明式事务的本质**：通过 AOP 拦截 `@Transactional` 方法，在 `@Around` 通知中调用 `transactionManager.getTransaction()` → 执行业务逻辑 → `commit/rollback`。巧妙之处在于 **将事务管理与业务逻辑完全分离**。
3. **事务与同步**：`TransactionSynchronization` 允许注册回调（如在事务提交后清理缓存），`@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)` 就是基于此实现。
4. **只读事务优化**：设置 `readOnly=true` 时，某些数据库驱动（如 MySQL）会跳过锁或减少日志，Spring 也会将 FlushMode 设为 MANUAL，提升性能。

### 4. Spring Boot 自动配置 —— 从条件化加载到可拔插架构

**背景（缺什么？）**  
传统 Spring 项目需要大量 XML 或 Java Config 显式声明 `DataSource`、`TransactionManager`、`ViewResolver` 等。每个新项目都要重复一遍，且第三方库集成困难。

**需要什么？**  
一个 **智能的、基于类路径和配置的默认配置系统**：当你在 classpath 中引入 H2 驱动，自动帮你创建 `DataSource`；当你引入 `spring-boot-starter-web`，自动配置 DispatcherServlet。同时允许用户通过少量配置覆盖默认行为。

**设计思路**  
- 所有自动配置类都标记 `@Configuration`，并且加上各种 `@Conditional`：
  - `@ConditionalOnClass`：存在某个类才生效（如 `DataSource` 存在）。
  - `@ConditionalOnMissingBean`：用户没有自定义 `DataSource` 才自动配置。
  - `@ConditionalOnProperty`：配置开关，如 `spring.datasource.enable=true`。
- Spring Boot 启动时扫描 `META-INF/spring.factories` 中 `org.springframework.boot.autoconfigure.EnableAutoConfiguration` 键值列表，加载这些配置类。
- 配置类中使用 `@EnableConfigurationProperties` 绑定 `application.yml` 中的属性（如 `DataSourceProperties`）。

**巧思**  
1. **排序与防冲突**：自动配置类之间可能有依赖（如 `DataSourceAutoConfiguration` 要在 `MyBatisAutoConfiguration` 之前执行）。Spring Boot 用 `@AutoConfigureOrder`、`@AutoConfigureAfter`、`@AutoConfigureBefore` 控制顺序，而不是依赖 `@Order`（因为自动配置数量庞大，显式声明更可靠）。
2. **灵活覆盖机制**：用户一旦在 `@Configuration` 中定义 `@Bean`，`@ConditionalOnMissingBean` 就会跳过默认配置。这让 **默认即合理，定制也简单**。
3. **starter 的设计哲学**：一个 starter 只包含一个自动配置模块 + 依赖管理，例如 `spring-boot-starter-web` 包含了 Tomcat、Spring MVC、Jackson 等。开发者引入一个 starter 就获得了一整套生态。
4. **自我排除**：有时自动配置可能因为某些条件不满足而被 disable，但日志中缺乏提示。Spring Boot 2.4+ 增加了 `ConditionEvaluationReport`，可以在 `/actuator/conditions` 端点查看每个自动配置的匹配/不匹配原因，极大提升了可调试性。

### 5. 嵌入式容器的设计 —— 打破传统部署模式的变革

**背景（缺什么？）**  
传统 Java Web 应用需要将 WAR 包部署到外部 Tomcat，配置繁琐，不同环境容器版本不一致容易出问题，开发调试也慢（每次重启容器）。

**需要什么？**  
一种 **将 Web 容器直接嵌入到应用内的方式**，让应用可以像普通 Java 程序一样通过 `main` 方法启动，并且容器的配置可以通过代码或配置文件统一管理，彻底消除“环境差异”。

**设计思路**  
Spring Boot 在 `spring-boot-starter-web` 中默认引入 Tomcat 的嵌入式版本（`tomcat-embed-core`）。`SpringApplication` 启动时会根据类路径判断 Web 应用类型，创建 `ServletWebServerApplicationContext`。该上下文在 `onRefresh()` 阶段调用 `createWebServer()`，初始化 Tomcat/Jetty 并绑定端口。

**巧思**  
1. **工厂模式解耦**：`ServletWebServerFactory`（如 `TomcatServletWebServerFactory`）负责创建具体的 `WebServer`，用户可以通过 `WebServerFactoryCustomizer` 修改配置（如端口、连接器、SSL），无需继承容器类。
2. **优雅关闭**：嵌入式容器通常随 JVM 一起停止，Spring Boot 2.3+ 支持 `graceful shutdown`（配置 `server.shutdown=graceful`），在关闭时不再接受新请求，等待现有请求处理完毕。这是对传统 `kill -9` 粗糙处理的一大改进。
3. **容器资源隔离**：每个 Spring Boot 应用有自己的类加载器，多个微服务可以运行在同一物理机而不会产生 Tomcat 全局配置冲突，这非常契合微服务架构。

### 6. 事件机制与异步 —— 从同步解耦到最终一致性

**背景（缺什么？）**  
在一个业务操作后往往需要触发其他动作（如发送短信、更新缓存），直接调用会导致模块间强耦合，且性能受影响。需要一个 **发布-订阅模型**，且支持异步甚至事务边界后的通知。

**需要什么？**  
Spring 的 `ApplicationEvent` 机制：发布者只负责发布事件，监听者处理自己的逻辑，并且可以选择同步或异步执行，还可以绑定到事务的某个阶段（提交前/后/回滚后）。

**设计思路**  
- 事件发布：`ApplicationEventPublisher.publishEvent()`，默认同步遍历所有监听器。
- 异步化：在监听器方法上加 `@Async`，并配置 `SimpleApplicationEventMulticaster` 的 `taskExecutor`。
- 事务绑定：`@TransactionalEventListener`，利用 `TransactionSynchronization` 注册回调，在事务提交、回滚等阶段触发。

**巧思**  
1. **避免内存泄漏**：事件监听器如果持有 heavyweight 对象，需要释放。Spring 的 `ApplicationListener` 默认是被容器持有的单例，若监听器订阅了海量不同类型的事件但未正确清理，可能造成内存问题。最佳实践是使用 `@EventListener` 注解且方法内部不保留不必要的引用。
2. **泛型事件**：`ResolvableType` 支持泛型事件类型匹配，例如 `EntityCreatedEvent<User>` 和 `EntityCreatedEvent<Order>` 可以分开监听。
3. **排序与优先级**：多个监听器可以指定 `@Order` 决定执行顺序。
4. **事件与异步结合的风险**：若事务监听器设置为 `AFTER_COMMIT` 并异步执行，此时当前线程已经返回，但异步任务中可能再次查询数据库，需要注意事务边界已结束（需要新事务或使用编程式事务）。
