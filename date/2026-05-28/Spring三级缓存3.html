Spring Framework Architecture Deep Dive

三级缓存的终极真相：
一场为了保全 BeanPostProcessor 的架构博弈

抛弃所有“二级存原始、三级存代理”的表面说辞。 本教程将从 Spring 团队的设计哲学出发，深度推演为什么 AOP 必须依赖 BPP，以及循环依赖是如何差点摧毁 Spring 的生命周期扩展体系的。

1 溯源：为什么创建代理必须放在 BeanPostProcessor 中？

要理解三级缓存，首先必须理解 Spring 最核心的架构成就：IoC 与 AOP 的彻底解耦。 Spring 的 IoC 容器（BeanFactory）在设计之初，完全不知道 AOP 的存在。

如果不用 BPP (强耦合架构)

IoC 容器核心代码 ❌ 硬编码 
public Object createBean() {
    Object bean = instantiate();
    populateBean(bean);
    initializeBean(bean);

    // 致命缺陷：IoC 容器被迫认识 AOP 模块
    if (AopConfig.isNeedProxy(bean)) {
        bean = AopProxyFactory.createProxy(bean);
    }
    return bean;
}

如果这样写，IoC 容器就死死绑定了 AOP。如果以后要加事务、异步、缓存，核心代码会被各种 if-else 撑爆。

Spring 的设计 (BPP 扩展点)

IoC 容器核心代码 ✅ 开闭原则 
public Object createBean() {
    Object bean = instantiate();
    populateBean(bean);
    Object wrapped = initializeBean(bean);

    // 完美解耦：容器只负责遍历回调
    for (BeanPostProcessor bpp : postProcessors) {
        wrapped = bpp.postProcessAfterInitialization(wrapped);
    }
    return wrapped;
}

IoC 容器提供 BeanPostProcessor 接口。AOP 模块自己实现这个接口（AbstractAutoProxyCreator），在回调时偷偷把原始对象换成代理对象。容器一无所知。

核心结论 1

创建代理放在 BPP 中，是为了实现 IoC 与 AOP 的彻底解耦。IoC 提供扩展点，AOP 作为插件接入。这是 Spring 能够无限扩展（事务、异步、校验）的基石。

2 推演：为什么代理必须“延迟”到生命周期的最后一步创建？

既然 BPP 是扩展点，Spring 提供了两个时机：postProcessBeforeInitialization (初始化前) 和 postProcessAfterInitialization (初始化后)。
为什么 Spring 团队坚决把 AOP 代理的创建放在 After (最后一步)？

代理的本质：包装一个“完全成熟”的目标对象

1. 实例化
空壳对象
无属性，无状态
2. 属性填充 + 初始化
半成品对象
有属性，未执行 @PostConstruct
3. 初始化后 (After)
100% 成熟对象
状态完整，可安全被代理

如果提前创建代理会发生什么？

假设在“属性填充”阶段（为了解决循环依赖）就提前创建了代理。此时代理内部的 Target 对象还是个半成品。

  • 状态不一致风险：如果 Target 在 @PostConstruct 中修改了自身状态，或者发布了事件，提前创建的代理可能无法正确拦截或传递这些早期行为。
  • 其他 BPP 的干扰：Spring 允许有多个 BPP。如果 AOP 在早期就创建了代理，后续的 BPP 拿到的就是代理对象。如果某个 BPP 试图通过反射修改 Target 的私有属性，它操作的是代理对象（CGLIB 子类），这会导致修改失效或报错。

Spring 的哲学：让所有的初始化逻辑（Aware回调、Init方法、自定义BPP）都在原始对象上执行。直到一切尘埃落定，再由 AOP 的 BPP 在最后一步套上代理壳。这样最安全、最符合直觉。

3 冲突爆发：循环依赖如何“破坏”了这个完美设计？

现在，我们迎来了终极矛盾。
正常的 AOP 代理在 第 4 步 (初始化后) 创建。
但是循环依赖发生在 第 2 步 (属性填充)！

场景：A 依赖 B，B 依赖 A，且 A 需要 AOP 代理

1. A 实例化 (Raw A)。
2. A 填充属性，发现需要 B。去创建 B。
3. B 实例化 (Raw B)。
4. B 填充属性，发现需要 A。此时 A 还在第 2 步，根本没有走到第 4 步（AOP BPP 还没执行）！
! 死局：B 需要 A 的代理，但 A 的代理按规矩必须在最后才能创建。如果现在给 B 原始 A，AOP 就失效了。

如果只有二级缓存，Spring 团队面临的两个“毒药”选择：

毒药 A：破坏生命周期 (提前创建)

为了解决循环依赖，在 A 的第 2 步（属性填充前），强行调用 AOP 逻辑创建代理，放入二级缓存。

后果： AOP 的创建时机被硬编码提前了。A 的 @PostConstruct 还没执行，代理就创建好了。如果初始化方法里有复杂逻辑，代理包装的就是一个半成品。这直接违背了前面说的“延迟创建”原则。

毒药 B：破坏解耦架构 (IoC 认识 AOP)

IoC 容器在发现循环依赖时，自己去判断是否需要 AOP，并手动调用 AopProxyFactory。

后果： IoC 核心代码里出现了 if (needAop)。BPP 机制被架空，Spring 退化为强耦合的单体框架。

三级缓存：天才的“拖延”战术

Spring 团队没有选择吞下毒药，而是引入了三级缓存（singletonFactories）。 它存的是 ObjectFactory (一个 Lambda 表达式)。这个设计的精妙之处在于：它把“是否提前创建代理”的决定权，推迟到了 B 真正来要 A 的那一瞬间，并且依然通过 BPP 机制来完成！

AbstractAutowireCapableBeanFactory.doCreateBean() ✅ 完美解法 
// 1. 实例化 A (此时 A 是原始对象)
Object bean = createBeanInstance();

// 2. 放入三级缓存 (注意：这里没有创建代理，只是存了一个动作)
addSingletonFactory(beanName, () -> {
    // 这个 Lambda 只有在发生循环依赖，B 来 getBean(A) 时才会执行！
    return getEarlyBeanReference(beanName, mbd, bean);
});

// 3. 属性填充 (如果触发循环依赖，B 会调用上面的 Lambda)
populateBean(beanName, mbd, instanceWrapper);

// 4. 初始化 (执行 @PostConstruct 等)
Object exposedObject = initializeBean(beanName, mbd, bean);

// 5. 正常的 AOP BPP 后置处理 (如果没有循环依赖，代理在这里创建)
for (BeanPostProcessor bpp : postProcessors) {
    exposedObject = bpp.postProcessAfterInitialization(exposedObject, beanName);
}

getEarlyBeanReference 的魔法 (保全架构的关键)

当 B 触发循环依赖，调用三级缓存的 Lambda 时，Spring 执行了 getEarlyBeanReference。这个方法做了什么？

protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
    Object exposedObject = bean;
    // 遍历所有的 SmartInstantiationAwareBeanPostProcessor
    for (BeanPostProcessor bp : getBeanPostProcessors()) {
        if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            SmartInstantiationAwareBeanPostProcessor ibp = (SmartInstantiationAwareBeanPostProcessor) bp;
            // AOP 的 AbstractAutoProxyCreator 在这里拦截，提前创建代理！
            exposedObject = ibp.getEarlyBeanReference(exposedObject, beanName);
        }
    }
    return exposedObject;
}

真相大白： 三级缓存的 Lambda 内部，依然是通过遍历 BPP 来决定是否创建代理的！ IoC 容器依然不知道 AOP 的存在。它只是提供了一个“紧急通道”，让 AOP 的 BPP 能够在循环依赖发生时，合法地提前介入，而不是由 IoC 容器硬编码去创建代理。

终极复盘：三级缓存到底解决了什么？

方案 能否解决循环依赖？ 是否破坏生命周期 (延迟创建原则)？ 是否破坏架构解耦 (BPP机制)？
一级缓存 ❌ 不能 (死循环) - -
二级缓存 (存原始对象) ✅ 能 (但 AOP 失效) ✅ 遵守 ✅ 遵守
二级缓存 (强行存代理) ✅ 能 ❌ 破坏 (被迫提前创建) ❌ 破坏 (IoC 需硬编码判断 AOP)
三级缓存 (Spring 方案) ✅ 能 ✅ 遵守 (无循环依赖时，依然在最后一步创建) ✅ 遵守 (通过 Lambda 回调 BPP，IoC 依然不知情)

三级缓存不是为了“能不能”解决循环依赖，而是 Spring 团队为了在解决这个极端 Bug 的同时，誓死捍卫其生命周期设计原则与 BPP 扩展架构 而做出的最优雅的妥协。
