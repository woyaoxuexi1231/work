# 第4道题：镜像队列的性能陷阱与反模式

> 你们的订单系统在促销高峰期出现消息堆积、P99 延迟飙升，配置是 `ha-mode: all` 且所有消息 `delivery-mode: 2`。请从镜像同步和磁盘 IO 角度解释性能瓶颈。为什么推荐使用 `ha-mode: exactly` 并指定参数而非 `all`？临时队列开启镜像会有什么问题？

---

## 实战类型：代码演示

**为什么选这种方式**：写放大、木桶效应 —— 都是可以精确量化的性能指标。必须在同一集群、同一时刻、同一条连接上压测三组配置，用平均延迟和 P99 的倍数差异来"实锤"。

---

## 演示场景设计

**场景**：促销期订单系统有三条队列，运维给它们配了三种策略：

- `q4.all.bench`  → `ha-mode: all`（每条消息写 3 个节点）
- `q4.ex2.bench`  → `ha-mode: exactly, ha-params: 2`（一主一从）
- `q4.none.bench` → 无镜像（裸奔基线）

同一个 Spring Boot 实例向三条队列各发 200 条消息，测量平均延迟和 P99。

**巧妙之处**：三条队列运行在同一个 3 节点集群上，**唯一变量是镜像数量**。你不换机器、不改网络、不换代码，只是队列名匹配了不同策略。压测结果会赤裸裸展示：`all` 的平均延迟是无镜像的 N 倍，`exactly:2` 几乎接近无镜像。

**"对比实验"设计**：

- **反面（/bench）**：all vs ex2 vs none 三组数据 → 写放大代价一目了然
- **正面（/anti-pattern）**：展示临时队列开镜像的反模式

---

## 核心代码

```java
// 【重点1】三个队列，名称前缀匹配不同策略
rabbitAdmin.declareQueue(new Queue("q4.all.bench",  true, false, false));  // → ha-mode:all
rabbitAdmin.declareQueue(new Queue("q4.ex2.bench",  true, false, false));  // → exactly:2
rabbitAdmin.declareQueue(new Queue("q4.none.bench", true, false, false));  // → 无镜像

// 【重点2】同一次调用、同一连接、同一时刻 → 唯一变量是镜像数量
BenchResult rAll  = measure("q4.all.bench",  200);
BenchResult rEx2  = measure("q4.ex2.bench",  200);
BenchResult rNone = measure("q4.none.bench", 200);

// 【重点3】用平均延迟倍数衡量写放大
// ha-mode:all → 每条消息等 3 个节点全部确认 → 最慢 mirror 拖死延迟
// exactly:2   → 只需等 1 个 mirror → 接近无镜像
double allVsNone = rNone.avgMs > 0 ? (double) rAll.avgMs / rNone.avgMs : 1.0;
```

### 第二步【反面实验】：三组压测

```bash
curl "http://localhost:8080/q4/bench?count=200"
```

**你应该关注的返回结果**：

```json
{
  "ha-mode_all_3副本": {
    "平均_ms": 9,
    "P99_ms (max)": 48
  },
  "exactly_2_1主1从": {
    "平均_ms": 4,
    "P99_ms (max)": 18
  },
  "无镜像_基线": {
    "平均_ms": 3,
    "P99_ms (max)": 10
  },
  "分析": {
    "all_avg_vs_无镜像": "3.0x",
    "ex2_avg_vs_无镜像": "1.3x",
    "all_P99_vs_无镜像": "4.8x",
    "结论": "ha-mode:all 平均延迟是无镜像的 3.0 倍，P99 是 4.8 倍。exactly:2 只慢 1.3 倍，却有备胎。"
  }
}
```

**观察点**：

1. **平均延迟**：无镜像 3ms < ex2 4ms < all 9ms。每多一个镜像副本，延迟成倍增长。
2. **P99 放大**：all 的 P99 是无镜像的 **4.8 倍**。因为 master 必须等**最慢的** mirror 确认。
3. **ex2 的性价比**：只比无镜像慢 1.3 倍，但有备胎 —— 这就是生产推荐配置。

> 如果三组差距不大 —— 可能所有 master 都在同一节点或磁盘性能一致。尝试把 `count` 调到 500 再测。

### 第三步：重复验证稳定性

```bash
for i in 1 2 3; do
  echo "===== 第${i}轮 ====="
  curl -s "http://192.168.3.4:8080/q4/bench?count=150" \
    | python3 -c "import sys,json;d=json.load(sys.stdin);[print(f'{k}: avg={v[\"平均_ms\"]}ms P99={v[\"P99_ms (max)\"]}ms') for k,v in d.items() if isinstance(v,dict) and '平均_ms' in v]"
  sleep 2
done
```

3 轮中 all 的延迟稳定最高 → 写放大是稳定可复现的，不是偶然。

### 第四步【正面实验】：临时队列反模式

```bash
curl http://localhost:8080/q4/anti-pattern
```

核心教训：RPC 回调临时队列（auto-delete，生命周期几秒）如果名字匹配了镜像策略 → 每条消息都做 GM 环广播 → 磁盘 IO 全浪费在马上要删的队列上。

---

## 原理点睛

面试题回答中有三段话被压测数据"实锤"了：

> "`ha-mode: all` 导致每条消息写入所有节点，吞吐量受限于最慢节点的磁盘 IO，写放大严重。"

你的压测：all 平均延迟 = 无镜像的 3.0x。每条 `convertAndSend` 要等 3 个节点全部 fsync。

> "镜像队列的吞吐量取决于最慢的那个镜像。如果其中一台机器磁盘在 GC，整个写入延迟就被拖高。"

all 的 P99=48ms vs 无镜像的 P99=10ms。那多出来的 38ms 很可能就是某个 mirror 的磁盘在吭哧。

> "临时队列开启镜像是典型的反模式。临时队列生命周期极短，镜像同步的开销远远大于它提供的所谓高可用。"

`/q4/anti-pattern` 给出了量化：1000 QPS × 3 副本 = 3000 次磁盘 IO/秒，全花在几秒后就删除的队列上。

---

## 面试要诀

> "我在同一个 3 节点集群上，用同一次压测对比了 ha-mode:all、exactly:2、无镜像。all 平均延迟是无镜像的 3.0 倍，P99 是 4.8 倍。从那以后我在生产只用 exactly:2，绝不用 all。"

面试官听到 "3.0 倍" 和 "4.8 倍" 这两个具体数字，知道你不是在背书。
