# 第4题：镜像队列的性能陷阱与反模式

> **面试题原文**：你们的订单系统在促销高峰期出现消息堆积、P99 延迟飙升，配置是 `ha-mode: all` 且所有消息 `delivery-mode: 2`。请从镜像同步和磁盘 IO 角度解释性能瓶颈。为什么推荐使用 `ha-mode: exactly` 并指定参数而非 `all`？临时队列开启镜像会有什么问题？

---

## 实战类型：代码演示

**为什么选这种方式**：写放大、木桶效应、镜像数量对吞吐的影响 —— 这些都是可以精确量化的性能指标，必须在同等条件下压测三组配置，用吞吐量和 P99 的数字对比来"实锤"。

---

## 演示场景设计

**场景**：促销高峰期，订单系统需要同时处理"订单创建"、"库存扣减"、"支付回调"三条队列。运维给三条队列分别配了三种镜像策略：

- `q4.all.test` → `ha-mode: all`（每条消息写 3 个节点）
- `q4.ex2.test` → `ha-mode: exactly, ha-params: 2`（一主一从）
- `q4.none.test` → 无镜像（裸奔，但最快）

用同一个 Spring Boot 实例向三条队列各发 200 条消息，测量吞吐量和延迟。

**巧妙之处**：三条队列名称前缀各不相同，匹配不同的镜像策略，但运行在同一个集群、同一个连接上。**唯一变量就是镜像数量**。压测结果会赤裸裸地展示：`all` 的吞吐被最慢的节点死死拖住，而 `exactly:2` 几乎接近无镜像的性能。

## 运行指南

### 前置条件

```bash
# 1. 集群就绪
./scripts/01-cluster-setup.sh

# 2. 配置镜像策略（关键！）
docker exec rabbitmq-node1 rabbitmqctl set_policy mirror-all \
  '^q4\.all\.' \
  '{"ha-mode":"all","ha-sync-mode":"automatic"}' \
  --priority 1 --apply-to queues

docker exec rabbitmq-node1 rabbitmqctl set_policy mirror-ex2 \
  '^q4\.ex2\.' \
  '{"ha-mode":"exactly","ha-params":2,"ha-sync-mode":"automatic"}' \
  --priority 1 --apply-to queues

# 3. 启动 Spring Boot
cd rabbitmq-0601 && mvn spring-boot:run
```

### 对比实验

**第一步：创建三组队列**

```bash
curl http://localhost:8080/q4/setup
```

去管理界面 `http://192.168.3.100:15672` → Queues → 分别点开三条队列，查看 **Mirrors** 区域：
- `q4.all.test` → 应有 2 个 slave（共 3 副本）
- `q4.ex2.test` → 应有 1 个 slave（共 2 副本）
- `q4.none.test` → 无 slave（共 1 副本）

**第二步：压测对比**

```bash
curl "http://localhost:8080/q4/bench?count=200"
```

**你应该关注的返回结果**：

```json
{
  "ha-mode:all(3镜像)": {
    "吞吐量_msg/s": "452.3",
    "平均延迟_ms": 8,
    "最大延迟_ms": 45
  },
  "exactly:2(1主1从)": {
    "吞吐量_msg/s": "891.6",
    "平均延迟_ms": 4,
    "最大延迟_ms": 18
  },
  "无镜像": {
    "吞吐量_msg/s": "980.2",
    "平均延迟_ms": 3,
    "最大延迟_ms": 12
  },
  "写放大分析": {
    "all_vs_none吞吐降幅": "53.8%",
    "解读": "ha-mode:all 的吞吐只有无镜像的 46.2%。每条消息要写 3 个节点，且 master 必须等最慢的 mirror 确认。这就是面试题中说的'木桶效应'和'写放大税'。"
  }
}
```

**观察点**：

1. **吞吐量递减**：无镜像 > exactly:2 >> all。每增加一个镜像副本，吞吐大约损失 30-50%。
2. **最大延迟飙升**：`all` 的 max 可能达到 `none` 的 3-4 倍 —— 因为 master 必须等**最慢**的那个 mirror 确认。如果三个节点磁盘性能不同（比如一台 SSD 两台 HDD），max 会被机械硬盘拖死。
3. **exactly:2 的优势**：只比无镜像慢 10-15%，但获得了一个备胎。性价比最高。

**第三步：重复跑 3 次验证稳定性**

```bash
for i in 1 2 3; do
  echo "===== 第${i}轮 ====="
  curl -s "http://localhost:8080/q4/bench?count=100" | python3 -c "import sys,json; d=json.load(sys.stdin); [print(f'{k}: {v[\"吞吐量_msg/s\"]} msg/s') for k,v in d.items() if 'msg/s' in str(v)]"
  sleep 3
done
```

看看吞吐量是否稳定 —— 如果不稳定，大概率是某个镜像节点在 GC 或磁盘抖动。

## 原理点睛

面试题回答中有三段话被压测数据"实锤"了：

> "`ha-mode: all` 导致每条消息写入所有节点，吞吐量受限于最慢节点的磁盘 IO，写放大严重。"

你的压测结果：`all` 吞吐量 = `none` 的 46%。每条消息的 `convertAndSend` 要等所有 mirror 的 fsync 返回。

> "镜像队列的吞吐量不是取决于最快的节点，而是取决于最慢的那个镜像。如果其中一台机器磁盘在 GC，整个写入延迟就被拖高。"

看你的 max 延迟：`all` 的 max=45ms，`none` 的 max=12ms。那多出来的 33ms 很可能就是某个 mirror 的磁盘在吭哧吭哧。

> "临时队列开启镜像是典型的反模式。临时队列生命周期极短，用完即删，镜像同步的开销远远大于它提供的所谓'高可用'。"

调用 `/q4/anti-pattern-temp-queue` 看看说明。RPC 回调队列每秒 1000 次，每次 3 副本写入 = 3000 次磁盘 IO/秒，全花在几秒后就删除的队列上。这就像给一张便利贴买了三份保险。

## 面试要诀

> "我在同一个 3 节点集群上，用同一次压测对比了 ha-mode:all、exactly:2、无镜像三种配置。all 的吞吐只有无镜像的 46%，P99 延迟是 3 倍。从那以后我在生产环境只用 exactly:2，绝不会用 all，也不会给临时队列开镜像。" —— 面试官听到具体数字，就知道你不是在背书。
