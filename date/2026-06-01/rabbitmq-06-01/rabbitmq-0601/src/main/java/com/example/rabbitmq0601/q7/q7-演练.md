# 第7题：代码设计 —— 统一可靠消息组件

> **面试题原文**：请设计一套基于 RabbitMQ 的可靠事件总线组件（EventBus），能够透明适配镜像队列和仲裁队列。要求：发送端支持同步/异步确认、失败自动重试（指数退避）、最终兜底写入数据库；消费端支持幂等、手动 ack、死信队列重试与告警；能够处理连接断开、队列不存在的异常，并自动恢复；组件内集成链路追踪，为每条消息注入 traceId。请画出核心接口和类图，并写出消息发布和消费的伪代码逻辑。

---

## 实战类型：代码演示

**为什么选这种方式**：这道题考的是**软件设计能力**——接口抽象、异常处理、幂等、重试策略。这些抽象概念必须通过可运行的代码来验证，让面试官看到你的设计不是纸上画图，而是能跑起来的工程实现。

---

## 演示场景设计

**场景**：一个电商订单系统，订单创建后发布 `order.created` 事件，下游多个服务（库存、支付、物流）消费该事件。这个事件必须**可靠送达**——不能丢、不能重复处理、失败要重试、最终失败要告警。

**巧妙之处**：`ReliableEventBus` 的实现把面试题 Q7 的 6 个考点全部"一口吞下"，而且你可以在 5 分钟内通过 4 个 HTTP 端点逐个验证：
1. `/q7/publish` → 发布 10 条，看 confirm 状态
2. `/q7/subscribe` → 消费验证幂等
3. `/q7/retry-demo` → 指数退避策略说明
4. `/q7/dead-letter-demo` → 死信队列机制
5. `/q7/full-demo` → 全链路一键演示

## 核心代码解读

### 1. 发送端 —— 发件箱模式 + 异步确认 + 指数退避

```java
// 【重点】先写发件箱，再发 MQ —— 消息绝不凭空消失
OutboxRecord record = new OutboxRecord();
record.msgId = msgId;
record.status = "PENDING";    // ← 先落地
outbox.put(msgId, record);

CorrelationData cd = new CorrelationData(msgId);
sendWithRetry(routingKey, messageBody, cd, 1, record);
return record.future;  // ← 调用方可选择同步等待

// 【重点】指数退避：2s → 4s → 8s → 16s → 32s
long delay = (long) Math.pow(2, attempt) * 1000;
scheduler.schedule(() -> sendWithRetry(..., attempt + 1, record), delay, ms);

// 【重点】ConfirmCallback：ack=true → 消息已落地，更新发件箱
record.confirmed = ack;
record.status = ack ? "SENT" : "NACKED";
```

### 2. 兜底扫描 —— confirm 超时救场

```java
// 【重点】每 10 秒扫描发件箱，PENDING 超过 30 秒 → 重新投递
// 解决：confirm 超时但消息实际已写入的边界情况
if ("PENDING".equals(record.status) && (now - record.createTime) > 30_000) {
    sendWithRetry(..., 1, record);  // 重新投递（幂等消费兜底）
}
```

### 3. 消费端 —— 幂等 + traceId 链路追踪 + 死信

```java
// 【重点】幂等检查：同一 messageId 只处理一次
if (processedIds.contains(event.messageId)) {
    log.info("重复消息，直接 ACK");
    return;  // ← 已经处理过，丢弃
}

// 【重点】traceId 串联全链路
MDC.put("traceId", event.traceId);
try {
    handler.handle(event);            // 业务处理
    processedIds.add(event.messageId); // 记录去重
} catch (Exception e) {
    // 【重点】失败 > 3 次 → 死信队列
    if (retries > 3) {
        // 消息通过 x-dead-letter-exchange 自动进入 q.dead.letter
        log.error("[DEAD_LETTER] msgId={} 进入死信", event.messageId);
    }
}
```

### 4. 队列声明 —— 透明适配镜像/仲裁

```java
// 【重点】一行参数切换镜像 vs 仲裁
Map<String, Object> args = new HashMap<>();
if ("quorum".equals(queueType)) {
    args.put("x-queue-type", "quorum");  // ← 仲裁队列
}
// 镜像队列只需匹配 ha-policy，不需要 x-queue-type
args.put("x-dead-letter-exchange", DLX);  // 死信配置
args.put("x-dead-letter-routing-key", "dead");
```

## 原理点睛

面试题回答中有四段话被你这段代码"实锤"了：

> "消息到达 publish 方法，首先写入 outbox 表（状态 PENDING），然后异步发送到 RabbitMQ。confirm 回调触发时更新为 SENT。"

代码第 92-104 行：先创建 `OutboxRecord`（status=PENDING），放入 `outbox` Map，再 `sendWithRetry`。ConfirmCallback 里 `record.status = "SENT"`。

> "confirm 超时或连接异常，进入重试循环（指数退避：1s, 2s, 4s... 上限 1 分钟）。每次重试用同一 messageId 重发。"

代码第 127-145 行：`delay = 2^attempt * 1000`，最多 5 次，通过 `scheduler.schedule` 延迟执行。

> "后台定时任务扫描 PENDING 超过一定时间的记录，重新投递，防止极端遗漏。"

代码第 105 行：`scheduler.scheduleWithFixedDelay(this::scanOutbox, 10, 10, SECONDS)`，每 10 秒扫描 `outbox`。

> "消费端取出 messageId，查询去重表，如果已存在直接 basicAck。处理成功记录去重键。"

代码第 190-196 行：`processedIds.contains(event.messageId)` 检查，`processedIds.add(event.messageId)` 记录。
