# 第6题：跨机房集群架构与网络分区处理

> **面试题原文**：公司业务国际化，要求在同城双机房部署 RabbitMQ 集群，单机房故障时消息服务整体切换时间 < 30s，且消息零丢失。镜像队列在跨机房（延迟 5-10ms）下出现严重性能抖动。请设计一套基于仲裁队列的多机房架构，并说明节点如何分布以满足"容忍一个机房整体故障"。网络分区发生时你会使用哪种处理策略？为什么？客户端如何自动切换到备用机房？

---

## 实战类型：操作文档 + 代码演示（混合）

**为什么选这种方式**：
- 模拟网络分区、观察 `pause_minority` → **操作文档**：必须通过 `docker network` 制造分区并观察节点行为

## 第一部分：操作文档 —— 模拟网络分区

### 操作背景（苦难时代）

你的集群跨了两个机房。一天，两个机房之间的光纤被挖断，节点 A 和 B、C 之间出现了网络分区。镜像队列时代，RabbitMQ 会选一个"获胜方"继续服务，**另一方全部暂停**，等网络恢复后**全量同步**（丢数据+阻塞队列）。你看着监控图上分裂的集群，手心冒汗。

现在换了仲裁队列 + `pause_minority` 策略：只有包含多数派（≥2 节点）的分区才能继续服务。少数派自动暂停，不产生脑裂，不会写出两份不一致的数据。

### 操作目标

用 `docker network disconnect` 人为制造 3 节点集群的分区，观察 `pause_minority` 策略下：少数派节点自动暂停，多数派继续服务，分区恢复后自动愈合。

### 环境准备

| 资源 | 说明 |
|---|---|
| 集群 | `01-cluster-setup.sh` 已运行，3 节点正常 |
| 策略 | 需要开启 `pause_minority` |

### 详细操作步骤

#### 步骤 1：开启 pause_minority

```bash
# 在 node1 上配置集群分区处理策略
docker exec rabbitmq-node1 rabbitmqctl set_cluster_partition_handling pause_minority

# 验证配置
docker exec rabbitmq-node1 rabbitmqctl cluster_status | grep -A2 "partition"
```

> **【原理验证点 1】** `pause_minority` 的含义：如果当前节点发现自己不在多数派分区中，就自动暂停服务。这和仲裁队列的 Raft 机制配合得天衣无缝 —— Raft 本身也需要多数派才能选举 Leader。

#### 步骤 2：创建仲裁队列并发布消息

```bash
# 创建仲裁队列
docker exec rabbitmq-node1 rabbitmqadmin declare queue \
  name=q6.partition.test durable=true \
  arguments='{"x-queue-type":"quorum"}'

# 发布 50 条消息
for i in $(seq 1 50); do
  docker exec rabbitmq-node1 rabbitmqadmin publish \
    routing_key=q6.partition.test \
    payload="PARTITION-TEST-$i"
done

echo "消息已发布，当前队列状态:"
docker exec rabbitmq-node1 rabbitmq-queues members q6.partition.test
```

**预期输出**：
```
Members of quorum queue 'q6.partition.test':
  rabbit@rabbitmq-node1 (leader)
  rabbit@rabbitmq-node2 (follower)
  rabbit@rabbitmq-node3 (follower)
```

#### 步骤 3：制造网络分区

**核心操作**：把 node3 从 Docker 网络中踢出去，让它和 node1、node2 形成分区。

```bash
# node3 脱离集群网络 → 形成分区
docker network disconnect rmq-cluster-net rabbitmq-node3

echo "等待分区检测..."
sleep 5
```

#### 步骤 4：观察分区行为

```bash
# 从多数派 (node1 + node2) 查看集群状态
echo "===== 多数派 (node1) 视角 ====="
docker exec rabbitmq-node1 rabbitmqctl cluster_status

echo ""
echo "===== 少数派 (node3) 视角 ====="
docker exec rabbitmq-node3 rabbitmqctl cluster_status 2>&1 || echo "(node3 可能因 pause_minority 暂停)"
```

> **【原理验证点 2】** node3 发现自己变成了孤家寡人（1/3 < 多数派 2/3），`pause_minority` 生效：node3 自动暂停服务，拒绝所有客户端请求。而 node1 + node2 的多数派继续正常工作。

**验证多数派仍可写入**：

```bash
# 从 node1 继续发送消息到仲裁队列
docker exec rabbitmq-node1 rabbitmqadmin publish \
  routing_key=q6.partition.test \
  payload="AFTER-PARTITION-TEST"

# 消费一条验证
docker exec rabbitmq-node1 rabbitmqadmin get \
  queue=q6.partition.test ackmode=ack_requeue_false
```

预期：message 正常返回。**多数派（2/3）存活，仲裁队列 Raft 继续服务。**

#### 步骤 5：恢复分区

```bash
# 把 node3 接回网络
docker network connect rmq-cluster-net rabbitmq-node3

echo "等待分区愈合..."
sleep 10

# 验证集群恢复
docker exec rabbitmq-node1 rabbitmqctl cluster_status
```

> **【原理验证点 3】** node3 重新加入后，它从 Leader 快照 + 后续日志增量追赶，**不阻塞队列读写**。对比镜像队列的全量同步阻塞，这就是 Raft 增量追赶的优雅之处。

### 验证客户端自动切换

```bash
# 开启一个持续发送的窗口
curl "http://localhost:8080/q6/stress-with-failover?count=20"
```

**同时**，在发送第 5-10 条之间，手动执行：

```bash
docker stop rabbitmq-node1  # 杀掉当前连接的节点
```

**观察返回结果**：

```json
[
  {"seq": 5, "status": "OK",    "latency_us": 3200},
  {"seq": 6, "status": "OK",    "latency_us": 2800},
  {"seq": 7, "status": "ERROR", "error": "Connection refused"},
  {"seq": 8, "status": "ERROR", "error": "Connection refused"},
  {"seq": 9, "status": "OK",    "latency_us": 85000},    ← 重连成功，但延迟突增
  {"seq":10, "status": "OK",    "latency_us": 4500},
  ...

```

> **【原理验证点 4】** 第 7-8 条 ERROR（node1 宕机），第 9 条恢复但延迟突增 85ms（自动重连 + 建立新通道）。这就是面试题中"客户端需配置多地址 + 自动故障切换"—— Spring Boot 的 `addresses` 配置就是这个机制的实现。

## 原理点睛

面试题回答中有三段话被你刚才的操作"实锤"了：

> "仲裁队列基于 Raft，只有包含多数派的分区能选举出 Leader 继续服务，少数派分区自动暂停。这比镜像队列的分区处理更优雅。"

你刚才的操作：node3（少数派）自动 pause，node1+node2（多数派）继续工作。如果是镜像队列，少数派节点直接被停掉，恢复后还要全量同步。

> "客户端配置多集群连接地址，主集群不可达时根据连接异常快速尝试备用集群。为防止消息重复，每条消息携带全局唯一 messageId。"

你看到的 `/q6/stress-with-failover` 中 ERROR→OK 的恢复过程，就是客户端自动切换到存活节点。messageId 在 Q7 中实现。

> "如果主站 3 节点整体物理毁灭，备站 2 节点无法自动提升为多数派，需要人工执行命令调整成员。"

你的分区实验中，如果改成 2+1（node1 + node2 在分区一侧，node3 在另一侧），少数派 (1/3) 只有 node3，它自动 pause。这就是 3 节点跨机房（主 2 + 备 1）的局限性。
