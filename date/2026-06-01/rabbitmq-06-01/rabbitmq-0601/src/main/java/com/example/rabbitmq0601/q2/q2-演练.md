# 第2题：集群模式下的消息路由与性能陷阱

> **面试题原文**：你的 RabbitMQ 集群有两个节点 A 和 B，生产者连接到节点 A 发送消息到交换器，但消息路由的目标队列 master 在节点 B 上。请描述消息从生产者到达消费者的完整路径。这种跨节点消息转发会带来什么性能影响？如果大量生产者连接到一个节点，而队列 master 都在另一个节点，会出现什么瓶颈？你如何通过客户端连接策略或队列主节点分布来优化？

---

## 实战类型：代码演示

**为什么选这种方式**：跨节点路由的性能代价是"多了一跳集群内部网络传输"，这种延迟差异必须通过**压测数据**来赤裸裸展现 —— P50/P99 延迟的数字对比，比任何文字描述都更有说服力。

---

## 演示场景设计

**场景**：一个订单系统有两条队列 —— `q2.local.bench` 和 `q2.remote.bench`。集群自动把它们的 master 分配到不同节点。然后从同一个 Spring Boot 实例向两条队列各发 100 条消息，测量每次 `convertAndSend` 的端到端延迟。

**巧妙之处**：这个实验利用了 RabbitMQ 的默认 `queue-master-locator` 策略 —— 多个队列的 master 会被"尽量分散"到不同节点。你不需要手动指定哪条队列在哪个节点，集群自己就把差异制造出来了。然后你只需要对比两条队列的 P99 延迟，跨节点路由的代价就会在数字里"跳出来"。

## 核心代码

```java
// 【重点1】测量单次 convertAndSend 的端到端延迟
long start = System.nanoTime();
rabbitTemplate.convertAndSend("", queueName, "BENCH-" + i);
long elapsed = System.nanoTime() - start;
latencies.add(TimeUnit.NANOSECONDS.toMillis(elapsed));

// 【重点2】排序后取分位数 —— P99 最能暴露跨节点延迟
Collections.sort(latencies);
long p50 = latencies.get(count / 2);
long p99 = latencies.get((int) (count * 0.99));

// 【重点3】对比结果 —— 跨节点 P99 通常比本地高 1-5ms
// 这就是"多了一跳集群内部网络传输"的代价
```

### 对比实验

**实验 A：创建队列 + 查看 master 分布**

```bash
curl http://localhost:8080/q2/setup
```

然后打开管理界面 `http://192.168.3.100:15672` → Queues → 分别点开 `q2.local.bench` 和 `q2.remote.bench`，查看各自的 **Node** 字段。

如果两个队列恰好在同一节点，删除重建即可让集群重新分配：

```bash
curl http://localhost:8080/q2/cleanup
curl http://localhost:8080/q2/setup
```

**实验 B：运行压测对比**

```bash
curl http://localhost:8080/q2/bench
```

**你应该关注的返回结果**：

```json
{
  "queueA_q2.local.bench": {
    "p50_ms": 2,
    "p99_ms": 5,
    "avg_ms": "2.3"
  },
  "queueB_q2.remote.bench": {
    "p50_ms": 4,
    "p99_ms": 11,
    "avg_ms": "4.1"
  },
  "跨节点额外延迟_ms": {
    "p50_diff": 2,
    "p99_diff": 6,
    "解释": "跨节点路由比本地路由 P99 慢 6ms —— 这就是多了一跳集群内部转发的代价"
  }
}
```

**观察点**：
1. **P50 差异**：通常 1-3ms，这是"集群内部网络传输 + 序列化/反序列化"的固定开销
2. **P99 差异**：可能更大（5-15ms），因为包含了网络抖动、GC 停顿等
3. 如果 `cross_node_p99` 和 `local_p99` 差异很小（<1ms），说明两个队列恰好 master 在同一节点 —— 这正是面试题说的"不确定性"，也是为什么你需要主动优化

**实验 C：重复跑 5 次，观察波动**

```bash
for i in $(seq 1 5); do
  echo "===== 第${i}次 ====="
  curl -s http://localhost:8080/q2/bench | python3 -m json.tool | grep -A2 p99
  sleep 2
done
```

对比 5 次结果中 `remote_p99` 和 `local_p99` 的差值波动 —— 网络不是稳定的，跨节点延迟会抖动。

## 原理点睛

面试题回答中有两段话被这段代码"实锤"了：

> "消息路径：生产者连节点 A → 交换器查路由键 → 目标队列 master 在 B → 节点 A 将消息通过集群内部连接转发到 B → 写入队列。跨节点入队的 P99 延迟通常比本地高 1-3ms，极端网络抖动时更大。"

你的代码跑出来的 P50_diff 和 P99_diff 就是对这段文字的数据化证明。`rabbitTemplate.convertAndSend` 的完成时间包含了从你的 Spring Boot → 连接节点 → 跨节点转发 → master 写入的全过程。

> "如果大量生产者都连到节点 A，而队列 master 全在节点 B，节点 A 会把所有流量甩到 B，B 的 CPU 很容易被打满，A 却比较空闲。"

试想一下：如果你把 100 条消息全部发到同一条"在另一节点"的队列，监控图上的表现就是：连接节点 CPU 30%，master 节点 CPU 95%。这就是负载倾斜。

---

## 面试要诀

> "我在 3 节点集群上做过压测：往 master 在本节点的队列发消息 P99 是 5ms，往 master 在另一节点的队列发消息 P99 是 11ms，差了 6ms。这就是 `queue-master-locator: client-local` 策略存在的意义 —— 让生产者和队列 master 在同一节点，省掉集群内部转发。" —— 这一句话，面试官就知道你**量过**，不是猜的。
