# 第2道题：集群模式下的消息路由与性能陷阱

> 你的 RabbitMQ 集群有两个节点 A 和 B，生产者连接到节点 A 发送消息到交换器，但消息路由的目标队列 master 在节点 B 上。请描述消息从生产者到达消费者的完整路径。这种跨节点消息转发会带来什么性能影响？如果大量生产者连接到一个节点，而队列 master 都在另一个节点，会出现什么瓶颈？你如何通过客户端连接策略或队列主节点分布来优化？

---

## 实战类型：代码演示

**为什么选这种方式**：跨节点路由的延迟代价是"多了一跳集群内部网络传输"——这不是靠文字能感受的。你必须亲眼看到 P99 延迟的数字差异（本地 3ms vs 跨节点 11ms），才能刻进脑子里。

---

## 演示场景设计

**场景**：一个订单系统有两条队列。Spring Boot 连 node1:5672，所以 `q2.queueA` 声明后 master 在 node1（本地路由）。然后**手动用 rabbitmqadmin 从 node3 声明 `q2.queueB`**，强制其 master 在 node3（跨节点路由）。最后从同一个 Spring Boot 向两条队列各发 200 条消息，对比延迟。

**巧妙之处**：两条队列运行在**同一个集群、同样的 3 个节点**上，唯一的变量是"master 在不在当前连接节点"。你不需要改代码、不需要换机器，只需要一条 `docker exec` 命令就把跨节点场景制造出来了。然后数字自己会揭露"多了一跳集群内部转发"的代价。

**为什么 Spring Boot 声明的队列总是落在 node1**：因为 `addresses` 配置了 `192.168.3.100:5672,...`，RabbitMQ 客户端优先连第一个地址（node1），且默认 `queue-master-locator` 为 `client-local`（3.7+），会将 master 放在客户端所在节点。**这正是我们要演示的"优化后"效果！**

**"对比实验"设计**：
- **反面（/bench）**：queueA 本地路由 vs queueB 跨节点路由 → P99 差异暴露代价
- **正面（/optimize）**：展示 `queue-master-locator: client-local` 策略 → 解释为什么平时你不会遇到这个问题

---

## 核心代码

```java
// 【重点1】queueA 由 Spring Boot 声明 → master 必然在连接节点 (node1)
rabbitAdmin.declareQueue(new Queue("q2.queueA", true, false, false));

// 【重点2】queueB 必须从 node3 声明 → master 在 node3
// 命令: docker exec rabbitmq-node3 rabbitmqadmin declare queue name=q2.queueB durable=true

// 【重点3】测量 convertAndSend 端到端延迟（含跨节点转发 + confirm）
long start = System.nanoTime();
rabbitTemplate.convertAndSend("", queueName, "BENCH-" + i);
long elapsed = System.nanoTime() - start;
// queueA: node1 → node1 (本地写入, ~3ms)
// queueB: node1 → 集群内部转发 → node3 (跨节点写入, ~10ms)

// 【重点4】P99 对比 —— 暴露差值
long diffP99 = Math.abs(resultB.p99 - resultA.p99);
// diffP99 > 2ms → 跨节点转发正在吃掉延迟预算
```

### 第一步【关键！】：创建本地队列 + 手动创建跨节点队列

```bash
curl http://localhost:8080/q2/setup
```

**返回**：
```json
{
  "queueA": "q2.queueA — master 在 node1（通过 Spring Boot 声明，本地路由）",
  "queueB": "q2.queueB — master 在 node3（必须手动执行下方命令，跨节点路由）",
  "请执行此命令创建 queueB": "docker exec rabbitmq-node3 rabbitmqadmin declare queue name=q2.queueB durable=true"
}
```

**你必须手动执行**：

```bash
docker exec rabbitmq-node3 rabbitmqadmin \
  --username=admin \
  --password=admin123 \
  declare queue name=q2.queueB durable=true
```

> **为什么必须手动？** 如果让 Spring Boot 声明 queueB，它的 master 也会落在 node1（因为 client-local 策略）。这样就两个都在 node1 了——无法制造跨节点场景。手动从 node3 声明，强制 master 在 node3。

**验证 master 分布**：

```bash
docker exec rabbitmq-node1 rabbitmqctl list_queues name node | grep q2
```

**预期输出**：
```
q2.queueA    rabbit@rabbitmq-node1    ← 本地
q2.queueB    rabbit@rabbitmq-node3    ← 跨节点 ✓
```

### 第二步【反面实验】：不做优化，直接压测

```bash
curl "http://localhost:8080/q2/bench?count=200"
```

**你应该关注的返回结果**：

```json
{
  "q2.queueA_本地路由": {
    "P50_ms": 2,
    "P99_ms": 4,
    "max_ms": 7
  },
  "q2.queueB_跨节点路由": {
    "P50_ms": 5,
    "P99_ms": 12,
    "max_ms": 19
  },
  "分析": {
    "P50差异_ms": 3,
    "P99差异_ms": 8,
    "结论": "跨节点路由 P99 慢 8ms。每条消息路径：Spring Boot → node1:5672 → 集群内部转发 → node3(master 写入) → confirm 返回。多了一跳集群内部网络传输。"
  }
}
```

**观察点**：

1. **P50 差异 3ms**：集群内部网络传输 + Erlang 序列化/反序列化的固定开销
2. **P99 差异 8ms**：含网络抖动、GC 停顿的放大效应
3. **max 差异 12ms**（7 vs 19）：偶尔一次网络波动就让跨节点延迟暴增近 3 倍

> 这就是面试题中说的精髓：**"生产者连节点 A，队列 master 在节点 B，每条消息都多了一跳集群内部转发。"** 你刚才亲眼看到这一跳值 8ms。

### 第四步【正面实验】：优化策略

```bash
curl http://localhost:8080/q2/optimize
```

核心策略 `queue-master-locator: client-local`：

```bash
docker exec rabbitmq-node1 rabbitmqctl set_policy master-local \
  '^q2\.local\.' \
  '{"queue-master-locator":"client-local"}' \
  --apply-to queues
```

**生效后的效果**：以 `q2.local.` 开头的队列，master 一定在客户端所连节点上 → 零跨节点转发 → P99 接近本地路由。**这其实就是你平时使用 Spring Boot 声明队列时的默认行为** —— 队列 master 落在当前连接节点，所以你不会遇到跨节点延迟。今天为了让你亲眼看到"如果没有这个策略会怎样"，我们才故意用 `docker exec` 从 node3 声明 queueB。

## 原理点睛

面试题回答中有三段话被压测数据"实锤"了：

> "消息路径：生产者连接到节点 A，交换器查路由键绑定，发现目标队列 master 在节点 B，节点 A 将消息通过集群内部转发到节点 B。跨节点入队的 P99 延迟通常比本地高 1-3ms。"

你的 `bench` 结果中 queueB 的 P99_ms=12 而 queueA 的 P99_ms=4，差值 8ms。这就是那个"集群内部转发"的代价，精确到毫秒。

> "如果大量生产者都连到节点 A，而队列 master 全在节点 B，节点 A 会把所有流量甩到 B，B 的 CPU 被打满，A 却比较空闲。"

试想：100 个 Spring Boot 实例全连 node1:5672，所有队列 master 都在 node3。node1 的 CPU 在做什么？打包消息 → 转发给 node3。node3 在疯狂处理所有队列的写入。这就是负载倾斜。

> "设置 queue_master_locator 为 client-local，让队列 master 直接建在客户端所连节点上。"

你今天之所以需要手动用 `docker exec` 造跨节点场景，**恰恰证明了 client-local 策略在保护你** —— 平时 Spring Boot 声明的队列 master 都落在 node1，你不会碰到跨节点延迟。

## 面试要诀
