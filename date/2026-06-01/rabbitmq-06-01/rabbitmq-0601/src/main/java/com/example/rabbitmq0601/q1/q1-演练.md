# 第1题：RabbitMQ 集群基础与节点元数据同步

> **面试题原文**：你现在要为公司搭建一个三节点的 RabbitMQ 集群。请你描述集群启动时节点之间如何发现彼此？集群的元数据是如何在各节点间同步的？为什么集群模式下队列的消息内容默认只存储在主节点上？如果你把一个普通队列的 master 节点直接关机，该队列能否在其他节点上继续提供服务？为什么？

---

## 实战类型：操作文档

**为什么选这种方式**：集群的节点发现、元数据同步、master 宕机后队列不可用 —— 这些都是系统行为，必须通过执行命令、观察管理界面、杀节点来亲眼验证，不是单靠代码能展现的。

---

## 操作背景（苦难时代）

早年大家都用单机 RabbitMQ，直到有一天机房断电，消息系统瞬间停摆。于是你想"多搞几台机器组集群，一台挂了另一台顶上"。你跑完 `join_cluster`，看着 `cluster_status` 显示 3 个节点，心满意足。然后你把订单队列的 master 所在节点关机测试 —— 订单消息全部卡死，生产端疯狂报错。你盯着监控图，消息入队速率从 2000/s 断崖式跌到 0。**你这才发现：集群只是共享元数据，队列数据还锁在宕机节点本地。**

---

## 操作目标

亲手搭建 3 节点集群，然后**杀掉一个普通队列的 master 节点**，亲眼看到队列不可用，深刻理解"元数据共享 ≠ 数据冗余"。

---

| 资源 | 说明 |

| 机器 | 192.168.3.100 Ubuntu + Docker |

| Spring Boot | `rabbitmq-0601` 项目 (端口 8080) |

---

## 详细操作步骤

### 步骤 1：部署 3 节点集群

```bash
cd scripts/
chmod +x 01-cluster-setup.sh
./01-cluster-setup.sh
```

**预期输出**：
```
[5/5] 验证集群状态
Running Nodes: rabbit@rabbitmq-node1, rabbit@rabbitmq-node2, rabbit@rabbitmq-node3
✅ 集群搭建完成!
```

打开管理界面 `http://192.168.3.100:15672` → 点击 **Nodes** 标签 → 看到 3 个节点，状态均为绿色 ✓。

> **【原理验证点 1】** 你看到的是 Mnesia 数据库里的节点列表被复制到了所有节点。每当你 `join_cluster`，新节点的 cookie 被校验通过后，集群把成员列表广播给所有人 —— 这就是"兄弟会的公告栏"：每个节点都知道有谁在会里。

---

### 步骤 2：声明一个普通队列并发布消息

启动 Spring Boot 项目，调用接口创建队列：

```bash
# 终端 1：启动 Spring Boot
cd rabbitmq-0601/
mvn spring-boot:run
```

```bash
# 终端 2：声明队列
curl http://localhost:8080/q1/declare

# 发布 10 条消息
curl http://localhost:8080/q1/publish
```

**预期输出**：
```json
{"queue":"q1.normal.test","declared":"q1.normal.test","published":10}
```

---

### 步骤 3：定位队列的 master 节点

回到管理界面 → **Queues** 标签 → 找到 `q1.normal.test` → 点击进去 → 查看 **Node** 字段。

或者在命令行查看：

```bash
docker exec rabbitmq-node1 rabbitmqctl list_queues name node --formatter=table | grep q1
```

**预期输出**：
```
q1.normal.test    rabbit@rabbitmq-node1
```

这意味着该队列的 master 在 node1。**注意**：你的队列可能在 node2 或 node3，取决于集群分配策略，记住它！

> **【原理验证点 2】** 队列的 `Node` 列就是它的 master。所有发往这个队列的消息，不管你的生产者连了哪个节点，最终都会被转发到这个 master 节点上处理。

---

### 步骤 4：杀死 master 节点 —— 亲眼看到灾难

```bash
# 假设 master 在 node1（替换为你实际看到的节点）
docker stop rabbitmq-node1
```

等待 5 秒，然后测试队列可用性：

```bash
curl http://localhost:8080/q1/test-availability
```

**预期输出**：
```json
{
  "available": false,
  "error": "... connection refused ...",
  "verdict": "✗ 队列不可用! master 宕机 → 消息锁在宕机节点磁盘上 → 其他节点救不了你 → 这就是为什么需要镜像队列/仲裁队列"
}
```

> **【原理验证点 3】** 这就是面试题 Q1 的终极答案。虽然 node2 和 node3 的 Mnesia 里还有 `q1.normal.test` 的元数据（队列名、属性），但**消息内容（Message Store）在 node1 的本地磁盘上**。Erlang 进程随容器终止，其他节点无法访问 node1 的磁盘文件。集群给了你"逻辑统一视图"，但没给你"数据冗余"。

---

### 步骤 5：对比 —— 元数据还在，数据没了

```bash
# 从存活节点查看元数据
docker exec rabbitmq-node2 rabbitmqctl list_queues name state --formatter=table | grep q1
```

输出：
```
q1.normal.test    down
```

看到了吗？队列的**元数据**显示 `down`，RabbitMQ 知道有这么一个队列，但它已经不可用了。就像兄弟会的公告栏上写着"302 室已坍塌"，但你拿不到里面的任何家具。

---

## 结果对比

| 操作 | 操作前 | 操作后 |
|---|---|---|
| 集群状态 | 3 节点 running | 2 节点 running，1 节点 down |
| 队列 `q1.normal.test` | 可用，有 10 条消息 | **不可用**，状态 `down` |
| 队列元数据 | 在 3 个节点 Mnesia 中 | 仍在 2 个存活节点 Mnesia 中 |
| 消息内容 | 在 master (node1) 磁盘 | **锁在 node1 磁盘，无法访问** |
| 消费者 | 正常消费 | `channel error`，连接断开 |

---

## 原理点睛

这道面试题回答中有两段话，刚好被你刚才的操作"实锤"了：

> "集群本质上是一个'元数据复制群组'，用最小的同步开销换来逻辑上的统一视图。队列的 master 节点宕机，队列就不可用。持久化只是把消息写到硬盘，而硬盘在宕机节点上，拔不出来。"

你刚才的操作完整复现了这段话：元数据确实在所有节点可见（`list_queues` 能查到），但消息锁在 node1 磁盘上，其他节点碰不到。

> "集群是共享元数据的兄弟会 —— 公告栏谁都看得见，但私人物品绝不混用。"

你看到的 `q1.normal.test down` 就是公告栏上的"302 室已坍塌"。

---

## 面试要诀

> 如果在面试中被问到 Q1，说一句："我在 3 节点 Docker 集群上实际测试过 —— 普通队列的 master 宕机后，其他节点上 `rabbitmqctl list_queues` 显示状态 `down`，消费者直接报 channel error。这让我刻骨铭心地理解了'集群只复制元数据、不复制数据'。" —— 面试官立刻知道你不是纸上谈兵。
