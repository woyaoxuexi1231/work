# 第1道题：RabbitMQ 集群基础与节点元数据同步

> 你现在要为公司搭建一个三节点的 RabbitMQ 集群。请你描述集群启动时节点之间如何发现彼此？集群的元数据（如交换器、队列、绑定关系）是如何在各节点间同步的？为什么即使在集群模式下，队列的消息内容默认只存储在其主节点上？如果你把一个普通队列的 master 节点直接关机，该队列能否在其他节点上继续提供服务？为什么？

---

## 实战类型：操作文档

**为什么选这种方式**：集群节点发现、Mnesia 元数据复制、master 宕机后队列 `down` —— 这些都是**系统运行时行为**，不是代码逻辑。你必须亲手执行 `join_cluster`、`docker stop`、`rabbitmqctl list_queues`，用眼睛看到"公告栏还在，东西却拿不到了"的瞬间，才能真正理解"元数据共享 ≠ 数据冗余"。

---

## 操作背景（苦难时代）

早年大家都用单机 RabbitMQ。一台机器跑得风生水起，直到有一天机房断电，消息系统瞬间停摆。于是你想"组个集群，一台挂了另一台顶上"。你跑完 `join_cluster`，看着 `cluster_status` 显示 3 个节点，心满意足。

然后你把订单队列所在的节点关机测试。瞬间，所有生产者报 `channel error`，消费者断连，消息入队速率从 2000/s 断崖式跌到 0。你盯着另一台正常节点的监控图 —— CPU 毫无波澜，因为它只是帮忙存了点元数据。故障持续 4 小时，直到运维手动恢复。

**你这才发现：集群只是共享元数据的兄弟会。公告栏谁都看得见，但私人物品绝不混用。**

---

## 操作目标

在 3 节点 Docker 集群上声明一个普通队列，发布消息，然后 **`docker stop` 杀掉它的 master 节点**，亲眼看到队列变成 `down` 状态、消息无法消费。用血淋淋的事实刻进肌肉记忆：集群不复制数据。

## 详细操作步骤

### 步骤 1：启动 Spring Boot 项目

```bash
cd rabbitmq-0601
mvn spring-boot:run
```

看到 `Started Rabbitmq0601Application` 即就绪。

---

### 步骤 2：声明一个普通队列并发布消息

```bash
# 声明普通队列（非镜像、非仲裁）
curl http://localhost:8080/q1/declare
```

**预期输出**：
```json
{
  "queue": "q1.normal.test",
  "declared": "q1.normal.test",
  "tip": "元数据已同步到 3 个节点，但消息内容只存 master 本地"
}
```

```bash
# 发布 10 条消息
curl http://localhost:8080/q1/publish
```

```json
{"published": 10, "queue": "q1.normal.test"}
```

> **【原理验证点 1】** 当你调用 `/q1/declare` 时，Spring Boot 通过 `RabbitAdmin.declareQueue()` 向集群中的某个节点（比如 node1:5672）声明队列。这个队列的**元数据**（名称、是否持久化、是否排他）通过 Mnesia 数据库立即复制到 node2 和 node3。但队列的 Erlang 进程只跑在 master 节点上。

---

### 步骤 3：定位队列的 master 节点

打开管理界面 → **Queues** 标签 → 找到 `q1.normal.test` → 查看 **Node** 列。

记下 master 是哪个节点（可能是 node2 或 node3，取决于集群分配）。

> **【原理验证点 2】** 这里的 `rabbit@rabbitmq-node1` 就是 Queue Master 的 Erlang 节点名。所有发往这个队列的消息，不管你的生产者连了 5672/5673/5674 哪个端口，最终都会被集群内部转发到这个 master 进程处理。

---

### 步骤 4：杀掉 master —— 亲手触发灾难

```bash
# 假设 master 在 node1（换成你在步骤 3 看到的节点）
docker stop rabbitmq-node1

# 等 5 秒让集群感知到节点离开
sleep 5
```

---

### 步骤 5：验证队列不可用

```bash
curl http://localhost:8080/q1/test-availability
```

**预期输出**：
```json
{
  "available": false,
  "error": "... connection refused ...",
  "verdict": "✗ 队列不可用! master 宕机 → 消息锁在宕机节点磁盘上 → 其他节点救不了你 → 这就是为什么需要镜像队列/仲裁队列"
}
```

> **【原理验证点 3】** 这就是这道面试题的终极答案。node2 和 node3 的 Mnesia 里还有 `q1.normal.test` 的元数据（名称、持久化属性），但消息内容在 node1 的本地磁盘上。Erlang 进程随容器一起死了，其他节点无法访问 node1 的磁盘文件。**集群给你逻辑统一视图，没给你数据冗余。**

---

### 步骤 6：对比 —— 元数据还在，数据没了

从存活节点查看队列状态：

```bash
docker exec rabbitmq-node2 rabbitmqctl list_queues name state --formatter=table | grep q1
```

**预期输出**：
```
q1.normal.test    down
```

看到了吗？队列的**元数据**还在（名字被 Mnesia 复制到了 node2），但状态是 `down`。RabbitMQ 知道有这么一个队列，但它已经不可用了。这就像兄弟会的公告栏上写着"302 室已坍塌"，但没人能拿出里面的家具。

---

### 步骤 7：恢复节点

```bash
docker start rabbitmq-node1
docker exec rabbitmq-node1 rabbitmqctl start_app
sleep 3
```

再次验证队列恢复：

```bash
curl http://localhost:8080/q1/test-availability
```

```json
{"available": true, "verdict": "队列 master 存活，可以正常消费"}
```

## 结果对比

| 维度 | 操作前（3 节点正常） | 操作后（master 宕机） | 恢复后 |
|---|---|---|---|
| 集群状态 | 3/3 running | 2/3 running, 1 down | 3/3 running |
| `q1.normal.test` 状态 | running | **down** | running |
| 队列元数据 | 3 节点 Mnesia 均有 | 2 个存活节点 Mnesia 仍有 | 3 节点均有 |
| 消息内容 | 在 master 磁盘 | **锁死在宕机节点磁盘** | 恢复，消息仍在 |
| 消费者 | 正常消费 | **channel error，连接断开** | 正常消费 |
| 管理界面 | 队列绿色 ✓ | 队列红色 ✗ | 队列绿色 ✓ |

## 原理点睛

面试题回答中有三段话，被你刚才的操作精准"实锤"了：

> "RabbitMQ 集群本质上是一个'元数据复制群组'。Mnesia 数据库存储元数据并复制到所有节点，但队列里的消息本身存在一个本地消息存储中。队列主节点宕机，队列就不可用。"

你杀 master 后从 node2 执行 `list_queues` —— 元数据在，状态 `down`。这就是 Mnesia 复制了元数据但没复制消息内容的直接证据。

> "持久化只是把消息写到硬盘，而硬盘在宕机节点上，拔不出来。"

你 `docker start` 恢复 node1 后，消息又回来了。因为消息持久化在 node1 的磁盘上，容器重启后数据仍在。但在宕机期间，其他节点碰不到这块磁盘。

> "集群是共享元数据的兄弟会——公告栏谁都看得见，但私人物品绝不混用。"

步骤 6 中 `q1.normal.test down` 就是公告栏上的"302 室已坍塌"。

---

## 面试要诀

> "我在 3 节点 Docker 集群上实际测试过：声明一个普通队列，发 10 条消息，然后 `docker stop` 掉 master 节点。从存活节点执行 `rabbitmqctl list_queues`，队列状态显示 `down`，Spring Boot 消费端直接报 `channel error`。这让我刻骨铭心地理解了'集群只复制元数据、不复制数据'—— 从那以后，所有需要高可用的队列我绝不会用普通队列。"

面试官听到"队列状态显示 `down`"和"channel error"这两个具体细节，立刻知道你不是在背书。
