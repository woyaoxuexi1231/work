# 第3题：镜像队列的主从同步与数据安全性

> **面试题原文**：在集群基础上，你启用了镜像队列。请描述消息发布到镜像队列时，master 与 slave 之间的同步过程（GM 协议）。`publisher confirm` 在镜像队列中何时返回？如果 master 节点宕机，存活的 slave 如何提升为新的 master？提升过程中可能会发生消息丢失或重复吗？你如何配置镜像策略来满足金融级消息零丢失的要求？

---

## 实战类型：操作文档 + 代码演示（混合）

**为什么选这种方式**：
- GM 协议的同步过程和故障切换 → **操作文档**：必须配策略、发消息、杀 master 来观察系统行为
- publisher confirm 的返回时机和零丢失配置 → **代码演示**：通过 ConfirmCallback 精确测量 confirm 延迟，对比镜像队列和普通队列的 confirm 差异

**执行顺序**：先跑操作步骤（脚本 02），再用 Spring Boot 代码验证 confirm 行为。

## 第一部分：操作文档 —— 镜像队列故障切换

### 操作背景（苦难时代）

你在集群上跑得好好的，直到有一次 node1 的电源模块烧了，上面"订单预处理"队列的 300 万条消息全部卡死。你痛定思痛，启用了镜像队列 `ha-mode: all`。结果一次网络抖动后 node3 重新加入集群，触发全量同步，**整个队列阻塞 20 分钟**，上游全部超时熔断。你这才明白：镜像队列解决了"master 宕机不可用"，却引入了"同步阻塞"这个新噩梦。

### 操作目标

配置 `ha-mode: exactly, ha-params: 2`（一主一从），发布消息到镜像队列，**杀掉 master**，亲眼看到 mirror 自动提升、消息不丢。

### 环境准备

| 资源 | 说明 |
|---|---|
| 集群 | `scripts/01-cluster-setup.sh` 已运行，3 节点正常 |
| 镜像策略脚本 | `scripts/02-mirrored-queue.sh` |

### 详细操作步骤

#### 步骤 1：运行镜像队列配置脚本

```bash
cd scripts/
chmod +x 02-mirrored-queue.sh
./02-mirrored-queue.sh
```

**关注脚本输出的这一段**：

```
[5/6] 定位镜像队列的 master 节点...
  mirrored.order.pending 的 master 在: rabbit@rabbitmq-node1
  master: rabbitmq-node1
  mirror: rabbitmq-node2
```

记下 master 和 mirror 分别是谁。

> **【原理验证点 1】** `rabbitmqctl list_queues name node` 显示的是队列的 master 节点，但 `ha-mode: exactly, ha-params: 2` 意味着还有一个隐藏的 mirror 在另一个节点上。脚本通过 `list_queues` 只显示 master，要看到 mirror 需要 `rabbitmqctl list_queues name slave_pids`。

#### 步骤 2：在管理界面确认镜像状态

打开 `http://192.168.3.100:15672` → Queues → `mirrored.order.pending` → 详情页。

找到 **Mirrors** 区域，应该看到一个 slave 节点，状态为 `Synchronised`。

> **【原理验证点 2】** `Synchronised` 意味着 mirror 已经和 master 完全同步。如果状态是 `Synchronising`，说明 mirror 正在追赶 master 的消息。此时杀 master，未同步的消息会丢失 —— 这就是面试题中"如果 slave 落后 master 很多消息，master 宕机会丢哪些消息"的答案。

#### 步骤 3：观察故障切换

脚本会自动执行这一步，但你应该仔细观察：

```
[6/6] ====== 模拟故障切换 ======
  ▸ 即将停止 master 节点: rabbitmq-node1
  ▸ 预期: rabbitmq-node2 上的 mirror 自动提升为新 master
```

脚本输出：

```
docker stop rabbitmq-node1
(等待 5 秒...)
✓ 故障切换成功! 队列仍可消费，消息未丢失
```

> **【原理验证点 3】** 这就是 GM 协议的核心能力：master 宕机后，存活的资格最老的 mirror 被提升为新 master。客户端只需重连即可继续消费。但注意：**如果 master 在宕机前已向生产者 confirm 但消息还没复制到 mirror，这些消息就丢了**。这就是镜像队列"已确认仍可能丢"的致命缺陷。

#### 步骤 4：恢复 master 节点

```bash
docker start rabbitmq-node1
docker exec rabbitmq-node1 rabbitmqctl start_app
```

> **【原理验证点 4】** node1 重新加入后，它会作为新 mirror 从当前 master（原 mirror）全量同步。如果队列里积压了大量消息，这个同步过程**会阻塞队列**。这就是面试题中"`ha-sync-mode: automatic` 生产环境绝对不敢用"的原因。

## 第二部分：代码演示 —— publisher confirm 行为

### 演示场景

Spring Boot 的 `RabbitTemplate` 注册了 `ConfirmCallback`。我们往镜像队列发 10 条消息，每条都带 `CorrelationData`，精确记录 confirm 返回的延迟。然后对比普通队列（单节点确认），让你亲眼看到：

- **镜像队列 confirm**：master 等到所有 mirror 确认后才回调
- **普通队列 confirm**：master 自己确认即可，更快但无冗余

### 核心代码

```java
// 【重点1】注册 ConfirmCallback —— 每条消息到达 Broker 后回调
rabbitTemplate.setConfirmCallback((CorrelationData cd, boolean ack, String cause) -> {
    // ack=true  → 消息已写入 master 且 mirror(s) 已确认
    // ack=false → 消息未成功（如队列不存在、镜像不足等）
    log.info("Confirm: msgId={}, ack={}, cause={}", cd.getId(), ack, cause);
});

// 【重点2】发送消息时绑定 CorrelationData，用于关联 confirm
CorrelationData cd = new CorrelationData(msgId);
rabbitTemplate.convertAndSend("", "mirrored.order.pending", payload, cd);

// 【重点3】同步等待 confirm，精确测量延迟
boolean ok = cd.getFuture().get(5, TimeUnit.SECONDS);
```

### 运行指南

#### 对比实验 A：镜像队列的 confirm 延迟

```bash
curl "http://localhost:8080/q3/publish-confirm-ok?count=20"
```

**关注返回结果中的 `latencyMs` 字段**：

```json
{
  "queue": "mirrored.order.pending",
  "total": 20,
  "results": [
    {"msgId":"a1b2...","result":"✓ CONFIRMED","latencyMs":12},
    {"msgId":"c3d4...","result":"✓ CONFIRMED","latencyMs":8},
    ...
  ],
  "verdict": "✓ 全部 confirm! 消息已安全写入 master + mirror"
}
```

**观察点**：每条消息的 confirm 延迟（8-20ms 正常），这个时间包含了"master 写入磁盘 + broadcast 到 mirror + mirror 写入磁盘 + 回复确认"的全程。

#### 对比实验 B：普通队列的 confirm 延迟

```bash
curl http://localhost:8080/q3/publish-classic
```

```json
{
  "queue": "q3.normal.test (普通队列)",
  "confirm_latency_ms": 3,
  "verdict": "普通队列 confirm 只需 3ms —— 只需 master 一个节点确认。对比镜像队列的 confirm，它不需要等 mirror，所以更快。但代价是：master 宕机 → 消息完蛋。"
}
```

#### 对比实验 C：模拟 confirm 超时（杀 master 后立即发消息）

```bash
# 第一步：杀掉 master
docker stop rabbitmq-node1

# 第二步：立即发消息到镜像队列
curl "http://localhost:8080/q3/publish-confirm-ok?count=5"
```

**预期结果**：部分消息的 confirm 状态显示 `✗ TIMEOUT` —— 因为 master 宕机后，新的 mirror 还在选举，confirm 无法在 5 秒内返回。但选举完成后，消息实际可能已写入新 master —— 这就是**重复的风险**。

---

## 结果对比

| 维度 | 普通队列 | 镜像队列 (exactly:2) |
|---|---|---|
| confirm 延迟 | 2-5ms | 8-20ms |
| confirm 含义 | master 已写入 | master + mirror 均已写入 |
| master 宕机 | 队列不可用，confirm 超时 | mirror 提升，confirm 短暂超时后恢复 |
| 数据安全 | 已 confirm 的消息在宕机磁盘上 | 已 confirm 的消息在 2 个节点上 |
| 最大风险 | 单点故障 | mirror 同步滞后时的 confirm 窗口 |

## 原理点睛

面试题回答中有三段话被这一整套操作"实锤"了：

> "master 接收消息，持久化，然后将消息通过 GM 环广播给所有 slave；slave 写入磁盘后返回确认；当所有 slave 确认后，master 向生产者发送 publisher confirm。"

你的 Spring Boot ConfirmCallback 里，`ack=true` 的时刻正是"所有 mirror 都确认"的时刻。普通队列的 confirm 更快（3ms），因为不需要等 GM 环上的广播和确认。

> "如果 slave 因同步滞后，master 宕机前已把消息写入并确认给生产者，但还没复制到 slave，那么这些已确认的消息就丢失了。"

这就是你在"杀 master 后立即发消息"实验中看到的 `TIMEOUT` 消息 —— 它们可能已经由旧 master 确认但还未复制，新 master 没有这些数据。

> "金融级零丢失配置：队列 durable、消息 delivery-mode:2、ha-mode:exactly ha-params:2、publisher confirm、手动 ack、消息幂等。"

你刚才做的每一步 —— 声明 durable 队列、发持久化消息、用 `exactly:2` 策略、通过 ConfirmCallback 等确认 —— 正是这套配置的前 4 项。剩下 2 项（手动 ack + 幂等）在 Q7 的 ReliableEventBus 中实现。
