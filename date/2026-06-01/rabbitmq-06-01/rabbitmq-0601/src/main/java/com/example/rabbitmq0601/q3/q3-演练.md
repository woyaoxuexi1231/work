# 第3道题：镜像队列的主从同步与数据安全性

> 在集群基础上，你启用了镜像队列。请描述消息发布到镜像队列时，master 与 slave 之间的同步过程（GM 协议）。`publisher confirm` 在镜像队列中何时返回？如果 master 节点宕机，存活的 slave 如何提升为新的 master？提升过程中可能会发生消息丢失或重复吗？你如何配置镜像策略来满足金融级消息零丢失的要求？

## 第一部分：操作文档 —— 镜像队列故障切换

### 操作背景（苦难时代）

你上一次在集群里把 master 节点关机后，队列变成 `down`，消息全部卡死（Q1 的惨痛教训）。于是你启用了镜像队列 `ha-mode: all`，每条消息写 3 个副本，觉得万无一失。结果一次 node3 网络抖动后被踢出集群，重新加入时触发全量同步，**整个队列阻塞 20 分钟**，上游全部熔断。

你这才发现：镜像队列解决了"master 宕机不可用"，却引入了"同步阻塞"这个新噩梦。

### 操作目标

配置 `ha-mode: exactly, ha-params: 2`（一主一从），发布 100 条消息，**杀掉 master**，亲眼看到 mirror 在 5 秒内提升为新 master、消息一条不丢（至少不丢已 confirm 的）。



### 一、ha-mode 副本分布（3种）
1. **all**：全集群节点都存副本，无需 `ha-params`
   优点：可用性最高；缺点：节点多同步开销大
2. **exactly**（你在用）：固定N个副本，`ha-params` 填数字
   例：`ha-params:2` = 1主1从，副本数量固定不变
3. **nodes**：指定节点存副本，`ha-params` 填节点名数组
   优点：精准控制；缺点：节点变动需改配置

### 二、ha-sync-mode 数据同步（2种）
1. **manual**（默认/你在用）：新从节点**只同步新消息，不同步历史**，不阻塞业务。风险：主节点宕机易丢旧消息。
2. **automatic**：新从节点**自动全量同步历史消息**，同步期间队列阻塞。数据安全，但大队列影响性能。

### 三、你的组合小结
`exactly + 2 + manual`：固定1主1从，新节点只追新消息、不同步历史，**性能优先，存在丢消息风险**。

### 详细操作步骤

#### 步骤 1：配置镜像策略

```bash
sudo bash scripts/02-mirrored-queue.sh
```

**预期输出**：
```
✅ 4 条镜像策略全部配置
  ^mirrored\\.      → exactly:2  (Q3)
  ^q4\\.all\\.      → all        (Q4 反面)
  ^q4\\.ex2\\.      → exactly:2  (Q4 推荐)
  ^q5\\.mirrored\\. → exactly:2  (Q5 对比)
```

> `02-mirrored-queue.sh` 只配策略，不创建队列。队列由 Spring Boot 创建。

#### 步骤 2：通过 Spring Boot 创建队列

```bash
curl http://localhost:8080/q3/setup
```

```json
{
  "mirrored.order.pending": "已创建 — 以 mirrored. 开头，自动匹配 ha-policy → 一主一从",
  "mirrored.order.confirmed": "已创建 — 同上",
  "q3.classic.control": "已创建 — 普通队列，无镜像"
}
```

> **【原理验证点 1】** 队列名 `mirrored.order.pending` 匹配策略模式 `^mirrored\\.`，自动获得 `ha-mode: exactly, ha-params: 2`。Spring Boot 只需命名，无需感知镜像细节。

#### 步骤 3：管理界面确认镜像状态

`http://192.168.3.100:15672` → Queues → `mirrored.order.pending` → **Mirrors** 区域有一个 slave，状态 `Synchronised`。

> **【原理验证点 2】** `Synchronised` = mirror 已和 master 完全同步。`Synchronising` = 正在追赶，此时杀 master 会丢未同步的消息。

#### 步骤 4：发布消息 + 杀 master 观察故障切换

```bash
# 发 10 条消息
curl "http://localhost:8080/q3/publish-mirrored?count=10"

# 定位 master 节点
docker exec rabbitmq-node1 rabbitmqctl list_queues name node --formatter=table 2>&1 | grep mirrored

# 杀 master
docker stop rabbitmq-node1
sleep 5

# 从存活节点消费
docker exec rabbitmq-node2 rabbitmqadmin -u admin -p admin123 get \
  queue=mirrored.order.pending ackmode=ack_requeue_false
```

**预期**：能消费到消息。mirror 在 5 秒内提升为新 master。

> **【原理验证点 3】** master 宕机 → 最老的 slave 提升 → 客户端重连继续消费。但 GM 协议窗口：**已 confirm 但未复制到 mirror 的消息会丢。**

#### 步骤 5：恢复 master

```bash
docker start rabbitmq-node1
docker exec rabbitmq-node1 rabbitmqctl start_app
```

> **【原理验证点 4】** node1 重新加入 → 作为新 mirror 全量同步。队列积压大时**会阻塞**——这就是 `ha-sync-mode: automatic` 不敢用于生产的原因。

## 第二部分：代码演示 —— publisher confirm 精确测量

### 演示场景

Spring Boot 注册了全局 `ConfirmCallback`。我们往镜像队列和普通队列各发 20 条消息，每条绑定 `CorrelationData`，精确记录 confirm 返回的毫秒级延迟。

**巧妙之处**：两条队列的 confirm 延迟差值，就是 GM 环广播 + mirror 写盘 + mirror 确认回传的时间。这是一个教科书级的"同步复制代价"量化实验。

### 核心代码

```java
// 【重点1】注册全局 ConfirmCallback
rabbitTemplate.setConfirmCallback((CorrelationData cd, boolean ack, String cause) -> {
    // ack=true  → 消息已写入 master + mirror(s) 已确认
    // ack=false → 镜像不足或队列不存在
});

// 【重点2】发送时绑定 CorrelationData，同步等待 confirm
CorrelationData cd = new CorrelationData(msgId);
rabbitTemplate.convertAndSend("", "mirrored.order.pending", payload, cd);
cd.getFuture().get(5, TimeUnit.SECONDS);  // 阻塞等待 confirm
// 这个 get() 返回的时刻 = 所有 mirror 确认完毕的时刻
```

### 运行指南

#### 对比实验 A：镜像队列 confirm 延迟

```bash
curl "http://localhost:8080/q3/publish-mirrored?count=20"
```

**关注返回的 `latency_ms`**：

```json
{
  "queue": "mirrored.order.pending (镜像队列, exactly:2)",
  "平均延迟_ms": 11,
  "最大延迟_ms": 25,
  "明细": [
    {"seq": 1, "latency_ms": 8,  "status": "✓ CONFIRMED"},
    {"seq": 2, "latency_ms": 12, "status": "✓ CONFIRMED"},
    {"seq": 3, "latency_ms": 25, "status": "✓ CONFIRMED"}   ← 某次 mirror 慢了
  ],
  "关键解读": "每条 confirm 的延迟包含了: master 写盘 → GM 环广播 → mirror 写盘 → mirror 确认 → master 汇总 → 回调。"
}
```

**观察点**：延迟 8-25ms，波动较大。第 3 条的 25ms 很可能是 mirror 节点磁盘 IO 抖动。

#### 对比实验 B：普通队列 confirm（基准线）

```bash
curl "http://localhost:8080/q3/publish-classic?count=20"
```

```json
{
  "queue": "q3.classic.control (普通队列，无镜像)",
  "平均延迟_ms": 3,
  "最大延迟_ms": 6,
  "关键解读": "普通队列的 confirm 只需 master 一个节点确认 —— 更快，但 master 宕机后消息锁死在磁盘上。"
}
```

#### 差值分析

| 指标 | 镜像队列 | 普通队列 | 差值 |
|---|---|---|---|
| 平均延迟 | 11ms | 3ms | **+8ms** |
| 最大延迟 | 25ms | 6ms | **+19ms** |

> 多出来的 8ms = GM 环广播到 mirror + mirror 写盘 + mirror 确认回传。多出来的 19ms（max）= 某个 mirror 磁盘 IO 抖动拖慢了整个确认链。"镜像队列的吞吐取决于最慢的那个 mirror"——**这里就是证据**。

## 结果对比

| 维度 | 普通队列 | 镜像队列 (exactly:2) |
|---|---|---|
| confirm 等待对象 | master 一个节点 | master + 1 mirror |
| confirm 平均延迟 | 3ms | 11ms |
| master 宕机 | 队列 `down`，消息不可用 | mirror 自动提升，5s 内恢复 |
| 已 confirm 消息安全性 | 锁在宕机节点磁盘 | 在 2 个节点上有副本 |
| 最大风险 | 单点故障 | mirror 同步滞后窗口 + 全量同步阻塞 |

## 原理点睛

面试题回答中有三段话被这一整套操作 + 代码"实锤"了：

> "master 接收消息，持久化，然后将消息通过 GM 环广播给所有 slave；slave 写入磁盘后返回确认；当所有 slave 确认后，master 向生产者发送 publisher confirm。"

你的 `/q3/publish-mirrored` 返回的 `平均延迟 11ms` 就是这句话的时间量化。普通队列只需 3ms，多出的 8ms 就是 GM 环广播 + mirror 写盘。

> "如果 slave 因同步滞后，master 宕机前已把消息写入并确认给生产者，但还没复制到 slave，这些消息就丢失了。"

你在 `/q3/confirm-during-failover` 实验中发的 TIMEOUT 消息就在这个窗口里。它们的 confirm 在旧 master 上可能已返回（或未返回），但新 master 没有这些数据。

> "金融级零丢失配置：队列 durable、消息 delivery-mode:2、ha-mode:exactly ha-params:2、publisher confirm、手动 ack、消息幂等。"

你刚才做的——声明 durable 队列、发持久化消息、用 `exactly:2` 策略、通过 ConfirmCallback 等确认——正是这套配置的前 4 项。剩下 2 项在 Q7 的 `ReliableEventBus` 中实现。
