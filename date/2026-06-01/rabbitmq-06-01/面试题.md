### 第1题：镜像队列的主从同步与消息可靠性

**【问题】**  
你负责的支付回调系统使用 RabbitMQ 镜像队列保证高可用。请详细描述一条消息从生产者发布到镜像队列，并被消费者成功消费的完整流程中，镜像队列的 master 与 slave 之间是如何协作的？`publisher confirm` 机制在镜像模式下何时返回确认给生产者？如果 master 节点突然宕机，剩余镜像提升为 master，如何保证消息不丢？你遇到过新 master 提升后大量消息重新投递的情况吗？

**【考察点】**
- 对镜像队列（HA queues）同步原理的理解（GM 协议）
- 消息可靠性的四个环节：生产者确认、broker 主从同步、消费者确认、故障恢复
- 能结合具体场景识别消息重复的风险点

**【追问方向】**
- 镜像队列的 `ha-sync-mode` 设为 automatic 和 manual 有何区别？全量同步会带来什么风险？
- 如果 slave 落后 master 很多消息，master 宕机，那些未同步的消息会丢失吗？
- 你提到新 master 会重新投递未确认消息，这一定安全吗？可能引发什么幂等性问题？

**【参考要点】**
1. 发布流程：master 接收消息，写入磁盘（视持久化配置），然后将消息广播给所有 slave（使用 GM 环），当所有 slave 返回确认后（或达到 `ha-sync-batch-size` 的批量确认），master 向生产者发送 `publisher confirm`。这保证了消息在 master 和所有镜像上一致。
2. 消费者确认：消费者从 master 消费，ack 只发送给 master，master 再广播给 slave 删除消息。
3. 故障切换：若 master 宕机，存活的 slave 中最早启动的（或最先完成同步的）会被提升为新 master，未确认的消息会重新入队（requeue），可能造成重复投递。
4. 关键风险：如果 `ha-sync-mode` 设为 manual，加入新节点时不会自动同步历史消息，该 slave 为空壳，若 master 此时宕机会丢失大量消息。automatic 同步会造成 IO 风暴，阻塞正常消息处理。

### 第2题：镜像队列的性能陷阱与反模式

**【问题】**  
你们的订单系统在活动高峰期突然出现消息堆积、P99 延迟飙升，排查发现使用了 `ha-mode: all`（所有节点镜像）而且 `delivery-mode: 2`（持久化消息）。请从镜像队列的同步特性和磁盘 IO 角度分析性能瓶颈原因。你会如何优化镜像策略和参数来平衡高可用和吞吐？为什么 `ha-mode: exactly` 并设置 `ha-params: 2` 通常比 `all` 更好？在大量临时队列的场景，镜像队列适用吗？

**【考察点】**
- 理解镜像队列的性能代价（写放大、所有节点写入延迟依赖最慢节点）
- 如何根据业务场景合理配置镜像参数
- 识别镜像队列的滥用场景（如大量短生命周期队列使用镜像）

**【追问方向】**
- 如果硬要使用全部镜像，有什么办法降低延迟？比如 `ha-sync-batch-size` 和 publisher confirm 的批量处理？
- 镜像队列搭配流控（Flow Control）时，日志里看到 connection blocked，如何快速定位是哪个队列引起的？
- 你提到临时队列不要用镜像，那类似 RPC 回调队列的高可用如何解决？

**【参考要点】**
1. 瓶颈原因：
    - `ha-mode: all` 导致每条消息必须同步到集群所有节点，写操作数量与节点数线性相关，受最慢节点磁盘性能拖累；
    - 持久化消息触发 fsync，所有镜像节点同时刷盘，加剧 IO 争用；
    - master 需要等待所有 slave 确认，增加发布延迟。
2. 优化：
    - 使用 `ha-mode: exactly` 并设置 `ha-params: 2` 或 3，保证一主一从（或一主两从），降低写放大；
    - 消息尽量使用 `delivery-mode: 1`（内存），结合队列持久化 `durable=true`，依赖镜像而非持久化来防止节点重启丢消息；
    - 调整 `ha-sync-batch-size`，减少同步批次数，降低网络往返。
3. 临时队列通常用于 RPC 或一次性回调，存活时间短，故障时直接重试即可，无需开启镜像，否则会严重拖慢集群。

### 第3题：集群网络分区处理与策略选择

**【问题】**  
你的 RabbitMQ 集群部署在两个机房，某天机房之间网络中断，触发了网络分区。请列举 RabbitMQ 提供的几种分区处理模式（`ignore`、`pause_minority`、`autoheal`、`pause-if-all-down`）。你们的业务是金融交易系统，不能接受任何消息丢失，也不能有脑裂，你会选择哪种模式？为什么？如果选了 `pause_minority`，分区恢复后，minority 侧的节点上未完成的消息如何保证不丢？你遇到过 `autoheal` 模式导致队列消失的惨案吗？

**【考察点】**
- 理解 RabbitMQ 集群网络分区的成因与后果
- 熟悉不同处理模式的机制及适用场景
- 能结合实际业务需求做出正确策略选择，并深知错误选择的代价

**【追问方向】**
- `pause_minority` 模式下，集群三分之二节点在分区中存活，剩下三分之一被暂停，此时生产者连接到被暂停的节点会怎样？
- 分区恢复时，`autoheal` 模式如何决定“赢家”节点？为什么它会导致数据丢失？
- 如何从日志和监控中发现网络分区？你们配置了分区告警吗？

**【参考要点】**
1. 模式解析：
    - `ignore`：无干预，可能导致脑裂，两侧各自有 master，产生消息不一致；
    - `pause_minority`：多数派正常服务，少数派节点自动暂停所有网络连接，不提供服务，避免脑裂；
    - `pause-if-all-down`：配置一组目标节点，当它们都不可达时暂停；
    - `autoheal`：分区恢复后自动选择一侧作为赢家，重启输家节点并同步数据，但输家侧所有状态（包括未持久化消息）会丢失。
2. 金融系统应选 `pause_minority`，确保只有多数派工作，牺牲小部分可用性，避免数据不一致与脑裂。
3. 关键点：被暂停的少数派节点上，未确认的消息会保留在队列中（持久化），当其恢复正常并重新加入集群后，如果该节点上的 master 队列已被多数派侧的镜像接管，旧 master 上的消息可能会被丢弃（取决于版本和同步状态）。因此需要确保消息持久化，并配合镜像策略使得多数派侧有备份。
4. `autoheal` 惨案：若选择输家是数据多的一侧，那一切状态清空，队列内容丢失，属于重大事故，仅适用于可重建设的开发或非关键系统。

### 第4题：从镜像队列到仲裁队列的演进

**【问题】**  
假设你正在设计一个新系统，需要选择队列的高可用方案。镜像队列和仲裁队列（Quorum Queues）都能提供数据冗余，请从内部共识算法、数据安全性、性能特点、运维复杂度等角度对比两者。为什么 RabbitMQ 官方推荐新应用使用仲裁队列？在什么场景下你仍然会选择镜像队列？迁移一个现有镜像队列到仲裁队列，你会如何设计平滑切换方案？

**【考察点】**
- 了解 RabbitMQ 3.8+ 引入的仲裁队列及其优势
- 理解 Raft 协议与镜像队列的 GM 协议在一致性和性能上的根本差异
- 能够根据实际业务需求进行合理技术选型，并规划迁移

**【追问方向】**
- 仲裁队列基于 Raft，如何选举 Leader？它如何处理网络分区？与镜像队列的 `pause_minority` 有何异同？
- 仲裁队列对消息吞吐量和延迟的影响如何？为什么它对消息大小有建议上限？
- 你如何说服团队从镜像队列迁移到仲裁队列？列出你的关键论据。

**【参考要点】**
1. 对比：
    - 算法：镜像队列基于主从异步（GM 虽是组播，但仍依赖 master 同步，从节点可能落后）；仲裁队列基于 Raft，写入需多数派确认，提供强一致性。
    - 数据安全：仲裁队列一旦确认，数据不会丢失，即使 Leader 宕机，新 Leader 保证已提交数据存在；镜像队列在同步未完成时可能丢失消息。
    - 性能：仲裁队列写入需多数派磁盘写入，延迟相对较高但稳定；镜像队列性能易受从节点影响，`all` 模式最差。
    - 运维：仲裁队列自动处理 Leader 选举和成员变更，无需配置镜像策略和同步模式，更简化。
2. 仍用镜像队列的场景：需要大量短生命周期队列（仲裁队列有一定的元数据开销）、需要支持低版本 RabbitMQ、需要跨广域网异步复制（Federation）。
3. 迁移方案：
    - 新建仲裁队列，消费者逐步从旧队列切换到新队列（按业务灰度），生产者双写或通过路由层动态切换；
    - 监控旧队列无消息后下线，期间注意幂等性。

### 第5题：故障排查——消息丢失与重复的根因分析

**【问题】**  
用户投诉“订单状态更新丢失”，你排查发现 RabbitMQ 集群的一个节点凌晨发生了重启，并且该节点上某个镜像队列的主副本在重启前有大量 unacked 消息。请你构造一个完整的故障链路，解释消息可能丢失或重复的几个关键点（包括生产者、broker、消费者）。你如何通过 RabbitMQ 的指标和日志（如 `rabbitmqctl list_queues`、`prometheus` 指标）验证你的猜测？并给出可靠的修复方案。

**【考察点】**
- 综合运用 RabbitMQ 原理定位生产事故的能力
- 对监控指标的熟悉程度（`messages_ready`、`messages_unacknowledged`、`disk_writes` 等）
- 可靠的端到端消息解决方案设计

**【追问方向】**
- 如果消费者用了 autoAck，会带来什么隐患？
- 你如何区分是生产者没发成功，还是 broker 丢了，还是消费者没正确处理？
- 如果要设计一个消息对账系统来发现这类丢失，你会怎么做？

**【参考要点】**
1. 可能丢失点：
    - 生产者：未启用 publisher confirm，投递到 broker 时连接断开，消息未入队；
    - 镜像队列：节点重启时，若队列为 `durable`，消息持久化到磁盘的可恢复，但未同步到镜像的消息会丢失；
    - 消费者：unacked 消息在节点重启后 requeue，若消费者没做幂等，重新处理可能导致重复；若消费者崩溃，消息返回队列但可能已经部分处理。
2. 监控验证：
    - 检查 `messages_ready` 和 `messages_unacknowledged` 趋势，若重启前后有突变；
    - 查看节点日志，确认有无 `node down` 和镜像提升事件；
    - 使用 `rabbitmq-diagnostics check_if_node_is_quorum_critical` 等工具检查队列状态。
3. 修复方案：
    - 必须开启 publisher confirm；
    - 消费者手动 ack，且处理业务后 ack；
    - 业务层实现幂等（如基于唯一消息 ID 去重表）；
    - 队列和消息都持久化，镜像策略设为 `exactly` 2，配合 quorum 队列更佳。

### 第6题：大规模集群设计与跨机房容灾

**【问题】**  
公司业务走向国际化，需要在同城双机房部署 RabbitMQ 集群，要求单机房故障时消息服务整体切换时间 < 30s，且消息零丢失。现有镜像队列方案在跨机房延迟 5-10ms 时出现严重性能抖动。请设计一套满足要求的架构，并说明为什么镜像队列不适合跨机房高可用？你会引入 Federation、Shovel 还是直接用 Quorum Queue 的多机房 Raft？如果必须自建集群，如何规划节点分布和负载均衡，以及如何处理脑裂？

**【考察点】**
- 对 RabbitMQ 跨机房架构的方案设计与权衡
- 理解镜像队列在跨地域高延迟下的致命缺陷
- 掌握 Federation/Shovel 的适用场景与 Quorum Queue 的 CAP 特性

**【追问方向】**
- 同城双机房，Quorum 队列要求多数派写入，假设你部署 3 节点分布 2 个机房，如何放置以容忍一个机房整体故障？
- Federation 插件如何解决消息顺序性问题？
- 跨机房切换时，客户端连接如何自动切换到备用机房？你考虑过 DNS 切换、负载均衡器还是客户端内置故障转移？

**【参考要点】**
1. 镜像队列问题：跨机房延迟导致所有节点同步写入的确认变慢，直接拉低整个集群吞吐，且容易触发网络分区。
2. 推荐方案：
    - 方案 A（强一致优先）：使用 Quorum Queue，3 节点跨两机房，主站点 2 节点，备站点 1 节点，保证主站点机房内部达到多数派即可写入，备站点异步跟随。主站点故障时，备站点单节点无法达成多数派，需手动介入或部署 5 节点（3+2）使备站点可自动达成多数派。
    - 方案 B（高可用异步）：主集群用镜像/Quorum队列，备机房部署独立集群，通过 Federation 插件进行异地异步镜像，接受延迟和少量消息重复，适合不要求强一致的场景。
3. 客户端切换：通过内部 DNS 或 LVS 指向主集群，切换时修改 DNS 指向备集群；或客户端同时连接两个集群 VIP，主集群故障时自动重连备用连接。需注意消息顺序和幂等。
4. 网络分区处理：用 `pause_minority` 策略，合理分布节点使任何分区都无法形成两个多数派。

- 能够透明适配镜像队列和仲裁队列，无需业务方修改代码。  
  请给出核心接口设计、消息生命周期状态机，以及处理连接断开、队列不存在的容错逻辑。写出发布消息和消费消息的核心伪代码。

**【考察点】**
- 生产级消息中间件封装能力
- 对连接管理、重试、幂等、死信等模式的掌握
- 抽象设计能力，隔离底层队列类型的差异

**【追问方向】**
- 如果发送时返回 `confirm` 超时，但实际消息已写入 broker，怎么避免重复？
- 消费者处理一条消息耗时过长，导致连接心跳超时，你如何避免？
- 如何在组件中集成链路追踪（tracing），让每条消息携带 traceId？

**【参考要点】**
1. 接口示例：
   ```java
   public interface EventBus {
       CompletableFuture<Void> publish(String routingKey, Event event);
       void subscribe(String queue, Consumer<Event> handler);
   }
   ```  
   内部维护 `rabbitmq.client.Channel` 池，`publish` 开启 confirm 模式，异步返回 Future。
2. 状态机：NEW → SENDING → SENT (confirm) → ACKED；若 confirm 超时或失败 → RETRYING，达到上限 → FAILED（写入 DB 兜底表，定时任务补偿）。

4. 连接容错：监听 `ShutdownListener`，触发时重新创建连接和拓扑声明；队列不存在异常捕获后声明队列（需保证配置同步）。
5. 适配镜像/仲裁：通过配置指定队列类型参数（`x-queue-type: quorum`），组件在声明队列时带上该参数，对业务无感。

### 第8题：集群扩容与不停机迁移

**【问题】**  
你的 RabbitMQ 集群线上有 6 个节点，由于业务增长需要扩容到 10 个节点。现有数百个镜像队列分布在原节点。请设计一个平滑扩容方案，要求不影响在线业务，迁移过程中不丢消息。你如何利用 RabbitMQ 的插件（如 `rabbitmq-consistent-hash-exchange`）或外部负载均衡来重新平衡队列？是否可以使用热迁移（queue master rebalance）？描述具体步骤、可能的风险和回滚策略。

**【考察点】**
- 对 RabbitMQ 集群扩容和数据迁移的实战经验
- 理解 queue master 分布与负载均衡
- 面对大规模集群的运维流程设计

**【追问方向】**
- 扩容后发现流量不均衡，如何重新平衡队列的 master 分布而不影响消息处理？
- 镜像队列在迁移 master 时会发生什么？为什么官方不建议频繁 `queue_master_locator` 策略变动？
- 如果用的是仲裁队列，扩容后新节点如何加入 Raft 成员组？数据会自动重分布吗？

**【参考要点】**
1. 扩容步骤：
    - 新节点加入集群，同步集群元数据；
    - 对于新创建的队列，通过 `x-queue-master-locator` 策略（如 `min-masters` 或 `client-local`）让新的队列 master 优先在新节点上；
    - 对于已有队列，通过制定 `ha-mode` 镜像策略，将新节点加入镜像列表（`ha-params` 增加），消息自动异步同步，同步完成后可考虑手动 rebalance master 分布，但要避开业务高峰，否则移动 master 会导致短暂队列不可用（需要停止队列，迁移主数据，重启）；
    - 更温和的方法是只利用新节点作为镜像分担读或备机，不强制迁移 master，长期通过队列自然创建达到均衡。
2. 风险：rebalance master 是阻塞操作，会导致队列短时不可服务；同步大量数据时可能引发网络和 IO 风暴。
3. 回滚：将新节点移出镜像策略即可，但需保证仍有足够镜像满足高可用。
4. 仲裁队列扩容：新节点加入集群后，可通过 `rabbitmq-queues add_member` 将新节点加入指定仲裁队列的 Raft 成员组，会触发数据同步，同步完成后旧节点可移除。该过程在线进行，对客户端透明。
