**【第1题】**

*   **本题包含的待答项**：
    -   主问题1：集群启动时节点之间如何发现彼此？
    -   主问题2：集群的元数据是如何在各节点间同步的？
    -   主问题3：为什么队列的消息内容默认只存储在其主节点上？
    -   主问题4：如果把一个普通队列的 master 节点直接关机，该队列能否在其他节点上继续提供服务？为什么？
    -   追问1：如果忘记设置统一的 Erlang Cookie，节点加入集群会报什么错？
    -   追问2：集群中每个节点都有一份完整的元数据拷贝吗？这在大规模集群中有什么潜在问题？
    -   追问3：如果要实现队列在节点宕机后仍可用，你会增加什么机制？这引出了镜像队列的必要性。

*   **主问题回答**：
    面试官好。关于三节点集群的搭建与元数据同步，这正好对应了我之前深度剖析的**“集群元数据全复制与消息存储主节点化”**思想。

    首先是节点发现。这就像我此前比喻的中世纪汉萨同盟。一个新城邦想入盟，不是自立旗号就行的。第一步是出示“护照”——即 `.erlang.cookie` 文件，这是集群所有节点共享的秘密。第二步是向宗主城邦报到，通过执行 `rabbitmqctl join_cluster` 命令，指定集群中任一在网节点。目标节点会验证护照，一旦通过，便会将新节点的信息广播给全网，最终所有节点之间都会建立起点对点的 TCP 长连接，形成一个全网状网络。

    其次是元数据同步。交换器、队列、绑定关系等这些“同盟协定”，都存放在一个内嵌的分布式数据库 Mnesia 中。这套数据会被复制到集群所有节点。这意味着，连接任何节点都能查到全集群的拓扑，这是“控制面全冗余”带来的便利。

    接着是第三点，为何消息内容只在主节点？因为我说的“同盟协定”只是商会名录，并不包含货物本身。货物依然存放在各个城邦的**私有仓库**里。在 RabbitMQ 中，消息内容存在 Message Store 组件中，只属于声明它的那个 Master 节点。这是**数据面分而治之**的设计，用隔离换来了写入的高性能。

    最后，如果我直接关机 Master，该队列**绝不能**在其他节点上恢复服务。虽然元数据还在，但货物已随仓库化为灰烬。所有生产者、消费者都会收到 `404 NOT_FOUND` 的报错。这正是普通队列的高可用边界。

*   **追问预判回答**：
    -   **追问1**：如果忘记统一 Erlang Cookie，加入集群时会报出非常明确的错误：`Connection attempt from disallowed node`。日志会清晰指出认证失败，节点间的 TCP 连接会在握手阶段就被直接切断。
    -   **追问2**：是的，默认情况下，每个节点都有一份完整的元数据拷贝。这在大规模集群中会引发我讲过的“写性能瓶颈”和“内存沉默杀手”问题。当节点数达到几十甚至上百时，任何一次元数据变更，比如声明一个新队列，都必须在所有节点构成的事务中提交，延迟取决于最慢的那个节点。同时，每个节点都要在内存里维护一份同样的百万级绑定表，巨大的内存浪费可能会打爆 Erlang 的原子内存，导致节点崩溃。
    -   **追问3**：要实现队列在节点宕机后仍可用，就必须打破“货物只存一处”的规则，引入**镜像队列**。通过配置策略，让消息在发布时就被实时抄送到其他节点的 Slave 上，给仓库找好替身。这就是镜像队列存在的根本必要性。

*   **【面试官视角点评】**
    这个回答最打动人的是，将集群架构的两个核心概念，用“共享商会名录”和“私有仓库”这个统一的比喻完美区分开。他没有只背诵命令，而是从设计思想的高度解释了“为何如此”，并把高可用缺陷作为引向镜像队列的自然过渡，逻辑非常清晰。

**【第2题】**

*   **本题包含的待答项**：
    -   主问题1：描述消息从生产者到达消费者的完整路径。
    -   主问题2：这种跨节点消息转发会带来什么性能影响？
    -   主问题3：如果大量生产者连接到一个节点，而队列 master 都在另一个节点，会出现什么瓶颈？
    -   主问题4：你如何通过客户端连接策略或队列主节点分布来优化？
    -   追问1：RabbitMQ 集群内部消息转发有没有压缩？对大消息有什么建议？
    -   追问2：如果使用一致性哈希交换器来分布队列，能解决这个问题吗？
    -   追问3：你会如何监控节点间的内部通信流量？

*   **主问题回答**：
    这题直击我深入讲解过的**“集群内部消息路由与节点间转发成本”**。

    消息的完整路径分四站。第一站：生产者连接到节点 A，将消息投递给交换器。第二站：节点 A 的交换器，基于本地那份完整的元数据名录，查询路由键的绑定，发现目标队列的 Master 在节点 B。第三站：节点 A 将消息通过集群内部专有通信信道，转发给节点 B。第四站：节点 B 收到消息，存入本地 Message Store。消费者如果也连在节点 B，就直接消费；如果连在节点 C，消息会被二次转发。

    这种逻辑上的透明路由，在物理上引入了巨大的“摩擦成本”。跨节点转发会带来序列化、TCP传输、反序列化、二次写入磁盘的完整开销，且内部通信**不做任何压缩**。这会直接增加处理延迟，消耗可观的网络带宽和 CPU 资源。

    当大量生产者连到节点 A，而队列 Master 全部在节点 B 时，节点 A 和 B 之间的内部链路就会成为唯一的“漏斗瓶颈”。我曾亲眼目睹内部带宽使用率飙到 90%，生产端 P99 延迟从 5ms 恶化到 500ms，CPU 软中断狂飙，最终消息大量堆积，系统被拖垮。

    优化策略，核心是执行我提出的“物理亲和性”铁律。第一，客户端优先本地连接，避免不必要的绕路。第二，对于动态队列，使用 `x-queue-master-locator: client-local` 策略，强制新队列的 Master 创建在生产者连接的那个节点上。第三，对于固定拓扑，主动进行容量规划，将相关业务的队列 Master 集中放在同一组节点上，实现流量本地化。

*   **追问预判回答**：
    -   **追问1**：RabbitMQ 内部转发消息没有压缩。因此对于大消息，强烈建议在应用层使用如 gzip、snappy 等算法压缩后发送，或者将大文件上传到对象存储，消息体只携带文件 URL，避免堵塞内部管道。
    -   **追问2**：一致性哈希交换器可以解决“队列在集群中如何分布”的问题，但它无法解决“生产者如何找到已存在队列的 Master”的问题。它能在创建队列时将其分散开，但无法修正一个已经存在的、且 Master 不在本地的队列的路由成本。
    -   **追问3**：我会重点监控 RabbitMQ 暴露的 Prometheus 指标，特别是 `rabbitmq_node_bytes_sent` 和 `rabbitmq_node_bytes_recv`。同时，结合 `iftop` 或 `nload` 等系统工具，观察集群节点间 25672 端口（Erlang 分发端口）的实时流量，一旦发现带宽使用率长期超过阈值，就说明拓扑存在热点。

*   **【面试官视角点评】**
    这个回答极具现场感。候选人将“瓶颈”用一个“漏斗”的比喻具象化，并引用了 P99 延迟从 5ms 恶化到 500ms 这个血泪案例，让性能影响不再是纸上谈兵。后续优化的“物理亲和性”原则是点睛之笔，体现了从原理推导出最佳实践的实战能力。

**【第3题】**

*   **本题包含的待答项**：
    -   主问题1：描述消息发布到镜像队列时，master 与 slave 之间的同步过程（GM 协议）。
    -   主问题2：`publisher confirm` 在镜像队列中何时返回？
    -   主问题3：如果 master 节点宕机，存活的 slave 如何提升为新的 master？
    -   主问题4：提升过程中可能会发生消息丢失或重复吗？
    -   主问题5：你如何配置镜像策略来满足金融级消息零丢失的要求？
    -   追问1：`ha-sync-mode: automatic` 的全量同步会阻塞整个队列吗？生产环境你敢用吗？
    -   追问2：如果 slave 节点落后 master 很多消息，master 宕机会丢哪些消息？
    -   追问3：你提到要消息零丢失，除了镜像队列配置，生产者和消费者端还需要做什么？

*   **主问题回答**：
    这正是我剖析过的**“镜像队列 GM 组播同步与确认语义”**中的核心问题。

    同步过程就像一场严谨的合唱。Master 是领唱。它收到消息并完成自己的“起音”——也就是本地持久化后，会通过 GM 协议构建的原子广播环，将消息依次传递给环上的每一个 Slave。每个 Slave 收到后写入磁盘，并向后传递确认。当这个“集体确认”回到 Master，意味着所有 Slave 都齐声唱完了，Master 才会转身，向台下的指挥（生产者）给出 `publisher confirm` 的手势。

    但在现实运行中，这里存在一个我反复强调的致命“异步窗口”。Master 可能在本地写入成功，并将消息放入发向 Slave 的缓冲区后，就**提前**向生产者返回了确认。它赌的是 Slave 能很快完成同步。如果 Master 恰在此时宕机，那些“已确认但未同步”的消息就会永久丢失。

    故障切换时，监控到 Master 宕机后，集群会推举最老的、或数据最完整的 Slave 为新 Master。这必然导致两类问题：一是**消息重复**，老 Master 上已发出但未收到消费者 `ack` 的消息，在新 Master 上会重新入队。二是**消息丢失**，如果 Slave 存在同步滞后，那些已确认但滞留在老 Master 缓冲区里的消息就彻底没了。

    要满足金融级“零丢失”，我的配置是一个纵深防御体系：第一，队列和消息必须 `durable`。第二，策略设 `ha-mode: exactly, ha-params: 2`，用最小成本保证一主一从。第三，生产者的 `publisher confirm` 是必要条件，但必须等待所有 Slave 确认，关上那个窗口。第四，消费者手动 `ack`。最后，业务层必须实现幂等，作为防重复的最后一道防线。

*   **追问预判回答**：
    -   **追问1**：`automatic` 的全量同步会严重冲击 Master 节点。新节点加入时，Master 会饿狼般地将所有积压消息一股脑塞过去，瞬间把 Master 的磁盘 IO 和网络带宽占满，导致主线程阻塞，队列停止服务。在生产环境，我绝对不敢用，必须设为 `manual`，在业务低谷期手动执行同步。
    -   **追问2**：如果 Slave 落后，Master 宕机时，所有滞后于 Slave 的、并且 Master 已向生产者确认过的消息，都会丢失。这正是“确认窗口”问题的直接体现。不丢那些生产者未收到确认的，只丢那些“已确认但未同步”的。
    -   **追问3**：生产者端必须实现**发件箱模式**。消息和业务数据在一个本地事务中写入数据库发件箱表，状态为 `PENDING`。只有当消息被中间件成功确认后才更新为 `SENT`。后台定时任务扫描 `PENDING` 记录进行重试。消费者端则必须实现**幂等**，通过全局唯一 `messageId` 去重。这才是真正的应用层兜底。

*   **【面试官视角点评】**
    这个回答的深度，在于他没有神话镜像队列。候选人精准地指出了 GM 协议中那个“异步确认窗口”的致命设计，并用“领唱提前给指挥手势”的比喻，把“已确认却丢失”的反直觉现象解释得极其透彻。最终给出的“零丢失”方案是一个从中间件到应用层的完整防御链条。

**【第4题】**

*   **本题包含的待答项**：
    -   主问题1：从镜像同步和磁盘 IO 角度解释性能瓶颈。
    -   主问题2：为什么推荐使用 `ha-mode: exactly` 并指定参数而非 `all`？
    -   主问题3：临时队列（如 RPC 回调）开启镜像会有什么问题？
    -   主问题4：如果必须保留镜像，有哪些参数可以调优以提升吞吐？
    -   追问1：如果镜像节点磁盘性能不一致（如一台 SSD、一台机械盘），会发生什么？
    -   追问2：`ha-sync-batch-size` 如何影响吞吐和延迟？
    -   追问3：你如何通过 RabbitMQ 管理界面或 Prometheus 指标发现镜像同步慢的节点？

*   **主问题回答**：
    这道题是一场典型的、由**“镜像队列写放大效应”**引发的生产事故。

    `ha-mode: all` 且消息持久化，导致每一条消息都会触发一场由集群所有节点共同参与的“多重 fsync 风暴”。Master 必须等待**每一个** Slave 都完成磁盘刷写，才向生产者确认。这造成了灾难性的写放大：1MB 消息在 N 个节点上产生 N MB 的物理写入。整个集群的写入吞吐，瞬间被性能最差的那块磁盘锁死。这就是我说的“木桶效应”——一个猪队友拖垮全队，最终引发消息堆积和 P99 延迟飙升。

    推荐 `exactly` 并设参数 `2`，本质是用最小的双倍写入代价，换得对最常见单节点故障的免疫。在全村抄写族谱，只抄两份，一份存东头，一份存西头，完全足以防火防盗，何必劳民伤财每家每户都抄。

    为临时队列开启镜像，是我称之为“最愚蠢反模式”的行为。这类 RPC 回调队列活不过几秒，为其建立和销毁镜像所带来的元数据同步开销，远超其本身的生命价值，纯粹是对集群资源的无谓消耗和内部干扰。

    如果必须保留镜像，我的调优三板斧是：第一，非关键业务用 `delivery-mode: 1` 非持久化消息，用内存副本换掉磁盘 IO。第二，`ha-sync-mode: manual`，把同步主动权交给运维，在低峰期进行。第三，适当增大 `ha-sync-batch-size`，让每次网络往返能多带一些消息，提升同步吞吐量。

*   **追问预判回答**：
    -   **追问1**：如果节点磁盘性能不一致，比如一台 SSD 一台 HDD，整个集群的性能就会被那块 HDD 节点拉低到机械盘的水平。Master 的 IO 会呈现周期性的阻塞等待，监控图上会看到规则的心跳式停顿和延迟尖刺。
    -   **追问2**：增大 `ha-sync-batch-size` 能提升同步吞吐，因为它减少了 Slave 确认和网络往返的次数。但代价是会增加同步过程的资源消耗和对 Master 内存的短期占用。需要压测找到一个平衡点，避免把 Master 撑垮。
    -   **追问3**：我会在管理界面直接观察队列的镜像同步状态和 Slave 的滞后消息数。更系统的方法是通过 Prometheus 指标，查看 `rabbitmq_queue_messages_ready` 在 Master 和 Slave 上的差异。结合各节点的 `disk_io_await` 指标，就能快速定位到那个拖后腿的“慢盘”节点。

*   **【面试官视角点评】**
    这个回答成功地将“写放大”从一个抽象概念，变成了“全村抄书”和“多重 fsync 风暴”的生动灾难。尤其是他明确指出临时队列镜像这个反模式，显示了对资源管理极其敏锐的嗅觉，这是一线架构师才有的实战智慧。

**【第5题】**

*   **本题包含的待答项**：
    -   主问题1：从内部共识算法、数据一致性保证、性能和运维复杂度几个维度，对比镜像队列与仲裁队列。
    -   主问题2：为什么官方建议新应用直接使用仲裁队列？
    -   主问题3：在哪些场景下你仍然会选择镜像队列？
    -   主问题4：如果要将现有镜像队列迁移到仲裁队列，你会设计怎样的平滑切换方案？
    -   追问1：仲裁队列如何处理网络分区？与镜像队列的 `pause_minority` 有何异同？
    -   追问2：仲裁队列有哪些性能限制？比如对消息大小、发布确认延迟的影响？
    -   追问3：如何说服团队使用仲裁队列，列出你的关键论据。

*   **主问题回答**：
    这道题是关于我讲解的**“仲裁队列 Raft 共识与强一致性模型”**的范式对比。

    首先，在算法思想上，镜像队列基于 GM 协议，是“主从异步复制”，一个人的独唱。而仲裁队列基于 Raft，是“议会式”的群体决策，一条法令必须多数派投票通过才生效。这是本质的“强一致性”对“最终一致性”的升级。

    其次，数据安全性上，这是最根本的分水岭。仲裁队列的 `publisher confirm` 是多数派投票通过且写入日志后的承诺，**一旦确认，绝不丢失**，新 Leader 必定包含所有已提交数据。而镜像队列的确认，如前所述，存在“异步窗口”导致丢失的风险。

    在性能上，镜像队列在小副本数时延迟极低，但方差大，受慢节点拖累。仲裁队列因为需要多数派的磁盘写入和网络往返，**延迟基线天然更高，但极其稳定**。这是一种用延迟换确定性的交易。

    运维上，镜像队列是一套精细的手工活，需管理策略、手动同步。仲裁队列则自动处理成员变更和 Leader 选举，运维心智负担大大降低。

    官方推荐仲裁队列，正因为它在默认情况下提供了“数据绝对安全”这个最宝贵的承诺。我仍会在两个场景选择镜像队列：一是需要管理海量短生命周期的临时队列，Raft 日志开销过重；二是需要跨广域网异步复制时，结合 Federation 插件使用。

    至于平滑迁移，我的方案是“双写灰度切换”。先创建新的仲裁队列，改造生产者，以双写模式同时写入新旧两套队列。然后将消费者逐步灰度切换至新仲裁队列。通过 `messageId` 和幂等机制确保无重复无丢失。待旧队列积压消息消费完毕后，平稳下线。

*   **追问预判回答**：
    -   **追问1**：仲裁队列通过 Raft 选举机制处理分区，只有拥有多数派节点的分区才能成功选举 Leader 并继续服务，少数派自然无法写入。这与 `pause_minority` 策略思想一致，但仲裁队列的实现更彻底、更自动化，能从协议层自动恢复，而镜像队列依赖的外部策略恢复更复杂。
    -   **追问2**：仲裁队列性能限制很明显。因为多数派磁盘写入，`publisher confirm` 延迟通常在 10-20ms 级别，显著高于镜像队列。且对消息大小有限制，不适合体量巨大的消息。吞吐量也受限于多数派节点中性能最弱的那一个。
    -   **追问3**：我会用一句话说服团队：“从我们迁移到仲裁队列后，对账差异归零了。”——这就是最强的论据。用业务的确定性、运维的零成本和数据绝对安全的承诺，足以压倒那几毫秒的延迟增加。

*   **【面试官视角点评】**
    这个回答的结构化对比非常精彩。候选人从“算法思想”这个根源切入，将安全、性能、运维三个维度的差异推导得环环相扣。最后用“对账差异归零”这个终极业务价值来为技术选型定性，极具说服力。这正是架构师做决策时的价值思考方式。

**【第6题】**

*   **本题包含的待答项**：
    -   主问题1：设计一套基于仲裁队列的多机房架构，并说明节点如何分布以满足“容忍一个机房整体故障”。
    -   主问题2：网络分区发生时你会使用哪种处理策略？为什么？
    -   主问题3：客户端如何自动切换到备用机房？
    -   追问1：如果使用 3 节点仲裁队列，两个机房如何分布才能让单个机房故障时仍能自动达成多数派？
    -   追问2：Federation 插件在此场景中是否适用？与 Quorum Queue 有何优劣？
    -   追问3：跨机房切换后，消息顺序如何保证？

*   **主问题回答**：
    这是对我讲解的**“多机房部署与网络分区处理策略”**思想的直接实战。

    镜像队列在跨机房 5-10ms 延迟下抖动，根源是其 GM 环的同步会把跨机房延迟在环上放大。我的架构设计将采用 5 节点仲裁队列，进行“3主站 + 2备站”的节点分布。

    法定人数为 (5/2)+1 = 3。我们将队列的 Leader 固定在主机房 A 的某个节点上。这样，正常写入只需主机房 A 内部多数派确认（2票），延迟等同于单机房。备机房 B 的节点作为“追随者”异步追赶日志。当主机房 A 整体灾难，备机房 B 仅剩 2 节点，无法满足法定人数 3，集群会自动暂停写入。这正是我们想要的效果：**宁可暂停服务，也绝不脑裂导致数据不一致**，满足“容忍一个机房整体故障”的架构目标。

    网络分区策略，我选择 `pause_minority`。当分区发生，少数派节点会自动暂停，拒绝服务。这与我设计的仲裁队列多数派选举机制相辅相成，构成了防止脑裂的双重保险。

    客户端切换，我会采用域名切换方案。客户端配置两个机房的负载均衡 VIP 域名。当主机房故障，运维通过监控确认后，将主域名解析指向备机房 VIP。配合合理的 TTL 和客户端重连机制，可实现 <30s 的切换目标。消费者端的**幂等机制**是这一切的基石，用以消除切换期间消息重复的风险。

*   **追问预判回答**：
    -   **追问1**：使用 3 节点，2主+1备的分布，在主机房整体宕机时，备站单节点无法达成多数派，不能自动选举，必须人工介入。因此它不满足自动化要求。要自动达成多数派，机房数必须为奇数，例如 3 机房各 1 节点，或使用 5 节点的 3+2 分布。
    -   **追问2**：Federation 适用于松耦合的、跨地域的异步消息转发，它不保证强一致性。而本方案是在一个强一致性集群内解决同城容灾，追求的是数据零丢失和高可用。两者可结合使用，例如 Quorum Queue 做同城容灾，Federation 做异地灾备。
    -   **追问3**：跨机房切换后，绝对的消息顺序保证是几乎不可能的，这是分布式系统的固有挑战。应对策略是让业务设计不依赖绝对的全局顺序，或者使用支持顺序性的仲裁队列单 Leader 特性，确保单个队列的消息相对有序，但全局顺序仍需业务层通过版本号等机制重建。

*   **【面试官视角点评】**
    这个设计方案老练而务实。候选人没有幻想“完美双活”，而是清晰地给出了 3+2 的分布和法定人数计算，并表达了“宁可暂停，绝不脑裂”的果断取舍。这是对 CAP 理论在实战中深刻理解的体现，知道什么时候该舍什么，该保什么。

**【第7题】**

*   **本题包含的待答项**：
    -   主问题1：请设计一套基于 RabbitMQ 的可靠事件总线组件（`EventBus`），能够透明适配镜像队列和仲裁队列。
    -   要求1（发送端）：支持同步/异步确认，失败自动重试（指数退避），最终兜底写入数据库；
    -   要求2（消费端）：支持幂等、手动 ack、死信队列重试与告警；
    -   要求3：能够处理连接断开、队列不存在的异常，并自动恢复；
    -   要求4：组件内集成链路追踪，为每条消息注入 `traceId`。
    -   要求5：请画出核心接口和类图，并写出消息发布和消费的伪代码逻辑。
    -   追问1：如果 `publisher confirm` 超时但消息实际已写入队列，如何避免重复发送？
    -   追问2：消费端处理消息耗时太长，导致连接心跳超时，如何设计长任务处理机制？
    -   追问3：你如何测试这个组件在各种故障下的行为（如网络抖动、broker 重启）？

*   **主问题回答**：
    这个组件的设计，正是我对**“可靠事件总线的抽象与容错设计”**思想的代码化落地。

    **核心接口与类图设计**：
    -   `IEventBus`：顶层抽象，定义 `publish(EventMessage)` 返回 `CompletableFuture<Void>` 和 `subscribe(consumer)` 方法。
    -   `RabbitEventBus`：核心实现，内部持有 `ConnectionManager`、`OutboxDispatcher`、`ChannelWorker` 等组件。
    -   `ConnectionManager`：封装连接工厂，监听 `ShutdownListener` 实现自动重连与拓扑恢复。
    -   `OutboxDispatcher`：负责从发件箱表抓取消息、投递、处理确认和重试。
    -   `ChannelWorker`：负责消费、调用业务处理器、处理 ack/nack 和死信。
    -   `QueueConfigurer`：声明队列时，根据配置自动注入 `x-queue-type: quorum` 或镜像策略参数，实现对两种队列的透明适配。

    **发送端伪代码逻辑（发件箱模式 + 指数退避）**：
    ```
    function publish(msg):
        // 1. 注入全局 traceId 和 messageId
        msg.headers.traceId = getCurrentTraceId()
        msg.headers.messageId = uuid()
    
        // 2. 兜底写入：在业务事务中落库，状态为 PENDING
        db.transaction(() => {
            saveBusinessData(msg.payload)
            saveOutboxRecord(msg.messageId, msg, status=PENDING, nextRetry=now())
        })
    
        // 3. 异步触发投递
        outboxDispatcher.dispatchAsync(msg)
    
    // OutboxDispatcher 内部
    function dispatchAsync(msg):
        try:
            channel.basicPublish(exchange, key, msg, confirmCallback)
            // confirmCallback: 成功 -> 更新发件箱状态为 SENT
            // 失败或超时 -> 指数退避计算 nextRetry = now + min(2^retryCount * 1s, 60s)
            //             -> 更新发件箱 next_retry_time
        catch (Exception e):
            // 连接异常等，同样计算退避重试时间
    ```
    后台独立扫描线程，定期捞取 `status=PENDING AND next_retry_time < now()` 的记录重新投递。

    **消费端伪代码逻辑（幂等 + 手动 ack + 死信）**：
    ```
    function handleMessage(channel, delivery):
        msg = deserialize(delivery.body)
        // 1. 链路追踪：traceId 放入 MDC
        MDC.put("traceId", msg.headers.traceId)
    
        // 2. 幂等检查
        if (dedupTable.exists(msg.headers.messageId)):
            channel.basicAck(delivery.envelope.deliveryTag, false)
            return
    
        // 3. 业务处理
        try:
            executeBusinessLogic(msg)
            channel.basicAck(delivery.envelope.deliveryTag, false)
            dedupTable.save(msg.headers.messageId) // 记录去重
        catch (TransientException e):
            channel.basicNack(delivery.envelope.deliveryTag, false, true) // 重回队列或进入延迟死信
        catch (FatalException e):
            channel.basicNack(delivery.envelope.deliveryTag, false, false) // 送入死信队列
            alertService.send(msg, e) // 触发告警
    ```

*   **追问预判回答**：
    -   **追问1**：这正是发件箱模式要解决的核心问题。`confirm` 超时但消息实际已入队，会导致重试发送，产生重复消息。解决方案不在于阻止重复发送，而在于消费端的**幂等机制**。消费者通过 `messageId` 去重，确保无论收到多少条相同消息，业务逻辑仅执行一次。
    -   **追问2**：对于长任务，绝不能在主消息处理线程阻塞。消费者收到消息后，应立即将其提交到一个专门的、独立的线程池进行异步处理，并快速返回 `ack` 确认 RabbitMQ。但这会丢失 `ack` 对处理结果的保证，因此需要结合发件箱或本地事件表，在长任务成功后，再通过业务手段进行后续处理或补偿。同时，需合理设置 `prefetch count` 和心跳超时时间。
    -   **追问3**：我会为此组件编写一套“混沌测试”套件。测试用例中，会使用 `iptables` 或 `tc` 命令模拟网络延迟、丢包和分区，使用 `docker kill` 模拟 Broker 宕机，使用 `kill -STOP` 模拟应用假死。通过断言测试前后，数据库发件箱和去重表中的记录状态，以及业务系统的最终数据，来验证消息的最终一致性和幂等性。

*   **【面试官视角点评】**
    这个回答的卓越之处在于，它将发件箱、重试、幂等、死信、链路追踪这些离散的最佳实践，有机地整合进了一个统一的 `EventBus` 组件中。候选人不仅给出了代码逻辑，更重要的是通过 `QueueConfigurer` 等设计，体现了“透明适配”和“封装复杂性”的架构思想。这是一个能直接落地的生产级方案。
