订单超时自动取消
如何优雅设计？

用户下单后 30 分钟未支付，系统需自动关闭订单并释放库存。 这不仅仅是一个定时任务，更是对延迟精度、系统解耦、幂等性的综合考验。

数据库轮询压力 延迟时间不准 重复取消/幂等 服务重启丢失

五种主流实现方案

Level 1

数据库定时轮询

每隔 N 秒扫描 order 表中 status=UNPAID 且 create_time < now()-30min 的记录。

实现极简，无外部依赖
DB 压力大，存在轮询间隔误差
数据量大时 SQL 性能急剧下降
适用: Demo / 日单量 < 1000
Level 2

JDK DelayQueue

基于优先级队列的内存级延迟队列，元素到期后才能被取出消费。

精度高，零外部依赖
致命缺陷：重启丢失所有任务
单机瓶颈，无法集群扩展
适用: 学习原理 / 非关键业务
Level 3

Redis ZSet / Keyspace

ZSet: score=过期时间戳，轮询取到期元素。
Keyspace: 利用 key 过期事件通知（不推荐）。

轻量，多数项目已有 Redis
ZSet 仍需轮询；Keyspace 不保证可靠
大数据量下 ZRANGEBYSCORE 阻塞风险
适用: 中小规模 / 已有Redis基础设施
Recommended
Level 4

MQ 死信队列 (TTL+DLX)

消息设置 TTL=30min → 超时后进入 Dead Letter Exchange → 路由到取消订单队列 → 消费者处理。

精准延迟，无需轮询
天然持久化，重启不丢失
解耦下单与取消逻辑
RabbitMQ 不支持动态 TTL 队列复用
🏆 生产环境首选方案
Level 5

RocketMQ 延迟消息

原生支持 18 个延迟等级 (1s~2h)。5.x 版本支持任意精度延迟。

开箱即用，无需 DLX 配置
高吞吐，阿里双11验证
4.x 仅支持固定等级，不够灵活
运维成本高于 RabbitMQ
适用: 已用RocketMQ / 超高并发
Bonus

Netty HashedWheelTimer

时间轮算法，O(1) 调度复杂度。适合海量短延迟任务。

内存级高性能调度
同样有重启丢失问题
常作为 MQ 内部调度引擎

生产环境三大铁律

幂等性保障

取消操作必须幂等！用户可能在第29分钟手动取消，延迟消息到达时需检查订单状态。
UPDATE ... WHERE status='UNPAID'
用乐观锁/CAS 防止重复扣减库存。

补偿兜底机制

MQ 不是 100% 可靠。必须保留一个低频定时任务（如每5分钟）扫描漏网之鱼作为最终一致性兜底。

避免 Keyspace 陷阱

Redis Keyspace Notification 是发布即忘模式，没有 ACK 机制。如果消费者宕机，消息永久丢失。生产环境严禁用于关键业务。

面试满分回答模板

"订单超时取消本质是一个分布式延迟任务问题。我推荐基于 RabbitMQ TTL + 死信队列的方案：下单时发送一条 TTL=30min 的消息到延迟队列，超时后自动路由到取消队列由消费者处理。相比 DB 轮询，它精准且不伤库；相比 Redis Keyspace，它有 ACK 保证可靠性。同时我会做三点加固：① 消费者通过乐观锁保证幂等；② 保留低频定时扫描作为兜底补偿；③ 如果团队已用 RocketMQ 5.x，可直接用其原生延迟消息简化架构。"
