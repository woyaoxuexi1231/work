1 制造一个大 Key (Hash 类型, 100万字段)

gen_bigkey.py 
import redis
import time

r = redis.Redis(host='host.docker.internal', port=6379)
KEY = "test:bigkey:hash"
COUNT = 1_000_000

print(f"🚀 Generating {COUNT} fields...")
start = time.time()

# 使用 pipeline 批量写入，速度更快
pipe = r.pipeline()
for i in range(COUNT):
    pipe.hset(KEY, f"field:{i}", f"value_{i}" * 10)
    if i % 10000 == 0 and i > 0:
        pipe.execute()
        print(f"  Progress: {i:,}")

pipe.execute()
print(f"✅ Done in {time.time()-start:.2f}s")
print(f"📏 Size: {r.memory_usage(KEY)} bytes")
gen_bigkey.sh (无需Python) 
# 生成100万条 HSET 命令并通过管道传入 redis-cli
# 约需 30-60 秒，取决于机器性能

echo "🚀 Generating big key via shell..."
for i in $(seq 1 1000000); do
  echo "HSET test:bigkey:hash field:$i value_$i"
done | redis-cli --pipe

echo "✅ Done. Verifying..."
redis-cli HLEN test:bigkey:hash
redis-cli MEMORY USAGE test:bigkey:hash

2 验证 & 体感测试

在 redis-cli 中执行以下命令 
# 1. 确认大小 (预期返回 ~1000000)
HLEN test:bigkey:hash

# 2. 确认内存占用 (预期返回 ~80,000,000+ bytes ≈ 80MB)
MEMORY USAGE test:bigkey:hash

# 3. ⚡ 体感测试: 先开一个终端持续 PING，观察延迟
#    终端A: redis-cli --latency-history -i 1
#    终端B: 执行下面的 DEL，观察终端A的延迟飙升
DEL test:bigkey:hash

# 4. 🔄 重新生成后，对比 UNLINK 的效果
UNLINK test:bigkey:hash
#    观察终端A的延迟是否几乎无变化

什么是 BigKey？为什么危险？

BigKey 通常指 String 类型 > 10KB 或 集合类型元素超过 5000 个 的键。 由于 Redis 是单线程模型（处理命令），对 BigKey 的读写或删除操作会长时间占用 CPU，导致其他请求排队超时，甚至引发集群节点故障转移。

网络拥塞

如何发现 BigKey

redis-cli --bigkeys

官方自带工具，扫描整个实例，给出每种数据类型的最大 Key 统计。

$ redis-cli -h 127.0.0.1 -p 6379 --bigkeys
Biggest string found 'user:json:992' has 5.2MB
生产环境慎用，会有性能损耗

RDB 离线分析

使用 rdb 或 redis-rdb-tools 解析备份文件，生成 CSV 报表，最安全。

$ rdb -c memory dump.rdb > result.csv
推荐：不影响线上服务

删除与优化方案

方案 A: UNLINK (首选)

Redis ≥ 4.0

UNLINK key 命令仅在主线程中移除键的引用（O(1)），实际的内存释放交给后台线程异步处理。

❌ 错误做法
DEL my_big_hash
✅ 正确做法
UNLINK my_big_hash

方案 B: 渐进式删除 (兼容旧版)

All Versions

对于 Hash/Set/ZSet，使用 HSCAN/SSCAN 分批获取成员，然后分批 HDEL。避免一次性删除造成的长尾延迟。

local cursor = "0"
repeat
  local res = redis.call("HSCAN", KEYS[1], cursor, "COUNT", 100)
  cursor = res[1]
  if #res[2] > 0 then
    redis.call("HDEL", KEYS[1], unpack(res[2]))
  end
until cursor == "0"

方案 C: 架构拆分 (治本)

将一个大 Key 拆分为多个小 Key。例如将一个包含用户所有信息的 Hash 拆分为：
user:{id}:base (基本信息) + user:{id}:orders (订单列表)

Hash user:1001 (100w fields)
String user:1001:info
ZSet user:1001:timeline

预防与规范

设定阈值监控

  • String 类型控制在 10KB 以内
  • Hash/List/Set/ZSet 元素数量控制在 5000 以内
  • 部署 Prometheus + Redis Exporter 监控内存使用率

代码审查规范

  • 禁止使用 HGETALL / SMEMBERS 等全量读取命令
  • 强制使用 HSCAN 或指定字段读取
  • 设置合理的 TTL，避免无用大 Key 永久驻留
