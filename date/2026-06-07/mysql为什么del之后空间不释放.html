MySQL 面试高频题

执行 DELETE 后
为什么磁盘空间没有释放？

面试官考察的不是命令本身，而是你对 InnoDB 存储引擎底层文件管理机制的理解。

一句话总结

InnoDB 的 DELETE 操作仅仅是把数据页标记为「可复用」（碎片化），这些空间会被记录在空闲列表中供后续 INSERT/UPDATE 使用，但不会归还给操作系统。只有 TRUNCATE 或 DROP 才会直接释放物理文件。

独立表空间 (.ibd)

当开启 innodb_file_per_table 时，每张表拥有独立的 .ibd 文件。DELETE 只是在文件内部维护了一个 Free List（空闲链表），文件句柄并未缩小。

性能与复用权衡

如果每次 DELETE 都向 OS 申请/释放空间，会导致大量的系统调用和文件碎片。保留空间是为了让后续的写入操作能以 O(1) 复杂度直接复用现有页面。

数据字典视角

通过 information_schema.tables 查看时，DATA_FREE 字段显示的就是这些已删除但未释放给 OS 的内部碎片空间大小。

💡 如何真正回收空间？

方案一：OPTIMIZE TABLE

OPTIMIZE TABLE your_table_name;

原理：创建临时表 → 拷贝有效数据 → 删除旧表 → 重命名新表。
缺点：大表耗时极长，期间可能锁表（取决于版本），会产生双倍临时空间占用。

方案二：ALTER TABLE (推荐)

ALTER TABLE your_table_name ENGINE=InnoDB;

原理：与 OPTIMIZE 类似，但在 MySQL 5.6+ 支持 Online DDL，对业务影响相对较小。
替代工具：生产环境建议使用 pt-online-schema-change 或 gh-ost 避免阻塞。

🎯 满分回答话术

"MySQL InnoDB 执行 DELETE 后不释放磁盘空间，是因为 InnoDB 采用了空间复用策略。被删除的记录所在的数据页会被标记为可用并加入 Free List，以便后续插入操作快速复用，避免频繁的系统级文件分配开销。这会导致表空间文件出现'空洞'或碎片。

如果需要物理回收空间，可以使用 OPTIMIZE TABLE 或 ALTER TABLE ... ENGINE=InnoDB 来重建表。但在生产环境中，对于大表我会优先考虑使用 gh-ost 等在线变更工具，以避免长时间锁表影响业务。另外，如果是清空全表，应直接使用 TRUNCATE 而非 DELETE。"
