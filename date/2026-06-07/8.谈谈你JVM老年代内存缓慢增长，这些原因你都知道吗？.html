六大核心成因

Full GC 的触发本质是「内存资源供需失衡」，即老年代对象增长速度超过回收速度

01

内存泄漏 — 头号元凶

对象不再使用却未被回收，长期驻留老年代，导致内存占用持续上升

静态集合未清理

static List、Map 等容器缓存大量过期数据，未设置过期淘汰机制

监听器未移除

注册的事件监听器、回调函数在对象销毁后未注销，引用链持续存在

资源未释放

数据库连接、文件流、Socket 等资源未关闭，关联对象长期存活

ThreadLocal 滥用

ThreadLocal 中的对象与线程生命周期绑定，线程池场景下未及时 remove() 导致对象永久驻留

// 典型内存泄漏代码示例
public class MemoryLeakExample {
    // 静态 Map 持续累积，无清理机制
    private static final Map cache =
        new HashMap<>();

    public void addCache(String key, Object value) {
        cache.put(key, value); // 只进不出！
    }

    // ThreadLocal 未清理
    private static final ThreadLocal tl =
        new ThreadLocal<>();

    public void process() {
        tl.set(new byte[1024 * 1024 * 10]); // 10MB
        // 用完未调用 tl.remove()
    }
}
02

大对象直接进入老年代

超过阈值的大对象（如大数组、大字符串）直接分配在老年代，快速耗尽空间

阈值说明

JVM 默认将大于 Eden 区剩余空间的对象直接分配到老年代，可通过 -XX:PretenureSizeThreshold 设置

大 JSON/ XML 字符串

序列化反序列化产生超大字符串对象

图片/文件字节数组

读取大文件到内存，byte[] 直接进入老年代

批量查询大数据集

一次查询返回百万级数据，List 对象巨大

// 每次请求创建 100MB 大数组
// 直接进入老年代！
byte[] data = new byte[1024 * 1024 * 100];

// 大对象分配参数调优
// -XX:PretenureSizeThreshold=1m
// 超过 1MB 直接进老年代

// 优化方案：分页处理、流式读取
public void processLargeData() {
    // 错误：一次性加载全部
    List all = dao.findAll();

    // 正确：分页流式处理
    dao.streamQuery(pageSize, record -> {
        process(record); // 处理完即释放
    });
}
03

对象过早晋升（Premature Promotion）

本应短期存活的对象因 Survivor 区不足或年龄阈值过低，提前进入老年代

Survivor 区不足

Minor GC 后存活对象超过 Survivor 容量，通过空间分配担保直接进入老年代

年龄阈值过低

MaxTenuringThreshold 设置过小，对象在 Survivor 区经历几次 GC 就晋升

动态年龄判定

Survivor 区中相同年龄对象大小超过 50% 时，大于等于该年龄的对象直接晋升

// JVM 参数调优建议
// 增大 Survivor 区比例
-XX:SurvivorRatio=6          // Eden:S0:S1 = 6:1:1

// 适当提高晋升年龄阈值
-XX:MaxTenuringThreshold=15  // 默认 15 (CMS 为 6)

// 观察晋升情况
jstat -gcutil PID 5000
// 关注 YGC 后 OU（老年代使用）是否持续增长
04

本地缓存对象累积

Guava Cache、Caffeine、HashMap 等缓存未设置过期策略，对象长期存活进入老年代

典型场景：每隔一段时间刷新缓存，旧缓存实例成为垃圾对象，但因未触发 Major GC 而长期占用老年代空间，最终呈现 Old Gen 内存上升趋势。 

// 错误：无界缓存，只进不出
private Map cache = new HashMap<>();

// 正确：使用带过期策略的缓存
LoadingCache cache = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最大条目数
    .expireAfterWrite(30, TimeUnit.MINUTES)  // 写入后过期
    .removalListener((key, value, cause) -> {
        // 清理资源
    })
    .build(key -> loadFromDB(key));

// 或 Guava Cache
Cache cache = CacheBuilder.newBuilder()
    .maximumSize(1000)
    .expireAfterAccess(10, TimeUnit.MINUTES)
    .weakValues()  // 允许 GC 回收值
    .build();
05

元空间（Metaspace）泄漏

动态生成类（如 CGLIB、反射、Groovy）未卸载，导致 Metaspace 持续增长，触发 Full GC

CGLIB / JDK 动态代理

频繁创建代理类，类加载器未释放

Groovy / JSP 动态编译

脚本引擎动态编译产生新类，类加载器累积

热部署 / 热加载

重复加载类，旧类加载器未卸载

Orika / MapStruct

字节码生成工具产生大量临时类

// 监控元空间
jstat -gcmetacapacity PID

// 设置元空间上限，避免无限增长
-XX:MetaspaceSize=128m
-XX:MaxMetaspaceSize=256m

// 开启类卸载日志
-XX:+TraceClassUnloading

// 检查类加载数量
jcmd PID VM.classloader_stats
06

GC 策略与参数配置不当

堆大小、分代比例、GC 收集器选择不合理，导致 GC 效率低下

老年代空间过小

NewRatio 过大导致老年代空间不足，频繁触发 Full GC

GC 收集器选择不当

Serial GC 在服务端应用性能差；CMS 浮动垃圾问题；G1 的 Region 大小不匹配大对象

堆内存未对齐

Xms ≠ Xmx 导致运行时堆扩展，触发 Full GC 整理内存

// 生产环境推荐基线配置 (JDK 8+)
-Xms4g -Xmx4g              // 堆大小固定，避免动态扩展
-XX:NewRatio=2             // 老年代:年轻代 = 2:1
-XX:SurvivorRatio=8         // Eden:S0:S1 = 8:1:1
-XX:MaxMetaspaceSize=512m   // 限制元空间
-XX:+UseG1GC               // 推荐 G1 收集器
-XX:MaxGCPauseMillis=200   // 目标最大停顿
-XX:InitiatingHeapOccupancyPercent=45

// 开启 GC 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log

代码定位与修复

结合 MAT 的 Path to GC Roots 找到引用链，定位到具体业务代码

修复 checklist

  • • 添加缓存过期策略（TTL / LRU）
  • • 移除无用监听器和回调
  • • ThreadLocal 使用完及时 remove()
  • • 关闭资源（try-with-resources）
  • • 分页处理大数据集
  • • 限制元空间大小，检查类卸载

面试回答要点

1

先判断是否正常：老年代缓慢上升不一定都是问题，可能是长期存活对象（如缓存）正常累积，只要最终能触发 Full GC 回收且不影响业务，属于正常现象。

2

区分泄漏与正常增长：若 Full GC 后老年代内存无法回落（仅降 3-5%），则存在内存泄漏；若能大幅回落，只是增长快，则考虑参数调优。

3

排查顺序：看 GC 日志 → 监控工具定位 → 堆快照分析 → 代码修复。常用工具：Arthas（在线）、MAT（离线堆分析）、jstat（命令行）。

4

高频根因：静态集合未清理、ThreadLocal 未 remove、大对象直接分配、缓存无过期策略、动态类生成导致 Metaspace 泄漏。
