Nacos 是什么？
注册 + 配置 双引擎

Nacos (Dynamic Naming and Configuration Service) 是阿里开源的微服务基础设施。 它不仅是 Eureka 的高性能替代者，更整合了 Spring Cloud Config 的功能，实现了“服务发现+动态配置”的一站式管理。

Nacos vs Eureka 核心差异

面试时请重点强调 CAP 模型、健康检查机制和性能表现这三个维度。

对比维度 Eureka (Netflix) Nacos (Alibaba)
CAP 模型 仅支持 AP (高可用优先) 支持 AP / CP 切换
(临时实例AP，持久实例CP)
健康检查 客户端心跳 (Client Heartbeat) 心跳 + TCP/HTTP 主动探测
(服务端双重检测)
配置管理 不支持 原生支持动态推送
负载均衡 Ribbon (客户端) Ribbon / LoadBalancer + 权重
雪崩保护 自我保护模式 (阈值触发) 无自我保护，依赖快速剔除
性能与规模 万级实例瓶颈，GC 频繁 十万级实例，异步推送优化

核心原理拆解

AP/CP 模式智能切换

Nacos 根据实例类型自动选择一致性协议：

  • 临时实例 (Ephemeral=true): 采用 Distro 协议 (AP)。数据存储在内存，节点间异步复制，允许短暂不一致，保证高可用。
  • 持久实例 (Ephemeral=false): 采用 Raft 协议 (CP)。数据持久化到 DB/Derby，强一致性选举 Leader，适合 DNS/K8s 场景。

推拉结合的服务发现

兼顾实时性与容错能力，避免纯推或纯拉的缺陷：

Pull: 客户端每 6s 定时拉取全量列表作为兜底
Push: 服务端 UDP 推送变更 (延迟 < 1s)
Check: 服务端主动 TCP 探活 + 客户端心跳

Eureka 自我保护机制 (AP 核心)

面试高频

Eureka 每分钟统计心跳续约数，当续约率低于阈值时触发保护：

预期续约数 = 实例数 × 2/分钟 (每30s一次心跳)
实际续约数 = 服务端实际收到的心跳次数
触发条件: 实际续约数 < 预期 × 0.85 → 激活保护
保护行为: 不剔除任何实例，宁可返回过时数据

关键误区 — 什么能/不能触发保护：

停 Eureka Server 节点 — 客户端还在给其他节点发心跳，续约率没降
大量客户端突然断心跳 — 续约率暴跌，低于 85% 阈值
网络分区 — Server 收不到心跳，续约率下降
设计理念：宁可返回过时的数据 (AP)，也不误删健康实例。对比 Nacos CP 模式：失去多数节点后直接拒绝写入。

动态配置长轮询 (Long Polling)

Nacos 摒弃了传统的短轮询和资源浪费的 WebSocket，采用长轮询机制实现准实时推送：

  1. 客户端发起 HTTP 请求，携带当前配置 MD5
  2. 服务端比对 MD5，若相同则挂起请求 29.5s (不立即返回)
  3. 期间若有配置变更 → 立即响应新值
  4. 若 29.5s 内无变更 → 返回 304 Not Modified
  5. 客户端收到响应后立即发起下一次长轮询
GET /listener?md5=xxx
Hold 29.5s
Response: New Config (or 304)
比短轮询节省 95% 无效请求
比 WebSocket 兼容性好且无需额外连接

实战验证

基于 3 节点集群的实际测试结果，面试时说出这些细节会非常加分。

Nacos AP/CP 集群故障实测

3节点集群: ephemeral 临时 vs 持久实例
场景 AP 写入 CP 写入 AP 读取 CP 读取
3/3 健康 ✓ ✓ ✓ ✓
停1节点 (2/3) ✓ ✓ ✓ ✓
停2节点 (1/3) ✓ ✗ ✓ 旧数据
全部恢复 ✓ ✓ 最终一致 强一致
核心差异: 失去多数节点后，AP (Distro) 仍可写入单节点内存；CP (Raft) 因无法选举 Leader 直接拒绝写入。
恢复后: CP 重新选举 Leader 并保证数据强一致；AP 通过异步复制最终一致。

3节点集群: 续约率暴跌实验

2

4
等待 2 分钟后假实例仍在 lease 已过期 (90s)，但保护模式下不剔除 → AP 设计
保护开启 (默认) 保护关闭
心跳停止后 实例保留 90s 后剔除
网络分区时 不会误删 可能误删
CAP 倾向 AP (可用性) 偏 CP (一致性)

Nacos vs Eureka: 节点故障时的行为对比

场景: 失去多数节点
Nacos AP: 仍可读写，最终一致
Nacos CP: 拒绝写入，保证一致性
Eureka AP: 自我保护，保留所有实例
设计哲学
Nacos: 按需选择 — 临时服务用 AP 保可用，核心配置用 CP 保一致
Eureka: 一刀切 AP — 宁可返回过时数据也不误删实例
面试加分点
• Nacos 的 AP/CP 不是配置切换，而是根据实例类型自动选择协议
• Eureka 自我保护不是 bug 而是设计，防止网络分区误删
• 停 Server 不触发 Eureka 保护，停 Client 心跳才会触发

面试一句话总结

"Nacos 是阿里开源的注册中心与配置中心二合一组件。相比 Eureka，它最大的优势在于同时支持 AP/CP 模型、原生集成配置管理以及更高的性能上限。Nacos 根据实例类型自动选择协议：临时实例用 Distro (AP) 保证高可用，持久实例用 Raft (CP) 保证强一致。实测中，当集群失去多数节点时，AP 仍可读写，CP 直接拒绝写入。Eureka 则是纯 AP 设计，通过自我保护机制实现高可用：当心跳续约率低于 85% 时触发保护，不剔除任何实例。需要注意，停 Server 不会触发保护，必须是客户端心跳丢失才会触发。在 Spring Cloud Alibaba 生态中，Nacos 是替代 Eureka+Config 的首选方案。"
