01

如何开启 GC 日志

JDK 1.8 的 GC 日志参数配置全解

核心参数（必开）

# JDK 1.8 推荐 GC 日志参数组合
-XX:+PrintGCDetails          # 打印详细 GC 信息（各区域回收详情）
-XX:+PrintGCDateStamps       # 打印绝对日期时间戳（推荐，便于对齐系统日志）
-XX:+PrintGCTimeStamps        # 打印 JVM 启动后的相对时间（秒）
-Xloggc:/path/to/gc.log       # 指定 GC 日志输出文件路径

# 生产环境完整推荐配置
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCTimeStamps \
     -Xloggc:/var/log/myapp/gc-%t.log \
     -XX:+UseGCLogFileRotation \
     -XX:NumberOfGCLogFiles=5 \
     -XX:GCLogFileSize=20M \
     -jar myapp.jar
-XX:+PrintGCDetails 必开

打印 GC 详细信息，包括各内存区域（Eden、Survivor、Old Gen、Metaspace）回收前后的使用量、耗时、触发原因等

-XX:+PrintGCDateStamps 必开

打印绝对日期时间戳（如 2024-06-08T21:04:33.123+0800），便于与系统日志、业务日志对齐排查

-XX:+PrintGCTimeStamps 建议

打印 JVM 启动后的相对时间（秒），便于计算 GC 间隔和频率

-Xloggc:<file> 必开

将 GC 日志输出到指定文件，避免与 stdout 混合。支持 %t 占位符生成带时间戳的文件名

日志轮转（生产环境必备）

生产环境 GC 日志可能非常大，开启日志轮转防止磁盘占满

# 日志轮转参数
-XX:+UseGCLogFileRotation     # 开启 GC 日志文件轮转
-XX:NumberOfGCLogFiles=5       # 保留 5 个历史日志文件
-XX:GCLogFileSize=20M          # 每个日志文件最大 20MB

# 完整示例：保留 5 个 20MB 的日志文件，循环覆盖
-Xloggc:/var/log/app/gc-%t.log \
-XX:+UseGCLogFileRotation \
-XX:NumberOfGCLogFiles=5 \
-XX:GCLogFileSize=20M

⚠️ JDK 9+ 注意

JDK 9 引入了统一日志框架（Unified Logging），上述参数在 JDK 9+ 已被移除，需使用 -Xlog:gc*:file=gc.log 替代。本指南仅针对 JDK 1.8。

进阶参数（按需开启）

-XX:+PrintHeapAtGC

每次 GC 前后打印堆内存详细分布，用于分析内存分配模式

-XX:+PrintTenuringDistribution

打印 Survivor 区中对象的年龄分布，排查对象过早晋升问题

-XX:+PrintGCApplicationStoppedTime

打印应用因 GC 导致的停顿时间（STW），评估对业务的影响

-XX:+PrintGCCause

打印触发 GC 的原因（如 Allocation Failure、Ergonomics 等）

-XX:+PrintReferenceGC

打印 Reference（软/弱/虚引用）处理时间，排查引用队列堆积

02

GC 日志怎么看

逐行拆解日志格式，读懂每一列含义

日志格式逐行解析

2024-06-08T21:04:33.123+0800: 120.456: [GC (Allocation Failure)
  [PSYoungGen: 153600K->25599K(179200K)]
  153600K->25601K(384000K), 0.0304152 secs]
  [Times: user=0.05 sys=0.01, real=0.03 secs]
2024-06-08T21:04:33.123+0800

绝对日期时间戳（由 PrintGCDateStamps 开启），精确到毫秒，带时区

120.456

JVM 启动后的相对时间（秒），由 PrintGCTimeStamps 开启

[GC / [Full GC

GC = Minor GC（只回收年轻代）；Full GC = 全堆回收（含老年代），会 STW

(Allocation Failure)

GC 触发原因。常见：Allocation Failure、Ergonomics、System.gc()、Metadata GC Threshold

153600K->25599K(179200K)

GC 前 -> GC 后（该区域总容量）。例如：年轻代从 153600K 降到 25599K，总容量 179200K

153600K->25601K(384000K)

整个堆内存的 GC 前 -> GC 后（堆总容量）。注意：堆的下降量 = 回收量

0.0304152 secs

本次 GC 耗时（秒）。Minor GC 通常 < 100ms，Full GC 可能几百 ms 到几秒

[Times: user=... sys=... real=...]

user = CPU 时间；sys = 系统调用时间；real = 实际 wall-clock 时间（STW 主要看 real）

Full GC 日志详解

2024-06-08T21:05:12.456+0800: 159.789: [Full GC (Ergonomics)
  [PSYoungGen: 25599K->0K(179200K)]
  [ParOldGen: 204801K->180234K(409600K)]
  230400K->180234K(588800K),
  [Metaspace: 34567K->34567K(107520K)],
  1.2345678 secs]
  [Times: user=3.50 sys=0.10, real=1.23 secs]

⚠️ 关键信号

出现 [Full GC 说明发生了全堆 STW，业务线程全部暂停

📊 回收效率

老年代从 204801K 降到 180234K，仅回收了约 24MB，回收率很低，可能存在内存泄漏

⏱️ 停顿时间

real=1.23 秒，业务停顿超过 1 秒，严重影响用户体验，需立即排查

命令行快速查看

# 1. 实时查看 GC 日志（类似 tail -f）
tail -f /var/log/app/gc.log

# 2. 只看 Full GC 行（快速定位问题）
grep "Full GC" /var/log/app/gc.log

# 3. 统计 Full GC 次数和平均耗时
awk '/Full GC/ {count++; sum+=$NF} END {print "Full GC 次数:", count, "平均耗时:", sum/count "s"}' gc.log

# 4. 提取每次 GC 的堆内存变化（年轻代 + 老年代）
grep -E "GC|Full GC" gc.log | awk '{print $1, $2, $(NF-1)}'

# 5. 找出耗时最长的 GC 事件
sort -t',' -k2 -nr gc.log | head -20

# 6. 统计每分钟 GC 次数（判断频率是否异常）
awk '{gsub(/T/, " "); print substr($1,1,16)}' gc.log | sort | uniq -c | sort -nr | head
03

GC 日志如何排查

从日志中提取关键指标，定位老年代内存问题

GC 频率

每分钟 GC 次数是否异常增加

正常：Minor GC < 10次/分
Full GC < 1次/小时

停顿时间

STW 时间是否影响业务

正常：Minor GC < 100ms
Full GC < 1s

回收效率

Full GC 后老年代是否回落

异常：Full GC 后只降 3-5%
→ 内存泄漏嫌疑

吞吐量

应用运行时间占总时间比例

目标：> 95%
< 90% 需优化

排查决策树

Step 1：看 Full GC 频率

grep -c "Full GC" gc.log          # 统计 Full GC 总次数
grep "Full GC" gc.log | wc -l     # 同上
正常

Full GC 很少（几小时一次），且每次回收后老年代大幅回落

异常

Full GC 频繁（几分钟一次），甚至连续触发 → 进入 Step 2

Step 2：看 Full GC 回收效率

# 提取老年代回收前后数据
grep "Full GC" gc.log | awk '{
    for(i=1;i<=NF;i++) {
        if($i ~ /ParOldGen|PSOldGen|Tenured/) {
            print $i
        }
    }
}'
# 示例输出：204801K->180234K(409600K)
# 回收率 = (204801-180234)/204801 ≈ 12% → 很低！
正常

Full GC 后老年代回收率 > 50%，内存能回落到较低水平

异常

回收率 < 20%，内存几乎不回落 → 内存泄漏嫌疑，进入 Step 3

Step 3：看 Minor GC 后老年代变化

# 观察每次 Minor GC 后老年代是否持续增长
# 如果每次 YGC 后老年代都在涨，说明对象在快速晋升
grep "GC (Allocation Failure)" gc.log | tail -20

# 关注整体堆的变化趋势
# 153600K->25601K(384000K)  → 堆从 153M 降到 25M，回收了 128M
# 如果每次回收量越来越少 → 老年代在累积对象

判断标准：如果 Minor GC 后老年代（Old Gen）持续增长，且 Full GC 也无法回收，基本可以确认存在 内存泄漏 或 对象过早晋升。

Step 4：结合 GC 原因定位根因

Allocation Failure

新生代空间不足，正常触发 Minor GC。若频繁出现，考虑增大年轻代或优化对象创建。

Ergonomics

JVM 自适应调整触发的 GC，通常伴随堆扩展。若频繁出现，建议固定堆大小（Xms=Xmx）。

Metadata GC Threshold

Metaspace 空间不足触发 Full GC。检查动态类生成、反射、CGLIB 代理等。

System.gc()

代码中显式调用了 System.gc()。搜索代码中是否有不当调用，建议禁用（-XX:+DisableExplicitGC）。

实战案例：老年代缓慢增长

假设你拿到以下 GC 日志片段，如何快速判断问题？

# 10:00 应用启动后 1 小时
2024-06-08T10:00:00.000+0800: 3600.000: [GC (Allocation Failure)
  [PSYoungGen: 153600K->2048K(179200K)]
  153600K->102400K(384000K), 0.0500000 secs]

# 11:00 应用启动后 2 小时
2024-06-08T11:00:00.000+0800: 7200.000: [GC (Allocation Failure)
  [PSYoungGen: 153600K->2048K(179200K)]
  153600K->204800K(384000K), 0.0500000 secs]

# 12:00 应用启动后 3 小时 —— 触发 Full GC
2024-06-08T12:00:00.000+0800: 10800.000: [Full GC (Ergonomics)
  [PSYoungGen: 2048K->0K(179200K)]
  [ParOldGen: 204800K->204000K(204800K)]
  206848K->204000K(384000K), 2.5000000 secs]
  [Times: user=5.00 sys=0.10, real=2.50 secs]

🚨 问题 1：老年代持续增长

1 小时时堆占用 102400K，2 小时时 204800K，老年代几乎翻倍增长，说明对象在不断累积进入老年代。

⚠️ 问题 2：Full GC 回收无效

Full GC 后老年代从 204800K 降到 204000K，仅回收了 800K（0.4%），几乎没回收！这是典型的 内存泄漏 特征。

💡 结论与下一步

确认存在内存泄漏。下一步：生成堆快照（heap dump），用 MAT 分析支配树，找到占用最大的对象及 GC Roots 引用链。

05

生产环境完整配置

直接可用的 JDK 1.8 GC 日志配置模板

# ============================================
# JDK 1.8 生产环境 GC 日志完整配置（推荐）
# ============================================

# --- 基础堆内存配置 ---
-Xms4g                          # 初始堆大小（建议 = Xmx，避免动态扩展触发 Full GC）
-Xmx4g                          # 最大堆大小
-XX:NewRatio=2                 # 老年代:年轻代 = 2:1（年轻代约 1.3g）
-XX:SurvivorRatio=8            # Eden:S0:S1 = 8:1:1

# --- GC 收集器选择（JDK 1.8） ---
-XX:+UseG1GC                   # 推荐 G1（停顿可控）
# -XX:+UseConcMarkSweepGC      # CMS（低延迟，但有浮动垃圾问题）
# -XX:+UseParallelGC           # Parallel（高吞吐，默认）

# --- GC 日志核心参数（必开） ---
-XX:+PrintGCDetails            # 详细 GC 信息
-XX:+PrintGCDateStamps         # 绝对日期时间戳
-XX:+PrintGCTimeStamps          # 相对时间戳
-Xloggc:/var/log/app/gc-%t.log # 日志文件路径

# --- 日志轮转（生产必备） ---
-XX:+UseGCLogFileRotation       # 开启轮转
-XX:NumberOfGCLogFiles=5        # 保留 5 个文件
-XX:GCLogFileSize=20M           # 单个文件 20MB

# --- 进阶参数（按需） ---
-XX:+PrintHeapAtGC              # GC 前后打印堆
-XX:+PrintTenuringDistribution  # 对象年龄分布
-XX:+PrintGCCause               # GC 触发原因
-XX:+PrintGCApplicationStoppedTime  # 应用停顿时间

# --- OOM 时自动 dump 堆（排查用） ---
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heapdump.hprof

# ============================================
# 完整启动命令示例
# ============================================
java -Xms4g -Xmx4g \
     -XX:NewRatio=2 -XX:SurvivorRatio=8 \
     -XX:+UseG1GC \
     -XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCTimeStamps \
     -Xloggc:/var/log/app/gc-%t.log \
     -XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=5 -XX:GCLogFileSize=20M \
     -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/app/heapdump.hprof \
     -jar myapp.jar

面试回答要点

1

开启参数：JDK 1.8 使用 -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:file 开启 GC 日志，生产环境必须加 -XX:+UseGCLogFileRotation 做日志轮转。

2

查看方式：命令行用 grep/awk 快速筛选 Full GC 行；可视化用 GCEasy（在线）、GCViewer（本地）生成图表报告。

3

排查思路：先看 Full GC 频率 → 再看回收效率（Full GC 后老年代是否回落）→ 最后看 GC 原因（Allocation Failure / Ergonomics / Metadata GC Threshold）。

4

关键判断：Full GC 后老年代回收率 < 20% 且持续增长 → 内存泄漏；回收率正常但频率高 → 参数调优（增大堆、调整分代比例）。
