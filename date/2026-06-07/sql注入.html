Java + MySQL 注入深度拆解

JDBC • MyBatis • JPA/Hibernate • PreparedStatement

原始漏洞代码 (Java JDBC)

以下是一段典型的 Java Servlet / Spring Controller 中的登录验证代码。所有后续攻击都基于这段代码。

public User login(String username, String password) throws SQLException { // ❌ 字符串拼接构建 SQL String sql = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"; Statement stmt = connection.createStatement(); ResultSet rs = stmt.executeQuery(sql); return rs.next() ? mapUser(rs) : null; }
正常输入时生成的 SQL
SELECT * FROM users
WHERE username='admin'
AND password='123456'
Java 特有的核心问题

使用 Statement + 字符串拼接。Java 在 JVM 中完成拼接后，发送给 MySQL 的已经是一条「被污染」的完整 SQL。MySQL 无法分辨原始代码与用户输入的边界。

1

万能密码 — 永真条件绕过

认证绕过
username 参数
admin
password 参数 (Payload)
' OR '1'='1') #
Java 层效果
rs.next() == true → 返回 admin
Java 拼接后发送给 MySQL 的实际 SQL
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1') #'
MySQL 解析器实际执行的语义
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1') # ' ← 注释截断

Java 视角：JVM 只是做了纯字符串拼接，对 SQL 语义毫无感知。拼接结果通过 executeQuery() 原样发给 MySQL。

MySQL 视角：' 闭合引号 → OR '1'='1' 永真 → # 注释掉尾部多余引号。WHERE 恒为 TRUE。

业务后果：rs.next() 返回 true，mapUser(rs) 取出第一行记录（通常是 admin），登录成功。

2

空格注释 — 跳过密码校验

认证绕过
username 参数 (Payload)
admin' --
⚠️ 末尾必须有至少一个空格
password 参数
任意值（被注释忽略）
Java 层效果
AND password=... 整段消失
MySQL 实际执行
SELECT * FROM users WHERE username='admin' -- ' AND password='anything'

与场景1的本质区别：场景1是「让条件恒真」，这里是「直接删除密码校验从句」。即使密码字段有复杂加密逻辑，也完全被绕过。

MySQL -- 规范：双横线后必须跟空白字符。如果攻击者提交 admin'--（无空格），MySQL 不会识别为注释，反而会报语法错误——这本身也是一个注入探测信号。

UNION 注入 — 窃取任意数据

数据泄露
username 参数 (Payload)
' UNION SELECT 1,table_name,3,4,5 FROM information_schema.tables WHERE table_schema=database() #
password 参数
任意值
Java 层效果
ResultSet 中包含所有表名
MySQL 实际执行
SELECT * FROM users WHERE username='' UNION SELECT 1, table_name, 3, 4, 5 FROM information_schema.tables WHERE table_schema=database() # ...

Java 层的致命配合：如果后端用 while(rs.next()) 遍历结果集并渲染到页面，UNION 查出的所有表名会直接回显给攻击者。

列数对齐：users 表有 5 列，所以 UNION SELECT 也必须 5 列。攻击者通过 ' UNION SELECT 1,2,3,4,5 # 逐步试探。

升级路径：表名 → information_schema.columns 获取列名 → UNION SELECT 1,password,3,4,5 FROM users 拖库。

堆叠注入 — 执行任意 SQL

数据破坏
username 参数 (Payload)
admin'; DROP TABLE users; #
Java 驱动关键差异
JDBC 默认禁止多语句
触发条件
需 allowMultiQueries=true
拼接后的 SQL
SELECT * FROM users WHERE username='admin'; DROP TABLE users; # ' AND password='...'

Java JDBC 特殊性：MySQL Connector/J 默认 allowMultiQueries=false，此时 executeQuery() 遇到分号会抛异常，攻击失败。

但风险依然存在：很多项目为了批量操作方便，在 JDBC URL 中加了 ?allowMultiQueries=true，此时堆叠注入完全生效。

替代攻击面：即使多语句被禁，前三种注入（永真/注释/UNION）仍然完全有效。堆叠注入不是必要条件。

Java 防御方案全览

Java 生态中 SQL 注入防御的核心原则：永远不要拼接 SQL 字符串。根据使用的数据访问技术，选择对应的参数化方案。PreparedStatement 在协议层面将 SQL 模板与参数分离，MySQL 服务端先编译模板再绑定数据，注入载荷只会被当作纯字符串值处理。

方案 A：原生 JDBC PreparedStatement

❌ Statement + 拼接
String sql = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"; Statement stmt = conn.createStatement(); ResultSet rs = stmt.executeQuery(sql);
✅ PreparedStatement + 占位符
String sql = "SELECT * FROM users WHERE username=? AND password=?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, username); ps.setString(2, password); ResultSet rs = ps.executeQuery();
关键点：JDBC 用 ? 作为占位符（不是 %s）。setString() 会自动处理转义和类型安全。绝对不要写成 "WHERE username='" + "?" + "'"'——把 ? 放在引号内会变成字面量问号。

方案 B：MyBatis #{} vs ${}

❌ ${} 字符串替换（等价于拼接）
<select id="findUser"> SELECT * FROM users WHERE username = '${username}' AND password = '${password}' </select>
✅ #{} 预编译参数绑定
<select id="findUser"> SELECT * FROM users WHERE username = #{username} AND password = #{password} </select>
${} 的唯一合法用途：动态表名、列名、ORDER BY 字段等不能用占位符的位置。此时必须加白名单校验，绝不能直接传入用户输入。

方案 C：JPA / Hibernate

❌ HQL/JPQL 拼接
String hql = "FROM User WHERE username='" + username + "'"; List<User> list = session.createQuery(hql).list();
✅ 命名参数 / Criteria API
String hql = "FROM User WHERE username = :name"; List<User> list = session.createQuery(hql) .setParameter("name", username) .list();
注意：HQL/JPQL 拼接同样存在注入！Hibernate 的 createQuery() 不自动防注入，只有 setParameter() 才安全。Criteria API / Specification 天然安全。

Java 项目安全检查清单

  • 全局搜索 createStatement() → 全部替换为 PreparedStatement
  • 全局搜索 MyBatis ${} → 确认是否可改为 #{}
  • 检查 JDBC URL 中的 allowMultiQueries=true → 非必要则移除
  • HQL/原生 SQL 查询全部使用 setParameter()
  • 集成 SonarQube / SpotBugs SQL 注入规则到 CI

Java 中这些方法不能防注入

  • StringEscapeUtils.escapeSql() — Apache Commons Lang 已废弃此方法，不可靠
  • 手动 replace("'","\\'") — GBK 等多字节编码可绕过
  • Spring MVC @RequestParam 参数校验 — 只验格式，不防注入
  • ORM 框架本身 — MyBatis ${} 和 HQL 拼接仍有漏洞
  • WAF / 黑名单过滤 — 仅作为纵深防御的辅助层
