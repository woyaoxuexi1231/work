Java + MySQL 注入深度拆解

JDBC • MyBatis • JPA/Hibernate • PreparedStatement

原始漏洞代码 (Java JDBC)

以下是一段典型的 Java Servlet / Spring Controller 中的登录验证代码。所有后续攻击都基于这段代码。

public User login(String username, String password) throws SQLException { // ❌ 字符串拼接构建 SQL String sql = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"; Statement stmt = connection.createStatement(); ResultSet rs = stmt.executeQuery(sql); return rs.next() ? mapUser(rs) : null; }
正常输入时生成的 SQL
SELECT * FROM users
WHERE username='admin'
AND password='123456'
Java 特有的核心问题

使用 Statement + 字符串拼接。Java 在 JVM 中完成拼接后，发送给 MySQL 的已经是一条「被污染」的完整 SQL。MySQL 无法分辨原始代码与用户输入的边界。

1

万能密码 — 永真条件绕过

认证绕过
username 参数
admin
password 参数 (Payload)
' OR '1'='1') #
Java 层效果
rs.next() == true → 返回 admin
Java 拼接后发送给 MySQL 的实际 SQL
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1') #'
MySQL 解析器实际执行的语义
SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1') # ' ← 注释截断

Java 视角：JVM 只是做了纯字符串拼接，对 SQL 语义毫无感知。拼接结果通过 executeQuery() 原样发给 MySQL。

MySQL 视角：' 闭合引号 → OR '1'='1' 永真 → # 注释掉尾部多余引号。WHERE 恒为 TRUE。

业务后果：rs.next() 返回 true，mapUser(rs) 取出第一行记录（通常是 admin），登录成功。

AND password=... 整段消失

与场景1的本质区别：场景1是「让条件恒真」，这里是「直接删除密码校验从句」。即使密码字段有复杂加密逻辑，也完全被绕过。

MySQL -- 规范：双横线后必须跟空白字符。如果攻击者提交 admin'--（无空格），MySQL 不会识别为注释，反而会报语法错误——这本身也是一个注入探测信号。

UNION 注入 — 窃取任意数据

Java 层的致命配合：如果后端用 while(rs.next()) 遍历结果集并渲染到页面，UNION 查出的所有表名会直接回显给攻击者。

列数对齐：users 表有 5 列，所以 UNION SELECT 也必须 5 列。攻击者通过 ' UNION SELECT 1,2,3,4,5 # 逐步试探。

升级路径：表名 → information_schema.columns 获取列名 → UNION SELECT 1,password,3,4,5 FROM users 拖库。

堆叠注入 — 执行任意 SQL

数据破坏

admin'; DROP TABLE users; #
Java 驱动关键差异
JDBC 默认禁止多语句

Java JDBC 特殊性：MySQL Connector/J 默认 allowMultiQueries=false，此时 executeQuery() 遇到分号会抛异常，攻击失败。

但风险依然存在：很多项目为了批量操作方便，在 JDBC URL 中加了 ?allowMultiQueries=true，此时堆叠注入完全生效。

替代攻击面：即使多语句被禁，前三种注入（永真/注释/UNION）仍然完全有效。堆叠注入不是必要条件。

Java 防御方案全览：知其然更知其所以然

所有安全方案的本质都是同一个：在协议或解析层面，将「SQL 代码结构」与「用户数据」彻底分离。不同的 Java 技术栈只是实现这一分离的手段不同。理解了底层原理，就能判断任何新写法是否安全。

方案 A：原生 JDBC PreparedStatement

❌ Statement + 拼接
String sql = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"; Statement stmt = conn.createStatement(); ResultSet rs = stmt.executeQuery(sql);
✅ PreparedStatement + 占位符
String sql = "SELECT * FROM users WHERE username=? AND password=?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, username); ps.setString(2, password); ResultSet rs = ps.executeQuery();
为什么这是安全的？（MySQL 协议层分离）

两阶段执行模型：prepareStatement() 时，JDBC 驱动向 MySQL 发送 COM_STMT_PREPARE 命令，MySQL 服务端此时就编译并缓存了 SQL 执行计划。此时 SQL 结构已经固定，? 被标记为纯数据槽位。

二进制参数绑定：setString() + executeQuery() 时，驱动发送 COM_STMT_EXECUTE，参数以独立的二进制数据包传输，永远不会被拼接到 SQL 字符串中。MySQL 直接将数据填入已编译的槽位。

即使输入 ' OR '1'='1：它也只是作为 username 字段的完整字符串值参与等值比较，不可能改变 WHERE 子句的逻辑结构。因为 SQL 结构在参数到达之前就已经编译完成了。

常见误区：绝对不要写成 "WHERE username='" + "?" + "'"'——把 ? 放在引号内，MySQL 会将其视为字面量问号字符，而非参数槽位，预编译完全失效。

方案 B：MyBatis #{} vs ${}

❌ ${} 字符串替换（等价于拼接）
<select id="findUser"> SELECT * FROM users WHERE username = '${username}' AND password = '${password}' </select>
✅ #{} 预编译参数绑定
<select id="findUser"> SELECT * FROM users WHERE username = #{username} AND password = #{password} </select>
为什么 #{} 是安全的而 ${} 不是？

${} 的本质是文本替换：MyBatis 在构建 SQL 字符串阶段，就把 ${username} 的值直接拼接进 SQL 文本。这与 Java 代码中用 + 拼接没有任何区别。拼接完成后才交给 JDBC，此时注入载荷已经成为 SQL 代码的一部分。

#{} 的本质是 PreparedStatement 封装：MyBatis 解析 XML 时，将 #{username} 替换为 ? 占位符，生成 WHERE username = ?。然后通过 JDBC 的 PreparedStatement.setXxx() 绑定参数。最终走的仍然是方案 A 的两阶段预编译协议。

一句话总结：#{} = PreparedStatement（安全），${} = Statement + 字符串拼接（危险）。MyBatis 本身不创造安全性，它只是 JDBC 的语法糖。

${} 的唯一合法用途：动态表名、列名、ORDER BY 字段等 SQL 语法上不允许使用占位符的位置。此时必须加白名单校验（如枚举匹配），绝不能直接传入用户输入。

方案 C：JPA / Hibernate

❌ HQL/JPQL/Native SQL 拼接
String hql = "FROM User WHERE username='" + username + "'"; List<User> list = session.createQuery(hql).list(); // Native SQL 同理 session.createNativeQuery("SELECT * FROM users WHERE name='" + name + "'");
✅ 命名参数 / Criteria API
String hql = "FROM User WHERE username = :name"; List<User> list = session.createQuery(hql) .setParameter("name", username) .list(); // Criteria API 天然免疫 CriteriaBuilder cb = em.getCriteriaBuilder(); CriteriaQuery<User> q = cb.createQuery(User.class); Root<User> root = q.from(User.class); q.where(cb.equal(root.get("username"), username));
为什么 setParameter 安全而 HQL 拼接不安全？

HQL/JPQL 拼接 = 双重注入风险：HQL 是 Hibernate 自定义的查询语言。拼接后的 HQL 字符串先被 Hibernate 的 HQL Parser 解析为 AST（抽象语法树），再翻译成 SQL。如果用户输入改变了 HQL 的 AST 结构，Hibernate 会把恶意语义忠实地翻译成对应的 SQL。ORM 层不会替你过滤注入，它只是忠实的翻译官。

setParameter() 的传递链：调用 setParameter("name", value) 时，Hibernate 将参数存入内部参数列表。当最终执行时，Hibernate 生成的 SQL 中使用 ? 占位符，并通过 JDBC PreparedStatement.setXxx() 绑定。最终仍然回归到方案 A 的预编译协议。

Criteria API 为何天然安全：它根本不构建查询字符串。查询结构通过 Java 对象方法调用（cb.equal(), root.get()）在内存中组装为 AST。用户输入只作为方法参数传入，从物理上就不存在「拼接字符串」这个动作，注入无从发生。

高危认知陷阱：很多开发者认为「用了 Hibernate/JPA 就自动防注入」。事实上，只要代码中出现 createQuery(string) 或 createNativeQuery(string) 且 string 包含拼接，注入风险与裸 JDBC 完全相同。

三种方案的统一本质

技术栈 安全写法 底层实际做的事 安全性来源
JDBC ? + setString() COM_STMT_PREPARE + COM_STMT_EXECUTE MySQL 协议层分离
MyBatis #{} 转换为 ? → 调用 PreparedStatement 委托给 JDBC 预编译
JPA/Hibernate setParameter() 生成带 ? 的 SQL → 调用 PreparedStatement 委托给 JDBC 预编译
Criteria API cb.equal()... 对象组装 AST → 生成带 ? 的 SQL → PreparedStatement 无字符串拼接 + 预编译

结论：Java 生态中所有安全的 SQL 操作，最终都归结为 JDBC PreparedStatement 的预编译协议。任何绕过这条路径的写法，都需要额外证明其安全性。

Java 项目安全检查清单

  • 全局搜索 createStatement() → 全部替换为 PreparedStatement
  • 全局搜索 MyBatis ${} → 确认是否可改为 #{}，不可改则加白名单
  • 全局搜索 createQuery( / createNativeQuery( → 检查是否有字符串拼接
  • 检查 JDBC URL 中的 allowMultiQueries=true → 非必要则移除
  • 集成 SonarQube / SpotBugs SQL 注入规则到 CI 流水线

Java 中这些方法不能防注入

  • StringEscapeUtils.escapeSql() — Apache Commons Lang 已废弃，多字节编码可绕过
  • 手动 replace("'","\\'") — GBK/BIG5 编码下可用宽字节注入绕过
  • Spring @Valid / @Pattern — 只验格式，合法格式的 payload 仍可注入
  • 认为「用了 ORM 就安全」— MyBatis ${} 和 HQL 拼接同样有漏洞
  • WAF / 黑名单过滤 — 仅作为纵深防御辅助层，不能作为唯一防线
