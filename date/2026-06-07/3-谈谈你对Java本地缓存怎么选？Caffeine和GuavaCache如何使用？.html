本地缓存适用场景

  • ✅ 高频读取、低写入频率的数据（配置、字典、用户会话）
  • ✅ 计算成本高且结果可复用的对象（复杂SQL聚合、RPC结果）
  • ✅ 允许一定数据延迟，需要大幅降低数据库/下游压力
  • ✅ 单机应用或分布式下无需跨节点强一致性的场景
💡 与分布式缓存（Redis）互补：本地缓存更接近应用，纳秒级延迟，无网络开销。

Caffeine 🆚 Guava Cache

对比维度 Caffeine Guava Cache
淘汰算法W-TinyLFU (近乎最优的命中率，兼顾频率与新鲜度)LRU (最近最少使用，实现简单但命中率较低)
读写性能极高，异步读写，支持并发优化（吞吐量高出Guava 2~3倍）优秀，但高并发下锁竞争略明显
异步加载原生支持 CacheLoader.asyncReloading，AsyncLoadingCache不支持异步加载，需自行包装CompletableFuture
统计监控内置 recordStats()，提供命中率、平均加载时间等需 CacheBuilder.recordStats()，功能基本一致
权重与大小支持基于权重驱逐、最大容量、弱/软引用同样支持权重、软/弱引用
刷新策略refreshAfterWrite 配合异步加载，支持并发安全刷新refreshAfterWrite，但刷新时可能阻塞读取线程
空值处理允许缓存null（通过Optional包装或自定义逻辑）支持，但加载器返回null时抛出异常
诞生背景Google开源于2018，Guava Cache的进化版，被Spring 5+ 默认采用Google Guava库，成熟稳定，广泛使用
🎯 选型结论： 新项目首选 Caffeine，性能与命中率均有明显优势；若项目已重度依赖Guava且缓存压力不大，可沿用Guava Cache。

Caffeine · 实战指南

高性能首选

Caffeine 提供类似 Guava 但更强大的 API，支持同步/异步加载，可配置过期策略、刷新策略、统计等。

1️⃣ 同步缓存 (LoadingCache)

import com.github.benmanes.caffeine.cache.*;

// 构建CacheLoader，自动加载
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10000) // 最大条目
    .expireAfterWrite(5, TimeUnit.MINUTES) // 写入后过期
    .refreshAfterWrite(1, TimeUnit.MINUTES) // 异步刷新
    .recordStats() // 开启统计
    .build(key -> userDao.findById(key));

// 使用方式：自动加载或手动放入
User user = cache.get("userId_123"); // 若不存在则调用loader
cache.put("key", manualValue); // 手动放入

2️⃣ 异步缓存 (AsyncLoadingCache)

AsyncLoadingCache<String, User> asyncCache = Caffeine.newBuilder()
    .maximumSize(5000)
    .expireAfterWrite(10, TimeUnit.SECONDS)
    .buildAsync(key -> fetchUserFromDB(key));

// 返回 CompletableFuture
CompletableFuture<User> future = asyncCache.get("userId");
future.thenAccept(user -> System.out.println(user));

// 异步+批量加载
AsyncLoadingCache<String, User> batchCache = Caffeine.newBuilder()
    .buildAsync(new AsyncCacheLoader<String, User>() {
      @Override
      public CompletableFuture<User> asyncLoad(String key, Executor executor) {
          return CompletableFuture.supplyAsync(() -> dao.find(key));
      }
    });
⏱️ 多种过期策略 
expireAfterAccess(1, TimeUnit.HOURS)  // 访问后过期
expireAfterWrite(1, TimeUnit.HOURS)   // 写入后过期
expireAfter(Expiry)                   // 自定义过期
📊 监控统计示例 
CacheStats stats = cache.stats();
System.out.println(stats.hitRate());      // 命中率
System.out.println(stats.loadSuccessCount());

Guava Cache · 经典实践

稳定可靠

Guava Cache 是本地缓存的老牌选手，API 设计简洁，适合中小规模项目。

基本构建方式

import com.google.common.cache.*;

CacheLoader<String, User> loader = new CacheLoader<String, User>() {
    @Override
    public User load(String key) throws Exception {
        return userService.findByKey(key); // 加载逻辑
    }
};

LoadingCache<String, User> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .recordStats()
    .build(loader);

User user = cache.get("userId"); // 自动加载
cache.put("userId2", customUser); // 手动填充

高级特性：权重/监听/刷新

// 基于权重的驱逐 (例如1KB为一个单位)
LoadingCache<String, String> weightCache = CacheBuilder.newBuilder()
    .maximumWeight(1000)
    .weigher((k, v) -> v.length())
    .build(loader);

// 移除监听器
CacheBuilder.newBuilder()
    .removalListener(notification -> {
        System.out.println("移除 key:" + notification.getKey());
    })
    .build(loader);

// 注意：Guava 的 refreshAfterWrite 需要重写 reload 方法以实现异步，否则依然同步。
⚠️ Guava 注意事项：当缓存同时设置 expireAfterWrite 和 refreshAfterWrite，refresh 会在 expire 之前尝试刷新，但刷新未完成时返回旧值；若需要完全异步加载，可配合 LoadingCache.refresh() 自行提交线程池。

缓存难题应对策略（本地缓存视角）

缓存穿透

大量请求查询不存在的数据 → 缓存无记录，一直穿透到DB。解决方案：缓存空对象（Caffeine允许put null）或布隆过滤器。

缓存击穿

热点key过期瞬间大量并发请求重建缓存。Caffeine的 CacheLoader 天然防并发，只允许一个线程加载，其他等待。

缓存雪崩

大量缓存同时失效。本地缓存可设置 expireAfterWrite 时加入随机偏移量，避免集体失效。

💡 Caffeine 的 refreshAfterWrite 机制可以在缓存过期后，返回旧值的同时异步刷新，有效防止击穿，并将雪崩风险降到最低。

从 Guava 迁移到 Caffeine

依赖替换 + API 几乎无缝兼容，修改少量import即可。

// Guava 写法
import com.google.common.cache.CacheBuilder;
CacheBuilder.newBuilder().maximumSize(1000);

// Caffeine 写法 (仅需改包名)
import com.github.benmanes.caffeine.cache.Caffeine;
Caffeine.newBuilder().maximumSize(1000);

// 其他 API 方法基本一一对应，但 Caffeine 额外提供异步和更丰富的驱逐。

最佳实践 & 避坑指南

✅预估容量：设置合理的maximumSize或maximumWeight，避免内存泄漏。
✅合理选择过期策略：写频繁数据用expireAfterWrite，读频繁用expireAfterAccess。
✅监控命中率：线上开启recordStats，结合指标系统告警命中率过低。
✅避免缓存大对象：使用weigher按内存权重驱逐，防止堆外压力。
⚠️刷新 vs 过期：refreshAfterWrite适合“容忍旧值”场景，需配合异步加载防止阻塞。
⚠️弱引用键：使用弱引用时GC可能造成数据丢失，谨慎使用。
📌 总结：Caffeine 是 Guava Cache 的现代替代品，除了极少数兼容场景外，都应优先采用。Spring Boot 2.x+ 的 CacheManager 默认就使用 Caffeine，足以说明其行业地位。
