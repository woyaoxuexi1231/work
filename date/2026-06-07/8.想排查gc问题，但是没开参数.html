生产环境应急指南

没开 GC 日志？
照样排查

应用正在运行，GC 日志没开，老年代内存还在涨 —— 别慌，用 JDK 自带工具在线诊断，无需重启

紧急排查 工具速查

⚠️ 紧急场景：GC 日志未开启，应用正在运行

生产环境最常见的情况：应用启动时没配 GC 日志参数，现在老年代内存缓慢增长，需要立即排查，但不能重启（重启会丢失现场）。

不能重启 不能改参数 需要在线诊断
01

黄金 5 分钟排查法

不重启、不改参数，纯在线诊断

1

找到 Java 进程 PID

10秒 
# 查看所有 Java 进程
jps -lvm

# 输出示例：
# 12345 myapp.jar -Xms2g -Xmx2g
# 12346 sun.tools.jps.Jps -lvm

# 或者用 ps 过滤
ps -ef | grep java | grep -v grep

拿到目标应用的 PID（如 12345），后续所有命令都基于这个 PID

2

实时监控 GC 与内存（jstat）

30秒 
# 最常用：查看 GC 统计（百分比）
jstat -gcutil PID 5000 10

# 每 5 秒采样一次，共 10 次
# 关注列：S0/S1/E/O/M/ YGC/FGC/ GCT
# 查看详细内存容量（KB）
jstat -gc PID 5000 10

# 关注列：
# S0C/S1C/EC/OC/MC = 各区域总容量
# S0U/S1U/EU/OU/MU = 各区域已使用

jstat -gcutil 输出解读

S0/S1

Survivor 0/1 区使用率

E

Eden 区使用率

O

Old Gen 使用率 ⭐

M

Metaspace 使用率

YGC

Young GC 次数

FGC

Full GC 次数 ⭐

YGCT

Young GC 总耗时

FGCT

Full GC 总耗时 ⭐

💡 判断技巧

连续执行 jstat -gcutil PID 5000（不加次数，持续观察），如果 O 列（老年代使用率）持续上涨，且 FGC 次数不断增加但 O 降不下来 → 确认内存泄漏或对象过早晋升

3

查看对象统计（jmap -histo）

1分钟 
# 查看存活对象统计（触发一次 Full GC 后统计，更准）
jmap -histo:live PID | head -n 30

# 不触发 Full GC，直接统计
jmap -histo PID | head -n 30

# 输出示例：
#  1:        123456     9876480  java.lang.String
#  2:         54321     8691360  [B
#  3:         12345     5925600  java.util.HashMap$Node
#  4:          8888     4555520  com.myapp.CacheEntry   <-- 重点关注！

📊 看什么？

找自己业务相关的类是否排名靠前。如果某个业务类（如 CacheEntry、UserData）实例数异常多，大概率是缓存泄漏或对象未释放

⚠️ 注意

jmap -histo:live 会触发 Full GC，生产环境慎用！建议先用不加 :live 的版本，确认问题后再用 :live 精确分析

4

查看堆配置与使用（jmap -heap）

30秒 
# 查看堆内存详细配置和使用情况
jmap -heap PID

# 输出关键信息：
# - 使用的 GC 收集器（G1/CMS/Parallel）
# - 堆内存各区域容量（NewSize/OldSize/MaxHeapSize）
# - 各区域当前使用量
# - 是否使用了压缩指针等

用途：快速确认当前 JVM 的堆配置是否合理。比如老年代容量是否过小、年轻代比例是否失衡，这些都会导致对象过早晋升到老年代

5

导出堆快照（jmap -dump）

2-5分钟 
# 导出堆快照（live 只保留存活对象，文件更小）
jmap -dump:live,format=b,file=/tmp/heap_dump.hprof PID

# 不区分存活/死亡，全部导出（文件较大）
jmap -dump:format=b,file=/tmp/heap_dump.hprof PID

# 如果 jmap 报错，用 gcore 替代（Linux）
gcore PID
jmap -dump:format=b,file=/tmp/heap_dump.hprof /path/to/java core.PID

⚠️ 生产环境注意

jmap -dump 会暂停应用（STW），堆越大停顿越久。建议在业务低峰期执行，或使用 gcore 先 dump 整个进程内存，再离线转换

📁 后续分析

导出后用 MAT（Memory Analyzer Tool）或 JVisualVM 打开，使用 Dominator Tree 找到占用最大的对象及 GC Roots 引用链

02

Arthas 在线诊断（推荐）

阿里巴巴开源，无需重启，功能最强大的在线工具

为什么用 Arthas？

  • 无需重启：attach 到运行中的进程即可
  • 功能全面：内存、线程、方法耗时、火焰图、反编译
  • 低侵入：对应用性能影响极小
  • 实时看板：dashboard 命令实时刷新内存和 GC 状态

快速安装启动

# 下载并启动（自动识别 Java 进程）
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

# 选择目标进程编号
# 进入 Arthas 命令行界面

排查老年代问题的核心命令

dashboard 实时看板

实时显示线程、内存、GC、运行时信息，自动刷新

$ dashboard -n 5 -i 2000
# -n 5: 刷新 5 次后退出
# -i 2000: 每 2 秒刷新一次

# 关注输出中的：
# HEAP-MEMORY: used / total / max
# G1-Heap: Eden / Survivor / Old Gen 使用量
# GC-COUNT: YGC / FGC 次数
memory 内存详情

查看 JVM 各内存区域的详细使用情况，比 jstat 更直观

$ memory

# 输出示例：
# Heap Memory:
#   used: 512MB, committed: 1GB, max: 2GB
# G1 Old Gen:
#   used: 380MB, committed: 800MB
# Metaspace:
#   used: 45MB, committed: 128MB
heapdump 堆快照

在线导出堆快照，无需 jmap，更稳定

$ heapdump /tmp/arthas_dump.hprof

# 或只 dump 存活对象（文件更小）
$ heapdump --live /tmp/arthas_dump.hprof

# 导出后用 MAT 分析 Dominator Tree
classloader 类加载器

查看 ClassLoader 统计，排查 Metaspace 泄漏

$ classloader

# 输出每个 ClassLoader 加载的类数量
# 如果发现大量自定义 ClassLoader（如 GroovyClassLoader）
# 且加载了成千上万个类 → Metaspace 泄漏
vmtool 强制GC

强制触发 GC，观察内存回收情况（类似 jmap -histo:live 的效果）

$ vmtool --action forceGc

# 强制 GC 后，再用 dashboard/memory 查看
# 如果 Old Gen 几乎没降 → 确认内存泄漏
03

动态开启 GC 日志（jinfo）

不重启应用，临时开启 GC 日志打印

ℹ️ 适用场景

虽然之前没开 GC 日志，但可以通过 jinfo 在运行时动态开启部分参数（注意：并非所有参数都支持动态修改，但 PrintGCDetails 和 PrintGC 通常可以）

# 1. 先查看当前是否开启了 GC 日志
jinfo -flag PrintGCDetails PID
# 输出：-XX:-PrintGCDetails  （减号表示未开启）

# 2. 动态开启 PrintGC（基础 GC 信息）
jinfo -flag +PrintGC PID
# 输出：成功则无返回值

# 3. 尝试开启 PrintGCDetails（详细 GC 信息）
jinfo -flag +PrintGCDetails PID

# 4. 验证是否开启成功
jinfo -flag PrintGCDetails PID
# 输出：-XX:+PrintGCDetails  （加号表示已开启）

# 5. 开启后，GC 信息会输出到控制台（stdout）
# 如果应用日志重定向到文件，可以在日志文件中看到 GC 信息

✅ 可以动态开启的参数

  • • PrintGC / PrintGCDetails
  • • PrintGCTimeStamps
  • • HeapDumpOnOutOfMemoryError
  • • MaxHeapFreeRatio / MinHeapFreeRatio

❌ 不能动态修改的参数

  • • Xmx / Xms（堆大小）
  • • Xloggc（日志文件路径）
  • • UseG1GC / UseConcMarkSweepGC（GC 收集器）
  • • NewRatio / SurvivorRatio（分代比例）

这些参数需要修改后重启才能生效

⚠️ 重要提示

动态开启的 GC 日志只会输出到 stdout/stderr，不会写入文件。如果应用日志做了重定向，可以在日志文件中找到；否则只能在控制台看到。建议配合 nohup 或 systemd 的日志收集查看

05

完整排查流程图

从发现到定位，一步不落

🔴 发现问题

监控告警：老年代内存使用率 > 80% 且持续增长，或应用响应变慢、偶发卡顿

1

🟡 第一步：jps 找 PID

jps -lvm

确认目标 Java 进程 PID

2

🔵 第二步：jstat 确认趋势

jstat -gcutil PID 5000

观察 O 列（老年代使用率）是否持续上涨，FGC 次数是否频繁增加

O 稳定 → 正常波动 O 持续上涨 → 进入第三步
3

🟣 第三步：jmap -histo 定位嫌疑类

jmap -histo PID | head -n 30

查看对象统计，找业务相关类是否异常多。如果找到嫌疑类 → 进入代码审查；如果找不到 → 进入第四步

4

🟢 第四步：heapdump + MAT 深度分析

jmap -dump:live,format=b,file=/tmp/heap.hprof PID

导出堆快照，用 MAT 分析 Dominator Tree 和 Leak Suspects 报告，找到 GC Roots 引用链，定位到具体代码

✅ 修复与验证

修复代码后重新部署，用 jstat 持续观察，确认老年代内存不再异常增长。同时补充 GC 日志参数，防止下次再踩坑

面试回答要点

1

首选 jstat：不重启、无侵入，jstat -gcutil PID 5000 实时监控老年代使用率和 GC 次数，是最快确认问题的手段。

2

推荐 Arthas：如果环境允许安装第三方工具，Arthas 是功能最全面的在线诊断方案，dashboard、memory、heapdump 命令一条龙。

3

jmap 慎用：jmap -histo:live 和 jmap -dump 都会触发 Full GC 或 STW，生产环境建议在低峰期执行，或用 gcore 替代。

4

jinfo 补救：可以通过 jinfo -flag +PrintGC PID 动态开启基础 GC 日志，虽然不能写文件，但至少能在控制台看到 GC 情况。
