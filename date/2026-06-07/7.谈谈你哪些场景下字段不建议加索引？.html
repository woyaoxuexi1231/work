⚡ 索引不是万能灵药 | 反模式陷阱

哪些场景下字段不建议加索引？

索引能加速查询，但滥用会引发写入暴跌、空间膨胀。
深入剖析7大反模式场景，让索引设计恰到好处。

💰

索引的代价

  • 📉 写放大：每次 INSERT/UPDATE/DELETE 需同步维护所有索引，额外磁盘IO
  • 💾 空间占用：索引可能占据表空间的50%甚至更多
  • 🧠 内存竞争：索引页挤占Buffer Pool，降低热数据命中率
  • 🔍 优化器负担：过多索引导致执行计划选择成本升高
💡 原则：索引是为查询服务的奢侈品，应保证收益远大于成本。
🎯

何时该加索引？

✅ 频繁作为 WHERE / JOIN / ORDER BY 条件的字段

✅ 高选择性字段（区分度 > 10%~15%）

✅ 读远多于写的表，且查询延迟敏感

✅ 需要保证唯一性的字段（唯一索引）

—— 反之，则应三思而后加
🚫

强烈不建议加索引的典型场景

📊 低选择性（低基数）字段

反模式 #1

例如：性别、状态枚举（只有0/1）、布尔字段、类型type固定几种值。

🧪 原理：当字段区分度很低（比如男/女），二级索引需要回表扫描大量重复数据，优化器可能直接放弃索引走全表扫描，索引形同虚设。
-- 低选择性字段索引几乎无效
CREATE INDEX idx_gender ON user(gender); -- gender只有'M','F'
SELECT * FROM user WHERE gender = 'M';
-- 优化器大概率全表扫描，因为访问超过一定比例(约20%)后回表代价更高。

✅ 替代方案：联合索引中放在最后一位，或结合其他高筛选条件。

✍️ 频繁更新的字段

反模式 #2

例如：登录时间、last_updated、计数字段、状态变更极高的列。

💥 每次更新都需要同步修改B+树索引，产生大量页分裂、合并操作，导致性能雪崩。
-- 高频率更新字段不宜单独索引
UPDATE product SET view_count = view_count + 1 WHERE id = ?;
-- 若view_count有索引，每次更新都要维护索引，浪费严重。

✅ 建议：仅对查询条件中的字段建索引，更新字段本身除非经常作为过滤条件。

🗄️ 极小表（几十到几百行）

反模式 #3

例如：配置表、行政区划表、码表等。

📌 全表扫描比索引查找更快：索引需要先查索引树再回表，而小表一次性顺序IO即可完成。MySQL优化器也会选择全表扫描。
-- 小表无需额外索引
CREATE TABLE config (id INT, key VARCHAR, value VARCHAR); -- 仅50行
SELECT * FROM config WHERE key='theme'; -- 全表扫描即可，索引开销大于收益。

📄 长文本、BLOB、超长VARCHAR

反模式 #4

例如：文章内容、JSON日志、评论详情。

🚨 直接索引会使得索引体积巨大，效率极低。即使使用前缀索引，若前缀区分度差也无法有效过滤。
-- 错误示范：对text字段完整索引
CREATE INDEX idx_content ON articles(content); -- content为TEXT，索引巨大
-- 改进：可使用全文索引（Full-Text）或前缀索引，但大多数场景不建议索引。

✅ 推荐：使用Elasticsearch等专门搜索引擎，或限制前缀长度但必须测试选择性。

❓ 大量NULL值字段

反模式 #5

例如：可选的外键、预留属性字段，90%以上为NULL。

🧩 B+树索引通常不存储全NULL值（取决于数据库实现），查询非NULL时效率仍差，且浪费空间。大部分数据库对NULL值索引优化有限。
-- 表中80%记录的deleted_at为NULL
CREATE INDEX idx_deleted ON soft_delete_table(deleted_at);
SELECT * FROM table WHERE deleted_at IS NOT NULL; -- 极少数查询，索引几乎无用。

📈 与查询模式不匹配的字段

反模式 #6

例如：只做 YEAR(date) 函数查询，对原始列加索引无效。

⚠️ 使用函数、计算或隐式类型转换会使得索引失效。如果查询条件总是包裹表达式，索引就是白建。
-- 索引不会被使用
CREATE INDEX idx_create_time ON orders(create_time);
SELECT * FROM orders WHERE DATE(create_time) = '2025-01-01';
-- 正确方式：范围查询 create_time BETWEEN ... AND ...

📝 极低查询负载的归档/日志表

反模式 #7

例如：操作日志、审计表、点击流数据，只做追加写入基本不查询。

🚀 索引会严重拖累写入TPS。如果业务很少查询这类表，应当不加索引，甚至考虑使用专门的时序数据库。
-- 每秒写入数千条的日志表，不应加任何二级索引
CREATE TABLE access_log (id BIGINT PRIMARY KEY, log_time DATETIME, detail TEXT);
-- 需要分析时可单独拉取到分析库，不要影响实时写入。

索引决策速查表

场景特征 是否建议索引 核心原因
选择性 > 20% 且频繁查询✅ 强烈推荐迅速过滤大部分数据
选择性 < 5% (性别/状态)❌ 不推荐索引几乎无效，回表开销大
频繁更新且非查询条件❌ 坚决不加写放大严重
表记录 < 1000 行❌ 没必要全表扫描更高效
长文本 / 超大字段❌ 禁止普通索引体积庞大、维护成本高
高频写 + 低频读 (日志表)❌ 避免写入性能大幅下降
唯一性约束 (业务主键)✅ 唯一索引保证数据完整性

深度原理解析

低选择性 → 回表地狱

InnoDB二级索引存储主键值，若字段重复度高，每条索引项几乎都要回表查聚集索引，随机IO巨大。优化器宁愿全表顺序IO也不愿走索引。

频繁更新 → 页分裂与合并

每次更新需维护B+树平衡，可能触发叶子节点分裂（分配新页）或合并，加剧锁竞争和redo日志量，对TPS影响可达30%~50%。

小表扫描 vs 索引搜索

全表扫描利用预读和顺序IO，成本约为1次随机IO；而索引搜索需2~3次随机IO（索引树+回表），当表数据量少于某个阈值（如几十个数据页）时，扫描更优。

🔬 实验数据：性别字段（二分）范围查询时，MySQL 5.7+ 优化器通常采用全表扫描；若强制使用索引，耗时反而增加2~5倍（取决于数据分布）。
🛠️

索引设计黄金法则

✔️ 给高选择性、高频查询字段建索引（如用户ID、订单号）
✔️ 优先考虑联合索引，遵循最左前缀原则，覆盖索引减少回表
❌ 避免在低基数列上建立独立索引，除非作为联合索引的末尾字段
❌ 不在频繁DML的表上建过多索引（一般建议<5个二级索引）
✔️ 定期通过 pt-index-usage 工具清理无用索引
✔️ 对于超大表，使用分区或归档历史数据减少索引体积
💡 面试点睛：当被问及索引设计，除了列举场景，更要强调“收益分析”——读性能提升是否值得写入与空间的牺牲。优秀的DBA会用Explain和实际业务负载驱动决策。
🎤 面试官追问：

"如果某个状态字段只有3种值，但有80%的查询都是查其中一种，是否应该建索引？"

✅ 参考答案： 不建议单独索引。因为查询占比过高，优化器依然会全表扫描。但可以考虑在联合索引中作为前导列，且配合其他高筛选字段。也可以使用过滤索引（部分索引）如 WHERE status='ACTIVE' 创建索引，只覆盖热点值，减少索引体积。
🧐 进阶思考：

"有没有可能索引反而使查询变慢？举个例子。"

示例： 一张订单表有status索引（0,1,2），执行 SELECT * FROM order WHERE status=1 命中60%数据，优化器若选错计划使用索引，就需要大量回表随机IO，比全表扫描慢数倍。这也是低选择性索引的经典危害。

索引是利器，但滥用便是债务。理解业务查询模式，从选择性、更新频率、表规模三维度审视每个潜在索引， 才能构建高性能、低维护成本的数据库系统。

📌 记住：删除无用索引和建立高效索引同样重要。
