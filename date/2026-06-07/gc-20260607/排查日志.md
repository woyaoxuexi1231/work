# gc-problem-01

## 我的排查过程

我先使用 jstat -gcutil pid 5000，观察了一会。

E 区一直在涨，YGC 5 次，FGC 0 次，说明可能最开始分配的内存不太够。

后续 O 一直缓慢涨。我推测可能有什么定时任务或者什么东西一直在跑，可能在往静态数组加对象，或者使用 ThreadLocal 导致内存泄漏。

## 对照排查指南后的完整结论

### jstat 核心数据
- O 列 45% → 67%，稳步上涨不回落
- FGC 一直为 0（还没触发 Full GC）

### 手动触发 GC 后
- jcmd <PID> GC.run 触发 Full GC
- O 从 67% → 64%，仅降 3% → 内存泄漏特征

### jmap 对象统计
- [B (byte[]) 实例数 3600+，占用 ~350MB
- 远超 128MB 堆空间，说明大量 byte[] 无法回收
- ConcurrentHashMap$Node 数量也异常

### 根因
ConcurrentHashMap 作为静态成员变量，每秒往里塞数据，但从未清理。
→ 对象全部存活，GC 无法回收 → 经典内存泄漏。
