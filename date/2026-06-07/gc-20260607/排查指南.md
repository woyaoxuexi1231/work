# GC 问题排查完整指南 — gc-20260607

---

## 前置准备：启动项目

每个项目都用以下命令启动（PowerShell 注意引号）：

```powershell
cd gc-problem-01
mvn spring-boot:run '-Dspring-boot.run.jvmArguments=-Xms128m -Xmx128m -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:./gc.log'
```

> 堆设 128MB 是为了让问题快速暴露，生产环境不会这么小。

---

## 通用排查步骤（每个项目都这样来一遍）

```
Step 1: jps -lvm           → 拿 PID
Step 2: jstat -gcutil PID 5000  → 观察 3-5 分钟，盯 O列和FGC列
Step 3: curl /trigger-gc   → 手动触发一次完整 GC，看老年代能不能降
Step 4: jmap -histo PID | head -n 30  → 看哪些对象实例数量异常
Step 5: 打开 gc.log        → 分析 Full GC 的回收效率
```

# gc-problem-01

## Step 1 — jstat 观察

启动后等待 1 分钟，然后执行：

```bash
jps -lvm | grep gc-problem-01
# 拿到 PID，假设是 12345

jstat -gcutil 12345 5000
```

你会看到类似这样的输出（每 5 秒一行）：

```
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
  0.00  99.99  78.12  45.33  92.18  88.11    12    0.156     0    0.000    0.156
  0.00  99.99  82.45  48.91  92.20  88.11    12    0.156     0    0.000    0.156
  0.00  99.99  93.11  52.50  92.20  88.11    14    0.182     0    0.000    0.182
  0.00  99.99  89.30  56.12  92.20  88.11    14    0.182     0    0.000    0.182
  0.00  99.99  85.67  59.77  92.22  88.11    16    0.208     0    0.000    0.208
  0.00  99.99  91.44  63.35  92.22  88.11    16    0.208     0    0.000    0.208
  0.00  99.99  87.56  66.90  92.22  88.11    18    0.234     0    0.000    0.234
```

**观察到什么？**
- O 列（老年代使用率）从 45% → 49% → 53% → 57% → 60% → 63% → 67%，稳步上升，从未下降
- YGC（Young GC）频繁触发，但每次 YGC 后 O 反而更高了
- FGC = 0，还没触发过 Full GC

> ✅ 你的判断是对的：确实有东西在不停创建对象，且这些对象逃逸到了老年代。

---

## Step 2 — 手动触发 GC，看回收率

```bash
curl http://localhost:8101/trigger-gc
```

等几秒后再看 jstat：

```
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
  0.00   0.00   15.12  64.22  92.24  88.11    20    0.260     1    0.045    0.305
```

**关键发现：** Full GC 后，O 从 66.90% → 64.22%，**几乎没降**！（就算再等一会也差不多）

这说明：**老年代里的对象几乎全都是"活"的，GC 收不掉。**

---

## Step 3 — jmap 对象统计

```bash
jmap -histo:live 12345 | head -n 30

# 输出类似：
#  num     #instances         #bytes  class name
#    1:         3600      368640000  [B
#    2:         3599          86376  java.lang.String
#    3:         3599         115168  java.util.concurrent.ConcurrentHashMap$Node
#    4:          ...
```

**观察到什么？**
- `[B`（byte[]）实例数 3600 个，占用约 350MB — 远超 128MB 堆空间！
- String 实例数和 byte[] 基本 1:1 对应
- ConcurrentHashMap$Node 数量也很高

→ **大量 byte[] 被一个 ConcurrentHashMap 持有。**

---

## Step 4 — GC 日志分析

```bash
cat gc.log | grep "Full GC"
```

输出类似：

```
2026-06-09T10:XX:XX.XXX+0800: XX.XXX: [Full GC (System.gc())
  [PSYoungGen: 1024K->0K(38400K)]
  [ParOldGen: 65000K->64000K(87552K)]
  66024K->64000K(125952K),
  [Metaspace: 35000K->35000K(107520K)],
  0.0451234 secs]
```

**关键数据：** `ParOldGen: 65000K->64000K` — 老年代回收前 65MB，回收后 64MB，**只回收了 1MB**。回收率不到 2%。

---

## Step 5 — 结论

| 排查维度 | 结论 |
|----------|------|
| jstat 观察 | O 持续上涨，无人为干预 |
| trigger GC | Full GC 后 O 几乎不降 |
| jmap -histo | byte[] 占比异常，与 ConcurrentHashMap 关联 |
| gc.log | Full GC 老年代回收率 < 2% |

**根因：`ConcurrentHashMap` 作为静态缓存容器，只 put 不 remove，对象永不释放 → 典型的内存泄漏。**

对应代码：

```java
// DataCollector.java 第 17 行
private final Map<String, byte[]> store = new ConcurrentHashMap<>();

// 第 24 行 — 每 2 秒往里塞 100KB，永不清除
@Scheduled(fixedRate = 2000)
public void collectData() {
    store.put("data-" + counter.incrementAndGet(), new byte[100 * 1024]);
}
```

**修复方向：** 要么设置容量上限（LRU 淘汰），要么定期清理过期条目。

```
  0.00  45.12  22.15  35.11  92.10  88.05     9    0.092     0    0.000    0.092
```

## Step 2 — 手动触发 GC


GC 后 O 没有明显变化，本身就低。

## Step 3 — jmap 对象统计

```bash
jmap -histo <PID> | head -n 30
```

```
#  num     #instances         #bytes  class name
#    1:         1200      188743680  [B
#    2:          ...
```

byte[] 占用约 180MB 实例数据，但存活时间很短。

---

## Step 4 — GC 日志分析

```bash
grep "GC (Allocation Failure)" gc.log | tail -10
```

```
[GC (Allocation Failure)
  [PSYoungGen: 28000K->800K(38400K)]
  30000K->5000K(125952K), 0.0081234 secs]

[GC (Allocation Failure)
  [PSYoungGen: 32000K->900K(38400K)]
  35000K->5200K(125952K), 0.0095678 secs]
```

**频繁的 Allocation Failure（分配失败）**，每次 Eden 区都接近满载 → GC。

---

## Step 5 — 结论

| 排查维度 | 结论 |
|----------|------|
| jstat 观察 | Eden 剧烈波动、YGC 极频繁，O 稳定 |
| trigger GC | O 不高 |
| jmap -histo | 大量临时 byte[] |
| gc.log | 大量 Allocation Failure，每次 Eden 满载触发 |

**根因：定时任务每次批量分配 15MB 的 `byte[]`（5 个 × 3MB），直接撑爆 128MB 堆的 Eden 区 → 频繁 Young GC。对象创建后很快 clear() 释放，所以 O 不涨。**

对应代码：

```java
// BatchProcessor.java 第 22 行 — 每 8 秒创建 15MB 大数组
@Scheduled(fixedRate = 8000)
public void processBatch() {
    for (int i = 0; i < 5; i++) {
        tempBuffer.add(new byte[3 * 1024 * 1024]); // 3MB × 5 = 15MB
    }
    tempBuffer.clear();
}
```

**修复方向：** 减少单次分配量（分片处理），或增大年轻代比例。

```bash
cat gc.log | grep "Full GC"
```

```
[Full GC (System.gc())
  [PSYoungGen: 1024K->0K(38400K)]
  [ParOldGen: 44000K->43000K(87552K)]
```

---

## Step 5 — 结论

| 排查维度 | 结论 |
|----------|------|
| jstat 观察 | O 稳步上涨，S1 常满载 |
| trigger GC | Full GC 回收率 < 3% |
| jmap -histo | SessionObject 大量驻留 |
| gc.log | 老年代回收极少 |

**根因：SessionManager 每 3 秒创建 20 个 SessionObject（每个 5KB），但每 200 轮才清理 30 个。创建速度（20×5KB/3s = 33KB/s）远超清理速度（30×5KB/60s = 2.5KB/s）→ 对象持续累积，且 Survivor 区兜不住，大量对象过早晋升到老年代。**

对应代码：

```java
// SessionManager.java
@Scheduled(fixedRate = 3000)
public void manageSessions() {
    // 每 3 秒加 20 个   → 净增 20 个/3秒
    for (int i = 0; i < 20; i++) {
        activeSessions.add(new SessionObject(...));
    }
    // 每 200 轮才清 30 个 → 净减 30 个
    if (sessionIdCounter.get() % 200 == 0) {
        // 只清 30 个...
    }
}
// 生产速度 >> 消费速度 → 永久累积
```

**修复方向：** 对齐生产和清理的速率，或者用 LRU 缓存替代。

## Step 4 — GC 日志分析

GC 日志中能发现一个重要线索，尝试：

```bash
grep "Full GC" gc.log
```

```
[Full GC (System.gc())
  [PSYoungGen: 1024K->0K(38400K)]
  [ParOldGen: 32000K->31000K(87552K)]
  33024K->31000K(125952K), 0.0321234 secs]
```

老年代回收率很低。再看看 GC 前的对象年龄分布：

```
Desired survivor size 6553600 bytes, new threshold 1 (max 15)
- age   1:    8388608 bytes,    8388608 total
```

`new threshold 1` — 意味着对象经历 1 次 GC 就晋升老年代（因为动态年龄判定：某个年龄的对象总大小超过了 Survivor 的 50%）。

---

## Step 5 — 结论

| 排查维度 | 结论 |
|----------|------|
| jstat 观察 | O 稳步上涨，每次 YGC 后都涨 |
| trigger GC | Full GC 几乎不回收 |
| jmap -histo | 大量大 byte[]（每个约 2MB） |
| GC 日志 | 对象 1 次 GC 就晋升老年代 |
| 隐秘线索 | 线程池复用导致数据不释放 |

**根因：`ThreadLocal<byte[]>` 分配 2MB 缓冲区，线程池复用 4 个核心线程，用完从未调用 `remove()`。每个线程的 ThreadLocal 里都锁着一个 2MB 的 byte[]，由于线程存活，这些对象永远无法被 GC → 内存泄漏。**

对应代码：

```java
// AsyncTaskService.java
private final ThreadLocal<byte[]> localBuffer = new ThreadLocal<>();

public void processTask(long taskId) {
    byte[] buf = new byte[2 * 1024 * 1024];
    localBuffer.set(buf);         // 设置进去
    // ... 业务处理 ...
    // ⚠️ 用完从未 localBuffer.remove()
}
```

线程池的 4 个线程永远不会销毁，每个线程的 ThreadLocalMap 里都持有 2MB 的 byte[] 引用 → 8MB 永远占着老年代。

**修复方向：** 在 `processTask` 的 finally 块中调用 `localBuffer.remove()`。

## 四个项目对比总结

| 项目 | 核心特征 | 表现 |
|------|----------|------|
| 01 | 静态 Map 只加不删 | O 持续上涨，FCG 无效 |
| 02 | 批量大对象临时分配 | E 剧烈波动，YGC 高频，O 正常 |
| 03 | 创建远快于清理 | O 慢涨，Survivor 满载 |
| 04 | ThreadLocal 线程池泄漏 | O 稳步上涨，每次 YGC 后跳升 |

## 排查黄金法则

```
1. jstat -gcutil  →  看 O 涨不涨、FGC 多不多
2. curl /trigger-gc →  看 Full GC 后 O 能不能降
    能降 → 参数调优（堆太小、晋升太快）
    不降 → 内存泄漏，进第 3 步
3. jmap -histo    →  哪个类实例数异常
4. gc.log         →  Full GC 回收率、年龄分布
5. 代码确认       →  找到对应的静态容器/ThreadLocal/缓存
```
