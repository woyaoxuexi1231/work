# 为什么 MyBatis 要设计四个拦截对象？

## 1. 核心答案：单一职责 + 可插拔

如果把所有事情都放在 Executor 里，代码会变成这样：

```java
// ❌ 错误设计：所有事情都塞到 Executor 里
public class Executor {
    public List<Object> query(MappedStatement ms, Object param) {
        // 1. 获取连接
        Connection conn = dataSource.getConnection();

        // 2. 创建 Statement（这是 StatementHandler 的职责）
        String sql = ms.getSqlSource().getBoundSql(param).getSql();
        PreparedStatement stmt = conn.prepareStatement(sql);

        // 3. 设置参数（这是 ParameterHandler 的职责）
        // 需要知道每个参数的类型、顺序、JDBC类型...
        stmt.setString(1, param.getName());
        stmt.setInt(2, param.getAge());
        // ... 可能有几十个参数

        // 4. 执行查询
        ResultSet rs = stmt.executeQuery();

        // 5. 处理结果集（这是 ResultSetHandler 的职责）
        // 需要知道如何把 ResultSet 映射成 Java 对象
        // 需要处理关联查询、集合映射、鉴别器...
        List<Object> results = new ArrayList<>();
        while (rs.next()) {
            Object obj = ms.getResultMaps().get(0).getType().newInstance();
            // 反射设置属性... 几十行代码
            results.add(obj);
        }

        return results;
    }
}
```

这个类会变得**几千行**，而且：
- 想换一种参数设置方式？改整个 Executor
- 想换一种结果映射方式？改整个 Executor
- 想加一个新数据库（如 Oracle）？改整个 Executor

## 2. 正确设计：分层解耦

```java
// ✅ 正确设计：每个对象只负责一件事

// Executor：只管执行策略和缓存
public class SimpleExecutor {
    public List<Object> query(MappedStatement ms, Object param) {
        // 1. 检查一级缓存
        CacheKey key = createCacheKey(ms, param);
        List<Object> cached = localCache.getObject(key);
        if (cached != null) return cached;

        // 2. 创建 StatementHandler（委托出去）
        StatementHandler handler = ms.getStatementHandler();

        // 3. 准备 Statement（委托给 StatementHandler）
        Statement stmt = handler.prepare(connection, timeout);

        // 4. 设置参数（委托给 ParameterHandler）
        handler.parameterize(stmt);

        // 5. 执行查询（委托给 StatementHandler）
        List<Object> result = handler.query(stmt, resultHandler);

        // 6. 放入缓存
        localCache.putObject(key, result);

        return result;
    }
}

// StatementHandler：只管 Statement 的创建和执行
public class PreparedStatementHandler {
    public Statement prepare(Connection conn, Integer timeout) {
        // 创建 PreparedStatement
        return conn.prepareStatement(boundSql.getSql());
    }

    public void parameterize(Statement stmt) {
        // 委托给 ParameterHandler
        parameterHandler.setParameters((PreparedStatement) stmt);
    }

    public List<Object> query(Statement stmt, ResultHandler handler) {
        // 执行查询
        ResultSet rs = ((PreparedStatement) stmt).executeQuery();
        // 委托给 ResultSetHandler
        return resultSetHandler.handleResultSets(rs);
    }
}

// ParameterHandler：只管参数设置
public class DefaultParameterHandler {
    public void setParameters(PreparedStatement stmt) {
        // 专注于：Java类型 → JDBC类型的转换
        // 使用 TypeHandler 进行转换
    }
}

// ResultSetHandler：只管结果映射
public class DefaultResultSetHandler {
    public List<Object> handleResultSets(ResultSet rs) {
        // 专注于：ResultSet → Java对象的映射
        // 处理关联查询、集合映射、鉴别器...
    }
}
```

## 3. 分层的好处

### 3.1 可替换性

```
想换参数设置方式？
├── 默认：DefaultParameterHandler
├── 加密：EncryptParameterHandler（自动加密敏感字段）
└── 只需替换 ParameterHandler，其他不动

想换结果映射方式？
├── 默认：DefaultResultSetHandler
├── 脱敏：MaskResultSetHandler（自动脱敏手机号）
└── 只需替换 ResultSetHandler，其他不动

想换执行策略？
├── 默认：SimpleExecutor（每次创建新Statement）
├── 复用：ReuseExecutor（复用Statement）
├── 批量：BatchExecutor（批量执行）
└── 只需替换 Executor，其他不动
```

### 3.2 可组合性

```
CachingExecutor（二级缓存）
    └── SimpleExecutor（一级缓存 + 执行策略）
            └── PreparedStatementHandler（Statement管理）
                    ├── DefaultParameterHandler（参数设置）
                    └── DefaultResultSetHandler（结果映射）
```

### 3.3 拦截器的灵活性

```
如果只有 Executor，你只能在一个地方拦截：

拦截点单一，想拦截参数设置？没办法
              想拦截结果处理？没办法
              想拦截SQL修改？没办法

有了四个对象，你可以在不同层次拦截：

Executor层：拦截执行（统计、审计）
StatementHandler层：拦截SQL（修改SQL、添加条件）
ParameterHandler层：拦截参数（加密、校验）
ResultSetHandler层：拦截结果（解密、脱敏）
```

## 4. 实际例子：为什么不能合并

假设你要实现：**查询时自动给 SQL 加租户条件**

### 方案1：在 Executor 层做

```java
// ❌ 问题：Executor 拿不到最终的 SQL 字符串
public class Executor {
    public List<Object> query(MappedStatement ms, Object param) {
        // ms.getSql() 返回的是原始SQL模板，可能包含 ${} #{}
        // 还没经过参数替换，无法直接修改
        String sql = ms.getSqlSource().getBoundSql(param).getSql();
        // 这个 sql 可能是: SELECT * FROM user WHERE id = #{id}
        // 你无法直接在末尾加 AND tenant_id = 1
        // 因为 #{id} 还没被替换，你不知道最终SQL长什么样
    }
}
```

### 方案2：在 StatementHandler 层做

```java
// ✅ 正确：StatementHandler 拿到的是最终 SQL
public class StatementHandler {
    public Statement prepare(Connection conn, Integer timeout) {
        // boundSql.getSql() 返回的是最终 SQL
        String sql = boundSql.getSql();
        // 这个 sql 是: SELECT * FROM user WHERE id = ?
        // 可以安全地修改为: SELECT * FROM user WHERE id = ? AND tenant_id = 1
        metaObject.setValue("delegate.boundSql.sql", newSql);
    }
}
```

## 5. 总结

| 问题 | 答案 |
|------|------|
| 为什么不分层？ | 代码会变成几千行的"上帝类"，无法维护 |
| 分层的好处？ | 单一职责、可替换、可组合、可拦截 |
| Executor 的职责？ | 执行策略、缓存、事务（不涉及SQL细节） |
| StatementHandler 的职责？ | 创建Statement、执行SQL（能拿到最终SQL） |
| ParameterHandler 的职责？ | 参数设置（Java类型 → JDBC类型） |
| ResultSetHandler 的职责？ | 结果映射（ResultSet → Java对象） |

**一句话**：就像公司分工一样，CEO（Executor）不关心具体怎么写代码，那是程序员（StatementHandler）的事；程序员不关心怎么发工资，那是财务（ParameterHandler）的事。
