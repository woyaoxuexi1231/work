自动装配 · 演化史

从"引入一个依赖要写 50 行 XML"，到"引入即用"

2004 第0代：Spring Framework —— 全手动

<!-- 引入 HikariCP，你需要手动配 4 个 Bean --> <bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource"> <property name="jdbcUrl" value="jdbc:mysql://..."/> </bean> <bean id="txManager" class="DataSourceTransactionManager"> <constructor-arg ref="dataSource"/> </bean> <bean id="jdbcTemplate" class="JdbcTemplate"> <constructor-arg ref="dataSource"/> </bean>
❌ 引入一个依赖 = 写一堆配置。DataSource、TransactionManager、JdbcTemplate——三个 Bean，缺一个都不行。新手引入 HikariCP 之后不知道怎么配，项目起不来。

2014 第1代：spring.factories + @Conditional —— "引入即用"

Spring Boot 的核心创新：你引入一个 starter，框架自己知道怎么配置。

// spring-boot-autoconfigure.jar 里的 spring.factories： org.springframework.boot.autoconfigure.EnableAutoConfiguration=\ org...DataSourceAutoConfiguration,\ org...DataSourceTransactionManagerAutoConfiguration,\ org...JdbcTemplateAutoConfiguration // 每个自动配置类都有条件注解： @Configuration @ConditionalOnClass(DataSource.class) // 类路径有 DataSource 才加载 @ConditionalOnMissingBean(DataSource.class) // 用户没手动定义才自动创建 public class DataSourceAutoConfiguration { ... }
✅ 核心机制：引入 HikariCP → AutoConfigurationImportSelector 从 spring.factories 拿到 180 个候选配置类 → 逐个检查 @Conditional → DataSourceAutoConfiguration 的 @ConditionalOnClass 满足 → DataSource 自动创建。用户可以随时用 @Bean 覆盖——@ConditionalOnMissingBean 会让自动配置自动退让。

核心洞察：自动装配的本质是"推断"——根据类路径、已有 Bean、配置项、应用类型，推断出用户大概率需要什么。它不是 AI，是一套规则引擎。条件注解就是规则。
