📋 Q12 · 事务传播行为 — REQUIRED / REQUIRES_NEW / NESTED。

面
说出 REQUIRED、REQUIRES_NEW、NESTED 的区别和适用场景？
我
区别在于对"已有事务"的处理：

REQUIRED（默认）：有事务就加入，没有就新建。内外层共享同一个事务——任何地方抛异常全部回滚。大部分业务用这个就够了。
REQUIRES_NEW：挂起当前事务，新建独立事务。内外互不影响——内层回滚不影响外层提交。典型场景：日志和审计。不管业务成功失败，日志必须独立写入。
NESTED：嵌套子事务，通过 JDBC Savepoint 实现。子事务回滚到 savepoint 不影响外层，但外层回滚会连子事务一起回滚。依赖数据库支持 Savepoint（H2/MySQL InnoDB/PG 支持）。
面
REQUIRES_NEW 挂起当前事务是怎么实现的？
我
通过 TransactionSynchronizationManager。unbindResource(dataSource) 解绑当前连接 → 保存到 SuspendedResourcesHolder → 获取新连接开启新事务 → 内层结束 → 恢复 Holder。全过程基于 ThreadLocal——这也是 @Transactional 不能跨线程的根本原因。
❓ 自问：@Async + @Transactional 为什么不能共享事务？
答：因为 @Async 用线程池执行，新线程有自己独立的 ThreadLocal，拿不到原线程的事务上下文。异步方法上的 @Transactional 是新事务，和外层无关。
