📋 Q14 · 事务实现原理 — @Transactional 如何织入 + afterCommit 回调。

面
@Transactional 是如何通过 AOP 织入事务的？
我
@EnableTransactionManagement → 注册一个 Advisor（Pointcut 匹配 @Transactional 方法，Advice 是 TransactionInterceptor）→ Bean 初始化时 BPP 检测到匹配 → 创建 AOP 代理。方法调用时 TransactionInterceptor.invoke() → getTransaction() 开启/加入事务 → 执行目标方法 → 成功 commit、异常 rollback。

PlatformTransactionManager 的三个核心方法：getTransaction()、commit()、rollback()。它是一个 SPI——JDBC、JTA、Hibernate 各有实现。
面
如何在一个事务提交后执行自定义逻辑？
我
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() { afterCommit() { ... } })。

为什么不在方法末尾直接发 MQ？因为 @Transactional 方法返回 ≠ 事务已提交——如果 commit 失败（数据库连接断开），消息已经发出去了。afterCommit 保证只在真正提交后才执行。
❓ 自问：TransactionSynchronizationManager 怎么管理事务上下文？
答：全部基于 ThreadLocal——当前连接、事务名、只读标志、隔离级别、以及 Synchronization 回调链。这就是事务不能跨线程的原因。
