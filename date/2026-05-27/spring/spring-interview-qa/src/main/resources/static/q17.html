参数绑定 · 演化史

从手写 getParameter + 手动 JSON 解析，到 @RequestBody 一行搞定

1998 第1代：Servlet 手写一切

读取查询参数：

// GET /user?id=1 String idStr = req.getParameter("id"); // 永远是 String if (idStr == null) { resp.sendError(400); return; } // 手动判空 Long id = Long.parseLong(idStr); // 手动转类型

读取请求体（JSON）：

// POST /user Body: {"id":1,"name":"Alice"} ← body 一直能用！只是手写很麻烦 BufferedReader reader = req.getReader(); StringBuilder body = new StringBuilder(); String line; while ((line = reader.readLine()) != null) { body.append(line); } // 手动读流 User user = objectMapper.readValue(body.toString(), User.class); // 手动调 Jackson
❌ 痛点：body 传参一直都能用——问题不在"能不能"，在"每次都要写 10 行"。4 个查询参数 + 1 个 body = 20 行样板代码。Controller 里 80% 的代码和业务无关。

2009 第2代：@RequestParam + @PathVariable 注解驱动

Spring MVC 3.0 解决的：

原来（Servlet）现在（@RequestParam）
String idStr = req.getParameter("id");
if (idStr == null) ...
Long id = Long.parseLong(idStr);		@RequestParam Long id
String pageStr = req.getParameter("page");
int page = pageStr != null ? Integer.parseInt(pageStr) : 1;		@RequestParam(defaultValue = "1") int page
✅ 进步：查询参数 + URL 路径参数的样板代码被消灭了。自动类型转换（TypeConverter）、自动判空（required=true）、默认值（defaultValue）——原来 12 行变 4 个注解。
⚠️ 还没解决的：body 还是要手写。@RequestParam 只能读查询参数和表单——读不了 JSON body。

现代 第3代：@RequestBody + HttpMessageConverter 链

// 原来：10 行手动读流 + 调 Jackson // 现在：1 行 public User createUser(@RequestBody User user) { ... }
✅ @RequestBody 的核心价值不是"能读 body"——Servlet 一直能读。它的价值是：策略模式（根据 Content-Type 自动选 Converter，JSON/XML/Protobuf 自动切换），零样板代码，可扩展（加一种新格式只需加一个 Converter——Controller 代码不用改）。

★ HttpMessageConverter 详解

它是什么？——一个策略接口

public interface HttpMessageConverter<T> { boolean canRead(Class<?> clazz, MediaType mediaType); // 请求：我能处理吗？ boolean canWrite(Class<?> clazz, MediaType mediaType); // 响应：我能处理吗？ T read(...); // HTTP Body 字节流 → Java 对象（反序列化） void write(...); // Java 对象 → HTTP Body 字节流（序列化） }

它不负责判断什么时候用它——那是 ArgumentResolver 的工作。它只负责：① 告诉上层"我能处理吗"，② 如果能，就把字节流变成对象（或反过来）。

Spring Boot 配了哪些？

Converter支持的 Content-Type能读写何时工作
StringHttpMessageConvertertext/plain, */*StringContent-Type: text/plain 时

💡 MultipartFile 不走 HttpMessageConverter。它走 MultipartResolver——一个完全不同的机制，在 DispatcherServlet 之前就处理了。

Converter 链是怎么选的？——按注册顺序遍历，先到先得

请求方向（@RequestBody 反序列化）：

POST /users Content-Type: application/json Body: {"id":1}

for (converter : converters)
├─ StringHttpMessageConverter.canRead(User, json) → ❌ false
├─ FormHttpMessageConverter.canRead(User, json) → ❌ false
├─ MappingJackson2HttpMessageConverter.canRead(User, json) → ✅ true!
└─ → read() → objectMapper.readValue(body, User.class)

响应方向（@ResponseBody 序列化）：

Controller 返回 User{id:1} Accept: application/json

for (converter : converters)
├─ StringHttpMessageConverter.canWrite(User, json) → ❌ false
├─ MappingJackson2HttpMessageConverter.canWrite(User, json) → ✅ true!
└─ → write() → objectMapper.writeValue(out, user) → {"id":1}

FormHttpMessageConverter 什么时候工作？

⚠️ 它处理的是老式 HTML 表单（application/x-www-form-urlencoded），不是 multipart/form-data（那个是 MultipartResolver 的活）。
// 传统 HTML 表单提交（没有文件）： // <form action="/login" method="POST"> // <input name="username"> <input name="password"> // </form> // → POST /login Content-Type: application/x-www-form-urlencoded // → Body: username=alice&password=123456 // → canRead(MultiValueMap, "application/x-www-form-urlencoded") → true // → 把 key=value&key=value 解析成 MultiValueMap

前后端分离项目基本用不到——前端发的是 application/json。

MappingJackson2HttpMessageConverter 什么时候工作？

前后端分离 99% 的场景。触发条件——请求 Content-Type = application/json；响应返回 POJO（非 String/byte[]）。

@PostMapping("/users") public User createUser(@RequestBody User user) { // ← MappingJackson2 读 JSON → User return userService.save(user); // ← MappingJackson2 写 User → JSON }

Converter 链 vs TypeConverter

TypeConverterHttpMessageConverter
数据来源URL 查询参数 / 路径变量 — 简单字符串HTTP Body 字节流 — 结构化数据
处理什么String → Integer/Long/BooleanJSON/XML/Protobuf → POJO
谁用@RequestParam / @PathVariable 的 ArgumentResolver@RequestBody / @ResponseBody 的 ArgumentResolver / ReturnValueHandler
举例"25" → 25 (Integer){"name":"Alice","age":25} → User{name="Alice", age=25}
💡 一句话区分：TypeConverter 处理"25 转成 int"（值级别）。HttpMessageConverter 处理"一段 JSON 转成一个对象"（结构级别）。各管各的，不是竞争关系。
