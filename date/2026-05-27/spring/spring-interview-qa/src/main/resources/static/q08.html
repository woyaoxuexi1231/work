📋 Q08 · 内部调用失效 — 同一个类中方法 A 调用 B，B 上的增强为什么失效？

面
同一个类中，方法 A 调用方法 B，B 上有 @Transactional 或 @Cacheable，为什么增强失效？
我
因为 Spring AOP 基于代理。外部调用 proxy.methodA() 时被代理拦截，但 methodA() 内部调用 this.methodB() 时——this 指向原始对象，不是代理。methodB 上的切面完全不会触发。

这是 Java 方法分派机制的物理限制，不是 Spring 的 bug。this.method() 在编译期就绑定到当前对象——不管外面包了多少层代理都改变不了这个事实。
面
怎么解决？
我
三种方案：

1. AopContext.currentProxy()：从 ThreadLocal 获取当前代理对象。需要 @EnableAspectJAutoProxy(exposeProxy = true)——有轻微性能开销。
2. 注入自身 @Autowired self：Spring 注入的是代理，调用 self.methodB() 走代理。但构造器注入时不能用（循环依赖）。
3. 重构到另一个类（最推荐）：把 methodB 抽到单独的 Bean 里。而且这通常说明原来的类职责太多了。
追问："AspectJ 能解决这个问题吗？"
答：能。AspectJ 编译期织入直接修改字节码，没有代理层，不存在 this 绕过的问题。但 AspectJ 配置复杂，实际项目用得少。
❓ 自问：怎么判断该用哪种方案？
答：methodB 如果是独立的业务逻辑（比如发送通知）→ 抽到另一个类。如果是类内部的辅助方法 → 用 AopContext 或注入自身。如果整个类大量内部调用 → 重新设计类结构。
