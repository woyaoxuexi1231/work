📋 Q05 · 作用域

singleton 注入 prototype 的经典陷阱——面试必问。

面
singleton、prototype、request、session 的作用域原理？
我
singleton：容器中唯一实例，启动时创建，所有请求共享。Spring 管理完整生命周期包括销毁。
prototype：每次 getBean() 新建实例。Spring 只负责创建，不负责销毁——@PreDestroy 不会被调用。
request/session：Web 环境专用。request 每个 HTTP 请求一个，session 每个会话一个。

prototype 不管理销毁是设计选择——如果 Spring 持有所有 prototype 实例引用，会导致内存泄漏。谁创建谁负责清理。
面
如果 singleton Bean 里注入 prototype Bean，如何保证每次拿到新实例？
我
直接 @Autowired 不行——singleton 只创建一次，依赖也只在创建时注入一次，prototype 被注入后就永远被 singleton 持有同一个引用。

三种解法：
1. ObjectProvider（推荐）：注入 ObjectProvider<PrototypeBean>，需要时 provider.getObject()。Spring 4.3 引入，支持 getIfAvailable() 和 stream()。
2. @Lookup：标注一个返回 PrototypeBean 的抽象方法，Spring 通过 CGLIB 生成子类重写它，每次调用执行 getBean()。但类不能是 final。
3. ApplicationContext.getBean()：最直接，但耦合了 Spring 容器。
追问："@Lookup 有什么限制？"
答：类不能是 final，方法不能是 private。而且如果你的类实现了接口，CGLIB 子类和 JDK 接口代理同时存在时可能有类型转换问题。
❓ 自问：什么时候该用 prototype？
答：很少。大多数时候 singleton + 无状态设计就够了。prototype 主要用于有状态的 Bean（比如一个 Bean 持有请求级别的数据）——但这类数据更应该放在 request scope 或方法参数里传递。
