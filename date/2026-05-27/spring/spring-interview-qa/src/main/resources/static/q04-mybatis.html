一个纯接口，MyBatis 是怎么把它变成对象的？

你写了一个 interface UserMapper { User findById(Long id); }
没有实现类、没有 XML（用了注解）、没有任何地方 new。然后 @Autowired UserMapper 就能用了。

这件事完全违反直觉。Java 里接口不能 new，这是编译器的铁律。
MyBatis 绕过它的方式，是 Spring 框架最精妙的机制之一。

1起点：@MapperScan 到底干了什么？

你只在配置类上写了一个 @MapperScan("com.example.mapper")，它是怎么找到所有接口的？

@MapperScan 的底层是一个 BeanFactoryPostProcessor——叫 MapperScannerConfigurer。

它在所有 Bean 创建之前运行（还记得 Q01 的 BFPP 吗？），扫描指定包下的所有 .class 文件，找到每个接口，然后——为每个接口注册一个 BeanDefinition。

📄 MapperScannerConfigurer.postProcessBeanDefinitionRegistry()（伪代码）
public class MapperScannerConfigurer implements BeanDefinitionRegistryPostProcessor { public void postProcessBeanDefinitionRegistry(BeanDefinitionRegistry registry) { ClassPathMapperScanner scanner = new ClassPathMapperScanner(registry); scanner.setAnnotationClass(null); // 不限定注解——只要是接口就扫 scanner.registerFilters(); // 过滤器：只接受接口（isInterface()） scanner.scan("com.example.mapper"); // 扫描指定包 } }
注意：MapperScannerConfigurer 实现了 BeanDefinitionRegistryPostProcessor（BFPP 的子接口），所以它能在所有 Bean 创建之前向容器注册新的 BeanDefinition。这和 ConfigurationClassPostProcessor 是同一级别的能力。

2关键：存进容器的不是"接口的实现类"，而是"一个工厂"

Spring 的 BeanDefinition 里必须有一个 beanClassName。对一个接口来说，这个 className 填什么？

答案是：不填接口名，填 MapperFactoryBean 的全限定名。

Spring 不需要知道接口怎么实现——它只知道："创建这个 Bean 的方式，是调用 MapperFactoryBean.getObject()"。BeanDefinition 里存的是工厂类，不是目标类。

📄 ClassPathMapperScanner 扫描到一个接口后做了什么
// 扫描到 com.example.mapper.UserMapper 接口后： BeanDefinition bd = new GenericBeanDefinition(); // ★ 关键：beanClassName 不是 UserMapper，而是 MapperFactoryBean！ bd.setBeanClassName("org.mybatis.spring.mapper.MapperFactoryBean"); // 把真正的接口类型存到构造器参数里 bd.getConstructorArgumentValues().addGenericArgumentValue( "com.example.mapper.UserMapper" ← 接口的 Class 对象 ); bd.setAutowireMode(AbstractBeanDefinition.AUTOWIRE_BY_TYPE); registry.registerBeanDefinition("userMapper", bd);
此时 Spring 容器里多了一条记录：名字是 "userMapper"，类型是 UserMapper，创建方式是 MapperFactoryBean.getObject()。Spring 不知道也不关心 getObject() 里面做了什么。

📄 AbstractBeanFactory.getObjectForBeanInstance() —— Spring 对 FactoryBean 的特殊处理
// 创建 Bean 时，Spring 拿到的是 MapperFactoryBean 实例 Object bean = createBean("userMapper", bd, args); // → MapperFactoryBean 实例 // ★ Spring 发现它实现了 FactoryBean 接口 if (bean instanceof FactoryBean) { FactoryBean fb = (FactoryBean) bean; // 调用 getObject() 拿真正的产物 bean = fb.getObject(); // → 这里返回了 JDK 代理对象！ // 产物还会经过完整的 BPP 链（包括 AOP） } return bean; // 返回的是代理对象，不是 MapperFactoryBean
这就是 getBean("userMapper") 返回代理对象而不是 MapperFactoryBean 的原因。Spring 在拿到 FactoryBean 实例后，自动多走了一步——调用 getObject() 并把产物当成最终的 Bean。

4核心：getObject() 里的 JDK 动态代理

MapperFactoryBean.getObject() 里面到底做了什么，能凭空变出一个对象？

📄 MapperFactoryBean.getObject() —— MyBatis-Spring 源码
public T getObject() throws Exception { return (T) Proxy.newProxyInstance( mapperInterface.getClassLoader(), // ① 用哪个 ClassLoader 加载代理类 new Class<?>[]{mapperInterface}, // ② 代理类要实现哪个接口 new MapperProxy<T>(sqlSession, mapperInterface) // ③ 所有方法调用转发给谁 ); }

三个参数决定了这个代理对象的一切行为：

  • ClassLoader：代理类是一个运行时生成的 $Proxy0.class，需要被加载到 JVM。用接口的 ClassLoader 是最自然的选择。
  • 接口数组：JDK 代理只能代理接口。这里只传了 mapperInterface 一个——生成的代理类实现了 UserMapper 接口。
  • MapperProxy：这是一个 InvocationHandler 实现。代理对象上的任何方法调用都会被转发到它的 invoke() 方法。

💡 这一刻，Java 的编译器铁律被绕过了

new UserMapper() 确实不行——编译器不允许。但 Proxy.newProxyInstance() 是 JDK 提供的内置 API，它在运行时动态生成字节码，不需要编译期检查接口是否有实现类。

JVM 加载这个生成的 $Proxy0.class 时，它里面已经有了 findById 和 insert 方法的具体实现——每个方法体就是一行：return h.invoke(this, method, args)。JVM 完全接受它，因为字节码层面一切合法。

📄 MapperProxy.invoke() —— MyBatis-Spring 源码（精简）
public class MapperProxy<T> implements InvocationHandler { private final SqlSession sqlSession; private final Class<T> mapperInterface; private final Map<Method, MapperMethod> methodCache; public Object invoke(Object proxy, Method method, Object[] args) { // ① Object 的方法 (toString/hashCode/equals) 直接执行 if (Object.class.equals(method.getDeclaringClass())) { return method.invoke(this, args); } // ② default 方法 (Java 8+) 用特殊方式执行 if (method.isDefault()) { return invokeDefaultMethod(proxy, method, args); } // ③ ★ Mapper 接口方法 → 从缓存获取或新建 MapperMethod MapperMethod mapperMethod = cachedMapperMethod(method); // ④ MapperMethod 执行 SQL return mapperMethod.execute(sqlSession, args); } }
注意第③步：MapperMethod 对象是缓存的。第一次调用 findById 时创建，后续调用直接取缓存——因为同一个方法的 SQL 语句是不变的。

6线程安全：为什么一个代理对象可以被所有线程共享？

代理是单例的——isSingleton() 返回 true。多个请求同时调用 userMapper.findById()，不会互相干扰吗？

代理对象本身没有任何状态。它持有的 SqlSession 引用，不是真正的数据库连接——而是 SqlSessionTemplate，一个线程安全的包装器。

每次调用 getSqlSession() 时，SqlSessionTemplate 内部通过 ThreadLocal 获取当前线程的真正 SqlSession。不同线程拿到的是不同的连接——所以一个代理对象可以被所有线程安全共享。

📄 SqlSessionTemplate —— 线程安全的 SqlSession 代理
public class SqlSessionTemplate implements SqlSession { private final SqlSessionFactory sqlSessionFactory; private final ExecutorType executorType; // ★ 这个 SqlSession 本身是 JDK 代理！ private final SqlSession sqlSessionProxy; public SqlSessionTemplate(SqlSessionFactory factory) { this.sqlSessionFactory = factory; this.sqlSessionProxy = (SqlSession) Proxy.newProxyInstance( SqlSession.class.getClassLoader(), new Class[]{SqlSession.class}, new SqlSessionInterceptor() // 拦截器负责从 ThreadLocal 取连接 ); } // 任何方法调用被 SqlSessionInterceptor 拦截： // ① 从 TransactionSynchronizationManager (ThreadLocal) 获取当前 SqlSession // ② 如果当前线程没有 → 从 SqlSessionFactory 新建一个 → 绑定到 ThreadLocal // ③ 调用真正的 SqlSession 方法 // ④ 如果不是事务中 → 调用后立即 commit/close }

💡 套娃！代理里面还有代理

MyBatis 的线程安全设计是一层套一层的代理：

userMapper（JDK 代理 → MapperProxy）
  → 内部持有 sqlSessionProxy（又是 JDK 代理 → SqlSessionInterceptor）
    → 拦截器从 ThreadLocal 取真正的 SqlSession
      → 每个线程拿到自己的数据库连接

两层代理，一层负责"接口方法→SQL"，一层负责"线程→数据库连接"。职责分离得清清楚楚。

7完整时间线：从启动到调用

① @MapperScan → MapperScannerRegistrar → 注册 MapperScannerConfigurer（一个 BFPP）
② Spring refresh() → invokeBeanFactoryPostProcessors() → MapperScannerConfigurer 执行
③ MapperScannerConfigurer 扫描包 → 找到 UserMapper.class、OrderMapper.class…
④ 为每个接口注册 BeanDefinition：beanClassName=MapperFactoryBean，构造器参数=接口Class
⑤ finishBeanFactoryInitialization() → 创建 userMapper Bean
⑥ Spring 实例化 MapperFactoryBean → 发现它实现 FactoryBean → 调用 getObject()
⑦ getObject() 内部：Proxy.newProxyInstance(ClassLoader, 接口, MapperProxy)
⑧ JDK 在内存生成 $Proxy0.class → 实现 UserMapper 接口 → 每个方法体调用 h.invoke()
⑨ Spring 把代理对象放入 singletonObjects → 类型 = UserMapper
⑩ 任何 @Autowired UserMapper 的地方 → 注入这个代理对象
⑪ 业务调用 userMapper.findById(1L) → $Proxy0 → MapperProxy.invoke() → 查缓存 → SqlSession → SQL
