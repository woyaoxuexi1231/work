📋 Q06 · Bean 覆盖与冲突

Spring Boot 从 1.x 到 2.x 改了一个默认值，背后是无数生产事故。

面
Spring Boot 中同名 Bean 默认允许覆盖吗？
我
默认不允许。Spring Boot 2.1+ 会直接报 ConflictingBeanDefinitionException。

但 Spring Framework 原生默认是允许的——后注册的静默覆盖前面的。Spring Boot 加了一层安全锁。为什么改？因为 1.x 时代太多生产事故——你引入一个 starter，它内部定义了一个同名 Bean，静默把你的替换了，没有警告、没有报错，上线出问题根本不知道怎么排查。改成 false 后 fail-fast——启动就报错，你必须显式解决冲突。

可以通过 spring.main.allow-bean-definition-overriding=true 开启——但我不推荐，这会掩盖问题。
面
你遇到过多模块同名 Bean 冲突怎么排错？
我
三步走：
1. 看错误日志——哪个 Bean 名称冲突了，是哪两个类
2. 决定谁优先——@Primary 让其中一个成为主选，或者 @Qualifier 精确指定
3. 如果冲突来自自动配置——用 @ConditionalOnMissingBean 让自动配置"温和注册"（用户定义了我就不注册）

实在排查不出来，用 --debug 启动。日志里会打印完整的自动配置报告——Positive matches（生效了）和 Negative matches（被条件注解跳过了）。找到冲突的那个 Bean，看它是从哪里注册的。
追问："allow-bean-definition-overriding=true 有什么风险？"
答：它是全局开关——影响所有 Bean，不是某一个。你没法只允许 A 覆盖 B 而不允许 C 覆盖 D。一旦打开，任何同名 Bean 都会静默覆盖，你完全不知道谁覆盖了谁。
❓ 自问：多模块项目怎么从设计上避免 Bean 冲突？
答：给 Bean 起唯一名字——加上模块前缀。自动配置类用 @ConditionalOnMissingBean。公共模块的 Bean 不要用通用名字（如 dataSource），让各模块自己定义。
