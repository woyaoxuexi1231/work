自定义 Starter · 演化史

从"copy-paste 配置类"到"一个依赖搞定"

2004 第0代：手动 copy-paste 配置

// 公司内部有个公共的短信 SDK。接入方式： // ① 复制 SmsConfig.java 到你的项目 // ② 复制 SmsProperties.java 到你的项目 // ③ 在 application.yml 里加 sms.apiKey=xxx // ④ 自己写 @Bean 注册 SmsClient
❌ 5 个项目用了短信 SDK → 5 份相同的 SmsConfig 代码。SDK 升级了？5 个项目分别改。配置项变了？5 个项目分别改。

2015 第1代：双模块 Starter —— "引入即用"

把配置代码也打成 jar，做到"引入一个依赖 = 自动配置完成"。

// 双模块结构： my-spring-boot-starter/ ← 空壳 POM，只做依赖聚合 pom.xml （引入 autoconfigure + 必要的第三方依赖） my-spring-boot-autoconfigure/ ← 核心：自动配置 SmsProperties.java @ConfigurationProperties("sms") SmsAutoConfiguration.java @Configuration + @ConditionalOnProperty SmsClient.java META-INF/spring.factories org...EnableAutoConfiguration=...SmsAutoConfiguration
✅ 为什么双模块？starter 只管依赖（用户引入就是全套），autoconfigure 只管配置。用户也可以只引入 autoconfigure 然后手动做依赖管理。spring-configuration-metadata.json 不是手写的——spring-boot-configuration-processor 编译时扫描 @ConfigurationProperties 自动生成，IDE 读它实现自动补全。
