📋 Q11 · 自研 AOP — 自己实现一个 AOP 引擎，核心接口怎么设计？

面
如果让你自己实现一个支持前置、后置、环绕通知的 AOP 引擎，核心接口怎么设计？
我
五个核心组件：

1. Advice 接口族：BeforeAdvice / AfterAdvice / AroundAdvice。为什么分开而不是一个？因为前置不需要返回值、后置需要接收返回值、环绕需要控制调用链——方法签名不同，合并会变成一堆 if-else。
2. Pointcut 接口：一个 matches(Method, Class) 方法。为什么是接口？因为切点可能是注解、表达式、正则——接口让每种方式独立实现。
3. Aspect 类：绑定 Advice + Pointcut + order。
4. AopEngine：注册切面 + 在 Bean 初始化时织入代理（用 JDK 动态代理）。
5. MethodInvoker 链：递归结构——每个 AroundAdvice 包裹下一层，形成洋葱模型。

200 行代码能完成一个最小可用版本。Spring 的实现更复杂——它多了异常通知、多切面排序、三级缓存与代理的协调。但核心思路完全一样。
追问："如何在 Bean 初始化时织入代理？"
答：对应 Spring 的 postProcessAfterInitialization()——Bean 初始化完成后、放入缓存前。此时 Bean 是成品，生成的代理能完整替代它。
❓ 自问：面试官想从这道题看到什么？
答：不是你的代码量，是你的接口设计思路——为什么这么分、每个接口的职责。能说清楚"Advice 按类型分接口是接口隔离原则，Pointcut 是接口是为了开闭原则"，比写 500 行代码更有说服力。
