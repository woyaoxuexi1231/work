### 第1题：Sentinel 故障转移的完整过程与生产问题

**【问题】**  
你负责的订单服务使用 Redis Sentinel 做主从高可用。某天凌晨主节点突然宕机，业务出现短暂不可用后自动恢复。请你详细描述一下从主节点宕机到服务恢复这期间，Sentinel 集群内部到底发生了什么？另外，如果让你设计一个压测场景去复现“脑裂”，你会怎么设计？脑裂的根本原因是什么？

**【考察点】**
- 对 Sentinel 内部机制的深度理解（主观下线 / 客观下线 / Leader选举 / 故障转移），而非死记硬背流程
- 理解分布式系统经典问题“脑裂”的成因，并能设计复现场景
- 能够将原理映射到实际生产故障

**【追问方向】**
- 如果当时 Sentinel 集群本身只有 2 个节点，会发生什么？
- `sentinel down-after-milliseconds` 和 `sentinel failover-timeout` 这两个参数到底如何影响切换时间？你会根据什么来调优？
- 有没有遇到过切过去之后新的主节点立刻又挂掉的情况？

**【参考要点】**
1. 流程：单个 Sentinel 检测到 master 不可达（SDOWN）→ 多个 Sentinel 在 `quorum` 数内确认 → 达成客观下线 ODOWN → 发起 Leader Sentinel 选举（Raft 协议子集，基于纪元 epoch）→ 由 Leader 选择新主、通知其他 Sentinel 和客户端。
2. 脑裂复现场景：网络分区导致旧主与 Sentinel 隔离，Sentinel 选新主，旧主还接收旧客户端写入。根本原因是**网络隔离后无有效的“多数派”保活机制**，仅靠 quorum 选举不够，还必须配合 `min-slaves-to-write`、`min-slaves-max-lag` 让旧主拒绝写入。
3. 参数调优：`down-after-milliseconds` 决定了检测故障的速度，过短会误判，过长则 RTO 变大，需根据网络抖动情况设定；`failover-timeout` 控制整个切换过程的超时。

### 第2题：Java 客户端与 Sentinel 的集成与坑点

**【问题】**  
你们目前线上用的 Redis 客户端是 Jedis 还是 Lettuce？请描述一下客户端如何通过 Sentinel 模式自动发现主节点地址，并能在主从切换后无缝连接到新主。你遇到过连接池耗尽、NoSuchElementException、或者“连接拿到了但写不进去”的情况吗？如果主从切换时客户端连续重试失败，你是如何兜底的？

**【考察点】**

- 客户端对 Sentinel 的拓扑感知机制（Pub/Sub 通知 vs 定期刷新）
- 连接池生命周期管理，故障时对象的无效化
- 生产级健壮性设计：重试策略、断路器、降级

**【追问方向】**
- 如果你们的 JedisSentinelPool 怎么知道当前主是谁？它内部是轮询 Sentinel 还是依赖订阅？
- 切换过程中客户端连续报 `READONLY` 错误，是什么原因？如何快速缓解？
- 如果要实现从库读的读写分离，结合 Sentinel 会有哪些额外复杂度？你们是怎么做的？

**【参考要点】**

1. 客户端一般先连接 Sentinel 节点列表，通过 `SENTINEL get-master-addr-by-name` 获取当前主地址，同时订阅 `+switch-master` 频道，当切换发生时 Sentinel 会推送新主地址，触发连接池重建。
2. 常见坑点：
    - 主库宕机瞬间，连接池里还缓存大量指向旧主的连接，请求会失败；
    - 新主升主期间，旧主可能短暂恢复成 slave 导致连接到旧主的客户端收到 `READONLY` 错误；
    - 连接池监控与刷新不及时，导致 NoSuchElementException。
3. 生产实践：
    - 尽量使用 Lettuce，天然支持自适应拓扑刷新（Cluster/Sentinel topology refresh）；
    - 结合重试机制与断路器，当读写失败超过阈值时，主动触发一次拓扑刷新，并对故障连接快速失效；
    - 做读写分离时，必须容忍从库延迟，并且从库变化也要感知 `+slave` / `-slave` 事件。

### 第3题：Redis Cluster 分片与客户端路由

**【问题】**  
你们的业务从 Sentinel 迁移到 Cluster 后，第一次出现了 `MOVED` 异常。请解释一下 MOVED 和 ASK 重定向的底层区别。为什么 Redis 集群选择 16384 个哈希槽，这个数字是怎么来的？Java 客户端（例如 Lettuce）在实现 Smart Routing 时，内部如何维护槽位映射？你有没有遇到过节点迁移时大量 `TRYAGAIN` 或 `MOVED` 导致延迟飙升的问题？怎么优化？

**【考察点】**
- 哈希槽、重定向的原理及差异
- 客户端分片路由的实现与性能代价
- 对 Cluster 伸缩过程中请求质量（P99延迟）的敏感度

**【追问方向】**
- 如果你用 Spring Data Redis 的 RedisTemplate，配置 Cluster 模式时，需要设置哪些关键参数来适应生产环境？
- 重定向次数有上限吗？如果没有限制会有什么后果？
- 假设你有一个批量操作（pipeline）中多个 key 不在同一节点，客户端如何处理？

**【参考要点】**
1. MOVED 是永久重定向，表示槽已迁移到目标节点，客户端应更新本地 slot 映射；ASK 是临时重定向，只对当前请求有效，槽正在迁移中，客户端不必更新槽表，但要带上 `ASKING` 命令。
2. 16384 个槽：一是集群心跳消息用位图结构携带本节点槽信息，2KB 大小刚好满足 16384 位，在消息大小和网络开销间平衡；二是节点数扩展到 1000 时每个节点负责的槽依然足够分散。
3. Smart 客户端如 Lettuce 会通过 `CLUSTER SLOTS` 或 `CLUSTER NODES` 初始化并定期刷新槽表，遇到 MOVED 会触发单点拓扑刷新，减少后续请求的错误。
4. 迁移期间优化：
    - 使用 `CLUSTER SETSLOT node TIMEOUT` 控制迁移暂停，降低抖动；
    - 客户端可配置最大重定向次数，防止死循环；
    - 批量操作最好按槽分组，通过 `slotHash` 计算后分节点发送。

### 第4题：Cluster 下的数据一致性与分布式事务

**【问题】**  
有一笔电商订单需要在一个 Lua 脚本中原子操作三个 Redis Key：库存、订单状态、用户已购列表。在 Cluster 模式下，这些 Key 大概率不在同一个 slot。你会怎么解决这个分布式事务问题？用 Hash Tag 强制分到同一 slot 会引入什么风险？如果业务强一致，你会选择 Redis Cluster 吗？请给出你的架构权衡。

**【考察点】**
- 对 Cluster 事务边界的认知（只能单节点原子）
- Hash Tag 的优势与副作用（数据倾斜、热点）
- 在 “AP 存储 + 业务补偿” 与 “CP 方案” 之间的决策能力

**【追问方向】**
- 如果非要用 Cluster，你会如何用 SAGA 或 TCC 模式来补偿？
- Hash Tag 下某热点订单占满一个 slot 所在节点的 CPU，你怎么处理？
- 有没有考虑过用 Redlock？在 Cluster 里能用吗？有什么问题？

**【参考要点】**
1. 对于多 Key 原子操作，Redis Cluster 本身不支持跨节点事务。要么用 Hash Tag（如 `{order:123}`）将它们映射到同一 slot，但会造成该订单所有数据集中在一个节点，若热点很容易压垮单节点；且扩展性变差。
2. 生产常见折衷：
    - 用分布式锁 + 应用层 check-and-set 逻辑完成多步操作，最终一致性通过消息队列补偿；
    - 订单状态流转可采用状态机，关键步骤用 Redis 单节点事务保护，其他非关键数据可异步同步。
3. 强一致场景（库存扣减不允许超卖）建议不单靠 Redis Cluster，而是使用 Redis 做缓存+高性能计数器，持久化由 DB 保证，或使用支持事务的中间件；Redis 应被定位为高性能、最终一致存储。

### 第5题：集群故障排查与性能优化

**【问题】**  
线上 Redis Cluster 某个节点 CPU 持续 90% 以上，同时该节点慢查询日志里出现大量 `KEYS *` 和 `HGETALL` 一个大 key。你一步步排查的思路是什么？如果这个节点恰好负责的 slot 中有热 key 出现流量倾斜，除了拆分大 key，还有哪些应急和长效方案？怎样在集群层面监控到这种倾斜并自动告警？

**【考察点】**
- 故障排查方法论（从监控 -> 慢查询 -> 大key/热key -> 业务代码定位）
- 热 key 解决方案的实战经验（读写分离、本地缓存、key 拆分）
- 运维监控体系搭建思路

**【追问方向】**
- 你如何快速临时限流该热 key 而不影响整个集群？
- 如果用 `redis-cli --hotkeys` 发现有 500 万 hits 的热 key，你敢直接删除吗？怎么安全降级？
- 集群模式下，`MEMORY USAGE` 和 `MEMORY STATS` 怎么用来诊断内存碎片？

**【参考要点】**
1. 排查链路：
    - 先看监控（CPU、QPS、命中率、网络出入流量），确认是否为单节点热点；
    - `SLOWLOG GET` 定位慢命令，分析是否由大 key、不恰当命令引起；
    - `redis-cli --bigkeys` / `--hotkeys` 找出 top key 以及内存占用；
    - 结合业务代码 review，看是否有 `KEYS` 或全量返回的命令。
2. 热 key 应急：
    - 读：临时增加从节点分担读压力（Cluster 下增加 slave 并手动将读路由过去）；
    - 写：无法简单分摊，需业务降级或排队；
    - 长效：key 拆分（如按用户 id 尾号分 100 份）、本地 Caffeine 缓存多级缓存、前端限流。
3. 监控体系：
    - 通过 `INFO` 命令采集每个节点的 key 数量、内存、CPU；
    - 自定义脚本周期性 `scan` 统计 keyspace 分布；
    - 接入 Prometheus + Grafana，配置 slot 维度的 QPS 倾斜告警。

### 第6题：架构选型与演进路线

**【问题】**  
假设你从零设计一个用户量千万级的社交 App 的缓存架构，初期用单机 Redis，中期引入 Sentinel，后期上线 Cluster。请分别说明每个阶段的分界点（数据量、QPS、RTO 要求）以及迁移过程中你的设计策略。如果现在让你重新选型，你会在什么条件下直接选择 Cluster，什么条件下坚持 Sentinel + 代理？你会自己运维还是用云服务，为什么？

**【考察点】**
- 架构演进的经验与判断标准
- 对容量规划、运维成本的全面考量
- 务实的技术选型思维，能权衡自建与云服务的利弊

**【追问方向】**
- Sentinel 模式下加一层 Proxy（如 Twemproxy/Codis/Tendis）与直接上 Cluster 相比，有什么优缺点？
- 迁移过程中如何保证数据的无缝切换？你们做的双写、回源、还是分步迁移？
- 如果业务突然爆发，Cluster 扩缩容的速度赶不上业务增长，你会如何临时应对？

**【参考要点】**
1. 阶段划分：
    - 单机：数据 < 10G，QPS < 5W，RTO 分钟级可接受，采用主从 + Keepalived；
    - Sentinel：数据 10-50G，QPS 5-15W，要求自动故障转移，但数据分片需求不迫切；
    - Cluster：数据 > 100G 或 QPS > 20W，需要水平扩展，业务容忍 Cluster 多节点间的轻度不一致。
2. Sentinel + Proxy 方案提供更好的屏蔽复杂性，适合不想改造客户端但需要分片平滑扩展的场景（如 Codis），但 Proxy 引入额外延迟，且需要维护 Proxy 集群。
3. 云服务与自建：如果团队有专职 DBA 且对成本敏感，可以选择自建；如果业务要求弹性伸缩、容灾、自动化运维，且对延迟不极端敏感，云服务更合适。但要注意云服务版 Cluster 支持的命令集可能被裁剪。

### 第7题：代码设计——统一客户端工厂

**【问题】**  
请设计一个 `RedisClientFactory`，能同时支持 Sentinel 模式和 Cluster 模式，并能通过配置动态切换。要求包括：连接池管理（连接数、空闲回收）、故障转移下的重试策略（指数退避）、拓扑刷新机制。请手绘或者描述出核心类与接口的结构，并写出连接获取和重试的伪代码逻辑。你如何保证在切换模式时，连接池能做到无中断？

**【考察点】**
- 抽象能力与设计模式（工厂、策略、装饰器等）
- 对 Lettuce 或 Jedis 底层 API 的熟悉程度
- 细节把控：重试与幂等、资源释放、线程安全

**【追问方向】**
- 如果 Sentinel 和 Cluster 两种客户端分别返回不同类型的连接（StatefulRedisConnection / StatefulRedisClusterConnection），你在外面如何统一封装？
- 怎么实现命令执行的超时控制和重试时的 idempotent 判断？
- 拓扑刷新如果失败，会影响到业务线程吗？你用什么机制隔离？

**【参考要点】**
1. 结构：
    - `RedisMode` 枚举（SENTINEL, CLUSTER）；
    - `RedisClientFactory` 根据 mode 创建对应的 `RedisClient` 实例；
    - `RedisClient` 接口提供 `execute(RedisCommand)` 方法；
    - `SentinelRedisClient` 和 `ClusterRedisClient` 分别封装 Lettuce 的 `RedisClient` 和 `RedisClusterClient`。
2. 连接池与拓扑刷新：
    - 使用 Lettuce 内置的连接池 `GenericObjectPool`，根据拓扑事件触发 `flushdb` 或 `reset`；
    - 拓扑刷新用 `TopologyRefreshScheduler` 定期执行，刷新失败仅记录日志，不影响业务；重试策略通过 `RetryListener` 实现指数退避。
3. 重试伪代码：
```java
<T> T executeWithRetry(RedisCommand<T> cmd, int maxRetries) {
    int attempts = 0;
    while (attempts < maxRetries) {
        try {
            return cmd.execute(getConnection());
        } catch (RedisException e) {
            if (isRetriable(e)) {
                attempts++;
                topologyRefreshIfNeeded(e);
                sleep(exponentialBackoff(attempts));
            } else {
                throw e;
            }
        }
    }
    throw new RedisRetryExhaustedException();
}
```
4. 模式切换时，通过 `AtomicReference` 持有当前模式，动态重建连接池，并在一段缓冲时间内逐步淘汰旧连接，做到平滑切换。

### 第8题：大规模集群运维与平滑扩缩

**【问题】**  
你现在负责一个有 100 个节点的 Redis Cluster 集群，业务计划大促前扩容 20 个节点。你会如何设计一个平滑且低影响的迁移方案？`redis-cli --cluster reshard` 背后的原理是什么？迁移过程中出现 MIGRATE 超时或者迁移中断，你怎么回滚？如何监控迁移进度与槽位平衡？如果要实现自动化扩容，你会如何设计其控制面？

**【考察点】**
- 对 Cluster 数据迁移机制的深度掌握（MIGRATE 命令、pipelined 迁移、ASK 转向）
- 大规模运维的流程设计、风险控制、回滚预案
- 自动化运维系统的架构设计能力

**【追问方向】**
- 迁移过程中，某个 key 非常大（如 500MB），会发生什么？你怎么提前发现并规避？
- 如果迁移过程中网络抖动导致部分 key 迁移成功部分失败，迁移状态机如何恢复？
- 怎样保证在 slot 迁移过程中，客户端遇到的 MOVED/ASK 错误不影响业务逻辑的正确性？

**【参考要点】**
1. `reshard` 原理：
    - 计算要迁移的 slot 列表，为每个 slot 生成迁移计划；
    - 源节点执行 `CLUSTER SETSLOT slot MIGRATING`，目标节点执行 `CLUSTER SETSLOT slot IMPORTING`；
    - 源节点用 `MIGRATE` 命令以 pipeline 方式逐个迁移 key，key 迁移完后通过 `CLUSTER SETSLOT slot NODE target` 通知全网更新；
    - 客户端遇到 ASK 重定向会到目标节点请求。
2. 生产平滑方案：
    - 选择业务低峰期，分批次迁移，每次只迁移少量 slot，并监控迁移延迟；
    - 设置 `MIGRATE` 的超时和 `replication` 参数，支持迁移期间 key 仍能服务；
    - 提前扫描并处理大 key（通过 `redis-cli --bigkeys`），采用拆分或先删除后重建等策略；
    - 准备回滚：若迁移中断，需手动将未完成的 slot 的 MIGRATING/IMPORTING 状态清除，并重置 slot 归属。
3. 自动化设计：
    - 控制面监听集群 metrics（CPU、内存、槽分布），触发 scale-out 事件；
    - 调用云 API 或自研 agent 启动新实例加入集群；
    - 自动运行 slot 平衡算法（如贪心迁移让节点流量相近）；
    - 迁移任务持久化到 DB，通过状态机推进，支持断点续传；
    - 迁移完成前持续监控 ASK 转向比例，达标后自动切换主从关系并回收资源。
