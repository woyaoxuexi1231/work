### 第1题：Sentinel 故障转移的完整过程与生产问题

好，这道题让我想起一个真实的凌晨三点。

当时我在一家社交公司，订单服务用的就是 Sentinel 架构。那个凌晨，主节点所在物理机因为内存条故障直接宕机了。整个过程，如果用慢镜头回放，哨兵集群里发生了这么一系列事情：

**第一阶段：主观下线（SDOWN）的发现。** 每个 Sentinel 节点都有一个独立的“心跳检测线程”，每秒向主库发 `PING`。我们的 `down-after-milliseconds` 设的是 3000ms。主库宕机后，第一个发现异常的 Sentinel，在 3 秒后标记主库为 SDOWN。但到这里，它还只是“一个人觉得皇帝驾崩了”，它不会轻举妄动，而是立刻通过 `SENTINEL is-master-down-by-addr` 命令，向其他哨兵询问：“你们觉得他还在吗？”

**第二阶段：客观下线（ODOWN）的裁定。** 我们设的 `quorum` 是 3，总共有 5 个 Sentinel。当这个 Sentinel 收集到包括自己在内一共 3 票“他不行了”时，就断定主库进入客观下线状态。这里有一个异常优雅的权衡：为什么是 3 而不是 5？因为要平衡“防误判”和“可用性”——如果设成 5，任何一个 Sentinel 网络抖动，整个集群就永远无法客观下线，服务就卡死了。但设成 2 又太敏感，哨兵自己打个喷嚏就可能误切。**本质是用多数派的票数来对抗网络分区带来的不确定性。**

**第三阶段：领袖选举（Leader Election）。** 客观下线后，不是所有哨兵都能动手，否则会出乱子。它们会进入一个基于 Raft 思想精简的选举：每个哨兵持有一个 `currentEpoch`（纪元号），想当 Leader 的哨兵会把自己的 `currentEpoch` 加 1，向其他哨兵拉票。只有拿到 `majority`（这里是 5/2+1=3 票）的哨兵，才能成为本轮故障转移的唯一执行者。如果这一轮没选出来，就增加纪元号再来一轮。这保证了**任何时刻，最多只有一个 Leader 在搞故障转移。**

**第四阶段：选新主与切换。** Leader 哨兵会从所有在线的从库中，按“优先级 > 复制偏移量最接近原主 > 运行 ID 字典序最小”的顺序，选出一个最优的。执行 `SLAVEOF NO ONE`，让它登基为新主。然后把其他从库指向新主。最后，哨兵更新自己维护的“主库地址映射”，并通过 Pub/Sub 频道 `+switch-master` 广播这条消息。我们的 Java 客户端订阅了这个频道，收到后立刻重建连接池，业务恢复。

整个过程，从宕机到服务恢复，我们实测通常在 10-15 秒。

**关于脑裂的复现与根治**

如果让我设计压测复现脑裂，我会这样做：用 iptables 在交换机上模拟网络分区，让原主库和所有 Sentinel 及从库之间网络断开，但原主和它的一部分老客户端之间还是通的——这就是经典的“孤岛主库”场景。Sentinel 这边，因为收不到原主心跳，判定客观下线，选新主。但原主那边，还在接收旧客户端的写入。等网络恢复，原主突然被降级为从库，全量同步新主，它那段时间接收的数据，全丢了。

**脑裂的根本原因，正如我刚才讲解哨兵时提到的，是一个“无共识的皇权”。** 一个节点之所以能“当皇帝”，是因为大家认它。网络隔离让两部分世界各自认了不同的“皇帝”。要根治，只靠 Sentinel 的选举不够，必须配合我给原主的“紧箍咒”——`min-slaves-to-write` 和 `min-slaves-max-lag`。我们的配置是：原主在发现自己连一个从库都没有，或者从库滞后超过 10 秒时，拒绝所有写请求。这样就算脑裂，孤岛主也写不进去，数据安全得以保全。

**【追问回答】**

- **2 个 Sentinel**：那将是一场灾难。任何一个哨兵挂掉或网络隔离，集群就只剩一个活口，永远拿不到 `majority=2` 票，永远无法客观下线，永远无法自动故障转移。所以我之前强调，哨兵部署必须是**奇数个且 ≥ 3**，这是在容忍少数派故障和防止脑裂间的黄金平衡。

- **`down-after-milliseconds` 和 `failover-timeout`**：前者是“发现故障的速度”，像我们汽车的安全气囊触发灵敏度。设得太短（比如 300ms），网络一个微抖动，就误判 SDOWN，频繁切换。设太长（比如 30秒），那故障转移太慢，业务等不起。我一般根据内网网络质量，P99 延迟在 1ms 内的，设 3-5 秒；跨机房或有波动的，设 10-15 秒。后者 `failover-timeout` 是整个过程的超时，包括选新主、数据同步。它必须大于 `down-after-milliseconds`，并为你预估的最长主从同步时间留够空间。我们一般设为 180 秒。

- **新主立刻又挂**：遇到过。是新主扛不住原主的流量洪峰，起来瞬间就被打趴。这暴露了容量规划问题——单节点规格不够。我们的解法是，对核心集群，保证所有从库与主库规格完全一致，不做“降配从库”，切过去才能接住流量。

【面试官视角点评】
*这个回答最打动人的地方，是候选人把枯燥的流程变成了一个有逻辑的“追凶”故事，清晰点出了 SDOWN 到 ODOWN 之间的“多数票”权衡，这是懂分布式的人才会讲出的内行话。对脑裂的复现设计和根治方案，展示了工程化的闭环思维——不仅能发现问题，更能设计实验，并给出有效解决手段。*

### 第2题：Java 客户端与 Sentinel 的集成与坑点

我们线上从 Jedis 一路磕磕绊绊，最终全面迁移到了 Lettuce。先说说 Jedis 怎么“知道”新主是谁，以及我们踩过的血坑。

Jedis 的 `JedisSentinelPool` 工作原理是两路并行：一路是在初始化时，通过 `SENTINEL get-master-addr-by-name` 直接问哨兵“主库在哪”，拿到 IP 和端口去建连接池；另一路更重要，它订阅了哨兵的 `+switch-master` 频道。切换发生那一刻，哨兵会在这个频道推送“`+switch-master mymaster 192.168.1.2 6379`”，Jedis 内部的监听器 `MasterListener` 会收到这条消息，然后调用 `initPool()` 销毁整个旧连接池，重建一个新的指向新主的连接池。

**我们踩过的最惨的坑，就是连接池耗尽和 `READONLY` 错误**。有一次主库宕机，瞬间所有 Jedis 连接都坏了。`JedisSentinelPool` 还在走内部逻辑判断、重建，那几百毫秒内，大量的业务线程涌到 `pool.getResource()`，拿不到连接，直接抛 `NoSuchElementException`，监控里一片红。还有一次更诡异，主库其实没宕机，而是 CPU 满载假死，哨兵误判切了。但我们的一个老服务用的 Jedis 版本没处理好，它拿到了新主连接，但新主那时还是从库状态，正在变为 `MASTER` 的过程中，那极短的窗口期，连接没报错，但一写就报 `READONLY`。

**我们的兜底策略进化了三个阶段：**

1. **原始的 Jedis 时期**：我们自定义了一个 `SafeJedisPool`，在 `getResource` 外层套了重试循环，捕获 `JedisConnectionException` 和 `NoSuchElementException` 后，强制休眠 1 秒，再次尝试获取。这是“用时间换可靠性”的粗暴做法，P99 会飙升。

2. **引入 Lettuce 的时期**：Lettuce 的核心理念是“连接是异步的、拓扑是自适应的”。它的 `SentinelTopologyRefresh` 机制，不仅在收到 `+switch-master` 时刷新，还会定时（默认 30 秒）全量拉取一次哨兵的最新拓扑。这意味着即使某次推送丢了，30 秒后也能自动纠正，这是典型的**用定期的冗余同步，来对抗消息丢失的不确定性**。并且它的连接池（`GenericObjectPool`）跟事件机制绑定，切换发生时，连接会被标记为失效，后续获取自动跳到新主。

3. **最终的健壮性方案**：在 Lettuce 之上，我们还封装了一层“断路器”和“强制刷新”。如果 Redis 操作异常率在 10 秒内超过 30%，不再依赖定时刷新，直接调用 `client.refreshTopology()` 强刷。同时结合 `spring-retry`，对可恢复异常（`RedisConnectionException`、`READONLY`）做指数退避重试（50ms → 100ms → 200ms），最多 3 次。这确保了短暂的切换抖动完全被应用层吸收。

**【追问回答】**

- **读写分离的额外复杂度**：哨兵模式下做读写分离，最大的坑是哨兵只负责主库的故障通知，对从库的上下线，你需要额外订阅 `+slave` 和 `-slave` 事件来动态调整“读连接池”的节点列表。其次，业务必须容忍“读延迟”，我们是在 SQL 中标记“可读从库”，结合 Spring 的 `@Transactional(readOnly=true)` 路由，但核心交易链路强制走主库。这就像你点菜和结账必须找同一个店员，但看菜单可以找任何一个人，并且要接受菜单可能不是最新版。

【面试官视角点评】
*候选人展现了从 Jedis 到 Lettuce 的完整踩坑与进化史，能讲出 NoSuchElementException 和 READONLY 的真实场景，证明是真枪实弹做过。特别是“强刷拓扑”和“断路器”的组合设计，说明他对客户端在故障切换时的不确定性有深刻理解，并设计了多层防御。*

### 第3题：Redis Cluster 分片与客户端路由

这个问题直击 Cluster 设计的核心——**用哈希槽的静态划分，来换取去中心化路由的动态稳定性**。我先讲 MOVED 和 ASK 的本质差异。

**MOVED 是“你永远不该再来找我”**。当一个 Key 的槽已经被永久迁移到节点 B，你请求节点 A，A 返回 `MOVED 3999 :7002>`。这告诉客户端两件事：槽 3999 归 7002 了，请更新你的本地槽位映射表，下次直接来 7002。这是**永久性的、要缓存的拓扑纠正**。

**ASK 是“这东西正在搬家，你这次去隔壁问问”**。这是槽迁移过程中的一个临时状态。源节点 A 的槽设为 `MIGRATING`，目标节点 B 的槽设为 `IMPORTING`。请求落到 A 时，如果 Key 还在，直接处理；如果不在了，A 返回 `ASK 3999 :7002>`。客户端必须临时向 B 发请求，并先发送 `ASKING` 命令，这相当于告诉 B：“我这是帮忙问一下，别给我甩个 MOVED 回来。”请求完，客户端**不更新**槽位映射表，因为槽迁移还没完。这是**临时性的、不缓存的路由转发**。

这精巧的区别，源于我讲 Cluster 时提到的核心矛盾：**迁移过程的原子性**。Redis 是按 Key 逐个迁移的，所以必然存在一个槽的 Key 一半在 A、一半在 B 的窗口期。MOVED 和 ASK 的组合，优雅地在客户端无感的情况下，解决了这个窗口期的路由问题。没有 ASK，迁移就得停写，这在生产上不可接受。

**关于 16384 个槽这个数字，就像我讲过的公社食堂的 16384 个固定饭碗，这个数字不是拍脑袋。** 心跳包里的槽位信息是个 2048 字节（2KB）的位图，16384 位正好填满。太小，比如 8192，节点数多时槽粒度过粗，数据迁移单位大，不灵活；太大，比如 65536，心跳包会膨胀到 8KB，在 1000 节点集群时，Gossip 消息的网络开销会急剧增加。这是一个经典的**空间与灵活度的权衡**。

在 Lettuce 里，Smart Routing 的实现异常精巧。它通过 `ClusterTopologyRefresh` 组件，定时或在收到 MOVED 时，执行 `CLUSTER SLOTS` 命令获取完整的槽-节点映射，存在本地 `Partitions` 对象里。每次请求，客户端本地计算 `CRC16(key) % 16384`，找到 `Partitions` 中对应的节点，直连过去。如果收到 MOVED，则作为事件触发一次即时拓扑刷新，把槽位映射纠正过来，然后对当前请求进行一次对齐到新节点的重试。

**我们迁移时遇到过大量 TRYAGAIN 和 MOVED 风暴。** 那是在一次大促前紧急扩容，运维用 `redis-cli --cluster reshard` 时用了默认的 `--pipeline 10000`，且没有限速。短时间内海量 Key 迁移，网卡打满，导致迁移中的槽返回大量 TRYAGAIN（源节点忙），部分请求连续收到 MOVED。P99 延迟从 2ms 飙到 300ms。

**优化：** 我们把 `reshard` 的 pipeline 降到 100，加上 `--timeout 3000`，让迁移更平滑。并且在客户端，把 `max-redirects` 从默认的 5 降到 2，避免在频繁 MOVED 时反复跳转，快速失败走兜底逻辑。更进一步，我们禁用了默认的自适应刷新，改由应用方在监控到异常时，通过控制面主动调用 `CLUSTER SLOTS` 并更新缓存的映射，把拓扑更新的时机牢牢控制在低峰窗口。这个教训就是：**去中心化的自动路由，在极端抖动时会带来级联的探索开销，有时需要有中心控制来平稳过渡。**

**【追问回答】**

- **Spring Data Redis 的关键配置**：`spring.redis.cluster.max-redirects=3`（防无限重定向），`spring.redis.lettuce.cluster.refresh.period=30s`（周期性确保最终正确），还有连接池的 `max-idle` 和 `min-idle` 要调小，因为 Cluster 下连接分散，池太大浪费资源。最重要的是，一定要开 `cluster.refresh.adaptive=true`，并设置 `adaptive-refresh-triggers` 为 MOVED_REDIRECT，只在必要时刷新。

- **批量操作跨节点**：客户端比如 Lettuce 会通过 `Partitions` 先按槽分组，组内 pipeline，跨组并发发请求，最后在应用层归并结果。代价是增加了一个分组和归并的 CPU 开销，且不再是一个原子的 pipeline。

【面试官视角点评】
*对 MOVED 和 ASK 的区别，候选人用“永远别来”和“临时问问”的表述，瞬间说清了永久和临时的本质差异。能讲出 16384 和心跳包 2KB 的限制关系，是看过源码或官方设计文档的人。迁移风暴的排查和客户端参数调优，展示了他对“去中心化自由与中心化控制”这一架构矛盾的深度思考。*

### 第4题：Cluster 下的数据一致性与分布式事务

这题问到我的心头痛——在 Redis Cluster 下做分布式事务，本身就是一个方向性的错误，我来一步步剖析我们是如何从“试图强上”到“理性接受现实”的。

**首先，Hash Tag 是“带毒的蜜糖”**。把三个 key 设计为 `{order:123}:inventory`、`{order:123}:status`、`{order:123}:purchased`，通过 `{order:123}` 这同一个 hash tag，能强制它们落入同一个槽。在单个槽内，Redis 的 Lua 脚本和 MULTI/EXEC 事务都能保证原子性，看起来很完美。但毒在哪里？用我之前的比喻——**这相当于把整个公社的粮食，全堆到了一个人的饭碗里**。这一个订单的所有压力，全集中在一个节点上。大促时，某个爆品订单的热点，会把单个节点 CPU 打满，导致这个节点上其他所有槽、所有 Key 的操作都慢下来，引发雪崩。

**我们怎么解决这个分布式事务？答案是：不解决。我们改变了问题。**

我们把“原子操作三个 Key”这个强一致性需求，解构成了一套**状态机 + 最终一致的补偿流**。订单处理变成一个状态机：`CREATED → INVENTORY_LOCKED → ORDER_CREATED → PURCHASED_LIST_UPDATED → COMPLETED`。每一个状态的转换，都是对**单个 Key** 的原子操作。比如“锁库存”，就是一个 Lua 脚本对一个 Key（比如 `stock:sku123`）进行原子扣减。如果扣减成功，订单状态这个单 Key 被更新为 `INVENTORY_LOCKED`。如果后续某一步失败，我们通过一个**订单事件表**（可放到 Mysql 或 Kafka 持久）和补偿 Worker，反向执行逻辑（比如库存回滚）。

这就是 SAGA 模式的思想——**把一个跨槽的强一致事务，拆成一串单槽的原子操作，并由异步协调者负责最终一致。** 我们当时用 Kafka 的 Exactly-Once 语义来做补偿消息的可靠投递。

**那如果我们业务就是要求强一致呢？比如绝对不允许库存超卖？** 那我会坦诚地告诉你：**不要用 Redis Cluster 做主存储。** 我会把 Redis 降级为“高性能缓存+计数器”，真正的库存扣减，由单机性能极高的账务 DB（比如自研的或经过极致优化的关系型 DB）在一个事务里完成，然后异步刷新到 Redis。这样，Redis 承受 99% 的读流量，DB 只处理核心的写和事务。在这个架构下，Redis Cluster 的高可用和水平扩展能力得到了最大发挥，而它的分布式一致性短板，被 DB 这一层兜住了。

**【追问回答】**

- **Hash Tag 下热点怎么处理？** 如果键已经被 `{orderId}` 困在一个节点，我们曾用过“热 Key 分身术”：应用层对 `{orderId}` 做读写分离，写仍走原节点，但读时，客户端会随机发到 `{orderId}-副本1`、`{orderId}-副本2` 等多个克隆 Key。这些克隆 Key 分布在集群的其他节点上，由异步任务负责从主 Key 同步过来。这牺牲了读的瞬时一致性，换来了读的横向扩展。

- **Redlock 在 Cluster 里能用吗？** 官方推荐在单实例或 Sentinel 下用。Cluster 下，锁的几个 Key（如 `lock:order123`）可能被分到不同槽，Redlock 算法要操作多个独立的 Redis 节点，在 Cluster 中难以保证这些节点是完全独立的（可能共用一个物理机），且时钟依赖问题依然无解。我的态度是：在 Cluster 里，用单实例锁保小范围临界区就好，别强求分布式锁。

【面试官视角点评】
*一针见血地指出 Hash Tag 是“带毒的蜜糖”，展示了不被技术表象迷惑的架构意识。核心亮点是“改变问题”的思维——不强上分布式事务，而是用状态机和 SAGA 模式把强一致降级为最终一致。最后的架构决策——“不用 Redis 做主存储”更是老手风范，知道工具的边界在哪里。*

### 第5题：集群故障排查与性能优化

一个 CPU 90% 的集群节点在报警，我的排查思路就像一个急诊医生的流程：**望闻问切，先救命后治病。**

**第一步：望——从外围监控定界。** 我会先看 Grafana 大盘：该节点 CPU 飙高，但 QPS 是否也同比飙高？如果 QPS 没变，CPU 高，可能是慢命令；如果 QPS 也高，可能是流量倾斜。再看网络出口流量，如果突然暴涨，大概率是有大量的数据返回。这个阶段，用 `redis-cli --stat` 快速看下连接数和命令处理速率的趋势。

**第二步：闻——直接查慢日志。** `SLOWLOG GET 20` 是这个时候的“听诊器”。你题目里说看到了 `KEYS *` 和 `HGETALL` 一个大 key。这就基本定位了：`KEYS *` 是 Redis 的“全表扫描”，它会遍历整个键空间，直接吃满 CPU。`HGETALL` 一个几 MB 的大 key，一次返回所有 field，把网络卡和 CPU 序列化全打满。根据我的血泪经验，**任何在生产环境出现 `KEYS *`，都必须立刻当作 P0 事故处理。**

**第三步：问——找哪个业务、哪个 key。** 慢日志会显示出客户端 IP 和命令。我会立刻找到负责该服务的开发，从代码里查。如果没法马上定位，用 `CLIENT LIST` 看当前连接，找出执行命令的客户端 IP。对于大 key，用 `redis-cli --bigkeys` 扫描，它会给出每种类型最大 Key 的名字和大小。

**第四步：切——先救命。** 如果这个节点已到 95%，随时可能 OOM 或宕机。应急方案：
- 对那个执行 `KEYS *` 的客户端，直接 `CLIENT KILL`。
- 如果是有用但太大的 key，比如题目里的 `HGETALL` 热点，我会临时在该节点上通过 `CONFIG SET` 开启 `lazyfree-lazy-server-del yes`（惰性删除，防阻塞），然后如果业务能容忍，立即把读该 key 的流量，通过本地缓存在应用层短路掉。

**长效方案：** 这是“治病”阶段。大 key 必须拆：像 `HGETALL` 的大 Hash，我通常建议业务按某个维度（如时间窗口、用户 ID 尾号）拆分：`user:1000` 变成 `user:1:000`、`user:1:001`……再用 HGET 取单个 field。热 key 则采用我前面提到的“读写分离+多副本”或“应用层本地缓存（Caffeine）+ 消息通知失效”的二级缓存架构。**本质是用多个缓存副本和更快的本地存储，去分担远端单点的压力。**

**监控倾斜与自动告警：** 我们有一套自研的 Redis 集群巡检程序，它伪装成一个 `redis-cli` 的客户端，周期性执行 `CLUSTER SLOTS` 拿到所有槽，然后采样 `INFO` 中每个节点的 `total_commands_processed` 差值，算出每个节点的实时 QPS。如果某个节点 QPS 持续超过集群平均 QPS 的 150%，就通过 Prometheus + AlertManager 发出 `slot_unbalance` 告警。我们还在客户端埋点，统计每个 Key 的请求频率，上报到监控系统，用于捕捉热 Key。

**【追问回答】**

- **紧急限流该热 key**：我不能直接改业务代码，最快的是在该节点前用一层 Nginx/Proxy 做个临时限流，或者更暴力但有效——在该 Redis 节点用 `CONFIG SET` 临时将这个热 key 的内存逐出策略调整为 `volatile-ttl`（如果允许它过期），或者对该热 key 做个极短的过期时间，用 `EXPIRE`。如果完全不能动，那就是典型的“事前没备份方案”的悲剧了，只有扩容和拆。

- **500 万 hits 热 key 怎么安全降级**：我不删。删除会导致流量一瞬间全打在 DB 上，瞬间击垮。我会先让业务给 DB 加个限流，然后把这个 Key 对应的值，直接推到所有应用节点的本地 Caffeine 缓存里，设置一个 1-2 秒的短暂失效。等应用监控显示本地缓存命中率达 99%，再逐步删除 Redis 中的热 key。这是“先建新路，再封旧桥”的平滑降级策略。

【面试官视角点评】
*非常成熟的排查套路：“望闻问切”四步法，兼顾了应急和长效。尤其是“先救命后治病”的理念和 CLIENT KILL、惰性删除的应急技巧，都是实战中才能学到的。“先建新路，再封旧桥”的降级思路，体现了对系统雪崩风险的敬畏。*

### 第6题：架构选型与演进路线

这是一个架构师必答题。我脑海中规划的社交 App 缓存架构演进，就像一个城市的交通系统扩建。

**第一阶段：单机时代。** 总数据 < 10G，QPS < 5 万。这阶段的核心是“简单”。一个主从复制 + Keepalived VIP 或者 DNS 指向就行。哪怕挂了，手动切从库，RTO 在 5 分钟内是可以接受的。这个阶段我只有一个原则：**为未来做最少的承诺**。客户端配置一个 Redis 地址就好，别把架构做重。

**分界点与迁移信号**：当 RTO 要求变成“1 分钟内必须恢复”，或单实例内存逼近 20G（启动慢、fork 卡），就进入 Sentinel 阶段。

**第二阶段：Sentinel + 主从。** 数据 20-50G，QPS 10-15 万。引入哨兵，实现自动故障转移。此时业务代码必须做第一次改造：从连单点 Redis 变成连哨兵。这个阶段的演进思想是：**用哨兵集群的共识，换取人工高可用的自动化。** 如果读压力大，在这个阶段就可以通过客户端简单判断，把读了从库——但要打上一个“容忍延迟”的标签。

**分界点与迁移信号**：当单主库写入 QPS 连续一周超过 8 万，或总数据超过 60G 导致 RDB 持久化时间超过业务窗口，就必须水平拆分了。这时，面临选择：Cluster 还是 Sentinel + Proxy？

**第三阶段：水平扩展（关键抉择）。**
如果我们的团队规模较小，业务迭代快，不想大规模改客户端，我会选择 **Sentinel + Proxy（如 Codis 或自研 Proxy）**。Proxy 对客户端屏蔽了分片，就像你打车不需要知道城市道路如何分区一样。客户端代码只需连 Proxy，迁移时只需改 Proxy 的路由表。代价是，引入了一层网络延迟，和一个需要自己维护的 Proxy 集群。

如果我们的团队有较强的中间件自研能力，且希望减少组件，**Redis Cluster** 是更彻底的方案。客户端直连，延迟最低，去中心化让它理论上无限扩展。代价是，客户端必须升级为 Smart Client，而且业务必须遵守“单 Key 操作”和“同槽原子”的约束。

以现在的经验，如果业务模型是典型的社交 Feed 流（Key 天然按用户 ID 分布，且多数是单 Key 操作），我会在数据量预计半年内超 100G 时，直接上 Cluster，一步到位。而如果是交易系统，Key 之间事务性强，我可能长期停留在 Sentinel + Proxy，甚至只用 Sentinel 做高可用，靠垂直升级单机内存来应对数据增长，直到单机物理极限。

**关于云服务还是自建**：如果公司处于 A/B 轮，人少事多，我一定选云服务（AWS ElastiCache / 阿里云 Redis）。让专业的团队做运维、扩缩容、备份恢复，我的精力聚焦在业务逻辑上。但当公司到了 C/D 轮，Redis 集群规模上百个，云服务的资源隔离、定制化不足、版本滞后、命令裁剪等问题会凸显。届时，我可能会组建一支 2-3 人的专职 DBA 团队去自建和深度定制，因为规模效应下，自建的成本和灵活性优势远大于运维负担。

【面试官视角点评】
*没有非黑即白的选型，而是根据团队能力、业务特性、成本做动态权衡。特别是对 Proxy 和 Cluster 的抉择分析，切中要害——不是哪个技术先进就用哪个，而是哪个匹配当前阶段的矛盾和约束。这种“顺势而为”的架构观，是成熟架构师区别于“技术原教旨主义者”的关键。*

### 第7题：代码设计——统一客户端工厂

这道题考究的是，如何将我之前讲过的两种截然不同的技术模式，用代码优雅地统一。

我的设计核心思想是：**接口隔离与策略封装**。上层业务只看到 `RedisOperator` 接口，底层的 Sentinel 和 Cluster 的差异，通过不同的实现类和工厂策略封装起来。

**核心类结构：**

- `RedisMode` 枚举：`SENTINEL` 和 `CLUSTER`。
- `RedisClientConfig`：包含模式、地址列表、连接池配置、重试配置等。
- `RedisOperator` 接口：封装 `get(String key)`、`set(String key, String value)` 等常用命令，不暴露底层连接类型。
- `RedisOperatorFactory`：工厂，根据 `RedisMode` 创建对应的 `RedisOperator` 实现，并用 `AtomicReference` 持有当前活跃的实例。
- `LettuceSentinelOperator` 和 `LettuceClusterOperator`：分别封装 `StatefulRedisSentinelConnection` 和 `StatefulRedisClusterConnection`。

**实现要点：**

**1. 连接获取与重试的伪代码逻辑（核心骨架）：**
```java
public <T> T execute(RedisCommand<T> command, String key) {
    int attempts = 0;
    // 指数退避的初始间隔
    long backoff = 50;
    
    while (attempts < maxRetries) {
        try {
            // 获取连接。对不同模式，getConnection 内部逻辑不同。
            // Sentinel 模式: 从连接池获取到当前主节点的连接。
            // Cluster 模式: 根据 key 算出 slot，获取到对应节点的连接。
            StatefulRedisConnection<String, String> conn = getConnection(key);
            return command.run(conn.sync());
        } catch (RedisConnectionException | ReadOnlyException e) {
            attempts++;
            if (attempts >= maxRetries) {
                throw new RedisOperationException("Redis retry exhausted for key: " + key, e);
            }
            
            // 关键：重试前，触发一次拓扑刷新，确保下次 getConnection 能找到对的节点。
            if (e instanceof RedisConnectionException || e.getMessage().contains("MOVED")) {
                topologyManager.triggerRefresh();
            }
            
            // 指数退避，加一点随机抖动，防止惊群
            sleep(backoff + ThreadLocalRandom.current().nextLong(50));
            backoff = Math.min(backoff * 2, 1000); // 上限 1 秒
        }
    }
}
```

**2. 拓扑刷新与隔离：**
`TopologyManager` 是一个独立的、单线程的 ScheduledExecutorService。它负责：
- 对于 Sentinel：订阅 `+switch-master` 频道，收到事件后异步重建 Sentinel 连接池。
- 对于 Cluster：定时（如 30s）或由 MOVED 异常触发，执行 `CLUSTER SLOTS`，更新本地 `Partitions` 缓存。

**这里的关键是异步和隔离。** 如果刷新失败，只记录日志和报警，绝对不能抛异常给业务线程。刷新用独立的、低优先级线程池，避免业务线程阻塞在拓扑更新上。

**3. 无中断切换策略：**
当运维通过配置中心下发一个模式切换指令（比如从 Sentinel 切换到 Cluster），工厂的 `refreshClient()` 方法被调用。
- 首先，根据新配置，在后台构建一个新的 `RedisOperator` 实例，并对其连接池进行预热（创建少量连接）。
- 然后，通过 `AtomicReference.compareAndSet` 原子地替换当前的 `operator`。
- 旧 `operator` 并不立即销毁，而是放入一个延时队列，5 分钟后才关闭其连接池。这个**“宽限期”**保证了那些正在执行中的、已经拿到旧连接的请求能够优雅完成，不会突然中断报错。
  这就是我前面说的，用空间（同时存在两套连接池的资源）换时间（业务无中断）。

**【追问回答】**

- **统一封装不同类型的连接**：我的 `getConnection` 方法内部，实际上是返回一个包装类 `ConnectionWrapper`，它持有通用的 `RedisCommands<String, String>` 接口，屏蔽掉底层是单节点还是集群连接。Lettuce 的好处就在这里，它的 `sync()` 方法返回的命令接口是一样的。

- **幂等性判断**：在重试伪代码中，`maxRetries` 适用于所有可恢复异常，但真正生产级要求更细粒度。我会在 `RedisCommand` 接口里加一个 `boolean isIdempotent()` 方法。写操作如 `SET`、`DEL` 是幂等的，`INCR`、`LPUSH` 则不是。非幂等写操作，默认不重试，直接抛异常，由上层业务决定是否补偿。这是必须从代码层面划清的边界。

【面试官视角点评】
*这个设计展现了良好的抽象功底。核心亮点有三：其一，重试前强制拓扑刷新，让重试逻辑和故障恢复紧密结合。其二，拓扑刷新完全异步隔离，不阻塞业务，体现了对生产环境“稳定性压倒一切”的追求。其三，平滑切换的“宽限期”设计，是真正做过线上变更的人才能想到的细节。*

### 第8题：大规模集群运维与平滑扩缩

管理 100 个节点的 Redis Cluster 扩容，这绝非敲敲 `reshard` 命令那么简单。这必须是一场精心策划的、分阶段执行的“外科手术”。

**我的平滑迁移方案分为五个阶段：**

**阶段一：术前检查与准备（T-3天）**
- **大 Key 扫描**：运行 `redis-cli --bigkeys` 扫描全集群，把所有超过 10MB 的 Key 列表发给业务方确认。因为我们知道，迁移大 Key 会阻塞。必须提前处理，要么删除，要么拆分。
- **流量建模**：获取过去一周的峰值 QPS 和分时段的流量分布，选择业务最低峰（通常是凌晨 3-5 点）作为迁移窗口。
- **节点预热**：新加的 20 个节点先加入集群作为“空主节点”，提前建立好从库和主从复制关系，确保到时不手忙脚乱。

**阶段二：迁移计划生成（T-1天）**
`reshard` 的原理，正如我讲解 Cluster 时提到的，是**按槽为单位，逐个原子地搬迁 Key**。`redis-cli --cluster reshard` 底层是一个精巧的循环：
1. 通过 `CLUSTER SETSLOT <slot> MIGRATING <source-id>` 和 `CLUSTER SETSLOT <slot> IMPORTING <source-id>`，把源和目标节点设置为迁移状态。
2. 源节点执行 `CLUSTER GETKEYSINSLOT <slot> <count>`，获取槽内的一批 Key。
3. 用 `MIGRATE` 命令，以原子 pipeline 的方式把这批 Key 推向目标节点。`MIGRATE` 的 RESTORE 是在目标节点上执行的，此时如果请求通过 ASK 转向到目标节点，就能拿到最新数据。
4. 槽内所有 Key 迁移完后，通过 `CLUSTER SETSLOT <slot> NODE <target-id>`，把槽的归属权正式从源转移到目标，并广播给全集群。

我会根据流量模型，写一个迁移脚本，**把迁移拆成几十个小批次，每批迁移 100-200 个槽，批间暂停观察 5 分钟**。这样，万一有抖动，能立刻停止，影响面极小。

**阶段三：手术进行（T 日凌晨）**
- 在低峰期，运行迁移脚本。我同时开着三个监控大盘：A 盘看迁移槽的源和目标节点的 CPU、网络和延迟；B 盘看业务整体 P99 延迟和错误率；C 盘看集群的 `cluster_slots_migrating` 和 `cluster_slots_importing` 状态数量。
- 一旦业务 P99 延迟涨幅超过 50%，立即暂停脚本，把已迁移一半的槽回滚或暂停，排查原因。

**阶段四：异常回滚与恢复**
如果 `MIGRATE` 超时，迁移会中断。此时，那个卡住的槽处于 `MIGRATING` / `IMPORTING` 状态。我的回滚步骤是：
- 先找到所有处于迁移状态的槽：`CLUSTER SLOTS` 结合 `CLUSTER NODES`。
- 对每个迁移中的槽，执行 `CLUSTER SETSLOT <slot> STABLE`，清除迁移状态，让它回归源节点。这样业务请求就不再 ASK 转向，暂时回到源节点。
- 待网络或节点稳定后，重新开始迁移这个槽。

**自动化扩容的控制面设计：**
我会设计一个 `Redis AutoScaler` 组件。它是一个独立部署的 Go 程序。
- **监听**：通过 Prometheus 获取每个节点的 CPU/内存，自定义 Exporter 获取槽级 QPS。
- **决策**：当触发“CPU > 70% 且节点数 > 3”规则时，发起扩容。决策算法是一个贪心算法：计算各节点 QPS，计算平均值，尝试将 QPS 最高节点的槽，迁移到新节点，平衡各节点 QPS。
- **执行**：调用云 API 创建实例，发送 `CLUSTER MEET`，然后通过类似 `redis-cli --cluster rebalance --weight` 的逻辑，分批次、有超时地执行迁移。全流程状态写入 MySQL 状态机，支持断点续传。
- **监控**：迁移过程中持续监控 ASK 错误比例，当比例为 0 且所有槽均未处于迁移态，任务完结。

这套自动化，就是把人工的谨慎和判断，沉淀为程序的规则和状态机，是运维经验的产品化。

**【追问回答】**

- **500MB 大 Key 怎么办？** 如果 `--bigkeys` 扫出这种庞然大物，硬迁移就是自杀。`MIGRATE` 时这个 Key 会被序列化并通过网络发送，500MB 会导致 `MIGRATE` 超时，源节点 CPU 和网络打满，还会阻塞其主线程。我的强制规则：生产集群不允许存在超过 10MB 的单个 Key。发现后，立刻和业务沟通重构，用哈希拆分或直接存入对象存储，Redis 中只存轻量索引。

- **迁移中断的恢复**：我之前提了用 `CLUSTER SETSLOT <slot> STABLE` 手动回滚。但在自动化设计中，更精细。迁移程序会为每个槽维护状态：`INIT`, `MIGRATING`, `IMPORTING`, `DONE`, `ERROR`。当中断时，该槽状态为 `ERROR`。恢复程序会重新检查源和目标节点上该槽的状态，尝试修复。如果无法自动修复，告警，人工介入执行 `STABLE`。

【面试官视角点评】
*这不仅是技术回答，更是一份可执行的迁移 SOP。候选人的“术前检查—计划—手术—回滚”分阶段方案，以及对大 Key 的零容忍规则，都体现出大规模运维中最重要的品质：对生产环境的敬畏心和风险管理意识。自动化控制面的设计，更是把经验提炼成了系统，是高级工程师向架构师进化的标志。*
