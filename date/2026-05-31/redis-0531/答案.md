### 苦难时代：那个“单点”决定生死的年代

要理解哨兵和集群，我们得先回到那个**“没有它们”的深夜**。

想象一下，你在维护一个初代的电商订单系统。所有 Redis 读写都指向一台性能强劲的单机。它像一个超级收银员，记忆力惊人，手脚麻利。突然有一天凌晨，物理内存条坏了，或是机房网络交换机一个端口闪断。你看见了什么？

**系统全景崩溃。** 所有依赖缓存的商品页、下单、库存查询，瞬间被巨大的流量直接击穿到后端的 MySQL。MySQL 连接池耗尽，线程堆积，整个服务雪崩。你从床上弹起来，披上衣服，唯一的操作就是：重启 Redis，或者手动把 DNS 切到一台冷备的从库上，然后祈祷数据没丢太多。那个年代，**恢复时间（RTO）是用“业务骂声的分贝”来衡量的，而数据丢失量（RPO）纯粹看运气。**

这便是苦难的根源：**状态服务需要高可用，但手工运维既慢又不准，分布式环境下的“主从”关系，如果靠人用眼睛看、用手切，一定会带来灾难。**

### 思想破局：从“瞎眼”到“瞭望塔”与“自治联邦”

Sentinel 的出现，解决了第一个核心矛盾：**如何让一群进程，对“谁是主”这个问题自动达成共识，并在主倒下时，选出一个新主。**

这里我给你讲一个故事，我叫它 **“老北京的胡同自治”**。

把 Redis 的主从节点想象成胡同里的“粮店”。主粮店（Master）负责卖粮（写），从粮店（Slave）只负责门口摆摊发粮（读），并且时刻从主粮店进货（复制）。而 Sentinel，就是胡同里几位**德高望重的、不卖粮的“老街坊”**。

他们只干三件事：
1.  **盯着主粮店**：隔几秒就派学徒去敲敲门，喊一声“掌柜的，还好吗？”（PING）。如果某个街坊发现主粮店半天没应声，他就在心里记下“我觉得老王不行了”（**主观下线 SDOWN**）。但他不会立刻嚷嚷，因为可能只是他自己耳朵背（网络抖动）。
2.  **街坊开会（投票）**：于是，这个街坊就去问其他几位街坊：“你觉着老王还行吗？” 当足够多（`quorum`）的街坊都说“对，我也叫不应他了”，这就成了公认的事实：“老王真的倒下了”（**客观下线 ODOWN**）。这个 `quorum` 机制，就是用“通信复杂度”换取“判断的可靠性”，避免了单人误判。
3.  **选出新掌柜**：一旦确认老王不行，这些街坊会内部推举出一个最有威望的人（**Leader 选举**，基于 Raft 协议的简化版），由他来决定把哪家从粮店升级为新主粮店，并昭告整个胡同（通知客户端、更新配置）。这个“瞭望塔 + 自治委员会”的设计，把**人工救火变成了系统自愈**。

而 Cluster 解决的是第二个矛盾：**当一个粮店囤不下全城粮食时怎么办？**

Cluster 的思想是 **“土地分封制”**。
它把全城划成 16384 块均等的“土地”（哈希槽 slot）。每把“粮食”（Key）根据名字（CRC16 哈希）都能算出自己属于哪块地。几家粮店（节点）瓜分这些地，每家负责其中几千块。当你去找粮食，随便问一家店，如果不在他家，他会直接告诉你：“出门左转，去老张家的第 9527 号地”（**MOVED 重定向**）。这就把**单点的容量瓶颈，用分片路由的方式横向扩展了**。为什么是 16384 块地？这是个优美的工程平衡：节点间心跳通信的位图大小正好 2KB，不多不少，网络开销极低；且就算发展到上千个节点，每个节点也能分到足够均匀的地，不会太碎。

这便是思想破局：**Sentinel 是高可用的“瞭望塔”，Cluster 是无限扩展的“分封制”。**

### 原理深剖与血泪案例

现在，我们把这两套哲学，代入到你的面试题中，用真实的血与泪来映照原理。

#### **第1题：Sentinel 故障转移的完整过程与脑裂之殇**

那天凌晨，你的订单服务主节点宕机，内部发生了一场精密的政治交接：

1.  **风暴前夜**：所有 Sentinel 每 1 秒给主节点发 `PING`。当其中一个 Sentinel 在 `down-after-milliseconds` （假设配了 3 秒）内没收到有效回复，它标记主节点 **SDOWN**。
2.  **达成共识**：这个 Sentinel 立刻向其他 Sentinel 询问。若大于等于 `quorum`（假设是 2 个）的 Sentinel 都同意，主节点被标记为 **ODOWN**。这里有个**致命陷阱**：如果你的 Sentinel 集群只有 2 个节点，`quorum` 为 2。一旦其中一个挂了，剩下的那个永远无法达到法定人数，**故障转移彻底失灵**。这就是为什么 Sentinel 必须部署**至少 3 个且为奇数个**的深刻原因。
3.  **领袖登基**：标记 ODOWN 的那个 Sentinel，会发起一场**领袖选举**。它增加自己的纪元（epoch），请求其他 Sentinel 投票。它用的是 Raft 协议的精髓：先到先得，一任内只能投一次。获得半数以上票数者成为领袖。
4.  **政权更迭**：领袖 Sentinel 从所有健康的 Slave 中，根据优先级、复制偏移量等规则，选出最优的一个作为新主，发送 `SLAVEOF NO ONE`。当新主成功上位，Sentinel 会通过 Pub/Sub 的 `+switch-master` 频道，向全城广播这一消息。

**血泪案例：脑裂（Split-Brain）**
有一年双十一，我们因机房内部一个核心交换机故障，导致机架 A（旧主）和机架 B（Sentinel 和新主）被网络隔离。Sentinel 们看不到旧主，果断完成了切换。但此时，旧主还活着，它与一部分还没来得及感知拓扑变化的旧客户端，组成了另一个“王国”，还在接收写请求。**两个主同时对外服务，订单号重复、库存错乱，这就是脑裂，数据一致性灾难。**

那次之后，我才深刻理解，**Sentinel 的 `quorum` 解决的是“能不能切换”的问题，但解决不了“旧主该不该活”的问题。** 真正的救命稻草是旧主自身配置的这两行：
```
min-slaves-to-write 1
min-slaves-max-lag 10
```
它告诉旧主：“如果你能联系上的小弟少于 1 个，或者小弟们落后你超过 10 秒，你就自觉闭嘴，拒绝写入。” 这就是**在数据完整性和可用性之间的终极权衡**。你必须在旧主上牺牲部分可能的可用性，来换取整个集群的一致性。

至于 `down-after-milliseconds` 和 `failover-timeout`，我的调优心法是：前者是**故障检测的敏感度**，设短了，网络打个喷嚏就全集群抖动；设长了，业务就得忍受更长的中断。我一般设 3-5 秒。后者是**整个切换事件的过期时间**，如果在这个时间内任何步骤（如新主同步）没完成，整个切换过程作废重来。根据你的网络和复制速度，可以设 30-120 秒。

#### **第2题：Java 客户端集成——那只“认死理”的连接池**

Java 客户端与 Sentinel 的集成，像极了一个**认路只认第一次的出租车司机**。JedisSentinelPool 就是典型。

它启动时，会连接你给的 Sentinel 地址，喊一声 `SENTINEL get-master-addr-by-name`，拿到当前主节点的 IP 和端口，然后建立连接池。同时，它会悄悄订阅 Sentinel 的 `+switch-master` 频道。这就是它的**两套导航系统：主动问路和被动听广播**。

**血泪案例：`NoSuchElementException` 风暴**
主库宕机，切换发生，新主诞生。但就在这毫秒级的瞬间，连接池里还躺着一大批指向旧主的“死连接”。业务线程去池子里借连接，拿到旧连接，一用就报错 `Connection refused`。如果连接池的“试借”逻辑不做校验，它就不断借出、报错、回收，直到耗尽，抛出 `NoSuchElementException`。

更惨的是，如果旧主没有彻底挂，而是复活成了一个 Slave，那些拿着旧连接的线程去写它，会收到一个冰冷的 `READONLY error`，错误一大片。

**生产级兜底策略**：
我们后来全面切换到 Lettuce。它原生支持基于拓扑事件的**自适应刷新**。一旦收到 `+switch-master` 或连接断开，它立刻失效旧连接，重建与新主的连接。但即便如此，也无法做到“零感知”。我们的最终方案是：
- **短链路，快失败**：设置合理的连接超时（`connect-timeout`）和命令超时（`command-timeout`），不死等。
- **上阵父子兵（重试+断路器）**：在连接获取层面，我们用指数退避重试（`exponential backoff`）。同时，在业务层引入断路器（Resilience4j），当 Redis 操作错误率超过阈值，直接熔断 10 秒，并主动降级，例如从数据库回源读数据，或返回默认空值。

#### **第3 & 4题：Cluster 分片、重定向与分布式事务之痛**

当数据大到必须分片，我们从“胡同”走进了“城市”。Cluster 带来了 `MOVED` 和 `ASK`，这是两个极其精妙的路标。

- **`MOVED`**：相当于路牌写着“李家庄已永久搬迁至幸福大街”。永久重定向。客户端收到后，必须更新自己本地的槽位映射表。Lettuce 这类“聪明”的司机，会立刻更新自己的“城市地图”。
- **`ASK`**：相当于临时路牌：“前方施工，9527号地块暂时在我这儿处理”。临时重定向。发生在 `MIGRATE` 槽迁移过程中。客户端不能更新地图，但必须带着 `ASKING` 标志，乖乖去临时节点处理这次请求。

**血泪案例：跨 Slot 的 Lua 原子梦碎**
你有一个完美的 Lua 脚本，原子地操作订单状态、库存、用户已购列表。这三把钥匙，被 CRC16 算法无情地分到了三个不同的节点。Lua 脚本发过去，Redis Cluster 会直接报错：`CROSSSLOT Keys in request don't hash to the same slot`。你试图跨节点保证原子性，**这本身就是在分布式系统中追求单机神话**。

**架构的权衡与妥协**：
- **Hash Tag 的诱惑与诅咒**：你用 `{order:123}` 把它们强行绑在一起。这立刻解决了原子性问题，但副作用如同饮鸩止渴：`order:123` 成了一个超级热 key，所有压力都打在一个节点上。一旦这个节点 CPU 飙升，整个订单链路瘫痪。数据倾斜是 Hash Tag 的原罪。
- **我们的最终信仰**：放弃 Redis 集群内的分布式事务。**绝不**让 Redis 去承担它不擅长的 CP 角色。我们采用了 **SAGA 长事务补偿模式**。
    1.  把核心状态流转（比如“下单”）设计成一个状态机。用 Redis 的 `SET NX` 或单节点 Lua 脚本，先原子地抢占“待支付”状态。
    2.  后续的库存扣减、已购列表更新，全部通过可靠消息队列异步完成。
    3.  如果某个环节失败，发送反向补偿消息，将订单状态回滚为“已取消”。
        **关键认知**：Redis 在集群模式下是 AP 系统，为高性能而设计。强一致性的底线，应该交给数据库或专门的事务协调器。Redis 只做它最擅长的事：以微秒级速度完成状态的暂存和校验。

#### **第5题：集群故障排查——那场 90% CPU 的午夜凶铃**

凌晨，监控告警：一个 Redis Cluster 节点 CPU 持续 90%+。

我的排查路径，就像一位老侦探：
1.  **现场勘察（监控）**：先看 Grafana。该节点 QPS 没有暴涨，但 CPU 飙升，网络出入口流量巨大。基本判定是**大 Key 或慢命令**。
2.  **寻找物证（慢日志）**：`SLOWLOG GET 100`，赫然发现大量 `KEYS user:session:*` 和 `HGETALL cart:10000001`。`KEYS *` 是线上绝对禁止的，它会遍历整个键空间；而那个 `HGETALL` 返回了一个十几 MB 的购物车数据。
3.  **锁定嫌疑（Big/Hot Key）**：用 `redis-cli --bigkeys` 和 `--hotkeys` 扫描，确认了那个巨型购物车 key 和另一个被疯狂访问的秒杀商品 key。
4.  **抓捕归案（业务代码）**：定位到是运营后台一个导出所有在线用户的功能，直接用了 `KEYS *`；而秒杀商品，前端没做任何缓存和限流。

**应急与长效方案**：
- **拆**：那个巨型购物车 key，我们按用户 ID 尾号拆成 10 份 `cart:10000001:{0..9}`。
- **分**：秒杀热 key，我们在本地 Caffeine 做了一级缓存，每 1 秒刷新一次，挡住了 99% 的读流量。写操作则用队列异步削峰。
- **限**：在 Redis 前，我们没有直接限流 Redis 命令，而是在业务网关层对特定 API 进行令牌桶限流，从源头掐断。
- **监**：建立了 slot 维度的 QPS 倾斜监控。通过 Prometheus 采集每个节点的 `node_info`，计算每个 slot 的 OPS。一旦某个 slot 的 OPS 超过平均值的 3 倍，立即告警。

#### **第6 & 7题：架构演进与代码设计——从作坊到工厂**

从单机 -> Sentinel -> Cluster，不是技术的堆砌，而是业务生命周期的必然。

- **单机期**：数据 < 10G，QPS < 5W。主从 + 手动切。这时的核心是**活下去**。
- **Sentinel 期**：数据 10-50G，QPS 5-15W。高可用变得和喝水一样重要，但数据还能装下。我们会引入 **Twemproxy/Codis 这类代理**，让业务不用改造客户端，就能实现初步的分片和平滑扩展。代理方案的优点是把复杂性隔离在中间层，缺点是增加了网络延迟。
- **Cluster 期**：数据 > 100G，QPS > 20W。必须直面分片复杂性。我们选择拥抱原生 Cluster，并要求客户端足够聪明。

**设计统一客户端工厂（第7题答案的伪代码）**
这个工厂的核心是**策略模式**，它需要把“获取连接”和“执行命令”这两个动作，从具体的集群模式中解耦出来。

```java
// 1. 定义顶层抽象：一个可执行命令的接口
public interface RedisCommandExecutor {
    <T> T execute(RedisCommand<T> command);
}

// 2. Sentinel 和 Cluster 模式的实现
public class SentinelExecutor implements RedisCommandExecutor {
    private final LettuceConnectionFactory factory; 
    @Override
    public <T> T execute(RedisCommand<T> command) {
        // 从支持拓扑刷新的连接池获取连接
        return factory.getConnection().sync().dispatch(command);
    }
}

public class ClusterExecutor implements RedisCommandExecutor {
    private final LettuceClusterConnection clusterConnection;
    @Override
    public <T> T execute(RedisCommand<T> command) {
        // 集群连接内部自动处理槽位计算和MOVED/ASK重定向
        return clusterConnection.sync().dispatch(command);
    }
}

// 3. 工厂类，根据配置创建对应的 Executor
public class RedisClientFactory {
    public static RedisCommandExecutor create(RedisConfig config) {
        if (config.getMode() == Mode.SENTINEL) {
            // 构建 Sentinel 连接，配置连接池、刷新策略
            return new SentinelExecutor(...);
        } else {
            // 构建 Cluster 连接，配置最大重试次数
            return new ClusterExecutor(...);
        }
    }
}

// 4. 重试与兜底伪代码
public class RedisTemplate {
    private RedisCommandExecutor executor;
    private RetryPolicy policy = new ExponentialBackoffRetry(2, 5, TimeUnit.SECONDS);

    public <T> T executeWithRetry(RedisCommand<T> cmd, int maxRetries) {
        int attempts = 0;
        while (attempts <= maxRetries) {
            try {
                return this.executor.execute(cmd);
            } catch (Exception e) {
                if (!isRetriable(e)) throw e; // 非重试异常（如数据错误）直接抛
                attempts++;
                if (attempts > maxRetries) throw new ExhaustedRetryException();
                // 异步触发拓扑刷新，不阻塞当前业务线程
                CompletableFuture.runAsync(() -> topologyRefresh.trigger());
                // 指数退避等待
                policy.backoff(attempts);
            }
        }
    }
}
```

**无中断切换**：我们用 `AtomicReference` 持有当前的 `Executor`。当配置变更时，我们**创建一套全新的连接池和 Executor**，然后原子地替换引用。旧连接池不会立刻关闭，而是等其内部的连接自然超时或通过 JMX 手动排空。这保证了模式切换时，连接池无缝交替。

#### **第8题：大规模运维——那场“换引擎”式的平滑扩容**

100 个节点，大促前扩容 20 个。这就像给一架正在高速飞行的客机换引擎。

`redis-cli --cluster reshard` 背后的原理，是**以槽为粒度的在线搬迁**：
1.  它在目标节点上 `CLUSTER SETSLOT <slot> IMPORTING <source-node>`，意为“我准备好引入这块地了”。
2.  在源节点上 `CLUSTER SETSLOT <slot> MIGRATING <target-node>`，意为“这块地要搬走了，别再来找我了”。
3.  接着，源节点执行 `MIGRATE` 命令，以 pipeline 方式，一个接一个地将这块地上的 key 搬到目标节点。搬一个，删一个。
4.  搬完后，向全网广播 `CLUSTER SETSLOT <slot> NODE <target-node>`，所有权正式交接。

**血泪案例：500MB 的迁移噩梦**
我们曾有一个存放用户行为记录的 key，体积高达 500MB，但我们监控不到位，没有被提前发现。在迁移它的时候，`MIGRATE` 命令持续了数秒，直接导致了 Redis 的**主线程阻塞**（`MIGRATE` 是同步操作），进而触发了该节点短暂的 Sentinel 误判和抖动。

**我们的平滑迁移与应急方案**：
- **防患未然**：**绝对**的底线是在迁移前，用自动化脚本扫描 `redis-cli --bigkeys`，对发现的大 key 强制要求业务方拆分。
- **分步迁移**：将迁移过程做成一个**状态机**。每次只迁移 1-2 个 slot，并观察集群的 P99 延迟和 CPU。一旦 P99 超过 10ms 或 CPU 抖动超过 20%，自动暂停 5 分钟。
- **自动化控制面**：我们自研了一个叫 `Redis-Operator` 的组件。它持续监听集群 Metrics，当发现新节点加入且槽位不均时，触发 `rebalance` 计划。它使用**贪心算法**，不断找出流量最高的节点和流量最低的节点，将它们之间的一个 slot 迁移过去，直到所有节点流量标准差小于预设阈值。
- **回滚**：如果迁移中断，状态机必须能清理遗留的 `MIGRATING/IMPORTING` 状态，将未完成迁移的 slot 强制 `STABLE` 回源节点。这套操作我们封装成脚本，确保在 1 分钟内一键回滚。
