## Q1: Sentinel 故障转移

> **演练方式**：手动 kill master → 盯 Sentinel 日志 → 看新主上线 → 复活旧主看降级

### 你需要的终端

| 终端 | 用途 |
|------|------|
| T1 | `redis-cli -h 192.168.3.100 -p 26379 -a 123456 MONITOR` |
| T2 | `redis-cli -h 192.168.3.100 -p 26380 -a 123456 MONITOR` |
| T3 | `redis-cli -h 192.168.3.100 -p 26381 -a 123456 MONITOR` |
| T4 | 执行演练脚本 |

### 步骤

**1) 查看初始拓扑**

```bash
curl http://localhost:8080/api/sentinel/topology | python3 -m json.tool
```

看 `masterFlags` — 应该是 `"master"`。

**2) 查看 Sentinel 配置**

```bash
curl http://localhost:8080/api/sentinel/drill-config | python3 -m json.tool
```

记下 `downAfterMs`（默认 30000=30s）和 `quorum`。

**3) 打开 3 个终端，MONITOR 三个 Sentinel**

```
redis-cli -h 192.168.3.100 -p 26379 -a 123456 MONITOR
redis-cli -h 192.168.3.100 -p 26380 -a 123456 MONITOR
redis-cli -h 192.168.3.100 -p 26381 -a 123456 MONITOR
```

**4) KILL MASTER**

```bash
redis-cli -h 192.168.3.100 -p 6379 -a 123456 SHUTDOWN
```

**5) 盯紧 Sentinel MONITOR 输出，你会依次看到：**

```
+sdown master mymaster 192.168.3.100 6379          ← 主观下线
+odown master mymaster 192.168.3.100 6379 #quorum  ← 客观下线（quorum 达成）
+new-epoch 1                                        ← 纪元递增
+vote-for-leader ...                                ← 投票
+elected-leader ...                                 ← Leader 当选
+switch-master mymaster 192.168.3.100 6379 192.168.3.100 6380  ← 切换完成，新主 6380
```

**6) 验证新拓扑**

```bash
curl http://localhost:8080/api/sentinel/who-is-master
# → master: "192.168.3.100:6380"

curl http://localhost:8080/api/sentinel/topology | python3 -m json.tool
# → masterFlags 应该是 "master"
# → slaves 中会看到 6379 已经不在了（等会复活后出现）
```

**7) 复活旧主——观察降级为 Slave**

```bash
# 重启旧主（根据你的部署方式）:
redis-server /path/to/redis-6379.conf &
# 或 Docker:
docker start <redis-6379-container>
```

回到 Sentinel MONITOR 终端，你会看到：

```
+slave slave 192.168.3.100:6379 192.168.3.100 6379 @ mymaster 192.168.3.100 6380
```

旧主以 **Slave 身份** 重新加入，从新主（6380）同步数据。

**8) 最终拓扑验证**

```bash
curl http://localhost:8080/api/sentinel/topology | python3 -m json.tool
```

现在 `slaves` 列表中应该能看到 `192.168.3.100:6379`，`flags: "slave"`。

### 面试要诀

> "我线上经历过一次凌晨 3 点的 Sentinel 故障转移。当时我在三个终端 tail 了三个 Sentinel 的日志，亲眼看到 `+sdown → +odown → +elected-leader → +switch-master` 这个过程。从 sdown 到切换完成总共约 33 秒，其中 30 秒是 down-after-milliseconds 的检测窗口。旧主恢复后 `+slave` 事件触发了它降级重新加入。我还验证了 Spring Boot 应用的 `/api/sentinel/topology` 接口，确认 Lettuce 的拓扑刷新在切换后自动更新了新主地址。"

## Q2: Java 客户端与 Sentinel 集成

> **演练方式**：用 JedisSentinelPool 和 Lettuce 连接真实 Sentinel，对比行为。

### 步骤

**1) 对比两种客户端**

```bash
curl http://localhost:8080/api/sentinel/client/compare | python3 -m json.tool
```

**2) 查看 Jedis 连接池状态**

```bash
curl -X POST http://localhost:8080/api/sentinel/client/jedis-init | python3 -m json.tool
```

看 `currentMaster` 字段——Jedis 通过 `SENTINEL get-master-addr-by-name` 发现的当前主。

**3) 查看 Lettuce 连接状态**

```bash
curl -X POST http://localhost:8080/api/sentinel/client/lettuce-init | python3 -m json.tool
```

**4) 断路器实验**

```bash
# 查看状态（CLOSED）
curl http://localhost:8080/api/sentinel/client/circuit-breaker

# 手动熔断（模拟 Redis 不可达）
curl -X POST http://localhost:8080/api/sentinel/client/circuit-breaker/open

# 再次查看——状态 OPEN，剩余 10 秒
curl http://localhost:8080/api/sentinel/client/circuit-breaker

# 等 10 秒后——HALF_OPEN
sleep 10
curl http://localhost:8080/api/sentinel/client/circuit-breaker

# 复位
curl -X POST http://localhost:8080/api/sentinel/client/circuit-reset
```

### 面试要诀

> "我们线上从 JedisSentinelPool 切到了 Lettuce。Jedis 的池化模型在主从切换瞬间有短暂不可用窗口——旧连接还没清空，新连接还没建好。Lettuce 的 TopologyRefreshScheduler 收到 +switch-master 后即时切换路由，切换窗口接近零。而且在 Spring Boot 2.x 中 Lettuce 是默认客户端，不需要额外依赖。我还手写了一个简单的断路器——连续 5 次失败熔断 10 秒，防止 Redis 不可用时业务线程全部阻塞。"

## Q3: Cluster 槽位路由

> **演练方式**：连真实 Cluster，手算 CRC16 与 Redis 结果对比。

### 步骤

**1) 查看真实集群的 slot 分布**

```bash
curl http://localhost:8080/api/ops/reshard/cluster-nodes | python3 -m json.tool
```

看 `nodesOutput`——这是 `CLUSTER NODES` 的原始输出。找 `[slots]` 部分，每行末尾的数字范围就是该节点负责的 slot。

**2) 手算 key 的 slot，与 Redis 对比**

```bash
# 用 Spring Boot 算
curl 'http://localhost:8080/api/cluster/slot/calc?key=order:123'

# 用 redis-cli 验证
redis-cli -c -h 192.168.3.100 -p 7000 -a 123456 CLUSTER KEYSLOT order:123
```

两个结果应该一致。

**3) Hash Tag 演示**

```bash
curl 'http://localhost:8080/api/cluster/slot/hash-tag-demo?tag=order:123' | python3 -m json.tool
```

看 `withoutHashTag`（3 个 key 散到不同 slot）vs `withHashTag`（3 个 key 全部同一 slot）。

**4) MOVED vs ASK 对比**

```bash
curl http://localhost:8080/api/cluster/slot/moved-vs-ask | python3 -m json.tool
```

### 面试要诀

> "16384 个槽是 CRC16 取低 14 位。我写了一个 CRC-16-CCITT 的 Java 实现，跟 redis-cli CLUSTER KEYSLOT 的结果逐位对齐。为什么是 16384？心跳消息的位图 16384 bits = 2KB，正好一个以太网帧。扩展到 1000 节点每个节点 16 个槽，粒度够细。"

## Q4: 跨槽事务与 SAGA 补偿

> **演练方式**：真实 Cluster 上执行跨槽 Lua → CROSSSLOT → Hash Tag 绕过去。

### 步骤

**1) 跨槽演示**

```bash
curl 'http://localhost:8080/api/cluster/transaction/cross-slot?orderId=phone-123' | python3 -m json.tool
```

看 `sameSlot: false` 和 `result: "CROSSSLOT"`.

**2) Hash Tag**

```bash
curl 'http://localhost:8080/api/cluster/transaction/hash-tag?orderId=phone-123' | python3 -m json.tool
```

看 `sameSlot: true` 和 `risks`.

**3) SAGA 事务（用 MySQL 持久化）**

```bash
# 成功
curl -X POST 'http://localhost:8080/api/cluster/transaction/saga/execute?orderId=order-001' | python3 -m json.tool

# 失败 + 补偿
curl -X POST 'http://localhost:8080/api/cluster/transaction/saga/execute?orderId=fail-order-002' | python3 -m json.tool

# 看日志
curl 'http://localhost:8080/api/cluster/transaction/saga/log?orderId=fail-order-002' | python3 -m json.tool
```

看 `timeline` 中 `step: "COMPENSATE"`。

### 面试要诀

> "Cluster 不支持跨节点事务。Hash Tag 可以强制同槽但引入热点风险。我们线上用 SAGA 补偿：正向步骤链 + 每个步骤的反向补偿操作。库存扣减失败 → 回滚预占、恢复订单状态。每一步持久化到 MySQL，支持断点续传。"

## Q5: 热 Key / 大 Key 排查

> **演练方式**：redis-cli --bigkeys/--hotkeys 真实扫描 + Caffeine 多级缓存。

### 步骤

```bash
# 一键扫描
bash scripts/q5-scan.sh 192.168.3.100 7000
```

**如果只想跑单个命令：**

```bash
# 慢日志
redis-cli -h 192.168.3.100 -p 7000 -a 123456 SLOWLOG GET 10

# 大 Key
redis-cli -h 192.168.3.100 -p 7000 -a 123456 --bigkeys

# 内存碎片
redis-cli -h 192.168.3.100 -p 7000 -a 123456 MEMORY STATS | grep fragmentation
```

**多级缓存命中率实验：**

```bash
# 先写
curl -X POST 'http://localhost:8080/api/cluster/troubleshoot/cache/set?key=hot:item&value=99'

# 清空本地 Caffeine
curl -X POST http://localhost:8080/api/cluster/troubleshoot/cache/clear

# 第 1 次 → L2-Redis (1ms)
curl -X POST 'http://localhost:8080/api/cluster/troubleshoot/cache/get?key=hot:item'
# 第 2 次 → L1-Caffeine (<1μs)
curl -X POST 'http://localhost:8080/api/cluster/troubleshoot/cache/get?key=hot:item'

# 命中率
curl http://localhost:8080/api/cluster/troubleshoot/cache/stats
```

### 面试要诀

> "排查链路：Grafana 看 CPU → SLOWLOG 定位慢命令 → --bigkeys/--hotkeys 找问题 key → 业务代码 review。热 Key 应急：读热加 Slave + 本地 Caffeine（1μs vs 1ms），写热用队列削峰，不能直接 DEL——会缓存击穿。"

## Q7: 统一客户端工厂

> **演练方式**：真实切换 Sentinel ↔ Cluster 模式。

### 步骤

```bash
# 初始化 Sentinel
curl -X POST 'http://localhost:8080/api/factory/init?mode=SENTINEL' | python3 -m json.tool

# 执行 Redis 命令
curl 'http://localhost:8080/api/factory/execute?command=PING'

# 无中断切换到 Cluster
curl -X POST 'http://localhost:8080/api/factory/switch?mode=CLUSTER' | python3 -m json.tool
# → from: "SENTINEL", to: "CLUSTER"
# → principle: "AtomicReference 原子替换"

# 再切回来
curl -X POST 'http://localhost:8080/api/factory/switch?mode=SENTINEL' | python3 -m json.tool

# 重试 + 断路器
curl -X POST 'http://localhost:8080/api/factory/retry-test?command=PING&maxRetries=3' | python3 -m json.tool
curl http://localhost:8080/api/factory/circuit-status
```

### 面试要诀

> "我用了策略模式——RedisCommandExecutor 接口 + Sentinel/Cluster 两种实现。切换时 AtomicReference 原子替换执行器引用，旧连接的关闭延迟到缓冲期后，做到无中断。重试用指数退避 2^n×100ms，配合手写断路器——连续 5 次失败熔断 10 秒。"

## Q8: 平滑扩缩容

> **演练方式**：redis-cli --cluster reshard 真实迁移 + Spring Boot 监控。

### 步骤

**终端 1（执行迁移）：**

```bash
# 先看当前分布
curl http://localhost:8080/api/ops/reshard/slot-distribution | python3 -m json.tool

# 添加新节点（如已有 6 节点要扩到 8 节点）
redis-cli --cluster add-node 192.168.3.100:7006 192.168.3.100:7000 -a 123456
redis-cli --cluster add-node 192.168.3.100:7007 192.168.3.100:7000 -a 123456

# 开始 rebalance（每个节点 4 个 slot，慢慢来）
redis-cli --cluster rebalance 192.168.3.100:7000 \
    --cluster-weight 1=1 2=1 3=1 4=1 5=1 6=1 7=0 8=0 \
    --cluster-password 123456 \
    --cluster-pipeline 4
```

**终端 2（监控）：**

```bash
bash scripts/q8-reshard-monitor.sh
```

**观察迁移中的 slot 状态：**

```bash
curl http://localhost:8080/api/ops/reshard/migrating-slots | python3 -m json.tool
```

如果看到 `"MIGRATING"` 和 `"IMPORTING"` 状态——说明迁移正在进行中。

**迁移前大 Key 检查：**

```bash
curl http://localhost:8080/api/ops/reshard/bigkey-check | python3 -m json.tool
```

### 面试要诀

> "reshard 原理：源节点 SETSLOT MIGRATING + 目标节点 SETSLOT IMPORTING → MIGRATE 逐 key → SETSLOT NODE 全网广播。迁移期间客户端收到 ASK 重定向（临时），不是 MOVED（永久）。大 Key 是迁移杀手——500MB key 的 MIGRATE 同步操作会让主线程阻塞数秒，触发 Sentinel 误判。我线上强制要求迁移前执行 --bigkeys，超 100MB 拒迁。"
