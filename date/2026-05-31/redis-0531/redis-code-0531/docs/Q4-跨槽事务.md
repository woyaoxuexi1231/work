# Q4: 跨槽事务——Hash Tag 的诱惑与 SAGA 的务实

> 在真实 Cluster 上执行跨槽 Lua → CROSSSLOT 被拒 → Hash Tag 绕过去但看到风险 → SAGA 补偿看回滚。

---

## 启动

```bash
sudo bash scripts/redis_cluster_start.sh
mysql -h 192.168.3.100 -u root -p123456 < src/main/resources/init.sql
mvn spring-boot:run -Dspring-boot.run.profiles=cluster
```

---

## 实验 1：跨槽 Lua → CROSSSLOT 被拒

```bash
curl -X POST 'http://localhost:8080/api/q4/cross-slot-lua?orderId=phone-123' | python3 -m json.tool
```

**返回**：

```json
{
  "orderId": "phone-123",
  "keys": [
    {"key": "stock:phone-123",        "slot": 5009},
    {"key": "order:status:phone-123", "slot": 655},
    {"key": "user:purchased:phone-123","slot": 14105}
  ],
  "status": "CROSSSLOT——被拒绝",
  "error": "CROSSSLOT Keys in request don't hash to the same slot",
  "explanation": "Redis Cluster 不支持跨节点 Lua。三个 key 的 slot 不同，Lua 脚本无法原子执行。"
}
```

**你做了什么**：向真实 Cluster 发送了一个操作 3 个 key 的 Lua 脚本。Redis 检查发现三个 key 的 slot 分别是 5009、655、14105——不在同一节点，直接拒绝。

**实锤了答案里的**："Redis Cluster 本身不支持跨节点事务"——这就是 CROSSSLOT 错误的现场。

---

## 实验 2：Hash Tag → 成功但看到风险

```bash
curl -X POST 'http://localhost:8080/api/q4/hash-tag-lua?orderId=phone-123' | python3 -m json.tool
```

**返回**：

```json
{
  "orderId": "phone-123",
  "keys": [
    {"key": "stock:{phone-123}",        "slot": 7969},
    {"key": "order:status:{phone-123}", "slot": 7969},
    {"key": "user:purchased:{phone-123}","slot": 7969}
  ],
  "allSameSlot": 7969,
  "luaResult": "✓ 成功——三个 key 在 slot 7969，Lua 原子执行",
  "risks": {
    "热点风险": "订单 phone-123 的所有数据集中在一个 slot → 该 slot 所在节点可能被压垮",
    "数据倾斜": "大量使用同一 tag 的业务 key 全部挤在同一节点",
    "扩展性丧失": "无法通过加节点分摊该 slot 的压力——slot 是原子迁移最小单位"
  },
  "verdict": "Hash Tag 是'饮鸩止渴'——短期解决原子性，长期引入热点。"
}
```

**你做了什么**：给三个 key 加上 `{phone-123}` Hash Tag，强制它们落入同一个 slot 7969，Lua 脚本成功执行。但同时——`risks` 字段告诉你这背后的代价。

**实锤了答案里的**："Hash Tag 强制同槽，但会造成数据集中在一个节点，若热点很容易压垮单节点"——`risks` 的三个条目就是证据。

## 实验 3：SAGA 成功路径

```bash
curl -X POST 'http://localhost:8080/api/q4/saga/execute?orderId=order-001' | python3 -m json.tool
```

**返回**：

```json
{
  "orderId": "order-001",
  "success": true,
  "timeline": [
    {"step": "PREPARE",       "action": "预占订单号 order-001",    "redisCmd": "SET NX lock:{order-001}", "result": "SUCCESS"},
    {"step": "STOCK_CHECK",   "action": "扣减库存",               "redisCmd": "DECR stock:{order-001}",  "result": "SUCCESS"},
    {"step": "STATUS_UPDATE", "action": "更新订单状态为 PAID",     "redisCmd": "SET order:status:{...}",   "result": "SUCCESS"},
    {"step": "LIST_APPEND",   "action": "追加用户已购列表",       "redisCmd": "SADD user:purchased:{...}","result": "SUCCESS"}
  ]
}
```

四个步骤全部 SUCCESS，`success: true`。

---

## 实验 4：SAGA 失败 → 触发补偿

```bash
curl -X POST 'http://localhost:8080/api/q4/saga/execute?orderId=fail-order-002' | python3 -m json.tool
```

**返回**（注意 orderId 包含 "fail" 会触发库存扣减失败）：

```json
{
  "orderId": "fail-order-002",
  "success": false,
  "timeline": [
    {"step": "PREPARE",     "redisCmd": "SET NX lock:{...}", "result": "SUCCESS"},
    {"step": "STOCK_CHECK", "redisCmd": "DECR stock:{...}",  "result": "FAILED"},
    {"step": "COMPENSATE",  "action": "【补偿】释放预占",    "redisCmd": "DEL lock:{...}", "result": "COMPENSATED"}
  ]
}
```

**你做了什么**：STOCK_CHECK 失败 → 触发 COMPENSATE——删除 PREPARE 创建的锁。这就是 SAGA 的反向补偿。

**实锤了答案里的**："SAGA 模式——把一个跨槽的强一致事务，拆成一串单槽的原子操作，并由异步协调者负责最终一致"——`COMPENSATE` 步骤就是补偿逻辑。

---

## 实验 5：查 MySQL 补偿日志

```bash
curl 'http://localhost:8080/api/q4/saga/log?orderId=fail-order-002' | python3 -m json.tool
```

每步都持久化到 `saga_transaction` 表，面试时可以证明"我们每一步都有审计日志"。

## 三个方案对比

| 方案 | 端点 | 效果 |
|------|------|------|
| 跨槽 Lua | `/api/q4/cross-slot-lua` | ✗ CROSSSLOT 被拒 |
| Hash Tag | `/api/q4/hash-tag-lua` | ✓ Lua 成功，但 `risks` 字段警告热点/倾斜 |
| SAGA | `/api/q4/saga/execute` | ✓ 正向成功 / 失败触发 COMPENSATE |

---

## 面试话术

> "我在真实 Redis 7.2.5 Cluster 上跑过跨槽 Lua——亲眼看到 `CROSSSLOT Keys in request don't hash to the same slot`。Hash Tag 能解决原子性，但三个 key 全挤在同一 slot——我在 `/api/q4/hash-tag-lua` 的返回里标注了热点、数据倾斜、扩展性丧失三个风险。生产上我们用 SAGA 补偿：每一步是单槽原子操作（Hash Tag 保证），失败时反向回滚。MySQL 持久化每步状态，支持审计和断点续传。"
