Redis 过期键删除策略：惰性删除 vs 定期删除

Redis 作为内存数据库，为了高效清理已经设置了过期时间（TTL）的键，采用两种策略协同工作：惰性删除 和 定期删除。两者目的都是释放内存，但机制、触发时机、资源消耗完全不同。下面从核心区别展开。

📋 核心区别总览

对比维度惰性删除 (Lazy Expiration)定期删除 (Active Expiration)
触发时机客户端访问（读/写）某个键时，检查该键是否过期，若过期则立即删除。Redis 内部周期性任务（默认每秒执行10次，即每隔100ms）主动抽取一批设置了过期时间的键进行检查删除。
删除方式被动删除，只有访问到才会触发，不访问则长期占用内存。主动扫描，在后台定期抽查过期键，无需等待客户端请求。
CPU 消耗低：仅在访问发生时检查，不额外占用后台资源。相对较高：周期性遍历部分过期键字典，消耗CPU时间片。
内存释放时效性差：如果过期键一直不被访问，会成为“僵尸键”长期滞留内存，造成内存浪费。较好：即便键不被访问，定期任务也会逐步清理，避免内存无限膨胀。
是否会遗漏过期键会大量遗漏（不访问就不删）。也会遗漏（因为每次只抽样部分键），但配合惰性删除可最大程度减少残留。
典型场景问题可能导致内存持续增长，有 OOM 风险。可能短暂增加CPU负载，但可控。

⚙️ 1. 惰性删除（Lazy Expiration） —— “用到才删”

  • 执行位置：所有读写命令（如 GET、SET、HGET、LPUSH 等）执行前，Redis 会检查目标键是否设置过期时间并已过期。
  • 删除动作：若过期，则先执行删除，再返回 key 不存在（或者对写操作视为该 key 不存在）。
  • 优点：CPU友好，只在必要时工作，不会额外增加系统负担。
  • 缺点：内存不友好。例如一个设置了 TTL 的 key 在过期后再也没有被访问，它将继续占用内存，造成“内存泄漏”假象。
  • 示例：
    > SET user:100 "john" EX 60 (60秒过期)
    > GET user:100 → 60秒内返回 "john"；60秒后再次 GET，Redis 发现已过期，删除键，返回 (nil)。
💡 重要：惰性删除保证了过期键不会通过访问“复活”，但无法主动清理死数据。因此必须配合定期删除。

⏲️ 2. 定期删除（Active Expiration） —— “后台主动扫描”

  • 执行机制：Redis 的 server.hz 配置（默认10，即每秒10次）触发 activeExpireCycle 函数。该函数每次在指定时间预算内，遍历各个数据库（db）的过期哈希表，随机抽取一批带有过期时间的键，检查并删除过期的键。
  • 扫描策略：每次选取一定数量（如20个）的过期字典槽位中的键，删除其中过期的键。如果本次过期键比例过高，会继续扫描，直至时间用完或遍历完。
  • 优点：弥补惰性删除不足，主动释放内存，避免大量过期键堆积。
  • 缺点：占用少量CPU；如果过期键很多，一次定期删除可能耗时稍长，但Redis做了严格的时间限制（默认不超过25%的CPU时间片）。
  • 配置相关：
    - hz 参数（1~500，默认10）影响定期删除的执行频率。调高会更快清理但增加CPU负载。
    - active-expire-effort (Redis 6.0+) 调节扫描积极度。
⚙️ 关键点：定期删除不是全量扫描所有过期键，而是随机抽样 + 自适应循环，防止阻塞主线程。在合理配置下，既能控制CPU开销，又能有效减少过期键残留。

🔄 3. 两者如何协同工作？—— 完整的过期键清理生态

Redis 实际采用 “惰性删除为主，定期删除为辅” 的组合策略：

  • ✅ 定期删除主动清理大部分过期键，保证内存不会无限增长。
  • ✅ 惰性删除作为兜底机制，清理那些被定期删除遗漏且后续又被访问到的过期键。
  • ✅ 如果过期键一直没人访问且定期删除每次抽样都恰好没抽到，那么该键依然可能滞留。为此 Redis 还提供了 内存淘汰策略（maxmemory-policy） 作为最后防线（如 volatile-lru、allkeys-lru 等）。

举例：一个 Redis 实例中有 10w 个设置了 TTL 的 key，大部分未被继续访问。定期任务每秒10次扫描，每次大约清理几百个 key，全部清理完毕需要一定时间。在此期间，如果客户端访问到某个还未被扫描到但已过期的 key，惰性删除会立刻删除，避免返回脏数据。两者结合达到 CPU/内存的平衡。

⚠️ 4. 极端情况：为什么还会出现过期键堆积？

  • 定期删除频率不足：如果 hz 太小（例如1），扫描次数少，新过期键产生速度快于清理速度，导致积压。
  • 每次扫描抽样数量有限：默认扫描20个字典槽位的key，若某个数据库有过亿级别的过期键，抽样无法覆盖全部。
  • 懒汉式缺陷：所有过期键永远不被客户端访问，且定期删除每次都“错过”，理论上存在极低概率长期残留，但 Redis 会通过多次迭代逐步覆盖大多数。
🔁 优化建议：适当调高 hz（比如20~50），或结合主动触发 SCAN + 手动删除，但一般默认配置足够。更严格的场景可增加内存淘汰策略兜底。

📌 5. 对比总结：面试可直接表达的核心区别

  • 触发方式不同：惰性删除 → 客户端访问触发；定期删除 → 服务器定时任务触发。
  • 删除时机不同：前者“即时检查”但被动；后者“主动扫描”但非实时。
  • 资源取舍：惰性删除节省CPU浪费内存；定期删除用适量CPU换内存释放。
  • 遗漏风险：惰性删除遗漏大部分过期键；定期删除减少遗漏但不能100%清除。
  • 最终依赖：两者必须共存，且最后还有内存淘汰策略（maxmemory）做保底。

🧠 补充：与内存淘汰策略的区别（容易混淆）

很多面试者会把“过期删除”和“内存淘汰”混淆，这里强调：

  • 惰性删除 + 定期删除 专门处理已经达到过期时间的键（TTL到期）。
  • 内存淘汰策略 (maxmemory-policy) 是在 Redis 内存使用超过 maxmemory 阈值时，根据某种算法（LRU/LFU/随机/ttl）强制删除任意键（无论是否过期），目的是防止 OOM。两者是完全不同的机制。

因此回答“Redis如何清理过期键”时，只需阐述惰性删除与定期删除即可；如果问“内存不足怎么处理”再引出淘汰策略。

📖 一句话记忆口诀

“惰性访问时清理，节约CPU；定期主动扫库，防止内存堆积；两者结合是王道，最后淘汰策略兜底。”
