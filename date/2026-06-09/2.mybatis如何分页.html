MyBatis 分页机制深度剖析 面试精准回答

从逻辑分页到物理分页｜PageHelper原理｜最佳实践
📌 核心结论

MyBatis 本身并没有提供完整的内置物理分页功能，但开发中有两种主流方案：逻辑分页（RowBounds） 和 物理分页（手写limit / 分页插件）。 实际生产环境 99% 的场景都推荐物理分页，避免全表查询导致性能雪崩。下面我会从原理、实现、源码层面和面试考察点系统拆解。

💡 面试加分技巧：主动对比两种方式的性能差异，并深入分页插件拦截器机制，展示你对 MyBatis 扩展点的掌握。
🧠 一、逻辑分页 · RowBounds（内存分页）

MyBatis 提供了 RowBounds 对象，通过设置 offset 和 limit 实现分页效果。 但本质是 先查询出所有符合条件的数据，再根据 offset 和 limit 在内存中截取。对于大表或高并发场景，会产生严重的性能瓶颈。

// 1. 在 Mapper 接口中正常定义查询方法，不需要额外参数 List selectAllUsers(); // 2. 调用时传入 RowBounds (offset = 起始行, limit = 每页条数) RowBounds rowBounds = new RowBounds((pageNum - 1) * pageSize, pageSize); List userList = sqlSession.selectList("com.demo.mapper.UserMapper.selectAllUsers", null, rowBounds); // 或者使用 Mapper 方式：(需要把 RowBounds 作为参数) // userMapper.selectAllUsers(rowBounds);
⚠️ 重大缺陷：例如一张表有 1000w 数据，查询第 100 页，RowBounds 依然会拉取 1000w 条到 JVM 内存再进行截取，导致 IO 和内存开销极大。因此 仅适用于数据量极小（如配置表）或快速原型阶段。
⚡ 二、物理分页 · 真正的数据库分页

物理分页通过数据库专属的 LIMIT / OFFSET、ROWNUM 或 TOP 等语法，在 SQL 执行层面就只返回需要的数据。 性能远优于逻辑分页。MyBatis 中实现物理分页常见两种手段：

🔨 方式1：手动拼接 LIMIT（原生 SQL 灵活）
// Mapper 接口 List selectUserPage(@Param("offset") int offset, @Param("limit") int limit); // XML 映射文件 <select id="selectUserPage" resultType="User"> SELECT * FROM user WHERE status = 1 ORDER BY id DESC LIMIT #{offset}, #{limit} </select> // 调用方式 int pageNum = 2, pageSize = 10; int offset = (pageNum - 1) * pageSize; List users = userMapper.selectUserPage(offset, pageSize);

优点：完全掌控 SQL，数据库方言精准；缺点：每个分页方法都需要手动计算 offset，代码冗余。 适合简单场景或需复杂联表优化的场景。

🚀 方式2：分页插件 PageHelper（最流行、无侵入）

PageHelper 是国内使用最广泛的 MyBatis 分页插件，基于 MyBatis 拦截器（Interceptor）自动改写 SQL，加上 LIMIT 方言。 完全零侵入，只需要在查询前调用 PageHelper.startPage 即可。

// 1. 引入 Maven 依赖 <dependency> <groupId>com.github.pagehelper</groupId> <artifactId>pagehelper</artifactId> <version>最新版(如 6.1.0)</version> </dependency> // 2. 配置 mybatis-config.xml 添加 plugin <plugins> <plugin interceptor="com.github.pagehelper.PageInterceptor"> <property name="helperDialect" value="mysql"/> <property name="reasonable" value="true"/> </plugin> </plugins> // 3. Service 层使用 PageHelper.startPage(pageNum, pageSize); // 紧跟着的查询会被分页 List userList = userMapper.selectAll(); PageInfo<User> pageInfo = new PageInfo<>(userList); // PageInfo 里封装了总记录数、当前页数据、页码、总页数等所有分页信息
✅ PageHelper 优势：自动识别数据库方言（MySQL/PostgreSQL/Oracle/SQL Server等），支持多表关联排序，并且分页参数通过 ThreadLocal 传递，保证线程安全，对业务代码几乎无侵入。

此外，MyBatis-Plus 也内置了强大的物理分页（selectPage方法），原理类似，基于拦截器注入 limit。不过面试常考原生 MyBatis 场景时，PageHelper 是必提的经典方案。

🧩 三、深入底层：分页插件实现原理（面试亮点）

PageHelper 为什么能无感改写 SQL 并添加 limit？ 关键在于 MyBatis 提供了强大的 Interceptor 接口， 允许开发者在 Executor 或 StatementHandler 执行 SQL 之前进行拦截改造。

// 自定义分页插件核心伪代码实现思路 @Intercepts({ @Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}) }) public class MyPageInterceptor implements Interceptor { @Override public Object intercept(Invocation invocation) throws Throwable { // 1. 从当前线程上下文中获取分页参数（ThreadLocal） PageParam page = PageContext.get(); if (page == null) return invocation.proceed(); // 2. 拿到 MappedStatement 和原始 SQL MappedStatement ms = (MappedStatement) invocation.getArgs()[0]; BoundSql boundSql = ms.getBoundSql(parameter); String originalSql = boundSql.getSql(); // 3. 拼接 limit 方言：例如 MySQL 加 LIMIT offset, limit String pageSql = originalSql + " LIMIT " + page.getOffset() + "," + page.getPageSize(); // 4. 利用反射替换 SQL，并继续执行 ... 改写后通过反射设置新的 BoundSql return invocation.proceed(); } }

⚙️ 核心流程：分页参数存入 ThreadLocal → 拦截器拦截查询 → 动态修改 SQL 增加分页语法 → 执行并返回分页数据 → 最后清理 ThreadLocal。 这也是 PageHelper 要求 startPage 后必须紧跟 Mapper 方法调用的原因。面试中如果能阐述拦截器签名以及 ThreadLocal 线程变量处理，会是巨大加分项。

📘 知识延伸：MyBatis 拦截器可拦截 Executor、ParameterHandler、ResultSetHandler、StatementHandler 四大组件。分页插件通常拦截 Executor 的 query 方法，从而实现对 SQL 的动态改写。
📊 四、逻辑分页 vs 物理分页 · 决战对比表
对比维度 RowBounds (逻辑分页) 物理分页 (LIMIT / PageHelper)
实现原理 查询全部结果，内存截取 SQL 执行时直接从数据库获取分页范围数据
性能 ❌ 极差，数据量越大越慢，占用JVM内存 ✅ 优秀，只传输需要的数据，利用索引
适用场景 仅适合非常小的表（几百行内） 所有生产环境，高并发、海量数据必备
复杂度 简单，无需修改 SQL 需手动offset/limit 或 集成插件
数据库兼容性 无差异（内存操作） 需适配不同数据库分页方言，PageHelper 自动处理
额外功能 只实现截取，无总条数 通常能同时获得总记录数、导航页码等
🧭 五、生产级最佳实践 & 避坑指南
✅ 推荐组合
  • 中小型项目/快速开发 → 直接使用 PageHelper 插件，完整分页对象 + 高开发效率。
  • 复杂动态SQL/极致优化 → 手动在 Mapper.xml 中使用 LIMIT，并注意 #{} vs ${} 防注入。
  • 微服务 + 多数据源 → 配置 PageHelper 时指定 dialect 属性，避免自动识别失效。
  • 大数据量深分页优化 → 避免传统 offset 过大导致的性能下降，使用 “游标/Seek Method” (比如 where id > lastId limit 10) 配合索引。
⚠️ 常见面试追问
❓ 追问1： “如果我用 RowBounds 配合分页插件会怎样？”
→ 分页插件会优先处理物理分页，RowBounds 参数可能会干扰，通常插件会忽略 RowBounds 或用插件自己的逻辑，不推荐混用。

❓ 追问2： “分页时如何获取总记录数？”
→ PageHelper 内部会额外发送一条 count 查询（自动优化，可配置），借助 PageInfo 获得 total。手动分页需要单独写 select count 语句。

❓ 追问3： “PageHelper 在多线程环境下线程安全问题？”
→ 用 ThreadLocal 存储分页参数，且每次执行完查询后会自动 clear，确保下一个请求不会串页。但是注意 startPage 后面如果不跟着查询，可能导致内存泄漏（一般推荐在 try/finally 或框架AOP控制）。
🛠️ 六、进阶：手写一个简化版分页插件

为了展示源码级理解，我们可以在面试中给出自定义分页插件的骨架：实现 Interceptor 接口，借助 Plugin.wrap 生成代理。 核心步骤：

@Intercepts(@Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})) public class SimplePageInterceptor implements Interceptor { @Override public Object intercept(Invocation invocation) throws Throwable { // 获取分页参数（从自定义上下文获取） PageParam page = PageParamContext.get(); if (page == null) return invocation.proceed(); MappedStatement ms = (MappedStatement) invocation.getArgs()[0]; Object parameter = invocation.getArgs()[1]; BoundSql boundSql = ms.getBoundSql(parameter); String sql = boundSql.getSql(); // 拼接分页 SQL (只做示例, 简单拼接) String newSql = sql + " LIMIT " + page.getOffset() + "," + page.getPageSize(); // 通过反射修改 BoundSql (实际需要创建新的MappedStatement或使用BoundSql复制) // ... 省略具体反射细节，面试阐述思路即可 return invocation.proceed(); } }

实际工程中，强烈建议直接使用成熟的 PageHelper，但手写插件展示能让你在面试中脱颖而出，证明你对 MyBatis 底层执行链和动态代理的熟悉程度。

🎯 面试回答总结（可直接背诵）

🎤 精炼回答版本（30秒～1分钟）：

“MyBatis 分页分为逻辑分页和物理分页。逻辑分页用 RowBounds，先查所有数据再内存截取，性能极差，不推荐生产使用。
物理分页是直接通过数据库 SQL 限制范围，有两种实现：1）手写 LIMIT，灵活直观；2）使用 PageHelper 分页插件，它基于 MyBatis 拦截器，自动改写 SQL 并支持多种数据库，是团队主流选择。 其核心原理是使用 ThreadLocal 存储分页参数，拦截 Executor 的 query 方法动态拼接方言，并且会自动查询总数。实际项目中应始终采用物理分页，注意深分页优化策略。”
