Redis 内存淘汰策略（Maxmemory Policy）

当 Redis 内存使用量达到 maxmemory 上限时，内存淘汰策略被触发，用于腾出空间容纳新数据。这与“惰性删除/定期删除”（针对已过期键）完全不同，淘汰策略可以删除任意键（无论是否设置 TTL）。

📌 1. 触发条件

  • 配置 maxmemory：设置 Redis 最大可用内存（例如 maxmemory 4gb）。若不设置，64位系统默认为物理内存上限，可能导致 OOM。
  • 写命令执行时：当尝试写入新数据（SET、LPUSH、SADD 等）导致内存超过 maxmemory，Redis 会按照配置的策略淘汰部分键，然后执行写入。如果淘汰后内存仍然不够，则写入失败（返回错误）。
  • 注意：读命令不触发淘汰。
💡 重要：内存淘汰是被动触发，仅在内存达到上限且需要写入时发生。可通过 INFO memory 查看 maxmemory 和当前内存使用。

⚙️ 2. 八种淘汰策略（Redis 4.0+ 支持）

通过 maxmemory-policy 配置项设置，常用策略如下：

策略名称适用数据范围淘汰算法/行为适用场景
noeviction所有键不淘汰任何键，当内存满时，写命令返回错误（如 OOM 错误）不允许丢数据的场景（但需保证内存绝对充足）
allkeys-lru所有键从所有键中淘汰最近最少使用（LRU）的键通用热点数据缓存，推荐
volatile-lru仅设置了过期时间的键从设置了 TTL 的键中淘汰 LRU 的键既需要持久化又需要淘汰过期数据的混合场景
allkeys-lfu (4.0+)所有键淘汰最不经常使用（LFU，访问频率最低）的键访问频率差异明显的缓存，LRU 容易被冷数据冲击
volatile-lfu (4.0+)仅有过期时间的键从设置了 TTL 的键中淘汰 LFU 的键同上，但仅限过期键集合
allkeys-random所有键随机淘汰缓存项访问分布均匀，无热点
volatile-random仅有过期时间的键从设置了 TTL 的键中随机淘汰配合 expire 使用的简单场景
volatile-ttl仅有过期时间的键淘汰剩余 TTL 最短（最快过期）的键希望优先删除即将过期的数据
🔍 LRU vs LFU 快速区分：LRU 关注最近一次访问时间，LFU 关注访问频率。LFU 能更好避免缓存污染（比如一次性读取大量冷数据挤掉热点）。

📖 3. 淘汰策略与过期删除的区别（高频面试题）

  • 触发维度不同：惰性/定期删除处理的是已到期的键；内存淘汰策略处理的是内存超限时，无论键是否到期。
  • 是否强制：过期删除是清理“合法过期”的键；淘汰策略是“强行驱逐”，可能删除未过期的键。
  • 执行顺序：通常先尝试惰性/定期删除来释放内存，如果释放后依然超过 maxmemory，则触发淘汰策略。
  • 配置独立：过期删除不可禁用（有 TTL 就会自动清理），淘汰策略可配置。
⚠️ 常见误区：很多人认为 Redis 满了会自动删除过期键，实际上过期键并不一定被立即删除（惰性删除不活跃），导致内存爆满时淘汰策略会删除任何键，包括未过期的热点数据。因此合理设置 maxmemory 与策略非常重要。

🔧 4. 配置示例 & 动态修改

配置文件 redis.conf：

maxmemory 2gb
maxmemory-policy allkeys-lru

运行时动态修改（无需重启）：

CONFIG SET maxmemory 4gb
CONFIG SET maxmemory-policy volatile-lfu

查看当前策略：

CONFIG GET maxmemory-policy

📊 5. 如何选择合适的内存淘汰策略？

  • 纯缓存场景（可丢数据）：allkeys-lru 或 allkeys-lfu。推荐 allkeys-lru，最通用。
  • 必须保留部分固定数据 + 缓存过期数据：volatile-lru 或 volatile-ttl。只对设置了 TTL 的键进行淘汰。
  • 禁止删除任何键：noeviction，但需搭配监控和充足内存，否则写入全失败。
  • 访问模式非常随机且无热点：allkeys-random 性能最佳。
  • 热点数据频繁访问，防止一次性扫描污染：LFU 策略（allkeys-lfu）优于 LRU。

⚠️ 6. 实战注意事项

  • maxmemory 不要设置过大：建议留出 10%~20% 内存给系统与碎片，避免 OOM Killer。
  • 预估写入吞吐：淘汰策略执行时会阻塞主线程（但 Redis 内部淘汰算法优化为近似 LRU/LFU，性能较高）。
  • 与持久化（RDB/AOF）的关系：内存淘汰会删除键，这些删除操作也会被记录到 AOF 文件，并在从库同步。所以从库也可能因为主库淘汰而丢失数据。
  • 监控淘汰情况：使用 INFO stats 查看 evicted_keys 计数器，如果频繁增长说明内存压力大。
🧠 面试回答模板：“Redis 内存淘汰策略在 maxmemory 达到上限时触发，通过 maxmemory-policy 配置。常用 allkeys-lru 淘汰最少使用的键。它与过期删除不同，过期删除只针对已 TTL 到期的键，而淘汰策略可删除任何键。选择策略需根据业务是否能容忍数据丢失以及访问模式来定。必要时还可调整 LFU 和采样精度参数。”

📚 扩展：近似 LRU 与精确 LRU

Redis 的 LRU 并非精确淘汰全局最久未访问的键，而是抽样淘汰（默认采样5个键，淘汰其中 idle 时间最长的）。
可通过 maxmemory-samples 配置提高精度（消耗更多 CPU）。LFU 也采用类似抽样方法。

CONFIG SET maxmemory-samples 10

这样在内存淘汰时更精确，但性能稍降。
