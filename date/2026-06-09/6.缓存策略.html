缓存策略：数据一致性 · 删除失败 · 缓存击穿

面试官：先更新数据库还是先更新缓存？如何保证一致性？删除缓存失败了怎么办？高并发下会不会出现缓存击穿？
本文逐一解答。

📌 1. 先更新数据库 vs 先更新缓存？

主流方案：Cache Aside Pattern（旁路缓存） —— 读：先读缓存，未命中则读数据库并写入缓存；写：先更新数据库，再删除缓存（而非更新缓存）。

策略风险结论
先更新缓存，后更新数据库更新缓存成功但DB失败 → 缓存与DB永久不一致❌ 不采用
先更新数据库，后更新缓存并发更新可能导致脏数据（如线程A写DB，线程B写DB并更新缓存，A再更新缓存覆盖）⚠️ 有竞态，不推荐
先更新数据库，后删除缓存删除失败时可能短暂不一致，但可通过重试/订阅binlog修复✅ 推荐
先删除缓存，后更新数据库删除后、更新前有读请求 → 缓存未命中，读取旧DB并重建缓存，导致脏数据⚠️ 需要延迟双删等补偿，复杂
✅ 最佳实践：先更新数据库，再删除缓存。因为数据库是唯一可靠数据源，缓存只是加速。删除操作比更新缓存更安全（避免并发写脏）。

🧹 2. 删除缓存失败怎么办？

即使采用“先DB后删缓存”，仍可能因网络闪断、Redis宕机等原因导致删除失败。此时缓存中仍是旧数据，造成短期不一致。解决方案：

  • ① 重试机制（同步+异步）：将删除失败的key放入本地内存队列或Redis重试队列，后台线程异步重试（如指数退避）。
  • ② 订阅数据库变更日志（Binlog）：使用 Canal / Debezium 监听MySQL binlog，当数据变更时，由消费者异步删除缓存（保证最终一致）。
  • ③ 设置合理的过期时间：即使删除失败，缓存也会在TTL后失效，保证最终一致（兜底）。
  • ④ 提供主动修复接口：运维或监控发现不一致时，手动调用清理接口。
💡 典型工程实现：更新DB后，立即发送MQ消息（如RocketMQ），消费者负责删除缓存；结合死信队列处理多次失败。或使用Canal + 监听binlog，无侵入保证最终一致。

🔥 3. 高并发下缓存击穿 & 如何应对？

缓存击穿：某个热点key在缓存中失效（过期或刚被删除），大量并发请求同时穿透到数据库，造成DB压力骤增。

  • 互斥锁（分布式锁）：只允许一个线程去查DB并重建缓存，其他线程等待或快速失败。常用Redis setnx或Redisson。
  • 逻辑过期（永远不设置物理过期，而是维护一个逻辑过期时间）：当发现逻辑过期时，异步去更新缓存，旧数据仍可返回，避免DB压力。
  • 布隆过滤器：针对不存在的key，提前过滤，避免无效穿透。但击穿是存在的key刚好失效，布隆过滤器无效，需用锁或异步重建。
  • 提前“续命”：对热点key后台线程定时刷新，防止失效瞬间高并发。
🧠 经典互斥锁代码示例（基于Redis）：
String value = redis.get(key);
if (value == null) {
  String lockKey = "lock:" + key;
  if (redis.setnx(lockKey, "1", 3)) { // 获取锁
    try {
      value = db.query(key);
      redis.setex(key, ttl, value);
    } finally { redis.del(lockKey); }
  } else {
    Thread.sleep(50); // 稍等重试
    return redis.get(key);
  }
}

注意区分：缓存穿透（查询不存在key）和雪崩（大量key同时失效）。击穿特指单个热点key失效。

📐 4. 整体一致性保证方案汇总

  • 最强一致性（低并发场景）：先更新DB，再同步删除缓存，若删除失败则回滚DB事务（需支持分布式事务，不推荐）。
  • 平衡方案（主流）：
    - 更新DB → 发送MQ → 消费者删除缓存（保证最终一致）
    - 或订阅binlog异步删除
    - 同时给缓存设置较短过期时间（如30秒），兜底不一致窗口。
  • 高并发优化：结合击穿防护（互斥锁）+ 异步删除 + 监控告警。
✅ 面试回答模板：
“我们采用先更新数据库，再删除缓存策略，避免脏数据。删除失败时，通过消息队列重试或binlog监听保证最终一致性。针对缓存击穿，对热点key使用分布式锁控制DB查询并发，或者采用逻辑过期异步更新方案。”
