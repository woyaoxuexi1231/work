线程池面试连环问：自定义参数 & 拒绝策略实战

面试官：你们系统用的什么线程池，自定义的还是JDK自带的？
回答：使用的是java.util.concurrent.ThreadPoolExecutor自定义线程池，并未使用Executors工具类的快捷方法（如newCachedThreadPool、newFixedThreadPool），因为这些预定义线程池存在队列无界或线程数无限制等风险，生产环境必须自己精细控制参数。

📌 1. 核心线程数 & 最大线程数 设置依据

没有“银弹”公式，必须结合业务类型、硬件资源、压测数据动态调整。以下是通用方法论：

🔹 区分任务类型

  • CPU密集型（加密、计算、图像处理）：核心线程数 ≈ CPU核心数 + 1，最大线程数一般与核心线程数持平，避免过多上下文切换。
  • IO密集型（数据库查询、RPC调用、文件读写）：核心线程数 ≈ CPU核心数 * 2（或更高），因为线程大部分时间在等待IO。最大线程数可以设置为核心线程数的2~4倍，取决于IO等待时长比例。
  • 混合型：根据压测找出最优值，或拆分为不同线程池隔离。

🔹 具体设置流程（我们系统的实践）

  • 基线计算：服务器配置（例如 8核32G）。核心线程数初始设为：IO密集型 → 16，CPU密集型 → 9。
  • 压测验证：使用JMeter或全链路压测，逐步增加并发，观察CPU利用率、系统负载、平均响应时间、拒绝策略触发次数。
  • 调优原则：
    - 核心线程数：保证系统在低负载时能快速响应，通常设为压测得出的最佳吞吐量对应的线程数。
    - 最大线程数：应对突发流量，通常设为核心线程数的2~4倍（结合队列容量）。但不能超过操作系统限制及内存承受能力，同时要预留一部分资源给其他应用。
  • 最终落地（例如某API业务，IO密集型）：corePoolSize = 16，maximumPoolSize = 64，经过压测后调整为core=24，max=80（前提是数据库连接池、下游系统能抗住）。
💡 关键点：线程数并非越大越好。过多线程会导致频繁上下文切换（每次约1~3微秒），反而降低吞吐量。需要结合CPU负载和任务平均等待/计算比来调整。

📦 2. 任务队列满了怎么处理？

队列的选择和满后的行为由阻塞队列类型 + 拒绝策略共同决定。以下是常见队列及其满后行为：

队列类型特点队列满后的行为
LinkedBlockingQueue (无界)默认Integer.MAX_VALUE，几乎不会满不会触发拒绝策略，但可能导致内存溢出（任务无限堆积）→ 不推荐生产使用
ArrayBlockingQueue (有界)固定容量，队列满后新任务尝试入队失败触发拒绝策略（由RejectedExecutionHandler处理）
SynchronousQueue不存储任务，每个插入必须等待一个移除若无空闲线程则立即创建新线程直到最大线程数，满后触发拒绝策略
LinkedBlockingQueue (有界，自定义容量)最常用，例如容量=2000队列满后新任务会尝试创建新线程（如果当前线程数 < maximumPoolSize），否则触发拒绝策略

我们系统的做法：使用有界的LinkedBlockingQueue（容量通常设置为2000~5000），并配合拒绝策略。原因：
- 无界队列风险太大（可能积压海量任务导致OOM）。
- 有界队列可以触发拒绝策略，让我们有机会快速失败或降级。
- 队列容量根据压测确定：
  ▪ 容量太小 → 频繁触发拒绝，浪费线程资源。
  ▪ 容量太大 → 任务积压延迟高，且内存占用高。
- 队列满后，ThreadPoolExecutor的执行逻辑：corePoolSize满 → 入队 → 队列满 → 新建线程（直至max） → 队列仍满 → 执行拒绝策略。

⚙️ 举例：core=20, max=80, 队列容量=1000。当20个核心线程都在忙，新任务进入队列；队列满后，会创建第21~80个线程来处理任务；若80个线程也全忙且队列已满，则后续任务走拒绝策略。

🚫 3. 拒绝策略 (RejectedExecutionHandler) 怎么做的？

JDK 内置四种拒绝策略，我们根据业务重要性选择了组合方案：

策略行为适用场景
AbortPolicy (默认)直接抛出 RejectedExecutionException 异常关键业务，不允许丢失任务，希望上游快速感知失败
CallerRunsPolicy由调用线程（提交任务的线程）自己执行该任务降低新任务提交速度，避免系统过载，但会阻塞调用方
DiscardPolicy静默丢弃该任务，不抛异常可容忍少量丢失的非核心业务（如日志上报）
DiscardOldestPolicy丢弃队列头部（等待最久）的任务，然后重试提交当前任务希望淘汰旧任务，保留新任务（如实时更新场景）

🔹 我们的实践与思考

  • 核心业务（订单处理、支付回调）：使用AbortPolicy + 统一异常捕获，并在上层做重试或降级（如写入消息队列，异步补偿）。绝对不能静默丢弃或让调用线程执行（避免阻塞Tomcat容器线程）。
  • 非核心业务（日志分析、统计上报）：使用DiscardPolicy，并记录监控指标（丢弃次数），允许高峰期丢失部分不敏感数据。
  • 需要削峰填谷但不想丢失任何任务：使用CallerRunsPolicy，但同时需要监控调用线程的耗时，防止雪崩。通常配合有界队列+限流器使用。
  • 自定义拒绝策略：对于极端重要场景，我们扩展了 RejectedExecutionHandler，将拒绝的任务持久化到本地磁盘或 Redis 队列，随后由单独补偿线程处理。
🧠 面试官追问：“CallerRunsPolicy 有什么潜在问题？”
答：如果调用线程是 Web 容器线程（如 Tomcat），该线程被阻塞执行耗时任务，会导致该请求响应变慢，甚至拖垮整个容器线程池，造成其他请求无法处理。因此一般不建议在 Web 请求处理链路中直接使用，除非任务非常轻量或专门用于限流。

📋 4. 完整自定义线程池配置示例（代码级）

实际项目中，我们会将线程池封装成 Spring Bean，并提供监控（通过 Micrometer 暴露指标）。

@Bean("bizExecutor")
public ThreadPoolExecutor bizExecutor() {
  int core = Runtime.getRuntime().availableProcessors() * 2; // IO密集 ≈ 16 (8核)
  int max = core * 4; // 64
  int queueCapacity = 2000;
  long keepAliveTime = 60L; // 空闲线程存活时间
  TimeUnit unit = TimeUnit.SECONDS;
  BlockingQueue<Runnable> workQueue = new LinkedBlockingQueue<>(queueCapacity);
  ThreadFactory factory = new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build();
  RejectedExecutionHandler handler = new ThreadPoolExecutor.AbortPolicy();
  return new ThreadPoolExecutor(core, max, keepAliveTime, unit, workQueue, factory, handler);
}

动态调优：通过 setCorePoolSize() 和 setMaximumPoolSize() 结合配置中心（Nacos/Apollo）实现运行时调整，无需重启。

🧪 5. 任务队列满 + 拒绝策略的实战案例

场景：双11大促，订单创建接口瞬时QPS飙升到平时5倍。线程池核心32，最大128，队列容量1000。瞬间打满后触发 AbortPolicy。为避免直接抛异常导致订单失败，我们做了如下设计：

  • 提交任务时捕获 RejectedExecutionException，将订单信息落入本地消息表 + 定时补偿。
  • 监控系统告警，立即触发弹性扩容（自动增加线程池最大线程数到200，并加大队列）。
  • 同时启用动态限流（Sentinel），控制进入线程池的流量，确保下游DB不会被压垮。

最终效果：少量订单短暂延迟，但无一丢失，系统平稳。

🎯 面试总结（快速回答模板）

✅ “我们使用自定义ThreadPoolExecutor，根据任务类型（IO/CPU）结合压测设置核心和最大线程数。使用有界队列来限制积压。队列满后会触发拒绝策略，核心业务用AbortPolicy并配合补偿机制，非核心用DiscardPolicy。同时我们通过监控和动态调优来应对突发流量。”
