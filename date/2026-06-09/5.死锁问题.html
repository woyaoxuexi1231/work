死锁：排查 → 预防 → 动态顺序锁

面试连环问：有没有遇到过死锁？怎么排查？jstat 是做什么的？如何预防？动态场景下如何保证加锁顺序？本文逐一回答。

🔍 1. 死锁排查实战（命令行 + 堆栈分析）

遇到过：典型场景如两个线程各自持有锁A、锁B，互相等待对方释放。排查步骤如下：

  • ① 定位进程：jps -l 或 ps -ef | grep java 获取 Java 进程PID。
  • ② 导出线程堆栈：jstack <pid> 或 jstack -l <pid>，控制台会直接打印所有线程状态。
    → 查找 "deadlock" 关键字，JVM 会自动检测死锁并输出线程信息、锁对象地址、以及哪两个线程互相等待。
  • ③ 分析锁持有情况：jstack 输出中会显示 waiting to lock ... 和 locked ...，可直接定位代码行号。
  • ④ 使用 jconsole 或 VisualVM：图形化界面，可以点击“线程”选项卡 → “检测死锁”。
⚠️ 关于 jstat：面试官可能故意混淆。jstat 是用于监控 JVM GC 和内存的工具（如 jstat -gc <pid>），不能直接排查死锁。死锁排查用 jstack、jconsole。区分清楚是加分点。

⚠️ 2. 导致死锁的四个必要条件 & 常见场景

  • 互斥：资源一次只能被一个线程占用。
  • 持有并等待：线程持有至少一个资源，同时等待其他资源。
  • 不可剥夺：资源只能由持有者主动释放。
  • 循环等待：线程间形成环形依赖。

常见产生死锁的代码场景：

  • 嵌套 synchronized 或 ReentrantLock，锁顺序不一致（如线程A先锁a再锁b，线程B先锁b再锁a）。
  • 数据库连接池 + 事务：一个连接持有锁，另一个事务等待，同时互相占用表锁或行锁。
  • 线程池任务提交自己（例如线程池内任务等待另一个任务结果，而另一个任务在队列中）。
  • 使用 CountDownLatch 或 CyclicBarrier 设置不当导致互相等待。

🛡️ 3. 预防死锁的通用策略

  • 破坏“请求与保持”：一次性申请所有资源（但降低并发度）。
  • 破坏“不可剥夺”：使用 ReentrantLock 的 tryLock(long timeout) 方法，超时则释放已持有的锁。
  • 破坏“循环等待”：按固定顺序获取锁（最常用）。
  • 使用更高级的并发工具：如 ConcurrentHashMap、BlockingQueue、AtomicReference 避免显式锁。
  • 静态代码分析工具：SpotBugs、SonarQube 检测潜在死锁。
✅ 重点：面试中被问到“两个线程互相等待锁怎么办？” → 最直接的解决方案是调整锁顺序，或者用 tryLock 超时 + 回退重试。

🔐 4. 按顺序获取锁：静态顺序 vs 动态顺序

静态顺序：例如两个锁对象 lockA、lockB，规定所有线程必须先锁A再锁B。这很简单，通过编码约束即可。

动态情况：锁的获取顺序依赖于运行时参数（如需要根据账户ID、订单ID等动态决定先锁哪个对象）。如何保证顺序？

✅ 动态顺序锁方案：基于哈希值或系统一致性排序

  • 方案1：System.identityHashCode 比较。对需要加锁的对象（如两个账户），比较其内存地址哈希值，总是先锁哈希值小的对象。
  • 方案2：使用对象的唯一ID（如数据库主键），总是按 ID 升序获取锁，可以完全避免循环等待。
  • 方案3：使用锁排序器（LockOrder），将需要锁的对象放入 TreeSet，按自然顺序排序后再依次加锁。
  • 方案4：使用 ReentrantLock 配合 tryLock 加超时，不固定顺序，但允许超时后释放已持有的锁并重试（类似死锁恢复机制）。
📌 动态顺序经典示例（转账业务）：
void transfer(Account from, Account to, int amount) {
  // 根据哈希值确定锁顺序，避免死锁
  Account first = System.identityHashCode(from) < System.identityHashCode(to) ? from : to;
  Account second = first == from ? to : from;
  synchronized(first) {
    synchronized(second) {
      // 执行转账
    }
  }
}
即使传入的 from/to 顺序不同，全局加锁顺序一致，打破循环等待。

🚀 5. 超时重试：应对无法固定顺序的复杂场景

如果动态排序无法应用（比如需要同时锁三个互相关联的缓存对象，且哈希冲突极少但理论存在），可以采用 ReentrantLock.tryLock 模式：

boolean acquired = false;
while (!acquired) {
  if (lockA.tryLock(10, TimeUnit.MILLISECONDS)) {
    try {
      if (lockB.tryLock(10, TimeUnit.MILLISECONDS)) {
        try {
          // 业务逻辑
          acquired = true; break;
        } finally { lockB.unlock(); }
      }
    } finally { if(!acquired) lockA.unlock(); }
  }
  Thread.sleep(5); // 退避重试
}

这种方式牺牲一定性能，但能彻底避免死锁。可用于分布式锁、数据库锁重试等场景。

🎤 面试核心答案总结（快速背诵版）

Q1：死锁排查？ → jstack 获取线程堆栈，查找 deadlock 字样，分析锁持有和等待关系。
Q2：预防死锁？ → 破坏四个必要条件之一，最常用固定顺序加锁，或 tryLock 超时重试。
Q3：两线程互相等待怎么办？ → 调整锁顺序，或使用 tryLock 超时 + 释放已持有锁。
Q4：动态情况下保证顺序？ → 根据对象哈希值或唯一ID排序，确保全局统一加锁顺序；极端情况使用超时重试。

额外说明：jstat 不是死锁排查工具，而是监控 GC 和内存，避免面试官挖坑。
