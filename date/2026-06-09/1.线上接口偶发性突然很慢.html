接口偶发性慢
怎么排查？

从监控告警到根因定位，系统性掌握偶发性接口延迟的排查方法论与工具链

⚠️ 偶发性问题的核心难点

偶发性慢接口最大的挑战是难以复现、现场易丢失。等你发现问题再去排查时，异常现场可能已经消失，传统的断点调试和手动 jstack 很难精准定位。

难以复现 现场丢失 无规律 可能涉及多服务

排查框架：先定界，再定位

偶发性问题最怕"大海捞针"，必须先缩小范围

第一步：界定问题范围

是单个接口还是全部接口？

单个接口慢 → 代码逻辑或该接口依赖的外部服务问题
全部接口慢 → 服务器资源瓶颈（CPU/内存/网络）

是偶发还是持续？

偶发 → 资源竞争、锁竞争、GC、网络抖动
持续 → 代码缺陷、配置错误、容量不足

是否有规律？

整点/高峰期慢 → 定时任务、流量突增
随机慢 → 资源竞争、GC、外部依赖抖动

是否特定参数触发？

特定用户/订单/参数慢 → 数据倾斜、大对象、慢SQL

📊 监控数据辅助判断

  • 查看 APM（SkyWalking/Prometheus）接口 P99/P95 延迟趋势
  • 对比不同实例的响应时间，确认是否单点问题
  • 查看错误率是否同步上升（超时/5xx）
  • 查看 QPS 曲线，确认是否与流量峰值相关

第二步：分布式链路追踪（最重要）

核心手段

偶发性问题最难的是"不知道慢在哪里"。链路追踪（SkyWalking / Zipkin / Jaeger）能还原一次请求的完整调用链，精准定位耗时环节。

TraceId 定位慢请求

在 APM 中找到耗时异常的 Trace，查看每个 Span 的耗时分布，一眼看出是数据库慢、RPC 慢、还是本地计算慢

识别耗时环节

链路图会标注每个调用的耗时，常见瓶颈：MySQL 查询、Redis 命令、HTTP/RPC 调用、消息队列

跨服务串联

微服务架构下，一个接口可能调用多个下游服务。链路追踪能穿透服务边界，定位是 A 服务慢还是 B 服务慢

# SkyWalking 查询示例
# 1. 进入 SkyWalking UI
# 2. 选择目标服务和时间段
# 3. 按耗时排序，找到 > P99 的 Trace
# 4. 查看 Trace 详情，分析每个 Span 的耗时

# 关键观察点：
# - 是否有某个 Span 耗时占比 > 80%？
# - 是否存在串行调用可以改为并行？
# - 是否有重复查询（N+1 问题）？

第三步：日志分析

# ELK 中搜索慢请求日志
curl -X GET "localhost:9200/logstash-*/_search" -H 'Content-Type: application/json' -d'
{
  "query": {
    "bool": {
      "must": [
        { "match": { "service": "order-service" }},
        { "range": { "responseTime": { "gt": 2000 }}}
      ]
    }
  },
  "sort": [{ "timestamp": "desc" }],
  "size": 100
}'

# 或者直接用 grep
grep "cost=\"[2-9][0-9]" application.log | tail -100

🔍 搜索关键词

Timeout、Slow、cost=、elapsed、exception、error

📊 关联 TraceId

日志中打印 TraceId，通过 TraceId 串联全链路日志，定位具体报错点

⏱️ 时间窗口对齐

将慢请求时间与 GC 日志、数据库慢查询日志、中间件监控对齐

第四步：JVM 与线程分析

🧵 线程 Dump 分析

# 导出线程栈
jstack PID > thread_dump.txt

# 查找 BLOCKED 线程
grep -A 20 "BLOCKED" thread_dump.txt

# 查找 WAITING 线程（可能等待锁或资源）
grep -A 20 "WAITING" thread_dump.txt | head -100

关注 BLOCKED（锁竞争）、WAITING/TIMED_WAITING（等待连接池、等待锁）状态的线程

🗑️ GC 影响排查

# 查看 GC 情况
jstat -gcutil PID 5000

# 如果开启了 GC 日志
grep "Full GC" gc.log | tail -20

# 关注：FGC 次数是否突增、GC 耗时是否 > 1s

Full GC 或长耗时 GC 会导致应用 STW，接口响应出现"毛刺"

🔥 Arthas 在线诊断（推荐）

# 1. 监控接口实时耗时
trace com.example.OrderService getOrder '#cost>1000' -n 5

# 2. 查看方法内部每行耗时（精准定位慢代码）
stack com.example.OrderService getOrder

# 3. 查看线程状态
thread -n 20 | grep TIMED_WAITING

# 4. 查看线程阻塞情况
thread -b  # 找出阻塞其他线程的罪魁祸首

# 5. 火焰图分析（需要 async-profiler）
profiler start --event cpu
profiler stop --file /tmp/flame.html

第五步：外部依赖排查

MySQL

  • • SHOW PROCESSLIST 看慢查询
  • • 慢查询日志分析
  • • 检查锁等待、死锁
  • • 连接池是否耗尽

Redis

  • • redis-cli --latency 测延迟
  • • 排查大 Key / 热 Key
  • • 检查连接池是否耗尽
  • • 集群节点是否故障

RPC / HTTP

  • • 下游服务是否超时
  • • 网络延迟 / 丢包
  • • 下游服务是否雪崩
  • • 重试策略是否合理

MQ / 定时任务

  • • 消息是否堆积
  • • 消费速度是否跟得上
  • • 定时任务是否抢资源
  • • 整点批量任务影响

经典案例与根因

面试中最常考的具体场景

案例一：数据库慢查询

最高频

现象：接口偶发响应时间从 50ms 飙到 3-5s，无规律，用户反馈"有时候卡"

🔍 排查过程

  1. SkyWalking 发现慢请求中 MySQL 查询耗时占 90%
  2. 慢查询日志定位到一条未走索引的 SQL
  3. EXPLAIN 分析：type=ALL，rows=5000000
  4. 发现该 SQL 的 WHERE 条件字段无索引

💡 根因分析

偶发性是因为：该 SQL 的查询条件对应的字段值分布不均匀，某些值命中大量数据时触发全表扫描。平时查询快是因为命中了索引字段的常见值。

-- 问题 SQL（status 字段无索引，且某些 status 值数据量极大）
SELECT * FROM orders WHERE status = 'PENDING' AND create_time > '2024-01-01';

-- EXPLAIN 结果：
-- type: ALL（全表扫描）
-- rows: 5000000
-- Extra: Using where

-- 修复方案：
-- 1. 添加复合索引
ALTER TABLE orders ADD INDEX idx_status_create_time(status, create_time);

-- 2. 或优化查询逻辑，增加时间范围限制
SELECT * FROM orders
WHERE status = 'PENDING'
  AND create_time BETWEEN '2024-06-01' AND '2024-06-08'
LIMIT 100;

案例二：锁竞争 / 死锁

高频

现象：订单扣库存接口偶发超时，高峰期更明显，jstack 发现大量线程 BLOCKED

🔍 排查过程

  1. jstack 导出线程栈，发现 50+ 线程 BLOCKED
  2. 所有 BLOCKED 线程都在等待同一把锁
  3. 锁的持有者在执行数据库更新操作
  4. 发现 synchronized 锁住了整个方法

💡 根因分析

synchronized 锁粒度太大，锁住了整个扣库存方法。高并发下，一个线程持有锁执行 DB 操作，其他线程全部排队等待，导致偶发性超时。

// 问题代码：锁粒度太大
public synchronized void deductStock(Long productId, int qty) {
    // 1. 查询库存（DB 操作，耗时 ~50ms）
    Stock stock = stockDao.selectById(productId);
    // 2. 校验库存
    if (stock.getQty() < qty) throw new BizException("库存不足");
    // 3. 扣减库存（DB 操作，耗时 ~50ms）
    stockDao.deduct(productId, qty);
    // 4. 记录日志（DB 操作，耗时 ~20ms）
    logDao.insert(...);
    // 整个方法被锁住，高并发下排队严重
}

// 修复方案：缩小锁粒度 + 使用乐观锁
public void deductStock(Long productId, int qty) {
    // 1. 先查询（无锁）
    Stock stock = stockDao.selectById(productId);
    if (stock.getQty() < qty) throw new BizException("库存不足");

    // 2. 使用乐观锁更新（数据库层面 CAS）
    int affected = stockDao.deductWithVersion(productId, qty, stock.getVersion());
    if (affected == 0) throw new BizException("并发冲突，请重试");
}

// SQL：利用 version 字段实现乐观锁
UPDATE stock SET qty = qty - #{qty}, version = version + 1
WHERE product_id = #{productId} AND version = #{version} AND qty >= #{qty};

案例三：连接池耗尽

高频

现象：接口偶发超时，不是每次都慢，监控显示连接池使用率偶尔飙到 100%

🔍 排查过程

  1. 监控发现连接池 Active Connections 偶尔达到 Max
  2. jstack 发现大量线程 WAITING on connection pool
  3. 检查代码发现某些接口未关闭连接（或连接泄漏）
  4. 还发现一个接口内部循环查询，每次查询都新建连接

💡 根因分析

偶发性是因为：连接池有 20 个连接，平时够用。但当某个大接口同时占用多个连接（或连接未释放），其他请求获取不到连接只能等待，导致偶发超时。

// 问题代码 1：连接未释放（Spring 中事务未正确关闭）
@Transactional
public void processOrder() {
    // 长时间持有连接，中间还调用了外部 HTTP 接口
    orderDao.update(...);  // 获取连接
    httpClient.call(...);   // 阻塞 2s，连接不释放！
    orderDao.insert(...);    // 复用同一连接
}

// 问题代码 2：循环内每次新建连接
public List getOrders(List ids) {
    List result = new ArrayList<>();
    for (Long id : ids) {
        // 每次循环都走数据库！N+1 问题
        result.add(orderDao.selectById(id));
    }
    return result;
}

// 修复方案 1：拆分事务，减少连接持有时间
public void processOrder() {
    orderDao.update(...);  // 事务 1，快速提交
    httpClient.call(...);   // 外部调用
    orderDao.insert(...);    // 事务 2
}

// 修复方案 2：批量查询
public List getOrders(List ids) {
    return orderDao.selectBatchIds(ids);  // 一条 SQL
}

案例四：GC 停顿（STW）

中频

现象：接口偶发延迟尖刺（从 50ms 跳到 2-3s），无规律，所有接口都受影响

🔍 排查过程

  1. SkyWalking 发现慢请求没有特定慢 Span
  2. 所有 Span 耗时都正常，但总耗时异常长
  3. jstat 发现 FGC 次数突增，且 GC 耗时 > 2s
  4. jmap -histo 发现某个缓存类实例数暴涨

💡 根因分析

偶发性是因为：Full GC 不是每次请求都触发，而是当老年代积累到一定阈值才触发。GC 期间应用 STW，所有线程暂停，导致接口出现"毛刺"。

// 问题代码：本地缓存无界增长
private static final Map cache = new HashMap<>();

public Object getData(String key) {
    if (!cache.containsKey(key)) {
        cache.put(key, loadFromDB(key));  // 只进不出！
    }
    return cache.get(key);
}

// 修复方案 1：使用有界缓存（Guava / Caffeine）
LoadingCache cache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(30, TimeUnit.MINUTES)
    .build(key -> loadFromDB(key));

// 修复方案 2：JVM 参数调优
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-Xms4g -Xmx4g  # 固定堆大小，避免动态扩展

案例五：下游服务超时 / 网络抖动

中频

现象：接口偶发超时，SkyWalking 显示下游 RPC 调用耗时异常，但下游服务监控正常

🔍 排查过程

  1. SkyWalking 显示下游服务 B 的某个接口偶发 5s+
  2. 但服务 B 的监控显示该接口平均 20ms
  3. 检查网络监控，发现偶发丢包 / 延迟尖刺
  4. 发现服务 A 的 HTTP 连接池配置超时时间为 5s

💡 根因分析

偶发性是因为：网络抖动导致个别请求 RTT 增加，加上连接池超时设置不合理（太长导致阻塞，太短导致频繁超时），偶发性的网络问题被放大。

// 问题配置：超时设置不合理
RestTemplate restTemplate = new RestTemplate();
// 默认超时很长，下游抖动时一直等待

// 修复方案 1：合理设置超时 + 熔断
@FeignClient(name = "order-service",
    fallbackFactory = OrderFallbackFactory.class,
    configuration = FeignConfig.class)
public interface OrderClient {
    @GetMapping("/orders/{id}")
    @Retryable(maxAttempts = 2, backoff = @Backoff(delay = 100))
    Order getOrder(@PathVariable Long id);
}

// 修复方案 2：Sentinel / Hystrix 熔断降级
// 当下游接口错误率 > 50% 或 RT > 1s 时，自动熔断，走降级逻辑

// 修复方案 3：异步调用 + 超时控制
CompletableFuture future = CompletableFuture.supplyAsync(
    () -> orderClient.getOrder(id),
    executor
).orTimeout(500, TimeUnit.MILLISECONDS)
 .exceptionally(ex -> getDefaultOrder());

面试回答要点

1

先定界再定位：先确认是单个接口还是全部接口、是偶发还是持续、是否有规律（整点/高峰/随机）。偶发性问题最怕"大海捞针"，必须先缩小范围。

2

链路追踪是核心：SkyWalking / Zipkin 等 APM 工具是排查偶发性问题的核心手段，通过 TraceId 还原完整调用链，精准定位耗时环节（数据库 / Redis / RPC / 本地计算）。

3

高频根因 TOP5：① 数据库慢查询（无索引/数据倾斜）② 锁竞争（synchronized 粒度太大）③ 连接池耗尽（连接泄漏/N+1查询）④ GC 停顿（Full GC STW）⑤ 下游服务超时/网络抖动。

4

工具链组合：APM（链路追踪）→ 日志（ELK 关联 TraceId）→ jstack（线程状态）→ Arthas（trace/stack 精准定位）→ 外部依赖监控（MySQL/Redis/网络）。
