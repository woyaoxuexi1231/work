这是一个典型的**并发 + 幂等**组合问题，回答时要展示出你既能抓住幂等的核心实现，又能讲清楚如何避免并发下的竞态条件。

以下是你可以参考的回答思路：

---

**1. 先点明核心矛盾**
> “幂等性设计要求重复请求返回一致结果且不产生副作用，而线程安全问题主要在于：当多个相同请求并发到达时，如何保证**只有一个被真正执行**，其余直接返回已有结果。”

**2. 针对不同场景，给出线程安全的实现方案**

- **场景一：写操作，有唯一业务单号（如创建订单、支付）**
  > 利用数据库的**唯一索引 + 插入**来做原子防重。先插入一条幂等记录（业务单号作为唯一键），利用数据库的行锁保证只有一个事务能插入成功；失败的线程捕获唯一键冲突异常，直接查询已有结果返回。  
  > 如果怕插入性能消耗大，可以用 `INSERT ... ON DUPLICATE KEY UPDATE` 但保证 UPDATE 部分是空操作或状态不变。

- **场景二：有状态流转（如订单状态机）**
  > 使用**乐观锁**，在更新时带上版本号或前置状态条件，例如 `UPDATE ... SET status = 'PAID' WHERE id = ? AND status = 'UNPAID'`。数据库的行锁会串行化执行，只有一个符合条件能更新成功，其他请求走到这里返回幂等成功。

- **场景三：无需持久化的轻量防重（如防止表单重复提交）**
  > 引入**分布式锁**，比如用 Redis 的 `SETNX` + 过期时间。在处理前尝试获取锁，获取成功的执行逻辑，失败的则直接返回“处理中/请勿重复提交”。注意锁的粒度是请求唯一标识（如 token），并要考虑锁续期和原子释放。  
  > 更好的方案是结合**请求 Token 机制**：客户端先获取一次 Token，提交时校验并删除 Token；利用 Redis 的 `DEL` 返回删除个数来判断是否为首次处理，保证原子性。

**3. 强调选型原则**
- 如果已经有数据库，优先用唯一索引/乐观锁，持久化且天然线程安全。
- 如果是纯缓存防重，用 Redis 原子命令，但要考虑缓存穿透和宕机丢失（可结合业务允许的小概率重复）。
- 避免“先查询再插入/更新”的模式，那会引发 TOCTOU 问题，在高并发下必然不线程安全。

**4. 最后总结一句话（可作收尾）**
> “线程安全的幂等设计本质上就是要把**判断是否处理过**和**执行业务逻辑**合并成一个原子操作，要么靠数据库的 ACID（行锁+唯一约束），要么靠中间件的原子命令，绝不让检查和执行之间存在竞态窗口。”
