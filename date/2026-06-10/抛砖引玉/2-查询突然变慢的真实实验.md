行，那我给你一个**可以直接复制执行**的实验，不需要你自己改 SQL。

目标：

```text
正常查询：10~50ms
异常查询：3~20秒
```

即使是 SSD，也能看出明显差异。

# 环境要求

建议：

```text
MySQL 8.x
内存 8G+
```

查看 Buffer Pool：

```sql
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
```

如果大于：

```text
4G
```

建议临时改小（测试库）：

```ini
innodb_buffer_pool_size=512M
```

重启 MySQL。

# 第二步：创建大表

这里故意把每行做成约 4KB。

这样 100 万行就是约 4GB 数据。

```sql
DROP TABLE IF EXISTS user_big;

CREATE TABLE user_big(
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    phone VARCHAR(20),
    status INT,
    create_time DATETIME,

    remark1 VARCHAR(1000),
    remark2 VARCHAR(1000),
    remark3 VARCHAR(1000),
    remark4 VARCHAR(1000),

    INDEX idx_phone(phone),
    INDEX idx_status(status)
) ENGINE=InnoDB;
```

# 第三步：插入100万行

```sql
INSERT INTO user_big(
    phone,
    status,
    create_time,
    remark1,
    remark2,
    remark3,
    remark4

SELECT
CONCAT('138',LPAD(n,8,'0')),

REPEAT(MD5(n),30),
REPEAT(MD5(n+1),30),
REPEAT(MD5(n+2),30),
REPEAT(MD5(n+3),30)

FROM numbers;
```

# 实验3：回表地狱

这里特别容易复现。

---

状态分布：

```text
status=1
999000条

status=0
1000条
```

---

查询：

```sql
SELECT *
FROM user_big
WHERE status=1;
```

执行计划：

```text
走 idx_status
```

但是：

```text
999000次回表
```

---

执行：

```sql
EXPLAIN ANALYZE
SELECT *
FROM user_big
WHERE status=1;
```

很多机器：

```text
3~15秒
```

# 实验4：执行计划变化

把 status 改成一半一半。

```sql
UPDATE user_big
SET status=0
WHERE id<=500000;
```

收集统计信息：

```sql
ANALYZE TABLE user_big;
```

---

再次执行：

```sql
EXPLAIN ANALYZE
SELECT *
FROM user_big
WHERE status=0;
```

你会发现：

```text
有些机器走索引

有些机器直接全表扫描
```

这就是线上经常发生的：

```text
昨天20ms
今天5秒
```

执行计划突变。

# 实验5：排序灾难（最容易看到差异）

建表：

```sql
DROP TABLE IF EXISTS article_big;

CREATE TABLE article_big(
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    category_id INT,
    title VARCHAR(200),
    content LONGTEXT,
    create_time DATETIME
);
```

---

插入100万条：

```sql
INSERT INTO article_big(
    category_id,
    title,
    content,
    create_time
)
SELECT
n % 100,
CONCAT('title_',n),
REPEAT(MD5(n),100),
NOW() - INTERVAL (n % 365) DAY
FROM numbers;
```

---

查询：

```sql
SELECT *
FROM article_big
ORDER BY create_time DESC
LIMIT 20;
```

执行计划：

```sql
EXPLAIN ANALYZE
SELECT *
FROM article_big
ORDER BY create_time DESC
LIMIT 20;
```

应该看到：

```text
Using filesort
```

耗时：

```text
几秒
```

---

加索引：

```sql
CREATE INDEX idx_create_time
ON article_big(create_time);
```

再次执行：

```sql
SELECT *
FROM article_big
ORDER BY create_time DESC
LIMIT 20;
```

耗时：

```text
几十毫秒
```

# 实验6：锁等待（100%复现）

准备：

```sql
DROP TABLE IF EXISTS account;

CREATE TABLE account(
    id BIGINT PRIMARY KEY,
    balance DECIMAL(10,2)
);

INSERT INTO account VALUES
(1,10000),
(2,10000);
```

窗口1：

```sql
BEGIN;

UPDATE account
SET balance=balance-100
WHERE id=1;
```

不要提交。

---

窗口2：

```sql
SELECT *
FROM account
WHERE id=1
FOR UPDATE;
```

直接卡住。

---

查看：

```sql
SHOW PROCESSLIST;
```

状态：

```text
Waiting for lock
```

---

这是线上最真实的：

```text
SQL执行10ms

等待锁5秒
```

---

如果你只有一台普通开发机，我建议先做：

1. 排序灾难（Using filesort）
2. 回表地狱（status=1 占 99.9%）
3. 锁等待

这三个实验最容易稳定复现，而且和线上实际故障最接近。
