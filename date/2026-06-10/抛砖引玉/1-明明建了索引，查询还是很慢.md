# 主问题



这是一个很经典的面试题，面试官其实是在考察你对索引原理、执行计划和数据库优化实战的理解。可以从“索引可能没用到”“用到了但效率低”“有其它瓶颈”这几个层面来答。下面模拟一段比较完整的回答，你可以根据自己的风格调整。

---

**面试回答参考：**

“这个问题我一般会从三个方向去排查：**索引本身有没有生效、索引生效了为什么还慢、以及是不是其他资源瓶颈导致的慢**。

**第一，索引失效，没走索引。**
明明建了索引，但 SQL 的写法可能导致索引失效。常见场景有：
- 在索引列上使用函数或运算，比如 `WHERE DATE(create_time) = '2026-06-12'`，这会绕过索引，应该改成范围查询。
- 隐式类型转换，比如 `WHERE phone = 13800138000`，如果 phone 是字符串类型，引号没加就会导致全表扫描。
- 前导模糊查询 `LIKE '%abc'`。
- 使用 `!=`、`NOT IN` 或者 OR 条件中有一边没索引，优化器容易放弃索引。
- 联合索引没遵循最左前缀原则，比如建了 `(a, b, c)`，查询却用 `WHERE b = 2`，索引就匹配不上。

**第二，索引用上了，但效果不好。**
最典型的是**回表代价太高**。二级索引找到主键后，要去聚簇索引查完整行。如果查询大量数据，随机 IO 比顺序扫描还慢，优化器可能直接选全表扫描。这时候如果建一个**覆盖索引**，让索引包含所有需要的列，避免回表，性能会明显提升。
还有索引**区分度低**的情况，比如性别、状态这类字段，索引过滤性差，优化器可能认为还不如全表扫描快。另外如果没及时更新统计信息或存在大量索引碎片，优化器也可能选错执行计划。

**第三，SQL 和业务本身的问题。**
- `SELECT *` 取了大量不需要的字段和行，即便走索引，数据传输和网络开销也很大。
- 深分页 `LIMIT 1000000, 10`，数据库需要扫描并丢弃前面的百万行，再取10行，越往后越慢。这种可以用“延迟关联”或基于主键的游标分页优化。
- 大表 JOIN、子查询没写好，导致临时表和文件排序，也很吃性能。

**第四，数据库层面的锁、并发和硬件瓶颈。**
- 查询被其他事务的锁阻塞，或者表锁/MDL 锁竞争。
- buffer pool 太小，热点数据没法全部缓存，大量磁盘读。
- 高并发下 CPU、磁盘 IOPS 被打满。

**总结来说，遇到这个问题，我第一步就是 `EXPLAIN` 看执行计划，确认 type、key、rows、Extra。**
如果 key 为 NULL，就是索引失效；如果 key 有值但 rows 很大，再看 Extra 里有没有 `Using where; Using index` 或 `Using filesort`。根据这些信息再去定位属于上面哪一类问题，再对症优化。”

---

这样回答既有体系，也展示了你具备从原理到排查的完整思路。如果想再出彩一点，可以主动提一句：“我可以根据具体执行计划举一个实际优化案例”，这会让面试官觉得你实战经验很足。



# 追问


这几个追问非常到位，正好是排查问题的核心。我按这个顺序来回答：

---

**如何确认走没走索引？**

最直接的办法就是用 `EXPLAIN` 或 `EXPLAIN ANALYZE`（部分数据库支持）看执行计划。比如 `EXPLAIN SELECT ...`，数据库会告诉你它打算怎么执行这条 SQL。在 MySQL 里，重点就看 `key` 字段是不是 NULL——如果是 NULL，基本就是没走索引。

另外还可以打开 `SET profiling = 1` 看实际执行时间和资源消耗，或者用 `SHOW STATUS LIKE 'Handler_read%'` 看 `Handler_read_rnd_next` 这类全表扫描计数器有没有异常增加。线上慢查询日志 `slow_query_log` 里也有 `Rows_examined` 和 `Rows_sent`，如果扫描行数远大于返回行数，即便用了索引，也可能是在“低效”地扫索引。

---

**EXPLAIN 怎么看？**

拿到 `EXPLAIN` 的输出，我会先看这几个关键字段，优先级从高到低：

- **type**：访问类型，是核心指标。从好到差大致是：
  `system` > `const` > `eq_ref` > `ref` > `range` > `index` > `ALL`  
  如果出现 `ALL`（全表扫描）或 `index`（全索引扫描但没过滤），就得警惕了。
- **key**：实际使用的索引名。为 NULL 就是没走索引。
- **rows**：优化器估算的需要扫描的行数。这个值越小越好，如果很大，就算走了索引，效率也可能不行。
- **Extra**：额外信息，非常关键。常见“坏”信号：
    - `Using filesort`：排序没用到索引，需要文件排序，很耗资源。
    - `Using temporary`：用了临时表，常见于 `GROUP BY`、`DISTINCT` 或 UNION。
    - `Using where; Using index`：是覆盖索引，没回表，这是好信号。
    - `Using index condition`：用了索引下推（ICP），也是一种优化。
- **possible_keys** 和 **key_len**：前者是“可以用的索引”，后者是实际用的索引占的字节数，可以帮助判断联合索引用到了哪几列。

如果有多个表关联，还要看 **table** 的顺序，驱动表和被驱动表的连接类型。

---

**possible_keys 和 key 的区别？**

`possible_keys` 是优化器在查询中**可能**选用的索引列表，是“候选者”。  
`key` 是优化器最终**实际选择**使用的索引，是“当选者”。

它们不一致的几种常见情况：
- possible_keys 里有索引，但 key 为 NULL —— 优化器认为全表扫描更划算（比如数据量小、区分度低）。
- possible_keys 里列了多个索引，key 只选了一个，优化器根据统计信息选了成本最低的那个。
- 连 possible_keys 都没有，说明 SQL 条件列上根本没索引可用。

所以出现 possible_keys 不为空但 key 为空，就是我们常说的“有索引但没用上”，需要重点排查。

---

**为什么有索引却不用？**

你追问到了现象的本质——这背后是优化器在计算成本后做的决定。我按出现频率从高到低说几个原因：

1. **回表代价太高，不如全表扫描**  
   二级索引找到主键后要回表，如果查询范围很大（比如要扫 30% 以上的数据），大量随机 IO 可能比顺序全表扫描还慢，优化器就会选 `ALL` 而不用索引。这时可以建覆盖索引来规避回表。

2. **统计信息不准确**  
   优化器基于表的统计信息（如 `innodb_stats_persistent`）估算 `rows` 和代价。如果表频繁大批量变更但没及时 `ANALYZE TABLE`，统计信息失真，优化器可能误判。

3. **索引区分度太低**  
   比如 `gender` 只有 0/1 两种值，通过索引过滤后还要扫一半数据，优化器会认为效率还不如直接扫表。

4. **隐式类型转换或索引列参与计算**  
   比如 `WHERE id = '123'`（id 是整数但传了字符串）在某些数据库中也可能导致索引失效，更常见的是 `WHERE DATE(create_time) = ...` 直接让索引列上用了函数，索引匹配不上。

5. **查询条件组合不满足最左前缀**  
   联合索引 `(a, b, c)`，只用 `b = ?` 和 `c = ?` 可能走不了索引，因为跳过了第一列。

6. **查询返回大部分列（SELECT \*）导致索引扫描成本变大**  
   如果索引很大且需要取全量字段，优化器评估后可能选全表扫描。

7. **SQL 中有 `ORDER BY` 或 `GROUP BY`，但索引顺序不一致**  
   优化器可能为了避免 `Using filesort` 而放弃过滤用的索引，去选择能够直接排序的索引；如果两个都没有，它可能会直接全表扫描再排序。

**实务上怎么验证？**  
可以用 `FORCE INDEX(索引名)` 强制走索引做对比测试，再用 `EXPLAIN` 看强制前后的 `rows` 和 `Extra` 差异，这样就能判断优化器放弃索引是否真的不划算。如果强制索引后查询确实更快，就说明统计信息或成本模型有问题，可以从优化参数或 SQL 写法上调整。

---

这样回答完，面试官通常会感受到你不只是会背“索引失效场景”，而是能走通从现象到执行计划再到底层原因的一整条链路。





# 什么情况下主动放弃索引

面试官这问是在深挖“有索引但不用”的根本原因，本质上就是**优化器基于成本估算，认为走索引的代价 ≥ 全表扫描**。可以从几个典型场景来回答，既讲原理，也讲验证方法。

---

**1. 二级索引回表成本过高**  
这是最常见的原因。二级索引找到主键后，要回表查聚簇索引拿完整行，这是随机IO。当扫描的数据量占比很大（比如超过全表的 20%~30%），大量随机IO的成本可能超过顺序扫描整张表，优化器就会直接 `type=ALL`。  
**例子**：`SELECT * FROM orders WHERE status = '已完成'`，如果90%的订单都是已完成，索引过滤性几乎为0，回表代价巨大。  
**验证**：`EXPLAIN` 看到 `possible_keys` 有值但 `key=NULL`，且 `rows` 很大。可以 `FORCE INDEX` 强制走索引，对比执行时间，往往会发现强制后反而更慢。

---

**2. 统计信息不准确**  
优化器依赖表的基数统计（如 `innodb_stats_persistent`）。如果表频繁大批量增删改后没有及时 `ANALYZE TABLE`，采样数据会失真，优化器可能误判。比如实际数据分布已变化，但统计信息仍认为某个值占比很大，导致优化器放弃索引。  
**验证**：`SHOW INDEX FROM table_name` 看 `Cardinality`，对比真实 `SELECT COUNT(DISTINCT col)` 的值。如果差距大，执行 `ANALYZE TABLE` 后再看 `EXPLAIN` 是否走索引。

---

**3. 索引列区分度极低**  
假设为性别、布尔字段建索引，优化器发现走索引后还需回表，且过滤掉的行数很少，不如直接扫描。  
**例子**：`WHERE is_deleted=0`，绝大部分数据都没删除，索引过滤性极差。  
**验证**：计算选择性 `SELECT COUNT(DISTINCT col)/COUNT(*)`，如果接近 0，即使提示强制索引，效率也很低。

---

**4. 索引列参与了函数或运算（导致索引降级，但优化器也可能直接放弃）**  
虽然严格来说是索引失效，但优化器判断后可能直接选全表扫描，因为即使走了索引（如部分函数索引可被利用），成本也可能更高。  
**例子**：`WHERE YEAR(create_time)=2026`，即使有的数据库支持表达式索引，优化器评估后可能觉得索引扫描范围大，直接全表扫描。

---

**5. 排序与过滤的索引冲突**  
当 SQL 同时有 `WHERE` 和 `ORDER BY`，优化器会评估两种成本：  
- 用一个索引过滤，然后排序（可能出现 `Using filesort`）  
- 用另一个索引避免排序，但过滤时需回表或扫更多数据  
如果避免排序的代价过高，优化器可能两个索引都不用，直接全表扫描再排序。  
**验证**：`EXPLAIN` 的 `Extra` 同时出现 `Using where; Using filesort`，且 `key` 可能不是最佳过滤索引。

---

**6. 查询返回绝大多数列（SELECT \* 且无覆盖索引）**  
索引里不包含 SELECT 的字段，必须回表。如果优化器估算需要回表的数据行非常多，它会倾向于全表扫描连续读取，而不在索引和聚簇索引之间随机跳跃。  
**例子**：表有 100 个字段，查询 `SELECT * FROM big_table WHERE create_time > '2026-01-01'`，即使索引用上，大量回表也会让性能骤降。

---

**7. 数据量太小（小表全表扫描更优）**  
几十到几百行的表，随机IO开销可能高于顺序扫，优化器直接全表扫。

---

**8. 优化器开关与成本模型设置**  
MySQL 的 `optimizer_switch` 可以控制是否启用某些索引优化，比如 `index_merge`、`mrr` 等。如果关闭了相关选项，或 `eq_range_index_dive_limit` 等参数使得优化器不深入评估，可能错误放弃索引。  
**验证**：检查 `SELECT @@optimizer_switch;` 和 `innodb_stats_persistent_sample_pages` 等，可能需要调参。

---

**总结落地**：  
一句话回答就是：**“优化器通过比较走索引和全表扫描的估算 I/O + CPU 成本来决定。当它认为回表、随机读或额外排序的代价超过顺序扫描时，就会主动放弃索引。核心是用 `EXPLAIN` 看 `type` 和 `rows`，并用 `FORCE INDEX` 做对照测试，再用 `SHOW STATUS` 和统计信息分析具体原因。”**

这样回答，既讲清了内部的决策逻辑，也给出了完整的问题定位闭环，面试官会认可你从原理到实战的深度。
