这个问题很典型，面试官想考察的是对**性能突然恶化**的应急排查能力，而不只是常规优化。

我会这样回答：

---

“如果线上查询突然从 100ms 变成 5 秒，说明大概率**不是 SQL 本身写错导致的慢性问题，而是某个环境或数据状态发生了突变**。我会立刻按以下顺序排查：

**第一步，确认影响范围。**
- 是所有调用都变慢，还是特定用户/参数变慢？用监控看接口耗时分布，或者 `SHOW PROCESSLIST` 抓当前执行的 SQL 和耗时。
- 如果只是个别参数变慢，可能是数据倾斜（如某个用户数据量暴增）。

**第二步，看执行计划是否变化。**  
这是最常见的原因：优化器可能因为统计信息变化，突然放弃了原来好用的索引，改走全表扫描。
- 立刻用 `EXPLAIN` 看当前执行计划，重点看 `type` 是不是从 `ref/range` 变成了 `ALL` 或 `index`，以及 `rows` 是否暴增。
- 如果当时没有保存历史计划，可以查 `performance_schema` 或 `sys` 库的 `statements_with_runtimes_in_95th_percentile`，或者之前慢查询日志里正常时的计划做对比。
- 如果确认是执行计划变差，**紧急方案**是加 `FORCE INDEX` 强制走原来的索引并快速发版，然后查原因。

**第三步，检查统计信息是否失效。**  
执行计划突变常常是因为统计信息过时或被错误更新。
- 查看最近有没有大量增删改操作（比如批量导入），有没有自动触发了 `ANALYZE TABLE`，或者 `innodb_stats_auto_recalc` 在变更超过 10% 的行后更新了统计信息，但采样不准确。
- 执行 `SHOW INDEX FROM table` 对比基数值是否合理，必要时 `ANALYZE TABLE` 修复。同时检查 `innodb_stats_persistent_sample_pages` 是否太小导致统计不准。

**第四步，排查锁和资源争用。**
- 用 `SHOW ENGINE INNODB STATUS` 或 `sys.innodb_lock_waits` 看是否有锁等待，尤其其他事务持有表锁或行锁阻塞了这条 SQL。
- 检查连接数是否打满、CPU/IO 是否突然飙高，可能其他业务在跑大查询或备份，导致整体性能下降，但看起来是这个查询变慢。
- 查看 buffer pool 命中率：`SHOW STATUS LIKE 'Innodb_buffer_pool_read_requests'` 和 `Innodb_buffer_pool_reads`，如果命中率陡降，说明热点数据被换出，查询大量读盘。

**第五步，检查数据量和索引碎片。**
- 数据量突然增长（比如日志表没及时清理）会导致索引树变高，即使走索引扫描也要更多 IO。
- 索引碎片化严重时，`SHOW TABLE STATUS` 看 `Data_free` 异常大，可能需要 `OPTIMIZE TABLE`（注意锁表风险）。

**第六步，从优化器成本模型找细节。**  
用 `EXPLAIN FORMAT=JSON` 看 `cost_info`，对比索引扫描和全表扫描的 `read_cost` 和 `eval_cost`，能定量知道优化器为什么做了错误选择。如果成本估算与真实情况不符，就反推统计信息或参数问题。

**总结**：这种突发慢，我一般先看执行计划是否变差，再看统计信息和锁，因为 80% 的场景是统计信息不准导致优化器选错索引，或者被锁堵住。快速恢复后，再深入调整统计信息采样参数或改写 SQL 预防复发。”

---

这样回答能把**应急思路**和**原理深度**结合起来，面试官会觉得你既懂底层，又能实战，而且有章法不乱。
