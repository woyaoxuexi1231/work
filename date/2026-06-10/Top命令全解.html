实时动态监控

持续刷新显示系统进程、CPU、内存、负载等关键指标，默认每3秒更新一次。

交互式操作

运行中可通过快捷键排序、筛选、终止进程、调整优先级，无需退出界面。

批处理模式

支持 `-b` 批处理模式，可将输出重定向到文件或管道，适合脚本和自动化监控。

01 基本语法

top [选项]
# 常用组合示例
# 基本启动（默认每3秒刷新）
top
# 指定刷新间隔为1秒
top -d 1
# 按内存排序，显示线程
top -o %MEM -H

启动参数大全

在启动 top 时通过命令行传入的选项

参数 说明 示例
-d seconds 设置刷新间隔（默认3秒，支持小数如0.5） top -d 1
-n count 设置刷新次数后自动退出 top -n 10
-b 批处理模式，非交互式，适合脚本和日志记录 top -b -n 1
-u user 只显示指定用户的进程 top -u root
-p pid1,pid2 只监控指定的进程ID（最多20个） top -p 1234,5678
-H 显示线程模式（默认显示进程） top -H -p 1234
-o field 按指定字段排序启动（如 %CPU, %MEM） top -o %MEM
-i 不显示闲置（idle）和僵尸进程 top -i
-c 显示完整命令行（而非仅程序名） top -c
-1 显示每个CPU核心的单独使用情况 top -1
-s 安全模式，禁用交互式命令 top -s
-S 累积时间模式，包含已终止子进程的时间 top -S
-w [cols] 覆盖输出宽度（最大512列） top -w 512
-E k/m/g/t 设置内存摘要区域的显示单位 top -E g
-e k/m/g/t 设置进程列表区域的内存单位 top -e m

内存与交换空间

MiB Mem : 7936.2 total, 1234.5 free, 3456.7 used, 3245.0 buff/cache
MiB Swap: 2048.0 total, 2048.0 free, 0.0 used. 4123.4 avail Mem

物理内存

total (总量) 7936.2 MiB
free (空闲) 1234.5 MiB
used (已用) 3456.7 MiB
buff/cache (缓冲/缓存) 3245.0 MiB

交换空间 & 可用内存

Swap total (交换总量) 2048.0 MiB
Swap used (已用交换) 0.0 MiB
avail Mem (可用内存) 4123.4 MiB
无需交换即可分配给新进程的内存量（包含可回收缓存）
内存压力信号：Swap used 持续大于0且 wa 值升高，说明系统存在内存压力，可能需要增加物理内存或优化进程。

进程列表字段详解

每个列的含义、单位及诊断价值

字段 全称 说明 诊断提示
PID Process ID 进程唯一标识符 用于 kill、renice、strace 等操作
USER User Name 进程所有者 排查异常用户进程
PR Priority 进程优先级（数值越小优先级越高） rt = 实时任务；数值越低越优先
NI Nice Value nice值（-20~19，负值越高优先级） renice 调整此值控制CPU份额
VIRT Virtual Memory 虚拟内存总量（含交换、共享库、映射文件） 通常远大于实际物理内存使用
RES Resident Memory 物理内存实际使用量（常驻集） 重点关注！ 实际内存占用
SHR Shared Memory 共享内存大小（共享库、tmpfs等） 可被多个进程共享，RES包含SHR
S State 进程状态：R运行 S睡眠 D不可中断睡眠 T停止 Z僵尸 D状态进程无法被kill -9；Z需处理父进程
%CPU CPU Percentage CPU时间占用百分比 按P排序快速定位CPU大户
%MEM Memory Percentage 物理内存占用百分比 按M排序快速定位内存大户
TIME+ CPU Time 进程累计消耗的CPU时间（百分之一秒精度） CPU密集型进程的累计耗时
COMMAND Command 进程名称或完整命令行（按c切换） 完整路径有助于定位程序位置

场景一：CPU 使用率飙升

Web服务器响应变慢，怀疑某个进程占用过多CPU

1
按 CPU 排序
进入 top 后按 P 键，或启动时 top -o %CPU
2
检查单核饱和
按 1 查看每个CPU核心，确认是否为单线程瓶颈
3
查看线程级细节
按 H 切换线程模式，查看具体哪个线程占用CPU
4
获取完整命令行
按 c 显示完整路径，确认是预期中的进程
决策：如果是异常进程，按 k 输入PID，先发送信号 15 (SIGTERM) 优雅终止；若无效再用 9 (SIGKILL) 强制终止。

场景二：内存泄漏排查

系统可用内存持续下降，怀疑存在内存泄漏

1
按内存排序
按 M 键按 %MEM 排序，关注 RES 列（物理内存）
2
持续观察 RES 变化
观察目标进程的 RES 值是否持续单调增长（排除缓存波动）
3
查看线程内存分布
使用 top -H -p <PID> 查看各线程内存占用
关联检查：同时关注 Swap used 是否增长，以及 dmesg 中是否有 OOM (Out of Memory) 日志。结合 pmap -x <PID> 可查看进程内存映射详情。

场景三：负载高但CPU不高

load average 很高，但 %CPU 并不高

1
检查 wa (I/O等待)
如果 wa 值持续较高，说明存在磁盘或网络I/O瓶颈
2
检查 D 状态进程
查看进程列表中是否有大量 S 列状态为 D（不可中断睡眠）的进程，通常是等待I/O
3
进一步工具
使用 iostat -x 1、iotop 或数据库慢查询日志定位具体瓶颈
核心原理：Linux 负载不仅包含运行队列，还包含不可中断睡眠（D状态）的进程。高负载+低CPU+高wa = I/O瓶颈；高负载+低CPU+低wa = 锁竞争或网络等待。

场景四：僵尸进程处理

Tasks 行显示 zombie 进程数大于0

1
定位僵尸进程
在 top 中查看 S 列状态为 Z 的进程，记录其PID
2
查找父进程
使用 ps -o ppid= -p <ZOMBIE_PID> 获取父进程ID
3
处理父进程
僵尸进程无法直接kill，必须修复或重启其父进程。如果父进程是init(PID 1)，说明系统存在问题
根本原因：僵尸进程是已终止但父进程未调用 wait() 回收的子进程。需要检查应用程序代码中的子进程处理逻辑，或重启有问题的服务。
