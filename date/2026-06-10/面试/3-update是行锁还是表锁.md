这个问题要分存储引擎和具体场景来看，最直接的回答是：

**InnoDB 引擎下，`UPDATE` 通常是行锁，但在没有用到索引的情况下，会把整张表的所有行都锁住，效果等同于表锁。MyISAM 引擎下则永远只有表锁。**

下面展开讲一下，面试官想听的关键点都在这里：

### 1. InnoDB 的行锁：基于索引实现的
InnoDB 的行锁不是锁在数据行上，而是锁在**索引记录**上。
- 如果 `WHERE` 条件命中了索引（主键或二级索引），MySQL 会精确锁定扫描到的索引记录，这就是**行锁**。
- 例如：`UPDATE user SET name='x' WHERE id=1;`  使用主键，只锁 id=1 这一行。

### 2. 什么时候会“锁全表”？——没走索引
当 `WHERE` 条件中的列**没有索引**，或者 SQL 写得不合理导致索引失效（如隐式类型转换），InnoDB 无法通过索引快速定位行，只能**全表扫描**。这时它会：
- 在 **RR（可重复读）隔离级别**下，对所有扫描到的行加**临键锁（Next-Key Lock）**，也就是行锁 + 间隙锁，导致整张表除了极少数边界之外几乎全被锁住，任何插入和更新都可能被阻塞。
- 在 **RC（读已提交）隔离级别**下，虽然不加间隙锁，但仍会对所有扫描行加行锁，依然是全表锁定的效果。

这就是很多人口中的“行锁变表锁”的本质。

### 3. 意向锁不是锁表的锁
执行 `UPDATE` 时，InnoDB 会在表级加一个**意向排他锁（IX）**，但它只是一个标识，用来和 LOCK TABLES 等表级锁协调，并不阻塞其他行级操作，所以不能算真正的表锁。

### 4. MyISAM 引擎：只有表锁
如果是 MyISAM 表，执行 `UPDATE` 时会直接对整个表加**写锁**，其他所有读写操作全部阻塞，直到更新结束。这也是生产环境几乎不用 MyISAM 的原因之一。

### 5. 如何避免“行锁变表锁”？
- **确保 `UPDATE` 走索引**：执行前用 `EXPLAIN` 看一下，`type` 不能是 `ALL`（全表扫描），`key` 列必须有值。
- **分批更新**：即使走了索引，一次锁太多行也会造成大范围阻塞，用 `LIMIT` 分批做。
- **调整隔离级别**：如果业务允许，RC 级别可以减少间隙锁，降低锁冲突概率，但无法解决无索引导致的锁全表问题。

**一句话总结：InnoDB 的 `UPDATE` 基于索引加行锁，没索引就等于锁全表；MyISAM 只能表锁。**
