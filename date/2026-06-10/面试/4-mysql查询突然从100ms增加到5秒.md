面对这个问题，面试官其实是想考察你**系统性排查性能突变**的能力，而不是只背一个“可能索引失效”的答案。一个从 100ms 突然跳到 5 秒的查询，核心线索在“突然”二字——它暗示大概率是**环境、数据或执行计划在某个时间点发生了突变**，而不是单纯的数据量平缓增长。

下面给你一个可以直接拿去用的回答框架，按这个逻辑说下来，会显得思路非常清楚。

---

### 回答时，可以这样分层展开

**1. 先快速确认 SQL 本身是否发生变化**  
“我会先确认这条 SQL 语句本身有没有被改过，比如是不是程序发布新增了函数包装、动态拼接了不同条件，或者被 ORM 框架修改了生成逻辑。如果 SQL 变了，那就直接对比新旧语句。”

**2. 如果 SQL 没变，立刻看执行计划**  
“这是最关键的一步。我会用 `EXPLAIN` 看当前执行计划，重点是：是否从走索引变成了全表扫描，或者索引选择是否发生了变化。  
这种执行计划突变，最常见的原因是**统计信息过期**，优化器对行数估算错误，导致它放弃了原来高效的索引，选了一个糟糕的执行路径。  
我会用 `EXPLAIN FORMAT=JSON` 看细节，或者用 `SHOW STATUS LIKE 'Last_query_cost'` 看代价值。如果确认是统计信息问题，先执行 `ANALYZE TABLE` 更新统计信息，看性能是否恢复；紧急情况下可以用 `FORCE INDEX` 临时固定索引。”

**3. 排查锁与阻塞**  
“如果执行计划没问题，我会怀疑这条查询是不是被阻塞了 5 秒。比如：
- 有没有未提交的长事务，持有行锁或间隙锁，把它卡住；
- 有没有人在同时做 DDL，导致等待元数据锁 (MDL)；
- 或者有没有 `FLUSH TABLES` 等操作。  
  我会执行 `SELECT * FROM sys.innodb_lock_waits`（或 8.0 的 `performance_schema.data_lock_waits`）看锁等待，结合 `SHOW ENGINE INNODB STATUS` 里的 `LATEST DETECTED DEADLOCK` 和事务列表来找阻塞源。如果是阻塞，就需要终止持有锁的源头连接。”

**4. 检查系统资源与 Buffer Pool 变化**  
“如果锁也没问题，我会转向资源层。
- 查看磁盘 I/O 是否突然飙升或出现毛刺，可以用 `iostat`，或者 MySQL 内部 `SHOW GLOBAL STATUS` 里的 `Innodb_buffer_pool_read_requests` 与 `Innodb_buffer_pool_reads` 的比率，看缓冲池命中率是否骤降。
- 比如 Buffer Pool 不够用，数据被大量换出，导致查询突然大量物理读，5 秒里可能 4.9 秒都在等磁盘。  
  也会检查是不是在慢的时间点刚好有备份、全量统计或大批量数据导入任务，把资源打满。”

**5. 关注数据和表结构的变更**  
“我还会确认：
- 这段时间内，表是否有大量 `UPDATE`、`DELETE` 或 `INSERT`，导致数据分布变化或产生大量碎片，间接影响执行计划或扫描效率；
- 是否有索引被误删除、约束改变，或者 `innodb_buffer_pool_size` 等关键参数被在线调小；
- 甚至查看 MySQL 错误日志，看有没有 OOM、磁盘写满等告警。”

**6. 善用历史和监控工具**  
“如果有慢查询日志和 `performance_schema`，我会直接查 `events_statements_summary_by_digest` 或 `events_statements_history`，对比突变前后该 SQL 的 `AVG_TIMER_WAIT`、锁时间、扫描行数等指标，这样很容易定位到突然变大的那项。再配合 `pt-query-digest` 分析慢日志，能很快缩小范围。”

---

### 面试中可以加分的总结收尾

“总结一下我的排查路径：
先看 SQL 本身 → 再查执行计划与统计信息 → 然后查锁阻塞 → 接着看系统和资源指标 → 最后回顾数据/配置变更。  
而且这类问题平时可以通过 **自动收集统计信息、设置 SQL 计划基线（SQL Plan Baseline）、增加查询监控告警** 来预防，比如索引使用变化和查询时间突增的报警，能让我们在 5 秒变 1 分钟之前就发现。”

---

这个回答既体现了你**有清晰的排查链路**（逻辑性），也抓住了“突然”这一关键点（敏感性），还顺带展示了你对锁、统计信息、资源、监控工具的熟悉程度，会是一个很扎实的面试表现。
