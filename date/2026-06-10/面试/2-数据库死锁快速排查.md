这道题其实是在考察你对锁机制的掌握，以及线上应急排查的实战能力。回答时最好能体现出清晰的排查路径和对日志的解读能力。

---

数据库死锁的快速排查，我通常分三步走：**确认现场 → 获取并分析死锁日志 → 定位原因并优化**。下面以最常见的 MySQL InnoDB 为例：

### 1. 确认是否发生了死锁
**死锁是瞬间的，InnoDB 会自动检测并回滚其中一个事务**，错误日志里会有记录。我一般会先看这两个地方：

- **`SHOW ENGINE INNODB STATUS\G`**  
  输出的 `LATEST DETECTED DEADLOCK` 部分会保留**最近一次**死锁的完整信息。如果死锁不频繁，这一招就能直接看到。
- **错误日志**  
  线上通常会开启 `innodb_print_all_deadlocks = ON`，这样**每一次死锁**都会详细打印到 error log，方便事后检索。否则死锁信息容易被后续新的死锁覆盖掉。我上去会直接 `grep "DEADLOCK" error.log`，或去日志平台搜关键字。

### 2. 解读死锁日志，找出互相等待的锁
死锁日志里最关键的两部分是 **事务1** 和 **事务2** 的信息。我会这样看：

```
*** (1) TRANSACTION:
    UPDATE t1 SET ... WHERE id = 10;  -- 正在等待什么锁
    (持有 S 锁 ...)
*** (1) WAITING FOR THIS LOCK TO BE GRANTED:
    RECORD LOCKS ... index PRIMARY of t1 ... lock_mode X locks rec but not gap waiting
*** (2) TRANSACTION:
    UPDATE t1 SET ... WHERE id = 10;
    (持有 X 锁 ...)
*** (2) HOLDS THE LOCK(S):
    RECORD LOCKS ... index PRIMARY of t1 ... lock_mode X locks rec but not gap
*** (2) WAITING FOR THIS LOCK TO BE GRANTED:
    ...
*** WE ROLL BACK TRANSACTION (1)
```

核心思路是：**看哪个事务持有了锁，又在等哪个锁，形成循环**。比如上面示例就是：事务2持有了记录 X 锁，想要另一个锁；事务1想要事务2持有的那个 X 锁，形成死锁，最终回滚了事务1。

**关键信息点**：
- 事务编号和 SQL 语句。
- 锁的类型：行锁（`locks rec but not gap`）、间隙锁（`lock_mode X locks gap before rec`）、插入意向锁等，能判断是间隙锁冲突还是行锁冲突。
- 涉及的索引名（`index PRIMARY` 还是二级索引），能看出是否因为索引缺失导致锁范围扩大。

### 3. 如果死锁正在发生，如何看当前锁等待链？
有时候线上是严重的**锁等待阻塞**（还没构成死锁循环），我需要快速看是谁阻塞了谁：

- **MySQL 5.7**：查 `information_schema.INNODB_LOCK_WAITS` 视图，配合 `INNODB_TRX` 和 `INNODB_LOCKS`。
- **MySQL 8.0**：直接用 `performance_schema.data_lock_waits` 和 `data_locks`，可以非常清晰地画出锁等待链：从 `WAITING_ENGINE_LOCK_ID` 追溯到 `BLOCKING_ENGINE_LOCK_ID`，找到源头阻塞者，再找出它的 `THREAD_ID`、SQL 文本。
- **快捷命令**：在 8.0 里可以直接 `SELECT * FROM sys.innodb_lock_waits;` 看到阻塞信息和被阻塞的 SQL。

### 4. 常见死锁原因与快速预防
根据日志分析，死锁无外乎这几种典型场景，排查方向非常明确：

- **多表操作顺序不一致**  
  比如事务A先更新订单表再更新库存表，事务B反过来。  
  **解决**：统一所有事务的访问顺序，或者将关键操作串行化。
- **索引缺失导致锁范围放大**  
  比如 `UPDATE` 的 `WHERE` 条件没走索引，导致扫描全表并加一堆间隙锁，极易跟其他插入语句发生死锁。  
  **解决**：看日志里等待的索引是不是“全表扫描”的聚簇索引，补上合适的索引即可。
- **大事务中的间隙锁冲突**  
  一个事务插入一条记录并持有了间隙锁，另一个事务也想插入同一间隙的记录，死锁。  
  **解决**：缩短事务，或用乐观锁/排队机制代替 `SELECT ... FOR UPDATE`。
- **唯一键冲突引起的死锁**  
  多个事务同时插入相同的唯一键值时，产生共享排他锁的等待。  
  **解决**：业务上去重，或使用 `INSERT ... ON DUPLICATE KEY UPDATE` 缩短加锁过程。

### 5. 应急处理与监控
如果死锁频繁导致业务报错，临时应急可以：
- 用 `KILL` 命令终止源头阻塞事务（但死锁本身已经自动回滚了，这只针对锁等待）。
- 在监控上加死锁计数器，比如 `SHOW GLOBAL STATUS LIKE 'innodb_deadlocks'`，一旦有增长就告警，配合日志快速介入。

---

**总结一句话：**
快速排查死锁就是“**看日志、读循环、找 SQL、看索引**”，InnoDB 已经把“案发现场”详细记录在了死锁日志里，我们要做的就是第一时间拿到它，然后从持锁和等锁的循环关系反推事务访问顺序和索引问题。
