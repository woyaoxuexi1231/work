MVC 请求生命周期

一个请求从前端发起，到拿到结果，后端经历了什么？ 基于 Spring MVC 源码级分析 · 附本项目实战对照

🧩 10 个核心阶段 ⚡ 6 个关键组件 🛡️ 2 条异常路径
💡

一句话概括

客户端发起 HTTP 请求后，DispatcherServlet（前端控制器）接管请求， 委派给各个组件协同处理：HandlerMapping 路由、HandlerAdapter 适配、 ArgumentResolver 解析参数、Controller 执行业务、 ResponseBodyAdvice 统一包装、HttpMessageConverter 序列化输出。 全程贯穿 @ControllerAdvice 异常拦截。

①

前端发起 HTTP 请求

Browser / Curl / Axios

用户点击、表单提交或前端框架（Vue/React）通过 Ajax 发出 HTTP 请求。 包含请求方法（GET/POST）、URL、Headers（Content-Type、Accept）、Body 等。

// GET 请求示例 GET /users?keyword=zhang HTTP/1.1 Host: localhost:8080 Accept: application/json
// POST 请求示例 POST /users HTTP/1.1 Content-Type: application/json {"name": "张三"}
②

DispatcherServlet 接收请求

前端控制器 · Front Controller

Spring MVC 的入口。所有请求经 web.xml（或 Spring Boot 自动注册）映射到 DispatcherServlet。 其在初始化（initStrategies()）时装配了所有核心组件。

HandlerMapping HandlerAdapter HandlerExceptionResolver ViewResolver HttpMessageConverter
// DemoApplication.java — 启动日志可见组件注册 @SpringBootApplication public class DemoApplication { public static void main(String[] args) {   SpringApplication.run(DemoApplication.class, args); // 日志: RequestMappingHandlerMapping: Mapped "{[/users],methods=[GET]}" } }
③

HandlerMapping 路由查找

URL → Controller 映射

DispatcherServlet.doDispatch() 调用 getHandler()， 遍历所有 HandlerMapping 寻找能处理该 URL 的处理器。 最常用的是 RequestMappingHandlerMapping —— 它维护着 @RequestMapping / @GetMapping 等注解的路由表。

// 路由表注册结果（启动日志） Mapped "{[/users],methods=[GET]}" →   listUsers(...) Mapped "{[/users],methods=[POST]}" →   createUser(...) Mapped "{[/users/profile],methods=[GET]}" →   profile(...)
// UserController.java — 路由定义 @RestController @RequestMapping("/users") public class UserController {   @GetMapping   public List<User> listUsers(     @RequestParam(defaultValue = "") String keyword) { ... } }
④

HandlerAdapter 适配执行

适配器模式

找到 Handler 之后，getHandlerAdapter() 根据 Handler 类型选择适配器。 对于 @Controller / @RestController， 选 RequestMappingHandlerAdapter。适配器统一了不同 Handler 的调用方式， 然后调用 ha.handle() 进入参数解析 + 方法执行阶段。

🔌 HandlerAdapter 是 MVC 中的"适配器模式"体现 —— 不管你的 Controller 是旧式 Controller 接口、HttpRequestHandler 还是注解驱动，适配器统一了调用入口。
⑤

ArgumentResolver 参数解析

HTTP → Java 方法参数

HandlerAdapter 调用 invokeHandlerMethod() 时， InvocableHandlerMethod 遍历所有 HandlerMethodArgumentResolver， 对 Controller 方法的每个参数逐个匹配、解析。Spring 内置了 20+ 个解析器， 也支持通过 WebMvcConfigurer.addArgumentResolvers() 注册自定义解析器。

注解 / 参数 解析器 来源
@RequestParam RequestParamMethodArgumentResolver Spring 内置
@RequestBody RequestResponseBodyMethodProcessor Spring 内置
@PathVariable PathVariableMethodArgumentResolver Spring 内置
@CurrentUser CurrentUserArgumentResolver 自定义 ✨
// 自定义解析器必须先注册到 WebMvcConfigurer！ @Configuration public class WebMvcConfig implements WebMvcConfigurer {   @Override   public void addArgumentResolvers(     List<HandlerMethodArgumentResolver> resolvers) {     resolvers.add(new CurrentUserArgumentResolver()); // ✅ 注册   } } // 不注册 → 解析器不生效（面试高频踩坑点）
⑥

Controller 执行业务逻辑

核心业务层

参数解析完成后，通过反射调用 Controller 的目标方法，执行业务逻辑。 方法可能调用 Service / DAO 层处理数据，然后返回结果（如 User、List<User>）。

// UserController.java — Controller 被执行 @GetMapping public List<User> listUsers(   @RequestParam(defaultValue = "") String keyword) {   log.info("📥 查询用户列表, keyword={}", keyword);   return Arrays.asList(     new User(1L, "张三", "zhangsan@example.com"),     new User(2L, "李四", "lisi@example.com")   ); // 返回 List<User> }
⑦

ReturnValueHandler + ResponseBodyAdvice

序列化前拦截

Controller 方法返回后，HandlerMethodReturnValueHandler 处理返回值。 对于 @ResponseBody（@RestController 自带）， 由 RequestResponseBodyMethodProcessor 处理。 这里 GlobalResponseBodyAdvice 会拦截返回值， 将原始对象（如 User）统一包装成 ApiResult { code, message, data } 格式。

// GlobalResponseBodyAdvice.java @RestControllerAdvice public class GlobalResponseBodyAdvice   implements ResponseBodyAdvice<Object> {   @Override   public Object beforeBodyWrite(...) {     return ApiResult.success(body); // 统一包装   } }
// 包装前（Controller 返回） { "id": 1, "name": "张三", ... } // 包装后（前端收到的格式） {   "code": 200,   "message": "success",   "data": { "id": 1, "name": "张三" } }
⑧

HttpMessageConverter 内容协商与序列化

序列化 · 内容协商

RequestResponseBodyMethodProcessor 调用 writeWithMessageConverters()， 遍历所有已注册的 HttpMessageConverter， 根据 Accept 请求头（内容协商 Content Negotiation）和返回值类型， 选择第一个 canWrite() 返回 true 的 Converter 进行序列化。

Converter 支持的 MediaType 说明
MappingJackson2HttpMessageConverter application/json 默认 JSON 序列化 ✅
StringHttpMessageConverter text/plain 字符串处理
CsvHttpMessageConverter ✨自定义 text/csv Accept: text/csv 时触发
🌐
内容协商演示： curl -H "Accept: text/csv" http://localhost:8080/negotiation/users → 返回 CSV 格式！同一个 URL，不同 Accept 头得到不同格式。
✓

HTTP 响应返回前端

正常结束

序列化后的字节流写入 HTTP 响应体，带上 Content-Type、 HTTP Status（200/201/400/500）等响应头， 通过 Servlet 容器（Tomcat）返回给客户端。前端（浏览器 / Axios / Curl）拿到响应结果进行渲染或处理。

// 前端收到的最终 HTTP 响应 HTTP/1.1 200 OK Content-Type: application/json ... {   "code": 200,   "message": "success",   "data": [     { "id": 1, "name": "张三", "email": "zhangsan@example.com" },     { "id": 2, "name": "李四", "email": "lisi@example.com" }   ] }
异常分支：@ExceptionHandler 拦截
⚠

@ExceptionHandler 全局异常拦截

@ControllerAdvice · AOP

若 Controller 方法执行过程中抛出异常（如参数校验失败、空指针），DispatcherServlet 在 processHandlerException() 中 遍历所有 HandlerExceptionResolver。 @ControllerAdvice 标注的 GlobalExceptionHandler 通过 @ExceptionHandler 匹配异常类型，返回统一的 ApiResult 错误响应。

// GlobalExceptionHandler.java @RestControllerAdvice public class GlobalExceptionHandler {   @ExceptionHandler(IllegalArgumentException.class)   @ResponseStatus(HttpStatus.BAD_REQUEST) // 400   public ApiResult<?> handleIllegalArgument(...) {     return ApiResult.error(400, "参数校验失败");   }   @ExceptionHandler(RuntimeException.class)   @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)   public ApiResult<?> handleRuntime(...) { ... } }
// 异常响应示例 HTTP/1.1 400 Bad Request {   "code": 400,   "message": "参数校验失败: 用户 ID 必须大于 0",   "data": null } // 前端收到统一错误结构，统一处理
